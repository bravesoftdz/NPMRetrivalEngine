    <h1 class="heading1" title="2. Modules &amp; Dependency Resolution"><span class="headingNumber"><a href="#2" title="2. Modules &amp; Dependency Resolution">2</a>. </span>Modules &amp; Dependency Resolution</h1> 
    <div class="contents1"> 
     <p>UIZE implements an elegant and clean system for defining JavaScript modules of many different varieties.</p> 
     <p>Defining a module in UIZE is as simple as calling the <code>Uize.module</code> method, as follows...</p> 
     <pre class="sample-code">Uize.module ({
  name:'MyNamespace.MyModule',
  required:[
    'Uize.Color',
    'Uize.Data.Matches',
    'MyNamespace.MyOtherModule'
  ],
  builder:function () {
    var myModule;

    // do stuff to initialize myModule
    // ... ... ... ... ... ... ...
    // ... ... ... ... ... ... ...

    return myModule
  }
});
</pre> 
     <a name="2_1"></a> 
     <h2 class="heading2" title="2. Modules &amp; Dependency Resolution -> 2.1. Dependency Resolution"><span class="headingNumber"><a href="#2" title="2. Modules &amp; Dependency Resolution">2</a>.<a href="#2_1" title="2. Modules &amp; Dependency Resolution -> 2.1. Dependency Resolution">1</a>. </span>Dependency Resolution</h2> 
     <div class="contents2"> 
      <p>The way that the module system works, any modules that are declared as direct dependencies in the <code>required</code> property of the module definition are loaded first.</p> 
      <p>The <code>builder</code> function for the module is called only once all dependencies - direct or indirect - are loaded. This means that you never have to worry about dependencies of other modules and figuring out comprehensive lists of all the code that a page might need. If you follow the practice of always declaring just your own direct dependencies for the module you're working on, then UIZE does all the recursive dependency tracing for you and the module loader takes care of loading what's still needed (not already loaded because something else had needed it).</p> 
     </div> 
     <a name="2_2"></a> 
     <h2 class="heading2" title="2. Modules &amp; Dependency Resolution -> 2.2. Built Into the Base Module"><span class="headingNumber"><a href="#2" title="2. Modules &amp; Dependency Resolution">2</a>.<a href="#2_2" title="2. Modules &amp; Dependency Resolution -> 2.2. Built Into the Base Module">2</a>. </span>Built Into the Base Module</h2> 
     <div class="contents2"> 
      <p>The system for loading module dependencies is built into the base <code>Uize.js</code> file, so all that's needed to bootstrap a page for basic development is to add a script tag to load that file.</p> 
      <pre class="sample-code">&lt;script src="http://www.uize.com/js/Uize.js"&gt;&lt;/script&gt;

&lt;script type="text/javascript"&gt;
  Uize.require (
    'Uize.Widgets.Calculator.Widget',
    function () {Uize.Widgets.Calculator.Widget ().insertUi ()}
  );
&lt;/script&gt;
</pre> 
      <p>If you copy-and-paste the above code into some HTML page (maybe you've got one you're working on), you will see a glorious calculator widget added to the page. Because of the highly modular nature of the UIZE codebase, there's actually a bunch of individual modules that will be loaded. All this is taken care of for you.</p> 
     </div> 
     <a name="2_3"></a> 
     <h2 class="heading2" title="2. Modules &amp; Dependency Resolution -> 2.3. Packaging"><span class="headingNumber"><a href="#2" title="2. Modules &amp; Dependency Resolution">2</a>.<a href="#2_3" title="2. Modules &amp; Dependency Resolution -> 2.3. Packaging">3</a>. </span>Packaging</h2> 
     <div class="contents2"> 
      <p>Rest assured, when it comes time to get a project ready for production, UIZE provides ways to build JavaScript packages using the same dependency resolution system so that you can reduce the number of HTTP requests made by your application and thereby improve its load time.</p> 
      <p>You can either rely on the built-in UIZE support for building JavaScript library files, or you can craft your own more sophisticated packager using the raw build methods for tracing dependencies of modules. And, because UIZE supports <a href="#2_4">all kinds of modules</a>, you can even build your <a href="#11_1">HTML templates</a> and <a href="#11_2">CSS templates</a> for UI widgets into the same JavaScript package as your JavaScript logic.</p> 
     </div> 
     <a name="2_4"></a> 
     <h2 class="heading2" title="2. Modules &amp; Dependency Resolution -> 2.4. All Kinds of Modules"><span class="headingNumber"><a href="#2" title="2. Modules &amp; Dependency Resolution">2</a>.<a href="#2_4" title="2. Modules &amp; Dependency Resolution -> 2.4. All Kinds of Modules">4</a>. </span>All Kinds of Modules</h2> 
     <div class="contents2"> 
      <p>Because UIZE's module mechanism is very generic in nature (you can define anything in your <code>builder</code> function), it lends itself to being used as a wrapper for all types of content.</p> 
      <p>In the UIZE widgets, for example, JavaScript modules are used to wrap <a href="#11_1">HTML templates</a> and <a href="#11_2">CSS templates</a>, allowing all the materials needed by a widget to be packaged into the same file (even the images if you implement <a href="http://en.wikipedia.org/wiki/Base64" target="_blank" class="externalSiteLink">base64 encoding</a> for inlining of images in the wrapped CSS modules).</p> 
      <p>Different flavors of JavaScript modules include...</p> 
      <table class="list"> 
       <tbody>
        <tr valign="top">
         <td style="white-space:nowrap;"><span class="bullet"></span> </td>
         <td>class modules</td>
        </tr> 
        <tr valign="top">
         <td style="white-space:nowrap;"><span class="bullet"></span> </td>
         <td>package modules</td>
        </tr> 
        <tr valign="top">
         <td style="white-space:nowrap;"><span class="bullet"></span> </td>
         <td>JSON / data modules</td>
        </tr> 
        <tr valign="top">
         <td style="white-space:nowrap;"><span class="bullet"></span> </td>
         <td>CSS template modules</td>
        </tr> 
        <tr valign="top">
         <td style="white-space:nowrap;"><span class="bullet"></span> </td>
         <td>HTML template modules</td>
        </tr> 
        <tr valign="top">
         <td style="white-space:nowrap;"><span class="bullet"></span> </td>
         <td>JavaScript library modules</td>
        </tr> 
       </tbody>
      </table> 
      <p>What's more, using the <a href="#18">powerful build system</a> and the <a href="#20">development server</a>, you can create your own URL handlers for new types of source files that should be ultimately wrapped in the form of JavaScript modules.</p> 
     </div>

     <div class="contents2"> 
      <p>While it's nice to have some of these types of utility methods always close at hand, we don't want the <code><a href="reference/Uize.html">Uize</a></code> base module to become bloatware and so we haven't added some of the more esoteric utilities that you might find in some other popular toolbelt JavaScript libraries.</p> 
      <p>The utility methods that make their way into the <code><a href="reference/Uize.html">Uize</a></code> base module are given careful consideration, and are often given the greenlight because they are needed by other core modules that are very likely to be used, so why not make them public for all to use.</p> 
     </div>

    <h1 class="heading1" title="4. Classes, Inheritance, OOP"><span class="headingNumber"><a href="#4" title="4. Classes, Inheritance, OOP">4</a>. </span>Classes, Inheritance, OOP</h1> 
    <div class="contents1"> 
     <p>UIZE makes it dead easy to create classes that come with support for events (thanks to a built-in <a href="#6">event infrastructure</a>) and <a href="#5">state properties and binding</a>.</p> 
     <a name="4_1"></a> 
     <h2 class="heading2" title="4. Classes, Inheritance, OOP -> 4.1. Creating a Class"><span class="headingNumber"><a href="#4" title="4. Classes, Inheritance, OOP">4</a>.<a href="#4_1" title="4. Classes, Inheritance, OOP -> 4.1. Creating a Class">1</a>. </span>Creating a Class</h2> 
     <div class="contents2"> 
      <p>Making a class in UIZE is as easy as calling the <code>subclass</code> static method of a class that you want to extend, as follows...</p> 
      <pre class="sample-code">var MyWidgetClass = Uize.Widget.V2.subclass ();
</pre> 
      <p>In the above example, <code>MyWidgetClass</code> is now a subclass of UIZE's V2 (version 2) widget base class. The new <code>MyWidgetClass</code> class inherits all of the instance methods and property, state properties, and inheritable static methods and properties from the <code><a href="reference/Uize.Widget.V2.html">Uize.Widget.V2</a></code> base class.</p> 
     </div> 
     <a name="4_2"></a> 
     <h2 class="heading2" title="4. Classes, Inheritance, OOP -> 4.2. Declaring Features"><span class="headingNumber"><a href="#4" title="4. Classes, Inheritance, OOP">4</a>.<a href="#4_2" title="4. Classes, Inheritance, OOP -> 4.2. Declaring Features">2</a>. </span>Declaring Features</h2> 
     <div class="contents2"> 
      <p>Declaring features for a newly created class can be done in a number of different ways, but by far the simplest and most elegant is to declare all the new features or overridden features in an optional features declaration object that is passed to the <code>subclass</code> method, as follows...</p> 
      <pre class="sample-code">var MyClass = Uize.Class.subclass ({
  instanceMethods:{
    someInstanceMethod1:function () {
      // do some stuff
    },
    someInstanceMethod2:function () {
      // do some stuff
    }
  },
  staticMethods:{
    someStaticMethod1:function () {
      // do some stuff
    },
    someStaticMethod2:function () {
      // do some stuff
    }
  }
});
</pre> 
     </div> 
     <p>For a more detailed discussion of the UIZE approach to OOP, consult the <a href="guides/classes-and-inheritance.html">Classes and Inheritance</a> guide.</p> 
    </div> 
    <a name="5"></a> 
    <h1 class="heading1" title="5. State Properties and Binding"><span class="headingNumber"><a href="#5" title="5. State Properties and Binding">5</a>. </span>State Properties and Binding</h1> 
    <div class="contents1"> 
     <p>UIZE offers a mature and powerful state properties system so that classes can expose a state interface for their instances.</p> 
     <p>The state properties system goes beyond regular object properties, and even beyond the newer <a href="https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Object/defineProperty" target="_blank" class="externalSiteLink">properties capabilities of ECMAScript 5</a>, providing powerful constructs like conformers, optimized onChange handlers, aliases (for deprecating old names), and value changed events using the built-in <a href="#6">event infrastructure</a>.</p> 
     <p><span class="allCaps">EXAMPLE</span></p> 
     <pre class="sample-code">function calculateVolume () {
  this.set ({volume:this.get ('width') * this.get ('depth') * this.get ('height')});
}

var Room = Uize.Class.subclass ({
  stateProperties:{
    width:{
      value:10,
      conformer:function (newValue) {return Uize.constrain (newValue,10,50)},
      onChange:calculateVolume
    },
    depth:{
      value:10,
      conformer:function (newValue) {return Uize.constrain (newValue,10,50)},
      onChange:calculateVolume
    },
    height:{
      value:10,
      conformer:function (newValue) {return Uize.constrain (newValue,9,25)},
      onChange:calculateVolume
    },
    volume:{
      onChange:function () {
        // code to display volume info in UI
      }
    }
  }
});
</pre> 
     <p>In this example we're creating a <code>Room</code> class with the state properties <code>width</code>, <code>depth</code>, <code>height</code>, and <code>volume</code>.</p> 
     <a name="5_1"></a> 
     <h2 class="heading2" title="5. State Properties and Binding -> 5.1. Range Type Conformers"><span class="headingNumber"><a href="#5" title="5. State Properties and Binding">5</a>.<a href="#5_1" title="5. State Properties and Binding -> 5.1. Range Type Conformers">1</a>. </span>Range Type Conformers</h2> 
     <div class="contents2"> 
      <p>The <code>width</code> and <code>depth</code> properties both have a conformer that constrains their value to a range of <code>10</code> to <code>50</code>, while the <code>height</code> property has a conformer that constrains its value to a range of <code>9</code> to <code>25</code>.</p> 
     </div> 
     <a name="5_2"></a> 
     <h2 class="heading2" title="5. State Properties and Binding -> 5.2. Optimized onChange Handler"><span class="headingNumber"><a href="#5" title="5. State Properties and Binding">5</a>.<a href="#5_2" title="5. State Properties and Binding -> 5.2. Optimized onChange Handler">2</a>. </span>Optimized onChange Handler</h2> 
     <div class="contents2"> 
      <p>All three of these properties share the same reference to the <code>calculateVolume</code> function, which gets called only once during a set if any or all of the properties change value during the set (this is a smart optimization of UIZE).</p> 
      <p>The <code>calculateVolume</code> function gets called as though an instance method (i.e. with the instance as context), so it can use the <code>this</code> keyword. It uses the <code>get</code> instance method to obtain the current values of the three dimension properties to calculate the volume of the room and then sets a new value on the <code>volume</code> state property, making this property effectively a derived property.</p> 
     </div> 
     <a name="5_3"></a> 
     <h2 class="heading2" title="5. State Properties and Binding -> 5.3. Looking at Some Outcomes"><span class="headingNumber"><a href="#5" title="5. State Properties and Binding">5</a>.<a href="#5_3" title="5. State Properties and Binding -> 5.3. Looking at Some Outcomes">3</a>. </span>Looking at Some Outcomes</h2> 
     <div class="contents2"> 
      <p>Now let's take a look at what happens when we try setting the values of the state properties in a few different situations.</p> 
      <pre class="sample-code">var livingRoom = Room ({width:15,depth:20,height:19}); // initialized with values for properties
alert (livingRoom.get ('volume'));              // alerts 5700

livingRoom.set ({width:15,depth:20,height:19}); // no change in properties, volume unchanged
alert (livingRoom.get ('volume'));              // alerts 5700

livingRoom.set ({width:18,depth:18,height:22}); // volume is only calculated once
alert (livingRoom.get ('volume'));              // alerts 7128

livingRoom.set ({width:55,depth:60,height:30}); // property values are constrained at upper bounds
alert (livingRoom.get ('width'));               // alerts 50
alert (livingRoom.get ('depth'));               // alerts 50
alert (livingRoom.get ('height'));              // alerts 25
alert (livingRoom.get ('volume'));              // alerts 62500

livingRoom.set ({width:99,depth:99,height:99}); // properties remain constrained, volume unchanged
alert (livingRoom.get ('width'));               // alerts 50
alert (livingRoom.get ('depth'));               // alerts 50
alert (livingRoom.get ('height'));              // alerts 25
alert (livingRoom.get ('volume'));              // alerts 62500
</pre> 
      <p>There are effectively five different places where we're setting values for the state properties: once as part of instance creation, and then on four subsequent occasions...</p> 
      <table class="list"> 
       <tbody>
        <tr valign="top">
         <td style="white-space:nowrap;">1. &nbsp;</td>
         <td>Immediately after the <code>livingRoom</code> instance is created, the value of the <code>volume</code> state property is <code>5700</code>, being calculated during construction of the instance from the values set for the <code>width</code>, <code>depth</code>, and <code>height</code> properties.</td>
        </tr> 
        <tr valign="top">
         <td style="white-space:nowrap;">2. &nbsp;</td>
         <td>Next, the <code>set</code> method is called in an attempt to set the values of the dimension properties to their exact current values. This results in no change in the dimension property values, the <code>calculateVolume</code> function is not called for any of the <code>onChange</code> handlers, and the value of the <code>volume</code> property remains unchanged.</td>
        </tr> 
        <tr valign="top">
         <td style="white-space:nowrap;">3. &nbsp;</td>
         <td>Next, the <code>set</code> method is called and different values are specified for each of the dimension properties. This results in the shared <code>calculateVolume</code> change handler function being called only once, and the value of the <code>volume</code> property is now set to <code>7128</code>.</td>
        </tr> 
        <tr valign="top">
         <td style="white-space:nowrap;">4. &nbsp;</td>
         <td>Next, the <code>set</code> method is called and values are specified for the dimension properties that are all outside of the range bounds enforced by their <code>conformer</code> functions. This results in the values being constrained at the upper limits of the ranges and there is a change in the values of all the properties. The shared <code>calculateVolume</code> change handler function is still called only once, and the value of the <code>volume</code> property is now set to <code>62500</code>.</td>
        </tr> 
        <tr valign="top">
         <td style="white-space:nowrap;">5. &nbsp;</td>
         <td>Finally, the <code>set</code> method is called once again in an attempt to set values for the dimension properties that are above the upper range bounds enforced by their <code>conformer</code> functions. This results in the values being constrained at the upper limits of the ranges, but this time there is no change in the values of the properties because they were already constrained at their upper limits. The shared <code>calculateVolume</code> change handler function is not called (none of the properties' values have changed), and the value of the <code>volume</code> property remains unchanged at <code>62500</code>.</td>
        </tr> 
       </tbody>
      </table> 
     </div> 
     <a name="5_4"></a> 
     <h2 class="heading2" title="5. State Properties and Binding -> 5.4. Changed Events"><span class="headingNumber"><a href="#5" title="5. State Properties and Binding">5</a>.<a href="#5_4" title="5. State Properties and Binding -> 5.4. Changed Events">4</a>. </span>Changed Events</h2> 
     <div class="contents2"> 
      <p>Every state property that is declared for a class gets the benefit of changed events that fire automatically when the value of the state property changes.</p> 
      <p><span class="allCaps">EXAMPLE</span></p> 
      <pre class="sample-code">var MyClass = Uize.Class.subclass ({
  stateProperties:{
    foo:{value:'bar'}
  }
});

var myClassInstance = MyClass ();

myClassInstance.wire (
  'Changed.foo',
  function () {alert (myClassInstance.get ('foo'))}
);

myClassInstance.set ({foo:'qux'});
</pre> 
      <p>In this example, we've created the class <code>MyClass</code> and declared that it has a state property named <code>foo</code> with an initial value of <code>'bar'</code>. After creating an instance of this class, <code>myClassInstance</code>, we wire a handler for the <code>Changed.foo</code> event, which is an event that will automatically be fired by UIZE whenever the value of the <code>foo</code> state property changes. After wiring the handler, we use the <code>set</code> method to set the <code>foo</code> property to the new value <code>'qux'</code>. This causes the <code>Changed.foo</code> event to be fired and the <code>alert</code> statement is hit, which alerts the new value <code>'qux'</code>.</p> 
     </div> 
     <a name="5_5"></a> 
     <h2 class="heading2" title="5. State Properties and Binding -> 5.5. Binding Properties Together"><span class="headingNumber"><a href="#5" title="5. State Properties and Binding">5</a>.<a href="#5_5" title="5. State Properties and Binding -> 5.5. Binding Properties Together">5</a>. </span>Binding Properties Together</h2> 
     <div class="contents2"> 
      <p>UIZE providers numerous utility modules for conveniently binding state properties together in straightforward or more elaborate ways.</p> 
      <p>Property binding utility modules include...</p> 
      <table class="list"> 
       <tbody>
        <tr valign="top">
         <td style="white-space:nowrap;"><span class="bullet"></span> </td>
         <td><code><a href="reference/Uize.Util.Coupler.html">Uize.Util.Coupler</a></code> - lets you couple two or more instances (of any class) together, synchronizing a set of any number of their state properties between all instances (e.g. synchronizing the <code>width</code>, <code>depth</code>, and <code>height</code> properties across any number of different <code>Room</code> instances)</td>
        </tr> 
        <tr valign="top">
         <td style="white-space:nowrap;"><span class="bullet"></span> </td>
         <td><code><a href="reference/Uize.Util.PropertyAdapter.html">Uize.Util.PropertyAdapter</a></code> - lets you connect any two properties of any two instance so that they remain synchronized bi-directionally, with an optional value transformer that can allows for the value to be transformed in different ways for each direction (e.g. synchronizing two color properties where the color is stored in different formats in each of the properties)</td>
        </tr> 
       </tbody>
      </table> 
     </div> 
     <p>For a more detailed discussion of the state properties system, consult the <a href="guides/state-properties.html">State Properties</a> guide.</p> 
    </div> 
    <a name="6"></a> 
    <h1 class="heading1" title="6. Event Infrastructure"><span class="headingNumber"><a href="#6" title="6. Event Infrastructure">6</a>. </span>Event Infrastructure</h1> 
    <div class="contents1"> 
     <p>The JavaScript language does not provide a built-in system for event-driven programming, so UIZE provides an event system.</p> 
     <p>Such events can be fired (aka published, dispatched, triggered, emitted, etc.) from anywhere in the code in an ad hoc fashion, and listenable events do not need to be registered formally. Multiple handlers can be registered for the same event. Handlers can be registered for events that are never fired, without ill effects. Conversely, events can be dispatched that are never listened for, without ill effects.</p> 
     <p><span class="allCaps">EXAMPLE</span></p> 
     <pre class="sample-code">var myWidget = Uize.Widget ();
myWidget.wire (
  'My Arbitrary Event',
  function (event) {alert (event.message)}
);
myWidget.fire ({name:'My Arbitrary Event',message:'Hello, world!'});
</pre> 
     <p>In the above example, an instance of the <code><a href="reference/Uize.Widget.html">Uize.Widget</a></code> class is created, a handler is registered on an arbitrarily named event (<code>'My Arbitrary Event'</code>), and then the <code>fire</code> instance method is called on the widget in order to fire that event. The event object contains an additional <code>message</code> property, whose value is alerted in the handler that was registered for the event.</p> 
     <p>The event system implemented by the UIZE JavaScript Framework is orthogonal to the event model of Web browsers that is provided as part of the standardized Document Object Model, and can be used outside of the context of Web browsers and other host environments for the JavaScript language that may implement their own proprietary event systems. As an example of this, the framework's event system can be utilized within Windows Script Host by scripts that use the <code><a href="reference/Uize.Class.html">Uize.Class</a></code> base class.</p> 
     <p>The <code><a href="reference/Uize.Class.html">Uize.Class</a></code> base class provides a convenient infrastructure for supporting both static and instance events. Events can conveniently be fired for a class or an instance of a class, and methods are provided to every class that subclasses the <code><a href="reference/Uize.Class.html">Uize.Class</a></code> base class to allow code to manage the wiring and unwiring of event handlers for static and instance events. This provides all <code><a href="reference/Uize.Class.html">Uize.Class</a></code> subclasses with a consistent event model.</p> 
     <p>For a more in-depth discussion on events, consult the guide <a href="guides/javascript-event-system.html">JavaScript Event System</a>.</p> 
    </div> 
    <a name="7"></a> 
    <h1 class="heading1" title="7. Conditions, Barriers, Needs"><span class="headingNumber"><a href="#7" title="7. Conditions, Barriers, Needs">7</a>. </span>Conditions, Barriers, Needs</h1> 
    <div class="contents1"> 
     <p>UIZE implements the useful programming constructs of conditions, barriers, and needs.</p> 
     <table class="list"> 
      <tbody>
       <tr valign="top">
        <td style="white-space:nowrap;"><span class="bullet"></span> </td>
        <td><b>conditions</b> - define conditions that are derived from one or more state properties using a properties derivation, and register code that is to be executed once a condition is met, each time a condition is met, or whenever a condition changes state</td>
       </tr> 
       <tr valign="top">
        <td style="white-space:nowrap;"><span class="bullet"></span> </td>
        <td><b>barriers</b> - register code that is to be executed once a condition is met, or immediately if the condition is already met - this can help with code that might be vulnerable to race conditions because of unpredictable asynchronous processes</td>
       </tr> 
       <tr valign="top">
        <td style="white-space:nowrap;"><span class="bullet"></span> </td>
        <td><b>needs</b> - register provider code that is to be executed the first time something is needed and that will provide that need, and register code that has a need and that should invoke an associated provider to be executed - register needs and providers optionally and in any order - this can help to manage loosely coupled code that needs to cooperate but can't agree on timing</td>
       </tr> 
      </tbody>
     </table> 
     <p>The conditions, barriers, and needs / providers mechanisms make it possible to manage tricky timing situations in applications where code is distributed and where it may not be feasible to enforce / orchestrate the desired sequencing. These mechanisms make it possible to explicitly define the state dependencies.</p> 
    </div> 
    <a name="8"></a> 
    <h1 class="heading1" title="8. Ajax / Asynchronous Processes"><span class="headingNumber"><a href="#8" title="8. Ajax / Asynchronous Processes">8</a>. </span>Ajax / Asynchronous Processes</h1> 
    <div class="contents1"> 
     <p>UIZE provides features to enable Web applications to more easily perform asynchronous communication with a Web server, a technique that is often referred to broadly as <a href="http://www.wikipedia.org/Ajax_(programming)" target="_blank" class="externalSiteLink">Ajax</a>.</p> 
     <p>The <code><a href="reference/Uize.Comm.html">Uize.Comm</a></code> base class provides a foundation for supporting asynchronous communication regardless of the mechanism employed, which could be a hidden <code>iframe</code>, the <code>XMLHttpRequest</code> object, or script tags (see <a href="http://www.wikipedia.org/AJAST_(Programming)" target="_blank" class="externalSiteLink">AJAST</a>). This base class provides a caching mechanism for requests, a queueing mechanism in order to sequence successive asynchronous requests through a single communication object, and a standard interface for issuing requests. Various subclasses of the <code><a href="reference/Uize.Comm.html">Uize.Comm</a></code> base class implement support for different communication mechanisms.</p> 
     <p><span class="allCaps">EXAMPLE</span></p> 
     <pre class="sample-code">Uize.Comm.Ajax ().request ({
  url:[
    '/service/search',
    {
      productFilter:'movies',
      genre:'comedy',
      sort:'popular',
      results:100
    }
  ],
  returnType:'json',
  callback:function (searchResults) {
    // do something with the search results
  }
});
</pre> 
     <p>In the above example, an instance of the <code><a href="reference/Uize.Comm.Ajax.html">Uize.Comm.Ajax</a></code> class is created and immediately used to perform an asynchronous request to a hypothetical search service using the <code>XMLHttpRequest</code> object. The anonymous function specified for the callback property is executed once the request has successfully executed, and the result from the server is passed as a parameter to the callback function as a JSON object.</p> 
     <p>For a more in-depth discussion of AJAX programming, consult the reference for the <code><a href="reference/Uize.Comm.html">Uize.Comm</a></code> base class.</p> 
    </div> 
    <a name="9"></a> 
    <h1 class="heading1" title="9. DOM Manipulation"><span class="headingNumber"><a href="#9" title="9. DOM Manipulation">9</a>. </span>DOM Manipulation</h1> 
    <div class="contents1"> 
     <p>While a design goal of the UIZE JavaScript Framework is to support multiple different host environments that support the JavaScript language, a key application of the framework is to facilitate the development of Rich Internet Applications that run with a web browser as the host environment for client-side functionality.</p> 
     <p>As such, there exist a number of modules designed specifically to assist with inspection of and manipulation of a Web page's Document Object Model. Among other things, these modules provide static methods for: iterating across a range HTML elements and performing operations on each, finding a set of HTML elements by means of a match expression, getting and setting Cascading Style Sheet (CSS) style properties, insertion of HTML markup, management of DOM event listeners, and querying and modifying coordinates.</p> 
     <p><span class="allCaps">EXAMPLE</span></p> 
     <pre class="sample-code">Uize.Dom.Basics.setStyle (
  ['myNode1','myNode2','myNode3'],
  {
    width:'100px',
    height:'20px',
    backgroundColor:'#000',
    fontFamily:'Times'
  }
);
</pre> 
     <p>The above example demonstrates how the <code>Uize.Dom.Basics.setStyle</code> static method of the <code><a href="reference/Uize.Dom.Basics.html">Uize.Dom.Basics</a></code> module is used to set the <code>width</code>, <code>height</code>, <code>backgroundColor</code>, and <code>fontFamily</code> CSS style properties for the three nodes with the ids "myNode1", "myNode2", and "myNode3".</p> 
     <p>For a more in-depth discussion of DOM manipulation, consult the references for the <code><a href="reference/Uize.Dom.Basics.html">Uize.Dom.Basics</a></code> module and other modules under the <code><a href="reference/Uize.Dom.html">Uize.Dom</a></code> namespace.</p> 
    </div> 
    <a name="10"></a> 
    <h1 class="heading1" title="10. Widgets Framework"><span class="headingNumber"><a href="#10" title="10. Widgets Framework">10</a>. </span>Widgets Framework</h1> 
    <div class="contents1"> 
     <p>UIZE provides a system to facilitate the development of widgets that can be embedded in HTML pages.</p> 
     <p>Functionality that can be inherited by widget classes is implemented in the <code><a href="reference/Uize.Widget.html">Uize.Widget</a></code> base class. The framework's widget system allows widgets to be arranged in a hierarchical tree structure, where each widget can have one parent widget and any number of child widgets. In this relationship, parent widgets can provide services to child widgets on their tree, and child widgets can inherit state from widgets higher up in the parent chain.</p> 
     <p>Examples of widgets include: buttons, sliders for selecting continuous values, progress bars, calendar / date pickers, color pickers, sortable collections, tabbed interfaces, modal dialogs, slide shows, data table sorters, tree list controls, and hierarchical menus.</p> 
     <p>For a more in-depth discussion of UIZE's widget framework, consult the guide <a href="guides/javascript-widgets.html">JavaScript Widgets</a>.</p> 
    </div> 
    <a name="11"></a> 
    <h1 class="heading1" title="11. HTML &amp; CSS Templating"><span class="headingNumber"><a href="#11" title="11. HTML &amp; CSS Templating">11</a>. </span>HTML &amp; CSS Templating</h1> 
    <div class="contents1"> 
     <p>The versatile <a href="#15">templating system</a> employed by UIZE is relied upon heavily for the <a href="#11_1">HTML templates</a> and <a href="#11_2">CSS templates</a> that are used by encapsulated widgets.</p> 
     <a name="11_1"></a> 
     <h2 class="heading2" title="11. HTML &amp; CSS Templating -> 11.1. HTML Templates"><span class="headingNumber"><a href="#11" title="11. HTML &amp; CSS Templating">11</a>.<a href="#11_1" title="11. HTML &amp; CSS Templating -> 11.1. HTML Templates">1</a>. </span>HTML Templates</h2> 
     <div class="contents2"> 
      <p>HTML templates are JST templates (<code>.js.jst</code> files) that are compiled to efficient JavaScript functions and wrapped in JavaScript modules so that they can be required as dependencies.</p> 
      <p>HTML template functions are called as instance methods of widget instances and can, therefore, call the widget's public instance methods as part of process of generating the HTML string for a widget. For instance, a widget's template can call the <code>childHtml</code> instance method on the widget in order to get the HTML for a child widget so that it can be added to the HTML in the desired place.</p> 
      <p>Below is an example of an HTML template taken from the <a href="reference/Uize.Widgets.ProgressBar.Widget.html">progress bar</a> widget...</p> 
      <p><span class="allCaps">EXAMPLE</span></p>

    <h1 class="heading1" title="13. Internationalization / Localization"><span class="headingNumber"><a href="#13" title="13. Internationalization / Localization">13</a>. </span>Internationalization / Localization</h1> 
    <div class="contents1"> 
     <p>UIZE provides facilities (in <code><a href="reference/Uize.Widget.html">Uize.Widget</a></code>) to ease <a href="http://en.wikipedia.org/wiki/I18n" target="_blank" class="externalSiteLink">i18n</a> (internationalization) and <a href="http://en.wikipedia.org/wiki/I18n" target="_blank" class="externalSiteLink">L10n</a> (localization) of JavaScript code.</p> 
     <p>The <code>localized</code> state property allows an arbitrary number of localized string templates to be specified in an object hash, and these string templates are then available to the widget and all its child widgets. The <code>localize</code> instance method allows a string template to be retrieved and will process the string, as necessary, to replace substitution tokens with dynamic data.</p> 
     <p><span class="allCaps">EXAMPLE</span></p> 
     <pre class="sample-code">myWidget = Uize.Widget ({
  localized:{welcomeMessage:'Welcome, {firstName} of {state}, {country}'}
});
alert (
  myWidget.localize (
    'welcomeMessage',
    {firstName:'Chris',state:'California',country:'USA'}
  )
);
</pre> 
     <p>In the above example, an instance of the <code><a href="reference/Uize.Widget.html">Uize.Widget</a></code> class is created, specifying a <code>localized</code> hash that contains just one localized string template named <code>welcomeMessage</code>. This string contains the substitution tokens <code>{firstName}</code>, <code>{state}</code>, and <code>{country}</code>. The <code>localize</code> instance method is called on the widget instance, with parameters specifying the name of the localized string to retrieve and dynamic data that should be substituted into its tokens. The result is then displayed to the user with the <code>alert</code> statement. Typically, localization will occur inside the implementation for a widget class, so this example is purely for illustrating the syntax.</p> 
     <p><b>TIP</b>: To aid in internationalizing legacy code, a script is provided that can be run in Windows Script Host and that will recurse through the folders of a Web site project in order to compile a report of all the string literals contained inside JavaScript files, using a heuristic algorithm to bucket them according to their likelihood of being internationalizable strings (see <a href="guides/javascript-build-scripts.html">JavaScript Build Scripts</a>).</p> 
     <p>For a more in-depth discussion of internationalization and localization, consult the guide <a href="guides/javascript-localization.html">JavaScript Localization</a>.</p>

    <h1 class="heading1" title="15. Templating System"><span class="headingNumber"><a href="#15" title="15. Templating System">15</a>. </span>Templating System</h1> 
    <div class="contents1"> 
     <p>UIZE implements a system for templating that allows the JavaScript language to be leveraged within templates, such templates being referred to as JavaScript templates.</p> 
     <p>This system allows for the use of programmatic logic and JavaScript control structures within a template for the purpose of rendering a result document. It's format and approach will be immediately familiar to anyone who has used the templating systems of other languages, such as Ruby's <code>.erb</code> templates, .NET's <code>.aspx</code> templates, Java's <code>.jsp</code> templates, etc.</p> 
     <p>The template engine is implemented in the <code><a href="reference/Uize.Template.html">Uize.Template</a></code> module, which provides a means for compiling a source template into a more performant JavaScript function. Once compiled into JavaScript functions, JavaScript templates can be used to generate text output for a variety of purposes, including, for example, the generation of HTML markup for widgets, the generation of complete HTML documents as part of a Web project build process, and the generation of RSS documents from a data source.</p> 
     <p><span class="allCaps">EXAMPLE</span></p> 
     <pre class="sample-code">var helloWorldTemplate = Uize.Template.compile (
  '&lt;% for (var i = 0; i &lt; input.repeats; i++) { %&gt;' +
  '&lt;p&gt;&lt;% .message %&gt;&lt;/p&gt;\n' +
  '&lt;% } %&gt;'
);

var myTemplateOutput = helloWorldTemplate ({repeats:5,message:'Hello World !'});

/* myTemplateOutput will now have the value...

&lt;p&gt;Hello World !&lt;/p&gt;
&lt;p&gt;Hello World !&lt;/p&gt;
&lt;p&gt;Hello World !&lt;/p&gt;
&lt;p&gt;Hello World !&lt;/p&gt;
&lt;p&gt;Hello World !&lt;/p&gt;

*/
</pre> 
     <p>In the above example, a JavaScript template in a string literal is being compiled to a JavaScript function, and then that function is being called with input data that is supplied to the template function. The result is assigned to a variable.</p> 
     <p>In practice, JavaScript templates are contained inside separate files with the extension <code>.jst</code> in order that the template source not be subject to a further level of escaping within JavaScript string literals.</p> 
     <p><span class="allCaps">AS A JST FILE</span></p> 
     <pre class="sample-code">&lt;% for (var i = 0; i &lt; input.repeats; i++) { %&gt;
&lt;p&gt;&lt;% .message %&gt;&lt;/p&gt;
&lt;% } %&gt;
</pre> 
     <p>JavaScript template files can be compiled into JavaScript modules with the use of JavaScript build scripts, and such modules can then be required as dependencies and treated as regular JavaScript modules. Alternatively, JavaScript template source can be embedded in non-visible elements of a Web page, such as a hidden <code>textarea</code> tag or a <code>script</code> tag with a mime type of <code>text/jst</code>, and then retrieved and compiled by client code when the document loads.</p> 
     <p>For a more in-depth discussion, consult the guide <a href="guides/javascript-templates.html">JavaScript Templates</a>.</p> 
    </div>

    <h1 class="heading1" title="16. Effects / Animation"><span class="headingNumber"><a href="#16" title="16. Effects / Animation">16</a>. </span>Effects / Animation</h1> 
    <div class="contents1"> 
     <p>UIZE provides powerful features to support slick animation effects, with easing, bounce, elasticity, pulsing, and much more.</p> 
     <p>Effects and animation in the UIZE JavaScript Framework are achieved through use of a suite of associated modules. The <code><a href="reference/Uize.Fade.html">Uize.Fade</a></code> module provides the underpinnings of time-based animation and compound value interpolation, the <code><a href="reference/Uize.Fade.xFactory.html">Uize.Fade.xFactory</a></code> extension module extends the <code><a href="reference/Uize.Fade.html">Uize.Fade</a></code> class with factory methods for creating, starting, and managing fades in a fade pool, and the <code><a href="reference/Uize.Fx.html">Uize.Fx</a></code> module provides static methods for initiating fades of CSS style properties.</p> 
     <p><span class="allCaps">EXAMPLE</span></p> 
     <pre class="sample-code">// fade from thin border/thick padding to thick border/thin padding over 1/4 second
Uize.Fx.fadeStyle ('myNode',{borderWidth:1,padding:20},{borderWidth:20,padding:1},250);

// fade from current colors to white text on black background over two seconds
Uize.Fx.fadeStyle ('myNode',null,{color:'#fff',backgroundColor:'#000'},2000);

// fade font size from 30px back to current size over a half second
Uize.Fx.fadeStyle ('myNode',{fontSize:30},null,500);
</pre> 
     <p>To add some pizazz to animations, the UIZE JavaScript Framework provides the <code><a href="reference/Uize.Curve.html">Uize.Curve</a></code>, <code><a href="reference/Uize.Curve.Mod.html">Uize.Curve.Mod</a></code>, and <code><a href="reference/Uize.Curve.Rubber.html">Uize.Curve.Rubber</a></code> modules that let you stray from the bland world of linear animations and into an exotic world of animations driven by arbitrary curve functions, with built-in support for easing curves as well as curves that emulate the complex properties of motion, letting you achieve effects like bounce, springiness, wobble, elasticity, etc. Beyond the built-ins, you have the freedom to roll your own curve functions in order to achieve some of the craziest motion effects you could imagine.</p> 
     <p>For a more in-depth discussion, consult the guide <a href="guides/javascript-animation-and-effects.html">JavaScript Animation and Effects</a>.</p>

    <h1 class="heading1" title="17. Documentation System"><span class="headingNumber"><a href="#17" title="17. Documentation System">17</a>. </span>Documentation System</h1> 
    <div class="contents1"> 
     <p>UIZE implements a system for building HTML from documentation that is written in a Wikitext like format called Simple Doc.</p> 
     <p>Document structure in this format is controlled through indentation, much like program structure can be governed by indentation in the <a href="http://www.wikipedia.org/Python_(programming_language)" target="_blank" class="externalSiteLink">Python programming language</a>. A documentation builder script is provided that can run in Windows Script Host. This build script can recurse through all the folders of a Web site project, building HTML files from all Simple Doc files it finds, and extracting Simple Doc formatted documentation from specially flagged comments inside JavaScript module files and synthesizing these Simple Doc fragments together into Simple Doc documents for conversion into HTML documentation files.</p> 
     <p>The following example shows the documentation for the <code>to</code> instance method of the <code><a href="reference/Uize.Color.html">Uize.Color</a></code> module. Notice how the documentation comment is indicated with the question mark immediately following the comment begin characters <code>/*</code>.</p> 
     <p><span class="allCaps">EXAMPLE</span></p> 
     <pre class="sample-code">/*?
  Instance Methods
    to
      Returns the current color of the instance, encoded to the specified format.

      SYNTAX
      .........................................
      encodedColor = colorOBJ.to (encodingSTR);
      .........................................

      The =encodingSTR= parameter supports a wide variety of different `color encodings`.

      EXAMPLES
      .............................................................................
      var fuchsia = Uize.Color ('fuchsia');
      fuchsia.to ('color');       // produces a new Uize.Color object
      fuchsia.to ('hex');         // produces 'ff00ff'
      fuchsia.to ('#hex');        // produces '#ff00ff'
      fuchsia.to ('name');        // produces 'fuchsia'
      fuchsia.to ('RGB array');   // produces [255,0,255]
      fuchsia.to ('RGB int');     // produces 16711935
      fuchsia.to ('RGB object');  // produces {red:255,green:0,blue:255}
      fuchsia.to ('RGB string');  // produces 'rgb(255,0,255)'
      fuchsia.to ('HSL array');   // produces [300,100,50]
      fuchsia.to ('HSL object');  // produces {hue:300,saturation:100,lightness:50}
      fuchsia.to ('HSL string');  // produces 'hsl(300,100,50)'
      .............................................................................

      NOTES
      - see the related =Uize.Color.to= static method
*/
</pre> 
     <p>For a more in-depth discussion of this documentation system, consult the guide <a href="guides/javascript-documentation-system.html">JavaScript Documentation System</a>.</p>

    <h1 class="heading1" title="19. Code Compression"><span class="headingNumber"><a href="#19" title="19. Code Compression">19</a>. </span>Code Compression</h1> 
    <div class="contents1"> 
     <p>UIZE provides a system for scrunching (minifying) JavaScript code - primarily to reduce its size, but also to obfuscate it.</p> 
     <p>While the network bandwidth that is used to transfer JavaScript code to the client can be reduced in order to reduce transfer time through the use of HTTP compression methods such as gzip, further code size reduction as well as a degree of code obfuscation can be achieved with the use of the Scruncher, a code minification utility that is provided as part of the UIZE JavaScript Framework. This utility can be accessed for individual use through a Web page interface, but can be more conveniently utilized as part of a build script that recurses through the folders of a Web site project and generates compressed and obfuscated code from all the JavaScript files it encounters.</p> 
     <p>Among the operations that are performed by the Scruncher are:</p> 
     <table class="list"> 
      <tbody>
       <tr valign="top">
        <td style="white-space:nowrap;"><span class="bullet"></span> </td>
        <td>remove unnecessary whitespace, such as line indentation, unnecessary whitespace between tokens, and linebreaks</td>
       </tr> 
       <tr valign="top">
        <td style="white-space:nowrap;"><span class="bullet"></span> </td>
        <td>remove comments, including documentation comments</td>
       </tr> 
       <tr valign="top">
        <td style="white-space:nowrap;"><span class="bullet"></span> </td>
        <td>rename private identifiers to shortened, cryptic forms</td>
       </tr> 
      </tbody>
     </table> 
     <p>Obfuscation that is provided by the Scruncher is an artifact of the code compression process, and the Scruncher is not designed to provide robust obfuscation.</p> 
     <p>The two examples below show first an extract of JavaScript code, and second the compressed and obfuscated form of that code. Notice how, in the compressed version of the code, private identifiers that were prefixed with an underscore in the source code have been reduced down to enumerated forms (e.g. <code>_a</code>, <code>_b</code>, <code>_c</code>). For the sake of readability in this example, the <code>LineCompacting</code> setting is turned off for the compressed code.</p> 
     <p><span class="allCaps">SOURCE CODE</span></p> 
     <pre class="sample-code">/*ScruncherSettings Mappings="=" LineCompacting="FALSE"*/

_package.toAbsolute = function (_baseUrl,_url) {

    <a name="20"></a> 
    <h1 class="heading1" title="20. Dev Server (NodeJS)"><span class="headingNumber"><a href="#20" title="20. Dev Server (NodeJS)">20</a>. </span>Dev Server (NodeJS)</h1> 
    <div class="contents1"> 
     <p>UIZE provides a NodeJS based development server that allows a project that involves many different types of built files to be developed without having to constantly run build scripts between changes.</p> 
     <p>The development server utilizes the <a href="#18">powerful build system</a> to build derived files whenever they are requested and are not current (i.e. some of their dependencies have been modified since the last time they were built). The development server is used in the development of the UIZE framework, itself, and it can be used for any project for which static files will be generated as part of a build process.</p> 
     <p>With the right configuration of your application server, you can pull all the static files that would be built as part of your build process from the development server running on your localhost. That way, you get the best of both worlds - you can keep using your existing server platform while using the UIZE development server for pre-processing of source code for static files.</p> 
    </div>
