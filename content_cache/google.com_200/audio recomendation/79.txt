       <section class="page-content"> 
        <h1 id="getting-started-with-raspberry-pi-3-and-node-js">Getting Started With Raspberry Pi 3 and Node.js</h1> 
        <h2 id="introduction">Introduction</h2> 
        <p>In this guide we will build a simple Node.js web server project on a Raspberry Pi 3. Along the way you will learn the basics of resin.io. We will walk through getting a Raspberry Pi 3 online and deploying some Node.js code to it.</p> 
        <p>We will also look into using the built in web terminal to run test commands and debug issues. Once you have a handle on the resin.io deployment workflow, we will introduce you to resin sync and our CLI which will allow you to increase your development speed on resin.io.</p> 
        <h2 id="what-you-will-need">What You Will Need</h2> 
        <p><img style="float: right;padding-left: 10px;" src="/img/raspberrypi/raspberrypi.jpg" width="25%"></p> 
        <ul> 
         <li>A <a href="https://www.raspberrypi.org/products/raspberry-pi-3-model-b/">Raspberry Pi 3 model B</a>. See our <a href="/hardware/devices/">supported devices list</a> for other boards.</li> 
         <li>A 4GB or larger microSD card. The <a href="https://en.wikipedia.org/wiki/Secure_Digital#Speed_class_rating">speed class</a> of the card also matters - class 10 card or above is the way to go.</li> 
         <li><strong>[Optional]</strong> An ethernet cable</li> 
         <li>A micro USB cable.</li> 
         <li><strong>[Optional]</strong> A <a href="https://www.raspberrypi.org/products/universal-power-supply/">2A USB micro power supply</a>.</li> 
         <li>A <a href="https://dashboard.resin.io/signup">resin.io account</a>.</li> 
        </ul> 
        <h2 id="getting-help">Getting Help</h2> 
        <p>Before we get started building something cool, lets just point out some places to get help. If at anytime while using resin.io you find yourself stuck, confused or alone remember that there is a team of helpful and friendly support engineers just one click away. </p> 
        <p>The best place to chat with us is in the <a href="https://forums.resin.io">resin.io forums</a>. Post your questions to the <a href="https://forums.resin.io/c/troubleshooting">Troubleshooting section</a>, where our engineers can help address any issues you may be having with resin.io. </p> 
        <p>We also love to meet users in our community chat at <a href="https://gitter.im/resin-io/chat">gitter.im</a>.</p> 
        <p>Read more about our approach to support at resin.io/support(<a href="https://resin.io/support">https://resin.io/support</a>).</p> 
        <p>To help us understand all the moving parts in resin.io, lets first define a few terms that will be used later in the guide.</p> 
        <ul> 
         <li><p><strong>Application:</strong></p> 
          <blockquote> 
           <p>This is a group of devices or "fleet" that will all run the same application code. When you provision a device, it will automatically be associated to the application. You can add as many devices to an application as you like, its also possible to migrate devices to other applications.</p> 
          </blockquote> </li> 
         <li><p><strong>resin remote:</strong></p> 
          <blockquote> 
           <p>A remote <a href="https://www.sbf5.com/~cduan/technical/git/git-1.shtml">git repository</a> that is associated to your application. Any code pushed to the <code>master</code> branch of this repo will be built and deployed as a container on all devices in the application. This git repo uses SSH keys to secure it, so don't forget to set up your SSH key.</p> 
          </blockquote> </li> 
         <li><p><strong>Container:</strong></p> 
          <blockquote> 
           <p>A <a href="https://docs.docker.com/engine/understanding-docker/#how-does-a-container-work">Docker container</a> that essentially is a bundle of your application code and all its dependencies. It runs as an isolated process in userspace on the resinOS host.</p> 
          </blockquote> </li> 
        </ul> 
        <h2 id="let-s-jump-in">Let's Jump In</h2> 
        <p>If you don't already have a resin.io account head over to our <a href="https://dashboard.resin.io/signup">signup page</a>, during the sign up process you will be asked to set up an SSH key so you can securely push code.</p> 
        <h3 id="adding-an-ssh-key">Adding an SSH Key</h3> 
        <p>SSH keys use the power of <a href="http://en.wikipedia.org/wiki/Public-key_cryptography">public-key cryptography</a> to secure your connection when sending your code to us. In order to secure your <a href="http://git-scm.com/">git</a> connection, you need to add your <strong>public</strong> <a href="http://en.wikipedia.org/wiki/Secure_Shell">SSH Key</a> (you should never share your <em>private</em> key with anyone.)</p> 
        <p><img src="/img/common/sign_up_flow/enter_ssh_key_cropped.png" width="80%"></p> 
        <p>Simply paste your <strong>public</strong> key into the box provided on the UI and click <code>save</code>. Alternatively you can import your key from <a href="http://github.com/">Github</a>, just click on the Octocat icon.</p> 
        <h4 id="don-t-have-a-ssh-key-">Don't have a SSH key?</h4> 
        <p>If you don't have an ssh key or have never used one, we recommend you take a look at <a href="http://github.com/">Github</a>'s <a href="https://help.github.com/articles/generating-a-new-ssh-key-and-adding-it-to-the-ssh-agent/">excellent documentation</a> on the subject and how to generate a key pair for your OS.</p> 
        <p>Once generated, SSH keys are easy to use. In fact you generally don't have to think about it at all. Once you're set up just <code>git push</code> your code to resin.io and it's taken care of automatically and securely.</p> 
        <h4 id="import-ssh-key-from-github">Import SSH key From GitHub</h4> 
        <p>For convenience we provide the ability to import your public SSH key from <a href="http://github.com/">GitHub</a> - just click on the Octocat icon in the bottom right-hand corner (<a href="http://resin.io/blog/email-github-public-ssh-key/">we use</a> GitHub's <a href="https://developer.github.com/v3/">public APIs</a> to retrieve this data.)</p> 
        <p>You will then have to enter your github username into the prompt:</p> 
        <p><img src="/img/common/sign_up_flow/enter_github_username_cropped.png" width="60%"></p> 
        <p>If you don't have a ssh key setup yet, but want to explore resin.io, just click <code>skip</code>. Note that you will not be able to push code to your Raspberry Pi 3 until you have a ssh key saved. This can be done at anytime from the <a href="https://dashboard.resin.io/preferences?tab=sshkeys"><code>Preferences</code></a> page on the dashboard.</p> 
        <h2 id="creating-an-application">Creating an Application</h2> 
        <p>To create an application simply type in a name, select the Raspberry Pi 3 type from the drop down list and click the create button. You should now be taken to the dashboard of your newly created application:</p> 
        <p><img src="/img/common/main_dashboard/select_fleet_type.png" width="80%"></p> 
        <p>This dashboard may not look like much right now, but this is where you will command and manage a whole fleet of Raspberry Pi 3s.</p> 
        <h2 id="adding-your-first-device">Adding Your First Device</h2> 
        <p>This is the application dashboard where all of the devices connected to your application will be shown, along with their statuses and logs.</p> 
        <p><img src="/img/common/app/app_dashboard_empty.png" width="80%"></p> 
        <p>Click the <code>Download resinOS</code> button to get the resin.io operating system (resinOS) image for your application. A list of available versions is available via the dropdown. In general, the most recent version is recommend. </p> 
        <p><strong>Warning:</strong> Versions ending in .dev are for development purposes only, and should not be used in production. More details on the differences between dev and prod images can be found <a href="/understanding/understanding-devices/2.0.0/#dev-vs-prod-images">here</a>.</p> 
        <p><strong>Note:</strong> The <code>.img</code> may seem large, but your browser will download a compressed version and decompress it on the fly using <a href="https://en.wikipedia.org/wiki/HTTP_compression">http compression</a>, so the download will be much, much faster than you expect!</p> 
        <p>Before your resinOS download begins, a prompt will appear asking you to specify how your device will connect to the internet. Currently there are two connectivity options:</p> 
        <ul> 
         <li>Ethernet cable, this option requires <em>NO</em> configuration as is the default.</li> 
         <li>Wifi, in which case you must specify the <a href="https://en.wikipedia.org/wiki/Service_set_(802.11_network)#Service_set_identification_.28SSID.29">network name or<code>SSID</code></a> and <code>passphrase</code> for the network your device will connect to.</li> 
        </ul> 
        <p>Once you have selected your connectivity option, click the <code>Download Device OS</code> button to get the resin.io operating system image configured for your application and network.</p> 
        <p><img src="/img/common/network/network_selection_wifi_cropped.png" width="60%"></p> 
        <p>When the download completes you should have a <code>.img</code> file with a name like <code>resin-myFleet-1.1.1-1.7.0-dc6c40fe05aa.img</code> where "myFleet" is the name you gave your application on the dashboard.</p> 
        <h3 id="create-a-bootable-sd-card">Create a Bootable SD card</h3> 
        <p>Now we have to flash the downloaded <code>.img</code> file onto our SD card. We recommend using <a href="https://etcher.io/">Etcher</a>, a simple, cross platform SD card writer and validator. Head over to the <a href="https://etcher.io/">Etcher homepage</a> and get install it, it only takes a few seconds :)</p> 
        <p>You can of course use any other SD card writing software you like, some options are:</p> 
        <ul> 
         <li><a href="http://sourceforge.net/projects/win32diskimager/">win32diskimager</a> for Windows.</li> 
         <li><a href="http://ivanx.com/raspberrypi/">piFiller</a> for osx.</li> 
         <li><a href="http://man7.org/linux/man-pages/man1/dd.1.html">dd or "Disk Destroyer"</a> for Linux.</li> 
        </ul> 
        <p><strong>Note:</strong> Before you flash resinOS to your SD card you may need to formatted it as <a href="http://en.wikipedia.org/wiki/Fat32#FAT32">FAT32</a>. <a href="http://www.wikihow.com/Main-Page">WikiHow</a> has great <a href="http://www.wikihow.com/Format-an-SD-Card">instructions</a> on how to do this.</p> 
        <p>For simplicity this tutorial will assume you are using <a href="https://etcher.io/">Etcher</a>. Once you have Etcher installed, start it up. You may be asked to allow Etcher administrative privileges. This is just so Etcher can access your SD card.</p> 
        <p>To create a bootable resinOS SD card follow these 3 easy steps:</p> 
        <ol> 
         <li>Click "Select image" button and find your applications resinOS <code>.img</code> file.</li> 
         <li>If you haven't already done so, insert your SD card into your computer. Etcher will automatically detect it. If you have more than one SD card inserted, you will need to select the appropriate one.</li> 
         <li>Click the "Flash!" button.</li> 
        </ol> 
        <p><img src="/img/common/etcher/etcher.gif" width="60%"></p> 
        <p>Etcher will now prepare a bootable SD card and validate that it was flashed correctly. Right! time for a spot of tea as flashing the SD card can take roughly 3 or more minutes depending on the quality of your SD card.</p> 
        <p>Etcher will give you a little ping! when it's done, and safely eject the SD card for you.</p> 
        <p><strong>Note:</strong> You can burn several SD cards with the same <code>.img</code> file and all the devices will boot and provision into your application's fleet. You can also disable the auto-ejecting or write validation steps from the Etcher settings panel.</p> 
        <h2 id="setting-up-your-device">Setting Up Your Device</h2> 
        <p>Insert the SD card into the Raspberry Pi 3 and connect the ethernet cable if necessary. Now power up the Raspberry Pi 3 by inserting the micro USB cable.</p> 
        <p><img src="/img/gifs/insert-sd.gif" alt="insert SD"></p> 
        <p>It will take a minute or two for the Raspberry Pi 3 to appear on your <a href="https://dashboard.resin.io/">resin.io dashboard</a>. While you wait the resinOS is expanding the partitions on your SD card to use all available space, installing a custom linux environment and establishing a secure connection with our servers.</p> 
        <p>You should now be ready to deploy some code!</p> 
        <p><strong>Note:</strong> Class 4 SD cards can take up to 3 times longer so it's well worth investing in the fastest card you can find.</p> 
        <h5 id="help-my-device-won-t-show-up-">Help! My device won't show up.</h5> 
        <p>If your device still hasn't shown up on your dashboard after 10 minutes, something is definitely wrong. First check that you entered the wifi credentials correctly and ensure that your network meets these <a href="/deployment/network/#network-requirements">basic requirements</a>. It may also be worth checking the <a href="/troubleshooting/error">LED error notifications</a></p> 
        <p>If you still can't get your device online, come on over and chat to us on our <a href="/support/">support channel</a>.</p> 
        <p><strong>Note:</strong> If you have an HDMI screen attached, you should see <code>"Booted - Check your resin.io dashboard."</code> on the screen when the device boots. If instead you see rainbow colours or a black screen with 4 raspberries on it, it could mean that the SD card was not burned correctly or is corrupted.</p> 
        <h2 id="deploying-code">Deploying Code</h2> 
        <!-- deploying Code to devices --> 
        <p>Now that we have a device or two connected to a resin.io application, lets deploy some code and actually start building something.</p> 
        <p>A nice first project to get your feet wet is a simple <a href="http://expressjs.com/">Express.js</a> web server which will serve a static page on port <code>:80</code>. All the project source code can be found <a href="https://github.com/resin-io-projects/simple-server-node">here on github</a>.</p> 
        <p>To clone the project, run the following command in a terminal or your preferred git client:</p> 
        <pre><code>$ git clone https://github.com/resin-io-projects/simple-server-node.git
</code></pre>
        <p>Once the repo is cloned, change directory into the newly created <code>simple-server-node</code> directory and add the resin git remote endpoint by running the command <code>git remote add</code> shown in the top-right corner of your application page:</p> 
        <pre><code>$ cd simple-server-node
$ git remote add resin &lt;USERNAME&gt;@git.resin.io:&lt;USERNAME&gt;/&lt;APPNAME&gt;.git
</code></pre>
        <p><strong>Note:</strong> On other git clients there may be an alternative way to add a remote repository.</p> 
        <p>So now we have set up a reference in our local git repository (the one on our development computer) to the resin.io application remote repository. So when we push new changes to this remote repository it will get compiled and built on our servers and deployed to every device in the application fleet.</p> 
        <p>Now to deploy this code to all device(s) in the application just run the command:</p> 
        <pre><code>$ git push resin master
</code></pre>
        <p>If you want to completely replace the source code of the application with a new source tree, you may need to force the push by running <code>git push resin master --force</code>, due to how git works.</p> 
        <p><strong>Note:</strong> On your very first push, git may ask you if you would like to add this host to your list of allowed hosts. Don't worry about this, just type 'yes' and carry on your merry way.</p> 
        <p>You'll know your code has been successfully compiled and built when our friendly unicorn mascot appears in your terminal:</p> 
        <p><img src="/img/common/pushing/success_unicorn_simple_nodejs.png" width="80%"></p> 
        <p>This means your code is safely built and stored on our image registry. It should only take about 2 minutes to build your code and subsequent builds will be quicker because of build caching.</p> 
        <p>Your application will now be downloaded and executed by all the devices you have connected in your application fleet. You may have to wait about 6 minutes for the first push... So time for more tea, but don't worry, all subsequent pushes are much, much faster due to <a href="https://docs.docker.com/engine/userguide/storagedriver/imagesandcontainers/">Docker layer sharing</a>. You can see the progress of the device code updates on the device dashboard:</p> 
        <p><img src="/img/common/device/device_dashboard_during_update_generic.png" width="80%"></p> 
        <p>You should now have a node.js web server running on your device and see some logs on your dashboard. If you go to the <code>Actions</code> page for your device, you can enable a public URL, this URL is accessible from anywhere in the world.</p> 
        <p><img src="/img/common/enable-public-URLs.png" width="80%"></p> 
        <p>If you follow the URL, you will be served a page saying "Hello, World!". Alternatively you can point your browser to your devices IP address.</p> 
        <p>You should now have a basic idea of how to deploy a node.js application on resin.io. If you feel like you have a handle on Docker and Node.js projects, then skip over the next section and go straight to <a href="#using-the-web-terminal">"Using the web terminal"</a>.</p> 
        <h4 id="let-s-dive-into-the-code">Let's dive into the code</h4> 
        <!-- project link: https://github.com/resin-io-projects/simple-server-node --> 
        <p>So in the root directory of our project we see a number of files, the most important ones to focus on are:-</p> 
        <ul> 
         <li><code>Dockerfile.template</code> : This is basically a recipe file on how to build and run our application container.</li> 
         <li><code>package.json</code> : This is a <a href="http://www.json.org/">JSON</a> file that describes how our node.js project is built, what dependencies it has and it's entry point. Read more about it on the <a href="https://docs.npmjs.com/files/package.json">npm docs</a>.</li> 
         <li><code>server.js</code> : This is the entry point to our application code and is where all the fun happens!</li> 
        </ul> 
        <p>The most important part of a resin.io project repo is usually the <code>Dockerfile</code> or <code>Dockerfile.template</code>. The <code>.template</code> version allows you to define template variables like <code>%%RESIN_MACHINE_NAME%%</code> which enables you to push the same repository to multiple different architecture fleets.</p> 
        <p>If we look at our <code>Dockerfile.template</code>, the first thing we see is:</p> 
        <pre><code class="lang-Dockerfile">FROM resin/%%RESIN_MACHINE_NAME%%-node:slim
</code></pre> 
        <p>This line has quite a bit packed into it. The first thing that happens is that the <code>%%RESIN_MACHINE_NAME%%</code> place holder gets stripped and replaced with the resin device name. For example if your application type is a Raspberry Pi 3, the line will be replaced with:</p> 
        <pre><code class="lang-Dockerfile">FROM resin/raspberrypi3-node:slim
</code></pre> 
        <p>Which tells the resin builder that this is the Docker image we want as our base. Checkout the full <a href="/devicetypes/">list of official resin device names</a> and the <a href="https://hub.docker.com/u/resin/">matching Docker Hub base images</a>.</p> 
        <p>We also have a <code>:slim</code> tag associated to the base image which denotes that we want the stripped down version only contains the minimal packages needed to run node, so no <a href="https://github.com/nodejs/node-gyp"><code>node-gyp</code></a> and other build-essentials. If you need to build some native modules, say node-i2c, you should switch to <code>:latest</code> tag. We also have a number of pinned version tags, which should be used for production devices. Checkout the full <a href="https://hub.docker.com/r/resin/raspberrypi3-node/tags/">list of -node tags</a>, if you want to target a specify node.js version or a fixed date build.</p> 
        <p>Next up we have 3 line which were commented out:</p> 
        <pre><code class="lang-Dockerfile">RUN apt-get update &amp;&amp; apt-get install -yq \
   alsa-utils libasound2-dev &amp;&amp; \
   apt-get clean &amp;&amp; rm -rf /var/lib/apt/lists/*
</code></pre> 
        <p>This is just a demonstration of how you can use <code>apt-get</code> to install dependencies in your container. In this case we would install some useful linux sound utilities.</p> 
        <p>The next two directives are pretty straight forward and key parts of using Docker.</p> 
        <pre><code class="lang-Dockerfile"># Defines our working directory in container
WORKDIR /usr/src/app

# Copies the package.json first for better cache on later pushes
COPY package.json package.json
</code></pre> 
        <p>As the comments say, <code>WORKDIR</code> set our working directory for any <code>RUN</code>, <code>COPY</code> or <code>CMD</code> commands following it. So the next line would effectively <code>COPY</code> our <code>package.json</code> in the root of our directory to <code>usr/src/app/package.json</code>. Check out the <a href="https://docs.docker.com/engine/reference/builder/">Docker reference</a> pages for more info on these commands.</p> 
        <p>We can now build all our node.js modules and dependencies, this is done using the <code>RUN</code> command. We also build with the <code>--production</code> flag and clear the cache in the same step in order to keep the final image size smaller.</p> 
        <pre><code class="lang-Dockerfile"># This install npm dependencies on the resin.io build server,
# making sure to clean up the artifacts it creates in order to reduce the image size.
RUN JOBS=MAX npm install --production --unsafe-perm &amp;&amp; npm cache clean &amp;&amp; rm -rf /tmp/*

# This will copy all files in our root to the working  directory in the container
COPY . ./

# Enable systemd init system in container
ENV INITSYSTEM on

# server.js will run when container starts up on the device
CMD ["npm", "start"]
</code></pre> 
        <p>After the <code>npm install</code> we copy the rest of our source code into the working directory, we do this after so that later builds can benefit from build caching. So we will only trigger a full npm install if we change something in <code>package.json</code>.</p> 
        <p>The last 2 commands are runtime directives. The <code>ENV INITSYSTEM on</code> is used to enable the <a href="https://en.wikipedia.org/wiki/Systemd">systemd</a> init within the container. This is useful for a number of reasons, like keeping the container open after application crash and handling <code>/dev</code> updates as new USB devices are plugged in. If you don't want an init system, just set it to <code>off</code> or remove the line for the <code>Dockerfile</code>.</p> 
        <p>The last command, <code>CMD</code> is perhaps one of the most important. This command defines what will run at container start on your Raspberry Pi 3, in our example we have told npm to start a process. It should be noted that you can only have <strong>one</strong> <code>CMD</code> per <code>Dockerfile</code>.</p> 
        <p>In our <code>package.json</code> the parts to focus on are our "scripts" and "dependencies":</p> 
        <pre><code class="lang-JSON">{
  "name": "simple-server-node",
  "version": "1.0.0",
  "description": "A simple Expressjs Web server on resin.io",
  "main": "server.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" &amp;&amp; exit 1",
    "start": "node server.js"
  },
  "dependencies": {
    "express": "*"
  }
}
</code></pre> 
        <p>The "dependencies" section allows us to define node modules and their versions we want to use in our application. For a production application we recommend pinning the versions.</p> 
        <p>The "scripts" allow us to point to the <code>server.js</code> as our starting point for the whole application, so we get our awesome "Hello, World!" server when the container starts up :)</p> 
        <h2 id="using-the-web-terminal">Using the Web Terminal</h2> 
        <p>Often while developing and deploying code, it helps to run a few test commands or check some log files. For this resin has a handy built in web terminal which you can use from the comfort of your dashboard, or command line (check the later section on CLI and resin sync for this).</p> 
        <p>To fire up a terminal session on your device you need to two things:</p> 
        <ol> 
         <li>An online device.</li> 
         <li>A running container.</li> 
        </ol> 
        <p>Number <code>.1</code> is usually pretty easy, but number <code>.2</code> catches people pretty often. Since if the main process of the Docker container crashes or ends, the container effectively stops and there is nothing for the web terminal to SSH into <code>:(</code> . For this reason we normally recommend using the systemd init system during development as this will ensure your container is always up and running, even if your application code crashes.</p> 
        <p><img src="/img/common/webterminal/terminal-raspberrypi3.png" width="80%"></p> 
        <p>To start a session, just navigate to the <code>&gt;_ Terminal</code> page for the device and hit the "Start the terminal session" button. It will take a few seconds to establish a connection and then you are good to go.</p> 
        <p><strong>Note:</strong> Currently if you navigate away from the <code>&gt;_ Terminal</code> page, you session will be killed. This is a known issue and will be remedied very soon.</p> 
        <h2 id="using-resin-sync-to-develop-fast">Using Resin Sync to Develop Fast</h2> 
        <p>Okay, so now we know how to provision a device and push code. There is just one glaring problem...</p> 
        <blockquote> 
         <p>Do I really have to go through the whole <code>git add</code>, <code>git commit</code>, <code>git push</code> dance every time I forget a semicolon?</p> 
        </blockquote> 
        <p>Luckily, our nifty little command line tool <code>resin sync</code> is here to save the day. It allows you to quickly sync source code and file changes across to one of the devices in your fleet, so you can rapidly iterate code on this test device before releasing it to the whole fleet.</p> 
        <p><strong>Note:</strong> Resin sync will only work on ResinOS v1.1.4+ and Agent v1.8.0+.</p> 
        <h4 id="setting-up-resin-sync-">Setting up resin sync.</h4> 
        <p>Resin sync is bundled in with our handy resin CLI. The CLI allows you to basically do all your resin.io management from the comfort of the command line. Read the <a href="/tools/cli/">CLI reference</a> more info on all the cool things it can do.</p> 
        <p><strong>Warning:</strong> Currently resin sync is <strong>NOT</strong> supported on Windows. Support is currently being worked on, you can check the progress on this on the <a href="https://github.com/resin-io-modules/resin-sync/blob/feat/windows-support/README.md#windows">git repository</a>.</p> 
        <p>To install resin CLI and sync you need at least <a href="https://nodejs.org/en/"><code>node.js 4.0.0</code></a> on your development machine, then run:</p> 
        <pre><code>$ npm install --global --production resin-cli
</code></pre>
        <p>You may need to run the install with admin privileges depending on how you have installed node.js.</p> 
        <p><strong>Note:</strong> If you already have resin CLI installed, you will need to upgrade to resin-cli v4.0.0+</p> 
        <p>Once the CLI is installed globally, login with your resin account:</p> 
        <pre><code>$ resin login
</code></pre>
        <p>You should then be presented with 3 options to login. The recommended method is <code>Web authorisation</code> which will open a dialog in your web browser and ask you to authorise the use of the CLI.</p> 
        <pre><code>resin:simple-server-node shaun$ resin login
______          _         _
| ___ \        (_)       (_)
| |_/ /___  ___ _ _ __    _  ___
|    // _ \/ __| | '_ \  | |/ _ \

\_| \_\___||___/_|_| |_(_)_|\___/

Logging in to resin.io
? How would you like to login? (Use arrow keys)

  Credentials
  Authentication token
  I don't have a Resin account!
</code></pre>
        <h4 id="setting-up-node-js-cross-compilation">Setting up Node.js cross-compilation</h4> 
        <p>Cross-compilation is not needed when using <code>resin sync</code> and Node.js as it is an interpreted language. Interpreted languages are programming languages in which programs may be run from source code form, they do not need to be compiled.</p> 
        <h4 id="using-resin-sync">Using resin sync</h4> 
        <p>You are now ready to start using <code>resin sync</code>, so open a terminal in the directory which we were using earlier in this guide. Make a trivial change to your source code and then run:</p> 
        <pre><code>$ resin sync --source . --destination /usr/src/app
</code></pre>
        <p>It should prompt you to select a device, then it will sync all the files in the root of your directory to <code>/usr/src/app</code> on your selected device and finally it will restart the container. You should see something similar to:</p> 
        <pre><code>resin:simple-server-node shaun$ resin sync --source . --destination /usr/src/app
Getting information for device: 5dc2c87ea2caf79bc4e25ae638c6d5a35a75cecf22e8f914331dcb2f588f4b
Application container stopped.
Synced /usr/src/app on 5dc2c87.
Application container started.

resin sync completed successfully!
</code></pre>
        <p>Your new code changes should be up an running in under 30 seconds, <strong>Great success!!</strong></p> 
        <h5 id="some-notes-and-caveats-on-resin-sync">Some Notes and Caveats on Resin Sync</h5> 
        <ul> 
         <li><p>Resin sync directly syncs all the files in the root of your directory to the selected destination directory on your device. If you are using a compiled language you will need to set up cross-compiling on your local development machine and include a <code>.resin-sync.yml</code> file that tells <code>resin sync</code> how to compile your code and sync the resulting executable.</p> </li> 
         <li><p>It is not possible(~easy) to install dependencies using resin sync. So if you need to do an <code>apt-get</code> or add something to either your <code>Dockerfile</code>, <code>package.json</code> or <code>requirements.txt</code>, then you will need to go through the standard <code>git push resin master</code> build pipeline.</p> </li> 
         <li><p>If you know the device UUID you can just run: <code>resin sync &lt;uuid&gt;</code>. This is useful for using resin sync programmatically, in say a <a href="http://gulpjs.com/">gulp</a> workflow, as it does not require interactive action to confirm the destination device.</p> </li> 
         <li><p>If no <code>--destination / -d</code> option is set, an interactive dialog will ask you to provide a destination directory. You can skip this by hitting 'enter' and <code>resin sync</code> will use the <code>/usr/src/app</code> device directory as the default sync destination.</p> </li> 
         <li><p>After every <code>resin sync</code> the updated settings are saved in <code>.resin-sync.yml</code> in your projects's local directory and will be used in later invocations. You can also change any option by editing '.resin-sync.yml' directly.</p> </li> 
         <li><p>A caveat, if you are using a DSA key, some newer openSSH clients do not allow them by default. So you may have to add the following option to <code>~/.ssh/config</code> : <code>PubkeyAcceptedKeyTypes=+ssh-dss</code></p> </li> 
         <li><p>Resin sync is currently permitted to device owners only. The owner of a device is the user who provisioned it, so you will not be able to use resin sync on devices of shared applications that you did not provision.</p> </li> 
        </ul> 
        <h5 id="one-last-tip-">One last Tip!</h5> 
        <p>One of the many useful features of <a href="/tools/cli/">resin-cli</a> is <code>resin ssh</code>, this command allows you to quickly SSH into a device's running container and run test commands or pull out some logs.</p> 
        <p>Provided you are already logged in on the CLI and you have a device online, you can use <code>resin ssh &lt;uuid&gt;</code> to access the container. Here is an example:</p>
