           <h1 class="article-title" itemprop="name"> No More Global Npm Packages </h1> 
          </header> 
          <p class="article-byline"> <a href="/javascript/no-more-global-npm-packages/" class="article-date"> <time datetime="2015-03-02T12:00:22.000Z" itemprop="datePublished">2015/03/02</time> </a> </p> 
          <div class="article-entry" itemprop="articleBody"> 
           <p>The JavaScript development community has been welcoming new and powerful tools by the dozens each year, almost too fast for mortals to keep up with. Many of these tools come with npm packages that you can install globally so you can use the command line tool from anywhere on your computer. This can be very convenient, but is it the <em>right</em> way to do it? Is there a better alternative?<a id="more"></a></p> 
           <div style="text-align:center;">
            <!-- Ezoic - In Content - under_first_paragraph -->
            <span id="ezoic-pub-ad-placeholder-107"></span>
            <span style="display:inline-block;float:none;margin-bottom:2px !important;margin-left:0px !important;margin-right:0px !important;margin-top:2px !important;min-height:300px;min-width:336px;" class="ezoic-ad medrectangle-3 adtester-container adtester-container-107" data-ez-name="joezimjs_com-medrectangle-3"><span id="div-gpt-ad-joezimjs_com-medrectangle-3-0" ezaw="336" ezah="280" style="position:relative;z-index:0;display:inline-block;" class="ezoic-ad"><script data-cfasync="false" type="text/javascript" style="display:none;">eval(ez_write_tag([[336,280],'joezimjs_com-medrectangle-3','ezslot_0']));</script></span></span>
            <!-- End Ezoic - In Content - under_first_paragraph -->
           </div> 
           <h2 id="Is-it-Right"><a href="#Is-it-Right" class="headerlink" title="Is it Right?"></a>Is it Right?</h2>
           <p>Yes, it sure is… but no. Well, which is it!?!? Both. If you’re experimenting with a tool or just using it in general, feel free to use it as a global command line tool. If your project is actually dependent on the tool, then installing it globally probably isn’t the best way to go.</p> 
           <p>Why not? Well, just like pretty much every package on npm, the command line tools are versioned. If your local/development machine has a different version of the tool than another machine that needs to run that tool for that project, then there could be incompatibilities. It’d be nice if we had an option to save global dependencies inside of <code>package.json</code>, but honestly a <em>package</em> file shouldn’t include <em>global</em> utilities, as <a href="https://github.com/npm/npm/issues/2949" target="_blank" rel="external">discussed in a GitHub issue for npm</a>.</p> 
           <p>Grunt did a good job of minimizing this issue by creating a separate npm package for the cli tool that is extremely minimal so you can keep it installed globally with minimal chances of incompatibility. There still are no guarantees, though.</p> 
           <h2 id="What’s-the-Alternative"><a href="#What’s-the-Alternative" class="headerlink" title="What’s the Alternative?"></a>What’s the Alternative?</h2>
           <p>npm is the alternative. Specifically, you should include your development tools in <code>devDependencies</code> in <code>package.json</code>. That way they can be versioned for each project. But then how do you run it on the command line? Nobody wants to have to type in <code>node ./node_modules/.bin/grunt</code> every time they want to run a Grunt command. While that would work for the majority of tools, there is a slightly simpler way: <a href="https://docs.npmjs.com/misc/scripts" target="_blank" rel="external">npm scripts</a>. You can use <code>npm run WHATEVER</code> to run any command you put into the <code>scripts</code> section of your <code>package.json</code> file, and if you reference a command that doesn’t exist on your computer’s PATH, then it’ll go searching through the node modules of your project for it.</p>

           <p>You can run it the same way, but it’ll always be verbose now. This is a great way to make sure some options that you always want set are used without having to specify them every time.</p> 
           <p>I’ve run into a significant snag, though when doing this in Powershell on Windows. Assume we are still using the <code>"grunt": "grunt"</code> script setup. If I try typing <code>npm run grunt -- -h</code> in Powershell, then the <code>-h</code> argument will actually be sent to <code>npm run</code> instead of to the <code>grunt</code> command. Any arguments you try to pass in that start with <code>-</code> or <code>--</code> will not be sent to the command in the <code>scripts</code> configuration, but to the <code>npm run</code> command. This doesn’t seem to be an issue with the standard cmd or Git Bash on Windows, nor does it look like Linux/Mac users are affected.</p> 
           <p>From countless comments I’ve read on the internet, it doesn’t sound like there are terribly many Powershell + npm users out there, but I <em>very highly doubt</em> that I’m the only one. In any case, this issue doesn’t come up all that often, since I tend to put all the commands I’m going to actually run into the <code>scripts</code>, so I rarely need to pass arguments and I tend to end up with nice aliases that actually tends to make it so that <code>npm run COMMAND</code> is shorter than the actual command it is running.</p> 
           <p>In fact, I’m looking around and trying to move beyond Grunt, Gulp, and the like, so I’m trying to use tools individually, instead of their task runner plugins (e.g. using the JSHint command line tool instead of the Grunt plugin). For more information, see <a href="http://blog.keithcirkel.co.uk/how-to-use-npm-as-a-build-tool/" target="_blank" rel="external">“How to use npm as a Build Tool”</a>, but I’m still experimenting with that and I’m certainly not trying to convince you of that just yet. This article is simply about removing the need for global npm packages.</p> 
           <h2 id="Pros-and-Cons"><a href="#Pros-and-Cons" class="headerlink" title="Pros and Cons"></a>Pros and Cons</h2>
           <p>There are certainly some pros and cons to this approach, so let’s take a look at some:</p> 
           <h3 id="Cons"><a href="#Cons" class="headerlink" title="Cons"></a>Cons</h3>
           <ul> 
            <li>It usually takes more keystrokes thanks to needing to throw <code>npm run</code> in front of everything and needing to add <code>--</code> in every time you want to throw some extra arguments in. Of course, this is negated when you use the scripts to convert long command names down to a short alias.</li> 
            <li>It takes a bit of extra work up front to add the scripts into the <code>package.json</code> file.</li> 
           </ul> 
           <h3 id="Pros"><a href="#Pros" class="headerlink" title="Pros"></a>Pros</h3>
           <ul> 
            <li>You have an easy place to find documentation on what command should be run. Just typing <code>npm run</code> by itself will show all the aliases and commands that are run for those aliases.</li> 
            <li>Depending on how you set up the npm scripts, they can act as an abstraction layer so that if you want to switch from Grunt to Gulp or something similar, you can just change what commands are being run inside the <code>scripts</code> configuration and never have to change the commands you actually type into the command line.</li> 
            <li>Obviously, one pro has to do with how I started this post: eliminating issues regarding version conflicts from global packages.</li> 
            <li>Finally, when someone clones your project, it increases the likeliness that all they will need to do is run <code>npm install</code> and they will actually get all of the dependencies installed.</li> 
           </ul> 
           <h2 id="Closing-Questions"><a href="#Closing-Questions" class="headerlink" title="Closing Questions"></a>Closing Questions</h2>
           <p>If I missed any pros or cons let me know. In any case, what do you think? Is it worth the fuss to prevent global package version conflicts? If you say yes, then how much do we rely on npm scripts versus letting other tools, like task runners do the real work? These are certainly questions you’ll have to answer for yourself or discuss with your team and not something I can just tell you the “right” answer to. I’d still love to hear your thoughts on using npm scripts to any extent, so let me hear it! God bless and happy coding!</p>
