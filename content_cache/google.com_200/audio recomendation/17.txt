     <h1 class="entry-title">Adding TypeScript Type Definitions to the Stream JavaScript API Client Library</h1> 
     <div class="post-meta">
      <div class="post-meta-byline author">
       <span class="label">Written by: </span> 
       <a href="https://getstream.io/blog/author/dwight/" rel="post-meta-author" class="fn">Dwight Gunning</a>
      </div>
      <div>
       <time class="post-meta-published" datetime="2017-09-05T13:22:16+00:00"><span class="label">Published on:</span> September 5, 2017 at 1:22 pm.</time>
      </div>
      <div>
       <time class="post-meta-updated" datetime="2017-09-28T21:51:34+00:00"><span class="label">Updated on:</span> September 28, 2017 at 9:51 pm.</time>
      </div>
      <div class="post-meta-tags">
       <span class="label">Tags: </span>
       <a href="https://getstream.io/blog/tag/angular/" rel="tag">angular</a>, 
       <a href="https://getstream.io/blog/tag/javascript/" rel="tag">javascript</a>, 
       <a href="https://getstream.io/blog/tag/typescript/" rel="tag">typescript</a>
      </div>
     </div> 
    </header> 
    <div class="entry-content"> 
     <p>TypeScript is a language that Stream has been interested in for quite some time. Over the past year, we’ve had an increasing number conversations with TypeScript and Angular developers. So naturally, it’s been our ambition to eventually distribute Type Definitions for our <a href="https://github.com/getstream/stream.js">JavaScript API Client</a> library. That time has come.</p> 
     <p>The process of coding Type Definitions is straight forward. However while developing Type Definitions, it’s useful to see them in action in a real project, so uncertainty about how the <code>.d.ts</code>&nbsp;file would eventually be distributed proved to be a sticking point when writing and testing Type Definitions for the first time.</p> 
     <p>Of the options available, which are detailed below, we chose to bundle Type Definitions within our JavaScript package. This post brings together what we learned and the steps we went through to get there. This should help other developers new to TypeScript and Type Definitions to do likewise.</p> 
     <h2>A TypeScript primer</h2> 
     <p>Let’s begin with a quick primer on the language and highlight why Type Definitions are of interest.</p> 
     <p>TypeScript is a modern programming language that is a superset of the JavaScript language. It is compiled (in fact, is&nbsp;<a href="https://www.stevefenton.co.uk/2012/11/compiling-vs-transpiling/">transpiled</a>) to plain JavaScript. This is quite similar to the <a href="http://coffeescript.org/">CoffeeScript</a> language and also to aspects of the <a href="https://babeljs.io/">Babel</a> project which transpiles between JavaScript versions.</p> 
     <p>The main features of <a href="https://www.typescriptlang.org/">TypeScript</a>:</p> 
     <ul> 
      <li>It is an Object Oriented Programming language</li> 
      <li>It provides for optional type hints <em>(which we’ll focus on in this post)</em></li> 
      <li>It offers the possibility of codebase integration between TypeScript and JavaScript</li> 
     </ul> 
     <p>If you’re looking for more, check this concise summary of the <a href="https://ionicframework.com/docs/developer-resources/typescript/">benefits of TypeScript</a>&nbsp;by our friends at <a href="https://ionicframework.com">Ionic</a>.</p> 
     <p>Type Definitions are the concept behind some of the most desirable benefits of TypeScript. They define the “shape” and types associated with a library’s API. They enable developers to add type hints to their code base and these are checked at compilation time which helps avoid bugs. They also facilitate advanced IDE features like “code completion” which can improve the developer experience and speed up development.</p> 
     <h2>Options for distributing Type Definitions</h2> 
     <p>Before we go too far, it’s important to note that there are a few ways that developers can create and distribute the Type Declaration File (identifiable by the <code>.d.ts</code>&nbsp;extension). <a href="https://www.npmjs.com/">npm.js</a>&nbsp;is the de facto tool and repository for JavaScript packages so it’s a logical place for our Type Definitions to end up.</p> 
     <p>The choice of how to get Type Definitions into <em>npm</em>&nbsp;depends on the project’s primary language (TypeScript or JavaScript) as well as one’s ability to commit/contribute changes, and ultimately publish new versions to <em>npm</em>.</p> 
     <h3>Developers and contributors of TypeScript projects</h3> 
     <p>It should be no surprise that for projects written in TypeScript, it is exceptionally easy to generate Type Definitions. It is simply a matter of including the <code>--declaration</code>&nbsp;and/or related&nbsp;<code>tsc</code>&nbsp;<a href="https://www.typescriptlang.org/docs/handbook/compiler-options.html">TypeScript compiler options</a>. From there the generated Type Definitions can be inspected and refined manually to ensure they match the developer’s intent.</p> 
     <p>From there it is simply a matter of including the generated <code>.d.ts</code>&nbsp;file in the resulting JavaScript package and publishing it to <em>npm</em>.</p> 
     <h3>Developers and contributors of JavaScript projects</h3> 
     <p>Developers and contributors working on JavaScript packages have more work to do. As stated in the TypeScript project documentation on <a href="https://www.typescriptlang.org/docs/handbook/declaration-files/publishing.html">publishing Declarations Files</a>, the <code>.d.ts</code>&nbsp;file should be bundled directly within the JavaScript project (emphasis added):</p> 
     <blockquote>
      <p>There are two main ways you can publish your declaration files to npm:</p> 
      <ol> 
       <li>bundling with your npm package, or</li> 
       <li>publishing to the&nbsp;@types organization&nbsp;on npm.</li> 
      </ol> 
      <p><strong>If you control the npm package you are publishing declarations for, then the first approach is favored.</strong></p>
     </blockquote> 
     <p>This bundling is convenient because the source and Type Definition is available in one place and are versioned together as part of one package.</p> 
     <h3>Everyone without the ability to commit or raise PRs and publish package versions</h3> 
     <p>The final option that’s available to everyone is to use the <a href="http://definitelytyped.org/">DefinitelyTyped</a> repository and the <a href="https://www.npmjs.com/~types">@types organization</a>. This is an independent, community managed collection of Type Definitions. When it is impractical to bundle Type Definitions with a JavaScript package, DefinitelyTypes is the alternative&nbsp;<a href="https://blogs.msdn.microsoft.com/typescript/2016/06/15/the-future-of-declaration-files/">recommended by the TypeScript project</a>.</p> 
     <p>The main downside to this approach is a side-effect of physically separating code that is otherwise closely related. It leaves the potential for version mismatches and also introduces lead-time associated with submitting Pull Requests through the DefinitelyTyped project repository.</p> 
     <p>Still, it’s considerably better than no Type Definitions at all.</p> 
     <h2>Coding the Type Definitions</h2> 
     <p>The&nbsp;<a href="https://www.typescriptlang.org/docs/handbook/declaration-files/introduction.html">Declaration Files</a>&nbsp;section of the TypeScript documentation provides guidance to both TypeScript and JavaScript developers. There’s not too much more to be said. In addition to the documentation, there are two related open source projects that offer additional support:</p> 
     <ul> 
      <li><strong>dtslint:</strong>&nbsp;a utility built on TSLint for linting TypeScript declaration (.d.ts) files [<a href="https://github.com/Microsoft/dtslint">dtslint on Github</a>]</li> 
      <li><strong>dts-gen:</strong>&nbsp;creates starter TypeScript definition files for any module or library [<a href="https://github.com/Microsoft/dts-gen">dts-gen on Github</a>]</li> 
     </ul> 
     <p>As we chose to add Type Definitions directly to our JavaScript package, we end up with four additional files in our JavaScript package:</p> 
     <ul> 
      <li>.tsconfig</li> 
      <li>tslint.json</li> 
      <li>tests-getstream.ts</li> 
      <li>getstream.d.ts</li> 
     </ul> 
     <p>In addition, the project’s&nbsp;<code>package.json</code>&nbsp;now declares the <code>"types"</code>&nbsp;property. This allows the TypeScript compiler and IDEs with TypeScript support to locate the Type Declaration file and apply the Type Definitions it contains.</p> 
     <h2>The end result</h2> 
     <p>With the Type Definitions bundled with the JavaScript API Client library, the benefits that come from type hints, static type checking and IDE code completion are immediately available.</p> 
     <p>&nbsp;</p>

     <p>&nbsp;</p> 
     <p>To start using Stream’s newly minted JavaScript API Client library with Type Definitions, simply install or upgrade to <a href="https://github.com/GetStream/stream-js/releases/tag/v3.6.0">v3.6.0</a>.</p> 
     <p>To see it in action in a demo project, take a look at the <a href="https://github.com/getstream/stream-angular-example">Stream Angular Example</a> project on Github.</p>
