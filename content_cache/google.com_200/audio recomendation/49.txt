     <h3><a name="jPlayer-fundamentals" class="anchor"></a>jPlayer Fundamentals</h3> 
     <section> 
      <h4><a name="jPlayer-essential-formats" class="anchor"></a>Essential jPlayer Media Formats</h4> 
      <p> The media formats that are essential to jPlayer are those that are supported by both the Flash solution and the HTML5 browsers that do not support Flash, such as iOS. It is important that one of these formats is supplied to jPlayer so that popular browsers will be able to play your media. After an essential format has been supplied, additional counterpart formats may be supplied to increase cross-browser support of the HTML5 solution. See the <code class="prettyprint">jPlayer({"<a href="#jPlayer-option-supplied">supplied</a>": "formats"})</code> option for more details. </p> 
      <dl> 
       <dt>
        Audio
       </dt> 
       <dd> 
        <p>One of the following audio formats must be supplied:</p> 
        <ul> 
         <li>MP3</li> 
         <li>M4A</li> 
        </ul> 
        <p class="note"> Note that, there is little to no benefit in providing both, due to the way support pans out with HTML5 browsers supporting either both or neither. </p> 
       </dd> 
      </dl> 
      <dl> 
       <dt>
        Video
       </dt> 
       <dd> 
        <p>The following video format must be supplied:</p> 
        <ul> 
         <li>M4V</li> 
        </ul> 
       </dd> 
      </dl> 
     </section> 
     <section> 
      <h4><a name="jPlayer-size" class="anchor"></a>Setting the size of jPlayer</h4> 
      <p> Set the width and height of jPlayer using the <code class="prettyprint">jPlayer({<a href="#jPlayer-option-size">size</a>:Object})</code> constructor option. </p> 
      <p> The full-screen size is set using the <code class="prettyprint">jPlayer({<a href="#jPlayer-option-sizeFull">sizeFull</a>:Object})</code> constructor option. </p> 
      <p> Note that the background-color may be set using the constructor option <code class="prettyprint">jPlayer({<a href="#jPlayer-option-backgroundColor">backgroundColor</a>:"#RRGGBB"})</code> </p> 
     </section> 
     <section> 
      <h4><a name="jPlayer-flash-security" class="anchor"></a>Flash Security Rules</h4> 
      <p> The security rules for jPlayer's SWF file have been relaxed using the following code and may be called from any domain. </p> 
      <pre class="prettyprint linenums">
flash.system.Security.allowDomain("*");
flash.system.Security.allowInsecureDomain("*");
</pre> 
      <p>Generally, you will upload the SWF file with the JavaScript file to a directory called "js" on your domain. Use the constructor option <code class="prettyprint">jPlayer({"<a href="#jPlayer-option-swfPath">swfPath</a>":path})</code> to change the path.</p> 
      <p> Technically, the plugin files may be linked to remotely at jplayer.org. However, we request that you don't link to the files on jplayer.com, since we do not have sufficient resources to become a CDN just now. Additionally, the Flash fallback on the remote server would require the use of absolute URLs for all <code class="prettyprint">jPlayer("<a href="#jPlayer-setMedia">setMedia</a>", media)</code> URLs. </p> 
      <p> Attempting to run jPlayer locally on your computer will generate Flash security violations and you would need to enable the local file access using the <a href="http://www.macromedia.com/support/documentation/en/flashplayer/help/settings_manager.html">Flash Settings Manager</a>. See the <a href="http://www.macromedia.com/support/documentation/en/flashplayer/help/index.html">Flash Player Help</a> for more information. </p> 
      <p>To develop locally, install a server on your system, such as Apache, to enable a localhost on your computer. </p>
     </section> 
     <section> 
      <h4><a name="jPlayer-media-encoding" class="anchor"></a>Media Encoding</h4> 
      <p> The media supplied must conform with the codecs supported by HTML5 browsers. The <a href="#jPlayer-essential-formats">essential jPlayer formats</a> (MP3 or M4A for audio and M4V for video) must also conform with the codecs supported by the Adobe Flash plugin. </p> 
      <p> jPlayer needs to distinguish between audio and video. This is because jPlayer behaves slightly differently for the two different media types. For example, the video needs to be shown. As a result, formats that are containers have an audio and video type included in their abbreviation to make them unique. For example, M4A and M4V are both MP4 files, and jPlayer knows to play them as audio and video respectively. </p> 
      <dl> 
       <dt>
        MP3
       </dt> 
       <dd> 
        <p>Since some browsers use the Flash element of jPlayer, the MP3 files used must be encoded according to the browser's Adobe Flash Plugin limitations:</p> 
        <ul> 
         <li>Constant Bitrate Encoded.</li> 
         <li>Sample Rate a multiple of 11,025Hz. ie., 22,050Hz and 44,100Hz are valid sample rates.</li> 
        </ul> 
       </dd> 
      </dl> 
      <dl> 
       <dt>
        M4A / M4V
       </dt> 
       <dd> 
        <p> The MP4 file is a container that supports both audio and video. The M4A format is an audio MP4, and the M4V format is a video MP4. The recommended encoding options for standard and mobile browsers are: </p> 
        <ul> 
         <li>H.264 Baseline Profile Level 3.0 video, up to 640 x 480 at 30 fps. Note that B frames are not supported in the Baseline profile.</li> 
         <li>AAC-LC audio, up to 48 kHz.</li> 
        </ul> 
        <p style="margin-top:1em;"> The metadata should be at the start of the encoded data, since the Flash solution must have the metadata in order to begin playling. The option for this varies from encoder to encoder, but usually refers to optimizing for either web or streaming. This 3rd party link might help if you are having problems with the metadata: <a href="http://rndware.info/products/metadata-mover.html">MetaData Mover</a> </p> 
       </dd> 
      </dl> 
      <dl> 
       <dt>
        OGA / OGV
       </dt> 
       <dd> 
        <p> The OGG file is a container that supports both audio and video. The OGA format is an audio OGG, and the OGV format is a video OGG. The Vorbis and Theora codecs appear to have full support. </p> 
       </dd> 
      </dl> 
      <dl> 
       <dt>
        WEBMA / WEBMV
       </dt> 
       <dd> 
        <p> The WebM file is a container that supports both audio and video. The WEBMA format is an audio WebM, and the WEBMV format is a video WebM. The Vorbis and VP8 codecs appear to have full support. </p> 
       </dd> 
      </dl> 
      <dl> 
       <dt>
        WAV
       </dt> 
       <dd> 
        <p> The WAV format is supported by many HTML5 browsers. We recommend that you avoid it though as a counterpart format. The recommended encoding options are: </p> 
        <ul> 
         <li>8-bit and 16-bit linear PCM</li> 
         <li>Only codec "1" (PCM) is supported.</li> 
        </ul> 
       </dd> 
      </dl> 
      <dl> 
       <dt>
        FLA / FLV
       </dt> 
       <dd> 
        <p> The Flash FLV file is a container that supports both audio and video. The FLA format is an audio FLV, and the FLV format is a video FLV. The FLA/FLV format is supported by the Flash fallback. </p> 
       </dd> 
      </dl> 
      <dl> 
       <dt>
        RTMPA / RTMPV
       </dt> 
       <dd> 
        <p> The RTMP protocol uses a Flash Media Server (FMS) to stream media to jPlayer's Flash solution. The RTMPA format is an audio RTMP stream, and the RTMPV format is a video RTMP stream. The HTML solution does not support RTMP. </p> 
       </dd> 
      </dl> 
      <dl> 
       <dt>
        M3U8A / M3U8V
       </dt> 
       <dd> 
        <p> The M3U8 playlist format is an Apple HLS (HTTP Live Streaming) file. m3u8a for audio and m3u8v for video. This format works well on Apple devices and browsers (Safari), but has limited support elsewhere. </p> 
       </dd> 
      </dl> 
      <dl> 
       <dt>
        M3UA / M3UV
       </dt> 
       <dd> 
        <p> The M3U playlist format is am MPEG URL file. m3ua for audio and m3uv for video. </p> 
       </dd> 
      </dl> 
      <dl> 
       <dt>
        FLAC
       </dt> 
       <dd> 
        <p> The flac audio format. </p> 
       </dd> 
      </dl>

      <h4><a name="jPlayer-server-response" class="anchor"></a>Server Response</h4> 
      <h5>MIME Type</h5> 
      <p>Your domain's server must give the correct MIME type (content type) for all media URLs.</p> 
      <p> Failure to give the correct MIME type will stop the media from working on some HTML5 browsers. This is a common cause of problems that only affect Firefox and Opera. Other browsers are less strict, but the MIME type should always be checked that it is correct if you having problems getting media to play on any browser. </p> 
      <dl> 
       <dt>
        Media MIME Types
       </dt> 
       <dd> 
        <ul> 
         <li>MP3: audio/mpeg</li> 
         <li>MP4: audio/mp4 video/mp4</li> 
         <li>OGG: audio/ogg video/ogg</li> 
         <li>WebM: audio/webm video/webm</li> 
         <li>WAV: audio/wav</li> 
        </ul> 
       </dd> 
      </dl> 
      <p> If you use a common extension for both audio and video media, for example audio.mp4 and video.mp4, then simply use the video version of the MIME type for both of them. ie., video/mp4 </p> 
      <p>On Apache servers, you can use the .htaccess file to set the MIME type based on the file extension:</p> 
      <pre class="prettyprint linenums">
# AddType TYPE/SUBTYPE EXTENSION

AddType audio/mpeg mp3
AddType audio/mp4 m4a
AddType audio/ogg ogg
AddType audio/ogg oga
AddType audio/webm webma
AddType audio/wav wav

AddType video/mp4 mp4
AddType video/mp4 m4v
AddType video/ogg ogv
AddType video/webm webm
AddType video/webm webmv
</pre> 
      <h5>Do Not GZIP the Media</h5> 
      <p>Disable GZIP encoding of all the media files. Media files are already compressed and the GZIP will just waste CPU on your server.</p> 
      <p>The Adobe Flash Plugin will experience issues if you GZIP the media.</p> 
      <p>Do not GZIP the Jplayer.swf file either. Feel free to GZIP the JavaScript.</p> 
      <h5>Byte-Range Requests</h5> 
      <p> Your server must enable <code class="prettyprint">Range</code> requests. This is easy to check for by seeing if your server's response includes the <code class="prettyprint">Accept-Ranges</code> in its header. Most HTML5 browsers enable seeking to new file positions during a download, so the server must allow the new <code class="prettyprint">Range</code> to be requested. </p> 
      <p> Failure to accept byte <code class="prettyprint">Range</code> requests will cause problems on some HTML5 browsers. Often the duration cannot be read from the file as some formats require that the start and end of the file is read to know its duration. Chrome tends to be the browser that has most problems if the <code class="prettyprint">Range</code> request is not enabled on the server, but all browsers will have some issue even if it is only that you have to wait for all the media to load before jumping close to the end. </p> 
      <p> This problem is known to affect Jetty 6 servers with their default configuration. </p> 
      <p> A PHP function has been written by the jPlayer community that can serve media files with support for Range requests. See this <a href="https://groups.google.com/d/msg/jplayer/nSM2UmnSKKA/bC-l3k0pCPMJ">jPlayer Support Group Post</a> on the topic. </p> 
      <h5>Protecting Your Media</h5> 
      <p> Be careful when trying to restrict access to your media files. The media URL must be accessible over the internet by the user and its response must be in the format expected. </p> 
      <p> Using the server response to disable the local cache of media can cause problems with some HTML5 browsers. This can cause the duration of the media to be unknown, which will show as a NaN in the duration of jPlayer. </p> 
      <p> If you do some magic on the backend to make it more secure, them make sure you are accepting the byte <code class="prettyprint">Range</code> requests described above. </p>

      <h4><a name="jPlayer-disable-by-css" class="anchor"></a>How to disable jPlayer with CSS or jQuery</h4> 
      <p> Since jPlayer uses Flash on some browsers, the jPlayer &lt;div&gt; must not be hidden. Mobile browsers do not appear to like their HTML5 media being hidden either. </p>
      <p> </p> You can change the size of the jPlayer &lt;div&gt; to zero if you want to make it invisible. You should also be careful that any jQuery animations, such as fadeIn() or fadeOut(), are not acting on the jPlayer &lt;div&gt; or any of its parents. 
      <p></p> 
      <p>jPlayer will attempt to recover and reconfigure the Flash solution when shown again after being hidden. However, this assumes that you did not issue any commands to jPlayer while it was hidden.</p>

   &lt;/script&gt;

&lt;body&gt;
   &lt;div id="jquery_jplayer"&gt;&lt;/div&gt;
&lt;/body&gt;
</pre>

      <h4><a name="jPlayer-security-restrictions" class="anchor"></a>Security Restrictions</h4> 
      <p> In order to eliminate cross site scripting (XSS), there are some limitations placed on the <code class="prettyprint">{<a href="#jPlayer-option-noConflict">noConflict</a>}</code> option and on the ID name of the jPlayer element. These restrictions are due to the communication between the Flash fallback and the JavaScript in the page, which require the jQuery variable name and the ID of the jPlayer element. This exposed a potential vulnerability through direct access to the Jplayer.swf file, which has been eliminated through the addition of these rules. </p> 
      <p> The whitelisted characters are: <code class="prettyprint">A-Z</code> <code class="prettyprint">a-z</code> <code class="prettyprint">0-9</code> <code class="prettyprint">_</code> <code class="prettyprint">-</code> <code class="prettyprint">.</code> Or in other words, the alpha-numeric characters, underscore, hyphen/minus and period. </p>

      <h4><a name="jPlayer-data" class="anchor"></a>jPlayer Data</h4> 
      <p> The jPlayer data is usually accessed via the <a href="#jPlayer-event-object">event object</a>, since the internal data tends to be updated before <a href="#jPlayer-events">events</a> are generated. An example of how a timeupdate event handler can be created through the constructor option is: </p> 
      <pre class="prettyprint linenums">
$('#jp').jPlayer({
   timeupdate: function(event) { // 4Hz
      // Restrict playback to first 60 seconds.
      if (event.jPlayer.status.currentTime &gt; 60) {
         $(this).jPlayer('stop');
      }
   }
   // Then other options, such as: ready, swfPath, supplied and so on.
});
</pre> 
      <p> The jPlayer object itself may also be accessed through <code class="prettyprint">$('#jp').data('jPlayer')</code>. However, since this gives full read/write access to all data and access to all the methods, even the internal methods, this is only recommended for advanced developers. Reading data is fine, but setting data might have unexpected side effects. </p> 
      <p> The previous timeupdate event code could be (crudely) emulated using an interval with the following code: </p> 
      <pre class="prettyprint linenums">
var jp = $('#jp'), jpData = jp.data('jPlayer');
setInterval(function() {
   // Restrict playback to first 60 seconds.
   if (jpData.status.currentTime &gt; 60) {
      jp.jPlayer('stop');
   }
},100); // 10Hz
</pre>

      <h4><a name="jPlayer-zepto" class="anchor"></a>jPlayer and Zepto</h4> 
      <p> jPlayer has been tweaked so that you can now use <a href="http://zeptojs.com/">Zepto</a> instead of <a href="http://jquery.com/">jQuery</a>. </p> 
      <p> Zepto 1.0+ compiled with the data module is required. </p> 
      <p> If you want to use jPlayer as an AMD module for Zepto, then you will need to edit lines of code in the source. By default, the AMD module has its dependencies set to jQuery, not Zepto. The following is the top of the jPlayer code and you can clearly see the two lines that need to be switched. </p> 
      <pre class="prettyprint linenums">
(function (root, factory) {
   if (typeof define === 'function' &amp;&amp; define.amd) {
      // AMD. Register as an anonymous module.
      define(['jquery'], factory); // jQuery Switch
      // define(['zepto'], factory); // Zepto Switch
   } else {
      // Browser globals
      if(root.jQuery) { // Use jQuery if available
         factory(root.jQuery);
      } else { // Otherwise, use Zepto
          factory(root.Zepto);
      }
   }
}(this, function ($, undefined) {
</pre> 
      <p> But to avoid presumption... These lines here: </p> 
      <pre class="prettyprint linenums">
      define(['jquery'], factory); // jQuery Switch
      // define(['zepto'], factory); // Zepto Switch
</pre> 
      <p> Change to: </p> 
      <pre class="prettyprint linenums">
      // define(['jquery'], factory); // jQuery Switch
      define(['zepto'], factory); // Zepto Switch
</pre> 
      <p> If you are not using the AMD module, then jPlayer is setup ready to use Zepto. Simply include Zepto in your page instead of jQuery. </p> 
      <p class="note"> Zepto support was officially released in jPlayer 2.4.0 </p>

     <h3><a name="jPlayer-constructor" class="anchor"></a>jPlayer Constructor</h3> 
     <h4>$(id).jPlayer( Object: options ) : jQuery</h4> 
     <h5>Description</h5> 
     <p>The jPlayer constructor is applied to the given jQuery selector (normally an ID css selector) and uses the options, if provided. The &lt;div&gt; used by jPlayer should be empty and not used for anything else.</p> 
     <p>In some cases, the jPlayer div can be placed at the top level of the &lt;body&gt; to help avoid <a href="#jPlayer-disable-by-css">effects on the page corrupting jPlayer's operation</a>. This applies to an audio player that does not use the poster feature of jPlayer.</p> 
     <p>Note that the most important option is the <code class="prettyprint">ready</code> event handler, which is a function defining the actions to perform once jPlayer is ready for use. Attempting to issue commands to jPlayer before the <code class="prettyprint">ready</code> event occurs will result in jPlayer raising error events if the Flash fallback is used and commands will be ignored.</p> 
     <p>The next most important option is the <code class="prettyprint">swfPath</code> element, which defines the path of jPlayer's SWF file. Remember to upload the SWF file to your server!</p> 
     <h5>Changing options after instancing jPlayer</h5> 
     <p> Options that may be changed using the <code class="prettyprint">jPlayer("<a href="#jPlayer-option">option</a>", key, value)</code> method after instancing jPlayer, are indicated by the <a href="#jPlayer-option">†</a> symbol. </p> 
     <h5><a name="jPlayer-constructor-event-handlers" class="anchor"></a>Event handlers created through the jPlayer options</h5> 
     <p> Event handlers that may be created through the jPlayer constructor options, are not really options. They are event handlers bound to the jPlayer element, with the ".jPlayer" jQuery event namespace, and are included in the options as an easy way to bind a handler to an event. Event handlers that are listed here in the options are indicated by the <a href="#jPlayer-constructor-event-handlers">‡</a> symbol. </p> 
     <h5>Parameters</h5> 
     <h6>options</h6> 
     <p>Object defining any changes to the default configuration of jPlayer.</p> 
     <dl> 
      <dt>
       <a name="jPlayer-option-ready" class="anchor"></a>ready 
       <a href="#jPlayer-constructor-event-handlers">‡</a>
      </dt> 
      <dd>
       <p>Function : (Default: undefined) : Defines an event handler function that is bound to the $.jPlayer.event.ready event. To reference the current instance, use <code class="prettyprint">$(this)</code>. Generally, it is recommended to use a function here to at least <code class="prettyprint">$(this).jPlayer("<a href="#jPlayer-setMedia">setMedia</a>",media)</code> the instance to a valid url ready for use.</p> 
       <p class="explanation"> The ready event handler is defined at creation to eliminate a race condition between the JavaScript code and Flash code. Thus ensuring the Flash function definitions exist when the JavaScript code is executed.<br> The handler is bound with the <code class="prettyprint">.jPlayer</code> namespace. </p> 
      </dd> 
      <dt>
       <a name="jPlayer-option-swfPath" class="anchor"></a>swfPath
      </dt> 
      <dd> 
       <p> String : (Default: "js") : Defines the path of jPlayer's Jplayer.swf file. This allows the developer to place the SWF file wherever they choose by using an alternative relative path, absolute path or server root relative path. </p> 
       <p> The swfPath option may be a path or a URL to the SWF file with extension <code class="prettyprint">.swf</code>. For example, <code class="prettyprint">jPlayer({swfPath:"/scripts/banana.swf"})</code> where you renamed the Jplayer.swf file to "banana.swf". Do not rename the <code class="prettyprint">.swf</code> extension, or a path will be assumed. </p> 
       <p> You can test that your swfPath is correct on an HTML5 browser that supports flash, by using the constructor option <code class="prettyprint">jPlayer({<a href="#jPlayer-option-solution">solution</a>:"flash, html")</code>.</p> 
       <p> An incorrect swfPath will generate <code class="prettyprint">$.jPlayer.event.<a href="#jPlayer-events">error</a></code> events with error type <code class="prettyprint">$.jPlayer.error.FLASH</code> when you issue commands like <code class="prettyprint">jPlayer("<a href="#jPlayer-setMedia">setMedia</a>", media)</code></p> 
       <p class="explanation"> Independent of trailing slash, ie., "myPath/" is the same as "myPath".<br> The URL given must conform to standard URL Encoding Rules. </p> 
      </dd> 
      <dt>
       <a name="jPlayer-option-solution" class="anchor"></a>solution
      </dt> 
      <dd>
       <p>String : (Default: "html, flash") : Defines the priority of the html and flash solutions. The default is to use HTML first, with a Flash fallback. Swapping the order to <code class="prettyprint">"flash, html"</code> causes the Flash to be used first, with an HTML fallback.</p> 
       <p>Depending on the <code class="prettyprint">supplied</code> formats, jPlayer may even use both solutions. For example, using jPlayer as a media player with <code class="prettyprint">{supplied:"mp3,oga,m4v"}</code> would on browsers like Firefox require the Flash solution to play the video media format, while the audio media can be played in the HTML solution.</p> 
       <p>While you could specify only one solution, we do not recommend that you do it!</p> 
       <p class="explanation">The solution picked by jPlayer depends on your brower and the formats in the supplied option.</p> 
       <p style="margin-bottom:0;"> There is an <strong>experimental aurora.js solution</strong>, that may be enabled by: </p> 
       <ul> 
        <li> Add the aurora.js JavaScript file to the HTML. The library core adds support for WAV: <pre class="prettyprint linenums">
&lt;script src="aurora.js"&gt;&lt;/script&gt;
</pre> </li> 
        <li> Add the Aurora Codec JavaScript files you require to the HTML. For example, add FLAC and OGG support. (OGGs use Vorbis or Opus): <pre class="prettyprint linenums">
&lt;script src="flac.js"&gt;&lt;/script&gt;
&lt;script src="ogg.js"&gt;&lt;/script&gt;
&lt;script src="vorbis.js"&gt;&lt;/script&gt;
&lt;script src="opus.js"&gt;&lt;/script&gt;
</pre> </li> 
        <li> Add <code class="prettyprint">aurora</code> to the <code class="prettyprint">solution</code> option. For example, <code class="prettyprint">{solution:"html,flash,aurora"}</code> will put the aurora player solution in 3rd priority. </li> 
        <li> Set the <code class="prettyprint">auroraFormats</code> option so jPlayer knows which codecs the Aurora solution can play. For example, <code class="prettyprint">{auroraFormats:"wav,flac,oga"}</code> tells jPlayer the Aurora player can play those formats. The formats defined should be in the same manner as jPlayers naming convention of the supplied options formats. For example, the Aurora codec <code class="prettyprint">aac.js</code> enables the jPlayer <code class="prettyprint">m4a</code> format, adding to the <code class="prettyprint">{auroraFormats:"m4a"}</code> option </li> 
        <li> The media must be on the same domain as your webpage for Aurora to access the data in the files. <span class="note">You can setup the CORS headers, but it did not work for me when I tried.</span> </li> 
       </ul> 
       <p> Remember that the inclusion of the Aurora player solution is experimental, and only experienced developers should try using it. The author doubts anyone else will need it, in part due to the Aurora Library needing its internal <code class="prettyprint">seek()</code> method fixed for external library usage. </p> 
      </dd> 
      <dt>
       <a name="jPlayer-option-supplied" class="anchor"></a>supplied
      </dt> 
      <dd> 
       <p> String : (Default: "mp3") : Defines the formats supplied to jPlayer. The order defines the priority, with the left most format being highest. Formats to the right are lower priority. </p> 
       <p> See <a href="#jPlayer-media-encoding">Media Encoding</a> for more details on the formats. See <a href="#jPlayer-essential-formats">Essential Formats</a> for more details on why some formats are manditory. </p> 
       <p> Essential Audio formats: <b>mp3</b> or <b>m4a</b>.<br> Essential Video format: <b>m4v</b>. </p> 
       <p>Counterpart formats: webma, webmv, oga, ogv, wav, fla, flv, rtmpa, rtmpv.</p> 
       <p> All formats in the supplied option must be given in every <code class="prettyprint">jPlayer("<a href="#jPlayer-setMedia">setMedia</a>",media)</code> command. </p> 
       <p> The exception to this rule applies to a Media Player. ie., A player that plays both video and audio. Both the video and audio formats are defined in the supplied option. Then use either the video or audio media when using <code class="prettyprint">jPlayer("<a href="#jPlayer-setMedia">setMedia</a>",media)</code> command. (You must give all the audio formats or all the video formats defined in the supplied option.) </p> 
       <p> The <code class="prettyprint">solution</code> priority dominates the priority of the <code class="prettyprint">supplied</code> formats. jPlayer works by finding the first working format in a given solution. So in Chrome with <code class="prettyprint">{solution:"html,flash"}</code> and <code class="prettyprint">{supplied:"mp3,oga"}</code> the mp3 file would play in native HTML, even though Chrome could play either. </p> 
       <p class="explanation"> The format picked by jPlayer depends on your brower and the solution option.<br> Essential formats are those supported by Flash and (luckily) by browsers that do not allow Flash. ie., iPad/iPhone<br> Counterpart formats are supplied to improve x-browser HTML5 media support. </p> 
      </dd>

       <a name="jPlayer-option-size" class="anchor"></a>size

      </dt>

        </dd>

      </dd>

       <a name="jPlayer-option-sizeFull" class="anchor"></a>sizeFull 
       <a href="#jPlayer-option">†</a>

repeat: function(event) {
  if(event.jPlayer.options.loop) {
    $(this).unbind(".jPlayerRepeat").bind($.jPlayer.event.ended + ".jPlayer.jPlayerRepeat", function() {

    });

    $(this).unbind(".jPlayerRepeat");
  }
}
</pre>

       <pre class="prettyprint linenums">
$("#my-jplayer").unbind($.jPlayer.event.repeat + ".jPlayer");
$("#my-jplayer").bind($.jPlayer.event.repeat + ".jPlayer", function() {
  // Your new repeat handler code
});
</pre>

       <pre class="prettyprint linenums">
$("#my-jplayer").unbind(".jPlayerRepeat");
</pre> 
       <p> The author cannot think of an example where you would want to change the repeat handler after instancing jPlayer, but it has been documented here just in case. </p>

     <h3><a name="jPlayer-methods" class="anchor"></a>jPlayer Methods</h3> 
     <p>jPlayer is controlled by sending the method name through the <code class="prettyprint">$(id).jPlayer()</code> plugin method.</p> 
     <h5><a name="jPlayer-methods-alias" class="anchor"></a>Methods that are an option alias</h5> 
     <p> Methods that are a shortcut/alias to change options using the <code class="prettyprint">jPlayer("<a href="#jPlayer-option">option</a>", key, value)</code> method, are indicated by the <a href="#jPlayer-methods-alias">†</a> symbol. </p> 
     <h4><a name="jPlayer-setMedia" class="anchor"></a>$(id).jPlayer( "setMedia", Object: media ) : jQuery</h4> 
     <h5>Description</h5> 
     <p>This method is used to define the media to play. The media parameter is an object that has properties defining the different encoding formats and a poster image.</p> 
     <p>The <code class="prettyprint">jPlayer("setMedia", media)</code> method must be used to define the media before jPlayer will be able perform other methods like <code class="prettyprint">jPlayer("<a href="#jPlayer-play">play</a>")</code>.</p> 
     <p> All media URLs are converted to their absolute URL. The converted URLs are stored in the <code class="prettyprint">event.jPlayer.status.media</code> object, even if you supplied relative URLs. <span class="note">This convertion to absolute URLs fixes a problem with the Flash MP4 player and the m4a/m4v/flv formats being relative to the SWF, not the page.</span> </p> 
     <p> All formats defined in the constructor option <code class="prettyprint">jPlayer({<a href="#jPlayer-option-supplied">supplied</a>: "f1, f2, fN"})</code> must be given a valid url to the media encoded in that format. The exception is when using both video and audio media types in a single jPlayer. Then either all the <code class="prettyprint">supplied</code> <strong>audio </strong>formats or all the <code class="prettyprint">supplied</code> <strong>video</strong> formats must be defined. </p> 
     <p>The constructor option <code class="prettyprint">jPlayer({<a href="#jPlayer-option-solution">solution</a>: "s1, s2")</code> can be used to define the priority of the HTML and Flash solutions used by jPlayer.</p> 
     <p class="explanation"> jPlayer distinguishes between the media types, and using both video and audio together in a single <code class="prettyprint">setMedia</code> command makes it difficult to determine what actually plays. The decision is based on the priority of <code class="prettyprint">solution</code> and <code class="prettyprint">supplied</code> and the browser being used. ie., The 1st one <code class="prettyprint">supplied</code> that can work in the <code class="prettyprint">solution</code>, will work, be that video or audio. </p> 
     <p> The point at which jPlayer begins downloading the media is complicated by changes in the by changes in the W3C HTML5 media spec for the <code class="prettyprint">media.load()</code> method and the <code class="prettyprint">media.src</code> property. In order for older HTML5 browsers to work with jPlayer, the <code class="prettyprint">media.load()</code> command is still used, where really it should be removed as it is no longer required. </p> 
     <p> The jPlayer constructor option <code class="prettyprint">jPlayer({<a href="#jPlayer-option-preload">preload</a>})</code> is used to control when jPlayer begins downloading new media. Some browsers will begin the download for the whole media when the preload option is "metadata". Others will require that "auto" is used. The only way to ensure download does not begin automatically is to use the preload option "none". Remember that, the media may not download as it depends on the browser you are using. Mobile browser such as Mobile Safari on iOS 4.2 require a gesture before any media is downloaded or played. ie., The user must press a button to initiate the load or play operations the 1st time. </p> 
     <p> Any media playing when the command is issued will be paused. The the download may not be cancelled as it depends on whether the browser follows the recent W3C HTML media spec. In practice, except for the first time setMedia is used in the ready event, the setMedia command is followed by a play command which stops the old download. The exception is with the media player, where the old media of different type (audio or video), will continue downloading. Otherwise the iPhone's built in video player hangs when the src is cleared to cancel the download. </p> 
     <p class="note">The URLs given must conform to standard URL Encoding Rules.</p>

     <h4><a name="jPlayer-play" class="anchor"></a>$(id).jPlayer( "play", [Number: time] ) : jQuery</h4> 
     <h5>Description</h5> 
     <p>This method is used to play the media specified using <code class="prettyprint">jPlayer("<a href="#jPlayer-setMedia">setMedia</a>", media)</code>.</p> 
     <p>If necessary, the file will begin downloading.</p> 
     <p> Without the <code class="prettyprint">time</code> parameter, new media will play from the start. Open media will play from where the play-head was when previously paused using <code class="prettyprint">jPlayer("<a href="#jPlayer-pause">pause</a>", [time])</code>. </p> 
     <p> The optional <code class="prettyprint">time</code> paramater will move the play-head position to the time given in seconds. <code class="prettyprint">jPlayer("play", 0)</code> is useful for forcing play from the start of the track, but is redundant for new media, which always starts from the beginning. This can be used to jump immediately to a play time afer changing media. jPlayer will seek to the time given and play when able to. </p> 
     <p class="explanation">If issued immediately after a <code class="prettyprint">setMedia</code> command, with the <code class="prettyprint">time</code> parameter, and when the browser is using the HTML5 solution, this command will initially fail and an internal timeout is setup to retry the command every 100ms until it succeeds.</p> 
     <h5>Parameters</h5> 
     <dl> 
      <dt>
       time
      </dt> 
      <dd>
       <p>[Optional] Number : Defines the new play-head position in seconds.</p> 
       <p class="explanation">If used while downloading, play will begin once the media is seekable to that point.</p>
      </dd> 
     </dl> 
     <p class="explanation">To make media playback loop, add <code class="prettyprint">jPlayer("play")</code> to the ended event. jPlayer deals with reseting the media to the start in the ended event, so the <code class="prettyprint">time</code> equals zero is not required.</p>

  },
  ended: function() { // The $.jPlayer.event.ended event
    $(this).jPlayer("play"); // Repeat the media
  },

     <h3><a name="jPlayer-options" class="anchor"></a>jPlayer Options</h3> 
     <p> jPlayer allows you to change certain options after instancing. These are indicated by the † symbol. All other options must be defined at creation through the constructor options. </p> 
     <p> Due to the nature of the cssSelector and cssSelectorAncestor options, there are described here in more detail. </p> 
     <h4><a name="jPlayer-option-set-cssSelector" class="anchor"></a>$(id).jPlayer( "option", "cssSelector", String: method, String: selector ) : jQuery</h4> 
     <h5>Description</h5> 
     <p>This option allows developers to change the cssSelector associations after jPlayer has been instanced on the page.</p> 
     <p>The <code class="prettyprint">cssSelector</code> method is used with the jPlayer constructor option <code class="prettyprint">cssSelectorAncestor</code> to create associations between jPlayer methods and CSS entities on the webpage. For example, this enables a play button graphic on the webpage to be associated with the method that executes the play command. </p> 
     <p>By default, jPlayer uses a <a href="#jPlayer-predefined-css-selectors">predefined set of cssSelectors</a>, where all the css selectors are class selectors. The strings are added to the end of the <code class="prettyprint">cssSelectorAncestor</code> string, with a space in between. The default <code class="prettyprint">cssSelectorAncestor</code> is an id selector. By using a single id and a common class structure, multiple instances of jPlayer are easy to add to the page. The idea is that jPlayer is associated with a unique interface, hence the id, and then the rest of the structure is common, hence the classes. </p> 
     <p>The <code class="prettyprint">method</code> may only have one <code class="prettyprint">selector</code> associated with it through jPlayer. An existing association will be removed if a new association is given. An empty string clears exisiting associations.</p> 
     <p class="explanation">jPlayer uses the <code class="prettyprint">".jPlayer"</code> event namespace for binding and unbinding methods to CSS entities.</p> 
     <h5>Parameters</h5> 
     <dl> 
      <dt>
       method
      </dt> 
      <dd>
       <p>String containing the name of the method to associate with the <code class="prettyprint">selector</code>.<br> See the <a href="#jPlayer-predefined-css-selectors">predefined cssSelectors</a> for the list of valid method names. </p>
      </dd> 
      <dt>
       selector
      </dt> 
      <dd>
       <p>String containing the CSS Selector to associate with the <code class="prettyprint">method</code>.<br> Usually a class selector. Eg., ".jp-play"</p>
      </dd> 
     </dl>

     <p>While the skin conforms with the <a href="http://www.w3.org/TR/REC-html40/">HTML 4.01 Spec</a>, unfortunately some browsers do not adhere to the spec.      <h3><a name="jPlayer-known-issues" class="anchor"></a>jPlayer Known Issues</h3> 
     <h4><a name="jPlayer-known-issues-android" class="anchor"></a>Android HTML5 Issues</h4> 
     <p>The incomplete implementation of the HTML5 Media Spec on Android 2.3 causes problems with jPlayer's behaviour.</p> 
     <p style="margin-bottom:0;">Issues with Android 2.3:</p> 
     <ul> 
      <li> Missing ended events. </li> 
      <li> Failure when <code class="prettyprint">setMedia</code> is followed by <code class="prettyprint">play</code>. </li> 
      <li> Failure when the media has finished playing. <span class="note">The audio that was just playing is corrupted by it ending.</span> </li> 
     </ul> 
     <p style="margin-top:1em;">Take a look at this <a href="../demo-01-android/">jPlayer Demo</a> to see a working solution to these issues.</p> 
     <h4><a name="jPlayer-known-issues-android-jquery-1-8-1" class="anchor"></a>Android Browser Crash with jQuery 1.8.1</h4> 
     <p>The GUI autohide animations crash the browser.</p> 
     <p style="margin-bottom:0;">Issues with Android 2.3.4 and jQuery 1.8.1 and jQuery 1.8.2. Fixed in jQuery 1.8.3</p> 
     <p style="margin-top:1em;">jQuery bug <a href="http://bugs.jquery.com/ticket/12497">Ticket #12497</a></p> 
     <h4><a name="jPlayer-known-issues-flash-metadata" class="anchor"></a>Flash and m4a/m4v/flv Metadata</h4> 
     <p> The Flash solution in jPlayer has problems playing the MP4 and FLV type formats when the metadata is missing or at the wrong end of the file. </p> 
     <p style="margin-bottom:0;">Issues with jPlayer's Flash solution:</p> 
     <ul> 
      <li> Symptom: <strong>Flash has to download the entire file before it plays.</strong><br> Reason: <i>Metadata at end of file.</i><br> Solution: Re-encode file for streaming, with metadata at the start. </li> 
      <li> Symptom: <strong>Flash will not pause an FLV</strong><br> Reason: <i>Metadata is missing.</i><br> Solution: Repair the FLV using FLVtool2 or YAMDI.<br> GitHub Issue: <a href="https://github.com/happyworm/jPlayer/issues/89">#89 Can not pause FLV files which have no metadata</a> </li> 
     </ul> 
     <h4><a name="jPlayer-known-issues-browsers" class="anchor"></a>Incompatible Browsers</h4> 
     <p>jPlayer does not work on the following browsers:</p> 
     <ul> 
      <li>Wii Opera</li> 
      <li>Playstation 3 Browser</li> 
     </ul> 
     <h4><a name="jPlayer-known-issues-event-driven" class="anchor"></a>Event Driven Mobile Browsers</h4> 
     <p>The following Mobile browsers require user gestures to trigger commands that effect media playback.</p> 
     <ul> 
      <li>Mobile Safari 
       <ul> 
        <li> The first time a media element is played must be initiated by a user gester. ie., The user must click the play button. This affects the operation of a <code class="prettyprint">jPlayer("<a href="#jPlayer-play">play</a>")</code> in the ready event handler. The browser will ignore the command. jPlayer will simply wait until the user presses the play button. <br> Once the first gesture has been received, JavaScript code is then allowed to do whatever you want with the media element. Note that a jPlayer media player instance uses a audio and a video element. Each require their own gesture. <br> Also affect: <code class="prettyprint">jPlayer("<a href="#jPlayer-load">load</a>")</code> and <code class="prettyprint">jPlayer("<a href="#jPlayer-pause">pause</a>", time)</code> </li> 
       </ul> </li> 
     </ul>

     <h3><a name="reference-html5-media" class="anchor"></a>Reference: HTML5 Browser Audio and Video Support</h3> 
     <p>Here are details on the supported formats by HTML5 browsers. All browser versions are their latest official release.</p> 
     <p class="note"> Format names relate to the <a href="#jPlayer-option-supplied">supplied</a> and <a href="#jPlayer-setMedia">setMedia</a> format property names used by jPlayer. For example: WEBMA is an audio WebM, OGV is a video OGG, and M4A is an audio MP4. For more details on the formats, see <a href="#jPlayer-media-encoding">Media Encoding</a>. </p> 
     <h4><a name="reference-html5-audio-format" class="anchor"></a>HTML5 Audio Format</h4> 
     <p class="note"> An audio WebM file uses the same Vorbis codec as an OGG file. </p> 
     <p>HTML5 browsers and their supported audio file formats:</p> 
     <ul> 
      <li>Firefox (OSX, Win): WEBMA, OGA</li> 
      <li>Safari (OSX, Win): MP3, M4A</li> 
      <li>Mobile Safari iOS4 (iPad, iPhone, iPod): MP3, M4A</li> 
      <li>Opera (OSX, Win): WEBMA, OGA</li> 
      <li>Chrome (OSX, Win): WEBMA, OGA, MP3, M4A</li> 
      <li>IE9 (Win): MP3, M4A <span class="note">(Can install the WebM codec.)</span></li> 
     </ul> 
     <h4><a name="reference-html5-video-format" class="anchor"></a>HTML5 Video Format</h4> 
     <p class="note"> The video WebM (VP8) codec is superior to the video OGG (Theora) codec. </p> 
     <p>HTML5 browsers and their supported video file formats:</p> 
     <ul> 
      <li>Firefox (OSX, Win): WEBMV, OGV</li> 
      <li>Safari (OSX, Win): M4V</li> 
      <li>Mobile Safari iOS4 (iPad, iPhone, iPod): M4V</li> 
      <li>Opera (OSX, Win): WEBMA, OGV</li> 
      <li>Chrome (OSX, Win): WEBMV, OGV, M4V. <span class="note">(Will drop support for M4V soon.)</span></li> 
      <li>IE9 (Win): M4V <span class="note">(Can install the WebM codec.)</span></li> 
     </ul> 
     <h4><a name="reference-html5-audio-streams" class="anchor"></a>HTML5 Audio Streams</h4> 
     <p>HTML5 browsers and their support for audio streams. (Note that, jPlayer's Flash fall-back for non-HTML5 browsers works with MP3 streams.)</p> 
     <p>Audio streams work on:</p> 
     <ul> 
      <li>Firefox (OSX, Win): OGA</li> 
      <li>Safari (OSX): MP3</li> 
      <li>Mobile Safari (iOS4 iPad/iPhone/iPod): MP3</li> 
      <li>Opera (OSX, Win): OGA</li> 
      <li>Chrome (OSX, Win): MP3, OGA</li> 
      <li>IE9 (Win): MP3</li> 
     </ul> 
     <p><br>Audio streams fail on:</p> 
     <ul> 
      <li>Safari (Win): MP3</li> 
     </ul> 
     <h4><a name="reference-html5-audio-resources" class="anchor"></a>External Resources</h4> 
     <p>Some links to resources about HTML5 video and audio support:</p> 
     <ul> 
      <li><a target="_blank" href="http://diveintohtml5.org/video.html">diveintohtml5.org</a> : Video on the Web.</li> 
      <li><a target="_blank" href="http://html5test.com/">html5test.com</a> : HTML5 Test - Where you can test your browser's HTML5 support.</li> 
      <li><a target="_blank" href="http://www.trygve-lie.com/blog/entry/html_5_audio_element_and">www.trygve-lie.com</a> : HTML5 audio element and streaming</li>
