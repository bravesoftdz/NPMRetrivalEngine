    <h1 style="margin: 0.2em 0;font-size: 3em;font-weight: 300;">ScalaTags <a class=" scalatex-site-Styles-headerLink" href="#ScalaTags" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a></h1>

    <p> ScalaTags is a small, <a href="#Performance">fast</a> XML/HTML/CSS construction library for <a href="http://www.scala-lang.org/">Scala</a> that takes fragments in plain Scala code that look like this: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">// import scalatags.Text.all._
// OR
// import scalatags.JsDom.all._
html(
  head(
    script(src:="..."),
    script(
      "alert('Hello World')"
    )
  ),
  body(
    div(
      h1(id:="title", "This is a title"),
      p("This is a big paragraph of text")
    )
  )
)</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/ExampleTests.scala#L41-L58" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> And turns them into HTML like this: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="xml scalatex-site-Styles-highlightMe">&lt;html&gt;
    &lt;head&gt;
        &lt;script src="..."&gt;&lt;/script&gt;
        &lt;script&gt;alert('Hello World')&lt;/script&gt;
    &lt;/head&gt;
    &lt;body&gt;
        &lt;div&gt;
            &lt;h1 id="title"&gt;This is a title&lt;/h1&gt;
            &lt;p&gt;This is a big paragraph of text&lt;/p&gt;
        &lt;/div&gt;
    &lt;/body&gt;
&lt;/html&gt;</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/ExampleTests.scala#L60-L72" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> This document should contain enough to get you started using Scalatags to start generating your HTML, whether on a web-server on in the browser using <a href="#Scala.js">Scala.js</a>. For more information checkout the: </p> 
    <ul> 
     <li> <a href="https://groups.google.com/forum/#!forum/scalatags">Mailing List</a></li>

    <div style="margin: 0px;color: #333;text-align: center;padding: 2.5em 2em 0;border-bottom: 1px solid #eee;display: block;" id="GettingStarted" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag">
     <h1 style="margin: 0.2em 0;font-size: 3em;font-weight: 300;">Getting Started<a class=" scalatex-site-Styles-headerLink" href="#GettingStarted" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a></h1>
     <br>
    </div> 
    <p></p> ScalaTags is hosted on 
    <a href="http://search.maven.org/#search%7Cga%7C1%7Cscalatags">Maven Central</a>; to get started, simply add the following to your 
    <code>build.sbt</code>: 
    <pre><code class="scala scalatex-site-Styles-highlightMe">libraryDependencies += "com.lihaoyi" %% "scalatags" % "0.6.7"
</code></pre> 
    <p> And you're good to go! Open up a <code>sbt console</code> and you can start working through the <a href="#Examples">Examples</a>, which should just work when copied and pasted into the console. </p> 
    <h1 id="Scala.js" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#Scala.js" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>Scala.js</h1> 
    <p> To use Scalatags with a ScalaJS project, add the following to the <code>built.sbt</code> of your ScalaJS project: </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">libraryDependencies += "com.lihaoyi" %%% "scalatags" % "0.6.7"
</code></pre> 
    <p> And you should be good to go generating HTML fragments in the browser! Scalatags has no dependencies, and so all the examples should work right off the bat whether run in Chrome, Firefox or Rhino. Scalatags 0.5.2 is currently only compatibly with ScalaJS 0.5.3+. </p> 
    <div style="margin: 0px;color: #333;text-align: center;padding: 2.5em 2em 0;border-bottom: 1px solid #eee;display: block;" id="WhyScalatags" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag">
     <h1 style="margin: 0.2em 0;font-size: 3em;font-weight: 300;">Why Scalatags<a class=" scalatex-site-Styles-headerLink" href="#WhyScalatags" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a></h1>
     <br>
    </div> 
    <p> The core functionality of Scalatags is a <a href="https://github.com/lihaoyi/scalatags/blob/master/scalatags/shared/src/main/scala/scalatags/generic/Core.scala">tiny amount of code</a>, and yet it provides all the functionality of large frameworks like Python's <a href="http://jinja.pocoo.org/docs/sandbox/">Jinja2</a> or C#'s <a href="http://msdn.microsoft.com/en-us/vs2010trainingcourse_aspnetmvc3razor.aspx">Razor</a>, and out-performs the competition by a <a href="#Performance">large margin</a>. It does this by leveraging the functionality of the Scala language to do almost <b>everything</b>, meaning you don't need to learn a second template pseudo-language just to stitch your HTML fragments together</p> 
    <p> Since ScalaTags is pure Scala, any editor that understands Scala will understand scalatags.Text. Not only do you get syntax highlighting, you also get code completion: </p> 
    <img style="width: 100%;" alt="Autocomplete" src="https://raw.githubusercontent.com/lihaoyi/scalatags/master/readme/resources/Autocomplete.png"> 
    <p> and Error Highlighting: </p> 
    <img style="width: 100%;" alt="Error Highlighting" src="https://raw.githubusercontent.com/lihaoyi/scalatags/master/readme/resources/ErrorHighlighting.png"> 
    <p> and in-editor documentation: </p> 
    <img style="width: 100%;" alt="Inline Documentation" src="https://raw.githubusercontent.com/lihaoyi/scalatags/master/readme/resources/InlineDocs.png"> 
    <p> And all the other good things (<b>jump to definition</b>, <b>extract method</b>, etc.) you're used to in a statically typed language. No more digging around in templates which mess up the highlighting of your HTML editor, or waiting months for the correct plugin to materialize.</p> 
    <p> Take a look at the <a href="#PriorWork">Prior Work</a> section for a more detailed analysis of Scalatags in comparison to other popular libraries. </p> 
    <div style="margin: 0px;color: #333;text-align: center;padding: 2.5em 2em 0;border-bottom: 1px solid #eee;display: block;" id="BasicExamples" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag">
     <h1 style="margin: 0.2em 0;font-size: 3em;font-weight: 300;">Basic Examples<a class=" scalatex-site-Styles-headerLink" href="#BasicExamples" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a></h1>
     <br>
    </div> 
    <p> This is a bunch of simple examples to get you started using Scalatags. These examples are all in the unit tests, so feel free to click the links to see them in full. For more examples, take a look at the <a href="http://lihaoyi.github.io/scalatags/demo.html">demo page</a>, backed by the code in <a href="https://github.com/lihaoyi/scalatags/blob/master/example/src/main/scala/example/ScalaJSExample.scala">example/</a> </p> 
    <h1 id="HelloWorld" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#HelloWorld" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>Hello World</h1> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">html(
  head(
    script("some script")
  ),
  body(
    h1("This is my title"),
    div(
      p("This is my first paragraph"),
      p("This is my second paragraph")
    )
  )
)</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/ExampleTests.scala#L75-L87" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> The core of Scalatags is a way to generate (X)HTML fragments using plain Scala. This is done by values such as <code>head</code>, <code>script</code>, and so on, which are automatically imported into your program when you <code>import scalatags.Text.all._</code>. See <a href="#ManagingImports">below</a> if you want more fine-grained control over what's imported.</p> 
    <p> This example code creates a Scalatags fragment. We could do many things with a fragment: store it, return it, etc. since it's just a normal Scala value. Eventually, though, you will want to convert it into HTML. To do this, simply use: </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">frag.toString

</code></pre> 
    <p> Which will give you a String containing the HTML representation: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="xml scalatex-site-Styles-highlightMe">&lt;html&gt;
    &lt;head&gt;
        &lt;script&gt;some script&lt;/script&gt;
    &lt;/head&gt;
    &lt;body&gt;
        &lt;h1&gt;This is my title&lt;/h1&gt;
        &lt;div&gt;
            &lt;p&gt;This is my first paragraph&lt;/p&gt;
            &lt;p&gt;This is my second paragraph&lt;/p&gt;
        &lt;/div&gt;
    &lt;/body&gt;
&lt;/html&gt;</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/ExampleTests.scala#L89-L101" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> If you want to add a doctype to your HTML fragment, simply add it as a string: </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">"&lt;!DOCTYPE html&gt;" + html(
  ...
)
</code></pre> 
    <p> The following examples will simply show the initial Scalatag fragment and the final pretty-printed HTML, skipping the intermediate steps. </p> 
    <h1 id="Attributes" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#Attributes" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>Attributes</h1> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">html(
  head(
    script("some script")
  ),
  body(
    h1("This is my title"),
    div(
      p(onclick:="... do some js")(
        "This is my first paragraph"
      ),
      a(href:="www.google.com")(
        p("Goooogle")
      ),
      p(hidden)(
        "I am hidden"
      )
    )
  )
)</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/ExampleTests.scala#L239-L258" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> In Scalatags, each attribute has an associated value which can be used to set it. This example shows how you set the <code>onclick</code> and <code>href</code> attributes with the <code>:=</code> operator. Attributes are all instances of the <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Attr</code> class. This prints: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="xml scalatex-site-Styles-highlightMe">&lt;html&gt;
    &lt;head&gt;
        &lt;script&gt;some script&lt;/script&gt;
    &lt;/head&gt;
    &lt;body&gt;
        &lt;h1&gt;This is my title&lt;/h1&gt;
        &lt;div&gt;
            &lt;p onclick="... do some js"&gt;
                This is my first paragraph&lt;/p&gt;
            &lt;a href="www.google.com"&gt;
                &lt;p&gt;Goooogle&lt;/p&gt;
            &lt;/a&gt;
            &lt;p hidden="hidden"&gt;
                I am hidden&lt;/p&gt;
        &lt;/div&gt;
    &lt;/body&gt;
&lt;/html&gt;</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/ExampleTests.scala#L280-L297" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> All the attributes included in HTML are included in the default import. <code>onclick</code>, <code>id</code>, <code>`class`</code>, <code>href</code>, etc.; the <a href="http://www.lihaoyi.com/scalatags/api/index.html#scalatags.generic.Attrs">Scaladoc</a> lists out what is available. </p> 
    <h1 id="InlineStyles&amp;Classes" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#InlineStyles&amp;Classes" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>Inline Styles &amp; Classes</h1> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">html(
  head(
    script("some script")
  ),
  body(
    h1(backgroundColor:="blue", color:="red")("This is my title"),
    div(backgroundColor:="blue", color:="red")(
      p(cls := "contentpara first")(
        "This is my first paragraph"
      ),
      a(opacity:=0.9)(
        p(cls := "contentpara")("Goooogle")
      )
    )
  )
)</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/ExampleTests.scala#L300-L316" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> In HTML, <code>style</code> attributes are often thought of not as normal attributes (which contain strings), but as lists of key-value pairs. Furthermore, there is a large but finite number of styles, and a style cannot simply be any arbitrary string. The above example shows how CSS classes and inline-styles are typically set. </p> 
    <p> Note that in this case, <code>backgroundColor</code>, <code>color</code> and <code>opacity</code> are all statically typed identifiers. These are all instances of <code>Style</code> and are defined by Scalatags. </p> 
    <p> Of course, <code>style</code> is in the end just an attribute, so you can treat it as such and bind them directly without any fuss: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">html(
  head(
    script("some script")
  ),
  body(
    h1(style:="background-color: blue; color: red;")("This is my title"),
    div(style:="background-color: blue; color: red;")(
      p(`class`:="contentpara first")(
        "This is my first paragraph"
      ),
      a(style:="opacity: 0.9;")(
        p(cls := "contentpara")("Goooogle")
      )
    )
  )
)</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/ExampleTests.scala#L317-L333" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> Both the above examples print the same thing: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="xml scalatex-site-Styles-highlightMe">&lt;html&gt;
    &lt;head&gt;
        &lt;script&gt;some script&lt;/script&gt;
    &lt;/head&gt;
    &lt;body&gt;
        &lt;h1 style="background-color: blue; color: red;"&gt;This is my title&lt;/h1&gt;
        &lt;div style="background-color: blue; color: red;"&gt;
        &lt;p class="contentpara first"&gt;This is my first paragraph&lt;/p&gt;
        &lt;a style="opacity: 0.9;"&gt;
            &lt;p class="contentpara"&gt;Goooogle&lt;/p&gt;
        &lt;/a&gt;
        &lt;/div&gt;
    &lt;/body&gt;
&lt;/html&gt;</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/ExampleTests.scala#L335-L349" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> A list of the attributes and styles provided by ScalaTags can be found in <a href="https://github.com/lihaoyi/scalatags/blob/master/scalatags/shared/src/main/scala/scalatags/generic/Attrs.scala">Attrs</a> and <a href="https://github.com/lihaoyi/scalatags/blob/master/scalatags/shared/src/main/scala/scalatags/generic/Styles.scala">Styles</a>. This of course won't include any which you define yourself. </p> 
    <h1 id="CustomAttributes,Styles,TagNames" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#CustomAttributes,Styles,TagNames" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>Custom Attributes, Styles, Tag Names</h1> 
    <p> The common HTML attributes all have static values to use in your fragments. This keeps things concise and statically checked. However, inevitably you'll want to set some attribute which isn't in the initial list defined by scalatags.Text. This can be done with the <code>attr(s: String)</code> functions that Scalatags provides. If you want to create an empty attribute, you can use <code>attr(...).empty</code>: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">html(
  head(
    script("some script")
  ),
  body(
    h1("This is my title"),
    div(
      p(attr("onclick"):="... do some js")(
        "This is my first paragraph"
      ),
      a(attr("href"):="www.google.com")(
        p("Goooogle")
      ),
      p(attr("hidden").empty)(
        "I am hidden"
      )
    )
  )
)</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/ExampleTests.scala#L259-L278" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="xml scalatex-site-Styles-highlightMe">&lt;html&gt;
    &lt;head&gt;
        &lt;script&gt;some script&lt;/script&gt;
    &lt;/head&gt;
    &lt;body&gt;
        &lt;h1&gt;This is my title&lt;/h1&gt;
        &lt;div&gt;
            &lt;p onclick="... do some js"&gt;
                This is my first paragraph&lt;/p&gt;
            &lt;a href="www.google.com"&gt;
                &lt;p&gt;Goooogle&lt;/p&gt;
            &lt;/a&gt;
            &lt;p hidden="hidden"&gt;
                I am hidden&lt;/p&gt;
        &lt;/div&gt;
    &lt;/body&gt;
&lt;/html&gt;</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/ExampleTests.scala#L280-L297" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> You can also use <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">css(...)</code> to create a custom inline style, if for some reason the style you want isn't available in the default set: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">div(
  attr("data-app-key") := "YOUR_APP_KEY",

    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">{

    <pre><code class="scala scalatex-site-Styles-highlightMe">val rssXml = {

    <h1 id="Auto-escapingandunsanitizedInput" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#Auto-escapingandunsanitizedInput" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>Auto-escaping and unsanitized Input</h1> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">val evilInput1 = "\"&gt;&lt;script&gt;alert('hello!')&lt;/script&gt;"

    <p> By default, any text that's put into the Scalatags templates, whether as a attribute value or a text node, is properly escaped when it is rendered. Thus, when you run the following snippet, you get this:</p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="xml scalatex-site-Styles-highlightMe">&lt;html&gt;

    <p> As you can see, the contents of the variables <code>evilInput1</code> and <code>evilInput2</code> have been HTML-escaped, so you do not have to worry about un-escaped user input messing up your DOM or causing XSS injections.     <p> If you <b>really</b> want, for whatever reason, to put unsanitized input into your HTML, simply surround the string with a <code>raw</code> tag: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">val evilInput = "&lt;script&gt;alert('hello!')&lt;/script&gt;"

    script("some script")
  ),
  body(
    h1("This is my title"),
    raw(evilInput)
  )
)</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/ExampleTests.scala#L531-L543" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> prints </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="xml scalatex-site-Styles-highlightMe">&lt;html&gt;
    &lt;head&gt;
        &lt;script&gt;some script&lt;/script&gt;
    &lt;/head&gt;
    &lt;body&gt;
        &lt;h1&gt;This is my title&lt;/h1&gt;
        &lt;script&gt;alert('hello!')&lt;/script&gt;
    &lt;/body&gt;
&lt;/html&gt;</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/ExampleTests.scala#L545-L554" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> As you can see, the <code class="xml scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">&lt;script&gt;</code> tags in <code>evilInput</code> have been passed through to the resultant HTML string unchanged. Although this makes it easy to open up XSS holes (as shown above!), if you know what you're doing, go ahead.</p> 
    <p> There isn't any way to put unescaped text inside tag names, attribute names, or attribute values. </p> 
    <h1 id="Layouts" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#Layouts" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>Layouts</h1>

    <h1 id="GroupingFragsandModifiers" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#GroupingFragsandModifiers" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>Grouping Frags and Modifiers</h1> 
    <p> Renderable fragments are of the type <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Frag</code>; these may represent a single HTML tag, multiple HTML tags, or even a mix of HTML tags and strings. For example, the <code>h1</code> and <code>p</code> in the below snippet can be thought of as a single <code>Frag</code>: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">{

    <p> These can be grouped together as a <code>Seq[Frag](...)</code> and used together: </p>

    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">{

    <p> Similarly, <code>Modifier</code>s such as <a href="#Attributes">Attributes</a> or <a href="#InlineStyles&amp;Classes">Inline Styles &amp; Classes</a> can also be grouped together as aggregate <code>Modifier</code>s. Taking the modifiers from this snippet: </p>

    <pre class=" scalatex-site-Styles-hoverContainer"><code class="xml scalatex-site-Styles-highlightMe">&lt;div class="my-bootstrap-class" style="color: red;"&gt;&lt;/div&gt;</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/ExampleTests.scala#L752-L753" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> Any "arbitrary HTML blob" that isn't guaranteed to have a single root tag is basically a <code>Frag</code>, which is a pretty common thing to take as an argument or return as the result of a function, and <code>frag</code> makes it convenient to deal with these.     <p> Similarly, the <code>Modifier</code> type lets you represent any ad-hoc collection of zero-or-more attributes, styles or children that you can apply to a HTML tag.     <p> More details about <code>Modifier</code>s and <code>Frag</code>s can be seen in the documentation of Scalatags' <a href="#Internals">Internals</a> </p> 
    <h1 id="ManagingImports" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#ManagingImports" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>Managing Imports</h1> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">// bundle is scalatags.Text or scalatags.JsDom
import bundle.short._
div(
  p(*.color:="red", *.fontSize:=64.pt)("Big Red Text"),

    <p> Apart from using <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">import scalatags.Text.all._</code>, it is possible to perform the imports manually, renaming whatever you feel like renaming. The example above provides an example which imports all HTML tags, but imports <code>Attrs</code> and <code>Styles</code> aliased rather than dumping their contents into your global namespace. This helps avoid polluting your namespace with lots of common names (e.g. <code>value</code>, <code>id</code>, etc.) that you may not use.</p> 
    <p> The main objects which you can import things from are:</p> 
    <ul> 
     <li> <code>tags</code>: common HTML tags</li> 
     <li> <code>tags2</code>: less common HTML tags</li> 
     <li> <code>attrs</code>: common HTML attributes</li> 
     <li> <code>styles</code>: common CSS styles</li> 
     <li> <code>styles2</code>: less common CSS styles</li> 
     <li> <code>svgTags</code>: SVG tags</li> 
     <li> <code>svgAttrs</code>: attributes only associated with SVG elements</li> 
     <li> <code>DataConverters</code>: convenient extensions (e.g. <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">10.px</code>) to create the CSS datatypes </li>
    </ul> 
    <p> You can pick and choose exactly which bits you want to import, or you can use one of the provided aggregates: </p> 
    <ul> 
     <li> <code>all</code>: this imports the contents of <code>Tags</code>, <code>Attrs</code>, <code>Styles</code> and <code>DataConverters</code> </li> 
     <li> <code>short</code>: this imports the contents of <code>Tags</code> and <code>DataConverters</code>, but aliases <code>Attrs</code> and <code>Styles</code> as <code>*</code></li>
    </ul> 
    <p> Thus, you can choose exactly what you want to import, and how: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">{

  import bundle.{attrs =&gt; attr, styles =&gt; css, _}

    <p> Or you can rely on a aggregator like <code>all</code>, which the rest of the examples use, or <code>short</code>. <code>short</code> imports <code>Attrs</code> and <code>Styles</code> as <code>*</code>, making them quick to access without cluttering the global namespace: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">// bundle is scalatags.Text or scalatags.JsDom

    <p> Both these examples print: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="xml scalatex-site-Styles-highlightMe">&lt;div&gt;
    &lt;p style="color: red;"&gt;Red Text&lt;/p&gt;
    &lt;img href="www.imgur.com/picture.jpg" /&gt;
&lt;/div&gt;</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/ExampleTests.scala#L31-L35" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> This custom import object also provides you a place to define your own custom tags that will be imported throughout your project e.g. a <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">js(s: String)</code> tag as shorthand for <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">script(src:=s)</code>.     <p> The lesser used tags or styles are generally not imported wholesale by default, but you can always import the ones you need: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">// bundle is scalatags.Text or scalatags.JsDom

    <p> This prints: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="xml scalatex-site-Styles-highlightMe">&lt;div&gt;

    <h1 id="CustomBundles" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#CustomBundles" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>Custom Bundles</h1> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">object CustomBundle extends Text.Cap with text.Tags with text.Tags2 with Text.Aggregate{

)</code></pre> 
    <p> In addition to importing things piecemeal from various pre-defined namespaces, Scalatags allows you to build a custom bundle which can be used to provide a single-import syntax for whatever import convention you're using. For example, the above snippet sets up a custom bundle which dumps <code>Tags</code> and <code>Tags2</code> in the local namespace, assigns <code>Styles</code> and <code>Styles2</code> to the <code>st</code> object, <code>Attrs</code> to the <code>at</code> object, and ignores Svg-related styles and tags entirely. This lets you enforce a particular convention without having to duplicate the same import-renamings in multiple files in your application.</p> 
    <p> The above snippet prints the following:</p> 
    <pre><code class="xml scalatex-site-Styles-highlightMe">&lt;html&gt;
    &lt;head&gt;
        &lt;script&gt;some script&lt;/script&gt;
    &lt;/head&gt;
    &lt;body&gt;
        &lt;h1 style="background-color: blue; color: red;"&gt;This is my title&lt;/h1&gt;
        &lt;div style="background-color: blue; color: red;"&gt;
        &lt;p class="contentpara first"&gt;This is my first paragraph&lt;/p&gt;
        &lt;a style="opacity: 0.9;"&gt;
            &lt;p class="contentpara"&gt;Goooogle&lt;/p&gt;
        &lt;/a&gt;
        &lt;/div&gt;
    &lt;/body&gt;
&lt;/html&gt;</code></pre> 
    <p> Despite its usefulness in enforcing a particular import convention, custom bundles are completely interoperable with each other or with the default <code>all</code> and <code>short</code> bundles, and the above snippet could equally be written as: </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">import CustomBundle.{st, at}
import Text.all._

CustomBundle.html(
  head(
    script("some script")
  ),
  CustomBundle.body(
    h1(backgroundColor:="blue", st.color:="red")("This is my title"),
    div(st.backgroundColor:="blue", color:="red")(
      p(cls := "contentpara first")(
        "This is my first paragraph"
      ),
      CustomBundle.a(st.opacity:=0.9)(
        p(at.cls := "contentpara")("Goooogle")
      )
    )
  )
)</code></pre> 
    <p> Mixing both things from the <code>all</code> bundle as well as <code>st</code> and <code>at</code> from our own <code>CustomBundle</code>. That's not to say you <b>should</b> do this, but if for some reason if e.g. you're using different conventions for different source files, you can be sure that they'll work together just fine. </p> 
    <div style="margin: 0px;color: #333;text-align: center;padding: 2.5em 2em 0;border-bottom: 1px solid #eee;display: block;" id="DOMBackend" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag">
     <h1 style="margin: 0.2em 0;font-size: 3em;font-weight: 300;">DOM Backend<a class=" scalatex-site-Styles-headerLink" href="#DOMBackend" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a></h1>
     <br>
    </div> 
    <p> Although Scalatags was originally a HTML-String generation library, it now ships with an additional backend that runs only on ScalaJS, available by replacing </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">import scalatags.Text._

</code></pre> 
    <p> with </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">import scalatags.JsDom._</code></pre> 
    <p> This gives you a version of Scalatags that renders directly to <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">dom.Element</code>s, rather than spitting out strings. With the DOM backend, <code>toString</code> still works and generates a HTML string, but is implemented by constructing a tree of <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">dom.Element</code>s and calling <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">.outerHTML</code> on them, and the resultant string may have subtle differences (e.g. order of attributes, or whitespace between styles) from the Text backend. </p> 
    <h1 id="UsingtheDOM" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#UsingtheDOM" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>Using the DOM</h1> 
    <p> The DOM backend provides an additional method <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">.render</code> on all Scalatags fragments, which converts the fragment into a DOM tree: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">val elem = div.render
assert(elem.children.length == 0)
elem.appendChild(p(1, "wtf", "bbq").render)
assert(elem.children.length == 1)
val pElem = elem.children(0).asInstanceOf[Paragraph]
assert(pElem.childNodes.length == 3)
assert(pElem.textContent == "1wtfbbq")</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/js/src/test/scala/scalatags/jsdom/DomTests.scala#L15-L22" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> As you can see, you can manipulate DOM elements directly, calling standard DOM APIs like <code>.children</code>, <code>.appendChild</code>, etc. </p> 
    <p> In addition to the default ability to splice <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">String</code>s, <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Boolean</code>s and <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Int</code>s into the Scalatags fragment, the DOM backend allows you to bind arbitrary <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">js.Any</code>-convertible objects, e.g. the function literal shown below: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">var count = 0
val elem = div(
  onclick := { () =&gt; count += 1},
  tabindex := 1
).render

assert(count == 0)
elem.onclick(null)
assert(count == 1)</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/js/src/test/scala/scalatags/jsdom/DomTests.scala#L66-L75" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> As you can see, the function literal is kept intact rather than being serialized (as you would expect in the Text backend). When called via the attribute of the element, you can see that it functions intact and properly increments the counter. </p> 
    <h1 id="UseCases" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#UseCases" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>Use Cases</h1> 
    <p> Having direct access to the DOM allows a great deal of flexibility that you do not have when working with strings. For example, the example below defines an input-label pair which clears the label when you focus on the input, a common pattern: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">val labelElem = label("Default").render

val inputElem = input(
  `type`:="text",
  onfocus := { () =&gt; labelElem.textContent = ""}
).render

val box = div(
  inputElem,
  labelElem
).render

assert(labelElem.textContent == "Default")
inputElem.onfocus(null)
assert(labelElem.textContent == "")</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/js/src/test/scala/scalatags/jsdom/DomTests.scala#L77-L93" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> This allows a very convenient direct-binding of behaviors without having to trawl through the DOM with jquery selectors or creating lots of IDs and worrying about collisions. </p> 
    <hr> 
    <p> The DOM backend currently requires you to separately add a dependency on <a href="https://github.com/scala-js/scala-js-dom">scala-js-dom</a> for it to function. </p> 
    <div style="margin: 0px;color: #333;text-align: center;padding: 2.5em 2em 0;border-bottom: 1px solid #eee;display: block;" id="Cross-backendCode" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag">
     <h1 style="margin: 0.2em 0;font-size: 3em;font-weight: 300;">Cross-backend Code<a class=" scalatex-site-Styles-headerLink" href="#Cross-backendCode" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a></h1>
     <br>
    </div> 
    <p> If you wish to, it is possible to write code that is generic against the Scalatags backend used, and can be compiled and run on both Text and JsDom backends at the same time! This is done by adding an explicit dependency on <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">generic.Bundle[Builder, Output, FragT]</code>, e.g. how it is done in the unit tests: </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">class ExampleTests[Builder, Output, FragT](bundle: Bundle[Builder, Output, FragT]) extends TestSuite{
  import bundle._
  ...
}
</code></pre> 
    <p> Inside this scope, you are limited to only using the common functionality defined in <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">generic.Bundle</code>, and can't use any Text or JsDom specific APIs. However, in exchange you can write code that works in either backend, by instantiating it with the respective bundle: </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">object ExampleTests extends generic.ExampleTests(scalatags.Text)
object ExampleTests extends generic.ExampleTests(scalatags.JsDom)
</code></pre> 
    <p> For example, here is an example defining a simple HTML widget in a cross-platform way, which can then be used to render both <code>String</code>s on both Scala.js and Scala-JVM, or <code>dom.Element</code>s on Scala.js: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">class SharedTemplates[Builder, Output &lt;: FragT, FragT](val bundle: scalatags.generic.Bundle[Builder, Output, FragT]){
  import bundle.all._
  val widget: Tag = div("hello")
}

object JsTemplates extends SharedTemplates(scalatags.JsDom)
object TextTemplates extends SharedTemplates(scalatags.Text)

val jsVersion: dom.Element = JsTemplates.widget.render
val txtVersion: String = TextTemplates.widget.render

assert(
  jsVersion.tagName.toLowerCase == "div",
  jsVersion.textContent == "hello",
  txtVersion == "&lt;div&gt;hello&lt;/div&gt;"
)</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/js/src/test/scala/scalatags/jsdom/DomTests.scala#L101-L117" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> There's a small amount of boilerplate in setting up the generic <code>SharedTemplates</code> class to take a generic <code>bundle</code>, but once done you can define as many templates inside it as you wish. You can then instantiate <code>SharedTemplates</code> twice, once for <code>scalatags.Text</code> and once for <code>scalatags.JsDom</code>, and can freely use any of the templates on either backend implementation. </p> 
    <p> In general, when writing cross-backend code against an abstract <code>bundle</code>, you are limited to the APIs that are common between both <code>Text</code> and <code>JsDom</code> backends. That means e.g. you cannot pass in callbacks to <code>onclick</code> functions, as that is only possible in <code>JsDom</code> and not in <code>Text</code>. Stepping outside the bounds of the common API is a compile error. </p> 
    <p> This is currently used to shared the bulk of unit tests between the Text and JsDom backends, and could be useful in other scenarios where you may want to swap between them (e.g. using Text on the server, and JsDom on the client where it's available) while sharing as much code as possible.</p> 
    <p> For a concrete, self-contained example of this, look at the <a href="https://github.com/lihaoyi/scalatags/blob/master/example/src/main/scala/example/ScalaJSExample.scala">demo page</a>. </p>

    <h1 id="scalatags.JsDomHTML" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#scalatags.JsDomHTML" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>scalatags.JsDom HTML</h1> 
    <div id="jsdom-html"></div> 
    <h1 id="scalatags.JsDomSVG" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#scalatags.JsDomSVG" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>scalatags.JsDom SVG</h1> 
    <div id="jsdom-svg"></div> 
    <script>example.ScalaJSExample().main("text-html", "text-svg", "jsdom-html", "jsdom-svg")</script> 
    <div style="margin: 0px;color: #333;text-align: center;padding: 2.5em 2em 0;border-bottom: 1px solid #eee;display: block;" id="CSSStylesheets" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag">
     <h1 style="margin: 0.2em 0;font-size: 3em;font-weight: 300;">CSS Stylesheets<a class=" scalatex-site-Styles-headerLink" href="#CSSStylesheets" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a></h1>
     <br>
    </div> 
    <p> Scalatags offsers an internal DSL for constructing CSS Stylesheets. The basic construction looks like: </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">import scalatags.stylesheet._

    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">object Simple extends StyleSheet {
  initStyleSheet()

  val x = cls(
    backgroundColor := "red",
    height := 125
  )
  val y = cls.hover(
    opacity := 0.5
  )

  val z = cls(x.splice, y.splice)

    <p> This generates a stylesheet that looks like </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="css scalatex-site-Styles-highlightMe">.$pkg-Simple-x{
  background-color: red;
  height: 125px;
}
.$pkg-Simple-y:hover{
  opacity: 0.5;
}
.$pkg-Simple-z{
  background-color: red;
  height: 125px;
  opacity: 0.5;

    <p> You can then access <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Simple.styleSheetText</code> in order to do things with the generated stylesheet text. Exactly what you want to do is up to you: In Scala-JVM you will likely want to save it to a file (to be served later) or inlined into some HTML fragment, while in Scala.js you may insert it into the page directly via a <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">style</code> tag.</p> 
    <p> Only <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">cls</code>s defined on <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">trait Simple</code> are gathered up as part of the generated stylesheet. You can override <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">customSheetName</code> to replace the <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">pkg-$class</code> part with something else.</p> 
    <p> Once the stylesheet is defined, you can then immediately start using styles within your Scalatags fragments, just like any other <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Modifier</code>: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">val x = div(
  Simple.x,
  Simple.y
)</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/StyleSheetTests.scala#L217-L221" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="xml scalatex-site-Styles-highlightMe">&lt;div class=" $pkg-Simple-x $pkg-Simple-y"&gt;&lt;/div&gt;</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/StyleSheetTests.scala#L222-L223" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> By default, Scalatag's <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">StyleSheet</code>s have no cascading: you only can define styles for a single class (and it's pseudo-selectors) at a single time. If you want to define styles for multiple tags without a larger HTML fragment, you should define multiple classes. The fact that Scalatag's <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">cls</code> definitions are just normal <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">def</code>s makes managing these classes very easy: you can use standard IDE's or tools to jump-to-definitions, auto-rename them, etc.. Many common mistakes in CSS, such as accidentally mis-spelling a class-name or botching a renaming, become compilation errors.</p> 
    <p> Since Scalatags <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">StyleSheet</code>s are just Scala, you can feel free to use normal Scala techniques (constants, functions, traits, etc.) to DRY up your code without having to learn any special Scalatags-specific mechanisms. </p> 
    <h1 id="InlineModifiers" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#InlineModifiers" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>Inline Modifiers</h1> 
    <p> The above example contains a single <code>hover</code> pseudo-selector, but what if you want the same class to have a whole range of pseudo-selectors? To do that, simply nest the selectors inline inside the <code>cls</code> fragment: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">object Inline extends StyleSheet{
  initStyleSheet()

  val w = cls(
    &amp;.hover(
      backgroundColor := "red"
    ),
    &amp;.active(
      backgroundColor := "blue"
    ),
    &amp;.hover.active(
      backgroundColor := "yellow"
    ),
    opacity := 0.5
  )
}</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/StyleSheetTests.scala#L28-L45" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> This generates the stylesheet: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="css scalatex-site-Styles-highlightMe">.$pkg-Inline-w{
  opacity: 0.5;

.$pkg-Inline-w:hover{
  background-color: red;

  background-color: blue;

    <h1 id="Cascading" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#Cascading" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>Cascading</h1> 
    <p> Occassionally, you really do want your CSS stylesheets to cascade. This may be e.g. because you want to define styles for a large HTML document, and don't want to repeat the "default" classes throughout your HTML. In order to do this with Scalatags, you have to opt in by inheriting from <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">CascadingStyleSheet</code>:</p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="scala scalatex-site-Styles-highlightMe">object Cascade extends CascadingStyleSheet{
  initStyleSheet()

  val y = cls()
  val x = cls(
    a(
      backgroundColor := "red",
      textDecoration.none
    ),
    a.hover(
      backgroundColor := "blue",
      textDecoration.underline
    ),
    (a.hover ~ div ~ y)(
      opacity := 0
    ),
    div.hover(
      div(
        y(
          opacity := 0
        )
      )
    )
  )
}</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/StyleSheetTests.scala#L45-L71" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> This generates: </p> 
    <pre class=" scalatex-site-Styles-hoverContainer"><code class="css scalatex-site-Styles-highlightMe">.$pkg-Cascade-x a{

  opacity: 0;
}
.$pkg-Cascade-x div:hover div .$pkg-Cascade-y{
  opacity: 0;
}</code><a class=" scalatex-site-Styles-headerLink" style="position: absolute;right: 0.5em;top: 0.5em;display: block;font-size: 24px;" href="https://github.com/lihaoyi/scalatags/tree/master/scalatags/shared/src/test/scala/scalatags/generic/StyleSheetTests.scala#L137-L151" target="_blank"><i class="fa fa-link "></i></a></pre> 
    <p> With Scalatags, even cascadin styles must start with a "root" class. However,inside that class, you then have two ways of defining cascades:</p> 
    <ul> 
     <li> Via nesting: you can nest Scalatags HTML tags such as <code>a</code> or <code>div</code>, or other pre-defined classes such as <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Cascade.y</code></li> 
     <li> Via the <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">~</code> or <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">&gt;</code> operators, for defining a chain with or without the <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">&gt;</code> child-selector</li>
    </ul> 
    <p> You shouldn't need cascading often, but when you do, it's useful to have around. Scalatags gives you the best of both worlds by exposing it where necessary but preventing you from using it by accident. </p> 
    <hr> 
    <p> Scalatags' CSS support is minimal and opinionated, and intentionally much less featureful than <a href="https://japgolly.github.io/scalacss/book/">Scala-CSS</a>. While the feature set will probably grow slightly to accommodate people's needs, it will always be a comparatively small, spartan API. </p>

    <div style="margin: 0px;color: #333;text-align: center;padding: 2.5em 2em 0;border-bottom: 1px solid #eee;display: block;" id="Internals" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag">
     <h1 style="margin: 0.2em 0;font-size: 3em;font-weight: 300;">Internals<a class=" scalatex-site-Styles-headerLink" href="#Internals" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a></h1>
     <br>
    </div> 
    <p> The primary data structure Scalatags uses are the <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">TypedTag</code>, representing some kind of HTML/XML tag: </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">trait TypedTag[Builder, +Output, FragT] extends Frag[Builder, FragT]{
  def tag: String

  /**
   * The modifiers that are applied to a TypedTag are kept in this linked-Seq
   * (which are actually WrappedArrays) data-structure in order for maximum
   * performance.
   */
  def modifiers: List[Seq[Modifier[Builder]]]

  /**
   * Add the given modifications (e.g. additional children, or new attributes)
   * to the [[TypedTag]].
   */
  def apply(xs: Modifier[Builder]*): Self

  /**
   * Collapses this scalatags tag tree and returns an [[Output]]
   */
  def render: Output
}

</code></pre> 
    <p> The <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Frag</code>, representing a renderable snippet, containing zero or more HTML/XML tags and string nodes: </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">trait Frag[Builder, FragT] extends Modifier[Builder]{
  def render: FragT
}
</code></pre> 
    <p> And the <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Modifier</code>, representing anything that can be applied to a <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Tag</code>: styles, classes, attributes, as well as any sort of <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Frag</code> which can be appended to it's children: </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">trait Modifier[Builder] {
  /**
   * Applies this modifier to the specified [[Builder]], such that when
   * rendering is complete the effect of adding this modifier can be seen.
   */
  def applyTo(t: Builder): Unit
}
</code></pre> 
    <p> A <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">TypedTag</code> is basically a tag-name together with a loose bag of <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Modifier</code>s, and is itself a <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Modifier</code> so it can be nested within other <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">TypedTag</code>s. A <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Modifier</code> is a tag, a sequence of tags, an attribute binding, a style binding, or anything else that can be used to modify how a tag will be rendered. Lastly, a <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Frag</code> represents the smallest standalone atom, which includes tags, loose strings, numbers, and other things.</p> 
    <p> In the text backend you have the alias </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">type Tag = TypedTag[String]</code></pre> 
    <p> Since everything renders to a string. In the DOM backend you have: </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">type HtmlTag = JsDom.TypedTag[dom.HTMLElement]
type SvgTag = JsDom.TypedTag[dom.SVGElement]
type Tag = JsDom.TypedTag[dom.Element]
</code></pre> 
    <p> These aliases help you keep your code short by letting you refer to the most common versions of <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">TypedTag</code> via concise names. </p> 
    <p> Each Scalatags backend has its own refinements, e.g. <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Text.TypedTag</code>, <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Text.Frag</code> and <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Text.Modifier</code> have the <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Builder</code> type-parameter fixed as <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">text.Builder</code>, and the <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Output</code> type-parameter fixed as <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">String</code>. Their <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">JsDom.*</code> counterparts have <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Builder</code> fixed as <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">dom.Element</code>, and <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Output</code> fixed to various subclasses of <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">dom.Element</code>. The various other classes/traits (e.g. <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Attr</code>, <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">AttrPair</code>, <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">StylePair</code>, etc.) are similarly abstract with concrete versions in each backend.</p> 
    <p> The current selection of <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">Modifier</code>s (or implicitly convertable) types include:</p> 
    <ul> 
     <li> <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">TypedTag</code>s and <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">String</code>s: appends itself to the parent's <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">children</code> list.</li> 
     <li></li> 
     <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">AttrPair</code>s: sets a key in the parent's 
     <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">attrs</code> list. 
     <li> <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">StylePair</code>s: appends the inline <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">style: value;</code> to the parent's <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">style</code> attribute.</li> 
     <li> <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">StringFrag</code>s: append the string as a child.</li> 
     <li> <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">RawFrag</code>s: append the string to the parent as unescaped HTML.</li>
    </ul> 
    <p> The bulk of Scalatag's ~5000 lines of code is static bindings (and inline documentation!) for the myriad of CSS rules and HTML tags and attributes that exist. The core of Scalatags lives in <a href="shared/main/scala/scalatags/Core.scala">Core.scala</a>, with most of the implicit extensions and conversions living in <a href="shared/main/scala/scalatags/package.scala">package.scala</a>.</p> 
    <h1 id="Architecture" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#Architecture" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>Architecture</h1> 
    <p> Scalatags has pretty odd internals in order to support code re-use between the Text and Dom packages. Essentially, each Scalatags package is an instance of </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">trait Bundle[Builder, Output &lt;: FragT, FragT]{
</code></pre> 
    <p> Which is parametrized on the <code>Builder</code> used to generate the output, the type generated by rendering a <code>Frag</code>, as well as the final <code>Output</code> type generated by rendering a <code>TypedTag</code>. The Text package is defined as </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">object Text extends Bundle[text.Builder, String, String] {
</code></pre> 
    <p> Since it uses a custom <code>text.Builder</code> object for maximum performance and spits out a <code>String</code>, while the Dom package is defined as as </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">object JsDom extends generic.Bundle[dom.Element, dom.Element, dom.Node]

</code></pre> 
    <p> Since it uses <code>dom.Element</code>s both as the intermediate builder as well as the final result. </p> 
    <p> This layout allows Scalatags to specify formally which types are common between the two backends (i.e. those in <code>generic.Bundle</code>) and which can vary. For example, both backends have a concept of <code>TypedTag</code>s, <code>Frag</code>s and <code>Modifier</code>s. On the other hand, The Text backend has <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">TypedTag[String]</code> aliased as <code>Tag</code> since it will always be <code>String</code>, while the Dom backend has it left as </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">TypedTag[+Output &lt;: dom.Element]
</code></pre> 
    <p> With helper types bound as: </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">type HtmlTag = TypedTag[dom.HTMLElement]
type SvgTag = TypedTag[dom.SVGElement]
type Tag = TypedTag[dom.Element]</code></pre> 
    <p> Since it is likely that working with the Dom backend you will want to distinguish these. </p> 
    <h1 id="Extensibility" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#Extensibility" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>Extensibility</h1> 
    <p> A mix of typeclasses and implicit conversions is used to control what you can put into a Scalatags fragment. For example, the typeclasses </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">trait AttrValue[Builder, T]{
  def apply(t: Builder, a: Attr, v: T)
}
trait StyleValue[Builder, T]{
  def apply(t: Builder, s: Style, v: T)
}</code></pre> 
    <p> Allow you to specify what (and how) types can be used as attributes and styles respectively, while implicit conversions to <code>Modifier</code> or <code>Frag</code> are used to allow you to use arbitrary types as children. The use of implicit conversions in this case is to allow it to work with variable length argument lists (i.e. <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">(mods: *Modifier)</code>), which is difficult to do with typeclasses.</p> 
    <p> Due to this design, and the parametrization of the bundles described earlier, it is possible to define behavior for a particular type only where it makes sense. e.g. there is a pair of typeclass instances </p> 
    <pre><code class="scala scalatex-site-Styles-highlightMe">implicit object bindJsAny extends generic.AttrValue[dom.Element, js.Any]
implicit def bindJsAnyLike[T &lt;% js.Any] = new generic.AttrValue[dom.Element, T]</code></pre> 
    <p> Which allows you to bind anything convertible to a <code>js.Any</code> into the JsDom fragments, since they can just be assigned directly to the attributes of the <code>dom.Element</code> objects. Doing the same thing for Text fragments doesn't make sense, and would correctly fail to compile.</p> 
    <p> You can easier add other typeclass instances to handle binding e.g. <code>Future</code>s (which will add a child or set an attr/style on completion), or reactive variables (which would constantly update the child/attr/style every time it changes). </p> 
    <div style="margin: 0px;color: #333;text-align: center;padding: 2.5em 2em 0;border-bottom: 1px solid #eee;display: block;" id="PriorWork" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag">
     <h1 style="margin: 0.2em 0;font-size: 3em;font-weight: 300;">Prior Work<a class=" scalatex-site-Styles-headerLink" href="#PriorWork" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a></h1>
     <br>
    </div> 
    <p> Scalatags was made after experience with a broad range of HTML generation systems. This experience (with both the pros and cons of existing systems) informed the design of scalatags.Text. </p> 
    <h1 id="Old-schoolTemplates" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#Old-schoolTemplates" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>Old-school Templates</h1> 
    <p> <a href="http://jinja.pocoo.org/docs/">Jinja2</a> is the templating engine that comes bundled with <a href="http://flask.pocoo.org/">Flask</a>, a similar (but somewhat weaker) system comes bundled with <a href="https://docs.djangoproject.com/en/dev/topics/templates/">Django</a>, and another system in a similar vein is <a href="http://guides.rubyonrails.org/layouts_and_rendering.html">Ruby on Rail's ERB</a> rendering engine. This spread more-or-less represents the old-school way of rendering HTML, in that they:</p> 
    <ul> 
     <li> Are effectively string-based</li> 
     <li> Use special syntax for both interpolating variables as well as for basic control flow logic</li> 
     <li> Have a ruby/python-like (but not quite!) language for logic within the template</li> 
     <li> Are generally with one template per file.</li>
    </ul> 
    <p> They also showcase many of the weaknesses of this style of templating system:</p> 
    <ul> 
     <li> The fact that it's string based means it's vulnerable to XSS injections, or plain-old malformed HTML output. </li> 
     <li> The one-template-per-file rule discourages you from building your page from small re-usable fragments, because who wants to keep track of hundreds of individual files. People are reluctant to make a file with 3-5 lines in it, which is understandable but unfortunate because factoring templates into re-usable 3-5 line snippets is a good way of staying sane.</li> 
     <li> The API is complex and novel: Jinja2 for example contains logic around <a href="http://jinja.pocoo.org/docs/api/#loaders">file-loading</a> &amp; <a href="http://jinja.pocoo.org/docs/api/#bytecode-cache">caching</a>, as well as custom Jinja2-specific ways of doing <a href="http://jinja.pocoo.org/docs/templates/#for">loops</a>, <a href="http://jinja.pocoo.org/docs/templates/#if">conditionals</a>, <a href="http://jinja.pocoo.org/docs/templates/#macros">functions</a>, <a href="http://jinja.pocoo.org/docs/templates/#comments">comments</a>, <a href="http://jinja.pocoo.org/docs/templates/#template-inheritance">inheritance</a>, <a href="http://jinja.pocoo.org/docs/templates/#block-nesting-and-scope">scoping</a>, <a href="http://jinja.pocoo.org/docs/templates/#import-context-behavior">imports</a> and other things. All these are things you would have to learn.</li> 
     <li> The syntax is completely new; finding an editor that properly supports all the quirks and semantics (or even simply highlighting things properly) is hard. You could hack together something quick and extremely fragile, or you could wait ages for a solid plugin to materialize.</li> 
     <li> Abstraction is clunky: inbuilt tags are used via <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">&lt;div /&gt;</code>, while user-defined components are called e.g. by <code>{{ component() }}</code>. You're left with a choice between not using much abstraction and mainly sticking to inbuilt tags, or creating components and accepting the fact that your templates will basically be totally composed of <code>{{ curly braces }}</code>. Neither choice is satisfying.</li> 
     <li> Have no sort of static checking at all; you're just passing dicts around, and waiting for silly mistakes to blow up at run-time so you can fix them. </li>
    </ul> 
    <h1 id="RazorandPlayTemplates" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#RazorandPlayTemplates" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>Razor and Play Templates</h1> 
    <p> <a href="https://github.com/Antaris/RazorEngine">Razor</a>, the ASP.NET MVC template engine, and the Play framework's <a href="http://www.playframework.com/documentation/2.3-SNAPSHOT/ScalaTemplates">template engine</a> go in a new direction. Their templates generally:</p> 
    <ul> 
     <li> Are statically-compiled</li> 
     <li> Re-use large chunks of the host language</li>
    </ul> 
    <p> Both templating systems generally use <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">@</code> to delimit "code"; e.g. <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">@for(...){...</code>} declares a for-loop. Nice things are:</p> 
    <ul> 
     <li> The API is far simpler: all the custom control-flow/logic/syntax basically collapses into the simple statement "do it the way C#/Scala does it".</li> 
     <li> Static checking in templates is nice.</li>
    </ul> 
    <p> However, they still have their downsides:</p> 
    <ul> 
     <li> Abstractions are less clunky to use than in old-school templates, e.g. <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">@component()</code> rather than <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">{{ component() }}</code>, but still not ideal.</li> 
     <li> The one-template-per-file rule is still there, making abstractions clunky to define.</li> 
     <li> They still require their own special build-system-integration, run-time compilation, caching, and other things, which a templating language really shouldn't need. <a href="http://scalate.fusesource.org/index.html">Scalate</a>, which has a runtime dependency on the entire Scala compiler, suffers from the same problem.</li> 
     <li> The syntax still poses a problem for editors; both HTML editors and C#/Scala editors won't want to work with these templates, so you still end up with sub-par support or waiting for plugins.</li> 
     <li> You still end up with weird <a href="http://stackoverflow.com/questions/13973009/complex-pattern-matching-on-templace-using-scala-play">edge</a> <a href="http://stackoverflow.com/questions/12070625/compilation-error-of-play-framework-templates">cases</a> due to the fact that you're squashing together two completely unrelated syntaxes.</li>
    </ul> 
    <h1 id="XHPandPyxl" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#XHPandPyxl" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>XHP and Pyxl</h1> 
    <p> <a href="https://github.com/facebook/xhp">XHP</a> and <a href="https://github.com/dropbox/pyxl">Pyxl</a> are HTML generation systems used at Facebook and Dropbox. In short, they allow you to:</p> 
    <ul> 
     <li> Embed sections of your HTML as literals within your PHP and Python code</li> 
     <li> Reference them as objects (e.g. calling methods and modifying fields)</li> 
     <li> Provide a way to interpolate values and combine them.</li>
    </ul> 
    <p> The Pyxl homepage provides this example: </p> 
    <pre><code class="python scalatex-site-Styles-highlightMe">image_name = "bolton.png"
image = &lt;img src="/static/images/{image_name}" /&gt;

text = "Michael Bolton"
block = &lt;div&gt;{image}{text}&lt;/div&gt;

element_list = [image, text]
block2 = &lt;div&gt;{element_list}&lt;/div&gt;
</code></pre> 
    <p> Which shows how you can generate HTML directly in your python code, using your python variables. These libraries are basically the same thing, and have some nice properties:</p> 
    <ul> 
     <li> The one-template-per-file rule is gone! This encourages you to make more, smaller fragments and then compose them, which is great.</li> 
     <li> They're no longer string-based, so you won't have problems with XSS or malformed output.</li> 
     <li> The API is really simple; maybe a dozen different things you need to remember, and the rest is "just use Python/PHP". E.g. they don't need to load templates from the filesystem any more (with all the associated discovery/loading/caching logic) since it's all just in your code.</li> 
     <li> The syntax is completely familiar too; apart from maybe one new rule (using <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">{...}</code> to interpolate values) the rest of your templates and logic are bog-standard HTML/Python/PHP.</li> 
     <li> Abstraction is almost seamless; both systems allow you to define custom components and have them called via <code class="xml scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">&lt;component arg="..." /&gt;</code></li>
    </ul> 
    <p> But they're not quite there:</p> 
    <ul> 
     <li> Although the syntax is familiar to you, it's not familiar to your editor, and probably will mess up your syntax highlighting (and other tooling) in your Python/PHP files.</li> 
     <li> Defining custom components (e.g. in <a href="https://github.com/dropbox/pyxl#ui-modules">Pyxl</a>) is much more verbose/tedious than it needs to be. Most of the time, all you want is a function; very rarely do you want a fragment that is long lived and has mutable state, which is where classes/objects are really necessary.</li> 
     <li> Even using custom components gets tedious; at some scale, everything you pass into a component will be a structured Python value rather than a string, and you end up with code like <code class="xml scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">&lt;component arg1="{value1}" arg2="{value2}" arg3="{value3}" arg4="{value4}" /&gt;</code>. It's nice to have inbuilt/custom components behave uniformly, but you wonder what the XML syntax is really buying you other than forcing you to only use keyword-arguments and to wrap arguments in <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;"> "{...}" </code>.</li> 
     <li> It's still XML! People spend <a href="https://github.com/visionmedia/jade">a lot</a> <a href="http://haml.info/">of time</a> trying to get XML out of their templates; it seems odd to spend just as much time trying to put it into your source code. </li>
    </ul> 
    <h1 id="Scalatags" style="display: block;" class=" scalatex-site-Styles-hoverContainer scalatex-site-Styles-headerTag"><a class=" scalatex-site-Styles-headerLink" href="#Scalatags" style="position: absolute;right: 0px;"><i class="fa fa-link"></i></a>Scalatags</h1> 
    <p> And that's why I created Scalatags:</p> 
    <ul> 
     <li> Structured and immune to malformed output/XSS.</li> 
     <li> No more one-template-per-file rule; make small templates to your heart's content! </li> 
     <li> Dead-simple API, which re-uses 100% of what you know about Scala's language and libraries.</li> 
     <li> 100% Scala syntax; no more bumping into weird edge cases with the parser.</li> 
     <li> Syntax highlighting, error-highlighting, jump-to-definition, in-editor-documentation, and all the other nice IDE features out of the box. No more waiting for plugins!</li> 
     <li> Inbuilt and custom tags are uniformly just function calls (e.g. <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">div("hello world")</code>)</li> 
     <li> Custom components are trivial to create (<code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">def component(x: Int, y: Int) = ...</code>) and trivial to use (<code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">component(value1, value2)</code>) because they're just functions.</li> 
     <li> No dependencies! No need for SBT integration, filesystem loading, runtime compilation, caching, and all that jazz. You create a Scala object, call <code>.toString()</code>, and that's it!</li>
    </ul> 
    <p> On top of fixing all the old problems, Scalatags targets some new ones: </p> 
    <ul> 
     <li> Typesafe-ish access to HTML tags, attributes and CSS classes and styles. No more weird bugs due to typos like <code class="scala scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">flaot: left</code> or <code class="xml scalatex-site-Styles-highlightMe" style="display: inline;padding: 0px;margin: 0px;">&lt;dvi&gt;</code>.</li> 
     <li> Cross compiles to run on both JVM and Javascript via <a href="https://github.com/scala-js/scala-js">ScalaJS</a>, which is a property few other engines (e.g. <a href="http://mustache.github.io/">Mustache</a>) have.</li> 
     <li> <a href="#performance">Blazing fast</a></li>
    </ul> 
    <p> Scalatags is still a work in progress, but I think I've hit most of the pain points I was feeling with the old systems, and hope to continually improve it over time. Pull requests are welcome! </p>
