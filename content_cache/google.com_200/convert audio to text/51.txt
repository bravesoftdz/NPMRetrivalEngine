      <h1 class="entry-title">Getting Audio File Information With HTML’s File API And Audio Element</h1> 
      <div class="entry-meta"> 
       <span class="meta-prep meta-prep-author">Posted by </span> 
       <span class="author vcard"><a class="url fn n" href="https://lostechies.com/derickbailey/author/derickbailey/" title="View all posts by Derick Bailey">Derick Bailey</a></span> 
       <span class="meta-sep"> on </span> 
       <a href="https://lostechies.com/derickbailey/2013/09/23/getting-audio-file-information-with-htmls-file-api-and-audio-element/" title="8:11 am" rel="bookmark"><span class="entry-date">September 23, 2013</span></a> 
      </div>
      <!-- .entry-meta --> 
      <div class="entry-content"> 
       <p>I’ve been cranking out features and capabilities on <a href="http://signalleaf.com"><strong>SignalLeaf</strong></a> for just over a month now, and it’s ever so close to being ready for some public beta testing. I do have one alpha tester, though. <a href="http://simpleprogrammer.com/">John Sonmez</a> is slowly moving some of his “<a href="http://getupandcode.com/">Get Up And Code</a>” podcast over to SignalLeaf, and is providing feedback to me along the way. One of the features that he asked for, after uploading his first file, was automatic file information gathering from the .mp3 file. It turns out information such as the file’s content type, size and duration can all be obtained quite easily with modern browsers – no server side code needed. This was a huge relief for me as I’m hosting SignalLeaf on Heroku and did not want to stand up a real server somewhere, to store files and process the audio file.</p> 
       <h2>Basic File Info With The File API</h2> 
       <p>Most of the file information that I need can be gathered with the <a href="http://www.w3.org/TR/FileAPI/">HTML File API</a>. File name, size and content type, for example, can be pulled from this easily. The only thing you need is a &lt;file&gt; input element and a little bit of JavaScript (jQuery in this case).</p> 
       <p><script src="https://gist.github.com/6670383.js?file=fileInfo.html"></script>
        <noscript> 
        </noscript></p>
       <pre><code class="language-html html">&lt;input type="file" id="file" /&gt;

&lt;p&gt;
  &lt;label&gt;File Name:&lt;/label&gt;
  &lt;span id="filename"&gt;&lt;/span&gt;
&lt;/p&gt;

&lt;p&gt;
  &lt;label&gt;File Type:&lt;/label&gt;
  &lt;span id="filetype"&gt;&lt;/span&gt;
&lt;/p&gt;

&lt;p&gt;
  &lt;label&gt;File Size:&lt;/label&gt;
  &lt;span id="filesize"&gt;&lt;/span&gt;
&lt;/p&gt;</code></pre> 
       <p></p> 
       <p><script src="https://gist.github.com/6670383.js?file=fileInfo.js"></script>
        <noscript> 
        </noscript></p>
       <pre><code class="language-javascript javascript">$("#file").change(function(e){
    var file = e.currentTarget.files[0];
   
    $("#filename").text(file.name);
    $("#filetype").text(file.type);
    $("#filesize").text(file.size);    
});</code></pre> 
       <p></p> 
       <p>When the “change” event on the file input fires, you can grab the file information using the .files attribute of the input element. In this case, I’m using jQuery to capture the “change” event, but I’m grabbing the e.currentTarget directly after that. This gives me the HTML file input element, which has the .files attribute on it.&nbsp;This attribute is an array of files, for scenarios where you are selecting multiple files. In this case, there is only a single file selected so I’m grabbing the first item in the array.</p> 
       <p>Once I have the file object, I can get the .name, .size and .type information and populate the HTML in my document, store these in &lt;input&gt; fields, or do whatever else I need to do with them. But while this information is great, it isn’t enough. I need to know the song duration as well, so that podcasters uploading an mp3 file to SignalLeaf won’t have to manually enter the song duration.</p> 
       <h2>Audio Element And Duration</h2> 
       <p>After some tweeting and asking if I there was a JavaScript library to get song duration in the browser, I found out that the HTML &lt;audio&gt; element has this built in to it. Once the &lt;audio&gt; element has loaded the file set in it’s “src” attribute, I can read the .duration of the element which returns the song duration in seconds.</p> 
       <p><script src="https://gist.github.com/6670383.js?file=audioElement.html"></script>
        <noscript> 
        </noscript></p>
       <pre><code class="language-html html">&lt;audio id="my-audio" src="http://example.com/somefile.mp3"&gt;&lt;/audio&gt;

&lt;script&gt;
  var myAudio = document.getElmentById("my-audio");
  myAudio.duration; // =&gt; duration, in seconds
&lt;/script&gt;</code></pre> 
       <p></p> 
       <p>Once I had that in place, I set up a “<a href="https://developer.mozilla.org/en-US/docs/Web/Reference/Events/canplaythrough">canplaythrough</a>” event listener to tell me when the song had been loaded. This event fires when the song is loaded and can play all the way through. From there, I read the duration and then using momentjs, I convert the duration from seconds in to a more useful “hh:mm:ss” format.</p> 
       <p><script src="https://gist.github.com/6670383.js?file=audio-duration.js"></script>
        <noscript> 
        </noscript></p>
       <pre><code class="language-javascript javascript">$("#my-audio").on("canplaythrough", function(e){
    var seconds = e.currentTarget.duration;
    var duration = moment.duration(seconds, "seconds");
    
    var time = "";
    var hours = duration.hours();
    if (hours &gt; 0) { time = hours + ":" ; }
    
    time = time + duration.minutes() + ":" + duration.seconds();
    $("#duration").text(time);
});
</code></pre> 
       <p></p> 
       <p>This worked well for files hosted somewhere on the web, but I needed to load the file from the local machine of the person using SignalLeaf, before it was uploaded anywhere. It turns out there are a couple of options for this.</p> 
       <h2>First Attempt: FileReader And Data URL</h2> 
       <p>The first thing I tried to do was read the file contents in to memory and create a Data URI with the FileReader API. I found <a href="http://www.html5rocks.com/en/tutorials/file/dndfiles/">this article on HTML5 Rocks</a>, and it gave me all the information I needed for this attempt. So I set up a FileReader and built a Data URL – a base64 encoded version of a binary file.</p> 
       <p><script src="https://gist.github.com/6670383.js?file=fileReader.js"></script>
        <noscript> 
        </noscript></p>
       <pre><code class="language-javascript javascript">$("#my-audio").on("canplaythrough", function(e){
    var seconds = e.currentTarget.duration;
    var duration = moment.duration(seconds, "seconds");
    
    var time = "";
    var hours = duration.hours();
    if (hours &gt; 0) { time = hours + ":" ; }
    
    time = time + duration.minutes() + ":" + duration.seconds();
    $("#duration").text(time);
});

$("#file").change(function(e){
  var file = e.currentTarget.files[0];

  var reader = new FileReader();
  
  reader.onLoad = function(encodedFile){
    $("#my-audio").prop(src, encodedFile);
  };
  
  reader.readAsDataURL(file);
});</code></pre> 
       <p></p> 
       <p>My first test was successful! I was able to get the file information that I wanted. But when I tried to use this on files that were more than a few seconds long, I noticed the browser was locking up. The larger the file, the longer it locked up.</p> 
       <p>It turns out this is a really bad idea for large audio files. Even with an audio file that is only 6 minutes long, my browser locked up for 3 or 4 seconds. Now imagine a podcast episode that is 30 or 40 minutes long. It would likely lock up the browser for 20 or 30 seconds, or even crash the browser. The problem is that using the Data URL encoded file gives you a base64 encoded version of the file, which is then stuffed in to the &lt;audio&gt; tag’s “src” property. You can imagine a browser not liking the 30 or 40 megs worth of data, and having a hard time encoding it and storing the string in this element.</p> 
       <h2>Fixing It With Object URLs</h2> 
       <p>Shortly after running in to this problem and complaining about it on twitter, <a href="https://twitter.com/chris_a_wagner/status/381470768648839168">Chris Wagner suggested I use URL.createObjectURL</a> instead. I’ve used URL.createObjectURL in the past, but had forgotten about it. The last time I used it was when I was helping to build <a href="http://hilojs.codeplex.com/">the Hilo.js sample app</a> for Microsoft Patterns &amp; Practices.</p> 
       <p>The gist of <a href="https://developer.mozilla.org/en-US/docs/Web/API/URL.createObjectURL">the URL.createObjectUrl function</a>, is that it returns a URL that points to a memory location in the browser, for an object. This object URL can be used in most places where a URL is supported. I used this to load large image files into memory for that project, and it makes sense to use it for a large audio file as well.</p> 
       <p><script src="https://gist.github.com/6670383.js?file=createObjectURL.js"></script>
        <noscript> 
        </noscript></p>
       <pre><code class="language-javascript javascript">$("#file").change(function(e){
    var file = e.currentTarget.files[0];
   
    var objectUrl = URL.createObjectURL(file);
    $("#audio").prop("src", objectUrl);
});</code></pre> 
       <p></p> 
       <p>Now when I select a file, the audio information is parsed and displayed nearly instantly. It doesn’t seem to matter whether I load a 5 meg or 50 meg audio file, either. The browser is pulling the file in to memory, and the &lt;audio&gt; element is pointed at that memory location.</p> 
       <p><strong>One important note:</strong> if you’re building a single page application and using createObjectURL, you will also need to know about <a href="https://developer.mozilla.org/en-US/docs/Web/API/URL.revokeObjectURL">revokeObjectURL</a>. I spent 2 weeks profiling the Hilo.js app memory usage, to figure out that we were leaking memory everywhere with our use of createObjectURL. Revoking the URL will release the memory, allowing your single page app to clean itself up.</p> 
       <h2>A Complete Demo</h2> 
       <p>WIth all that said and done, It’s fairly easy to get the complete set of audio information that I need from a .mp3 file. Here’s a complete demo of the code, which can be found at <a href="http://jsfiddle.net/derickbailey/s4P2v/">this JSFiddle</a>.</p> 
       <p><iframe width="100%" height="300" src="http://jsfiddle.net/derickbailey/s4P2v/embedded/result,js,html" allowfullscreen frameborder="0"></iframe></p> 
       <p>Find a .mp3 file, or other audio file that the &lt;audio&gt; input element supports. Once you have selected it with the file chooser, you will see the file name, type, size and duration loaded in to the HTML – all thanks to the File API, &lt;audio&gt; element and URL.createObjectURL.</p> 
       <p>By the way, if you want to know more about SignalLeaf and how it is simplifying podcast audio hosting, be sure to sign up for the mailing list at the bottom of <a href="http://signalleaf.com">SignalLeaf.com</a>.</p>
