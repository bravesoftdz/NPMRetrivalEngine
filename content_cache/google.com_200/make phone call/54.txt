         <div class="w-blog-post-body">
          <h1 class="w-blog-post-title entry-title" itemprop="headline">A guide to npm: The node.js package manager</h1>
          <div class="w-blog-post-meta"> 
           <time class="w-blog-post-meta-date date updated" itemprop="datePublished" datetime="2017-04-13 10:00:58">April 13, 2017</time>
           <span class="w-blog-post-meta-author vcard author"><a href="https://go.tinymce.com/blog/author/martingouws/" class="fn">Martin Gouws</a></span>
           <span class="w-blog-post-meta-category"><a href="https://go.tinymce.com/blog/category/guest/" rel="category tag">Guest</a></span>
          </div>
         </div>
        </div>
       </div>
      </article>
      <section class="l-section">
       <div class="l-section-h i-cf" itemprop="text">
        <p>JavaScript is easily the most used language when it comes to the development of websites and web applications. The multitude of resources are astounding, and even more so, the number of libraries available.</p>
        <p>At first, these libraries are few and easy to maintain; however, soon enough dependency hell sets in and a more mature solution is required.</p>
        <p><img class="aligncenter size-full wp-image-28383" src="https://www.ephox.com/wp-content/uploads/2017/03/toptal-blog-image-1485880745952-0d808da7348344ead96210bb0b7e9e51.png" alt="npm package manager" width="640" height="640"></p>
        <p>Enter the Node Package Manager (npm) – a JavaScript package manager most notably used in conjunction with Node.js, although it can be used independently as well. It gives you exceptional control over your project’s dependencies and provides a great way to contribute to the open-source world.</p>
        <p>You can get started by simply running <code class="prettyprint">npm install</code>&nbsp;and injecting it into your JavaScript file.</p>
        <p>Want to install a specific version? No problem. Run <code class="prettyprint">npm install@1.2.3</code>.</p>
        <p>Want to install a package globally (like Mocha, or Angular-CLI)? Just add a <code class="prettyprint">-g</code> like so:<code class="prettyprint">npm install -g angular-cli mocha</code>.</p>
        <p>Admittedly, most use cases stop at an npm install, and there isn’t a need for anything else. However, npm has loads of additional features, which I’ll walk you through, highlighting those I consider essential, really useful, or just plain awesome.</p>
        <h2>CLI commands</h2>
        <p>The CLI is where users spend most of their time interacting with npm, and its help interface is actually helpful.</p>
        <p>Querying help (<code class="prettyprint">npm help</code>) spews out an entire array of options, and running <code class="prettyprint">npm help-search</code>&nbsp;gives you a list of search results direct from the npm markdown.</p>
        <p>Here are the core commands that stand out.</p>
        <ul>
         <li><code class="prettyprint">Install</code>: It’s mentioned here because of its sheer necessity when working with npm. Used to either install a new package locally or globally (when adding <code class="prettyprint">-g</code> ) or to install dependencies listed in the <code class="prettyprint">package.json</code> file (more on that later).</li>
         <li><code class="prettyprint">uninstall</code> : This is also an essential. It’s used to purge a specific package from the &nbsp;<code class="prettyprint">node_modules</code> directory either locally or globally (when adding <code class="prettyprint">-g</code>).</li>
         <li><code class="prettyprint">access</code>: This is the playground of npm user permission administrators within the context npm-organizations and scoped (private) packages. Seriously powerful stuff. Used in conjunction with &nbsp;<code class="prettyprint">adduser</code>,&nbsp; <code class="prettyprint">owner</code>, <code class="prettyprint">team</code>, etc., it gives fine grained control over who has access to what.<code class="prettyprint">bin</code>: Where on earth are packages installed? Run this command to see the absolute file path.</li>
         <li><code class="prettyprint">cache</code>: If you start installing packages from npm left, right, and center, this command is quite useful. Either call it with the &nbsp;<code class="prettyprint">ls</code> subcommand to see a list of locally cached packages or with the &nbsp;<code class="prettyprint">clean</code> subcommand to clear all packages that are in the cache. Back when the npm registry was still a bit unstable, this was essential to get back to a stable environment or to reset things when you didn’t properly set up npm permissions.</li>
         <li><code class="prettyprint">config</code>: We will get into the different configuration options later, but this command deals primarily with persisting configuration properties in the local or global configuration file by using the &nbsp;<code class="prettyprint">set</code>, &nbsp;<code class="prettyprint">get</code> or &nbsp;<code class="prettyprint">delete</code> subcommands.</li>
         <li><code class="prettyprint">dedupe</code>&nbsp;or&nbsp; <code class="prettyprint">ddp</code>: When working on a project over an extensive period of time and installing packages straight from npm, this command will walk the local package tree and attempt to simplify dependencies.</li>
         <li><code class="prettyprint">link</code> : When you are developing your own npm package, this allows you to create a symlink to the global context so it can be tested as if it was installed globally from the npm registry. For example, if you are writing an assembly tool in node that has a CLI installed globally, you can run this command and test your CLI’s behavior without needing to deploy it first.</li>
         <li><code class="prettyprint">ls</code> : It’s used to visualize package dependencies and their dependencies, in a tree structure. It’s cool to see and is also useful for comparisons with other projects.</li>
         <li><code class="prettyprint">outdated</code> : This is used to evaluate the current state of installed dependencies and whether or not they are outdated. In projects where the root dependency list is hundreds of lines long, a manual check on packages is close to impossible. Adding <code class="prettyprint">-g --depth=0</code> to this command allows you to also check your globally installed packages.</li>
         <li><code class="prettyprint">publish</code> : This command is essential when developing your own package for npm. It does exactly as the name suggests; it publishes your package to the npm registry.</li>
         <li><code class="prettyprint">search</code>: Use this to search the registry for all packages containing the text provided in the third argument.</li>
         <li><code class="prettyprint">shrinkwrap</code>: In short, this commands allows you to lock down specific dependency versions in a package in order, so that a relaxed semver&nbsp;(<a href="https://docs.npmjs.com/getting-started/semantic-versioning">semantic versioning</a>) number doesn’t break production code.</li>
         <li><code class="prettyprint">star</code> : Do you really like a package you are using? Use this command to show your appreciation directly from the terminal, which then reflects on the package’s page on the npm registry.</li>
         <li><code class="prettyprint">update</code>: This usually follows the <code class="prettyprint">outdated</code> command to update any outdated packages.</li>
         <li><code class="prettyprint">version</code>: This gives you a shorthand to bump the&nbsp; <code class="prettyprint">package.json</code> version property, and do a git tag all in one.</li>
        </ul>
        <p>Note that most of these commands can take subcommands and/or configurations, and this list is by no means an end-all discussion on the CLI.</p>
        <h2>npm-config</h2>
        <p>Configuration is a major part of npm, and there are multiple ways configuration variables can be set.</p>
        <h3>Configuration via CLI and Environmental Variables</h3>
        <p>First, configuration can be set via the CLI from the terminal.</p>
        <p>Typically it would look something like this: <code class="prettyprint">npm &lt;command&gt; --&lt;configuration option&gt; [&lt;optional value&gt;].</code></p>
        <p>If the value is not specified, the option will be set to true by default.</p>
        <p>For example, let’s say that you are working on a scoped (private) npm package, and you decide to publish it as a public package.</p>
        <p>This is easily done by appending <code class="prettyprint">--access=public</code> to your <code class="prettyprint">publish</code> command. If we did not specify the property to be public, the default would be restricted (private).</p>
        <p>Configuration appended to other commands like this does not persist everywhere, so it can become tiresome to set an array of configurations via the CLI.</p>
        <p>In those cases, it may be better to set configuration using environmental variables.</p>
        <p>Any environmental variable set with the <code class="prettyprint">npm_config_</code> prefix will be used to configure npm.</p>
        <p>For example: <code class="prettyprint">export npm_config_registry=localhost:4321</code> would set the registry configuration option globally, and when npm is executed it will use the npm registry located at localhost on port 4321.</p>
        <h3>Configuration via the npmrc file</h3>
        <p>You can also set configuration options using the special <code class="prettyprint">.npmrc file</code>, which can be set at different levels depending on your requirements:</p>
        <ul>
         <li style="list-style-type: none;">
          <ul>
           <li>Project level: In the root of a project’s code along with its <code class="prettyprint">package.json</code> file, typically <code class="prettyprint">path/to/project/.npmrc</code></li>
           <li>User level: The directory that configures a specific user’s account, typically <code class="prettyprint">~/.npmrc</code>.</li>
           <li>Global level: The directory where npm looks for global configurations, typically <code class="prettyprint">$PREFIX/etc/npmrc</code></li>
           <li>Built-in level: Be cautious. This configuration is not only global but it is also part of the npm source code, and best practice recommends (actually demands) that we do not change code we are not responsible for maintaining. It can typically be found at <code class="prettyprint">/path/to/npm/npmrc</code>.</li>
          </ul></li>
        </ul>
        <p>Configuration settings in the <code class="prettyprint">.npmrc</code> file can be modified and persisted using the CLI by running a command in this format: <code class="prettyprint">npm config set &lt;key&gt; &lt;value&gt;</code> .</p>
        <p>For example, you can run <code class="prettyprint">npm config set access public</code> to make the publishing configuration of a scoped (private) package persistently public.</p>
        <p>By default, this command would persist the configuration locally (the user level configuration as described above), but you can add <code class="prettyprint">-g</code> to persist it globally.</p>
        <p>When persisted configuration needs to happen on the project level, or on the built-in level, the <code class="prettyprint">.npmrc</code> file must be modified using a text editor.</p>
        <h3>Configuration via package.json</h3>
        <p>Finally, configuration can be set from the <code class="prettyprint">package.json</code> file. However, this is seldom used (and should only be used if explicitly required), because a project level <code class="prettyprint">.npmrc</code> file is the conventionally preferred place to set package configuration.</p>
        <h3>Notable configuration settings</h3>
        <ul>
         <li><code class="prettyprint">access</code>: As discussed above, it’s used to set permissions.</li>
         <li><code class="prettyprint">always-auth</code>: It’s important to note that this setting defaults to false. When it is set to true, npm will always require authentication when contacting the registry.</li>
         <li><code class="prettyprint">ca</code>: Defaults to the npm certificate authority (CA). It can be changed to null to allow access to only known registrars, or to a specific CA certificate to only grant access to that specific one. This setting, along with <code class="prettyprint">cafile</code>, <code class="prettyprint">cert</code> and <code class="prettyprint">strict-ssl</code>, are seldom used, but speak toward the security and reliability aspect of npm, allowing peace of mind in knowing that the package you are installing is coming from the source you expect.</li>
         <li><code class="prettyprint">color</code>: This defaults to true, giving you a break from the standard bleakness of the terminal by coloring the <code class="prettyprint">stdout</code> that is allowed by tty file descriptors. If set to false the terminal remains dull. When it is set to <code class="prettyprint">always</code>, it always outputs in color.</li>
         <li><code class="prettyprint">depth</code>: This setting allows for granular control over what you see with recursive commands, such as <code class="prettyprint">ls</code> and <code class="prettyprint">outdated</code>, by assigning how deeply they’re executed. A value of 0 will only evaluate the first level of dependencies whereas infinity (the default) will cause all levels of dependencies to be evaluated. The exception to this rule is when using it with <code class="prettyprint">outdated</code>; in that case, infinity is interpreted as 0 to ensure more relevant output.</li>
         <li><code class="prettyprint">dev</code>: This is set to false by default, but when it is set to true (when doing an npm install) all development dependencies in the package.json file will be installed along with the normal dependencies.</li>
         <li><code class="prettyprint">dry-run</code>: When this setting is set to true, npm will not make any changes to your package but will instead tell you what it would have done. This can be very useful when running certain commands, such as dedupe or update.</li>
         <li><code class="prettyprint">git-tag-version</code>: It is set to true by default. This setting tags a version in git when running the npm version command. If you are using npm as the package manager for a large project that has tagged versions in git, it can save you time, and remember to update the package.json file for you.</li>
         <li><code class="prettyprint">loglevel</code>: By default, this is set to warn, which gives an error and warning output when running npm commands. Other settings include <code class="prettyprint">silent</code>, which provides no output; <code class="prettyprint">error</code>, which only logs errors to the output; <code class="prettyprint">http</code>, which only announces http request errors; <code class="prettyprint">info</code>, for want informative output); <code class="prettyprint">verbose</code>, which logs almost everything; and <code class="prettyprint">silly</code>, which, like the name suggests, gives a silly amount of output and then some.</li>
         <li><code class="prettyprint">production</code>: When this is set to true, npm acts accordingly and runs all commands in production mode. This means that development or optional dependencies will not be installed, nor will any development related tasks be executed.</li>
         <li><code class="prettyprint">rollback</code>: When set to true, all failed installs are removed. This comes in handy when an installation of dependencies fail. Depending on your level of logging, you should be able to see what installations failed, make a note of these, and run the npm install command with the rollback option set to true. Then with your notes and a dry-run installation (as described above), you can then debug the problem.</li>
         <li><code class="prettyprint">save</code>: When installing a package directly from the registry, you can append <code class="prettyprint">–save</code> to the command which will add the installed package to the dependencies option in the package.json file. For example, npm install lodash` will add lodash to your dependencies.</li>
         <li><code class="prettyprint">save-dev</code>: Similar to the save configuration option, add <code class="prettyprint">--save-dev</code> when installing a package, and it will then be added to the <code class="prettyprint">devDependencies</code> option in the <code class="prettyprint">package.json</code> file.</li>
         <li><code class="prettyprint">save-optional</code>: Similar to the save configuration option, add <code class="prettyprint">--save-optional</code> when installing a package, and it will then be added to the <code class="prettyprint">optionalDependencies</code> option in the package.json file.</li>
         <li><code class="prettyprint">save-exact</code>: When installing packages, the <code class="prettyprint">save</code>, <code class="prettyprint">save-dev</code> and <code class="prettyprint">save-optional</code> options modify the <code class="prettyprint">package.json</code> file by inserting the installed package into its respective property with a semver range operator. When invoking the ‘save-exact’ configuration setting with a value of true, in conjunction with one of the mentioned settings above, a specific version number is used, ignoring the semver range.</li>
         <li><code class="prettyprint">save-prefix</code> This sets the semver range operator when using <code class="prettyprint">save</code>, <code class="prettyprint">save-dev</code> or <code class="prettyprint">save-optional</code>. The default is <code class="prettyprint">^</code>, allowing for minor upgrades on packages to happen on install. This can be set to any valid prefixed semver range operator.</li>
         <li><code class="prettyprint">tag-version-prefix</code>: The conventional default is <code class="prettyprint">v</code>, specifying what is prepended to the git tag version when running <code class="prettyprint">npm version</code>.</li>
        </ul>
        <h2>Updating npm,&nbsp;using npm</h2>
        <p>You can also use npm to update itself.</p>
        <p>Simply run <code class="prettyprint">npm install -g npm@latest</code>, and npm will be updated to the latest stable release. It is important to note that each version of <a href="https://www.toptal.com/nodejs">Node.js</a> ships with a specific version of npm, and in my experience, you shouldn’t mess with that pairing too much.</p>
        <p>At the end of the day, my recommendation is to stick with the pairing as they are intended.</p>
        <p>When using npm as a standalone tool, make sure you understand the implications of using whatever version you choose. There is a great tool for managing different Node.js versions (and in turn npm versions) on the same system called <a href="https://github.com/creationix/nvm">nvm</a>.</p>
        <h3>The package.json File</h3>
        <p>The <code class="prettyprint">package.json</code> file is the crucial element that links everything together.</p>
        <p>It’s a requirement for publishing a package to the npm registry, and it’s where the management part of dependencies comes to life.</p>
        <p>It has two required fields, namely “name” and “version,” and together these properties should be a unique identifier.</p>
        <p>The name field should adhere to certain rules, as defined by the <a href="https://docs.npmjs.com/files/package.json">npm documentation on naming</a>, and the version field is subject to the <a href="https://docs.npmjs.com/getting-started/semantic-versioning">semver specificiations</a>.</p>
        <p><img class="aligncenter size-full wp-image-44371" src="https://www.ephox.com/wp-content/uploads/2017/04/toptal-blog-image-1485880802196-510038b388c3c19c55001d613004ed65.png" alt="screenshot of npm reading the package.json file for dependency management" width="640" height="640"></p>
        <p>Over and above that, you can have a list of dependencies a mile long, and define a specific version to be used for each of them, using the semver versions and range operators. Here’s a list of other notable properties.</p>
        <h4>“main”</h4>
        <p>“main” defines the entry point to your application, which defaults to <code class="prettyprint">index.js</code>. Depending on convention or your framework, it might be <code class="prettyprint">app.js</code> or <code class="prettyprint">main.js</code>. You can, of course, make it anything you want.</p>
        <h4>“scripts”</h4>
        <p>This is an underrated property.</p>
        <p>First, it can be used to do things on prepublish.</p>
        <p>Second, it provides a place where you can alias an array of frequently used commands, ranging from build tasks (defined in gulp or grunt), triggering the installation of other dependencies (with something like bower), starting a development server with webpack, or running a set of bash commands.<br></p>
        <div id="1" class="widgets_on_page wop_tiny1  wop_small1  wop_medium1  wop_large1  wop_wide1">

        <h4>“dependencies”</h4>
        <p>This property is a list of packages needed by your application, along with the compatible semver number. It is a notable property because it can be modified from the terminal when you install local packages.</p>
        <p>This is done by adding <code class="prettyprint">--save</code> (or the shorthand -S) at the end of the <code class="prettyprint">npm install command</code>.</p>
        <p>When you do this, the newly install package(s) gets added to the list of dependencies in your package.json file.</p>
        <p>Similarly, a dependency can also be removed by adding <code class="prettyprint">--save</code> when running the <code class="prettyprint">npm uninstall</code> command.</p>
        <p>It is important to be aware of the semver versioning patterns of each of the dependencies and what they mean.</p>
        <p>If the semver rule is too strict, you lose out on new features and improvements, whereas if the semver rule is too relaxed, a breaking version of a package can be installed along the line.</p>
        <p>A broken package install can prove to be quite difficult to resolve, especially when the minified version of the package is used.</p>
        <h4>“devDependencies”</h4>
        <p>Separate from the dependencies property, the “devDependencies” property allows you to define dependencies that are only used during the development phase and are not required for the production build (like ESLint, grunt-contrib packages, and Protractor). Just as with dependencies, this property can be modified from the terminal by adding <code class="prettyprint">--save-dev</code> (or the shorthand <code class="prettyprint">-D</code>) to the end of the <code class="prettyprint">npm install</code> command or the <code class="prettyprint">npm uninstall</code> command. The same caution applies to versioning as mentioned under dependencies.</p>
        <h4>“bin”</h4>
        <p>This is where you can specify your package’s executable file(s), like the path to a CLI utility. This property tells npm to create local or global symlinks to your executables when your package is installed.</p>
        <h4>“config”</h4>
        <p>As discussed earlier, this is where you define configuration settings though your package.json file.</p>
        <h4>“private”</h4>
        <p>When set to true, npm will refuse to publish the package.</p>
        <p>This should not be confused with the access configuration setting.</p>
        <p>This is a handy setting when you have a project that utilizes npm along with its package.json but it is not intended to be published to the npm registry, either scoped or public.</p>
        <p>If your intention changes, simply change the setting to false, and you will be able to publish your package.</p>
        <h4>Custom Properties</h4>
        <p>The <code class="prettyprint">package.json</code> file also accepts custom properties, as long as the name is not already defined or reserved.</p>
        <h2>Developing your own npm package</h2>
        <p>The npm ecosystem is filled with packages, written by thousands of different developers around the world. Each solves some sort of problem, providing an abstraction, or presenting an implementation of something.</p>
        <p>Chances are that, at some point, you too will want to develop your own package to share.</p>
        <p>First, you need to author a <code class="prettyprint">package.json</code> file with the minimum required properties of “name” and “version”, and then the “main” property to specify the entry point, for example, index.js.</p>
        <p>Write your code in that index.js file, login with your npm user account, or create a new user from the terminal, and you are ready to publish it to the npm registry.</p>
        <p>Packages can be public or private.</p>
        <p>Public packages are free to publish and are available for everyone to utilize.</p>
        <p>Private packages, called scoped packages, can only be published if you paid private modules user, and they can be identified by the distinct <code class="prettyprint">@username/</code> that is prepended to the package name.</p>
        <p>Scoped packages can also be published publically by calling the publish command with <code class="prettyprint">--access=public</code>.</p>
        <p>Furthermore, if you spend some more time expanding and improving your package’s code base, and it is time for a new version to be published, you simply alter the version (as per the semver rules and convention) of the package in the <code class="prettyprint">package.json</code> file and type npm publish.</p>
        <p>You can also use the command line interface and call <code class="prettyprint">npm version &lt;update_type&gt;</code>, where update_type is either <code class="prettyprint">patch</code>, <code class="prettyprint">minor</code>, or <code class="prettyprint">major</code>, as described by semver, and this then automatically increments the version number in the package.json file.</p>
        <h2>npm&nbsp;organizations</h2>
        <p>Again, the <a href="https://www.npmjs.com/docs/orgs/">npm documentation</a> for this is excellent, and it would be futile to just repeat their words.</p>
        <p>What can be said about organizations in the npm context is that it is extremely fine grained, and when managed correctly, large teams and individuals, working on scoped or public packages under one name, can be very well managed and restricted.</p>
        <p>While it is complex to master, it’s very rewarding.</p>
        <h2>The power of npm</h2>
        <p>Ultimately the documentation that npm provides is extensive and should be consulted for specifics, but this article provides a useful overview of both basic and more advanced, involved functionality, conveying the awesomeness of npm.</p>
        <p>As with all things, strong opinions exist and many faults can be found. But if you have never tried npm (or node, for that matter), dive in, and explore it for yourself. Chances are you’ll enjoy it more than you think.</p>
