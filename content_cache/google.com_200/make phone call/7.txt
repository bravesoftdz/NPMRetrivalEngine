           <h1 name="3f48" id="3f48" class="graf graf--h3 graf--leading graf--title">How To Combine REST API calls with JavaScript Promises in node.js or OpenWhisk</h1>
           <h2 name="6c88" id="6c88" class="graf graf--h4 graf-after--h3 graf--subtitle">A familiar scenario for node.js developers: you want to make an HTTP request to a REST API, and depending on the response of the first request, you need to make more requests.</h2>
           <p name="c463" id="c463" class="graf graf--p graf-after--h4">I came across a scenario like this just the other day when I read <a href="http://redmonk.com/sogrady/2017/01/13/the-state-of-open-source-licensing/" data-href="http://redmonk.com/sogrady/2017/01/13/the-state-of-open-source-licensing/" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">Stephen O’Grady’s blog post on the state of Open Source licensing</a>.</p>
           <blockquote name="9121" id="9121" class="graf graf--blockquote graf-after--p">
            the unfortunate reality is that increasingly open source repositories are populated not by code carrying one of the aforementioned types of licenses, but no license at all.
           </blockquote>
           <p name="231c" id="231c" class="graf graf--p graf-after--blockquote">Thinking of my own GitHub repositories, I wondered how many would not have a license, and if there was an easy way to find out. I decided <a href="https://xkcd.com/1319/" data-href="https://xkcd.com/1319/" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">against the obvious solution</a> and set out to write some code to find the answer.</p>
           <p name="131a" id="131a" class="graf graf--p graf-after--p">The traditional approach would have been to simply make a request, handle the response in a callback function, make another request, handle the response in a nested callback function, and keep on nesting callbacks as deeply as necessity requires or sanity permits.</p>
           <p name="8073" id="8073" class="graf graf--p graf-after--p">With my exploration of <a href="https://github.com/openwhisk/openwhisk" data-href="https://github.com/openwhisk/openwhisk" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">OpenWhisk</a>, I wanted to see if there was a more elegant way to solve this problem. And there is: the <a href="https://github.com/request/request-promise" data-href="https://github.com/request/request-promise" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">request-promise</a> library (a variation of the <a href="https://github.com/request/request" data-href="https://github.com/request/request" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">popular request HTTP client library for node.js</a>).</p>
           <h3 name="d06b" id="d06b" class="graf graf--h3 graf-after--p">Getting Started with request-promise</h3>
           <p name="4106" id="4106" class="graf graf--p graf-after--h3">To start: note that request-promise requires the Bluebird promise implementation and the original request library. If you don’t already have them, install using npm.</p>
           <pre name="4b17" id="4b17" class="graf graf--pre graf-after--p">npm install request request-promise</pre>
           <p name="8bb7" id="8bb7" class="graf graf--p graf-after--pre">With the dependencies installed, you can make your first request.</p>

           <p name="4210" id="4210" class="graf graf--p graf-after--p">To chain the requests, we define a new function <code class="markup--code markup--p-code">getUserReposUrl</code>, which is using the <code class="markup--code markup--p-code">repos_url</code> key on the passed JSON object.</p>
