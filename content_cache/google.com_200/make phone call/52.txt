     <h1 class="Page-title">Intercom Destination</h1>
     <div class="Page-contents" data-swiftype-name="body" data-swiftype-type="text">
      <p>Segment makes it easy to send your data to <a href="http://intercom.com?utm_source=segmentio&amp;utm_medium=docs&amp;utm_campaign=partners">Intercom</a> (and lots of other destinations). Once you've tracked your data through our open source <a href="/libraries">libraries</a> we'll translate and route your data to Intercom in the format they understand. <a href="/integrations/intercom">Learn more about how to use Intercom with Segment.</a></p>
      <h2 id="getting-started">Getting Started</h2>
      <ol>
       <li>Authorize your Intercom account in Segment and select the Intercom Account you want to sync with Segment. You can choose which account to sync in the drop down menu in the top right.</li>
      </ol>
      <p><img src="https://i.gyazo.com/733dad7de64f4db9be7af3c34a53f222.png" alt=""></p>
      <ol>
       <li>Now that you have authenticated, activate the Intercom destination.</li>
      </ol>
      <p><img src="https://i.gyazo.com/9c21f88ec9c4011e8316a5151e36d125.png" alt=""></p>
      <ol>
       <li>Once you have activated the destination this is what happens:</li>
      </ol>
      <ul>
       <li>Our CDN is updated within 5-10 minutes. Then our snippet will start asynchronously loading Intercom’s library.js onto your page. This means you should remove Intercom’s snippet from your page.</li>
       <li>Because Intercom only associates Track events with known users, an <a href="#identify">Identify</a> call with a userId is required before Track events will be associated properly.</li>
      </ul>
      <hr>
      <h2 id="page">Page</h2>
      <p>Calling <a href="/docs/spec/page/"><code>page</code></a> in Analytics.js triggers the Intercom update method, which will look for new Intercom messages that should be displayed to the current user.</p>
      <p>The <a href="/docs/spec/page/"><code>page</code></a> and <a href="/docs/spec/screen/"><code>screen</code></a> methods don’t have any effect on Intercom when used from our server-side or mobile SDKs.</p>
      <p>If you have the Intercom’s <strong>Respond</strong> package, simply calling <code>.page()</code> (which is already included in our snippet) will trigger the chat widget to appear. Otherwise, you would need to call <code>identify</code> to make the chat widget appear.</p>
      <p><strong>Note:</strong> If you have the <strong>Respond</strong> package and calling <code>page</code> still does not show your chat widget, be sure to check your “Visitors on your website” setting inside your Intercom account!</p>
      <h2 id="identify">Identify</h2>
      <h3 id="from-the-browser">From the Browser</h3>
      <p>When you call <a href="/docs/spec/identify/"><code>identify</code></a> on <code>analytics.js</code>, we create the <code>intercomSettings</code> object and load Intercom’s javascript into the page. Here’s an example <code>identify</code> call that you can send:</p>
      <pre><code class="lang-javascript">analytics.identify('user1234', {
  email: 'teemo@segment.com',
  name: 'Teemo',
  company: {
    id: '813',
    name: 'Teemo Holding Company'
    createdAt: 'Thu Mar 24 2016 17:44:51 GMT+0000 (UTC)'
  },
  createdAt: 'Thu Mar 24 2016 17:46:45 GMT+0000 (UTC)'
});
</code></pre>
      <p>Here’s how we create the <code>intercomSettings</code> object:</p>
      <ul>
       <li>The <code>userId</code> parameter (the first argument) is set to <code>intercomSettings.user_id</code>.</li>
       <li>The <code>traits</code> parameter (the second argument) is set to <code>intercomSettings.custom_attributes</code>.</li>
       <li>If you send <code>traits.email</code> we set it as <code>intercomSettings.email</code>.</li>
       <li>If you send <code>traits.name</code> we set it as <code>intercomSettings.name</code>.</li>
       <li>If you send <code>traits.company</code> we set it as <code>intercomSettings.company</code>.</li>
       <li>If you send <code>traits.createdAt</code> as a javascript Date object, a valid date string, milliseconds since the epoch, or as seconds like a Unix timestamp, we set <code>intercomSettings.created_at</code> as the unix timestamp in seconds for that created date.</li>
      </ul>
      <p><strong>Note:</strong> -<code>traits.avatar</code> is read-only. Intercom determines the avatar based on the email address of the user.</p>
      <ul>
       <li>Passing <code>traits.company</code> will create a new Intercom Company if the <code>company_id</code> does not match a known company. If company is a string, we will set the <code>company_id</code> as a hash of <code>company_name</code> as an id is required to <a href="https://developers.intercom.io/reference#user-model">associate the user to the company</a>.</li>
      </ul>
      <p>The <a href="/docs/spec/group/"><code>group</code> call</a> may be used to create/update company profiles explicitly.</p>
      <h3 id="collect-context">Collect Context</h3>
      <p>With this option selected, <code>identify</code> calls will include contextual information collected by <a href="/docs/sources/#mobile">Segment’s mobile libraries</a> if it is available. This info will be set as Custom Attributes on the Intercom user.</p>
      <p>The fields collected from the <a href="/docs/spec/common/">context object</a> are device.type, device.manufacturer, device.model, os.name, os.version, app.name, app.version and will populate in Intercom as device_type, device_manufacturer, etc.</p>
      <h3 id="server-side-mobile">Server-Side &amp; Mobile</h3>
      <p>When you call <a href="/docs/spec/identify/"><code>identify</code></a> from any of the server-side or mobile sources, we create or update a user in Intercom. We’ll map our <a href="/docs/spec/identify#special-traits">special traits</a> (<code>email</code>, <code>firstName</code>, <code>lastName</code>, <code>createdAt</code>, and <code>company</code>) to Intercom special properties.</p>
      <p>If you want to use Intercom’s <code>last_request_at</code>, you’ll need to pass in <code>active: true</code> in the context object. Then, by default we will set <code>last_request_at</code> to the current time; however, if you pass in your own timestamp, pass it in as <code>lastRequestAt</code> (in camelCase), and we’ll set <code>last_request_at</code> to that value in our server-side sources (coming soon on mobile!).</p>
      <p>If you want to include <code>last_seen_user_agent</code> then include it in the <code>context.userAgent</code>. Similarly with <code>last_seen_ip</code> which is used for geolocation, you can include the IP address at <code>context.ip</code>. <a href="#last-seen">Click here for an example</a>.</p>
      <h2 id="track">Track</h2>
      <p>Segment supports Intercom events in all of our sources through the <a href="/docs/spec/track/"><code>track</code></a> method.</p>
      <p>If you send <code>properties.revenue</code> and <code>properties.currency</code>, we will format that according to <a href="https://developers.intercom.io/reference#event-metadata-types">Intercom’s Monetary Amount</a> and send it as:</p>
      <pre><code>price: {
  amount: &lt;properties.revenue&gt; * 100, // since Intercom requires this in cents
  currency: &lt;properties.currency&gt; // will default to 'usd'
}
</code></pre>
      <h3 id="limited-properties">Limited Properties</h3>
      <p>Intercom can only store <a href="http://docs.intercom.io/Intercom-for-user-analysis/Tracking-User-Events-in-Intercom#metadata-support">5 event properties</a> per event. That means if you send an event to Segment with more than 5 properties, Intercom will only show the first 5 properties.</p>
      <h3 id="limited-events">Limited Events</h3>
      <p>Intercom only allows a total of <a href="http://docs.intercom.io/Intercom-for-user-analysis/Tracking-User-Events-in-Intercom#events-faqs">120 unique event names</a> (see the questions titled “Is there a limit on the number of event types I can send?”). That means if you are sending Segment more than 120 unique event names, Intercom will only accept the first 120 events that hit their servers.</p>
      <h3 id="server-side-race-condition">Server-side Race Condition</h3>
      <p>Because our server-side libraries batch calls by default, it may happen that an Identify call intended to create a user arrives simultaneously with a Track event associated with this user. If the Track event is processed before the user is created, an error will result and the event will not be recorded.</p>
      <p><a href="/docs/sources/server/node/#batching">Adding a Flush method</a> immediately following the Identify, and before any subsequent Track event, will help the Identify call reach Intercom first to create the user. Generally, this prevents the race condition, but an additional timeout can be used if necessary.</p>
      <h2 id="group">Group</h2>
      <p>Segment supports Intercom companies in all of our sources through the <a href="/docs/spec/group/"><code>group</code></a> method. Users can be put into multiple groups, which will associate them to multiple companies in Intercom.</p>
      <pre><code class="lang-javascript">analytics.group('companyId123', {
  name: 'Segment'
});
</code></pre>
      <p>Segment also maps to Intercom’s special properties <code>monthly_spend</code> and <code>plan</code>. All you have to do is set <code>traits.monthlySpend</code> and <code>traits.plan</code>.</p>
      <p><strong>Note:</strong> In order for the Company Sessions Count to update within Intercom, the company must first be recorded in an <code>identify</code> call.</p>
      <hr>
      <h2 id="features">Features</h2>
      <h3 id="arrays-and-objects">Arrays and Objects</h3>
      <p>Intercom does <strong>not</strong> support custom arrays or objects. Per Intercom’s request, we removed support for this feature starting Aug 21st, 2017. This means that if you want a certain user <code>trait</code> or event <code>property</code> to be sent to Intercom, you must send them at the top level.</p>
      <p>This limitation does not apply however for mapping <code>company</code> objects. We will continue to handle that in the same way. This is only applicable for any custom traits or properties.</p>
      <h3 id="disassociating-users-from-a-company">Disassociating Users from a Company</h3>
      <p>You can unassociate a user from a company by passing in a field inside the <code>company</code> trait with <code>remove: true</code> in your <code>identify</code> calls.</p>
      <pre><code class="lang-javascript">analytics.identify({
  userId: '019mr8mf4r',
  traits: {
    name: 'Michael Bolton',
    email: 'mbolton@initech.com',
    plan: 'Enterprise',
    company: {
      id: 12345,
      remove: true
    }
  }
});
</code></pre>
      <h3 id="identity-verification">Identity Verification</h3>
      <p>If you want to enable Intercom <a href="https://docs.intercom.com/configure-intercom-for-your-product-or-site/staying-secure/enable-identity-verification-on-your-web-product">identity verification</a> for <code>analytics.js</code>, you can pass in the <code>user_hash</code> variable inside the context object.</p>
      <p>The <code>user_hash</code> should be a SHA256 hash of your Intercom API secret and the <strong>userId</strong>. The hash is not based on the email, it’s based on the <code>userId</code>. Here’s an example rendering an identify call with identity verification:</p>
      <pre><code class="lang-javascript">analytics.identify('&lt;%= current_user.id %&gt;', {
    email: '&lt;%= current_user.email %&gt;',
    createdAt: '&lt;%= current_user.created %&gt;'
}, {
     Intercom: {
        user_hash: '&lt;%= OpenSSL::HMAC.hexdigest("sha256", "YOUR_INTERCOM_APP_SECRET", current_user.id) %&gt;'
     }
});
</code></pre>
      <p><code>YOUR_INTERCOM_APP_SECRET</code> is found in Intercom’s identity verification setup guide.</p>
      <h4 id="identity-verification-plus-filtering-via-destinations-object">Identity verification plus filtering via Destinations Object</h4>
      <p>If using Intercom identity verification AND <a href="https://segment.com/docs/sources/website/analytics.js/#selecting-integrations">selective destinations functionality</a>, the context object will look like this:</p>
      <pre><code class="lang-json">{
     integrations: {
         All: false,
         Intercom: {
            user_hash: '&lt;%= OpenSSL::HMAC.hexdigest("sha256", "YOUR_INTERCOM_APP_SECRET", current_user.id) %&gt;'
         }
     }
}
</code></pre>
      <h3 id="unsubscribe-users">Unsubscribe Users</h3>
      <p>To unsubscribe users from emails, you may set a flag from <strong>server side</strong> libraries, <code>unsubscribedFromEmails</code>, inside <code>context</code> object:</p>
      <p><code>node.js</code> example:</p>
      <pre><code class="lang-javascript">analytics.identify({
  userId: '4832094283057439285723523452345',
  anonymousId:'43254364571',
  context:{
    Intercom: { unsubscribedFromEmails: true }
  },
  traits: {
    firstName: 'John ',
    lastName: 'Jacob',
    email: 'jingleheimer@schmidt.com'
  }
});
</code></pre>
      <p><strong>Note</strong>: This will only work from server side libraries. NOT <code>analytics.js</code>!</p>
      <h3 id="last-seen">Last Seen</h3>
      <p>By default Intercom updates the <strong>Last Seen</strong> user trait whenever a user’s profile is updated by <code>identify</code> calls, or if a group call is sent with a user’s userId. If you want to update a user without updating their <strong>Last Seen</strong> time, pass <code>active</code> with a value of <code>false</code> into the context (see example below) of your <code>identify</code> or <code>group</code> calls. Note that this only works server-side; <strong>Last Seen</strong> will always be updated client-side. Please note that id or name are necessary to update company.</p>
      <p>Here’s a full <code>python</code> example of an <code>identify</code> call with <code>active</code> set to <code>false</code>:</p>
      <pre><code class="lang-python">analytics.identify(user_id='some_user_id', traits={
    "email": "ben@intercom.io",
    "firstName": "Ben",
    "lastName": "McRedmond"
    "createdAt": 1363902294011,
    "plan": "Premium"
}, context={
    "ip": "192.168.0.1",
    "active": False,
    "userAgent": "Mozilla/5.0 (Macintosh; Intel Mac OS X 10_7_5) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/27.0.1453.47 Safari/537.36"
})
</code></pre>
      <h3 id="intercom-tags">Intercom Tags</h3>
      <p>Our API doesn’t support Intercom tags. Traits can be used instead of tags to create segments of users, and the advantage is you can use those traits in other destinations like Segment.</p>
      <h3 id="conditionally-show-the-intercom-chat-widget">Conditionally show the Intercom chat widget</h3>
      <p>You can take advantage of Intercom’s <code>hide_default_launcher</code> option to selectively show the chat widget. According to Intercom’s <a href="https://docs.intercom.io/configure-intercom-for-your-product-or-site/customize-the-intercom-messenger/customize-the-intercom-messenger-technical#show-the-intercom-messenger-to-selected-users-for-web-">docs</a>, you want to first hide the Messenger for all users inside their UI via Messenger settings. Then think about how you want to programmatically decide which users you’d like to show the widget to. Then you can pass an Intercom specific destination setting like this:</p>
      <pre><code class="lang-js">// with analytics.js
analytics.identify('teemo', { someTrait: 'x'}, {
  Intercom: { hideDefaultLauncher: true }
});
</code></pre>
      <p><strong>Note</strong>: You can pass in the Intercom specific option via all supported calls for this destination (<code>page</code>, <code>identify</code>, and <code>group</code>)!</p>

      <h2 id="troubleshooting">Troubleshooting</h2>
      <h3 id="i-m-seeing-a-403-forbidden-error">I’m seeing a <code>403 Forbidden</code> error</h3>
      <p>You probably have <a href="#identity-verification">Intercom’s identity verification</a> setting turned on but are not passing the <code>user_hash</code> correctly or at all.</p>
      <p>You may also have to <a href="https://docs.intercom.io/configuring-for-your-product-or-site/customizing-the-intercom-messenger#whitelist-the-domains-you-want-to-use-intercom-with">whitelist your domain</a> in Intercom’s dashboard. Otherwise, events on non-whitelisted pages may be rejected with a 403 error.</p>
      <h3 id="my-intercom-widget-doesn-t-show-up">My Intercom Widget doesn’t show up</h3>
      <p>Make sure you are sending a <code>page</code> and <code>identify</code> call when the page is loaded. This allows Intercom to register the page and the user, which would enable the widget to appear.</p>
      <p>If you are sending those two calls, then check that the CSS selector for the widget is correct. The default is <code>#IntercomDefaultWidget</code>, but if you <a href="http://docs.intercom.io/configuring-Intercom/in-app-messenger#custom-style">customize your widget</a>, then be sure to update this field accordingly.</p>
      <h3 id="my-client-side-and-server-side-calls-are-going-to-one-segment-source-but-different-intercom-projects">My client-side and server-side calls are going to one Segment source, but different Intercom projects</h3>
      <p>Server-side calls go the the project selected when you authenticated your Intercom account while setting up the destination. Client-side calls go to the project referenced with the <a href="#app-id-required-for-analytics-js-">App ID setting</a>. Make sure those projects are the same.</p>

      <h2 id="settings">Settings</h2>
      <p>Segment lets you change these destination settings via your Segment dashboard without having to touch any code.</p>
      <h3 id="custom-inbox-button-selector">Custom Inbox Button Selector</h3>
      <p>By default, Intercom will inject their own inbox button onto the page, but you can choose to use your own custom button instead by providing a CSS selector like: <code>#my-button</code>. <em>Note: you must have enabled the <strong>Show the Intercom Inbox</strong> setting for this to work.</em> The default value is <code>#IntercomDefaultWidget</code></p>
      <h3 id="enter-your-api-key">Enter your API Key</h3>
      <p>Note: this is <strong>required</strong> to send data to Intercom from your servers. You can find your API key after activating your account under the <strong>Configure &gt; API Keys</strong> at <a href="https://intercom.io">Intercom.io</a>. It should be 40 characters long, and look something like this: <code>4ad6ce80fc1d441324cfb11cdf8d3ade8fc7e8fd</code>.</p>
      <h3 id="app-id-required-for-analytics-js-">App ID (required for analytics.js)</h3>
      <p>Note: this is <strong>required</strong> to send data to Intercom from your browser. You can find your App ID after activating your account under the <strong>Gear Menu &gt; App Settings &gt; API Keys</strong> at <a href="https://intercom.io">Intercom.io</a>. It should look something like this: <code>9iefb489.</code></p>
      <h3 id="include-context-with-identify-calls">Include Context with Identify Calls</h3>
      <p>Pass context data with Identify calls as Custom Attributes. Includes device.type, device.manufacturer, device.model, os.name, os.version, app.name, app.version</p>
      <h3 id="isbulkapienabled">isBulkAPIEnabled</h3>
      <h3 id="oauth">OAuth</h3>
      <p>Intercom uses an OAuth-like interface for reading/writing to their system. This option stores the various values that are needed by Intercom to validate requests (replaces API keys), and the user-specific values for UI display (e.g. user’s email, avatar, list of accounts, etc.). These fields are unique to Intercom, and are cleared out en masse when disconnecting (revoking authorization).</p>
      <h3 id="rich-link-properties">Rich Link Properties</h3>
      <p>To use Intercom’s Rich Link object, list the name of the exact Segment property name and make sure the value of those properties only include <code>url</code> and <code>value</code> keys.</p>
