       </svg> </button> How To Use npm to Manage Node.js Packages on a Linux Server </h1> 
     <span class="meta-section timestamp"><span class="tutorial-date-text">Posted</span><span class="tutorial-date">May 14, 2014</span></span> 
     <span class="meta-section pageviews"><span class="icon icon-eye v-mid"></span><span class="views-count v-mid">100.5k</span><span class="sr-only"> views</span></span> 
     <span class="meta-section tags"> <a class="tag" href="/community/tags/node-js?type=tutorials">Node.js</a> <a class="tag" href="/community/tags/ubuntu?type=tutorials">Ubuntu</a> </span>

     <h3 id="introduction">Introduction</h3> 
     <p>Node.js is a popular Javascript platform for server-side programming that allows you to build and run web apps quickly. In a past guide, we discussed <a href="https://www.digitalocean.com/community/articles/how-to-install-node-js-on-an-ubuntu-14-04-server">how to install Node.js on an Ubuntu 14.04 server</a>.</p> 
     <p>In this guide, we will talk about how to use <code>npm</code>, a Node.js package management system, to manage Node.js packages on your server. Using this utility, you can easily satisfy application dependencies, search for packages, and manage installations.</p> 
     <p>We'll assume that you have both Node.js and <code>npm</code> installed. You can follow the guide above to get these running on Ubuntu 14.04. I recommend using the <code>nvm</code> method covered in that tutorial.</p> 
     <p>If you just want to get started quickly, you can also find the packages in Ubuntu 14.04's repositories and install them by typing:</p> 
     <pre class="code-pre "><code langs="">sudo apt-get update
sudo apt-get install nodejs npm
</code></pre> 
     <h2 id="getting-familiar-with-npm">Getting Familiar with npm</h2> 
     <p>Now that you have the necessary components installed, we can get started.</p> 
     <p>The first thing we are going to do is add some <code>npm</code> auto-complete functionality to our shell by adding some information to the bottom of our shell configuration file.</p> 
     <p>You can do that by typing:</p> 
     <pre class="code-pre "><code langs="">npm completion &gt;&gt; ~/.bashrc
</code></pre> 
     <p>Now, just source the configuration file to re-read the new changes you just added:</p> 
     <pre class="code-pre "><code langs="">source ~/.bashrc
</code></pre> 
     <p>Now, npm should be able to automatically complete its commands by hitting the <code>[TAB]</code> key.</p> 
     <p>To get ourselves oriented, the best thing to do is get familiar with the help functionality that npm provides. To get an idea of the subcommands that are available within npm, you can type:</p> 
     <pre class="code-pre "><code langs="">npm help
</code></pre> 
     <hr> 
     <pre class="code-pre "><code langs="">Usage: npm &lt;command&gt;

where &lt;command&gt; is one of:
    add-user, adduser, apihelp, author, bin, bugs, c, cache,
    completion, config, ddp, dedupe, deprecate, docs, edit,
    explore, faq, find, find-dupes, get, help, help-search,
. . .
</code></pre> 
     <p>To get a brief summary of what these subcommands mean, you can type:</p> 
     <pre class="code-pre "><code langs="">npm -l
</code></pre> 
     <hr> 
     <pre class="code-pre "><code langs="">Usage: npm &lt;command&gt;

where &lt;command&gt; is one of:

    adduser     npm adduser
                Then enter stuff at the prompts

    bin         npm bin
                npm bin -g
                (just prints the bin folder)
. . .
</code></pre> 
     <p>You can get a much more extensive overview of each of these commands by using the following format:</p> 
     <pre>
npm help <span class="highlight">subcommand</span>
</pre> 
     <p>This will bring up a separate man page for the desired subcommand.</p> 
     <p>If you don't know which command covers the action you wish to perform, you can also search within the help topics by using something like:</p> 
     <pre>
npm help-search <span class="highlight">search_term</span>
</pre> 
     <h2 id="finding-information-about-your-system">Finding Information about your System</h2> 
     <p>Now that you have set up and know how to access its built-in help, we can explore. Let's start by learning how to ask for information about our system that relates to npm.</p> 
     <p>Before we do that, we should talk about the difference between local and global packages.</p> 
     <h3 id="local-vs-global-packages">Local vs Global Packages</h3> 
     <p>When creating a Node.js application, you create a directory structure to hold your application. Within the project's directory, there will be a subdirectory called <code>node_modules</code>. This holds all of the modules that you can install <strong>locally</strong> for your project.</p> 
     <p>Let's create an empty directory structure that will serve as our package directory as an example:</p> 
     <pre class="code-pre "><code langs="">mkdir -p ~/projects/test_package/node_modules
cd ~/projects/test_package
</code></pre> 
     <p>Modules installed locally can be used within your project by using the regular Javascript <code>require()</code> method. This is what you need most of the time for a package.</p> 
     <p>If you need to use a package's command line capabilities, you should install it <strong>globally</strong>. Globally installed applications are kept in the <code>~/.npm/</code> directory. Their command line functions will be available to all of the Node.js applications. Globally installed packages cannot be used with the Javascript <code>require()</code> though.</p> 
     <p>Knowing this, you can see how we will have functionality that is associated with each specific type of package. By default, most npm commands will assume that you are talking about local packages, unless you pass the <code>-g</code> or <code>--global</code> flags.</p> 
     <h3 id="listing-packages-installed-on-our-system">Listing Packages Installed on our System</h3> 
     <p>Now that you know the difference between local and global packages, we can move on. So, to list the (locally installed) Node.js packages available in our <code>test_package</code> directory, we can type:</p> 
     <pre class="code-pre "><code langs="">npm ls
</code></pre> 
     <hr> 
     <pre class="code-pre "><code langs="">/home/demouser/projects/test_package
??? (empty)
</code></pre> 
     <p>Since we haven't installed any Node.js packages for our application yet, this shows up empty.</p> 
     <p>However, npm installs some packages globally automatically. Let's pass the <code>-g</code> flag to see:</p> 
     <pre class="code-pre "><code langs="">npm ls -g
</code></pre> 
     <hr> 
     <pre class="code-pre "><code langs="">/home/demouser/.nvm/v0.11.13/lib
??? npm@1.4.9
  ??? abbrev@1.0.4
  ??? ansi@0.2.1
  ??? ansicolors@0.3.2
  ??? ansistyles@0.1.3
  ??? archy@0.0.2
. . .
</code></pre> 
     <p>As you can see, there are quite a few global packages. They are shown in a dependency tree format.</p> 
     <p>There are some other options you can use to modify the output. You can use the <code>-l</code> or <code>-a</code> flags to get additional information. You can view this information in JSON by optionally passing the <code>--json</code> flag.</p> 
     <h3 id="querying-for-other-information">Querying for Other Information</h3> 
     <p>Apart from simply asking npm about installed packages, we can ask it some other information about its environment.</p> 
     <p>We can see what npm refers to as the root module directory within the local package hierarchy by typing:</p> 
     <pre class="code-pre "><code langs="">npm root
</code></pre> 
     <hr> 
     <pre class="code-pre "><code langs="">/home/demouser/projects/test_package/node_modules
</code></pre> 
     <p>You can get the top-level path for the package you're working on by using the <code>prefix</code> subcommand:</p> 
     <pre class="code-pre "><code langs="">npm prefix
</code></pre> 
     <hr> 
     <pre class="code-pre "><code langs="">/home/demouser/projects/test_package
</code></pre> 
     <p>These can also be used with the <code>-g</code> flag to see where the global files are stored.</p> 
     <p>To see where npm will install executables, you can find the <code>bin</code> directory for the current project:</p> 
     <pre class="code-pre "><code langs="">npm bin
</code></pre> 
     <hr> 
     <pre class="code-pre "><code langs="">/home/demouser/projects/test_package/node_modules/.bin
</code></pre> 
     <h3 id="exploring-a-package-39-s-files">Exploring a Package's Files</h3> 
     <p>To quickly go to the directory location where a package is installed, you can actually just use the <code>explore</code> subcommand. This will open a subshell in the directory where the package is installed.</p> 
     <p>For instance, we can open a subshell in the directory where npm itself is installed by typing:</p> 
     <pre class="code-pre "><code langs="">npm explore -g npm
</code></pre> 
     <hr> 
     <pre class="code-pre "><code langs="">Exploring /home/demouser/.nvm/v0.11.13/lib/node_modules/npm
Type 'exit' or ^D when finished
</code></pre> 
     <p>You can manipulate or explore the files in this directory and then exit out of the subshell to return to what you were previously doing.</p> 
     <pre class="code-pre "><code langs="">exit
</code></pre> 
     <h2 id="using-npm-as-a-package-manager">Using npm as a Package Manager</h2> 
     <p>We've covered a lot of ground so far, but we've skirted the main functionality of npm until now. The main reason that people use npm is for its package management capabilities.</p> 
     <h3 id="searching-for-and-installing-packages">Searching for and Installing Packages</h3> 
     <p>When you need a Node.js package, you can search for it using npm. Let's pretend we're going to need the <code>express</code> package for our project. Express is a web framework.</p> 
     <p>We can search for this package in the npm package archives, which are located at <a href="https://www.npmjs.org">the npm website</a>:</p> 
     <pre class="code-pre "><code langs="">npm search express
</code></pre> 
     <p>This will actually return an extremely long list, since express is a very popular package. You can pipe this into a pager like <code>less</code> to view it easier. We can also use regular expressions (with some funky characters to escape shell interpretation) like this:</p> 
     <pre class="code-pre "><code langs="">npm search /^express\[\ \]
</code></pre> 
     <hr> 
     <pre class="code-pre "><code langs="">NAME    DESCRIPTION                                AUTHOR          DATE       VERSIO
express Sinatra inspired web development framework =tjholowaychuk… 2014-05-12 4.2.0
</code></pre> 
     <p>If you are getting too many results though, it might be easier just to go to the npm site to see what the exact package is called.</p> 
     <p>When you know the name of the package, you can install it. Since we need the command line functionality of express, we'll install it globally:</p> 
     <pre class="code-pre "><code langs="">npm install -g express
</code></pre> 
     <p>This will download and install the express package and all of its dependencies into the global npm root directory.</p> 
     <p>If we want the package in our local application's module directory so that it can be included in the app with a <code>require()</code> line, you can type:</p> 
     <pre class="code-pre "><code langs="">cd ~/projects/test_package
npm install express
</code></pre> 
     <p>We don't actually need to install the package and dependencies in both places however, and we'll show you how to avoid this below.</p> 
     <h3 id="uninstalling-de-duplicating-pruning-and-linking-packages">Uninstalling, De-duplicating, Pruning, and Linking Packages</h3> 
     <p>Since we want this package in both our global and local scope, we need to provide it with a way to have this package available in both places.</p> 
     <p>First, let's uninstall our local version of the package, like this:</p> 
     <pre class="code-pre "><code langs="">npm uninstall express
</code></pre> 
     <p>Now, we can tell npm that we want to link our global copy to the local package, like this:</p> 
     <pre class="code-pre "><code langs="">npm link express
</code></pre> 
     <p>This will create a symbolic link within our <code>node_modules</code> directory to our global package, making it available in both places.</p> 
     <p>We could have done the reverse of this by uninstalling the global version, going into the local package and linking globally. We could do that like this.</p> 
     <p>First, uninstall the local link and the global package:</p> 
     <pre class="code-pre "><code langs="">npm uninstall express       # This removes the symbolic link in local
npm uninstall -g express    # This removes the actual package in global
</code></pre> 
     <p>Reinstall the package as a local package:</p> 
     <pre class="code-pre "><code langs="">npm install express
</code></pre> 
     <p>The package is in our local application directory (<code>~/projects/test_package/node_modules</code>), but not installed globally. We can link it globally to going to the package directory and calling the link subcommand with no additional arguments:</p> 
     <pre class="code-pre "><code langs="">cd ~/projects/test_package/node_modules/express
npm link
</code></pre> 
     <p>We could use this same procedure to make our own <code>test_package</code> available to other packages we are creating (assuming it had the proper directory structure and files, which it does not now).</p> 
     <p>While we are installing and shuffling our packages around, we should take advantage of the <code>dedupe</code> function.</p> 
     <p>Our packages have dependencies installed under them in the hierarchy. However, some packages may have shared dependencies. If our package directory contains two modules that each require a package called "blah", it will be installed twice, once under "package1/node<em>modules/blah" and one under "package2/node</em>modules/blah".</p> 
     <p>The <code>dedupe</code> subcommand looks for these cases and moves shared dependencies up a level, where both packages will find the single installation of the dependency. </p> 
     <p>For example, we can try to dedupe the dependencies of our packages in the express directory. We can do this in the package directory like this:</p> 
     <pre class="code-pre "><code langs="">cd ~/projects/test_package/node_modules/express
npm dedupe
</code></pre> 
     <p>This will find two packages (mime and debug). The debug module is actually also a dependency of another express module dependency called "send". This would normally be deduplicated, but there is a version conflict, so nothing is done in this instance.</p> 
     <p>Let's say that you have deduped your packages, which moves some dependencies upwards, but then you uninstall the packages that formerly owned the dependency package. The dependency will not be removed because it is not within the packages subdirectory anymore.</p> 
     <p>In these cases, we can tell npm to "prune" packages that aren't needed in our system. This means packages that aren't listed in our parent package's dependency list.</p> 
     <p>You can remove the unwanted dependencies by calling:</p> 
     <pre class="code-pre "><code langs="">npm prune
</code></pre> 
     <p>This will remove the unneeded packages.</p> 
     <h3 id="finding-outdated-packages-and-updating-them">Finding Outdated Packages and Updating them</h3> 
     <p>If you need to find out which packages are out of date, you can do so by typing something like this:</p> 
     <pre class="code-pre "><code langs="">npm outdated
</code></pre> 
     <hr> 
     <pre class="code-pre "><code langs="">. . .
Package  Current  Wanted  Latest  Location
type-is    1.1.0   1.1.0   1.2.0  express &gt; type-is
debug      0.8.0   0.8.0   0.8.1  express &gt; send &gt; debug
</code></pre> 
     <p>This will give you a list of packages that have more recent versions available. It will give you a chart showing what is installed, what your application requested, and how that differs from the current version.</p> 
     <p>If you want to update your packages, you should edit the version requested in the package's <code>package.json</code> and then issue the update command:</p> 
     <pre class="code-pre "><code langs="">npm update
</code></pre> 
     <h2 id="conclusion">Conclusion</h2> 
     <p>We've covered some of the more common ways to use npm as a package manager. You should now have a good idea of how to acquire and manage other people's packages for use in your own development.</p> 
     <p>In our <a href="https://www.digitalocean.com/community/articles/how-to-use-npm-to-build-and-publish-node-js-packages-on-a-linux-server">next guide on npm</a>, we'll go over how to configure npm to alter its behavior, how to create packages with npm, and how to publish your packages and interact with the npm website.</p>
