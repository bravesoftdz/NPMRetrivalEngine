      <h1 id="calling-kotlin-from-java">Calling Kotlin from Java</h1> 
      <p>Kotlin code can be called from Java easily.</p> 
      <h2 id="properties">Properties</h2> 
      <p>A Kotlin property is compiled to the following Java elements:</p> 
      <ul> 
       <li>A getter method, with the name calculated by prepending the <code>get</code> prefix;</li> 
       <li>A setter method, with the name calculated by prepending the <code>set</code> prefix (only for <code>var</code> properties);</li> 
       <li>A private field, with the same name as the property name (only for properties with backing fields).</li> 
      </ul> 
      <p>For example, <code>var firstName: String</code> gets compiled to the following Java declarations:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-java">private String firstName;

public String getFirstName() {
    return firstName;
}

public void setFirstName(String firstName) {
    this.firstName = firstName;
}
</code></pre> 
      <p>If the name of the property starts with <code>is</code>, a different name mapping rule is used: the name of the getter will be the same as the property name, and the name of the setter will be obtained by replacing <code>is</code> with <code>set</code>. For example, for a property <code>isOpen</code>, the getter will be called <code>isOpen()</code> and the setter will be called <code>setOpen()</code>. This rule applies for properties of any type, not just <code>Boolean</code>.</p> 
      <h2 id="package-level-functions">Package-Level Functions</h2> 
      <p>All the functions and properties declared in a file <code>example.kt</code> inside a package <code>org.foo.bar</code>, including extension functions, are compiled into static methods of a Java class named <code>org.foo.bar.ExampleKt</code>.</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">// example.kt
package demo

class Foo

fun bar() {
}

</code></pre> 
      <pre><code class="code _highlighted" data-lang="text/x-java">// Java
new demo.Foo();
demo.ExampleKt.bar();
</code></pre> 
      <p>The name of the generated Java class can be changed using the <code>@JvmName</code> annotation:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">@file:JvmName("DemoUtils")

package demo

class Foo

fun bar() {
}

</code></pre> 
      <pre><code class="code _highlighted" data-lang="text/x-java">// Java
new demo.Foo();
demo.DemoUtils.bar();
</code></pre> 
      <p>Having multiple files which have the same generated Java class name (the same package and the same name or the same @JvmName annotation) is normally an error. However, the compiler has the ability to generate a single Java facade class which has the specified name and contains all the declarations from all the files which have that name. To enable the generation of such a facade, use the @JvmMultifileClass annotation in all of the files.</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">// oldutils.kt
@file:JvmName("Utils")
@file:JvmMultifileClass

package demo

fun foo() {
}
</code></pre> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">// newutils.kt
@file:JvmName("Utils")
@file:JvmMultifileClass

package demo

fun bar() {
}
</code></pre> 
      <pre><code class="code _highlighted" data-lang="text/x-java">// Java
demo.Utils.foo();
demo.Utils.bar();
</code></pre> 
      <h2 id="instance-fields">Instance Fields</h2> 
      <p>If you need to expose a Kotlin property as a field in Java, you need to annotate it with the <code>@JvmField</code> annotation. The field will have the same visibility as the underlying property. You can annotate a property with <code>@JvmField</code> if it has a backing field, is not private, does not have <code>open</code>, <code>override</code> or <code>const</code> modifiers, and is not a delegated property.</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">class C(id: String) {
    @JvmField val ID = id
}
</code></pre> 
      <pre><code class="code _highlighted" data-lang="text/x-java">// Java
class JavaClient {
    public String getID(C c) {
        return c.ID;
    }
}
</code></pre> 
      <p><a href="properties.html#late-initialized-properties">Late-Initialized</a> properties are also exposed as fields. The visibility of the field will be the same as the visibility of <code>lateinit</code> property setter.</p> 
      <h2 id="static-fields">Static Fields</h2> 
      <p>Kotlin properties declared in a named object or a companion object will have static backing fields either in that named object or in the class containing the companion object.</p> 
      <p>Usually these fields are private but they can be exposed in one of the following ways:</p> 
      <ul> 
       <li><code>@JvmField</code> annotation;</li> 
       <li><code>lateinit</code> modifier;</li> 
       <li><code>const</code> modifier.</li> 
      </ul> 
      <p>Annotating such a property with <code>@JvmField</code> makes it a static field with the same visibility as the property itself.</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">class Key(val value: Int) {
    companion object {
        @JvmField
        val COMPARATOR: Comparator&lt;Key&gt; = compareBy&lt;Key&gt; { it.value }
    }
}
</code></pre> 
      <pre><code class="code _highlighted" data-lang="text/x-java">// Java
Key.COMPARATOR.compare(key1, key2);
// public static final field in Key class
</code></pre> 
      <p>A <a href="properties.html#late-initialized-properties">late-initialized</a> property in an object or a companion object has a static backing field with the same visibility as the property setter.</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">object Singleton {
    lateinit var provider: Provider
}
</code></pre> 
      <pre><code class="code _highlighted" data-lang="text/x-java">// Java
Singleton.provider = new Provider();
// public static non-final field in Singleton class
</code></pre> 
      <p>Properties annotated with <code>const</code> (in classes as well as at the top level) are turned into static fields in Java:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">// file example.kt

object Obj {
    const val CONST = 1
}

class C {
    companion object {
        const val VERSION = 9
    }
}

const val MAX = 239
</code></pre> 
      <p>In Java:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-java">int c = Obj.CONST;
int d = ExampleKt.MAX;
int v = C.VERSION;
</code></pre> 
      <h2 id="static-methods">Static Methods</h2> 
      <p>As mentioned above, Kotlin represents package-level functions as static methods. Kotlin can also generate static methods for functions defined in named objects or companion objects if you annotate those functions as <code>@JvmStatic</code>. If you use this annotation, the compiler will generate both a static method in the enclosing class of the object and an instance method in the object itself. For example:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">class C {
    companion object {
        @JvmStatic fun foo() {}
        fun bar() {}
    }
}
</code></pre> 
      <p>Now, <code>foo()</code> is static in Java, while <code>bar()</code> is not:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-java">C.foo(); // works fine
C.bar(); // error: not a static method
C.Companion.foo(); // instance method remains
C.Companion.bar(); // the only way it works
</code></pre> 
      <p>Same for named objects:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">object Obj {
    @JvmStatic fun foo() {}
    fun bar() {}
}
</code></pre> 
      <p>In Java:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-java">Obj.foo(); // works fine
Obj.bar(); // error
Obj.INSTANCE.bar(); // works, a call through the singleton instance
Obj.INSTANCE.foo(); // works too
</code></pre> 
      <p><code>@JvmStatic</code> annotation can also be applied on a property of an object or a companion object making its getter and setter methods be static members in that object or the class containing the companion object.</p> 
      <h2 id="visibility">Visibility</h2> 
      <p>The Kotlin visibilities are mapped to Java in the following way:</p> 
      <ul> 
       <li><code>private</code> members are compiled to <code>private</code> members;</li> 
       <li><code>private</code> top-level declarations are compiled to package-local declarations;</li> 
       <li><code>protected</code> remains <code>protected</code> (note that Java allows accessing protected members from other classes in the same package and Kotlin doesn't, so Java classes will have broader access to the code);</li> 
       <li><code>internal</code> declarations become <code>public</code> in Java. Members of <code>internal</code> classes go through name mangling, to make it harder to accidentally use them from Java and to allow overloading for members with the same signature that don't see each other according to Kotlin rules;</li> 
       <li><code>public</code> remains <code>public</code>.</li> 
      </ul> 
      <h2 id="kclass">KClass</h2> 
      <p>Sometimes you need to call a Kotlin method with a parameter of type <code>KClass</code>. There is no automatic conversion from <code>Class</code> to <code>KClass</code>, so you have to do it manually by invoking the equivalent of the <code>Class&lt;T&gt;.kotlin</code> extension property:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">kotlin.jvm.JvmClassMappingKt.getKotlinClass(MainView.class)
</code></pre> 
      <h2 id="handling-signature-clashes-with-jvmname">Handling signature clashes with @JvmName</h2> 
      <p>Sometimes we have a named function in Kotlin, for which we need a different JVM name the byte code. The most prominent example happens due to <em>type erasure</em>:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">fun List&lt;String&gt;.filterValid(): List&lt;String&gt;
fun List&lt;Int&gt;.filterValid(): List&lt;Int&gt;
</code></pre> 
      <p>These two functions can not be defined side-by-side, because their JVM signatures are the same: <code>filterValid(Ljava/util/List;)Ljava/util/List;</code>. If we really want them to have the same name in Kotlin, we can annotate one (or both) of them with <code>@JvmName</code> and specify a different name as an argument:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">fun List&lt;String&gt;.filterValid(): List&lt;String&gt;

@JvmName("filterValidInt")
fun List&lt;Int&gt;.filterValid(): List&lt;Int&gt;
</code></pre> 
      <p>From Kotlin they will be accessible by the same name <code>filterValid</code>, but from Java it will be <code>filterValid</code> and <code>filterValidInt</code>.</p> 
      <p>The same trick applies when we need to have a property <code>x</code> alongside with a function <code>getX()</code>:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">val x: Int
    @JvmName("getX_prop")
    get() = 15

fun getX() = 10
</code></pre> 
      <h2 id="overloads-generation">Overloads Generation</h2> 
      <p>Normally, if you write a Kotlin function with default parameter values, it will be visible in Java only as a full signature, with all parameters present. If you wish to expose multiple overloads to Java callers, you can use the <code>@JvmOverloads</code> annotation.</p> 
      <p>The annotation also works for constructors, static methods etc. It can't be used on abstract methods, including methods defined in interfaces.</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">class Foo @JvmOverloads constructor(x: Int, y: Double = 0.0) {
    @JvmOverloads fun f(a: String, b: Int = 0, c: String = "abc") {
        ...
    }
}
</code></pre> 
      <p>For every parameter with a default value, this will generate one additional overload, which has this parameter and all parameters to the right of it in the parameter list removed. In this example, the following will be generated:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-java">// Constructors:
Foo(int x, double y)
Foo(int x)

// Methods
void f(String a, int b, String c) { }
void f(String a, int b) { }
void f(String a) { }
</code></pre> 
      <p>Note that, as described in <a href="classes.html#secondary-constructors">Secondary Constructors</a>, if a class has default values for all constructor parameters, a public no-argument constructor will be generated for it. This works even if the <code>@JvmOverloads</code> annotation is not specified.</p> 
      <h2 id="checked-exceptions">Checked Exceptions</h2> 
      <p>As we mentioned above, Kotlin does not have checked exceptions. So, normally, the Java signatures of Kotlin functions do not declare exceptions thrown. Thus if we have a function in Kotlin like this:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">// example.kt
package demo

fun foo() {
    throw IOException()
}
</code></pre> 
      <p>And we want to call it from Java and catch the exception:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-java">// Java
try {
  demo.Example.foo();
}
catch (IOException e) { // error: foo() does not declare IOException in the throws list
  // ...
}
</code></pre> 
      <p>we get an error message from the Java compiler, because <code>foo()</code> does not declare <code>IOException</code>. To work around this problem, use the <code>@Throws</code> annotation in Kotlin:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">@Throws(IOException::class)
fun foo() {
    throw IOException()
}
</code></pre> 
      <h2 id="null-safety">Null-safety</h2> 
      <p>When calling Kotlin functions from Java, nobody prevents us from passing <em class="keyword">null</em> as a non-null parameter. That's why Kotlin generates runtime checks for all public functions that expect non-nulls. This way we get a <code>NullPointerException</code> in the Java code immediately.</p> 
      <h2 id="variant-generics">Variant generics</h2> 
      <p>When Kotlin classes make use of <a href="generics.html#declaration-site-variance">declaration-site variance</a>, there are two options of how their usages are seen from the Java code. Let's say we have the following class and two functions that use it:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">class Box&lt;out T&gt;(val value: T)

interface Base
class Derived : Base

fun boxDerived(value: Derived): Box&lt;Derived&gt; = Box(value)
fun unboxBase(box: Box&lt;Base&gt;): Base = box.value
</code></pre> 
      <p>A naive way of translating these functions into Java would be this:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-java">Box&lt;Derived&gt; boxDerived(Derived value) { ... }
Base unboxBase(Box&lt;Base&gt; box) { ... }
</code></pre> 
      <p>The problem is that in Kotlin we can say <code>unboxBase(boxDerived("s"))</code>, but in Java that would be impossible, because in Java the class <code>Box</code> is <em>invariant</em> in its parameter <code>T</code>, and thus <code>Box&lt;Derived&gt;</code> is not a subtype of <code>Box&lt;Base&gt;</code>. To make it work in Java we'd have to define <code>unboxBase</code> as follows:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-java">Base unboxBase(Box&lt;? extends Base&gt; box) { ... }  
</code></pre> 
      <p>Here we make use of Java's <em>wildcards types</em> (<code>? extends Base</code>) to emulate declaration-site variance through use-site variance, because it is all Java has.</p> 
      <p>To make Kotlin APIs work in Java we generate <code>Box&lt;Super&gt;</code> as <code>Box&lt;? extends Super&gt;</code> for covariantly defined <code>Box</code> (or <code>Foo&lt;? super Bar&gt;</code> for contravariantly defined <code>Foo</code>) when it appears <em>as a parameter</em>. When it's a return value, we don't generate wildcards, because otherwise Java clients will have to deal with them (and it's against the common Java coding style). Therefore, the functions from our example are actually translated as follows:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-java">// return type - no wildcards
Box&lt;Derived&gt; boxDerived(Derived value) { ... }
 
// parameter - wildcards 
Base unboxBase(Box&lt;? extends Base&gt; box) { ... }
</code></pre> 
      <p>NOTE: when the argument type is final, there's usually no point in generating the wildcard, so <code>Box&lt;String&gt;</code> is always <code>Box&lt;String&gt;</code>, no matter what position it takes.</p> 
      <p>If we need wildcards where they are not generated by default, we can use the <code>@JvmWildcard</code> annotation:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">fun boxDerived(value: Derived): Box&lt;@JvmWildcard Derived&gt; = Box(value)
// is translated to 
// Box&lt;? extends Derived&gt; boxDerived(Derived value) { ... }
</code></pre> 
      <p>On the other hand, if we don't need wildcards where they are generated, we can use <code>@JvmSuppressWildcards</code>:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">fun unboxBase(box: Box&lt;@JvmSuppressWildcards Base&gt;): Base = box.value
// is translated to 
// Base unboxBase(Box&lt;Base&gt; box) { ... }
</code></pre> 
      <p>NOTE: <code>@JvmSuppressWildcards</code> can be used not only on individual type arguments, but on entire declarations, such as functions or classes, causing all wildcards inside them to be suppressed.</p> 
      <h3 id="translation-of-type-nothing">Translation of type Nothing</h3> 
      <p>The type <a href="exceptions.html#the-nothing-type"><code>Nothing</code></a> is special, because it has no natural counterpart in Java. Indeed, every Java reference type, including <code>java.lang.Void</code>, accepts <code>null</code> as a value, and <code>Nothing</code> doesn't accept even that. So, this type cannot be accurately represented in the Java world. This is why Kotlin generates a raw type where an argument of type <code>Nothing</code> is used:</p> 
      <pre><code class="code _highlighted" data-lang="text/x-kotlin">fun emptyList(): List&lt;Nothing&gt; = listOf()
// is translated to
// List emptyList() { ... }
</code></pre>
