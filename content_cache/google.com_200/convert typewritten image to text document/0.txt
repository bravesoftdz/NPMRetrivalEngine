       </svg></a>Mammoth .docx to HTML converter</h1> 
     <p>Mammoth is designed to convert .docx documents, such as those created by Microsoft Word, and convert them to HTML. Mammoth aims to produce simple and clean HTML by using semantic information in the document, and ignoring other details. For instance, Mammoth converts any paragraph with the style <code>Heading 1</code> to <code>h1</code> elements, rather than attempting to exactly copy the styling (font, text size, colour, etc.) of the heading.</p> 
     <p>There's a large mismatch between the structure used by .docx and the structure of HTML, meaning that the conversion is unlikely to be perfect for more complicated documents. Mammoth works best if you only use styles to semantically mark up your document.</p> 
     <p>The following features are currently supported:</p> 
     <ul> 
      <li> <p>Headings.</p> </li> 
      <li> <p>Lists.</p> </li> 
      <li> <p>Customisable mapping from your own docx styles to HTML. For instance, you could convert <code>WarningHeading</code> to <code>h1.warning</code> by providing an appropriate style mapping.</p> </li> 
      <li> <p>Tables. The formatting of the table itself, such as borders, is currently ignored, but the formatting of the text is treated the same as in the rest of the document.</p> </li> 
      <li> <p>Footnotes and endnotes.</p> </li> 
      <li> <p>Images.</p> </li> 
      <li> <p>Bold, italics, underlines, strikethrough, superscript and subscript.</p> </li> 
      <li> <p>Links.</p> </li> 
      <li> <p>Line breaks.</p> </li> 
      <li> <p>Text boxes. The contents of the text box are treated as a separate paragraph that appears after the paragraph containing the text box.</p> </li> 
      <li> <p>Comments.</p> </li> 
     </ul>

     <p>You can convert docx files by passing the path to the docx file and the output file. For instance:</p> 
     <pre><code>mammoth document.docx output.html
</code></pre> 
     <p>If no output file is specified, output is written to stdout instead.</p> 
     <p>The output is an HTML fragment, rather than a full HTML document, encoded with UTF-8. Since the encoding is not explicitly set in the fragment, opening the output file in a web browser may cause Unicode characters to be rendered incorrectly if the browser doesn't default to UTF-8.</p>

       </svg></a><code>mammoth.convertToHtml(input, options)</code></h4> 
     <p>Converts the source document to HTML.</p> 
     <ul> 
      <li> <p><code>input</code>: an object describing the source document. On node.js, the following inputs are supported:</p> 
       <ul> 
        <li><code>{path: path}</code>, where <code>path</code> is the path to the .docx file.</li> 
        <li><code>{buffer: buffer}</code>, where <code>buffer</code> is a node.js Buffer containing a .docx file.</li> 
       </ul> <p>In the browser, the following inputs are supported:</p> 
       <ul> 
        <li><code>{arrayBuffer: arrayBuffer}</code>, where <code>arrayBuffer</code> is an array buffer containing a .docx file.</li> 
       </ul> </li> 
      <li> <p><code>options</code> (optional): options for the conversion. May have the following properties:</p> 
       <ul> 
        <li> <p><code>styleMap</code>: controls the mapping of Word styles to HTML. If <code>options.styleMap</code> is a string, each line is treated as a separate style mapping, ignoring blank lines and lines starting with <code>#</code>: If <code>options.styleMap</code> is an array, each element is expected to be a string representing a single style mapping. See "Writing style maps" for a reference to the syntax for style maps.</p> </li> 
        <li> <p><code>includeEmbeddedStyleMap</code>: by default, if the document contains an embedded style map, then it is combined with the default style map. To ignore any embedded style maps, set <code>options.includeEmbeddedStyleMap</code> to <code>false</code>.</p> </li> 
        <li> <p><code>includeDefaultStyleMap</code>: by default, the style map passed in <code>styleMap</code> is combined with the default style map. To stop using the default style map altogether, set <code>options.includeDefaultStyleMap</code> to <code>false</code>.</p> </li> 
        <li> <p><code>convertImage</code>: by default, images are converted to <code>&lt;img&gt;</code> elements with the source included inline in the <code>src</code> attribute. Set this option to an <a href="#image-converters">image converter</a> to override the default behaviour.</p> </li> 
        <li> <p><code>ignoreEmptyParagraphs</code>: by default, empty paragraphs are ignored. Set this option to <code>false</code> to preserve empty paragraphs in the output.</p> </li> 
        <li> <p><code>idPrefix</code>: a string to prepend to any generated IDs, such as those used by bookmarks, footnotes and endnotes. Defaults to an empty string.</p> </li> 
        <li> <p><code>transformDocument</code>: if set, this function is applied to the document read from the docx file before the conversion to HTML. The API for document transforms should be considered unstable. See <a href="#document-transforms">document transforms</a>.</p> </li> 
       </ul> </li> 
      <li> <p>Returns a promise containing a result. This result has the following properties:</p> 
       <ul> 
        <li> <p><code>value</code>: the generated HTML</p> </li> 
        <li> <p><code>messages</code>: any messages, such as errors and warnings, generated during the conversion</p> </li> 
       </ul> </li> 
     </ul>

       </svg></a>Writing style maps</h2> 
     <p>A style map is made up of a number of style mappings separated by new lines. Blank lines and lines starting with <code>#</code> are ignored.</p> 
     <p>A style mapping has two parts:</p> 
     <ul> 
      <li>On the left, before the arrow, is the document element matcher.</li> 
      <li>On the right, after the arrow, is the HTML path.</li> 
     </ul> 
     <p>When converting each paragraph, Mammoth finds the first style mapping where the document element matcher matches the current paragraph. Mammoth then ensures the HTML path is satisfied.</p> 
     <h3><a id="user-content-freshness" class="deep-link" href="#freshness" aria-hidden="true">
       <svg aria-hidden="true" class="deep-link-icon" height="16" version="1.1" viewbox="0 0 16 16" width="16">
        <path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path>
       </svg></a>Freshness</h3> 
     <p>When writing style mappings, it's helpful to understand Mammoth's notion of freshness. When generating, Mammoth will only close an HTML element when necessary. Otherwise, elements are reused.</p> 
     <p>For instance, suppose one of the specified style mappings is <code>p[style-name='Heading 1'] =&gt; h1</code>. If Mammoth encounters a .docx paragraph with the style name <code>Heading 1</code>, the .docx paragraph is converted to a <code>h1</code> element with the same text. If the next .docx paragraph also has the style name <code>Heading 1</code>, then the text of that paragraph will be appended to the <em>existing</em> <code>h1</code> element, rather than creating a new <code>h1</code> element.</p> 
     <p>In most cases, you'll probably want to generate a new <code>h1</code> element instead. You can specify this by using the <code>:fresh</code> modifier:</p> 
     <p><code>p[style-name='Heading 1'] =&gt; h1:fresh</code></p> 
     <p>The two consective <code>Heading 1</code> .docx paragraphs will then be converted to two separate <code>h1</code> elements.</p> 
     <p>Reusing elements is useful in generating more complicated HTML structures. For instance, suppose your .docx contains asides. Each aside might have a heading and some body text, which should be contained within a single <code>div.aside</code> element. In this case, style mappings similar to <code>p[style-name='Aside Heading'] =&gt; div.aside &gt; h2:fresh</code> and <code>p[style-name='Aside Text'] =&gt; div.aside &gt; p:fresh</code> might be helpful.</p>

     <p>To specify a separator to place between the contents of paragraphs that are collapsed together, use <code>:separator('SEPARATOR STRING')</code>.</p> 
     <p>For instance, suppose a document contains a block of code where each line of code is a paragraph with the style <code>Code Block</code>. We can write a style mapping to map such paragraphs to <code>&lt;pre&gt;</code> elements:</p> 
     <pre><code>p[style-name='Code Block'] =&gt; pre
</code></pre>
     <p>Since <code>pre</code> isn't marked as <code>:fresh</code>, consecutive <code>pre</code> elements will be collapsed together. However, this results in the code all being on one line. We can use <code>:separator</code> to insert a newline between each line of code:</p> 
     <pre><code>p[style-name='Code Block'] =&gt; pre:separator('\n')

        <path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path>
       </svg></a>1.0.0</h3> 
     <p>The <code>convertUnderline</code> option is no longer supported. Use style mappings to control how underlines are handled.</p> 
     <h3><a id="user-content-030" class="deep-link" href="#030" aria-hidden="true">
       <svg aria-hidden="true" class="deep-link-icon" height="16" version="1.1" viewbox="0 0 16 16" width="16">
        <path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path>
       </svg></a>0.3.0</h3> 
     <p>If you've defined custom style maps or used a document transform, you will likely need to change your usage slightly. Otherwise, you should be able to continue using Mammoth as before.</p> 
     <h4><a id="user-content-custom-style-maps" class="deep-link" href="#custom-style-maps" aria-hidden="true">
       <svg aria-hidden="true" class="deep-link-icon" height="16" version="1.1" viewbox="0 0 16 16" width="16">
        <path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path>
       </svg></a>Custom style maps</h4> 
     <p>Prior to 0.3.0, Mammoth matched docx paragraphs using style IDs e.g. <code>p.Heading1</code>. These IDs are used internally in the docx format, and are distinct from the style name i.e. the name shown by Microsoft Word or LibreOffice. Although Mammoth still supports matching styles by ID, matching styles by name is preferred. For instance, instead of:</p> 
     <p><code>p.AsideHeading =&gt; h1</code></p> 
     <p>prefer:</p> 
     <p><code>p[style-name='Aside Heading'] =&gt; h1</code></p> 
     <h4><a id="user-content-document-transforms-1" class="deep-link" href="#document-transforms-1" aria-hidden="true">
       <svg aria-hidden="true" class="deep-link-icon" height="16" version="1.1" viewbox="0 0 16 16" width="16">
        <path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path>
       </svg></a>Document transforms</h4> 
     <p>Prior to 0.3.0, Mammoth (misleadingly) assigned the style ID to a property called <code>styleName</code>. The style ID is now assigned to a more appropriate property, <code>styleId</code>. The <code>styleName</code> property is now set to the name of the style. To preserve existing behaviour, any existing document transforms should be rewritten in one of two ways:</p> 
     <ul> 
      <li> <p>Set the <code>styleId</code> property instead of the <code>styleName</code> property</p> </li> 
      <li> <p>Set the <code>styleName</code> property to the name of the style, rather than the ID</p> </li> 
     </ul> 
     <h3><a id="user-content-020" class="deep-link" href="#020" aria-hidden="true">
       <svg aria-hidden="true" class="deep-link-icon" height="16" version="1.1" viewbox="0 0 16 16" width="16">
        <path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path>
       </svg></a>0.2.0</h3> 
     <p>The function <code>mammoth.style()</code> was renamed to <code>mammoth.styleMapping()</code>.</p>
