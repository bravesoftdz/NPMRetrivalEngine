       This article is about the web tracking technique. For bugs in web software, see 
       <a href="/wiki/Software_bug" title="Software bug">software bug</a>.
      </div> 
      <p>A <b>web beacon</b> or <b>web bug</b> is one of various techniques used on <a href="/wiki/Web_page" title="Web page">web pages</a> or <a href="/wiki/Email" title="Email">email</a>, to unobtrusively (usually invisibly) allow checking that a user has accessed some content.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup> Common uses are <a href="/wiki/Email_tracking" title="Email tracking">email tracking</a> and <a href="/wiki/Web_analytics#Page_tagging" title="Web analytics">page tagging</a> for <a href="/wiki/Web_analytics" title="Web analytics">web analytics</a>. Alternative names are <b>tracking bug</b>, <b>tag</b>, or <b>page tag</b>, <b>tracking pixel</b>, <b>pixel tag</b>, <b>1×1 gif</b>, and <b>clear <a href="/wiki/GIF" title="GIF">gif</a></b>. <sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup> When implemented using <a href="/wiki/JavaScript" title="JavaScript">JavaScript</a>, they may be called <b>JavaScript tags</b>.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup></p> 
      <p>There is a work in progress to standardize an interface that web developers can use to asynchronously transfer small HTTP data from the User Agent to a web server that call it simply <b>beacons</b> (in the context of web development) which can be used to send data to a web server prior to the loading of the document without delaying the load and affecting the perception of page load performance for the next navigation.<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup></p>

      <h2><span class="mw-headline" id="Overview">Overview</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_beacon&amp;action=edit&amp;section=1" title="Edit section: Overview">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>A web beacon is any of a number of techniques used to <a href="/wiki/Website_visitor_tracking" title="Website visitor tracking">track who is reading a web page</a> or email, when, and from which computer. They can also be used to see if an email was read or forwarded to someone else, or if a web page was copied to another website. The first web beacons were small images.</p> 
      <p>Some emails and web pages are not wholly self-contained. They may refer to content on another <a href="/wiki/Server_(computing)" title="Server (computing)">server</a>, rather than including the content directly. When an <a href="/wiki/Email_client" title="Email client">email client</a> or <a href="/wiki/Web_browser" title="Web browser">web browser</a> prepares such an email or web page for display, it ordinarily sends a request to the server to send the additional content.</p> 
      <p>These requests typically include the <a href="/wiki/IP_address" title="IP address">IP address</a> of the requesting computer, the time the content was requested, the type of web browser that made the request, and the existence of <a href="/wiki/HTTP_cookie" title="HTTP cookie">cookies</a> previously set by that server. The server can store all of this information, and associate it with a unique tracking token attached to the content request.</p> 
      <h3><span class="mw-headline" id="On_web_pages">On web pages</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_beacon&amp;action=edit&amp;section=2" title="Edit section: On web pages">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p>Web bugs are typically used by <a href="/wiki/Third_parties" class="mw-redirect mw-disambig" title="Third parties">third parties</a> to monitor the activity of customers at a site.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup></p> 
      <p>As an example of the way web beacons can make user logging easier, consider a company that owns a network of sites. This company may have a network that requires all images to be stored on one host computer while the pages themselves are stored elsewhere. They could use web beacons in order to count and recognize users traveling around the different servers on the network. Rather than gathering statistics and managing cookies on all their servers separately, they can use web beacons to keep them all together.</p> 
      <h2><span class="mw-headline" id="Implementation">Implementation</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_beacon&amp;action=edit&amp;section=3" title="Edit section: Implementation">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>Originally, a web beacon was a small (usually 1×1 <a href="/wiki/Pixel" title="Pixel">pixel</a>) <a href="/wiki/Transparent_pixels" class="mw-redirect" title="Transparent pixels">transparent</a> <a href="/wiki/Graphics_Interchange_Format" class="mw-redirect" title="Graphics Interchange Format">GIF</a> or <a href="/wiki/Portable_network_graphic" class="mw-redirect" title="Portable network graphic">PNG</a> image (or an image of the same color as the background) that was embedded in an <a href="/wiki/HTML" title="HTML">HTML</a> page, usually a page on the web or the content of an <a href="/wiki/Email" title="Email">email</a>. Modern web beacons also use the HTML <a href="/wiki/Iframe" class="mw-redirect" title="Iframe">IFrame</a>, style, script, input link, embed, object, and other tags to track usage.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup> Whenever a user opens a page with a graphic <a href="/wiki/Web_browser" title="Web browser">browser</a> or <a href="/wiki/Email_client" title="Email client">email reader</a>, such image and other information is downloaded. This download requires the browser to send a request to the <a href="/wiki/Server_(computing)" title="Server (computing)">server</a> storing that image or information, allowing the organization running that server to keep track of the HTML page.</p> 
      <p>Images and other content do not have to be invisible: any element can be used for tracking. Typically advertisements, banners and buttons are fetched from their site, not from the main site. This allows a third party site to gather information about visitors when they pull HTML content from the main site. Companies or organisations, buttons or images of which are included on many sites, can thus track (part of) the browsing habits of a significant share of web users. Earlier, this included mainly ad- or counter-serving companies, but nowadays buttons of social media sites are becoming common.</p> 
      <p>While web beacons are used in the same way in web pages or emails, they have different purposes:</p> 
      <ol> 
       <li>If the beacon is embedded in an email, the image is requested when the user reads the email for the first time, and can also be requested every time that the user subsequently loads the email;</li> 
       <li>Whenever a web page (with or without beacons) is downloaded, the server holding the page knows and can store the <a href="/wiki/IP_address" title="IP address">IP address</a> of the computer requesting the page; this information can therefore be retrieved from the server <a href="/wiki/Computer_data_logging" class="mw-redirect" title="Computer data logging">log files</a> without the need of using beacons. Beacons are used when the monitoring party does not have easy access to the logs of the main web server and also because IP addresses are often shared amongst multiple users. This may happen when a web site owner does not control its web servers (such as in web hotels), because monitoring is done by a third party, or a greater level of detail needs to be recorded than is possible from web log analysis alone.</li> 
      </ol> 
      <p>As with any files transferred using the <a href="/wiki/Hypertext_Transfer_Protocol" title="Hypertext Transfer Protocol">Hypertext Transfer Protocol</a>, web beacons are requested by sending the server their <a href="/wiki/Uniform_Resource_Locator" class="mw-redirect" title="Uniform Resource Locator">URL</a>, and possibly the URL of the page containing them. Both contain information that can be useful for the gatherer:</p> 
      <ol> 
       <li>The URL of the web page with the beacon allows a server to determine which page is accessed;</li> 
       <li>The URL of the beacon can be appended with an arbitrary string in various ways while still identifying the same object. It can be used to better identify the conditions under which the beacon has been loaded. It can be added while sending the page or by JavaScript scripts after the download.</li> 
      </ol> 
      <p>For example, an email sent to the address <code>somebody@example.org</code> can contain the embedded image of URL <code>http://example.com/bug.gif?somebody@example.org</code>. Whenever the user reads the email, the image at this URL is requested. The part of the URL after the question mark is ignored by the server for the purpose of determining which file to send, but the complete URL is stored in the server's log file. As a result, the file <code>bug.gif</code> is sent and shown in the email reader; at the same time, the server stores the fact that the particular email sent to <code>somebody@example.org</code> has been read. Using this system, a <a href="/wiki/Spam_(electronic)" class="mw-redirect" title="Spam (electronic)">spammer</a> or <a href="/wiki/Email_marketing" title="Email marketing">email marketer</a> can send similar emails to a large number of addresses to check which ones are valid and read by the users.</p> 
      <p>Web beacons can be used in combination with <a href="/wiki/HTTP_cookie" title="HTTP cookie">HTTP cookies</a> like any other object transferred using the HTTP protocol.</p> 
      <h2><span class="mw-headline" id="Email_web_beacons">Email web beacons</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_beacon&amp;action=edit&amp;section=4" title="Edit section: Email web beacons">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>Web beacons embedded in emails have greater privacy implications than beacons embedded in web pages. Through the use of unique identifiers contained in the URL of the web beacons, the sender of an email containing a web beacon is able to record the exact time that a message was read, as well as the <a href="/wiki/IP_address" title="IP address">IP address</a> of the computer used to read the mail or the <a href="/wiki/Proxy_server" title="Proxy server">proxy server</a> that the user went through. In this way, the sender can gather detailed information about when and where each particular recipient reads email. Every subsequent time the email message is displayed can also send information back to the sender.</p> 
      <p>Web beacons are used by email marketers, <a href="/wiki/Spam_(electronic)" class="mw-redirect" title="Spam (electronic)">spammers</a>, and <a href="/wiki/Phishing" title="Phishing">phishers</a> to verify that email addresses are valid, that the content of emails has made it past the spam filters, and that the email is actually viewed by users. When the user reads the email, the email client requests the image, letting the sender know that the email address is valid and that the email was viewed. The email need not contain an advertisement or anything else related to the commercial activity of the sender. This makes detection of such emails harder for <a href="/wiki/Mail_filter" class="mw-redirect" title="Mail filter">mail filters</a> and users.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">[7]</a></sup></p> 
      <p>Tracking via web beacons can be prevented by using <a href="/wiki/Email_client" title="Email client">email clients</a> that do not download images whose URLs are embedded in HTML emails. Many graphical email clients can be configured to avoid accessing remote images. Examples include the <a href="/wiki/Gmail" title="Gmail">Gmail</a>, <a href="/wiki/Yahoo!" title="Yahoo!">Yahoo!</a>, and <a href="/wiki/SpamCop" title="SpamCop">SpamCop</a>/<a href="/wiki/Horde_(software)" title="Horde (software)">Horde</a> <a href="/wiki/Webmail" title="Webmail">webmail</a> clients; <a href="/wiki/Mozilla_Thunderbird" title="Mozilla Thunderbird">Mozilla Thunderbird</a>, <a href="/wiki/Opera_(web_browser)" title="Opera (web browser)">Opera</a>, <a href="/wiki/Pegasus_Mail" title="Pegasus Mail">Pegasus Mail</a>, <a href="/w/index.php?title=IncrediMail&amp;action=edit&amp;redlink=1" class="new" title="IncrediMail (page does not exist)">IncrediMail</a>, <a href="/wiki/Apple_Mail" class="mw-redirect" title="Apple Mail">Apple Mail</a>, later versions of <a href="/wiki/Microsoft_Outlook" title="Microsoft Outlook">Microsoft Outlook</a>, and <a href="/wiki/KMail" class="mw-redirect" title="KMail">KMail</a> mail readers. Other HTML techniques (such as <a href="/wiki/Iframe" class="mw-redirect" title="Iframe">IFrames</a>) can still be used to track email viewing.</p> 
      <p>Text-based mail readers (such as <a href="/wiki/Pine_(email_client)" title="Pine (email client)">Pine</a> or <a href="/wiki/Mutt_(email_client)" title="Mutt (email client)">Mutt</a>) and graphical <a href="/wiki/Email_client" title="Email client">email clients</a> with purely text-based HTML capabilities (such as <a href="/wiki/Mulberry_(email_client)" title="Mulberry (email client)">Mulberry</a>) do not interpret HTML or display images, so their users are not subject to tracking by email web beacons. <a href="/wiki/Plain-text" class="mw-redirect" title="Plain-text">Plain-text</a> email messages cannot contain web beacons because their contents are interpreted as display characters instead of embedded HTML code, so opening messages does not initiate communication. Some email clients offer the option to disable all HTML in every message (thus rendering all messages as plain text), which prevents any web beacons from loading.</p> 
      <p>Many modern email readers and web-based email services will not load images when opening an HTML email from an unknown sender or that is suspected to be spam mail. The user must explicitly choose to load images. Web beacons can also be filtered out at the server level so that they never reach the end user. <a href="/wiki/MailScanner" title="MailScanner">MailScanner</a> is an example of gateway software that can disarm IFrames as well as web beacons. Disconnecting from the Internet before reading any downloaded messages and then deleting those messages suspected of containing web beacons before reconnecting may also eliminate the threat.</p> 
      <p><a href="/wiki/Message_Disposition_Notification" class="mw-redirect" title="Message Disposition Notification">Disposition-Notification-To</a> email headers may be seen as another form of web beacons. See <a class="external mw-magiclink-rfc" rel="nofollow" href="//tools.ietf.org/html/rfc4021">RFC 4021</a>.</p> 
      <h2><span class="mw-headline" id="Beacon_API">Beacon API</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Web_beacon&amp;action=edit&amp;section=5" title="Edit section: Beacon API">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>The Beacon API is a newer technique which does not require the use of invisible images or similar tactics. It is a candidate recommendation of the <a href="/wiki/World_Wide_Web_Consortium" title="World Wide Web Consortium">World Wide Web Consortium</a>, the standards organization for the web.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">[8]</a></sup> It is designed to allow web developers to track the activity of users, by sending information (such as analytics or diagnostic data) back to the web server once the user navigates away from the page.<sup id="cite_ref-9" class="reference"><a href="#cite_note-9">[9]</a></sup> Use of the web beacon API allows tracking without interfering with or delaying navigation away from the site, and is invisible to the end-user.<sup id="cite_ref-10" class="reference"><a href="#cite_note-10">[10]</a></sup> Support for the Beacon API was introduced into Mozilla's Firefox browser in February 2014<sup id="cite_ref-11" class="reference"><a href="#cite_note-11">[11]</a></sup> and in Google's Chrome browser in November 2014<sup id="cite_ref-12" class="reference"><a href="#cite_note-12">[12]</a></sup>.</p>
