            <h2><a href="/project/ctools">Chaos tool suite (ctools)</a></h2> 
            <div class="submitted">
              Posted by 
             <a href="/u/merlinofchaos" title="View user profile." data-uid="26979" class="username">merlinofchaos</a> on 
             <time pubdate datetime="1228505870">5 December 2008</time>, updated 
             <time pubdate datetime="1493410331">28 April 2017</time> 
            </div> 
            <div class="content"> 
             <div class="field field-name-body field-type-text-with-summary field-label-hidden">
              <div class="field-items">
               <div class="field-item even">
                <p>This suite is primarily a set of APIs and tools to improve the developer experience. It also contains a module called the Page Manager whose job is to manage pages. In particular it manages panel pages, but as it grows it will be able to manage far more than just Panels.</p> 
                <p>For the moment, it includes the following tools:</p> 
                <ul>
                 <li> Plugins -- tools to make it easy for modules to let other modules implement plugins from .inc files.</li> 
                 <li> Exportables -- tools to make it easier for modules to have objects that live in database or live in code, such as 'default views'.</li> 
                 <li> AJAX responder -- tools to make it easier for the server to handle AJAX requests and tell the client what to do with them.</li> 
                 <li> Form tools -- tools to make it easier for forms to deal with AJAX.</li> 
                 <li> Object caching -- tool to make it easier to edit an object across multiple page requests and cache the editing work.</li> 
                 <li> Contexts -- the notion of wrapping objects in a unified wrapper and providing an API to create and accept these contexts as input.</li> 
                 <li> Modal dialog -- tool to make it simple to put a form in a modal dialog.</li> 
                 <li> Dependent -- a simple form widget to make form items appear and disappear based upon the selections in another item.</li> 
                 <li> Content -- pluggable content types used as panes in Panels and other modules like Dashboard.</li> 
                </ul>
               </div>
              </div>
             </div> 
            </div>

            <h2><a href="/project/views">Views</a></h2> 
            <div class="submitted">
              Posted by 
             <a href="/u/merlinofchaos" title="View user profile." data-uid="26979" class="username">merlinofchaos</a> on 
             <time pubdate datetime="1132950847">25 November 2005</time>, updated 
             <time pubdate datetime="1503070575">18 August 2017</time> 
            </div> 
            <div class="content"> 
             <div class="field field-name-body field-type-text-with-summary field-label-hidden">
              <div class="field-items">
               <div class="field-item even">
                <h3>You need Views if</h3> 
                <ul>
                 <li> You like the default front page view, but you find you want to sort it differently. </li>
                 <li> You like the default taxonomy/term view, but you find you want to sort it differently; for example, alphabetically. </li>
                 <li> You use /tracker, but you want to restrict it to posts of a certain type. </li>
                 <li> You like the idea of the 'article' module, but it doesn't display articles the way you like. </li>
                 <li> You want a way to display a block with the 5 most recent posts of some particular type. </li>
                 <li> You want to provide 'unread forum posts'. </li>
                 <li> You want a monthly archive similar to the typical Movable Type/Wordpress archives that displays a link to the in the form of "Month, YYYY (X)" where X is the number of posts that month, and displays them in a block. The links lead to a simple list of posts for that month. </li>
                </ul>
                <p>Views can do a lot more than that, but those are some of the obvious uses of Views. </p> 
                <h3>Views for Drupal 8</h3> 
                <p>Views is in Drupal 8 Core!</p>

            <h2><a href="/project/libraries">Libraries API</a></h2> 
            <div class="submitted">
              Posted by 
             <a href="/u/sun" title="View user profile." data-uid="54136" class="username">sun</a> on 
             <time pubdate datetime="1242651119">18 May 2009</time>, updated 
             <time pubdate datetime="1493674134">1 May 2017</time> 
            </div> 
            <div class="content"> 
             <div class="field field-name-body field-type-text-with-summary field-label-hidden">
              <div class="field-items">
               <div class="field-item even">
                <p>The common denominator for all Drupal modules/profiles/themes that integrate with external libraries.</p> 
                <p>This module introduces a common repository for libraries in <code class="language-php">sites/all/libraries</code> resp. <code class="language-php">sites/&lt;domain&gt;/libraries</code> for contributed modules.</p> 
                <dl>
                 <dt>
                  External libraries
                 </dt> 
                 <dd>
                  Denotes libraries (
                  <em>"plugins"</em>) that are neither shipped nor packaged with a project on drupal.org. We do not want to host third-party libraries on drupal.org for a multitude of reasons, starting with licensing, proceeding to different release cycles, and not necessarily ending with fatal errors due to conflicts of having the same library installed in multiple versions.
                 </dd> 
                </dl>
                <p>Drupal 7 only has built-in support for non-external libraries via hook_library(). But it is only suitable for drupal.org projects that bundle their own library; i.e., the module author is the creator and vendor of the library. Libraries API should be used for externally developed and distributed libraries. A simple example would be a third-party jQuery plugin.</p> 
               </div>
              </div>
             </div>

               <div class="field-item even">
                <h2>Module description</h2> 
                <p>The link module provides a standard custom content field for links. With this module, links can be easily added to any content types and profiles and include advanced validating and different ways of storing internal or external links and URLs. It also supports additional link text title, site wide tokens for titles and title attributes, target attributes, css class attribution, static repeating values, input conversion, and many more.</p> 
                <h2>Requirements / Dependencies</h2> 
                <ol>
                 <li><strong>Drupal 6:</strong> Custom content module <a href="http://drupal.org/project/cck" rel="nofollow">(CCK)</a></li> 
                 <li><strong>Drupal 7:</strong> Fields API is provided already by core [no dependencies].</li> 
                 <li><strong>Drupal 8:</strong> Link module is in core now. No module installation needed. Yay! Don't forget to activate it. It's deactivated by default.</li> 
                </ol>
                <p><u>INFO</u> Since some misleading user reports we need to clarify here - Link module is <em>NOT</em> about to add links to any menus or the navigation nor primary/secondary menu. This can be done with default menu module (part of Drupal core). The Link module provides an additional custom field for storing and validating links to be added with any content type, which means another input block additional to your text-body, title, image and any other input you can make on new content creation.</p>

             <div class="field field-name-body field-type-text-with-summary field-label-hidden">
              <div class="field-items">
               <div class="field-item even">
                <p><a href="http://www.jacklmoore.com/colorbox/" rel="nofollow">Colorbox</a> is a light-weight customizable lightbox plugin for jQuery. This module allows for integration of Colorbox into Drupal.</p> 
                <p>Images, iframed or inline content etc. can be displayed in a overlay above the current page.</p> 
                <h3>Features</h3> 
                <p><strong>The Colorbox module:</strong></p> 
                <ul>
                 <li>Works as a Formatter in entities and in views. </li>
                 <li>Excellent integration with core image field and image styles and the <a href="/project/insert" rel="nofollow">Insert</a> module</li> 
                 <li>Choose between a default style and a number of other styles that are included.</li> 
                 <li>Style the Colorbox with a custom Colorbox style in your theme.</li> 
                 <li>Drush command, drush colorbox-plugin, to download and install the Colorbox plugin in sites/all/libraries.</li> 
                </ul>
                <p><strong>The Colorbox plugin:</strong></p> 
                <ul>
                 <li>Compatible with: jQuery 1.3.2+ in Firefox, Safari, Chrome, Opera, Internet Explorer 7+</li> 
                 <li>Supports photos, grouping, slideshow, ajax, inline, and iframed content.</li> 
                 <li>Lightweight: 10KB of JavaScript (less than 5KBs gzipped).</li> 
                 <li>Appearance is controlled through CSS so it can be restyled.</li> 
                 <li>Can be extended with callbacks &amp; event-hooks without altering the source files.</li> 
                 <li>Completely unobtrusive, options are set in the JS and require no changes to existing HTML.</li> 
                </ul>
               </div>
