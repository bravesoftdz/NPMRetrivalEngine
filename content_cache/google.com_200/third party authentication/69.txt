             <h3><a href="/node/32178#comment-6990484" class="permalink active" rel="bookmark">Custom Module Configuration File Best Practice?</a></h3> 
             <div class="submitted"> 
              <a href="/u/mducharme" title="View user profile." data-uid="616996" class="username">mducharme</a> commented 
              <time pubdate datetime="2013-01-27T20:35:58+00:00">27 January 2013 at 20:35</time> 
             </div> 
             <div class="content"> 
              <div class="field field-name-comment-body field-type-text-long field-label-hidden">
               <div class="field-items">
                <div class="field-item even">
                 <p>I'd like to see "Module Integration" added to this list but would like feedback on this problem:</p> 
                 <p>There is a discussion about how best to handle <a href="http://drupal.org/node/1297134" title="Node.js multi-site" rel="nofollow">Node.js multi-site</a> management for node.js configuration files. What is the best way to handle flat-file configurations that have different values per-site?</p> 
                 <p>nodejs/nodejs.config.js &lt;- stores site-specific configuration variables read by nodejs.</p> 
                 <p>This structure blocks us from having the module in /sites/all/modules since each site must have a unique configuration. The only option is to place the module with the site-specific modules folder.</p> 
                 <p>The other issue is that we have to modify the files in the module folder which could be considered a hack. Is there a best practice for managing this scenario?</p> 
                 <p>Some ideas:</p> 
                 <ol>
                  <li>place the configuration file in the root of each multi-site (where settings.php is) and have the third-party app load from there.</li> 
                  <li>create a protected folder in the /files/ directory and load from there. (cache style)</li> 
                  <li>move parts of the module code and configuration file to /libraries/ where code changes are not considered a hack.</li> 
                  <li>The current practice is fine?:<br> sites/mysite1/modules/nodejs/[config 1]<br> sites/mysite2/modules/nodejs/[config 2]<br> sites/mysite3/modules/nodejs/[config 3] </li> 
                 </ol>
                </div>
