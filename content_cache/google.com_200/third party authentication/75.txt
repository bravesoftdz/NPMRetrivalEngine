          <div class="medium-11 large-9 xlarge-8 xxlarge-7 xxxlarge-6 col content medium-col-first"> 
           <p>If you are using some of the currently common frontend toolchains, chances are you are using <a href="http://bower.io/" rel="nofollow">Bower</a>. In short, Bower is a package manager for JavaScript libraries and other frontend dependencies. Its something that usually sneaks in with other build tools. Its lightweight and does its job, so youll probably just set it up and forget it. Turns out you dont need Bower and, in fact, it may actually be causing some of your problems. Lets take a closer look.</p> 
           <h2 id="WhyWeShouldStopUsingBower-AndHowtoDoIt-WhatiswrongwithBower">What is wrong with Bower</h2> 
           <p>As a Bower user, you probably are more than familiar with this sight:</p> 
           <p><a href="http://gofore.com/wp-content/uploads/2015/05/bower-unable-to-find-suitable1.png"><img class="lazy-load alignnone size-large wp-image-4777" src="/wp-content/themes/flatsome/assets/img/lazy.png" data-src="http://gofore.com/wp-content/uploads/2015/05/bower-unable-to-find-suitable1-1024x391.png" alt="bower-unable-to-find-suitable" width="625" height="238"></a></p> 
           <p>You may be accustomed to giving little hints to Bower so it can figure out which package versions are compliant. When you skim through the listed requirements, you cannot escape the feeling that&nbsp;<em>something feels inherently wrong when your package manager needs your help for doing its core job.</em></p> 
           <p>The reason, as it turns out, is that <strong>Bower doesnt support nested dependencies. </strong>If the packages you use contain subdependencies, Bower resolves them into a flat dependency list, which must satisfy all subdependency requirements. If Bower cannot find a version of the dependency that meets all the conditions, then you get a conflict and must resolve it by hand. Bower can persist this choice into a resolutions list and its often not a big deal when you are only dealing with one or two of these resolutions. But the more dependencies you add, the more difficult it becomes to find versions that satisfy all of your dependencies.</p> 
           <p>If your packages are large and few in number, flat dependency management may even be a plus. After all, it means that Bower is checking you dont mix non-compliant modules. But thats also the problem: in an attempt to avoid conflicts, flat dependency management encourages you to use fewer dependencies that do more things at once. A good example of this is jQuery and its plugins. jQuery contains many commonly required features like AJAX and data manipulation methods. Libraries that require these features, often register themself as jQuery plugins. This allows these plugins to share some common functionality. But this approach doesnt scale very well: If you need to share code between several plugins, you will find yourself reinventing dependency management on top of jQuery.</p> 
           <p><strong>Bower is also redundant. </strong>Given the recent popular trends in adopting node and npm in frontend build tooling, you probably already have a package management system. Practices in asset modularization have also evolved,&nbsp;<strong>leaving Bower with a lot of boilerplate you can get rid of</strong>. This includes not only your own code, but the repositories you reference. Bower <a href="https://github.com/Heisenbergjs/heisenberg/pull/18" rel="nofollow">blindly pulls out all of their files</a>, including internal tooling and build scripts you dont actually need or want. While later on <a href="https://github.com/bower/bower.json-spec#ignore" rel="nofollow">ignore support has been added</a>, it is an opt-in configuration not all modules obey.</p> 
           <p>The problems dont end here. <strong>Bower is also unreliable.</strong> It is actually not a real package repository, but rather just a way to handle metadata. So when&nbsp;<a href="http://arstechnica.com/security/2015/04/ddos-attacks-that-crippled-github-linked-to-great-firewall-of-china/" rel="nofollow">Github gets DDOSed</a>, your build breaks. In Bowers defense, things have been improving on this front. Bower does local caching and is also beginning to support <a href="https://www.npmjs.com/package/private-bower" rel="nofollow">private registries</a>. But things on this front are still far from perfect. Bowers package cache is global by default so CI configuration is easy to get wrong, which may lead to <a href="https://github.com/bower/bower/issues/933" rel="nofollow">parallel builds breaking randomly</a>. Also, in the absense of SaaS-provided private registries, hosting a private Bower yourself is still an endeavour not to be undertaken lightly.</p> 
           <p>Just dont get me wrong: back in 2012 Bower was a good tool and probably the only good one to manage frontend assets.&nbsp;But things have moved forward. And so should we.</p> 
           <h2 id="WhyWeShouldStopUsingBower-AndHowtoDoIt-Step1-Movedependenciesfrombower.jsontopackage.json">Step 1  Move dependencies from bower.json to package.json</h2> 
           <p>You are probably already using npm as part of your frontend build tooling. Most of your frontend packages are probably already available on npm too, so theres little reason to pull them from Bower instead of npm.</p> 
           <p>Lets consider the following <b>bower.json</b> as an example:</p> 
           <pre id="file-gistfile1-json-LC10">{
  "name": "my-bower-project",
  "version": "0.1.0",
  "dependencies": {
    "angularjs": "~1.3.15",
    "jquery": "~2.1.3",
    "lodash": "~3.6.0"
  }
}</pre> 
           <p>The equivalent file in npm is <b>package.json</b>. If you dont have it already, generate it with:</p> 
           <pre>npm init</pre> 
           <p>Now, proceed to move your current bower dependencies from <b>bower.json</b> to your&nbsp;<b>package.json</b>.</p> 
           <p>For example, when can add jQuery as npm dependency by invoking:</p> 
           <pre>npm install --save jquery</pre> 
           <p>Npm loads jquery into your local node_modules folder, and stores relevant metadata to <b>package.json</b>.</p> 
           <p>Repeat this for all of the dependencies that are available on npm and you end up with something like:</p> 
           <div title="Hint: double-click to select code"> 
            <pre>{
  ...
  "dependencies": {
&nbsp;   "angular": "^1.3.15",
&nbsp;   "jquery": "^2.1.3",
&nbsp;   "lodash": "^3.6.0"
  }
}</pre> 
           </div> 
           <p>You can now load the dependencies straight from node_modules folder just like you loaded your Bower components from bower_components. Just with this small change, you already gain benefits. Unlike Bower, npm provides you with a real package repository. Also, since you dont need to load packages from third party repositories, your builds are less likely to break.</p> 
           <h2 id="WhyWeShouldStopUsingBower-AndHowtoDoIt-Step2-Userequireinsteadofwiredep">Step 2  Use require instead of wiredep</h2> 
           <p>Dependencies in Bower are commonly wired together with <a href="https://github.com/taptapship/wiredep" rel="nofollow">wiredep</a>. Wiredep is a tool that determines the right order of loading for your Bower packages and injects them into your source file inside placeholder comments.</p> 
           <p>Npm doesnt work like that. With npm, you just write your application as a collection of CommonJS modules. Any module can import its dependencies and export a public API for consumption by its clients.</p> 
           <p>Say you got <em>app.js</em> that depends on several modules: jQuery, Lodash and Angular.js. With require you simply add these as dependencies:</p> 
           <div title="Hint: double-click to select code"> 
            <pre>var $ = require('jquery'),
&nbsp;&nbsp;&nbsp; _ = require('lodash'),
&nbsp;&nbsp;&nbsp; angular = require('angular');</pre> 
           </div> 
           <p>Note that require doesnt work in browsers out-of-the-box. Now, instead of wiredep, you now need another additional tool for resolving and bundling together these modules.</p> 
           <p>One of the popular options for bundling is <strong>Browserify.</strong></p> 
           <p>First, start by installing Browserify globally as a command-line tool:</p> 
           <pre>npm install -g browserify</pre> 
           <p>We can now pass our app script to browserify:</p> 
           <pre>browserify app.js -o bundle.js</pre> 
           <p>And out comes a full, browser-friendly bundle which can be included with a single script tag:</p> 
           <div title="Hint: double-click to select code"> 
            <pre>&lt;script src="bundle.js"&gt;&lt;/script&gt;</pre> 
           </div> 
           <p>Stay productive by installing and using&nbsp;<a href="https://github.com/substack/watchify" rel="nofollow">watchify</a> to efficiently recompile your bundle on each change:</p> 
           <pre>watchify app.js -o bundle.js</pre> 
           <h2>Step 3  Addressing any further concerns</h2> 
           <p>Often thats all you need to do. Just remove Bower from your dependencies and start building on your new stack.</p> 
           <p>However, you may still have some lingering concerns that well attempt to address next.</p> 
           <h3 id="WhyWeShouldStopUsingBower-AndHowtoDoIt-Butbutbut...Iuseprivaterepositories!">But but but I use private repositories!</h3> 
           <p>No worries! You can generally still use them with npm:</p> 
           <div title="Hint: double-click to select code"> 
            <pre>{
  ...
  "dependencies": {
&nbsp;  &nbsp; "lib1": "git+https://jsalonen@github.com/jsalonen/lib1.git",
&nbsp;  &nbsp; "lib2": "git+https://[TOKEN]:x-oauth-basic@bitbucket.org/jsalonen/lib2.git"
  }
}</pre> 
           </div> 
           <p>When it comes to popular repository hosting services, Github is the easiest to use as it has support for basic authentication. Bitbucket doesnt support that, but you can still pull repositories off it <a href="https://confluence.atlassian.com/display/BITBUCKET/OAuth+on+Bitbucket" rel="nofollow">by supplementing your dependency with an OAuth access token</a>. Similar methods apply to other Git repository hosting services.</p> 
           <p><strong>Note that npm also supports private registries.</strong> When you find yourself getting accustomed to npm, you may find yourself splitting your code into a number of smaller, private modules. With a private npm registry like <a href="https://www.npmjs.com/enterprise" rel="nofollow">npm Enterprise</a> or <a href="https://books.sonatype.com/nexus-book/reference/npm-private-registries.html" rel="nofollow">Sonatype Nexus</a>, you can publish and depend upon these modules with minimal hassle. Additionally, you can use private registries to mirror public npm modules to speed up module downloads and make your infrastructure more resilient to problems in the public part of the registry.</p> 
           <h3 id="WhyWeShouldStopUsingBower-AndHowtoDoIt-Butbutbut...Ican'tgetridofallBowerdependencies!">But but but I cant get rid of all Bower dependencies!</h3> 
           <p>If you want to start using browserify but dont want to port all your packages over to npm, <strong>you can still pull some of the packages from Bower with <a href="https://github.com/eugeneware/debowerify" rel="nofollow">debowerify</a></strong>.</p> 
           <p>Lets say you want to add a dependency to <a href="http://twitter.github.io/typeahead.js/" rel="nofollow">typeahead.js</a>. Heres how to install it with Bower:</p> 
           <pre>bower install --save typeahead.js</pre> 
           <p>The npm way would be to load it using require:</p> 
           <div title="Hint: double-click to select code"> 
            <pre>var typeahead = require('typeahead.js');</pre> 
           </div> 
           <p>To enable this way of loading, install and add debowerify as a dependency:</p> 
           <pre>npm install --save-dev debowerify</pre> 
           <p>Then just use it as a browserify transformation:</p> 
           <pre>browserify -t debowerify app.js -o bundle.js</pre> 
           <p>Note that you still need to specify any such dependencies in <em>bower.json</em> and install them with Bower.</p> 
           <p>Debowerify only works if the&nbsp;<a href="https://github.com/eugeneware/debowerify#notes" rel="nofollow">main entry is defined in the components bower.json</a>.</p> 
           <h3 id="WhyWeShouldStopUsingBower-AndHowtoDoIt-Butbutbut...IneedmodulesoutsideBowerandnpm!">But but but I need modules outside Bower and npm!</h3> 
           <p>Vendor scripts relying on browser DOM can be adapted to Browserify with <a href="https://github.com/thlorenz/browserify-shim" rel="nofollow">browserify-shim.</a></p> 
           <p>If you need to use scripts relying on Asynchronous module definition (AMD), check out <a href="https://github.com/jaredhanson/deamdify" rel="nofollow">deAMDify</a>.</p> 
           <h3 id="WhyWeShouldStopUsingBower-AndHowtoDoIt-Butbutbut...myfrontendandbackenddependenciesgetmixed!">But but but my frontend and backend dependencies get mixed!</h3> 
           <p>If you want to keep your server-side and browser-side applications decoupled, you can just create two separate npm modules and manage dependencies separately.</p> 
           <p>Sometimes decoupling server and client is not desirable. This could be the case if you are writing an isomorphic application (e.g. a single-page app that uses server-side prerendering).</p> 
           <p>If this is the case, specify dependencies in <b>package.json</b> as usual. Proceed to use the&nbsp;<a href="https://github.com/substack/node-browserify#browser-field" rel="nofollow">browser field</a>&nbsp;in package.json to override file resolution for browser-specific versions of files.</p> 
           <p>Say that you want to use <em>lodash</em>&nbsp;on the client and server, but want to use a modern build on the server and compat build in the browser.</p> 
           <p>Begin by adding both lodash versions as npm dependencies:</p> 
           <div title="Hint: double-click to select code"> 
            <pre>npm install --save lodash
npm install --save lodash-compat</pre> 
           </div> 
           <p>Proceed to add a browser field entry for lodash to hint that you want the compat build whenever&nbsp;<em>lodash</em> is required in the browser:</p> 
           <div title="Hint: double-click to select code"> 
            <pre>{
  ...
  "browser": {
 &nbsp;&nbsp; "lodash": "lodash-compat"
&nbsp; }
}</pre> 
           </div> 
           <p>Different versions of lodash are now used on the server and in the browser, allowing you to adapt your code to both environments!</p> 
           <h2 id="WhyWeShouldStopUsingBower-AndHowtoDoIt-Finalwords">Final words</h2> 
           <p>I have to be honest with you: Bower isnt really completely evil and I actually think it really works for a variety of tasks. Bower is also being actively developed, and some of the issues I pointed out in the beginning will probably be addressed in the future.</p> 
           <p>The bigger thing is that the JavaScript ecosystem has changed. Tools like npm and browserify allow us to modularize our applications without coupling our modules to the DOM and relying on global state. With the upcoming ES6 module standard even more is yet to come. So why dont you jump on the module bandwagon today and even more great things will soon come your way!</p>
