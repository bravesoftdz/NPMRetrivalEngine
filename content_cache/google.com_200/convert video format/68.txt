           <h1 id="ibm-pagetitle-h1" class="ibm-h1">Use jspm to harness the future of JavaScript modules</h1>
           <p class="dw-article-subhead">Automate module resolution, and get ready now for language features to come</p> 
           <!-- Article Top Bar --> 
           <div class="ibm-columns dw-article-topbar"> 
            <!-- Author and article info. --> 
            <div class="ibm-col-6-2 ibm-col-medium-6-4 dw-article-metadata">

           <p>JavaScript packaging straddles two worlds: the client side and the system/server side. Client-side JavaScript dependency management and packaging have long been a web development sore spot, with a wide variety of piecemeal solutions in use in the wild. <a href="http://es6-features.org/">ECMAScript 6</a> (ES6), which is still in standards preview at the time of writing, tries to take the best ideas from these solutions and merge them into the language. </p>
           <p><a href="http://jspm.io/">jspm</a> is a progressive framework that makes it possible to use the ES6 dependency-handling syntax for JavaScript now. jspm combines:</p>
           <ul class="ibm-bullet-list">
            <li><span data-widget="tooltip" title="A transpiler takes the source code of a program written in one programming language as its input and produces the equivalent source code in another programming language that operates at approximately the same abstraction level.">Transpiling</span> (via <a href="https://babeljs.io/">Babel</a>) to enable use of coming language features</li>
            <li><a href="https://github.com/systemjs/systemjs">SystemJS</a>, a module abstraction layer that can understand the several packaging styles that are in use</li>
           </ul>
           <p>jspm sits atop Babel and SystemJS and addresses them with a consistent syntax, providing automated module resolution through registries such as npm and GitHub. The framework also includes support for CSS loading and a development server. </p>
           <p>This tutorial gives you a practical introduction to jspm and an overview of what the framework can offer your projects. For context, I start with a quick example of server-side dependency management via the Node Package Manager (npm). Then you'll see how jspm provides comparable ease of use  along with other benefits  on the browser side. See <a href="#artdownload">Download</a> to get the sample code.</p>
           <h2 id="npm-Local" class="ibm-h2">npm local</h2>
           <p>You're probably familiar with Node.js and its application of JavaScript as a general language that's free of the browser. You can run JavaScript files with Node, but almost everything of interest in Node happens via <em>modules</em> (also known as <em>dependencies</em>) that you must include. npm makes it super easy to install and control server-side dependencies at both the global system level and the local project level.</p>
           <p>If you don't already have Node.js and npm on your system, <a href="https://docs.npmjs.com/getting-started/installing-node">install</a> them now. After installation, two commands are on your global path: <code>node</code> and <code>npm</code>.</p>
           <p>Your project is to create a command-line app that includes a spinner  an animated icon that shows the user that processing is in progress and to please hold on. This formerly computing-intensive functionality has been simplified for everyday use and now exists as a node module.</p>
           <p>Install node modules into global context by running <code>npm install <em>module-name</em> -g</code>, but for this app you <strong>don't</strong> want the context to be global. Rather, you want your dependencies to be explicitly defined for the project and contained within the project. That way, you can check out the project from version control and install the dependencies from within the project itself, and you can see the dependencies listed therein.</p>
           <p>So, instead, use package.json  the file that controls npm in a project  to keep things clean and contained. Listing 1 shows a simple package.json for your app. Put this file in the root of a new folder for your project.</p>
           <h5 id="code1" class="ibm-h5">Listing 1. package.json with dependency</h5>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">{
    "name": "jspm-intro",
    "preferGlobal": true,
    "version": "1.0.0",
    "author": "user",
    "description": "JSPM Intro - Demo App",
    "license": "",
    "engines": {
        "node": "&gt;=0.10"
    },
    "scripts": {
        "go": "node app-cl.js"
    },
    "dependencies": {
        "cli-spinner": "0.2.2"
    }
}</pre></span>
           <p>Notice that the <code>dependencies</code> entry in package.json references the <code>cli-spinner</code> module, giving it a version number. (Most JavaScript packages give you the latest version on their npm page  for example, the <a href="https://www.npmjs.com/package/cli-spinner"><code>cli-spinner</code></a> page.) You can also get the current version by running <code>npm install <em>package-name</em></code> from the command line.</p>
           <p>To install the dependencies, run <code>npm install</code> from the command line. Watch as npm downloads the dependencies and places them into /node_modules.</p>
           <p>Notice also in Listing 1 that <code>scripts</code> contains a <code>go</code> command. In the <code>scripts</code> property, you can run arbitrary commands within the defined npm context. So, the dependencies are available to the <code>node app-cl.js</code> command. (The /node_modules directory is put onto the path during execution of <code>npm run-script</code> commands.)</p>
           <p>Listing 2 shows the simple yet profound app-cl.js program.</p>
           <h5 id="code2" class="ibm-h5">Listing 2. Requiring and using a module in a Node app</h5>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">"use strict";
var Spinner = require('cli-spinner').Spinner;

var spinner = new Spinner('This is gonna blow your mind... %s');
spinner.setSpinnerString('|/-\\');

spinner.start();

setTimeout(function() {
    spinner.stop();s
}, 3000);</pre></span>
           <p>Go to the command line and run <code>npm run-script go</code>. You see the spinner in action, which demonstrates clean dependency usage.</p>
           <h2 id="N100F3" class="ibm-h2">jspm for web app use</h2>
           <p>Now suppose that you're writing a web app and want to include <a href="https://jquery.com/">jQuery</a>. If you have npm, you can install the JQuery module the same way that you installed the spinner module: by adding <code>"jquery": "2.2.0"</code> to the <code>dependencies</code> section of package.json. Make that addition now and then run <code>npm install</code> to place jQuery in /node_modules. Regardless of how you install jQuery, though, you need a way to get the jQuery module into your web app.</p>
           <p>Now imagine that your web app has an index.html page, and that's the page that you want to use jQuery from. A simple approach is to create the index.html file and include the jQuery file that's in node_modules/jquery/dist via a <code>&lt;script&gt;</code> tag. Listing 3 shows an example of this approach. (In the demo app, this file is index-script.html; see <a href="#artdownload">Download</a>.)</p>
           <h5 id="code3" class="ibm-h5">Listing 3. Typical old-school index.html with <code>&lt;script&gt;</code> tag include</h5>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">&lt;!doctype html&gt;

&lt;html lang="en"&gt;
&lt;head&gt;
  &lt;meta charset="utf-8"&gt;

  &lt;title&gt;JSPM Intro&lt;/title&gt;
  &lt;meta name="description" content="JSPM Intro"&gt;
  &lt;meta name="author" content="MTyson"&gt;
&lt;/head&gt;

&lt;body&gt;
  &lt;script src="node_modules/jquery/dist/jquery.js"&gt;&lt;/script&gt;
  &lt;script src="app.js"&gt;&lt;/script&gt;
&lt;/body&gt;
&lt;/html&gt;</pre></span>
           <p>Now suppose that in app.js, you want to append some content to the document body. You can do a <code>&amp;("body").append("&lt;div&gt;Ground control to major Tom.&lt;/div&gt;");</code>, because you included the jQuery file and therefore have access to the <code>$</code> global.</p>
           <p>This approach works but has a host of shortcomings, especially as a project becomes more complex and relies on more-complex modules. You need to keep all of the various (potentially deeply nested) dependencies written out in the HTML. And you must separate the include from the use of the package. This approach also requires that modules be exposed in the global namespace.</p>
           <p>You don't want this naive, global-variable approach of yore. Instead, you want nicely modular includes, similar to Java JARs and <code>import</code> statements, that you can manage with a dependency manager that's similar to the Java world's <a href="https://maven.apache.org/">Maven</a> or <a href="http://gradle.org/">Gradle</a>.</p>
           <div class="ibm-pull-quote ibm-h3">
            <blockquote>
             <p><em class="dw-pullquote"><span class="dw-pullquote-open"></span>You can write your includes in ES6 and can pull in packages in many formats, including AMD, CommonJS, ES6, and old-school global packaging.<span class="dw-pullquote-close"></span></em></p>
            </blockquote>
           </div>
           <p>A few syntactical approaches to modularization are available, with <a href="http://www.commonjs.org/">CommonJS</a> recently taking center stage. npm packages use CommonJS. Tools such as <a href="http://browserify.org/">Browserify</a> convert npm packages into browser-consumable ones.</p>
           <p>jspm supports CommonJS but goes a step further to support the coming ES6 include syntax via SystemJS. SystemJS is a kind of meta-import compatibility layer. (In ES6, the <code>System</code> object will be a native object in the browser, which is how SystemJS gets its name.) When you use SystemJS, you can write your includes in ES6 and can pull in packages in many formats, including AMD, CommonJS, ES6, and old-school global packaging.</p>
           <p>Now you'll see how you can use jspm to include jQuery  specifically, how you can use jspm to connect your dependencies to the app.js file.</p>
           <h3 id="N1014A" class="ibm-h3">Installing jspm</h3>
           <ol>
            <li>Add jspm to your project. You can hand-code that addition in the dependencies in package.json, but I want to show you another way. Go to the command line and type <code>npm install jspm --save-dev</code>. This command adds the jspm package to the <code>devDependencies</code> field in your package.json. <code>devDependencies</code> work the same as dependencies but are used only for development tools, not the functioning of the application in question. jspm is only for your build, so you don't need it in <code>dependencies</code>.</li>
            <li>Enable jspm commands. Again, rather than install jspm globally, use npm to keep things clean. Add <code>"jspm": "jspm"</code> to the <code>scripts</code> field, and then drop into the command prompt and run <code>npm run-script jspm init</code>.</li>
            <li>Accept all the defaults for jspm questions. jspm creates a config.js file for you (this file chiefly creates a dependency map), updates package.json, and creates a jspm_modules directory.</li>
           </ol>
           <h3 id="N1016B" class="ibm-h3">The package.json <code>jspm</code> entry</h3>
           <p>You can see in package.json that installation of jspm added a <code>jspm</code> object. Nested in that object are new <code>devDependencies</code>, which function like the other npm dependencies but for jspm only. jspm is keeping things clean for you. Notice that the jspm dependency syntax differs slightly: The package names are prefixed with <code>npm</code>, as in <code>"npm:babel-core@^5.8.24"</code>. The prefix is used because jspm supports multiple <em>registries</em>, of which npm is one. (The other built-in registry is GitHub. jspm's registry system is extensible, so you can define new registry endpoints  for example, to use an organization-specific registry.) The <code>"npm:babel-core@^5.8.24"</code> include tells jspm to get the <code>babel-core</code> package from npm. Babel is what jspm uses to transpile ES6 syntax to JavaScript that the browsers of today can read.</p>
           <p>Add in jQuery. At the command line, run <code>npm run-script jspm install jquery</code>. Look in package.json and find the new entry: <code>"jquery": "npm:jquery@^2.2.0"</code>. You see the same pattern of <em>name</em>, <em>registry</em>, <em>module name</em>, <em>version requirement</em>. jspm maintains a small registry of common requirements such as jQuery; that's how it knew to translate from <code>jquery</code> to <code>npm:jquery</code>. For less common modules, you must specify both the registry and module when you install the module at the command line.</p>
           <p>jspm put the assets in place for you; now you're ready to see how to use them.</p>
           <p>Instead of including the app.js file in index.html, you can now import it as a module. But first you must include a couple of support files: the config.js file that jspm created for you and the system.js file, which is SystemJS. You'll use ES6 syntax to include dependencies, and SystemJS makes it possible for you do so across the various packages that are out there in the wild.</p>
           <p>Listing 4 shows an index.html file that uses the jspm-generated build. Don't be concerned about the extra includes; you'll see how to bundle things for production shortly.</p>
           <h5 id="code4" class="ibm-h5">Listing 4. index.html using jspm-generated build</h5>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">      &lt;!doctype html&gt;

&lt;html lang="en"&gt;
&lt;head&gt;
  &lt;meta charset="utf-8"&gt;

  &lt;title&gt;JSPM Intro&lt;/title&gt;
  &lt;meta name="description" content="JSPM Intro"&gt;
  &lt;meta name="author" content="MTyson"&gt;

  &lt;head&gt;
    &lt;script src="jspm_packages/system.js"&gt;&lt;/script&gt;
    &lt;script src="config.js"&gt;&lt;/script&gt;
    &lt;script&gt;
      System.import('./app');
    &lt;/script&gt;
  &lt;/head&gt;
&lt;/head&gt;

&lt;body&gt;
&lt;/body&gt;
&lt;/html&gt;</pre></span>
           <p>Create an app.js file consisting of:</p>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">import $ from 'jquery';

&amp;("body").append("&lt;div&gt;Ground control to major Tom.&lt;/div&gt;");</pre></span>
           <p>Notice the <code>import $ from 'jquery';</code> import. That is a <em>named import</em>. You can use ES6 syntax in a few different ways to require a package. In this case you import the default module, the classic dollar sign.</p>
           <h3 id="N101C1" class="ibm-h3">Exported functions</h3>
           <p>Look a little deeper into the ES6 syntax. In ES6, a module can define multiple exports. So, in your app.js file, you can create an exported function:</p>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">import &amp; from 'jquery';
export function init(){
      &amp;("body").append("&lt;div&gt;Good morning!.&lt;/div&gt;");
      &amp;("body").append("&lt;div&gt;How are you?.&lt;/div&gt;");</pre></span>
           <p>And in index.html, you can make use of this export, as in Listing 5. Notice that I use a <code>.then(...)</code> promise  similar to AMD syntax  to define a callback function, because the modules are loaded asynchronously. </p>
           <h5 id="code7" class="ibm-h5">Listing 5. Using an exported module</h5>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">  &lt;script&gt;
      System.import('./app').then(function(app){
        app.init();
      });
    &lt;/script&gt;</pre></span>
           <h3 id="N101DC" class="ibm-h3">Loader plugins for CSS and other assets</h3>
           <p>Suppose that you have some CSS files that you want to incorporate into the app. To begin, run <code>npm run jspm install css=npm:jspm-loader-css</code> from the command line to install the jspm CSS loader named <code>css</code>.</p>
           <p>Create a simple app.css file with the following contents:</p>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">div {
  background-color: red;
}</pre></span>
           <p>And now, you can import the CSS <strong>into your JavaScript</strong> by adding this line to app.css: <code>import './app.css!';</code>. Notice that the CSS is also loaded at evaluation time (that is, when the script runs in the browser), so you can be clever about loading CSS modules on demand when needed, if you want.</p>
           <p>You can achieve the same kinds of results for SASS, the CSS preprocessor, with <code>npm run jspm install scss=sass</code>. See the <a href="https://github.com/systemjs/systemjs#plugins">full list</a> of loader plugins that are compatible with SystemJS and jspm.</p>
           <h2 id="N10200" class="ibm-h2">Testing with the jspm dev server</h2>
           <p>To test the app, you need a way to host the files in a server, because the browser can't load the packages from local disk. You can use NGINX or Apache or an Express server  whatever you want. But you have jspm running, so you can install and use the jspm dev server instead.</p>
           <p>As a utility, the jspm dev server prefers to be globally installed. Global installation makes sense for this dependency, because in another environment like staging, test, or production, the app will likely be hosted by a specific server. Run <code>npm install -g jspm-server</code>. Type <code>jspm-server</code> at the project root. Wait for jspm to pop open a browser window and direct you to http://127.0.0.1:8080/, where the app is running and greeting you.</p>
           <p>Note that the jspm dev server supports proxying of requests. So if you have a back-end service to integrate with, you can map certain URI paths (for example, <code>/api</code>) to the back end during development.</p>
           <h2 id="N10216" class="ibm-h2">Bundling for production</h2>
           <div class="dw-article-sidebar ibm-background-cool-white-20">
            <h5>HTTP/2 and bundling</h5>
            <p>For the moment, you still need to bundle packages for best performance. But ES6 syntax is designed to work within the context of <a href="https://http2.github.io/">HTTP/2</a>, which will handle multiple file loading optimization at the protocol level.</p>
           </div>
           <p>You can bundle your dependency tree for production. If you go to the command line and run <code>npm run jspm bundle app --inject</code>, jspm distills the entire dependency graph down to a single file, build.js, based on the app.js entry point. You can then load everything in one fell swoop, instead of asynchronously as the system encounters the includes.</p>
           <p>At that point, you can include build.js instead of app.js, like so: <code>&lt;script src="build.js"&gt;&lt;/script&gt;</code>. You'll still need system.js and config.js, however.</p>
           <h2 id="N1022F" class="ibm-h2">Conclusion</h2>
           <p>The great thing about using jspm is that going forward, browsers will pick up support for the SystemJS syntax natively, and your applications will be ready. You can make use of the wide range of existing bundles, with a future-ready syntax, and take advantage of build tools like CSS inclusion and a dev server in the process. Although SystemJS might be largely replaced by the native browser support, the packaging and bundling aspects of jspm will remain useful and necessary.</p>
