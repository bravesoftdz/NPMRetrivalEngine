     <h1 class="content-title Tutorial-header"> <button type="button" class="Button--reset Tutorial-headerButton " data-toggle="tooltip" title="Mark as Complete" data-js="markAsComplete" data-tutorialinteraction data-tutorialsingle>

       </svg> </button> How To Use npm to Build and Publish Node.js Packages on a Linux Server </h1> 
     <span class="meta-section timestamp"><span class="tutorial-date-text">Posted</span><span class="tutorial-date">May 14, 2014</span></span> 
     <span class="meta-section pageviews"><span class="icon icon-eye v-mid"></span><span class="views-count v-mid">42.8k</span><span class="sr-only"> views</span></span> 
     <span class="meta-section tags"> <a class="tag" href="/community/tags/node-js?type=tutorials">Node.js</a> </span>

     <h3 id="introduction">Introduction</h3> 
     <p>Node.js is a popular Javascript platform for server-side programming that allows you to build and run web apps quickly.</p> 
     <p>In this guide, we will talk about how to use <code>npm</code>, a Node.js package management system. In a past guide, we discussed <a href="https://www.digitalocean.com/community/articles/how-to-install-node-js-on-an-ubuntu-14-04-server">how to install Node.js on an Ubuntu 14.04 server</a>. We have also discussed <a href="https://www.digitalocean.com/community/articles/how-to-use-npm-to-manage-node-js-packages-on-a-linux-server">how to use npm as a package consumer by searching for and acquiring packages</a>.</p> 
     <p>Now, we'll cover how to modify configuration settings, create and publish a package, and how to interact with the npm package repository. Let's jump right in.</p> 
     <h2 id="managing-configuration-settings">Managing Configuration Settings</h2> 
     <p>To control the behavior of npm, you can adjust the settings for the tool. There are quite a few ways built into npm to do this.</p> 
     <h3 id="listing-the-current-configuration-settings">Listing the Current Configuration Settings</h3> 
     <p>First, let's see what our current settings are. We can do this by typing:</p> 
     <pre class="code-pre "><code langs="">npm config ls
</code></pre> 
     <hr> 
     <pre class="code-pre "><code langs="">; cli configs
registry = "https://registry.npmjs.org/"
user-agent = "npm/1.4.9 node/v0.11.13 linux x64"

; node bin location = /home/demouser/.nvm/v0.11.13/bin/node
; cwd = /home/demouser/projects/test_package
; HOME = /home/demouser
; 'npm config ls -l' to show all defaults.
</code></pre> 
     <p>This list of settings is pulled from a variety of different places, including the current operating environment. It is built when the command is run. You can test this by passing the <code>-g</code> flag. It will suddenly show that the "global" option is set:</p> 
     <pre>
npm config ls -g
</pre> 
     <pre>
; cli configs
<span class="highlight">global = true</span>
registry = "https://registry.npmjs.org/"
user-agent = "npm/1.4.9 node/v0.11.13 linux x64"
. . .
</pre> 
     <p>Right now, npm is not pulling any information from its configuration files because there <em>are</em> no configuration files that have been created yet. It is simply gathering information from its environment (current directory, etc.) and using the application defaults.</p> 
     <p>We aren't shown most of the defaults with this command though. To get more extensive information, we'll have to pass the <code>-l</code> flag:</p> 
     <pre class="code-pre "><code langs="">npm config ls -l
</code></pre> 
     <hr> 
     <pre class="code-pre "><code langs="">; cli configs
long = true
registry = "https://registry.npmjs.org/"
user-agent = "npm/1.4.9 node/v0.11.13 linux x64"

; default values
always-auth = false
bin-links = true
browser = null
ca = null
. . .
</code></pre> 
     <h3 id="getting-and-setting-specific-values">Getting and Setting Specific Values</h3> 
     <p>As you can see, there are many key-value pairs of information. This is how we work with configuration values within npm. For instance, if we want to see the current value of the "editor" key, we can type:</p> 
     <pre class="code-pre "><code langs="">npm config get editor
</code></pre> 
     <hr> 
     <pre class="code-pre "><code langs="">vi
</code></pre> 
     <p>If we want to change this value, we can easily do that by typing something like this:</p> 
     <pre>
npm config set editor <span class="highlight">vim</span>
</pre> 
     <p>This will do two things. It will override the default value (vi) with our new value (vim), and it will <em>also</em> create our configuration file, since it is our first custom value.</p> 
     <p>Where is our configuration file written? Why don't we ask npm:</p> 
     <pre class="code-pre "><code langs="">npm config get userconfig
</code></pre> 
     <hr> 
     <pre class="code-pre "><code langs="">/home/demouser/.npmrc
</code></pre> 
     <p>If we set values using the <code>config set -g</code> syntax, they will be written to our <code>globalconfig</code> file. We can find out where this is stored by typing:</p> 
     <pre class="code-pre "><code langs="">npm config get globalconfig
</code></pre> 
     <hr> 
     <pre class="code-pre "><code langs="">/home/demouser/.nvm/v0.11.13/etc/npmrc
</code></pre> 
     <p>Your file location will differ depending on the method you used to install Node.js. Again, this file won't exist until you stray from the default values on the global level.</p> 
     <p>For the file that <em>does</em> exist, if you want to edit it with your editor, you can do that by typing:</p> 
     <pre class="code-pre "><code langs="">npm config edit
</code></pre> 
     <p>This will not work if the file hasn't already been created yet (manually or by setting a non-default value within npm).</p> 
     <p>You can also set different configuration options for a specific command as you are calling it by using this syntax:</p> 
     <pre>
npm <span class="highlight">subcommand</span> --<span class="highlight">key</span> <span class="highlight">value</span>
</pre> 
     <p>Other areas where values are pulled from are the <code>package.json</code> file for the project itself and environmental variables.</p> 
     <h2 id="creating-a-package-with-npm">Creating a Package with npm</h2> 
     <p>We've learned a bit about how to use npm as a standard package manager to acquire and manage dependencies and packages that your application will need. However, you can also use npm to help you package and share your application or library.</p> 
     <p>One of the first things that npm helps with is generating a package.json file for your project. Before we begin, you should take a look at the help page for this so that you are familiar with the fields that you will be asked about:</p> 
     <pre class="code-pre "><code langs="">npm help json
</code></pre> 
     <p>Once you are in your application directory, you can get started:</p> 
     <pre class="code-pre "><code langs="">cd ~/projects/test_package
npm init
</code></pre> 
     <p>This will prompt you for information that it needs to build a package file for you. It will take some cues from the environment to populate default values. Your session might look something like this:</p> 
     <pre class="code-pre "><code langs="">name: (test_package) 
version: (0.0.0) 0.0.1
description: A test package
entry point: (index.js) index.js
test command: echo "test"
git repository: http://fake.git.repo.com
keywords: test
author: me
license: (ISC) GLP
About to write to /home/demouser/projects/test_package/package.json:

{
  "name": "test_package",
  "version": "0.0.1",
  "description": "A test package",
  "main": "index.js",
  "dependencies": {
    "express": "^4.2.0"
  },
  "devDependencies": {},
  "scripts": {
    "test": "echo \"test\""
  },
  "keywords": [
    "test"
  ],
  "author": "me",
  "license": "GLP"
}

Is this ok? (yes) yes
</code></pre> 
     <p>If you have to install additional dependencies after you have generated your <code>package.json</code> file, you should install them with the <code>--save</code> flag, which will update the dependencies list in your <code>package.json</code> file.</p> 
     <p>If your package requires very specific versions, you can use the <code>shrinkwrap</code> subcommand. Usually, the package file sets your current packages as the minimum viable versions that are needed to build. With a shrinkwrap file, these versions are specific and locked down.</p> 
     <p>You can create this file with:</p> 
     <pre class="code-pre "><code langs="">npm shrinkwrap
</code></pre> 
     <p>This will generate a file called <code>npm-shrinkwrap.json</code> with very detailed package information for each dependency and in a recursive manner. If your application environment is delicate, this is an essential tool.</p> 
     <h2 id="interacting-with-the-npm-website-through-npm">Interacting with the npm Website through npm</h2> 
     <p>The npm project maintains a website of packages at <a href="https://www.npmjs.org">npmjs.org</a>. You can actually interact with this site through the npm command line tool. We've already discussed how you can search for packages, but there is more functionality as well.</p> 
     <p>First, if you haven't signed up with an account yet, you can do so from <em>within</em> the npm interface by typing:</p> 
     <pre class="code-pre "><code langs="">npm adduser
</code></pre> 
     <p>You will be asked to select a username, a password, and an email address. These will be posted to the site, just as if you were registering through the web browser.</p> 
     <p>If you check your configuration, you should see your new account details:</p> 
     <pre class="code-pre "><code langs="">npm config ls
</code></pre> 
     <hr> 
     <pre class="code-pre "><code langs="">. . .
; userconfig /home/demouser/.npmrc
editor = "vim"
email = "demouser@domain.com"
username = "demouser"
. . .
</code></pre> 
     <h3 id="publish-your-package">Publish your Package</h3> 
     <p>When you have created a package that you wish to share with the greater Node.js community, the npm site is a great place to start.</p> 
     <p>Fortunately, npm comes with some helper functions that will get you started.</p> 
     <p>If you are in your package's directory, you can simply type:</p> 
     <pre class="code-pre "><code langs="">npm publish
</code></pre> 
     <p>This will push your package onto the npm website. The success of this operation is dependent on your package having a unique name and version number. If you go there and search for your package, it should be listed just like any other Node.js package.</p> 
     <h3 id="tagging-your-uploads">Tagging your Uploads</h3> 
     <p>You can tag specific releases of your packages by using the <code>tag</code> subcommand. These can be used in place of version numbers during installation and dependency specifications.</p> 
     <p>By default, npm will include a tag labeled "latest" to any package you publish. This just marks the most recent push. You can see this by typing:</p> 
     <pre class="code-pre "><code langs="">npm config get tag
</code></pre> 
     <hr> 
     <pre class="code-pre "><code langs="">latest
</code></pre> 
     <p>This means that you can specifically request the latest version by typing things like:</p> 
     <pre class="code-pre "><code langs="">npm install package@latest
</code></pre> 
     <p>If you want to add an additional tag to one of your package/version combinations, you can type something like:</p> 
     <pre>
npm tag <span class="highlight">package</span>@<span class="highlight">version</span> <span class="highlight">tag_name</span>
</pre> 
     <p>This will let you now reference the version you tagged by the "tag_name" instead of the version number.</p> 
     <h3 id="altering-ownership-of-packages">Altering Ownership of Packages</h3> 
     <p>By default, when you publish a package, you will be given ownership of the package.</p> 
     <p>You can verify this by typing:</p> 
     <pre>
npm owner ls <span class="highlight">package_name</span>
</pre> 
     <pre>
demouser &lt;demouser@domain.com&gt;
</pre> 
     <p>If you want to add additional people to the project, which will allow them to make modifications, you can issue a command like this:</p> 
     <pre>
npm owner add <span class="highlight">username</span>
</pre> 
     <p>You can also do the reverse of this and remove rights from users like this:</p> 
     <pre>
npm owner rm <span class="highlight">username</span>
</pre> 
     <p>At the time of this writing, all owners for a package have the same rights, so it is not possible to assign more fine-grained access control.</p> 
     <h3 id="removing-and-deprecating-packages">Removing and Deprecating Packages</h3> 
     <p>Sometimes, there is an important change in your package that make it necessary to deprecate older versions. For example, maybe there was a security problem.</p> 
     <p>With npm, you can easily deprecate a version or a version range of your package by typing something like this:</p> 
     <pre>
npm deprecate <span class="highlight">package_name</span>@"<span class="highlight">version_or_range</span>deprecation_message_to_users"
</pre> 
     <p>The message that you choose to include will be shown as a warning to any user installing the package versions you've marked.</p> 
     <p>If you want to take your package down entirely, you can do that by unpublishing it. You should probably not do this if your package has been used by other people and npm encourages you to deprecate the package instead. You can also unpublish a specific version, but the same suggestion holds.</p> 
     <p>An important thing to note is that even if you unpublish a package and it is removed from the site, the name/version combinations it held cannot be reused ever. This is to ensure that people are not accidentally pulling the wrong packages.</p> 
     <p>The syntax for unpublishing your work is:</p> 
     <pre>
npm unpublish <span class="highlight">package_name</span>@<span class="highlight">version</span>
</pre> 
     <p>You can leave off the <a href="https://www.digitalocean.com/community/users/version" class="username-tag">@version</a> if you want to remove the entire package. In this case, you'll have to add the <code>--force</code> flag so avoid accidental deletion of your entire project:</p> 
     <pre>
npm unpublish --force <span class="highlight">package_name</span>
</pre> 
     <h2 id="conclusion">Conclusion</h2> 
     <p>You should now have a fairly in-depth grasp of how to leverage the power of npm in your Node.js projects. The npm utility can be used to assist in most of the areas of project development, from building files and acquiring dependencies, to publishing packages and deploying into production.</p>
