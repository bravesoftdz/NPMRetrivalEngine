       <h2>Brick</h2>
       <div class="markdown_content">
        <div class="anchor-navigation"></div> 
        <p>Brick is Paymentwall's premiere credit card solution available for businesses in 200+ countries.</p> 
        <h2>Getting Started</h2> 
        <h3>API Credentials</h3> 
        <p>Paymentwall provides you with the following API credentials:</p> 
        <ul> 
         <li>Public API Key - used for building the Frontend credit card form. This can be publicly available to your end-users.</li> 
         <li>Private API Key - used for charging cards from your backend. Keep this private at all times.</li> 
        </ul> 
        <p>A pair of test API keys can be retrieved at the <a href="https://www.paymentwall.com/en/documentation/getting-started">Getting Started</a> page or by Signing Up.</p> 
        <h3>Data Flow</h3> 
        <p><a href="https://api.paymentwall.com/content/design/files/developers/documentation/apis/brick/BrickFlow.png">Link to data flow chart</a>.</p> 
        <h3>Step 1: Frontend</h3> 
        <p>To charge a credit card, you will first need to collect the credit card details from the end-user and pass them to your backend. Brick helps you to do this in a PCI DSS compliant manner by creating a one-time token that you can further use to charge the card on Step 2.<br> Please note that payment pages are <strong>required</strong> to be loaded via <strong>https://</strong> (At least TLSv1.1 is recommended).<br> The code sample below will build the credit card form (or add event listeners to your own custom form) and will tokenize the credit card data once the form is submitted and then will send the token to your backend controller specified by <em>action</em> parameter. More details about the client-side library for tokenization can be found in <a href="#brickjs">Brick.js documentation</a>.</p> 
        <p>If you're getting error "<strong>Card tokenization failed</strong>" - make sure you are testing on <strong>https</strong>. Known limitation in IE9 and IE10 - the request would not go through if you're testing on http and not https. But in any case this is the security requirement for going live, but this issue is known to affect testing on development and staging environment.</p>

        <h2>Brick.js<span id="brickjs"></span></h2> 
        <p>Brick.js is a powerful JavaScript library for building Brick standard credit card forms, tokenizing credit card data, adding submit listeners for tokenizing your custom credit card forms, as well as generating fingerprints for risk scoring.</p> 
        <p><strong>Brick</strong>(<em>options</em>[, <em>type</em>])<br> Initializes Brick.js</p> 
        <ul> 
         <li><em>options</em> - required - object - list of options for Brick.js: 
          <ul> 
           <li>public_key - required - Your project's Public Key</li> 
           <li>container - required - ID of the container where payment form will be placed</li> 
           <li>amount - required - Payment amount</li> 
           <li>currency - required - Payment currency</li> 
           <li>action - required - Pyment form action URL</li> 
           <li>form - optional - Payment form configuration object 
            <ul> 
             <li>id - optional - ID of the pyment form. Used for <a href="#step-1-frontend">one-line tokenization</a> of custom forms</li> 
             <li>merchant - Your company name</li> 
             <li>product - Your product name</li> 
             <li>pay_button - Text for submit button</li> 
             <li>lang - optional - ISO alpha-2 language code to localize the payment form into</li> 
             <li>zip - If set to true, zipcode field will be visible. Result is submitted in brick_zip POST parameter. Can be further passed as customer[zip] into the Charge API for AVS checks</li> 
            </ul></li> 
           <li>onTokenized(<em>data</em>) - optional - Callback after custom payment form is tokenized and hidden inputs <em>brick_token</em> and <em>brick_fingerprint</em> are added to the form. Used for <a href="#step-1-frontend">one-line tokenization</a> of custom forms. 
            <ul> 
             <li>data - optional - object - Data format is similar to response from <a href="#one_time_token">Tokenization</a> API.</li> 
            </ul></li> 
           <li>onError(<em>errors</em>) - optional - Callback for errors that occurred during one-line tokenization. 
            <ul> 
             <li>errors - optional - object - Returns error response object. For example, {"success": "0", "error": {"code": "3004", "message": "Expiration month is invalid"}}</li> 
            </ul></li> 
          </ul></li> 
         <li><em>type</em> - optional - payment form type. Possible value: <em>default</em>, <em>custom</em></li> 
        </ul> 
        <p><strong>Brick.showPaymentForm</strong>(<em>onSuccess</em>, <em>onError</em>)</p> 
        <ul> 
         <li><em>onSuccess</em>(<em>data</em>) - required - object - Callback on successful payment 
          <ul> 
           <li>data - optional - object - Returns object. For example, {"success": "1"}</li> 
          </ul></li> 
         <li><em>onError</em>(<em>errors</em>) - required - Callback when something went wrong 
          <ul> 
           <li>errors - optional - object - Returns error response object. For example, {"success": "0", "error": {"code": "3004", "message": "Expiration month is invalid"}}</li> 
          </ul></li> 
        </ul> 
        <p><strong>Brick.tokenizeCard</strong>(<em>data</em>, <em>callback</em>)</p> 
        <ul> 
         <li><em>data</em> - required - Tokenization data. Possible values: HTMLElement | jQuery object | JavaScript Object // HTMLElement 
          <ul> 
           <li>document.getElementById('my-payment-form') // JavaScript object</li> 
           <li>card_number - Card number</li> 
           <li>card_expiration - Card expiration date. Format (MM / YY), for example: "10 / 22"</li> 
           <li>card_expiration_month - Card expiration month</li> 
           <li>card_expiration_year - Card expiration year</li> 
           <li>card_cvv - Card CVV</li> 
          </ul></li> 
         <li><em>callback</em>(<em>data</em>) - required - Callback returning tokenization data on success. 
          <ul> 
           <li>data - optional - object - Data format is similar to response from <a href="#one_time_token">Tokenization</a> API</li> 
          </ul></li> 
        </ul> 
        <p><strong>Brick.localize</strong>(<em>form</em>, <em>language</em>, <em>options</em>)<br> Localizes a custom payment form.</p> 
        <ul> 
         <li><em>form</em> - required - Payment form HTML Element / jQuery element</li> 
         <li><em>language</em> - optional - string - Language used for translations. Available options: 'de', 'ja', 'fr'. English is set by default.</li> 
         <li><em>options</em> - optional - object - Config for translations. Format: {element: 'label|span|div', position: 'adjacent|inner'} </li> 
        </ul> 
        <p>Expected response types from your backend script:</p> 
        <ul> 
         <li><code>{"success":1}</code> - for success</li> 
         <li><code>{"success":"0","error":{"code":"3004","message":"Expiration month is invalid"}}</code> - for errors</li> 
         <li><code>{"success":"0","secure":"..."}</code> - for 3dsecure scenario</li> 
        </ul> 
        <h2>API Reference</h2> 
        <h2>Authentication</h2> 
        <h3>HTTP Header</h3> 
        <p>All API endpoints except One-time Token expect the Private API key to be sent as custom HTTP header "X-ApiKey". For example, "X-ApiKey: YOUR_PRIVATE_KEY". The API libraries supporting Brick automatically add this API header. </p> 
        <p>Please note that the Private API key can only be used for server-to-server calls and should never be exposed to end-users. For this reason One-time Token API accepts Public API key via <em>public_key</em> parameter instead of the Private API key header. Paymentwall is also expecting header Content-Type: application/x-www-form-urlencoded <a href="https://github.com/paymentwall">Client-side libraries</a> supporting Brick automatically add X-ApiKey and Content-Type into API requests.</p>

        <h3>3D Secure</h3> 
        <p>If 3D Secure step is required, the following response is returned as a response to create Charge request. User then needs to be presented with an HTML page where this form is automatically submitted, so that he can complete the 3D Secure step. Developers need to embed this formHTML into the page's HTML. The form will submit itself.</p> 
        <pre><code class="language-json">{
  "secure":{"formHTML":"..."}
}</code></pre> 
        <p>After completing the 3D Secure step, the user will be redirected back to the website with parameters brick_charge_id and brick_secure_token that need to be appended to the request parameters of the original Charge request. <a href="https://api.paymentwall.com/content/design/files/developers/documentation/apis/brick/BrickFlow3ds.png">Brick 3D Secure data flow diagram</a>.</p> 
        <h4>Default Form</h4> 
        <p>If you're using default brick.js form, it takes care of this step itself and re-submits the request to your backend script. </p> 
        <h4>Custom Form / Redirect Flow</h4> 
        <p>You should pass secure_redirect_url in the Charge request where the user will be redirected after completing the 3D Secure step and where parameters brick_secure_token and brick_charge_id will be passed via POST. Once you get the 3D Secure form HTML, you can submit the form via something like the following code in PHP.</p> 
        <pre><code class="language-php">echo '&lt;div id="3ds_form_container"&gt;' . $responseData['secure']['formHTML'] . '&lt;/div&gt;&lt;script&gt;document.getElementById("3ds_form_container").getElementsByTagName("form")[0].submit();&lt;/script&gt;';</code></pre> 
        <p>Once the user completes the step, he will be redirected back to secure_redirect_url from where you can complete the Charge. brick_fingerprint and brick_token of the original request should be embedded into secure_redirect_url along with the order ID, so that they can be subsequently passed into the Charge.</p> 
        <h4>Custom Form / Popup Flow</h4> 
        <p>To open the 3D Secure form, you can open a popup via <code>window.open</code> and submit the 3D Secure form into that popup via <code>target</code> attribute of the form. Capturing the 3D Secure step completion You can , and catch the 3D Secure step completion into the original via <code>window.postMessage</code> using the following message format: <code>{"event":"3dSecureComplete","data":{"secure_token":"...","charge_id":"..."}}</code>.</p> 
        <h3>Preventing Fraud</h3> 
        <p>Paymentwall has an in-house risk scoring system assesing the risk level of every transaction in realtime based on various such as velocity metrics, geolocation, transaction history, engagement, device information. Paymentwall also has a 24/7 in-house team of risk analysts that perform agent review of transactions with high risk score on behalf of merchants. To ensure that Paymentwall's risk engine has all the necessary data to score transactions accurately, please ensure to pass as much <a href="https://www.paymentwall.com/en/documentation/User-Profile-API/3125">User Profile API</a> information as possible along with every Charge or Subscription API creation request.</p> 
        <p>In case a payment is currently under risk review by Paymentwall, the following fields will be returned in the charge object among the others:</p> 
        <pre><code class="language-json">{
  ...
  "risk":"pending",
  "captured":false,
  ...
}</code></pre> 
        <p>At the same time, pingback with type=200 is sent to your server letting you know that the payment is pending risk review. Once the risk review is over (which normally takes only a few minutes), Paymentwall will send a <a href="https://www.paymentwall.com/en/documentation/Digital-Goods-API/710#pingback_types_risk">Pingback</a> with the respective decision: type=201 if the payment is accepted; type=202 if the payment is declined.<br> You can use <a href="#step-3-testing">Test CVV numbers</a> 555 and 556 to emulate accepted and declined after risk review scenarios, respectively.</p> 
        <h3>Storing a card after a Charge</h3> 
        <p>If you want to store the credit card for future use, you can store the permanent token (<em>card.token</em> field of <a href="#charge_object">charge response object</a>) and use it for a new charge later on. For this you can use the following example utilizing <a href="https://github.com/paymentwall/paymentwall-php">Paymentwall PHP library</a>:</p> 
        <pre><code class="language-php">// Library initialization section
$charge = new Paymentwall_Charge(/* charge details */);
if ($charge-&gt;isSuccessful()) {
   store($charge-&gt;card-&gt;token);
}</code></pre> 
        <p>Please note that during <a href="#charge_create">charge</a> you can pass <em>options[capture]=0</em>, thus only creating a temporary authorization without charging the funds from the user's card. You can then <a href="#charge_void">void</a> the charge and return funds to the user.<br> You can pass amount=0 for zero authorization so that the user is not charged, however this may be declined by some issuing banks depending on where your business entity is registered. Please contact your account manager for the possibility of activating zero authorization. </p>
