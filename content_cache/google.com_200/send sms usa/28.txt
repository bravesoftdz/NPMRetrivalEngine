        <h1 id="overview">Overview</h1> 
        <p>The Sendwithus API is based on the REST paradigm, and thus features resource based URLs with standard HTTP response codes to indicate errors. We use standard HTTP authentication and request verbs, and all responses are JSON formatted.</p> 
        <p>To make the integration process as easy as possible, we include test API keys with all accounts. Any requests made with test API keys will handle data and give responses, but not actually send any emails.</p> 
        <h3 id="baseurl">Base URL</h3> 
        <p>All API URLs referenced in this document are relative to the following base URL:</p> 
        <pre><code class="language-sh">https://api.sendwithus.com/api/v1
</code></pre> 
       </div> 
       <hr> 
       <div> 
        <h1 id="authentication">Authentication</h1> 
        <p>The Sendwithus API uses <a href="http://en.wikipedia.org/wiki/Basic_access_authentication">HTTP Basic Authentication</a>.</p> 
        <p>All requests require you to authenticate using a Sendwithus API Key as the username with an empty string as the password:</p> 
        <pre><code class="language-sh">curl -u live_1234qwerzxcv7890: https://api.sendwithus.com/api/v1
</code></pre> 
        <p>All API calls must use HTTPS. Any calls made using HTTP will return an appropriate error code.</p> 
       </div> 
       <hr> 
       <div> 
        <h1 id="errors">Errors</h1> 
        <p>Any API request will receive a standard HTTP response code.</p> 
        <h4 id="responseranges:">Response Ranges:</h4> 
        <ul> 
         <li>2xx -- Successful Request</li> 
         <li>4xx -- Failed Request (Client error)</li> 
         <li>5xx -- Failed Request (Server error)</li> 
        </ul> 
       </div>

        <h1 id="versioning">Versioning</h1> 
        <p>Any time we make backwards-incompatible changes to our API, we will release a new version. Our current API is v1, with v2 currently under development.</p>

}
</code></pre>

        <pre><code class="language-json">{

        <h1 id="sendapi">Send API</h1> 
        <p><em>NOTE</em> -- All parameters are mandatory unless otherwise noted.</p> 
        <p><em>NOTE</em> -- If a customer does not exist by the specified email, this call will create a customer.</p> 
        <h2 id="sendanemail">Send an Email</h2> 
        <p><code>POST /send</code> </p>
        <h4 id="params:">Params:</h4> 
        <ul> 
         <li>template -- Template ID to send</li> 
         <li>recipient 
          <ul> 
           <li>address -- The recipient's email address</li> 
           <li>name (optional) -- The recipient's name</li> 
          </ul></li> 
         <li>cc (optional) -- An array of CC recipients, of the format {"address":"cc@email.com"}</li> 
         <li>bcc (optional) -- An array of BCC recipients, of the format {"address":"bcc@email.com"}</li> 
         <li>sender (optional) 
          <ul> 
           <li>address -- The sender's email address</li> 
           <li>reply_to -- The sender's reply-to address</li> 
           <li>name -- The sender's name</li> 
          </ul></li> 
         <li>template_data (optional) -- Object containing email template data (maximum 128KB)</li> 
         <li>tags (optional) -- Array of tags (as strings). <em>Tags are passed to your ESP as Categories, Tags, etc.</em></li> 
         <li>headers (optional) -- Object contain SMTP headers to be included with the email</li> 
         <li>inline (optional) -- Inline attachment object (see example)</li> 
         <li>files (optional) -- List of file attachments (combined maximum 7MB, see example)</li> 
         <li>esp_account (optional) -- ID of the ESP Account to send this email through. ex: esp_1a2b3c4d5e</li> 
         <li>locale (optional) -- Template locale to send (ie: en-US)</li> 
         <li>version_name (optional) -- Name of the template version to send (overrides A/B tests)</li> 
        </ul> 
        <h4 id="samplerequest:">Sample Request:</h4> 
        <pre><code class="language-json">{
    "template": "tem_A5RHVP6CnRbS34UysLjYHx",
    "recipient": {
        "name": "John",
        "address": "user@email.com"
    },
    "template_data": { "amount": "$12.00" },
    "cc": [
        {"address": "cc_one@email.com"},
        {"address": "cc_two@email.com"}
    ],
    "bcc": [
        {"address": "bcc_one@email.com"},
        {"address": "bcc_two@email.com"}
    ],
    "sender": {
        "name": "Company",
        "address": "company@company.com",
        "reply_to": "info@company.com"
    },
    "tags": [
        "tag1",
        "tag2",
        "tag3"
    ],
    "headers": {
        "X-HEADER-ONE": "header-value"
    },
    "inline": {
        "id": "cat.png",
        "data": "{BASE_64_ENCODED_FILE_DATA}"
    },
    "files": [
        {
            "id": "doc.txt",
            "data": "{BASE_64_ENCODED_FILE_DATA}"
        },

        <h1 id="internationalizationi18napi">Internationalization (i18n) API</h1> 
        <h3 id="howitworks">How it works</h3> 
        <p>Sendwithus handles internationalization by using <em>Jinja2</em> style trans blocks.</p> 
        <p>The easiest way to set up templates for translation is to use the trans block syntax. A string can be marked for translation by simply wrapping it in a trans block. For example, the internationalized version of "Hello World" is "{% trans %}Hello World{% endtrans %}".</p> 
        <p>Variables in trans block can be added like normal template variables. For example "Hello {{ name }}" will become "{% trans %}Hello {{ name }}{% endtrans %}". Multiple variables can be used in the same trans block.</p> 
        <p>Translation packages (.pot files) can be downloaded via API, and then corresponding string files (.po) can be uploaded via API, which will trigger translated template variants to be generated.</p> 
        <h2 id="gettranslationpackage.potfile">Get translation package (.pot file)</h2> 
        <p>This call will fetch a .pot translation package for all templates that have been assigned the corresponding tag.</p> 
        <p><code>GET /i18n/pot/(:tag)</code> </p>
        <h4 id="params:">Params:</h4> 
        <ul> 
         <li>tag -- String will return a .pot file only for templates that have the corresponding tag. Tags are set in the <a href="https://www.sendwithus.com/#/emails">dashboard</a></li> 
        </ul> 
        <h3 id="examples">Examples</h3> 
        <p>Fetch .pot for template with tag "international".</p> 
        <p><code>GET /i18n/pot/international</code> </p>
        <h4 id="sampleresponse:">Sample Response:</h4> 
        <p>The reponse is a valid .pot file.</p> 
        <pre><code class="language-sh">  msgid "Welcome to Sendwithus, %(first_name)s!"
  msgstr ""
    ?
  msgid "Confirm your email"
  msgstr ""
</code></pre> 
        <h2 id="posttranslatedstringsand.pofiles">Post translated strings and .po files</h2>

        <h2 id="sendingtranslatedtemplates">Sending translated templates</h2> 
        <p>At send time, Sendwithus uses the <em>locale</em> parameter in the Send API, and matches it against the template version for that locale. This is why .po files must have a filename that matches the locale.</p> 
        <p>For an example sending with a translated template, please see the <a href="https://www.sendwithus.com/docs/api#send">send api</a> documentation.</p> 
        <h2 id="localecodestandards">Locale Code Standards</h2> 
        <p>We follow the IETF Language Tag standard which consists of a 2 character language code and a 2 character country code separated by a hyphen. Example: 'en-US' is the code for English (United States)</p> 
        <p>IETF Language Tags: <a href="http://en.wikipedia.org/wiki/IETF_language_tag">http://en.wikipedia.org/wiki/IETF_language_tag</a></p> 
        <p>The prefix, in this case 'en', is a language code following the ISO 639-1 standard.</p> 
        <p>ISO 639-1: <a href="http://en.wikipedia.org/wiki/List_of_ISO_639-1_codes">http://en.wikipedia.org/wiki/List_of_ISO_639-1_codes</a></p> 
        <p>The suffix, in this case 'US', is a country code following the ISO 3166-1 Alpha-2 standard.</p> 
        <p>ISO 3166-1 Alpha-2: <a href="http://en.wikipedia.org/wiki/ISO_3166-1_alpha-2">http://en.wikipedia.org/wiki/ISO_3166-1_alpha-2</a></p>
