         <p>When you’re collecting data from users, there are two key challenges; collecting that information, and validating it. Some types of information are straightforward – someone’s age, for example, couldn’t really be simpler to collect and to validate. Names aren’t as straightforward as they sound, but provided you cater for edge cases and international variations – for example patronymics, the mononymous, or even just people with hyphenated surnames – you can’t go too far wrong (although plenty of applications and services do!). Email addresses, while theoretically very easy to validate, have their own challenges – yet nevertheless, there are plenty of regular expressions in the wild that aren’t quite right.</p> 
         <p>And then there are telephone numbers. These are hard. Really hard. In this article I’ll discuss some of the challenges around collecting, validating, and displaying telephone numbers.</p> 
         <h2 id="why-telephone-numbers-are-different">Why Telephone Numbers are Different</h2> 
         <p>Perhaps you’re thinking that since telephone numbers tend to follow a pretty rigid format, such as this:</p> 
         <pre><code class="language-">202-456-1111</code></pre> 
         <p>…that it ought to be simple to construct a simple regular expression to validate them. In fact, here’s one:</p> 
         <pre><code class="language-">^(\([0-9]{3}\)|[0-9]{3}-)[0-9]{3}-[0-9]{4}$</code></pre> 
         <p>Well, stop right there. For starters, here are just some variations of the number above, all of which are perfectly valid:</p> 
         <pre><code class="language-">202 456 1111
(202) 456 1111
2024561111
1-202-456-1111
1-202-456-1111 x1234
1-202-456-1111 ext1234
1 (202) 456-1111
1.202.456.1111
1/202/456/1111
12024561111
+1 202 456 1111</code></pre> 
         <p>So based on that, we know that the regular expression apparoach isn’t as simple as we first thought – but that’s only the half of it. These examples are just for a US-based number. Sure, if you know that the number you’re collecting is going to be for a specific country, you may be able to use a regular expression. Otherwise, this approach won’t cut it.</p> 
         <p>Let’s look at some of the other issues around telephone numbers, and why they make our job even harder.</p> 
         <h3 id="numbers-change">Numbers Change</h3> 
         <p>All sorts of external factors can have implications for telephone numbering. Whole countries come and go, introducing new country prefixes. New classifications of numbers introduce new numbering systems – premium-rate, local-rate, toll-free, and so on. When a carrier runs out of one set of numbers – like, sadly, premium-rate – they simply introduce a new prefix. </p> 
         <p>Some changes have enormous implications; in the United Kingdom some years ago, for example, the entire regional numbering system underwent a drastic change, with virtually every area code getting an additional “1” inserted. Even then, the capital had a subtly different system. It was probably a decade before signage was changed across the country to reflect the changes. </p> 
         <p>Then, of course, there was the enormous and unprecedented growth in mobile. No longer was the number of telephone numbers required largely limited to the number of households, but many times over. The continued strain on the pool of available numbers can only increase the likelihood of further changes.</p> 
         <h3 id="international-dialing-codes">International Dialing Codes</h3> 
         <p>It’s often important to capture a number’s international dialing code. In some cases, the context might mean they aren’t required. For example if you operate in a single country, and telephone numbers are captured to be used by a human operator, you might not need them. But for anything remotely automated – such as sending SMS messages – or to validate them effectively, you’ll need to capture the country prefix.</p> 
         <p>The <a href="https://github.com/mledoze/countries">countries</a> library contains a bunch of geographical information which includes international dialing codes. Here is an excerpt from <code>countries.json</code> from that library:</p> 
         <pre><code class="language-">{
    "name": {
        "common": "Austria",
        "official": "Republic of Austria",
        // ... //
    },
    // ... //
    "callingCode": ["43"],
    // ... //
},</code></pre> 
         <p>As you can see, this demonstrates that Austria uses the international dialing code 43.</p> 
         <p>So how might we use this information? Well, using the magic of Lodash (or Underscore), there are a few ways in which we can query dialing code-related information.</p> 
         <p>For example, to find out whether a given dialing code is valid:</p> 
         <pre><code class="language-javascript">var _ = require('lodash')
    , data = require('world-countries')

module.exports = {
    /**
     * Determines whether a given international dialing code is valid
     * 
     * @param  string  code
     * @return bool
     */
    isValid : function(code) {

        var codes = _.flatten(_.pluck(data, 'callingCode'));

        return _.contains(codes, code);

    }
    // ...
}</code></pre> 
         <p>There are a more efficent ways of doing this, of course, so this and the following examples aren’t necessarily optimized for production.</p> 
         <p>We can look up the countries which use a particular dialing code:</p> 
         <pre><code class="language-javascript">/**
 * Gets a list of countries with the specified dialing code
 * 
 * @param  string   code
 * @return array        An array of two-character country codes
 */
getCountries : function(code) {

    var countryEntries = _.filter(data, function(country){
        return (_.contains(country.callingCode, code));
    })

    return _.pluck(countryEntries, 'cca2');

}</code></pre> 
         <p>Finally, we can get the dialing codes for given country:</p> 
         <pre><code class="language-javascript">/**
 * Gets the dialing codes for a given country
 * 
 * @param  string   country     The two-character country code
 * @return array        An array of strings representing the dialing codes
 */
getCodes : function(country) {
        
    // Get the country entry
    var countryData = _.find(data, function(entry) {
        return (entry.cca2 == country);
    });

    // Return the code(s)
    return countryData.callingCode;

}</code></pre> 
         <p>You’ll find these functions packaged up as a module, along with unit tests, in <a href="https://github.com/jsprodotcom/telephones-js">the repository that accompanies the article</a>.</p> 
         <p>Even international dialing codes, however, aren’t as straightforward as you may think. The format can vary – 1, 43, 962 1868 are all valid codes. There isn’t necessarily a one-to-one mapping; 44 for example, is used not just for the United Kingdom but for the Isle of Man, Guernsey and Jersey.</p> 
         <p>Numbers must also be altered according to where you’re dialing from. From abroad, to call a UK number you need to drop the leading zero and prefix with the dialing code 44:</p> 
         <pre><code class="language-">020 7925 0918</code></pre> 
         <p>…becomes…</p> 
         <pre><code class="language-">+44 20 7925 0918</code></pre> 
         <p>You can also replace the “+” with a double zero:</p> 
         <pre><code class="language-">0044 20 7925 0918</code></pre> 
         <p>To complicate things even further, some numbers vary when called from outside of a country depending on which country you’re dialing <em>from</em>. In the US, for example, numbers must also be prefixed with the US exit code <code>011</code>, so the example above becomes:</p> 
         <pre><code class="language-">011 44 20 7925 0918</code></pre> 
         <p>Thankfully, there is a format we can use which enable us to get around these variations.</p> 
         <h2 id="e164">E.164</h2> 
         <p>Luckily for developers there is an unambiguous, internationally recognized standard for telephone numbers anywhere in the World called E.164. The format is broken down as follows:</p> 
         <ul> 
          <li>A telephone number can have a maximum of 15 digits</li> 
          <li>The first part of the telephone number is the country code</li> 
          <li>The second part is the national destination code (NDC)</li> 
          <li>The last part is the subscriber number (SN)</li> 
          <li>The NDC and SN together are collectively called the national (significant) number</li> 
         </ul> 
         <p>(<a href="http://electronics.howstuffworks.com/telephone-country-codes3.htm"><em>source</em></a>)</p> 
         <p>Here’s the number from earlier, in E.164 format:</p> 
         <pre><code class="language-">+12024561111</code></pre> 
         <p>We can use the same format for, as an example, a London-based UK number:</p> 
         <pre><code class="language-">+442079250918</code></pre> 
         <p>We can represent any valid telephone number using the E.164 format. We know what country it refers to, and it’s unabmiguous – making it the ideal choice for storage. It’s also commonly used for telephony based services such as SMS providers, as we’ll see a little later.</p> 
         <p>There’s a catch, of course. The E.164 standard might be great for storage, but terrible for two things. First, virtually no one would type or read out their number in that format. Second, it’s hopeless in terms of its readability. Later though, when we look at <code>libphonenumber</code>, we’ll see that there are ways of formatting numbers for humans.</p> 
         <h2 id="collecting-telephone-numbers">Collecting Telephone Numbers</h2> 
         <p>First though, let’s look at the issue of collecting telephone numbers.</p> 
         <h3 id="html5-and-the-tel-input">HTML5 and the “tel” input</h3> 
         <p>HTML5 introduced a new “tel” input type. However, because of the issues around the variations in format, it doesn’t actually place any restrictions on what the user can type, nor does it perform any validation in the same way as, say, the email element. Nevertheless, there are some advantages – when used on a mobile site a user’s telephone keypad will usually be displayed, rather than a conventional keyboard layout.</p> 
         <p>You can use a single element to collect a number:</p> 
         <pre><code class="language-markup">&lt;input type="tel" name="number"&gt;</code></pre> 
         <p>Alternatively, you can break a number down into separate elements:</p> 
         <pre><code class="language-markup">&lt;!-- area code and number --&gt;
&lt;input type="tel" name="number"&gt;

&lt;!-- country code, area code and number --&gt;
&lt;input type="tel" name="country" size="4"&gt; &lt;input type="tel" name="area" size="6"&gt; &lt;input type="tel" name="number" size="8"&gt;

&lt;!-- US-style --&gt;
(&lt;input type="tel" size="3"&gt;) &lt;input type="tel" size="3"&gt; - &lt;input type="tel" size="4"&gt;</code></pre> 
         <p>Browser support is pretty good (e.g. Chrome 6+, Firefox 4+, Safari 5+, IE 10+), but even in an older browser it will simply fall back to a plain old text field.</p> 
         <p>Should we decide that a regular expression is sufficient – and remember, there are issues – then we can use the <code>pattern</code> attribute to add some validation:</p> 
         <pre><code class="language-markup">&lt;input type="tel" name="number" pattern="^(?:\(\d{3}\)|\d{3})[- ]?\d{3}[- ]?\d{4}$"&gt;</code></pre> 
         <h3 id="masked-inputs">Masked Inputs</h3> 
         <p>Masked inputs are a common technique for restricting user input or providing hints as to the expected format. But again, unless you can be confident that numbers will always be for a particular country, it’s very difficult to cater to international variations. However, it’s one thing to annoy users by making assumptions – asking a non-US user to provide a state and a zip-code. It’s quite another to make a form completely unusable, for example by forcing people to provide numbers in a certain country’s format.</p> 
         <p>Nevertheless, they can be effective if you know that certain numbers will be within a particular range. Here is an example of a <a href="https://jsfiddle.net/dKRGE/3/">masked input</a> for US telephone numbers.</p> 
         <h3 id="a-better-way">A Better Way</h3> 
         <p>There is a better and more flexible way to collect telephone numbers, in the form of an excellent <a href="https://github.com/Bluefieldscom/intl-tel-input">jQuery plugin</a>. It’s illustrated below.</p> 
         <p><img src="https://dab1nmslvvntp.cloudfront.net/wp-content/uploads/2014/10/141369377668747470733a2f2f7261772e6769746875622e636f6d2f426c75656669656c6473636f6d2f696e746c2d74656c2d696e7075742f6d61737465722f73637265656e73686f742e706e67.png" alt="The plugin in action" width="439" height="245" class="alignnone size-full wp-image-92839"></p> 
         <p>You can also play with a live demo <a href="http://jackocnr.com/intl-tel-input.html">here</a>.</p> 
         <p>Usage is simple – make sure you’ve included jQuery, the library, and the CSS file, and that the flag sprite is available and properly referenced from the CSS – you’ll find it in <code>build/img/flags.png</code>.</p> 
         <p>Next, create an element:</p> 
         <pre><code class="language-markup">&lt;input type="tel" id="number"&gt;</code></pre> 
         <p>Finally, intialize it as follows:</p> 
         <pre><code class="language-javascript">$("#number").intlTelInput();</code></pre> 
         <p>For a full list of configuration options, <a href="https://github.com/Bluefieldscom/intl-tel-input">consult the documentation</a>. Later, we’ll look at the <code>utilsScript</code> option, but first, we need to delve into another useful library.</p> 
         <h2 id="introducing-libphonenumber">Introducing libphonenumber</h2> 
         <p>Luckily, there’s a solution to many of our validation and formatting woes. Originally developed for the Android operating system, Google’s libphonenumber library offers all sorts of methods and utilities for working with telephone numbers. Better still, it’s been ported from Java to Javascript, so we can use it in web or Node.js applications.</p> 
         <h3 id="installation">Installation</h3> 
         <p>You can download the library from the <a href="https://code.google.com/p/libphonenumber/">project homepage</a> on – as you might expect – Google Code.</p> 
         <p>You can also get it via npm. Here’s the <a href="https://www.npmjs.org/package/google-libphonenumber">project page</a>, and to install from the command-line:</p> 
         <pre><code class="language-">npm install google-libphonenumber</code></pre> 
         <p>You can also install it using Bower:</p> 
         <pre><code class="language-">bower install libphonenumber</code></pre> 
         <p>If you’re thinking of using it in a front-end project, be warned though – even when minified and compressed, it comes in at over 200Kb.</p> 
         <h3 id="parsing-numbers">Parsing Numbers</h3> 
         <p>In order to demonstrate the library’s key features, I’m going to assume you’re writing a Node.js application. You can find some example code in the <a href="https://github.com/jsprodotcom/telephones-js">repository which complements this article</a>.</p> 
         <p>First, import <code>phoneUtil</code>:</p> 
         <pre><code class="language-javascript">var phoneUtil = require('google-libphonenumber').phoneUtil;</code></pre> 
         <p>Now you can use its <code>parse()</code> method to interpret a telephone number:</p> 
         <pre><code class="language-javascript">var tel = phoneUtil.parse('+12024561111');</code></pre> 
         <p>There are a number of things we can do with this. Let’s first import some constants from the library. Change your <code>require</code> declaration to the following:</p> 
         <pre><code class="language-javascript">var phoneUtil = require('google-libphonenumber').phoneUtil
        , PNF = require('google-libphonenumber').PhoneNumberFormat
        , PNT = require('google-libphonenumber').PhoneNumberType;</code></pre> 
         <p>Now we can do the following:</p> 
         <pre><code class="language-javascript">var tel = phoneUtil.parse('+12024561111');

console.log(phoneUtil.format(tel, PNF.INTERNATIONAL));
console.log(phoneUtil.format(tel, PNF.NATIONAL));
console.log(phoneUtil.format(tel, PNF.E164));</code></pre> 
         <p>The output from this will be as follows:</p> 
         <pre><code class="language-">+1 202-456-1111
(202) 456-1111
+12024561111</code></pre> 
         <p>Now try parsing the number without the international dialing code:</p> 
         <pre><code class="language-javascript">var tel = phoneUtil.parse('2024561111');</code></pre> 
         <p>This will throw the following exception:</p> 
         <pre><code class="language-">Error: Invalid country calling code</code></pre> 
         <p>This is because without explictly telling it what country the number is for, it’s impossible to interpret. The <code>parse()</code> method takes an optional second parameter, which is the ISO 3166-1 alpha-2 (i.e., two character) country code.</p> 
         <p>If you try the line again, but this time passing “US” as the second argument, you’ll find that the results are as before:</p> 
         <pre><code class="language-javascript">var tel = phoneUtil.parse('2024561111', 'US');</code></pre> 
         <p>You can also play around with the formats; all of these will work, too:</p> 
         <pre><code class="language-javascript">var tel = phoneUtil.parse('202-456-1111', 'US');
var tel = phoneUtil.parse('(202) 456 1111', 'US');</code></pre> 
         <p>To interpret a United Kingdom number:</p> 
         <pre><code class="language-javascript">var tel = phoneUtil.parse('(0) 20 7925 0918', 'GB');
console.log(phoneUtil.format(tel, PNF.INTERNATIONAL));
console.log(phoneUtil.format(tel, PNF.NATIONAL));
console.log(phoneUtil.format(tel, PNF.E164));</code></pre> 
         <p>This will output the following:</p> 
         <pre><code class="language-">+44 20 7925 0918
020 7925 0918
+442079250918</code></pre> 
         <p>Once you’ve parsed a number, you can validate it – as we’ll see in the next section.</p> 
         <h3 id="validating-a-number">Validating a Number</h3> 
         <p>Validation follows a similar pattern; again, there is a second optional argument, but one which you’re going to need if the country isn’t implictly stated.</p> 
         <p>Here are some examples of valid numbers, where the country code is either provided as the second argument, or contained within the first argument:</p> 
         <pre><code class="language-javascript">console.log(phoneUtil.isValidNumber(phoneUtil.parse('+12024561111')));
// =&gt; outputs true

console.log(phoneUtil.isValidNumber(phoneUtil.parse('202-456-1111', 'US')));
// =&gt; outputs true

console.log(phoneUtil.isValidNumber(phoneUtil.parse('(0) 20 7925 0918', 'GB')));
// =&gt; outputs true</code></pre> 
         <p>If you don’t supply the country code, or it’s not implied, you get the same error as before:</p> 
         <pre><code class="language-javascript">console.log(phoneUtil.isValidNumber(phoneUtil.parse('(0) 20 7925 0918')));
// =&gt; throws exception "Error: Invalid country calling code"

console.log(phoneUtil.isValidNumber(phoneUtil.parse('2024561111')));
// =&gt; throws exception "Error: Invalid country calling code"</code></pre> 
         <p>Here are some examples where validation fails, returning <code>false</code>:</p> 
         <pre><code class="language-javascript">console.log(phoneUtil.isValidNumber(phoneUtil.parse('573 1234 1234', 'US')));
// =&gt; outputs false

console.log(phoneUtil.isValidNumber(phoneUtil.parse('555-555-5555', 'US')));
// =&gt; outputs false (this is often used as a placeholder, but it's not a valid number)

console.log(phoneUtil.isValidNumber(phoneUtil.parse('295-123-1234', 'US')));
// =&gt; outputs false (there is no 295 area code in the US)</code></pre> 
         <p>Be warned, however, as an invalid number can throw an exception:</p> 
         <pre><code class="language-javascript">console.log(phoneUtil.isValidNumber(phoneUtil.parse('NOT-A-NUMBER', 'US')));
// =&gt; throws exception "Error: The string supplied did not seem to be a phone number"</code></pre> 
         <h3 id="determining-a-numbers-type">Determining a Number’s Type</h3> 
         <p>Sometimes, it’s useful to know the <em>type</em> of a telephone number. For example, you may wish to ensure that you’ve been provided with a mobile number – perhaps you plan to send SMS messages, for example to implement two-factor authentication – or attempt to weed out premium rate numbers.</p> 
         <p>The library’s <code>getNumberType()</code> function does just that. Let’s take a look.</p> 
         <p>The function takes a parsed telephone number as its argument:</p> 
         <pre><code class="language-javascript">var tel = phoneUtil.parse('(0) 20 7925 0918', 'GB');
var type = phoneUtil.getNumberType(tel)</code></pre> 
         <p>The return value is a constant defined in the PhoneNumberType sub-module – you’ll recall we’ve <code>require</code>‘d this in as PNF.</p> 
         <p>As an example, let’s query whether the number in question is a mobile or a fixed line:</p> 
         <pre><code class="language-javascript">if (type === PNT.MOBILE) {
    console.log("It's a mobile number");
} else if (type === PNT.FIXED_LINE) {
    console.log("It's a fixed line");
}</code></pre> 
         <p>As seems to be the theme of the topic, naturally there’s a catch. Sometimes, even the libphonenumber library can’t be sure. US numbers, for example, cannot be easily distinguished; hence the constant <code>PNT.FIXED_LINE_OR_MOBILE</code>.</p> 
         <p>We’ll just have to change our example code to reflect this uncertainty:</p> 
         <pre><code class="language-javascript">if (type === PNT.MOBILE) {
    console.log("It's a mobile number");
} else if (type === PNT.FIXED_LINE) {
    console.log("It's a fixed line");
} else if (type === PNT.FIXED_LINE_OR_MOBILE) {
    console.log("Your guess is as good as mine");
}</code></pre> 
         <p>There are a number of other possibilities, too. Here is the full list currently:</p> 
         <ul> 
          <li><code>PNT.FIXED_LINE</code></li> 
          <li><code>PNT.MOBILE</code></li> 
          <li><code>PNT.FIXED_LINE_OR_MOBILE</code></li> 
          <li><code>PNT.TOLL_FREE</code></li> 
          <li><code>PNT.PREMIUM_RATE</code></li> 
          <li><code>PNT.SHARED_COST</code></li> 
          <li><code>PNT.VOIP</code></li> 
          <li><code>PNT.PERSONAL_NUMBER</code></li> 
          <li><code>PNT.PAGER</code></li> 
          <li><code>PNT.UAN</code></li> 
          <li><code>PNT.UNKNOWN</code></li> 
         </ul> 
         <p>As you can see, the <code>PNT.UNKNOWN</code> reflects the fact that we can’t necessarily glean any information with any certaintly. So in summary, whilst this feature can be useful as a quick initial check, we can’t rely on it.</p> 
         <h2 id="is-the-number-in-service">Is the Number in Service?</h2> 
         <p>There are plenty of telephone numbers which will validate, but which are not in use. They may have been disconnected, not yet allocated, or perhaps a SIM card has been dropped down a toilet.</p> 
         <p>If you need to ensure that a number is not just valid but also active, there are a number of options open to you.</p> 
         <p>One approach is to require users’ confirm their number, in much the same way as you might require users confirm their email address. You can use a service such as <a href="https://www.twilio.com/">Twilio</a> to <a href="https://www.twilio.com/docs/api/rest/sending-messages">send an SMS</a>, or even <a href="https://www.twilio.com/docs/api/rest/making-calls">place a call</a>. </p> 
         <p>Here is a very simple code snippet for generating and sending a confirmation code by SMS using Twilio:</p> 
         <pre><code class="language-javascript">// You'll need to install the Twilio library with "npm install twilio"
var client = require('twilio')('YOUR-SID', 'YOUR-AUTH-TOKEN');

// Generate a random four-digit code
var code = Math.floor(Math.random()*8999+1000); 

// Send the SMS
client.sendMessage({
    to: phoneUtil.format(tel, PNF.E164),  // using libphonenumber to convert to E.164
    from: 'YOUR-NUMBER',
    body: 'Your confirmation code is ' + code
}, function(err, respons) {
    // ...do something
});</code></pre> 
         <p>It’s then a trivial exercise to ask users to enter the code into a form in your web app to verify it – or you could even allow people to validate their number by replying to the message.</p> 
         <p>There are also (paid) services which will check whether a number is in service for you in realtime, such as <a href="http://www.phone-validator.net/phone-number-online-validation-api.html">this one from Byteplant</a>.</p> 
         <h2 id="other-issues">Other Issues</h2> 
         <h3 id="legal">Legal</h3> 
         <p>As with any personal information, there are also plenty of legal issues to be mindful of. In the UK, for example, the <a href="http://www.tpsonline.org.uk/tps/index.html">Telephone Preference Service (TPS)</a> is a national register of telephone numbers which have explictly been registered by people not wishing to receive marketing communications. There are paid services which offer APIs to check a number against this register, such as <a href="http://t2a.co/services/telephone_numbers/tps_check_a_telephone_number">this one</a>.</p> 
         <h3 id="usability-considerations">Usability Considerations</h3> 
         <p>It’s very common to request up to three different telephone numbers in a single form; for example daytime, evening and mobile.</p> 
         <p>It’s also worth remembering that asking for telephone numbers over the internet can come across as rather intrusive. If someone is unwilling to provide that information despite you having made it a required field, they will probably do one of two things:</p> 
         <ul> 
          <li>Attempt to “fool” the validation. Depending on the approach, they may type something like “ex directory”, or enter an invalid number – such as one which contains only numbers.</li> 
          <li>Walk away.</li> 
         </ul> 
         <h2 id="combining-the-jquery-plugin-with-libphonenumber">Combining the jQuery Plugin with libphonenumber</h2> 
         <p>You might remember that the jQuery plugin has a rather cryptically named option called <code>utilsScript</code>.</p> 
         <p>This option allows us to take advantage of the validation and formatting features of <code>libphonenumber</code>. Having selected a country – either using the drop-down or by typing the dialing code – it will transform the textfield into a masked input which reflects that country’s numbering format.</p> 
         <p>The plugin contains a packaged version of libphonenumber; pass the path to this file to the constructor as follows:</p> 
         <pre><code class="language-javascript">$("#number").intlTelInput(
    {
        utilsScript : '/bower_components/intl-tel-input/lib/libphonenumber/build/utils.js'
    }
);</code></pre> 
         <p>As I’ve previously mentioned, do bear in mind that this approach should be used with caution, owing to the file size of the <code>libphonenumber</code> library. Referencing it here in the constructor does however mean that it can be loaded on demand.</p> 
         <h2 id="displaying-telephone-numbers">Displaying Telephone Numbers</h2> 
         <p>We’ve looked at how we can format numbers when displaying them to be more “friendly”, using formats such as <code>PNF.INTERNATIONAL</code> and <code>PNF.NATIONAL</code>.</p> 
         <p>We can also use the <code>tel</code> and <code>callto</code> protocols to add hyperlinks to telephone numbers, which are particulary useful on mobile sites – allowing users to dial a number direct from a web page. </p> 
         <p>To do this, we need the E.164 format for the link itself – for example:</p> 
         <pre><code class="language-markup">&lt;a href="tel:+12024561111"&gt;+1 202-456-1111&lt;/a&gt;</code></pre> 
         <p>Of course, you could use the <code>libphonenumber</code> library’s <code>format()</code> method to render both the E.164 version (<code>PNF.E164</code>) and the more user-friendly display version.</p> 
         <h2 id="microdata">Microdata</h2> 
         <p>We can also use Microdata to mark up telephone numbers semantically. Here’s an example; note the use of <code>itemprop="telephone"</code> to mark-up the link:</p> 
         <pre><code class="language-markup">&lt;div itemscope itemtype="http://schema.org/LocalBusiness"&gt;
    &lt;h1 itemprop="name"&gt;Acme Corp, Inc.&lt;/h1&gt;
    Phone: &lt;span itemprop="telephone"&gt;&lt;a href="tel:+12024561111"&gt;202-456-1111&lt;/a&gt;&lt;/span&gt;
&lt;/div&gt;</code></pre> 
         <h2 id="summary">Summary</h2> 
         <p>In this article, we’ve opened up the hornets nest that is telephone numbers. It should be pretty apparent by now that there are all sorts of complexities, subtleties and gotchas you need to be aware of if you need to collect, validate and display them.</p> 
         <p>We’ve looked at a few methods for collecting numbers – the “tel” input type, masked inputs and finally the <code>intl-tel-input</code> jQuery plugin.</p> 
         <p>We then looked at some of the issues around validation, and why common approaches such as regular expressions are often inadequate, particularly when you go international.</p> 
         <p>We took a look at Google’s <code>libphonenumber</code> library; using it to parse, validate, display and determine the type of telephone numbers.</p> 
         <p>We combined the <code>intl-tel-input</code> plugin with <code>libphonenumber</code> for an even better user experience, albeit one which comes at a cost in terms of performance.</p> 
         <p>Finally we looked at how we might mark up telephone numbers in our HTML.</p> 
         <p>There are a few recommendations I would make for dealing with telephone numbers:</p> 
         <ul> 
          <li>Unless you only operate in a single country, be aware of the international differences.</li> 
          <li>Use masked inputs with caution.</li> 
          <li>Be very careful with regular expression-based validation.</li> 
          <li>Where possible, use E.164 for storage.</li> 
          <li>Use Google’s libphonenumber library.</li> 
          <li>When displaying numbers, format them where possible, use the tel: or callto: link type, and use Microdata.</li> 
         </ul>
