       </svg></a>Node Application Metrics</h1> 
     <p>Node Application Metrics monitoring and profiling agent</p> 
     <p><a href="https://www.codacy.com/app/dancunnington/appmetrics?utm_source=github.com&amp;utm_medium=referral&amp;utm_content=RuntimeTools/appmetrics&amp;utm_campaign=Badge_Grade"><img src="https://api.codacy.com/project/badge/Grade/8caec03a401f4a37823ac547d7a0a272" alt="Codacy Badge"></a> <a href="https://travis-ci.org/RuntimeTools/appmetrics"><img src="https://travis-ci.org/RuntimeTools/appmetrics.svg?branch=master" alt="Build Status"></a> <a href="https://codebeat.co/projects/github-com-runtimetools-appmetrics-master"><img src="https://codebeat.co/badges/9e9229c4-dcfa-4a98-a9a2-4770f3a2dd59" alt="codebeat badge"></a> <a href="https://codecov.io/github/RuntimeTools/appmetrics?branch=master"><img src="https://codecov.io/github/RuntimeTools/appmetrics/coverage.svg?branch=master" alt="codecov.io"></a> <img src="https://img.shields.io/badge/license-Apache2-blue.svg?style=flat" alt="Apache 2"> <a href="https://developer.ibm.com/node/monitoring-post-mortem/application-metrics-node-js/"><img src="https://img.shields.io/badge/homepage-Node%20Application%20Metrics-blue.svg" alt="Homepage"></a></p> 
     <p>Node Application Metrics instruments the Node.js runtime for performance monitoring, providing the monitoring data via an API. Additionally the data can be visualized by using the <a href="https://github.com/RuntimeTools/appmetrics-dash">Node Application Metrics Dashboard</a>.</p> 
     <p>The data can also be visualized in Eclipse using the <a href="https://marketplace.eclipse.org/content/ibm-monitoring-and-diagnostic-tools-health-center">IBM Monitoring and Diagnostics Tools - Health Center</a> client. Profiling data is available in Health Center, but is not yet available in the Dashboard. See <a href="https://www.ibm.com/developerworks/java/jdk/tools/healthcenter/">https://www.ibm.com/developerworks/java/jdk/tools/healthcenter/</a> for more details.</p>

     <h2><a id="user-content-performance-overhead" class="deep-link" href="#performance-overhead" aria-hidden="true">
       <svg aria-hidden="true" class="deep-link-icon" height="16" version="1.1" viewbox="0 0 16 16" width="16">
        <path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path>
       </svg></a>Performance overhead</h2> 
     <p>Our testing has shown that the performance overhead in terms of processing is minimal, adding less than 0.5 % to the CPU usage of your application. The additional memory required is around 20 MB to gather information about your system and application.</p> 
     <p>We gathered this information by monitoring the sample application <a href="https://github.com/RuntimeTools/appmetrics/wiki">Acme Air</a>. We used MongoDB as our datastore and used JMeter to drive load though the program. We have performed this testing with Node.js version 6.10.3</p>

     <h3><a id="user-content-installation" class="deep-link" href="#installation" aria-hidden="true">

       </svg></a>Installation</h3> 
     <p>You can get Node Application Metrics from 3 different places:</p> 
     <ul> 
      <li>npmjs.org (install by running <code>npm install appmetrics</code>. Native libraries are prebuilt)</li> 
      <li>Github (<a href="https://github.com/RuntimeTools/appmetrics/wiki/Install-direct-from-github-source">install from source</a> by cloning the git repository. Requires a compiler)</li> 
      <li><a href="https://developer.ibm.com/node/sdk/">IBM SDK for Node.js</a> (packaged with the SDK, native libraries are prebuilt)</li> 
     </ul> 
     <p>Using <strong>npm</strong> you can install Node Application Metrics either locally or globally.</p> 
     <p><strong>When installed locally</strong> you can access monitoring data via both the API and the Health Center client by modifying your application to use appmetrics (see <em><a href="#modifying-your-application-to-use-the-local-installation">Modifying your application to use the local installation</a></em>).</p> 
     <p>To perform a local install:</p> 
     <div class="highlight sh">
      <pre class="editor editor-colors">
       <div class="line">
        <span class="source shell"><span>$&nbsp;npm&nbsp;install&nbsp;appmetrics</span></span>
       </div></pre>
     </div> 
     <p>A local install will put the module inside "<em><code>./node_modules</code> of the current package root</em>" (see the <a href="https://docs.npmjs.com/files/folders">npm documentation</a> for more information); usually this is the current directory and in that case the module installation directory will be <code>./node_modules/appmetrics</code>.</p> 
     <p><strong>When installed globally</strong> you can access monitoring data via the Health Center client (but not the API) by using the <code>node-hc</code> command-line utility (see <em><a href="#the-node-hc-command">The <code>node-hc</code> command</a></em>).</p> 
     <p>To perform a global install:</p> 
     <div class="highlight sh">
      <pre class="editor editor-colors">
       <div class="line">
        <span class="source shell"><span>$&nbsp;npm&nbsp;install&nbsp;-g&nbsp;appmetrics</span></span>
       </div></pre>
     </div> 
     <p>A global install will put the module inside a directory tied to your Node.js SDK.</p> 
     <ul> 
      <li>On Windows, either: 
       <ul> 
        <li><code>&lt;UserDirectory&gt;\AppData\Roaming\npm\node_modules</code></li> 
        <li>or: <code>&lt;NodeInstallDirectory&gt;\node_modules</code></li> 
       </ul> </li> 
      <li>On other platforms: 
       <ul> 
        <li><code>&lt;node_install_directory&gt;/lib/node_modules</code></li> 
       </ul> </li> 
     </ul> 
     <p>It also adds the <code>node-hc</code> command to another directory tied to your Node.js SDK, one that was added to your executable search path by the Node.js SDK installer.</p> 
     <ul> 
      <li>On Windows, either: 
       <ul> 
        <li><code>&lt;UserDirectory&gt;\AppData\Roaming\npm</code></li> 
        <li>or: <code>&lt;NodeInstallDirectory&gt;</code></li> 
       </ul> </li> 
      <li>On other platforms: 
       <ul> 
        <li><code>&lt;node_install_directory&gt;/bin</code></li> 
       </ul> </li> 
     </ul>

        <path d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path>
       </svg></a>Configuring Node Application Metrics</h3> 
     <p>Node Application Metrics can be configured in two ways, by using the configuration file described below or via a call to configure(options).</p> 
     <p>Node Application Metrics comes with a configuration file inside the <a href="#installation">module installation directory</a> (<code>.../node_modules/appmetrics/appmetrics.properties</code>). This can be used to configure connection options, logging and data source options.</p> 
     <p>Node Application Metrics will attempt to load <code>appmetrics.properties</code> from one of the following locations (in order):</p> 
     <ol> 
      <li>the application directory</li> 
      <li>the current working directory</li> 
      <li>the appmetrics module installation directory</li> 
     </ol> 
     <p>The default configuration has minimal logging enabled, will attempt to send data to a local MQTT server on the default port and has method profiling disabled.</p> 
     <p>Many of the options provide configuration of the Health Center core agent library and are documented in the Health Center documentation: <a href="https://www-01.ibm.com/support/knowledgecenter/SS3KLZ/com.ibm.java.diagnostics.healthcenter.doc/topics/configproperties.html">Health Center configuration properties</a>.</p> 
     <p>The following options are specific to appmetrics:</p> 
     <ul> 
      <li><code>com.ibm.diagnostics.healthcenter.data.profiling=[off|on]</code> Specifies whether method profiling data will be captured. The default value is <code>off</code>. This specifies the value at start-up; it can be enabled and disabled dynamically as the application runs, either by a monitoring client or the API.</li> 
     </ul>

       </svg></a>The <code>node-hc</code> command</h3> 
     <p>If you <a href="#installation">globally installed</a> this module with npm, you can use the <code>node-hc</code> command to run your application instead of the <code>node</code> command. This will run your application as it would normally under node (including any node options) but additionally load and start <code>appmetrics</code>.</p> 
     <div class="highlight sh">
      <pre class="editor editor-colors">
       <div class="line">
        <span class="source shell"><span>$&nbsp;node-hc&nbsp;app.js</span></span>
       </div></pre>
     </div> 
     <p>The purpose of this mode of operation is to provide monitoring of the application without requiring any changes to the application code. The data is sent to the Health Center Eclipse IDE client.</p>

       </svg></a>Connecting to the client</h3> 
     <p>Connecting to the Health Center client requires the additional installation of a MQTT broker. The Node Application Metrics agent sends data to the MQTT broker specified in the <code>appmetrics.properties</code> file or set via a call to configure(options). Installation and configuration documentation for the Health Center client is available from the <a href="http://www.ibm.com/support/knowledgecenter/SS3KLZ/com.ibm.java.diagnostics.healthcenter.doc/topics/connecting.html">Health Center documentation in IBM Knowledge Center</a>.</p> 
     <p>Note that both the API and the Health Center client can be used at the same time and will receive the same data. Use of the API requires a local install and application modification (see <em><a href="#modifying-your-application-to-use-the-local-installation">Modifying your application to use the local installation</a></em>).</p> 
     <p>Further information regarding the use of the Health Center client with Node Application Metrics can be found on the <a href="https://github.com/RuntimeTools/appmetrics/wiki">appmetrics wiki</a>: <a href="https://github.com/RuntimeTools/appmetrics/wiki/Using-Node-Application-Metrics-with-the-Health-Center-client">Using Node Application Metrics with the Health Center client</a>.</p>

     <p>Method profiling data is not collected by default, check <em><a href="#configuring-node-application-metrics">Configuring Node Application Metrics</a></em> for information on how to enable it.</p> 
     <p>If collection is enabled, an absence of method profiling data from a Node.js application could be caused by the type of tasks that are being run by your application -- it may be running long, synchronous tasks that prevent collection events from being scheduled on the event loop.</p> 
     <p>If a task uses the Node.js thread exclusively then shuts down the Node.js runtime environment, the Health Center agent may not get the opportunity to obtain <em>any</em> profiling data.
