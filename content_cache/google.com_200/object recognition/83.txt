       <h1>Addressing Objects in Windows Store Applications</h1> 
       <div class="tree-button"> 
        <button class="btn btn-lg btn-primary" data-toggle="offcanvas"><i class="fa fa-bars"></i> Addressing Objects in Windows Store Applications</button> 
       </div> 
      </div> 
      <div class="doc-info">
       <i class="fa fa-code-fork"></i> Applies to TestComplete 12.31, last modified on July 10, 2017
      </div> 
      <!-- Topic start --> 
      <div id="sbTopicBody">
       <img src="/testcomplete/docs/_assets/commonImages/minus-btn.gif" style="display: none;">
       <p>The way you address objects in Windows Store applications depends on the application type.</p> 
       <div style="margin-top: 16px; margin-left: 16px; "> 
        <p><a href="#process-object-names"><img src="/testcomplete/docs/_assets/commonImages/go.gif" border="0" class="aqInlineImg">Process Object Names</a></p> 
        <p><a href="#addressing-objects-in-javascript"><img src="/testcomplete/docs/_assets/commonImages/go.gif" border="0" class="aqInlineImg">Addressing Objects in JavaScript-Based Windows Store Applications</a></p> 
        <div class="aqIndent"> 
         <p class="aqTextInLink">? <a href="#addressing-objects-in-javascript-object-hierarchy">Object Hierarchy</a></p> 
         <p class="aqTextInLink">? <a href="#addressing-objects-in-javascript-object-names">Object Names</a></p> 
        </div> 
        <p><a href="#addressing-objects-in-xaml-based"><img src="/testcomplete/docs/_assets/commonImages/go.gif" border="0" class="aqInlineImg">Addressing Objects in XAML-Based Windows Store Applications</a></p> 
        <p><a href="#shortening-object-names"><img src="/testcomplete/docs/_assets/commonImages/go.gif" border="0" class="aqInlineImg">Shortening Object Names</a></p> 
       </div>
       <a id="process-object-names" name="process-object-names"></a>
       <h3 style="">Process Object Names</h3> 
       <div style=""> 
        <p>TestComplete recognizes Windows Store applications running in the system and displays them as <code>Process(<i>name</i>)</code> nodes in the Object Browser:</p> 
        <div class="aqImageEx" style="width : 368px;">
         <a href="../../../_images/app-testing/desktop/windows-store/process-name.png" target="_blank"><img src="../../../_images/app-testing/desktop/windows-store/process-name.png" border="0" align="center" title="Testing Windows Store applications: A process name in the Object Browser" alt="Testing Windows Store applications: Process name in the Object Browser" width="368" height="156"></a>
         <p>Click the image to enlarge it.</p> 
        </div> 
        <p>The <span class="aqParam">name</span>&nbsp;parameter is a process identifier. If an application’s package contains only one application, it is the package name as it is specified in the package manifest. If the package contains several applications, it is the application's ID (it can be a GUID).</p> 
        <p>Both these values differ from the application name that users see in the operating system. They are chosen because they are “stable” parameters that can be used as scripting identifiers. The application’s display name cannot be used for this purpose, as it depends on the operating system’s language. It can include local symbols and can vary from one computer to another. For example, it can be Weather in English and Väder in Swedish.</p> 
        <p>To learn the application’s display name, use the <a href="../../../reference/test-objects/members/window-and-process/friendlyname-property.html"><code>FriendlyName</code></a> property. TestComplete provides it for the <a href="../../../reference/test-objects/controls/desktop/general/process/index.html"><code>Process</code></a> objects that correspond to Windows Store applications.</p> 
       </div>
       <a id="addressing-objects-in-javascript" name="addressing-objects-in-javascript"></a>
       <h3>Addressing Objects in JavaScript-Based Windows Store Applications</h3> 
       <div> 
        <p>JavaScript-based Windows Store applications include a built-in web browser. TestComplete recognizes the application controls as web page elements displayed in this browser. It names the test objects that correspond to the controls in the same way as it would name elements on a web page displayed in Internet Explorer, Chrome or other supported browser.</p> 
        <p>The control names depend on the <a href="../../../working-with/managing-projects/properties/open-apps/web-testing/general.html"><span class="aqOption">Tree model</span></a> property of your project. The default property value is <i>Tree</i>. This section provides a brief overview of this default model. For detailed information on it and on addressing objects in other models, see topics of the <a href="../../web/general/object-identification/index.html">Understanding Web Object Identification and Object Models</a> section.</p>
        <a id="addressing-objects-in-javascript-object-hierarchy" name="addressing-objects-in-javascript-object-hierarchy"></a>
        <h4>Object Hierarchy</h4> 
        <div> 
         <p>The full name of a test object includes the names of its parent objects, that is, it depends on the hierarchy of test objects. And the hierarchy of web test objects is defined by the <span class="aqOption">Tree model</span> project property mentioned above. Below is a sample view of the object hierarchy in a JavaScript-based Windows Store application:</p> 
         <div class="aqImageEx" style="width : 450px;">
          <a href="../../../_images/app-testing/desktop/windows-store/js-win-store-app-ob.png" target="_blank"><img src="../../../_images/app-testing/desktop/windows-store/js-win-store-app-ob.png" border="0" align="center" title="Testing Windows Store applications: Properties of a JavaScript-based application in the Object Browser" alt="Testing Windows Store applications: Properties of a JavaScript-based application in the Object Browser" width="450" height="296"></a>
          <p>Click the image to enlarge it.</p> 
         </div> 
         <p>As you can see, the <code>Process</code> item that corresponds to the tested application contains a <code>BrowserWindow</code> window, which is a substrate control for the built-in browser. Some levels below <code>BrowserWindow</code>, there is a <code>Page</code>&nbsp;object that corresponds to a web page displayed in the built-in browser.</p> 
         <p>The items below the <code>Page</code>&nbsp;object correspond to the application objects. Their hierarchy depends on the <span class="aqOption">Tree model</span> property. If the property value is <i>Tree</i>, the hierarchy reflects the nesting elements in the HTML markup of the page. For simplicity, the hierarchy does not include some elements that are considered to be insignificant. A typical example of such elements can be substrate containers that do not have any text. For complete information on what controls are included in and excluded from the hierarchy, see <a href="../../web/general/object-identification/tree-model-elements.html">Web Elements Included in and Excluded From the Tree Model</a>.</p> 
        </div>
        <a id="addressing-objects-in-javascript-object-names" name="addressing-objects-in-javascript-object-names"></a>
        <h4>Object Names</h4> 
        <div> 
         <p>If you use <a href="../../../testing-with/object-identification/name-mapping/index.html">Name Mapping</a>, the names used to refer to web page objects from tests are specified in Name Mapping. For example, the full name of a user name input field on a page can be as follows:</p> 
         <div class="aqCodeSegment" filtered="NO"> 
          <div id="IDUG1CZJ15DLOSJUOZ3PXJUUVKBJ1X14T3M0TJ12CSHKCVAINMWETE_divcodeseg" class="aqCodeSegmentInternal"> 
           <p> Aliases.MyWinStoreApp.BrowserWindow.WebPlatformEmbedding.browser.pageMywinstoreapp.textboxNameinput </p> 
          </div> 
         </div> 
         <p>Name mapping is enabled by default, so if you record a test on a Windows Store application, the test commands will use the mapped names (aliases). You can modify the default generated names in the <a href="../../../testing-with/object-identification/name-mapping/editor/about.html">Name Mapping</a> editor to make them more descriptive (see <a href="../../../testing-with/object-identification/name-mapping/managing/modifying-names.html">Renaming Objects in the Name Mapping Repository</a>).</p> 
         <p>Here is a sample view of the mapped object names in the <a href="../../../testing-with/object-identification/name-mapping/editor/about.html">Name Mapping</a> editor:</p> 
         <div class="aqImageEx" style="width : 400px;">
          <a href="../../../_images/app-testing/desktop/windows-store/win-store-app-name-mapping.png" target="_blank"><img src="../../../_images/app-testing/desktop/windows-store/win-store-app-name-mapping.png" border="0" align="center" title="Testing Windows Store applications: Object names in the Name Mapping editor" alt="Testing Windows Store applications: Object names in the Name Mapping editor" width="400" height="318"></a>
          <p>Click the image to enlarge it.</p> 
         </div> 
         <p style="text-align : justified; ">The object structure in <a href="../../../testing-with/object-identification/name-mapping/index.html">Name Mapping</a> matches the object structure in the Object Browser. However, if the objects are mapped while the <a href="../../../reference/options/engines/name-mapping.html"><span class="aqOption">Use extended find when possible</span></a> option is enabled, Name Mapping does not include some layout containers like divs and tables, which are not used for identification of tested controls.</p> 
         <p>For more information on mapping web object names, see <a href="../../web/general/object-identification/about.html#NameMapping">Addressing Web Objects Using Name Mapping and Aliases</a>.</p> 
         <p style="margin-top : 32px; text-align : justified; ">In the <a href="../../../testing-with/exploring-apps/object-browser/index.html">Object Browser</a> and in tests where Name Mapping is not used, object names have the <code>format </code><code>ObjectType(<i>Object_Identifier</i>)</code>, for example –</p> 
         <div class="aqCodeSegment" filtered="NO"> 
          <div id="IDTCLZTHVYPGXGFQMDFUCWRJYGNBKNWCY4EWUQTPJLMNJAUGAJXH4M_divcodeseg" class="aqCodeSegmentInternal"> 
           <p>Panel(0)<br> Image("logo")<br> Link("firstTab")</p> 
          </div> 
         </div> 
         <p>TestComplete analyzes the web element’s type and gives the test object the name that corresponds to the type. For example, the A elements (links) are named as <code>Link(...)</code> and DIV elements are named as <code>Panel(...)</code>. For complete information on correspondence between the types and names, see <a href="../../web/general/object-identification/object-types.html">Web Object Types Used by the Tree Model and Hybrid Mobile Applications</a>.</p> 
         <p>The identifier is typically, the value of the <code>id</code> or <code>name</code> attribute that is specified for the web control in the application. However, there are some exceptions. For IMG elements, for example, TestComplete uses the file name specified in the <code>src</code> attribute; for AREA elements, it uses the element’s text.</p> 
         <p>If TestComplete is unable to determine the id of a web element, it uses the zero-based index of the object among the sibling objects of the same type.</p> 
         <p>For detailed information on naming web test objects, see <a href="../../web/general/object-identification/tree-model-elements.html">Web Elements Included in and Excluded From the Tree Model</a>.</p> 
        </div> 
       </div>
       <a id="addressing-objects-in-xaml-based" name="addressing-objects-in-xaml-based"></a>
       <h3>Addressing Objects in XAML-Based Windows Store Applications</h3> 
       <div> 
        <p>To expose internal objects of Windows Store applications that are not JavaScript-based, TestComplete uses the <a href="../../../testing-with/object-identification/improving/using-ui-automation/about.html">UI Automation</a> technology. Below is a sample view of exposed objects:</p> 
        <div class="aqImageEx" style="width : 450px;">
         <a href="../../../_images/app-testing/desktop/windows-store/cs-win-store-app-ob.png" target="_blank"><img src="../../../_images/app-testing/desktop/windows-store/cs-win-store-app-ob.png" border="0" align="center" title="Testing Windows Store applications: Test objects in a C# application" alt="Testing Windows Store applications: Test objects in a C# application" width="450" height="296"></a>
         <p>Click the image to enlarge it.</p> 
        </div> 
        <p>As you can see, the object names have the format <code>UIAObject(<i>Object_Identifier</i>)</code>. Here, <code><i>Object_Identifier</i></code> is the value of the object’s <code>ObjectIdentifier</code> property, which, in its turn, is based on the UI Automation <code>Name</code> property. Note that this property’s name coincides with the <code>Name</code> property that TestComplete provides for all test objects, so you can access it through the <code>NativeUIAObject</code>&nbsp;<a href="../../../scripting/namespaces.html">namespace</a>.</p> 
        <p>The name must be a valid script identifier, so it must start with a letter and contain only alphanumeric characters and underscores. If some character is not allowed in the identifier, TestComplete replaces it with an underscore. However, in some cases, even replacements cannot make the name a valid script identifier.</p> 
        <p>If TestComplete is unable to obtain the <code>Name</code> property or if it cannot make the name a valid identifier valid, it addresses the object in the usual way, that is, by class name, caption and index: <code>Window("<span class="aqParam">WndClassName</span>", "<span class="aqParam">WndCaption</span>", <span class="aqParam">WndIndex</span>)</code>.</p> 
        <p style="margin-top : 32px; text-align : justified; ">If you use <a href="../../../testing-with/object-identification/name-mapping/index.html">Name Mapping</a>, then TestComplete uses the names specified in the <a href="../../../testing-with/object-identification/name-mapping/editor/about.html">Name Mapping</a> editor. The object names in your tests can be as follows –</p> 
        <div class="aqCodeSegment" filtered="NO"> 
         <div id="IDDZXYXBASQT0KKU4UPBK0TYAHDBFO3U5REETS5JHGYNW5CZFAQHIK_divcodeseg" class="aqCodeSegmentInternal"> 
          <p>Aliases.App1.wndApp1.Back<br> Aliases.App1.wndApp1.Items_In_Group.Group_Title_2</p> 
         </div> 
        </div> 
        <p>Name mapping is enabled by default, so if you record a test on a Windows Store application, the test commands will use the mapped names (aliases).</p> 
        <p>To form an alias, TestComplete uses the value of the <code>ObjectIdentifier</code> property. You can check the aliases and the properties that TestComplete uses for object recognition in the <a href="../../../testing-with/object-identification/name-mapping/editor/about.html">Name Mapping</a> editor.</p> 
       </div>
       <a id="shortening-object-names" name="shortening-object-names"></a>
       <h3>Shortening Object Names</h3> 
       <div> 
        <p>As the object hierarchy reflects the control parent-child relationships, the full object names can be rather long. This is frequent for JavaScript-based Windows Store applications, where the object hierarchy reflects the nesting HTML elements, but also can happen with objects of XAML-based applications:</p> 
        <div class="aqCodeSegment" filtered="NO"> 
         <div id="ID342ABVYOWUBJPYZP3HLH1VFY2EE5PSEFHHTBCUF3GX4GZOLLY2VC_divcodeseg" class="aqCodeSegmentInternal"> 
          <p><i>// With name mapping</i><br> Aliases.Hilo_JavaScript.BrowserWindow.WebPlatformEmbedding.browser.pageHilo.panelContenthost.panelPagecontrol.panelFragmentHub.Section("image_nav").Panel("appbar").Button("rotate").TextNode(1)<br><br><i>// Without name mapping</i><br> Sys.Process("Hilo JavaScript").BrowserWindow(0).Window("Web Platform Embedding", "", 1).Window("Internet Explorer_Server", "", 1).Page("ms-appx://microsoft.sdksamples.hilo.js/default.html").Panel("contenthost").Panel(0).Panel(0).Section("image_nav").Panel("appbar").Button("rotate").TextNode(1)</p> 
         </div> 
        </div> 
        <p>To make them shorter, you can do the following:</p> 
        <ul type="disc"> 
         <li> <p>If you use <a href="../../../testing-with/object-identification/name-mapping/index.html">Name Mapping</a>, you can use the <a href="../../../testing-with/object-identification/name-mapping/managing/modifying/extended-search-criteria.html"><i>Extended Find</i></a> feature to skip intermediate hierarchy levels. See <a href="../../../testing-with/object-identification/name-mapping/managing/modifying/extended-search-criteria.html">Using Extended Search Criteria in Name Mapping Repository</a>.</p> </li> 
         <li> <p>You can get references to target objects by using search functions like <a href="../../../reference/test-objects/members/common-for-all/findchild-method.html"><code>FindChild</code></a> or <a href="../../../reference/test-objects/members/page/evaluatexpath-action-page-object.html"><code>EvaluateXPath</code></a> (see <a href="../../../app-objects/common-tasks/searching.html">Searching for an Object</a> and <a href="../../web/general/common-tasks/finding-objects/about.html">About Finding Objects on Web Pages</a>). Once you get a reference and store it to a variable, you use the variable to work with the object, there is no need to use the long object name.</p> </li> 
         <li> <p>In <a href="../../../keyword-testing/index.html">keyword tests</a>, you can use the <a href="../../../reference/options/panels/kdt-editor/groups.html"><span class="aqOption">Maximum depth</span></a> option to reduce the number of <i>displayed</i> levels. When editing operations, you see the full object hierarchy.</p> </li> 
        </ul> 
       </div>
