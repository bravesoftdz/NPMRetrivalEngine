     <h1 class="Page-title">Node.js Source</h1>
     <div class="Page-contents" data-swiftype-name="body" data-swiftype-type="text">
      <p>Our Node.js library lets you record analytics data from your node code. The requests hit our servers, and then we route your data to any analytics service you enable on your destinations page.</p>
      <p>This library is open-source, so you can <a href="https://github.com/segmentio/analytics-node">check it out on Github</a>.</p>
      <p>All of our server-side libraries are built for high-performance, so you can use them in your web server controller code. This library uses an internal queue to make <code>identify</code> and <code>track</code> calls non-blocking and fast. It also batches messages and flushes asynchronously to our servers.</p>
      <p>Want to stay updated on releases? Subscribe to the <a href="https://github.com/segmentio/analytics-node/releases.atom">release feed</a>.</p>
      <h2 id="getting-started">Getting Started</h2>
      <h3 id="install-the-module">Install the Module</h3>
      <p>Run:</p>
      <pre><code>npm install --save analytics-node
</code></pre>
      <p>This will add our Node library module to your <code>package.json</code>. The module exposes an <code>Analytics</code> constructor, which you need to initialize with your Segment source’s <strong>Write Key</strong>, like so:</p>
      <pre><code class="lang-javascript">var Analytics = require('analytics-node');
var analytics = new Analytics('YOUR_WRITE_KEY');
</code></pre>
      <p>Of course, you’ll want to replace <code>YOUR_WRITE_KEY</code> with your actual <strong>Write Key</strong> which you can find in Segment under your source settings.</p>
      <p>This will create an instance of <code>Analytics</code> that you can use to send data to Segment for your project. The default initialization settings are production-ready and queue 20 messages before sending any requests. In development you might want to use <a href="/docs/sources/server/node/#development">development settings</a>.</p>
      <h2 id="identify">Identify</h2>
      <p><code>identify</code> lets you tie a user to their actions and record traits about them. It includes a unique User ID and any optional traits you know about them.</p>
      <p>We recommend calling <code>identify</code> a single time when the user’s account is first created, and only identifying again later when their traits change.</p>
      <p>Example <code>identify</code> call:</p>
      <pre><code class="lang-javascript">analytics.identify({
  userId: '019mr8mf4r',
  traits: {
    name: 'Michael Bolton',
    email: 'mbolton@initech.com',
    plan: 'Enterprise',
    friends: 42
  }
});
</code></pre>
      <p>This call is identifying Michael by his unique User ID (the one you know him by in your database) and labeling him with <code>name</code>, <code>email</code>, <code>plan</code> and <code>friends</code> traits.</p>
      <p>The <code>identify</code> call has the following fields:</p>
      <table class="api-table">
       <tbody>
        <tr>
         <td><code>userId</code> _String</td>
         <td>The ID for this user in your database.</td>
        </tr>
        <tr>
         <td><code>traits</code> <em>Object, optional</em></td>
         <td>A dictionary of <a href="/docs/spec/identify#traits">traits</a> you know about the user. Things like: <code>email</code>, <code>name</code> or <code>friends</code>.</td>
        </tr>
        <tr>
         <td><code>timestamp</code> <em>Date, optional</em></td>
         <td>A Javascript date object representing when the identify took place. If the identify just happened, leave it out and we’ll use the server’s time. If you’re importing data from the past make sure you to send a <code>timestamp</code>.</td>
        </tr>
        <tr>
         <td><code>context</code> <em>Object, optional</em></td>
         <td>A dictionary of extra <a href="">context</a> to attach to the call. <em>Note: <code>context</code> differs from <code>traits</code> because it is not attributes of the user itself.</em></td>
        </tr>
        <tr>
         <td><code>anonymousId</code> <em>String, optional</em></td>
         <td>An ID to associated with the user when you don’t know who they are (eg., <a href="/docs/sources/website/analytics.js/#anonymous-id">the anonymousId generated by <code>analytics.js</code></a>)</td>
        </tr>
       </tbody>
      </table>
      <p>Find details on the <strong>identify method payload</strong> in our <a href="/docs/spec/identify/">Spec</a>.</p>
      <h2 id="track">Track</h2>
      <p><code>track</code> lets you record the actions your users perform. Every action triggers what we call an “event”, which can also have associated properties.</p>
      <p>You’ll want to track events that are indicators of success for your site, like <strong>Signed Up</strong>, <strong>Item Purchased</strong> or <strong>Article Bookmarked</strong>.</p>
      <p>To get started, we recommend tracking just a few important events. You can always add more later!</p>
      <p>Example <code>track</code> call:</p>
      <pre><code class="lang-javascript">analytics.track({
  userId: '019mr8mf4r',
  event: 'Item Purchased',
  properties: {
    revenue: 39.95,
    shippingMethod: '2-day'
  }
});
</code></pre>
      <p>This example <code>track</code> call tells us that your user just triggered the <strong>Item Purchased</strong> event with a revenue of $39.95 and chose your hypothetical ‘2-day’ shipping.</p>
      <p><code>track</code> event properties can be anything you want to record. In this case, revenue and shipping method.</p>
      <p>The <code>track</code> call has the following fields:</p>
      <table>
       <tbody>
        <tr>
         <td><code>userId</code> _String</td>
         <td>The ID for this user in your database.</td>
        </tr>
        <tr>
         <td><code>event</code> <em>String</em></td>
         <td>The name of the event you’re tracking. We recommend human-readable names like <code>Song Played</code> or <code>Status Updated</code>.</td>
        </tr>
        <tr>
         <td><code>properties</code> <em>Object, optional</em></td>
         <td>A dictionary of properties for the event. If the event was <code>Product Added</code>, it might have properties like <code>price</code> or <code>product</code>.</td>
        </tr>
        <tr>
         <td><code>timestamp</code> <em>Date, optional</em></td>
         <td>A Javascript date object representing when the track took place. If the track just happened, leave it out and we’ll use the server’s time. If you’re importing data from the past make sure you to send a <code>timestamp</code>.</td>
        </tr>
        <tr>
         <td><code>context</code> <em>Object, optional</em></td>
         <td>A dictionary of extra <a href="">context</a> to attach to the call. <em>Note: <code>context</code> differs from <code>traits</code> because it is not attributes of the user itself.</em></td>
        </tr>
        <tr>
         <td><code>anonymousId</code> <em>String, optional</em></td>
         <td>An ID to associated with the user when you don’t know who they are (eg., <a href="/docs/sources/website/analytics.js/#anonymous-id">the anonymousId generated by <code>analytics.js</code></a>)</td>
        </tr>
       </tbody>
      </table>
      <p>Find details on <strong>best practices in event naming</strong> as well as the <strong><code>track</code> method payload</strong> in our <a href="/docs/spec/track/">Spec</a>.</p>
      <h2 id="page">Page</h2>
      <p>The <a href="/docs/spec/page/"><code>page</code></a> method lets you record page views on your website, along with optional extra information about the page being viewed.</p>
      <p>If you’re using our client-side setup in combination with the Node.js library, page calls are <strong>already tracked for you</strong> by default. However, if you want to record your own page views manually and aren’t using our client-side library, read on!</p>
      <p>Example <code>page</code> call:</p>
      <pre><code class="lang-js">analytics.page({
  userId: '019mr8mf4r',
  category: 'Docs',
  name: 'Node.js Library',
  properties: {
    url: 'https://segment.com/docs/libraries/node',
    path: '/docs/libraries/node/',
    title: 'Node.js Library - Segment',
    referrer: 'https://github.com/segmentio/analytics-node'
  }
});
</code></pre>
      <p>The <code>page</code> call has the following fields:</p>
      <table>
       <tbody>
        <tr>
         <td><code>userId</code> _String</td>
         <td>The ID for this user in your database.</td>
        </tr>
        <tr>
         <td><code>category</code> <em>String, optional</em></td>
         <td>The category of the page. Useful for things like ecommerce where many pages often live under a larger category.</td>
        </tr>
        <tr>
         <td><code>name</code> <em>String, optional</em></td>
         <td>The name of the of the page, for example <strong>Signup</strong> or <strong>Home</strong>.</td>
        </tr>
        <tr>
         <td><code>properties</code> <em>Object, optional</em></td>
         <td>A dictionary of properties of the page. A few properties specially recognized and automatically translated: <code>url</code>, <code>title</code>, <code>referrer</code> and <code>path</code>, but you can add your own too!</td>
        </tr>
        <tr>
         <td><code>timestamp</code> <em>Date, optional</em></td>
         <td>A Javascript date object representing when the track took place. If the track just happened, leave it out and we’ll use the server’s time. If you’re importing data from the past make sure you to send a <code>timestamp</code>.</td>
        </tr>
        <tr>
         <td><code>context</code> <em>Object, optional</em></td>
         <td>A dictionary of extra <a href="">context</a> to attach to the call. <em>Note: <code>context</code> differs from <code>traits</code> because it is not attributes of the user itself.</em></td>
        </tr>
        <tr>
         <td><code>anonymousId</code> <em>String, optional</em></td>
         <td>An ID to associated with the user when you don’t know who they are (eg., <a href="/docs/sources/website/analytics.js/#anonymous-id">the anonymousId generated by <code>analytics.js</code></a>)</td>
        </tr>
       </tbody>
      </table>
      <p>Find details on the <strong><code>page</code> payload</strong> in our <a href="/docs/spec/page/">Spec</a>.</p>
      <h2 id="group">Group</h2>
      <p><code>group</code> lets you associate an <a href="/docs/sources/server/node/#identify">identified user</a> with a group. A group could be a company, organization, account, project or team! It also lets you record custom traits about the group, like industry or number of employees.</p>
      <p>This is useful for tools like <a href="/docs/integrations/intercom/">Intercom</a>, <a href="/docs/integrations/preact/">Preact</a> and <a href="/docs/integrations/totango/">Totango</a>, as it ties the user to a <strong>group</strong> of other users.</p>
      <p>Example <code>group</code> call:</p>
      <pre><code class="lang-javascript">analytics.group({
  userId: '019mr8mf4r',
  groupId: '56',
  traits: {
    name: 'Initech',
    description: 'Accounting Software'
  }
});
</code></pre>
      <p>The <code>group</code> call has the following fields:</p>
      <table class="api-table">
       <tbody>
        <tr>
         <td><code>userId</code> _string</td>
         <td>The ID for the user that is a part of the group.</td>
        </tr>
        <tr>
         <td><code>groupId</code> _string</td>
         <td>The ID of the group.</td>
        </tr>
        <tr>
         <td><code>traits</code> <em>dict, optional</em></td>
         <td>A dict of traits you know about the group. For a company, they might be things like <code>name</code>, <code>address</code>, or <code>phone</code>.</td>
        </tr>
        <tr>
         <td><code>context</code> <em>dict, optional</em></td>
         <td>A dict containing any context about the request. To see the full reference of supported keys, check them out in the <a href="/docs/spec/common/#context">context reference</a></td>
        </tr>
        <tr>
         <td><code>timestamp</code> <em>datetime, optional</em></td>
         <td>A <code>datetime</code> object representing when the group took place. If the group just happened, leave it out and we’ll use the server’s time. If you’re importing data from the past make sure you send <code>timestamp</code>.</td>
        </tr>
        <tr>
         <td><code>anonymousId</code> <em>String, optional</em></td>
         <td>An ID to associated with the user when you don’t know who they are (eg., <a href="/docs/sources/website/analytics.js/#anonymous-id">the anonymousId generated by <code>analytics.js</code></a>)</td>
        </tr>
        <tr>
         <td><code>integrations</code> <em>dict, optional</em></td>
         <td>A dictionary of destinations to enable or disable</td>
        </tr>
       </tbody>
      </table>
      <p>Find more details about <code>group</code>, including the <strong><code>group</code> payload</strong>, in our <a href="/docs/spec/group/">Spec</a>.</p>
      <h2 id="alias">Alias</h2>
      <p><code>alias</code> is how you associate one identity with another. This is an advanced method, but it is required to manage user identities successfully in <em>some</em> of our destinations.</p>
      <p>In <a href="/docs/integrations/mixpanel/#alias">Mixpanel</a> it’s used to associate an anonymous user with an identified user once they sign up. For <a href="/docs/integrations/kissmetrics/#alias">KISSmetrics</a>, if your user switches IDs, you can use ‘alias’ to rename the ‘userId’.</p>
      <p>Example <code>alias</code> call:</p>
      <pre><code class="lang-javascript">analytics.alias({
  previousId: 'old_id',
  userId: 'new_id'
});
</code></pre>
      <p>The <code>alias</code> call has the following fields:</p>
      <table class="api-table">
       <tbody>
        <tr>
         <td><code>userId</code> <em>String</em></td>
         <td>The ID for this user in your database.</td>
        </tr>
        <tr>
         <td><code>previousId</code> <em>String</em></td>
         <td>The previous ID to alias from.</td>
        </tr>
       </tbody>
      </table>
      <p>Here’s a full example of how we might use the <code>alias</code> call:</p>
      <pre><code class="lang-javascript">// the anonymous user does actions ...
analytics.track({ userId: 'anonymous_user', event: 'Anonymous Event' })
// the anonymous user signs up and is aliased
analytics.alias({ previousId: 'anonymous_user', userId: 'identified@gmail.com' })
// the identified user is identified
analytics.identify({ userId: 'identified@gmail.com', traits: { plan: 'Free' } })
// the identified user does actions ...
analytics.track({ userId: 'identified@gmail.com', event: 'Identified Action' })
</code></pre>
      <p>For more details about <code>alias</code>, including the <strong><code>alias</code> call payload</strong>, check out our <a href="/docs/spec/alias/">Spec</a>.</p>

      <h2 id="configuration">Configuration</h2>
      <p>The second argument to the <code>Analytics</code> constructor is an optional dictionary of settings to configure the module.</p>
      <pre><code class="lang-javascript">var analytics = new Analytics('YOUR_WRITE_KEY', {
  flushAt: 20,
  flushInterval: 10000
});
</code></pre>
      <table>
       <tbody>
        <tr>
         <td><code>flushAt</code> <em>Number</em></td>
         <td>The number of messages to enqueue before flushing.</td>
        </tr>
        <tr>
         <td><code>flushInterval</code> <em>Number</em></td>
         <td>The number of milliseconds to wait before flushing the queue automatically.</td>
        </tr>
       </tbody>
      </table>
      <h2 id="development">Development</h2>
      <p>You can use this initialization during development to make the library flush every time a message is submitted, so that you can be sure your calls are working properly before pushing to production.</p>
      <pre><code class="lang-javascript">var analytics = new Analytics('YOUR_WRITE_KEY', { flushAt: 1 });
</code></pre>
      <h2 id="selecting-destinations">Selecting Destinations</h2>
      <p>The <code>alias</code>, <code>group</code>, <code>identify</code>, <code>page</code> and <code>track</code> calls can all be passed an object of <code>integrations</code> that lets you turn certain destinations on or off. By default all destinations are enabled.</p>
      <p>Here’s an example with the <code>integrations</code> object shown:</p>
      <pre><code class="lang-javascript">analytics.track({
  event: 'Membership Upgraded',
  userId: '97234974',
  integrations: {
    'All': false,
    'Vero': true,
    'Google Analytics': false
  }
})
</code></pre>
      <p>In this case, we’re specifying that we want this <code>track</code> to only go to Vero. <code>All: false</code> says that no destination should be enabled unless otherwise specified. <code>Vero: true</code> turns on Vero, etc.</p>
      <p>Destination flags are <strong>case sensitive</strong> and match <a href="/docs/integrations">the destination’s name in the docs</a> (i.e. “AdLearn Open Platform”, “awe.sm”, “MailChimp”, etc.).</p>
      <p><strong>Note:</strong></p>
      <ul>
       <li><p>Available at the business level, filtering track calls can be done right from the Segment UI on your source schema page. We recommend using the UI if possible since it’s a much simpler way of managing your filters and can be updated with no code changes on your side.</p></li>
       <li><p>If you are on a grandfathered plan, events sent server-side that are filtered through the Segment dashboard will still count towards your API usage.</p></li>
      </ul>
      <h2 id="historical-import">Historical Import</h2>
      <p>You can import historical data by adding the <code>timestamp</code> argument to any of your method calls. This can be helpful if you’ve just switched to Segment.</p>
      <p>Historical imports can only be done into destinations that can accept historical timestamp’ed data. Most analytics tools like Mixpanel, Amplitude, Kissmetrics, etc. can handle that type of data just fine. One common destination that does not accept historical data is Google Analytics since their API cannot accept historical data.</p>
      <p><strong>Note:</strong> If you’re tracking things that are happening right now, leave out the <code>timestamp</code> and our servers will timestamp the requests for you.</p>
      <h2 id="batching">Batching</h2>
      <p>Our libraries are built to support high performance environments. That means it is safe to use our Node library on a web server that’s serving hundreds of requests per second.</p>
      <p>Every method you call <strong>does not</strong> result in an HTTP request, but is queued in memory instead. Messages are then flushed in batch in the background, which allows for much faster operation.</p>
      <p>By default, our library will flush:</p>
      <ul>
       <li>The very first time it gets a message.</li>
       <li>Every 20 messages (controlled by <code>options.flushAt</code>).</li>
       <li>If 10 seconds has passed since the last flush (controlled by <code>options.flushInterval</code>)</li>
      </ul>
      <p>There is a maximum of <code>500kb</code> per batch request and <code>15kb</code> per call.</p>
      <p>If you don’t want to batch messages, you can turn batching off by setting the <code>flushAt</code> option to <code>1</code>, like so:</p>
      <pre><code class="lang-javascript">var analytics = new Analytics('YOUR_WRITE_KEY', { flushAt: 1 });
</code></pre>
      <p>Batching means that your message might not get sent right away. But every method call takes an optional <code>callback</code>, which you can use to know when a particular message is flushed from the queue, like so:</p>
      <pre><code class="lang-javascript">analytics.track({
  userId: '019mr8mf4r',
  event: 'Ultimate Played'
}, function(err, batch){
  if (err) // There was an error flushing your message...
  // Your message was successfully flushed!
});
</code></pre>
      <p>You can also flush on demand. For example, at the end of your program, you need to flush to make sure that nothing is left in the queue. To do that, call the <code>flush</code> method:</p>
      <pre><code class="lang-javascript">analytics.flush(function(err, batch){
  console.log('Flushed, and now this program can exit!');
});
</code></pre>
      <h2 id="multiple-clients">Multiple Clients</h2>
      <p>Different parts of your application may require different types of batching, or even sending to multiple Segment sources. In that case, you can initialize multiple instances of <code>Analytics</code> with different settings:</p>
      <pre><code class="lang-javascript">var Analytics = require('analytics-node');
var marketingAnalytics = new Analytics('MARKETING_WRITE_KEY');
var appAnalytics = new Analytics('APP_WRITE_KEY');
</code></pre>
      <h2 id="troubleshooting">Troubleshooting</h2>
      <p>If you’re having trouble we have a few tips that help common problems.</p>
      <h3 id="no-events-in-my-debugger">No events in my debugger</h3>
      <ol>
       <li><p>Double check that you’ve followed all the steps in the <a href="#getting-started">Quickstart</a>.</p></li>
       <li><p>Make sure that you’re calling one of our API methods once the library is successfully installed—<a href="#identify"><code>identify</code></a>, <a href="#track"><code>track</code></a>, etc.</p></li>
      </ol>
      <h3 id="no-events-in-my-end-tools">No events in my end tools</h3>
      <ol>
       <li><p>Double check your credentials for that destination.</p></li>
       <li><p>Make sure that the destination you are troubleshooting can accept server-side API calls. Compatibility is shown on the <a href="/docs/integrations/">destination docs</a> pages and on the sheets on your Segment source Destinations page.</p></li>
       <li><p>Check out the <a href="/docs/integrations/">destination’s documentation</a> to see if there are other requirements for using the method and destination you’re trying to get working.</p></li>
      </ol>

  /**
   * getObjectValues
   * Return an array of values for an object
   * @param  {Object} obj - an object
   * @return {Array}
   */
  function getObjectValues(obj) {
    var values = []
    for(var key in obj){
      if (obj.hasOwnProperty(key)) values.push(obj[key])
    }
    return values
  }

  /**
   * highlightScrolledTitles
   * Highlight 'active' H2 titles when they are scrolled within the top 100px
   * of the window.
   * @param  {Event} e - scroll
   */
  function highlightScrolledTitles(e) {
    sectionTitles.map(function (title) {
      var titlePosition = title.getBoundingClientRect().top
      var slug = title.getAttribute('id') // getting-started

      if (pageBody.scrollTop > lastScrollTop) {
        // scrolling down
        // find a navigation link with matching data-slug value + set it highlighted
        // cache this title in `cachedTitles`
        if (titlePosition <= 100) {
          resetHighlightedNavItems()
          highlightPageNavItemLinkBySlug(slug)
          cachedTitles[slug] = title
        }
      } else {
        // scrolling up
        // delete this title from `cachedTitles`
        // locate the previous title and find a navigation link with
        // matching data-slug value + set it highlighted
        if (titlePosition > 100 && cachedTitles[slug]) {
          delete cachedTitles[slug]
          resetHighlightedNavItems()
          var values = getObjectValues(cachedTitles)
          var previousTitle = values[values.length - 1]
          if (previousTitle) {
            var previousSlug = previousTitle.getAttribute('id')
            highlightPageNavItemLinkBySlug(previousSlug)
          }
        }
      }
    })

    lastScrollTop = pageBody.scrollTop
  }

  pageBody.addEventListener('scroll', highlightScrolledTitles, {passive: false})

  var menu = source.nextSibling
  if (!menu && !menu.classList) return
  menu.classList.add('visible')

  source.scrollIntoView()

  // check the position of the source. If flush (60) apply 100px offset
  if (source.getBoundingClientRect().top === 60) {
    var offset = 100
    var sideBar = document.querySelector('.Page-sidebar')
    sideBar.scrollTop = sideBar.scrollTop - offset
