             <span class="field-content"><a href="/blog/2013/02/11/willow-garage-changing">Willow Garage is changing</a></span> 
            </div> 
            <div class="views-field-created"> 
             <span class="field-content">February 11, 2013</span> 
            </div> 
            <div class="views-field-body"> 
             <div class="field-content">
              <p><span style="color: #222222; font-family: arial, sans-serif; font-size: 13px;">Willow Garage has decided to enter the world of commercial opportunities with an eye to becoming a self-sustaining company. This is an important change to our funding model.</span><br style="color: #222222; font-family: arial, sans-serif; font-size: 13px;"><br style="color: #222222; font-family: arial, sans-serif; font-size: 13px;"><span style="color: #222222; font-family: arial, sans-serif; font-size: 13px;">The success of the PR2 personal robot and of ROS will continue. There are close to 50 PR2 robots in the world and Willow Garage support of the platform will not diminish. And of course, ROS, as an open source platform, will continue independent of our business model choices. In addition to Willow Garage, its supporters include the <a href="http://www.osrfoundation.org">Open Source Robotics Foundation</a></span><span style="color: #222222; font-family: arial, sans-serif; font-size: 13px;">&nbsp;and all the other contributors in the ROS community (academic, industrial and individual) who have made it the platform of choice for Robotics.</span></p>

             <span class="field-content"><a href="/blog/2013/01/29/real-time-modular-3d-mapping">Real-Time Modular 3D Mapping</a></span> 
            </div> 
            <div class="views-field-created"> 
             <span class="field-content">January 29, 2013</span> 
            </div> 
            <div class="views-field-body"> 
             <div class="field-content">
              <p><iframe width="640" height="360" frameborder="0" src="http://www.youtube.com/embed/NbHnRqe8oLk?rel=0"></iframe></p> 
              <p>During his sabbatical at Willow Garage, Stéphane Magnenat from the <a href="http://www.asl.ethz.ch" title="http://www.asl.ethz.ch" target="_blank">Autonomous Systems Lab, ETH Zurich</a> integrated a new Simultaneous Localization and Mapping (SLAM) solution.&nbsp; SLAM allows a robot to build a map of its environment, and to localize itself in this map.&nbsp; This system is based on a modular ICP algorithm, a collaboration with François Pomerleau and Francis Colas at the Autonomous Systems Lab, ETH Zurich.</p>
              <p>The ICP is a classical algorithm to find a transformation between two point clouds, representing the same environment but from different viewpoints.&nbsp; This algorithm forms the basis of most SLAM systems working with laser or depth data.&nbsp; While the classical ICP algorithm is simple, it does not work well in most real 3D environments, and therefore hundreds of variants have been published in the last 20 years.&nbsp; These variants address specific problems, but lack a common comparison ground.</p>
              <p>The work of Stéphane and his colleagues provides a framework in which different ICP variants can be tested, combined, and evaluated.&nbsp; The integration with ROS provides a real-time 2D and 3D SLAM system that can fit a large variety of robots and application scenarios without any code change or recompilation.</p>
              <p>A paper describing this work will appear in the forthcoming "Autonomous Robots special issue on open source software for robotics research."&nbsp; A dataset paper proposing a variety of environments along with ground-truth poses was recently published in IJRR (link to <a href="http://ijr.sagepub.com/content/31/14/1705.abstract" title="http://ijr.sagepub.com/content/31/14/1705.abstract">http://ijr.sagepub.com/content/31/14/1705.abstract</a>) &nbsp;(datasets freely available (link to <a href="http://projects.asl.ethz.ch/datasets/doku.php?id=laserregistration:laserregistration" title="http://projects.asl.ethz.ch/datasets/doku.php?id=laserregistration:laserregistration">http://projects.asl.ethz.ch/datasets/doku.php?id=laserregistration:laser...</a>)).&nbsp; The open-source ROS SLAM node is available in the ethzasl_icp_mapping stack (link to <a href="http://ros.org/wiki/ethzasl_icp_mapping" title="http://ros.org/wiki/ethzasl_icp_mapping">http://ros.org/wiki/ethzasl_icp_mapping</a>).</p>
              <p>&nbsp;</p>

             <span class="field-content"><a href="/blog/2013/01/22/pr2-actuator-diagnostic-tool">PR2 Actuator Diagnostic Tool</a></span> 
            </div> 
            <div class="views-field-created"> 
             <span class="field-content">January 22, 2013</span> 
            </div> 
            <div class="views-field-body"> 
             <div class="field-content">
              <p><iframe width="640" height="360" frameborder="0" src="http://www.youtube.com/embed/uRg7E26ujyI"></iframe></p>
              <p>During his recent internship, Rahul Udasi, an undergraduate student at the <a href="http://uwaterloo.ca" title="http://uwaterloo.ca" target="_blank">University of Waterloo</a>&nbsp;created a motor diagnostic tool for the PR2. The tool allows PR2 users to find motor faults quickly by getting diagnostic data from the motors along with analysis of the collected information. The results are presented in graphs and messages that may indicate what might be wrong with the PR2's joint motors, all without having to remove any covers.&nbsp;</p>
              <p>The first step in the tool is to select which joint motors the users wants to diagnose. Next, the user moves the joints for the specific motors for about 4 seconds to record enough data then moves onto the next joint. For convenience, the PS3 joystick is used to indicate when the user is done with one joint and when to move to the next. After the data is recorded, the user has access to comprehensive logs and analysis of motor performance. If something looks wrong, the software will tell the user where to look for issues with the joint motor.&nbsp;</p>
              <p>By using the tool, users can quickly diagnose motor faults without the need to remove the motors from the assembly, a time-consumering and specialized task.&nbsp;</p>
              <p>For more information please visit the <a href="http://www.ros.org/wiki/pr2_motor_diagnostic_tool" title="http://www.ros.org/wiki/pr2_motor_diagnostic_tool" target="_blank">PR2 motor diagnostic page</a>.</p>

             <span class="field-content"><a href="/blog/2013/01/17/improving-perception-pcl">Improving Perception in PCL</a></span> 
            </div> 
            <div class="views-field-created"> 
             <span class="field-content">January 17, 2013</span> 
            </div> 
            <div class="views-field-body"> 
             <div class="field-content">
              <p><iframe src="http://www.youtube.com/embed/ypH0J_yU-8U" frameborder="0" height="360" width="640"></iframe></p> 
              <p>Stefan Holzer of <a target="_blank" title="http://www.tum.de/" href="http://www.tum.de/">TU Munich</a> spent last spring working on including new algorithms into the <a target="_blank" title="http://pointclouds.org/" href="http://pointclouds.org/">Point Cloud Library</a> (PCL). His main focus was on creating a recognition and machine learning library in PCL.&nbsp;</p>
              <p>With respect to the recognition library, Stefan assisted in implementing LINEMOD [1], a highly efficient template-matching approach for detecting texture-less objects in heavily cluttered scenes. LINEMOD is based on Kinect data. For detecting objects, it uses color gradients computed from image data, as well as surface normals, computed from depth data.</p>
              <p>Within the machine learning library, Stefan worked on implementing a flexible and efficient decision tree learning framework. This was applied to obtain keypoint detectors with improved repeatability and efficiency. Additionally, tools were created to evaluate existing detectors and to create the data for learning.</p>
              <p>Having a state-of-the-art object detection method in PCL enables efficient detection of texture-less objects with kinect data. Thanks to the new learning framework there is now the ability to create keypoint detectors with improved detection characteristics, which are useful for localization and object detection.</p>

             <span class="field-content"><a href="/blog/2013/01/15/congratulations-ompl-winning-open-source-software-world-challenge">Congratulations To OMPL For Winning The Open Source Software World Challenge</a></span> 
            </div> 
            <div class="views-field-created"> 
             <span class="field-content">January 15, 2013</span> 
            </div> 
            <div class="views-field-body"> 
             <div class="field-content">
              <p>Willow Garage would like to congratulate the team behind the <a href="http://ompl.kavrakilab.org/">Open Motion Planning Library (OMPL)</a>, which was recently awarded the grand prize from the Open Source Software World Challenge. The OSSWC is the annual competition hosted by the <a href="http://www.mke.go.kr/language/eng/index.jsp">Ministry of Knowledge and Economy of Korea</a> with the goal of promoting open source software and expanding exchanges among open source software developers all over the world. Fifty-five teams from twenty-three countries participated in this year's competition with OMPL coming out on top.</p> 
              <p>OMPL is used for motion planning in ROS, as part of the <a href="http://www.ros.org/wiki/arm_navigation">arm navigation</a> stack and as part of the upcoming release of <a href="http://moveit.ros.org">MoveIt!</a></p> 
              <p>OMPL is developed and maintained by the <a href="http://www.kavrakilab.org/">Physical and Biological Computing Group</a> at Rice University, led by <a href="http://www.cs.rice.edu/~kavraki">Dr. Lydia Kavraki</a>. The development is coordinated by <a href="http://www.cs.rice.edu/~mmoll">Dr. Mark Moll</a>, Dr. Lydia Kavraki (Rice) and <a href="http://ioan.sucan.ro/">Dr. Ioan ?ucan</a>.</p> 
              <p>Willow Garage is proud of our role in supporting OMPL, particularly through the contributions of Dr. Ioan&nbsp;?ucan, <a href="/pages/people/sachin-chitta-research-scientist">Dr. Sachin Chitta</a> and <a href="/pages/people/e-gil-jones">Dr. Gil Jones</a>. The award-winning code is based on an initial version of OMPL written by Ioan ?ucan while he was an intern at Willow Garage. Sachin, Gil and Ioan presented aspects of OMPL at <a href="http://roscon.ros.org/2012/">ROSCon 2012</a>.</p>

             <span class="field-content"><a href="/blog/2013/01/11/unsupervised-object-discovery-megaworldmodel">Unsupervised Object Discovery With Megaworldmodel</a></span> 
            </div> 
            <div class="views-field-created"> 
             <span class="field-content">January 11, 2013</span> 
            </div> 
            <div class="views-field-body"> 
             <div class="field-content">
              <p><iframe src="http://www.youtube.com/embed/i5TrXEg3CGw" frameborder="0" height="360" width="640"></iframe></p>
              <p>During his summer internship at Willow Garage, Julian "Mac" Mason, a Ph.D. candidate from <a target="_blank" title="http://www.google.com/url?sa=t&amp;rct=j&amp;q=&amp;esrc=s&amp;source=web&amp;cd=1&amp;cad=rja&amp;sqi=2&amp;ved=0CDEQFjAA&amp;url=http%3A%2F%2Fwww.duke.edu%2F&amp;ei=bRreUMOAFe7yiQLYtYD4Bw&amp;usg=AFQjCNGLnHgAKc9wpb4SQmaqtAgyrGZtvg&amp;bvm=bv.1355534169,d.cGE" href="http://www.google.com/url?sa=t&amp;rct=j&amp;q=&amp;esrc=s&amp;source=web&amp;cd=1&amp;cad=rja&amp;sqi=2&amp;ved=0CDEQFjAA&amp;url=http%3A%2F%2Fwww.duke.edu%2F&amp;ei=bRreUMOAFe7yiQLYtYD4Bw&amp;usg=AFQjCNGLnHgAKc9wpb4SQmaqtAgyrGZtvg&amp;bvm=bv.1355534169,d.cGE">Duke University</a>, worked on the <a target="_blank" title="http://www.ros.org/wiki/megaworldmodel" href="http://www.ros.org/wiki/megaworldmodel">Megaworldmodel</a>: a framework for large-scale, long-term semantic maps.&nbsp; In constrast to occupancy maps (which model free and occupied space), semantic maps model the location of objects, and (when possible) their identities.&nbsp; Determining these identities is difficult: object recognition remains an open problem.&nbsp; For this reason, the Megaworldmodel provides a generic interface to object recognition systems, allowing existing tools to be easily integrated.&nbsp; Two such tools have already been included: Willow Garage's <a target="_blank" title="http://www.ros.org/wiki/textured_object_detection" href="http://www.ros.org/wiki/textured_object_detection">textured_object_detection</a>, and Hilton Bristow's implementation of Deva Ramanan's <a target="_blank" title="https://github.com/wg-perception/PartsBasedDetector" href="https://github.com/wg-perception/PartsBasedDetector">deformable-parts model</a>.</p>
              <p>The Megaworldmodel cleanly encapsulates the capture, processing, and mapping of recognizable objects, and the querying of the resulting map.&nbsp; However, not all objects are recognizable! State-of-the-art object recognition algorithms require extensive supervised training to accurately recognize objects.&nbsp; In large, general environments, manual training is intractable.&nbsp; There are simply too many objects.&nbsp; To enable large-scale semantic mapping, the Megaworldmodel includes tools for active object search (using a Kinect-equipped PR2) and for unsupervised object discovery.&nbsp; While autonomously exploring an environment, the robot will encounter objects (which it cannot yet recognize) from many different viewpoints.&nbsp; Using unsupervised segmentation, these objects can be detected, and then clustered into training examples for existing object recognition techniques.&nbsp; Although this does not provide semantic labels (you get "object 6," not "coffee cup"), it does allow object instances to be recognized in other locations, and at other times.&nbsp; Ongoing work seeks to scale this technique to extremely large datasets, permitting the entirely unsupervised creation of a large object database.</p>
              <p>More information about megaworld is available <a target="_blank" title="http://www.ros.org/wiki/megaworldmodel" href="http://www.ros.org/wiki/megaworldmodel">here</a>.</p>

              <p class="line867" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">&nbsp;</p>
              <h3 id="Mass_migration_of_code_to_GitHub" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 1.375em; font-weight: normal; counter-reset: subsection 0; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">Mass migration of code to GitHub</h3>
              <p><span class="anchor" id="line-23" style="font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; font-size: 15px;"></span></p>
              <p class="line874" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">Traditionally, ROS code has been scattered across numerous version control systems (git, svn, hg, etc) across different hosting services throughout the world. Though the ROS wiki has acted as a central point of documentation, issue/ticket tracking has been just as disparate as the usage of VCS tools.<span class="anchor" id="line-24"></span><span class="anchor" id="line-25"></span></p>
              <p class="line862" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">With ROS Groovy, an effort has been made to move core packages to&nbsp;<a class="http" href="http://github.com/" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">GitHub</a>&nbsp;along with all issue tracking. This has brought several benefits including making ROS more available to the wider open source community and providing VCS consistency for ROS packages. Most importantly, utilizing GitHub has involved the ROS community more and given it more ownership of the codebase. GitHub's pull requests have made it much easier for the core ROS development team to apply patches from the community as well as respond to design feedback more rapidly. In the last development cycle there have been 71 people who have contributed to the rosdistro file, using 239 pull requests and there exist 115 public forks of the rosdistro project on GitHub. The ease of forking and submitting pull requests has made this higher level of community engagement possible. GitHub additionally provides excellent tools for navigating code, managing issues/milestones, and increasing developer communication. All ROS package developers are highly encouraged, but not required, to utilize GitHub for their own repositories so as to further unify both the ROS codebase and community.<span class="anchor" id="line-26"></span><span class="anchor" id="line-27"></span></p>
              <p class="line862" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">On GitHub, ROS repositories are spread across several GitHub "organizations" to group related repositories in an intuitive manner:<span class="anchor" id="line-28"></span><span class="anchor" id="line-29"></span></p>
              <ul style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 20px; border-image: initial; font-size: 15px; list-style-position: outside; list-style-image: initial; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; background-repeat: no-repeat repeat; padding: 0px; border: 0px initial initial;">
               <li style="border-image: initial; font-size: 1em; padding: 0px; margin: 0px; border: 0px initial initial;"><p class="line891" style="margin-top: 0.25em; margin-right: 0px; margin-bottom: 0.25em; margin-left: 0px; border-image: initial; font-size: 1em; padding: 0px; border: 0px initial initial;"><a class="https" href="https://github.com/ros-infrastructure" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">ROS Infrastructure</a>&nbsp;-&nbsp;<em>Foundational tools for ROS ecosystem such as package release (<a href="http://www.ros.org/wiki/bloom" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">bloom</a>), build farm scripts, documentation generators. Most of these are backend, non-user facing repositories.</em><span class="anchor" id="line-30"></span></p></li>
               <li style="border-image: initial; font-size: 1em; padding: 0px; margin: 0px; border: 0px initial initial;"><p class="line891" style="margin-top: 0.25em; margin-right: 0px; margin-bottom: 0.25em; margin-left: 0px; border-image: initial; font-size: 1em; padding: 0px; border: 0px initial initial;"><a class="https" href="https://github.com/ros" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">ROS Core</a>&nbsp;-&nbsp;<em>Core ROS libraries and tools such as standard messages, the build system&nbsp;<a href="http://www.ros.org/wiki/catkin" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">catkin</a>, the ROS clients<a href="http://www.ros.org/wiki/roscpp" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">roscpp</a>&nbsp;and&nbsp;<a href="http://www.ros.org/wiki/rospy" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">rospy</a>,&nbsp;<a href="http://www.ros.org/wiki/actionlib" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">actionlib</a>,&nbsp;<a href="http://www.ros.org/wiki/pluginlib" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">pluginlib</a>,&nbsp;<a href="http://www.ros.org/wiki/nodelet" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">nodelet</a>, etc</em><span class="anchor" id="line-31"></span></p></li>
               <li style="border-image: initial; font-size: 1em; padding: 0px; margin: 0px; border: 0px initial initial;"><p class="line891" style="margin-top: 0.25em; margin-right: 0px; margin-bottom: 0.25em; margin-left: 0px; border-image: initial; font-size: 1em; padding: 0px; border: 0px initial initial;"><a class="https" href="https://github.com/ros-visualization" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">ROS Visualization</a>&nbsp;-&nbsp;<em>Graphical tools to visualize data including&nbsp;<a href="http://www.ros.org/wiki/rviz" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">rviz</a>&nbsp;and&nbsp;<a href="http://www.ros.org/wiki/rqt" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">rqt</a>.</em><span class="anchor" id="line-32"></span></p></li>
               <li style="border-image: initial; font-size: 1em; padding: 0px; margin: 0px; border: 0px initial initial;"><p class="line891" style="margin-top: 0.25em; margin-right: 0px; margin-bottom: 0.25em; margin-left: 0px; border-image: initial; font-size: 1em; padding: 0px; border: 0px initial initial;"><a class="https" href="https://github.com/ros-perception" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">ROS Perception</a>&nbsp;-&nbsp;<em>Tools for perception and vision</em><span class="anchor" id="line-33"></span></p></li>
               <li style="border-image: initial; font-size: 1em; padding: 0px; margin: 0px; border: 0px initial initial;"><p class="line891" style="margin-top: 0.25em; margin-right: 0px; margin-bottom: 0.25em; margin-left: 0px; border-image: initial; font-size: 1em; padding: 0px; border: 0px initial initial;"><a class="https" href="https://github.com/ros-drivers" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">ROS Drivers</a>&nbsp;-&nbsp;<em>Drivers for sensors and other hardware</em><span class="anchor" id="line-34"></span></p></li>
               <li style="border-image: initial; font-size: 1em; padding: 0px; margin: 0px; border: 0px initial initial;"><p class="line891" style="margin-top: 0.25em; margin-right: 0px; margin-bottom: 0.25em; margin-left: 0px; border-image: initial; font-size: 1em; padding: 0px; border: 0px initial initial;"><a class="https" href="https://github.com/ros-planning" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">ROS Planning</a>&nbsp;-&nbsp;<em>Navigation and path planning related packages including&nbsp;<a href="http://www.ros.org/wiki/navigation" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">navigation</a>&nbsp;and&nbsp;<a href="http://www.ros.org/wiki/moveit" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">moveit</a></em></p></li>
              </ul>
              <p class="line867" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">&nbsp;</p>
              <h3 id="New_Build_System_-_catkin" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 1.375em; font-weight: normal; counter-reset: subsection 0; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">New Build System - catkin</h3>
              <p><span class="anchor" id="line-39" style="font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; font-size: 15px;"></span></p>
              <p class="line862" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">The most striking and significant change in Groovy is the introduction of a new build system called&nbsp;<a href="http://www.ros.org/wiki/catkin" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">catkin</a>&nbsp;which will eventually fully replace the original&nbsp;<a href="http://www.ros.org/wiki/rosbuild" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">rosbuild</a>&nbsp;build system.&nbsp;<a href="http://www.ros.org/wiki/catkin" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">catkin</a>&nbsp;was developed to address several drawbacks in<a href="http://www.ros.org/wiki/rosbuild" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">rosbuild</a>&nbsp;so that ROS can continue to grow and scale. As opposed to rosbuild&nbsp;<a href="http://www.ros.org/wiki/catkin" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">catkin</a>&nbsp;promotes Filesystem Hierarchy Standard (FHS) compliance, making it much easier to distribute ROS packages on other operating systems and architectures.<span class="anchor" id="line-40"></span><span class="anchor" id="line-41"></span></p>
              <p class="line862" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">Though a prototype version was already introduced in&nbsp;<a href="http://www.ros.org/wiki/fuerte" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">fuerte</a>&nbsp;for a only few packages, catkin is now the official build system of ROS.&nbsp;<a href="http://www.ros.org/wiki/rosbuild" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">rosbuild</a>&nbsp;will continue to be supported for the forseeable future, but in time will be deprecated. catkin has been designed to be backwards compatible with existing&nbsp;<a href="http://www.ros.org/wiki/rosbuild" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">rosbuild</a>&nbsp;packages and stacks that have not yet migrated. Most of the core packages have already been migrated and new packages should utilize&nbsp;<a href="http://www.ros.org/wiki/catkin" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">catkin</a>.<span class="anchor" id="line-42"></span><span class="anchor" id="line-43"></span></p>
              <p class="line862" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">The workflow of&nbsp;<a href="http://www.ros.org/wiki/catkin" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">catkin</a>&nbsp;is somewhat different from&nbsp;<a href="http://www.ros.org/wiki/rosbuild" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">rosbuild</a>, but great care has been taken to make the transition as smooth as possible. Here are some resources for getting started with&nbsp;<a href="http://www.ros.org/wiki/catkin" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">catkin</a>:<span class="anchor" id="line-44"></span><span class="anchor" id="line-45"></span></p>
              <ul style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 20px; border-image: initial; font-size: 15px; list-style-position: outside; list-style-image: initial; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; background-repeat: no-repeat repeat; padding: 0px; border: 0px initial initial;">
               <li style="border-image: initial; font-size: 1em; padding: 0px; margin: 0px; border: 0px initial initial;"><p class="line891" style="margin-top: 0.25em; margin-right: 0px; margin-bottom: 0.25em; margin-left: 0px; border-image: initial; font-size: 1em; padding: 0px; border: 0px initial initial;"><a href="http://www.ros.org/wiki/catkin" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">catkin</a>&nbsp;-&nbsp;<em>Main wiki page</em><span class="anchor" id="line-46"></span></p></li>
               <li style="border-image: initial; font-size: 1em; padding: 0px; margin: 0px; border: 0px initial initial;"><p class="line891" style="margin-top: 0.25em; margin-right: 0px; margin-bottom: 0.25em; margin-left: 0px; border-image: initial; font-size: 1em; padding: 0px; border: 0px initial initial;"><a href="http://www.ros.org/wiki/catkin/conceptual_overview" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">catkin/conceptual_overview</a>&nbsp;-&nbsp;<em>Reasoning behind the move from rosbuild to catkin</em><span class="anchor" id="line-47"></span></p></li>
               <li style="border-image: initial; font-size: 1em; padding: 0px; margin: 0px; border: 0px initial initial;"><p class="line891" style="margin-top: 0.25em; margin-right: 0px; margin-bottom: 0.25em; margin-left: 0px; border-image: initial; font-size: 1em; padding: 0px; border: 0px initial initial;"><a href="http://www.ros.org/wiki/catkin/Tutorials" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">catkin/Tutorials</a>&nbsp;-&nbsp;<em>Tutorials</em><span class="anchor" id="line-48"></span></p></li>
               <li style="border-image: initial; font-size: 1em; padding: 0px; margin: 0px; border: 0px initial initial;"><p class="line891" style="margin-top: 0.25em; margin-right: 0px; margin-bottom: 0.25em; margin-left: 0px; border-image: initial; font-size: 1em; padding: 0px; border: 0px initial initial;"><a href="http://www.ros.org/wiki/catkin/migrating_from_rosbuild" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">catkin/migrating_from_rosbuild</a>&nbsp;-&nbsp;<em>Migrating from rosbuild to catkin</em><span class="anchor" id="line-49"></span><span class="anchor" id="line-50"></span></p></li>
              </ul>
              <p class="line867" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">&nbsp;</p>
              <h4 id="Removal_of_Stacks" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-weight: normal; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">Removal of Stacks</h4>
              <p><span class="anchor" id="line-51" style="font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; font-size: 15px;"></span></p>
              <p class="line874" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">With the new version of catkin in Groovy, the concept of Stacks has been removed. One of the main reasons for this was that there was a duplication of dependency tracking between packages and stacks. Previously packages only depend on packages, and stacks only depend on stacks, so the stack which contains a package must depend on all of the stacks which contain the dependencies of that package. Additionally, by removing stacks the code base has become more modular because you can install any package and its dependencies, where as in the past a package would often unnecessarily pull in its stack and other unused dependencies.<span class="anchor" id="line-52"></span><span class="anchor" id="line-53"></span></p>
              <p class="line874" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">To preserve some of the benefits of stack, the concept of "metapackages" replaces the concept of stacks. A metapackage is not a container of packages like rosbuild stacks were, but a named set of references to packages. It can be used to preserve some structure using a catkin metapackage replacing an old rosbuild stacks (this helps with for backwards compatibility of documentation with rosbuild, e.g. in wikis). A metapackage can depend on packages which don't necessarily reside in the same folder or source repository. This allows for more flexibility of development workflows, allowing to have separate repositories for packages that belonged to a single stack.<span class="anchor" id="line-54"></span><span class="anchor" id="line-55"></span></p>
              <p class="line862" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">This is captured more formally in&nbsp;<a class="http" href="http://ros.org/reps/rep-0127.html" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">REP 127</a><span class="anchor" id="line-56"></span><span class="anchor" id="line-57"></span><span class="anchor" id="line-58"></span></p>
              <p class="line867" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">&nbsp;</p>
              <h3 id="New_Package_Release_System_-_bloom" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 1.375em; font-weight: normal; counter-reset: subsection 0; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">New Package Release System - bloom</h3>
              <p><span class="anchor" id="line-59" style="font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; font-size: 15px;"></span></p>
              <p class="line862" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">With catkin also comes a new package release system known as&nbsp;<a href="http://www.ros.org/wiki/bloom" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">bloom</a>.&nbsp;<a href="http://www.ros.org/wiki/bloom" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">bloom</a>&nbsp;not only simplifies your workflow when releasing ROS packages to the world, but also provides features to make it easier to maintain released packages, version them, and patch/backport changes.<span class="anchor" id="line-60"></span><span class="anchor" id="line-61"></span></p>
              <p class="line867" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;"><a href="http://www.ros.org/wiki/bloom" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">bloom</a>&nbsp;takes your catkin packages and releases them to a "release repository". All of the ROS release repositories are hosted in the ros-gbp organization on&nbsp;<a class="https" href="https://github.com/" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">https://github.com</a>:&nbsp;<a class="https" href="https://github.com/ros-gbp" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">https://github.com/ros-gbp</a>. The release repository contains snapshots of your source repository (the repository you develop on) and provides tags to the snapshots. Additionally it creates a "release" branch for each package in your upstream repository on which you can make patches and it provides tags for each released version of your packages. Finally,&nbsp;<a href="http://www.ros.org/wiki/bloom" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">bloom</a>&nbsp;can generate files for building Debian .debs on the ROS build farm. Because we host our release repositories on&nbsp;<a class="https" href="https://github.com/" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">https://github.com</a>, there are known urls for each release of each package. This makes the release repositories a good source for building released ros packages from source.<span class="anchor" id="line-62"></span><span class="anchor" id="line-63"></span></p>
              <p class="line867" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;"><a href="http://www.ros.org/wiki/bloom" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">bloom</a>&nbsp;allows a source code repository to contain any number of packages, related or not. The only caveat to having multiple packages in a single repository is that they must be released with a common version number, i.e. one release version number per repository.<span class="anchor" id="line-64"></span><span class="anchor" id="line-65"></span><span class="anchor" id="line-66"></span></p>
              <p class="line867" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">&nbsp;</p>
              <h3 id="New_GUI_Tools_-_rqt" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 1.375em; font-weight: normal; counter-reset: subsection 0; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">New GUI Tools - rqt</h3>
              <p><span class="anchor" id="line-67" style="font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; font-size: 15px;"></span></p>
              <p class="line862" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">In Groovy, core ROS GUI tools (rxconsole, rxgraph, etc) have been significantly refined. These tools have been replaced by a single new tool called&nbsp;<a href="http://www.ros.org/wiki/rqt" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">rqt</a>. rqt is a software framework that implements the various GUI tools in the form of plugins. One can run all the existing GUI tools as dockable windows within rqt -- even&nbsp;<a href="http://www.ros.org/wiki/rviz" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">rviz</a>! The tools can still run in a traditional standalone method, but&nbsp;<a href="http://www.ros.org/wiki/rqt" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">rqt</a>&nbsp;makes it easier to manage all the various windows on the screen at one moment.<span class="anchor" id="line-68"></span><span class="anchor" id="line-69"></span></p>
              <p class="line862" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">Users can create their own plugins for&nbsp;<a href="http://www.ros.org/wiki/rqt" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">rqt</a>&nbsp;with either Python or C++.&nbsp;<a href="http://www.ros.org/wiki/rqt/Plugins" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">Over 20 plugins</a>&nbsp;have already been created and more are slated for development.<span class="anchor" id="line-70"></span><span class="anchor" id="line-71"></span></p>
              <p class="line867" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">&nbsp;</p>
              <h3 id="rviz" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 1.375em; font-weight: normal; counter-reset: subsection 0; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">rviz</h3>
              <p><span class="anchor" id="line-72" style="font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; font-size: 15px;"></span></p>
              <p class="line867" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;"><a href="http://www.ros.org/wiki/rviz" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">rviz</a>&nbsp;has been significantly redesigned with a cleaner user interface and and a new plugin API covering all major functions. RViz is now "on its own" and so can be installed directly (on Ubuntu for example) as ros-groovy-rviz. It no longer depends on visualization_common, instead depends directly on standard OS installations of OGRE, easing compilation from source. Python support has been greatly expanded, so rviz windows, displays, viewpoints, and tools can all be manipulated from Python now.<span class="anchor" id="line-73"></span><span class="anchor" id="line-74"></span></p>
              <p class="line874" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">The "save" file format has also changed: it now saves into "*.rviz" files which are in YAML format, easing manual editing, comparison, and automatic generation.<span class="anchor" id="line-75"></span><span class="anchor" id="line-76"></span></p>
              <p class="line867" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">&nbsp;</p>
              <h3 id="pluginlib_and_class_loader" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 1.375em; font-weight: normal; counter-reset: subsection 0; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">pluginlib and class_loader</h3>
              <p><span class="anchor" id="line-77" style="font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; font-size: 15px;"></span></p>
              <p class="line867" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;"><a href="http://www.ros.org/wiki/pluginlib" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">pluginlib</a>, the library for creating plugins in C++ code, has been almost completely rewritten while retaining 100% backwards compatibility. New features include thread-safety, simplified API, and true library class introspection.&nbsp;<a href="http://www.ros.org/wiki/pluginlib" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">pluginlib</a>is now built on top of a new ROS-independent package called&nbsp;<a href="http://www.ros.org/wiki/class_loader" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">class_loader</a>&nbsp;which can be used to work with plugins in software that does not utilize the ROS build system.<span class="anchor" id="line-78"></span><span class="anchor" id="line-79"></span></p>

              <h3 id="Tutorials" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 1.375em; font-weight: normal; counter-reset: subsection 0; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">Tutorials</h3>

It is recommended to review them again, even if you are a ROS veteran, at&nbsp;<a class="http" href="http://www.ros.org/wiki/ROS/Tutorials" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">http://www.ros.org/wiki/ROS/Tutorials</a>.               <h3 id="Automatic_Documentation_Jobs" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 1.375em; font-weight: normal; counter-reset: subsection 0; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">Automatic Documentation Jobs</h3>
              <p><span class="anchor" id="line-93" style="font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; font-size: 15px;"></span><span class="anchor" id="line-94" style="font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; font-size: 15px;"></span></p>
              <p class="line862" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">One of the valuable services provided to the ROS community is the public indexing of packages and automatic documentation generation onto ros.org. The entire system has been overhauled to parallelize and isolate failures of individual repositories from stopping other repositories from being documented. The new infrastructure now also provides properly versioned documentation for each ROS distro. For more information on how to be indexed in the new system see the&nbsp;<a class="nonexistent" href="http://www.ros.org/wiki/Get%2520Involved#Documenting_Your_.2A-ros-pkg_Repository_on_ROS.org" style="color: gray; border-image: initial; border: 0px initial initial;">Get Involved page.</a><span class="anchor" id="line-95"></span><span class="anchor" id="line-96"></span><span class="anchor" id="line-97"></span><span class="anchor" id="line-98"></span></p>

              <h2 id="Migrating" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-weight: normal; counter-reset: section 0; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">Migrating</h2>
              <p><span class="anchor" id="line-99" style="font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; font-size: 15px;"></span><span class="anchor" id="line-100" style="font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; font-size: 15px;"></span></p>
              <p class="line874" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">As with any release there have been some areas which will require updating. In this cycle most of the changes have been adding APIs while maintaining backwards compatibility for the old APIs.<span class="anchor" id="line-101"></span><span class="anchor" id="line-102"></span></p>
              <p class="line867" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">&nbsp;</p>
              <h3 id="Moving_From_rosbuild_To_catkin" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 1.375em; font-weight: normal; counter-reset: subsection 0; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">Moving From rosbuild To catkin</h3>
              <p><span class="anchor" id="line-103" style="font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; font-size: 15px;"></span><span class="anchor" id="line-104" style="font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; font-size: 15px;"></span></p>
              <p class="line862" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">The biggest change is the switch from&nbsp;<a href="http://www.ros.org/wiki/rosbuild" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">rosbuild</a>&nbsp;to&nbsp;<a href="http://www.ros.org/wiki/catkin" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">catkin</a>. This switch has been done while providing backwards compatibility, such that legacy packages can depend on converted packages without being changed. However catkin based packages cannot depend on rosbuild based packages. This leads to what has been called the rising tide where the conversion to catkin must propogate up the dependency tree. Wet packages, catkin based, are always below the dry packages, rosbuild based, as the tide rises.<span class="anchor" id="line-105"></span><span class="anchor" id="line-106"></span></p>
              <p class="line862" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">A detailed guide to converting packages can be found in the catkin documentation.&nbsp;<a href="http://www.ros.org/wiki/catkin/migrating_from_rosbuild" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">catkin/migrating_from_rosbuild</a><span class="anchor" id="line-107"></span><span class="anchor" id="line-108"></span></p>
              <p class="line867" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">&nbsp;</p>

              <p><span class="anchor" id="line-109" style="font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; font-size: 15px;"></span><span class="anchor" id="line-110" style="font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; font-size: 15px;"></span></p>
              <p class="line862" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">One significant change is the transition from Wx to Qt. If you have been using Wx graphical toolkits it is recommended to switch. Most of our primary visualization tools have been ported to Qt. And there are&nbsp;<a href="http://www.ros.org/wiki/rqt/Plugins" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">many plugins for them.</a>&nbsp;There are tutorials on the [[rqt] page on how to write new Qt based plugins.<span class="anchor" id="line-111"></span><span class="anchor" id="line-112"></span></p>

              <p><span class="anchor" id="line-127" style="font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; font-size: 15px;"></span><span class="anchor" id="line-128" style="font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; font-size: 15px;"></span></p>
              <p class="line862" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">The planning for the Groovy release was coordinated on the&nbsp;<a href="http://www.ros.org/wiki/groovy/Planning" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">planning page</a>.<span class="anchor" id="line-129"></span><span class="anchor" id="line-130"></span></p>
              <p class="line867" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">&nbsp;</p>

              <p><span class="anchor" id="line-131" style="font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; font-size: 15px;"></span><span class="anchor" id="line-132" style="font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; font-size: 15px;"></span></p>
              <p class="line862" style="margin-top: 0px; margin-right: 0px; margin-bottom: 0.75em; margin-left: 0px; border-image: initial; font-size: 15px; font-family: Helvetica, Verdana, Arial, 'Lucida Grande', sans-serif; padding: 0px; border: 0px initial initial;">A significant amount of the Groovy development was coordinated by Special Interest Groups. These groups are summarized at the bottom of the&nbsp;<a href="http://www.ros.org/wiki/groovy/Planning#SIGs" style="color: #2b7fcf; border-image: initial; border: 0px initial initial;">planning page</a>.<span class="anchor" id="line-133"></span><span class="anchor" id="line-134"></span></p>

            <div class="views-field-title"> 
             <span class="field-content"><a href="/blog/2012/12/11/exploring-ros-turtlecore">Exploring ROS With The TurtleCore</a></span> 
            </div> 
            <div class="views-field-created"> 
             <span class="field-content">December 11, 2012</span> 
            </div> 
            <div class="views-field-body"> 
             <div class="field-content">
              <p><iframe src="http://www.youtube.com/embed/cQ8nbz8Yupg" width="640" frameborder="0" height="360"></iframe></p> 
              <p>During his internship at Willow Garage, Rob Linsalata from <a target="_blank" title="http://www.tufts.edu" href="http://www.tufts.edu">Tufts University</a> worked on running ROS on small, low power ARM-based processors. &nbsp;Rob focused on driving the <a target="_blank" title="http://turtlebot.com" href="http://turtlebot.com">TurtleBot</a> around using the TurtleCore -- a small embedded computer produced by <a target="_blank" title="https://www.gumstix.com/store/product_info.php?products_id=280" href="https://www.gumstix.com/store/product_info.php?products_id=280">Gumstix</a>.&nbsp;</p>
              <p>Rob worked to integrate the TurtleCore with the TurtleBot throughout his time at Willow Garage. &nbsp;After initially integrating the TurtleCore and TurtleBot he focused on getting ROS running on the ARM processor in the TurtleCore. &nbsp;He has provided documentation and procedures to bring up the TurtleCore on the Overo ARM processor.&nbsp; Rob then helped test and debug ROS functionality running on the ARM architecture. &nbsp;</p>
              <p>The TurtleBot comes by default with an Asus netbook. &nbsp;The netbook accounts for a significant fraction of the computation, cost, and power consumption when the TurtleBot is operating. &nbsp;By using a smaller ARM-based processor the TurtleBots can be made to run longer&nbsp; using less expensive parts.&nbsp;</p>
              <p>For more details, see <a target="_blank" title="http://ros.org/wiki/TurtleCore" href="http://ros.org/wiki/TurtleCore">ros.org/wiki/TurtleCore</a></p>
             </div>

             <span class="field-content"><a href="/blog/2012/12/04/advancements-connecting-ros-web-rosbridge-suite">Advancements In Connecting To ROS Via Web With The Rosbridge Suite</a></span> 
            </div> 
            <div class="views-field-created"> 
             <span class="field-content">December 5, 2012</span> 
            </div> 
            <div class="views-field-body"> 
             <div class="field-content">
              <p><iframe src="http://www.youtube.com/embed/aTfhRYMWhHM" frameborder="0" height="360" width="640"></iframe></p>
              <p>During his internship at Willow Garage, Jonathan Mace from <a target="_blank" title="http://robotics.cs.brown.edu" href="http://robotics.cs.brown.edu">Brown University</a> worked on building an industrial strength successor to <a target="_blank" title="http://www.ros.org/wiki/rosbridge" href="http://www.ros.org/wiki/rosbridge">rosbridge</a>, a popular ROS package for connecting to ROS from a Web browser. &nbsp;His internship concluded with the release of the rosbridge suite, a robust and extensible collection of packages that facilitate Web-based and non-ROS connection to ROS.</p>
              <p>Web browsers are a compelling choice for writing front ends to robot applications. &nbsp;In particular, they offer a ubiquitous, interoperable platform for robot interaction. &nbsp;Given that end users of robot applications may require little or no knowledge of the underlying robot middleware, decoupling a Web-based front end from a ROS-dependent back end is a promising direction to pursue.</p>
              <p>In order to facilitate this decoupling, the rosbridge suite provides an access point for Web browsers (and other WebSocket-compatible systems) to access ROS. &nbsp;The rosbridge suite also provides components to automate install and runtime linking of Web components. &nbsp;As such, the rosbridge suite makes it much easier for ROS developers to include a Web component to their work.</p>
              <p>The rosbridge suite primarily contains a Web server which runs inside the ROS environment. &nbsp;This Web server listens for incoming WebSocket connections, and exchanges JSON-based messages with connected clients. &nbsp;Clients can instruct rosbridge to call ROS services, subscribe and publish to ROS topics, or introspect the ROS runtime. &nbsp;Response messages originating in the ROS runtime are propagated back to the client. &nbsp;Thus, Web browsers and middleware separate from ROS can still fully interact with a running ROS system.</p>
              <p>The structure of the JSON messages exchanged between clients and the rosbridge server are defined in the rosbridge protocol. &nbsp;In order to make the protocol more extensible and pluggable, it was redefined and formally specified. &nbsp;It is similar in spirit to the protocol used by the original rosbridge, but offers more customization.</p>
              <p>Other packages in the rosbridge suite include: roswww, an HTTP Web server that runs in the ROS runtime; rosapi, a node that advertises services that introspect the ROS runtime; and tf smart throttle, a node that intelligently throttles tf messages for low-bandwidth connections.</p>
              <p>The rosbridge suite has been released as a ROS stack and is available <a target="_blank" title="http://www.ros.org/wiki/rosbridge_suite" href="http://www.ros.org/wiki/rosbridge_suite">here</a>. &nbsp;In addition, tutorials, client implementations, and more information is available at <a target="_blank" title="http://www.rosbridge.org/doku.php" href="http://www.rosbridge.org/doku.php">www.rosbridge.org</a>.</p>
              <p>&nbsp;</p>
