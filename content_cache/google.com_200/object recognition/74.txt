      <h1>Buildpacks</h1>
      <p class="last-updated"><span class="icon-clock"></span>Last updated 12 September 2017</p>
      <div id="table-of-contents">
       <h3><span class="icon-list"></span>Table of Contents</h3>
       <ul>
        <li><a href="#officially-supported-buildpacks">Officially supported buildpacks</a></li>
        <li><a href="#setting-a-buildpack-on-an-application">Setting a buildpack on an application</a></li>
        <li><a href="#using-multiple-buildpacks">Using multiple buildpacks</a></li>
        <li><a href="#creating-a-buildpack">Creating a buildpack</a></li>
       </ul>
      </div> 
      <p>Buildpacks are responsible for transforming deployed code into a slug, which can then be executed on a dyno. Buildpacks are composed of a set of scripts, and depending on the programming language, the scripts will retrieve dependencies, output generated assets or compiled code, and more. This output is assembled into a slug by the <a href="https://devcenter.heroku.com/articles/slug-compiler">slug compiler</a>.</p> 
      <p>Heroku’s support for Ruby, Python, Java, Clojure, Node.js, Scala, Go and PHP is implemented via a set of open source buildpacks.</p> 
      <h2 id="officially-supported-buildpacks">Officially supported buildpacks</h2> 
      <p>Heroku maintains a collection of officially supported buildpacks that are available by default to all Heroku apps during slug compilation.</p> 
      <div class="callout"> 
       <p>These buildpacks are open-source and available on GitHub. If you have a change that would be useful to all Heroku developers, we encourage you to submit a pull request.</p>

      <p>By default, these buildpacks will be searched in this order until a match is detected and used to compile your app. If the build succeeds, <strong>the detected buildpack will be permanently set for future pushes</strong> to your application as if you had run <code>heroku buildpacks:set</code> manually as <a href="#setting-a-buildpack-on-an-application">explained below</a>.</p> 
      <p>Custom buildpacks can be used to support languages or frameworks that are not covered by Heroku’s officially supported buildpacks. For a list of known third-party buildpacks, see <a href="https://devcenter.heroku.com/articles/third-party-buildpacks">Third-Party Buildpacks</a>.</p> 
      <h2 id="setting-a-buildpack-on-an-application">Setting a buildpack on an application</h2> 
      <p>You can change the buildpack used by an application by setting the buildpack value. When the application is next pushed, the new buildpack will be used.</p> 
      <pre class="language-term"><code class="language-term">$ heroku buildpacks:set heroku/php
Buildpack set. Next release on random-app-1234 will use heroku/php.
Run `git push heroku master` to create a new release using this buildpack.
</code></pre> 
      <div class="callout"> 
       <p>It used to be possible to set a <a href="https://devcenter.heroku.com/articles/config-vars">config var</a> named <code>BUILDPACK_URL</code> to declare which buildpack to use. If defined, its value will still be used, but a buildpack configured through <code>heroku buildpacks:set</code> will take precedence, and the use of <code>BUILDPACK_URL</code> is now deprecated.</p> 
      </div> 
      <p>You may also specify a buildpack during app creation:</p> 
      <pre class="language-term"><code class="language-term">$ heroku create myapp --buildpack heroku/python
</code></pre> 
      <h3 id="setting-a-buildpack-in-app-json">Setting a buildpack in <code>app.json</code> </h3> 
      <p>The buildpack can be explicitly set in <a href="https://devcenter.heroku.com/articles/app-json-schema"><code>app.json</code></a> so that applications created via Heroku buttons can use custom buildpacks.</p> 
      <h3 id="buildpack-modifications">Buildpack modifications</h3> 
      <p>You can change or remove a previously set buildpack again later:</p> 
      <pre class="language-term"><code class="language-term">$ heroku buildpacks:set heroku/nodejs
$ heroku buildpacks:remove heroku/nodejs
</code></pre> 
      <div class="note"> 
       <p>When no buildpack is set on your application, the <a href="#officially-supported-buildpacks">detection process</a> will be run again on your next <code>git push heroku</code>.</p> 
      </div> 
      <p>The <code>heroku buildpacks</code> command provides a range of other capabilities around addition, modification and removal of buildpacks, some of which are particularly useful when employing <a href="#using-multiple-buildpacks">multiple buildpacks</a>. The <code>heroku help buildpacks</code> command provides further details:</p> 
      <pre class="language-term"><code class="language-term">$ heroku help buildpacks
Usage: heroku buildpacks

Additional commands, type "heroku help COMMAND" for more details:

  buildpacks:add BUILDPACK_URL       #  add new app buildpack, inserting into list of buildpacks if neccessary
  buildpacks:clear                   #  clear all buildpacks set on the app
  buildpacks:remove [BUILDPACK_URL]  #  remove a buildpack set on the app
  buildpacks:set BUILDPACK_URL       #  set new app buildpack, overwriting into list of buildpacks if neccessary
</code></pre> 
      <h3 id="detection-failure">Detection failure</h3> 
      <p>If the configured buildpack cannot handle your application (as determined by its <a href="https://devcenter.heroku.com/articles/buildpack-api#bin-detect">detection script</a>), you will receive an error. For example, Heroku’s Ruby buildpack expects a <code>Gemfile</code> to be present in the root folder of an application to correctly identify its type, but if the buildpack of an application is set to <code>heroku/ruby</code> and no <code>Gemfile</code> is present, the application will fail to build.</p> 
      <div class="callout"> 
       <p>This situation may also occur if you remove or rename a file that previously led to the automatic detection of your application type and thus the automatic setting of the detected buildpack on your application.</p> 
       <p>For example, had you pushed an application written in PHP, which in addition to a <code>composer.json</code> contained a <code>package.json</code> with NPM packages for asset handling, the application would have been detected as a Node.js application on first push due to the <a href="#officially-supported-buildpacks">order in which default buildpacks are invoked for detection</a>.</p> 
       <p>The removal of <code>package.json</code> in this case would not automatically reconfigure the buildpack used for the application - the buildpack is “pinned” to <code>heroku/nodejs</code>, and attempting a <code>git push</code> without a <code>package.json</code> would result in an error:</p> 
       <pre><code>-----&gt; Using set buildpack heroku/nodejs
 !     Push rejected, failed to detect set buildpack heroku/nodejs
</code></pre> 
       <p>You would now need to manually <a href="#buildpack-modifications">clear the buildpack</a> or <a href="#setting-a-buildpack-on-an-application">set the desired buildpack</a> - in this example, to <code>heroku/php</code>.</p> 
      </div> 
      <h3 id="using-an-official-buildpack">Using an official buildpack</h3> 
      <p>If Heroku’s auto-detection of buildpacks is not sufficient, or if you need <a href="#using-multiple-buildpacks">multiple buildpacks</a> you can configure your app to run with one of the default buildpacks by executing a command such as this:</p> 
      <pre class="language-term"><code class="language-term">$ heroku buildpacks:set heroku/ruby
Buildpack set. Next release on random-app-1234 will use heroku/ruby.
Run `git push heroku master` to create a new release using this buildpack.
</code></pre> 
      <p>This example forces Heroku to use the latest release version of the official Ruby buildpack on your next deployment (and every deployment after unless you change it). Instead of <code>heroku/ruby</code>, you may use any of the other shorthand identifiers listed above in the <a href="#officially-supported-buildpacks">table of official buildpacks</a>.</p> 
      <div class="callout"> 
       <p>The shorthand format does not support specifying branches, tags or versions of the buildpack with the <code>#</code> notation. You must use a buildpack URL if you require anything other than the latest version of the official buildpack.</p> 
      </div> 
      <h3 id="using-a-custom-buildpack">Using a custom Buildpack</h3> 
      <p>You can use any unofficial or third-party buildpack by specifying a URL with the <code>buildpacks:set</code> command:</p> 
      <pre class="language-term"><code class="language-term">$ heroku buildpacks:set https://github.com/heroku/heroku-buildpack-erlang
Buildpack set. Next release on random-app-1234 will use https://github.com/heroku/heroku-buildpack-erlang.
Run `git push heroku master` to create a new release using this buildpack.
</code></pre> 
      <div class="note"> 
       <p>You can also set the full URL of any official buildpack, which will cause the <code>master</code> Git branch of that buildpack to be used instead of the last released version. As the latest Git version may contain unexpected changes, it is highly recommended to use the <code>heroku/…</code> syntax for any of the official buildpacks.</p> 
      </div> 
      <h3 id="buildpack-references">Buildpack references</h3> 
      <p>Your buildpack value can point to either git repositories or a tarball URL. Hosting a buildpack on S3 can be a good way to ensure it’s highly available.</p> 
      <div class="callout"> 
       <p>When the source of the buildpack you wish to use is a Git repository, you can specify a specific tag or branch of that buildpack to be used by appending a Git <a href="https://git-scm.com/docs/gitrevisions">object</a> (e.g. a commit SHA, branch name or tag name) to the URL when invoking the <code>buildpacks:set</code> command; for example:</p> 
       <ul> 
        <li><code>https://github.com/heroku/heroku-buildpack-nodejs.git#somedevbranch</code></li> 
        <li><code>https://github.com/heroku/heroku-buildpack-ruby.git#v9861</code></li> 
       </ul> 
      </div> 
      <h2 id="using-multiple-buildpacks">Using multiple buildpacks</h2> 
      <p>There are many scenarios in which a single buildpack is not sufficient when building an application. Typical scenarios include database connection pooling using <a href="https://elements.heroku.com/buildpacks/heroku/heroku-buildpack-pgbouncer">PgBouncer</a>, or asset handling using a Node.js library in combination with a Ruby, Python or PHP application.</p> 
      <p>When this is the case, please follow our guide to <a href="https://devcenter.heroku.com/articles/using-multiple-buildpacks-for-an-app">Using Multiple Buildpacks for an App</a>.</p> 
      <h2 id="creating-a-buildpack">Creating a buildpack</h2> 
      <p>If you’d like to use a language or framework not yet supported on Heroku you can create a custom buildpack. To get started, see the following articles:</p> 
      <ul> 
       <li>To learn about the structure of a buildpack, see <a href="https://devcenter.heroku.com/articles/buildpack-api">Buildpack API</a>.</li> 
      </ul>
