           <h2 id="1Changeschallengeautomatedtesting" class="ibm-h2">Changes challenge automated testing</h2>
           <p>The Internet and World Wide Web have become integral parts everyday life and business. Of course, most Web pages and Web applications that we see today are dynamic. Those changing pages make automating the testing of Web interfaces quite challenging. IBM® Rational® Functional Tester can help you automate tests even while the applications under test and the pages they link to continue to change.</p>
           <h2 id="2Includingactionsonobjectswithoutaddingtotheobjectrepository" class="ibm-h2">Including actions on objects without adding to the object repository</h2>
           <p>Your test might include selecting random dates in a date-selection window. For the test to pass, the specific date chosen at random must be recognized. The date selection object shown in <a href="#calendar">Figure 1</a> and the typical strategy of adding the links to the object repository poses problems:</p>
           <ul class="ibm-bullet-list">
            <li>To click a date link, the date links must be added to the object repository; otherwise, no dates can be selected during the test run. </li>
            <li>If the chosen dates are random, then you need to add to the repository all 365 days as date links. </li>
           </ul>
           <p>Adding all these dates would require that you spend your test-development time adding the links and filling with the object repository with hundreds of date links. </p>
           <h5 id="calendar" class="ibm-h5">Figure 1. Date-selection window</h5>
           <img src="image001.jpg" class="ibm-downsize" alt="A calendar display for selecting a month and day" height="258" width="261">
           <h3 id="N10074" class="ibm-h3">A smarter solution for objects</h3>
           <p>Rather than relying on so many links, you can you create the objects dynamically and use them to include this date-selection window, plus include the possibility of using any date in a year. This also makes your script resilient enough to permit using different dates. This method does not require that you add any date objects to the object repository.</p>
           <p><strong>Important:</strong><br> Unregister the dynamically created objects after you have used it and no longer need the object. Rational Functional Tester unregisters bound references when the entire playback ends, not when the script ends. When a bound reference to the object exists, Rational Functional Tester prevents the object in the application from being entirely free. If the object continues as unregistered, the garbage collector does not collect the object because Rational Functional Tester refers to it. This reference prevents the release of memory, which can lead to unpredictable behavior</p>
           <p>To create the date-selection object dynamically, you need to see the properties of the object first:</p>
           <ol type="1">
            <li>To add the object so that you can see the properties of the object, select <strong>19</strong> for 19 July 2009.</li>
            <li>Open the object repository of the script where you added the link. The properties are shown in <a href="#_Ref238277945">Figure 2</a>. The date link is shown as <code>javascript:dateHandler(%2019,7,2009%20)</code>; </li>
           </ol>
           <h5 id="_Ref238277945" class="ibm-h5">Figure 2. The object map editor with the test object hierarchy and the object properties</h5>
           <img src="image002.jpg" class="ibm-downsize" alt="The object map editor shows the link value" height="406" width="555">
           <p class="ibm-ind-link"><a class="ibm-popup-link" onclick="IBMCore.common.widget.overlay.show('N10097');return false;" href="#N10097">View image at full size</a></p>
           <div class="ibm-common-overlay ibm-overlay-alt-three" data-widget="overlay" id="N10097">
            <img alt="The object map editor shows the link value" src="image002.jpg" width="555">
           </div>
           <p>You can use a regular expression for the date link, and determine the object at run time. In the <strong>Value</strong> column, type this value for the object: <code>javascript:dateHandler(%20\\d+,\\d+,\\d+%20);<br></code><code><br></code>The regular expression includes the 35 possible date links that are available in <a href="#calendar">Figure 1</a>. However, you need to add all possible dates in a year.</p>
           <p><strong>Note: </strong><br>See the Rational Functional Tester Information Center for details about using regular expressions. </p>
           <ol start="3" type="1">
            <li>Expand the regular expression to include all possible dates in a year (see Listing 1). By writing the regular expression this way, the test can accommodate any specific date.</li>
           </ol>
           <h5 id="_Ref238279888" class="ibm-h5">Listing 1. Sample code for clicking the date link</h5>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">publicvoid clickReqDate(String date, String month, String year) 
throws Exception{
	
	RegularExpression dateHrefReProp = new
	RegularExpression("javascript:dateHandler\\
	(%20"+date+","+month+",\\d+%20\\);",true);
		
	//Finding the object from the dialog which has properties of
	//class as Html.A and href as dateHrefProp
	TestObject [] dateObjs = find(atDescendant
	(".class","Html.A", ".href", dateHrefReProp));
		
	System.out.println(dateObjs.length);
		
	//converting the testObject to GuiTestObject
	GuiTestObject dateObj = (GuiTestObject)dateObjs[0];
	dateObj.waitForExistence();
		
	//getting the text property of the date at runtime.
	String runtimeDate= (String) dateObj.getProperty(".text");
	System.out.println("Date: "+runtimeDate);
		
	//Comparing the String that is passed through the method with that of the 
	//date String that we got at runtime.
		
	if(runtimeDate.equalsIgnoreCase(date)){
		dateObj.click();
	}
		
	//unregistering of the object.
	dateObj.unregister();
		
	}</pre></span>
           <ol start="4" type="1">
            <li>Delete the object that you used to select the single date of 19 July 2009.</li>
            <li>For selecting the date of 19 July 2009, call the method with these parameters specified: <code>clickReqDate("19","7","2009"); </code></li>
            <li>Unregister the object that you create dynamically, which is also good coding practice. See the final section of <a href="#_Ref238279888">Listing 1</a> for the command to unregister the object.</li>
           </ol>
           <h2 id="3Navigatingtopageswithoutfillingscriptswithlinks" class="ibm-h2">Navigating to pages without filling scripts with links</h2>
           <p>Specifying navigation clicks can clutter a test script and waste your valuable time. For example, navigating to the page shown in <a href="#_Ref238284727">Figure 3</a>, requires that you click four to five links: <strong>Rational </strong>&gt; <strong>Downloads </strong>&gt; <strong>Plug-ins</strong> &gt; <strong>Ready for Rational Plug-in Central</strong>. Your test would include the code in <a href="#_Ref238282304">Listing 2</a>. That code represents the navigation to one page. Consider how many pages users go to in applications under test.</p>
           <h5 id="_Ref238284727" class="ibm-h5">Figure 3. Sample site showing the navigation</h5>
           <img src="image003.jpg" class="ibm-downsize" alt="Web page shows selections for opening another page" height="424" width="572">
           <p class="ibm-ind-link"><a class="ibm-popup-link" onclick="IBMCore.common.widget.overlay.show('N10100');return false;" href="#N10100">View image at full size</a></p>
           <div class="ibm-common-overlay ibm-overlay-alt-three" data-widget="overlay" id="N10100">
            <img alt="Web page shows selections for opening another page" src="image003.jpg" width="572">
           </div>
           <h5 id="_Ref238282304" class="ibm-h5">Listing 2. Links in a test script</h5>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">link_rational().click();
link_downloads().click();
link_plugins().click();
link_rationalPluginCentre();</pre></span>
           <p>After selecting the final link, a tester might need to navigate to a similar link in WebSphere. The test requires clicking four to five additional links. Most of the script contains link coding.</p>
           <h3 id="N1010F" class="ibm-h3">A smarter solution for navigation</h3>
           <p>In a storage domain product that I currently work on, we spend most of the time navigating to different pages. If we follow the typical procedure of recording clicks in the test script, clicking the links would constitute 70% of the script code. </p>
           <p>However, you can reduce the links in the script and create a resilient script that can function even as various sites are chosen. We created XML code that maintains the parent-child structure of the links and tabs. Furthermore, we developed a method to click the required link by getting the parent-child information for the link from the developed XML code.</p>
           <p>The XML code for clicking the Ready for IBM Rational Plug-in Central link, as shown earlier in <a href="#_Ref238284727">Figure 3</a>, has the hierarchy shown in Listing 3.</p>
           <h5 id="_Ref239570178" class="ibm-h5">Listing 3. XML code that shows the link hierarchy of a Rational Web page</h5>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">&lt;link_IBM_HomePage&gt;
	&lt;link_rational&gt;	
	&lt;link_downloads&gt;
		&lt;link_trialDownloads&gt;&lt;/link_trialDownloads&gt;
		&lt;link_onlineTrials&gt;&lt;/link_onlineTrials&gt;
		&lt;link_supportDownloads&gt;&lt;/link_supportDownloads&gt;
		&lt;link_plugins&gt;
		   &lt;link_ibmPlugins&gt;&lt;/link_ibmPlugins&gt;
    		   &lt;link_rationalPluginCentral&gt;&lt;/link_rationalPluginCentral&gt;
               &lt;link_webspherePluginCentral&gt;&lt;/link_webspherePluginCentral&gt;
		&lt;/link_plugins&gt;
	
	&lt;/link_downloads&gt;
	&lt;/link_rational&gt;
&lt;/link_IBM_HomePage&gt;</pre></span>
           <p>In your test, suppose that you want to navigate to the Ready for IBM Rational Plug-in Central page. You would follow these steps:</p>
           <ol type="1">
            <li>Create XML code that captures the hierarchy of links. (See <a href="#_Ref239570178">Listing 3</a>.)</li>
            <li>In your test script, use this method: <br><code>gotoPage("link_rationalPluginCentral");</code></li>
           </ol>
           <p><br>Adding this single-line method replaces five link lines similar to those lines in <a href="#_Ref238282304">Listing 2</a></p>
           <p>The <code>gotoPage(String)</code> method first checks whether <code>link_rationalPluginCentral</code> is available on the current Web page. If the link is available, the method clicks the link. If the link is not available, the method checks for the parent of the object, <code>link_plugins</code>. If that page is not available, then the method checks for its parent, <code>link_downloads</code>, and so on up the hierarchy.</p>
           <p>You do not need to include all of these links in your script, and you can reuse the XML hierarchy code and methods to further enhance efficiency.</p>
           <ol start="3" type="1">
            <li>After you perform the required operation on the Plug-in Central page, to move to a similar page for WebSphere, we can just call this method: <code>gotoPage("link_webspherePluginCentral");</code></li>
           </ol>
           <p><strong>Tip:</strong><br> To open the Plug-in Central page for WebSphere page, you must specify the navigation in the XML hierarchy similar to the one shown in <a href="#_Ref239570178">Listing 3</a>. </p>
           <h2 id="4Summary" class="ibm-h2">Summary</h2>
           <p>You can build clean, lean, and resilient scripts by incorporating these practices:</p>
           <ul class="ibm-bullet-list">
            <li>Add only the required objects to the object repository to keep the repository clean. Creating dynamic objects and acting upon them as required reduces test dependency on the object repository, which enhances object recognition and reduces maintenance significantly. </li>
            <li>Make your scripts resilient when applications change by including regular expressions and XML link hierarchies. The tests can tolerate constantly changing objects and navigation.</li>
            <li>Increase flexibility and reduce the package maintenance that dynamic Web applications demand by rethinking the way that you include objects and navigation.</li>
           </ul>
           <p>Spend more time testing and less time adjusting scripts to the inevitable changes in Web applications.</p>
