      <h2>Data-URI base64 converter</h2> 
      <p>Helps improve page load performance over <abbr title="HyperText Transfer Protocol">HTTP</abbr>/1 connections by bundling multiple images into a single cacheable style sheet.</p> 
      <h2>When using base64 data-URIs</h2> 
      <h3>Optimise first</h3> 
      <p>Most designers don't tend to care too much about optimisation, and <a href="http://www.photoshop.com/products/photoshop">Photoshop's</a> <q>Save for web</q> just isn't very good. My personal choice for web output is the now defunct <a href="http://www.adobe.com/uk/products/fireworks.html">Fireworks</a>. That said there are many online tools which may help and remember image optimisation is <q>best practice</q>:</p> 
      <ul> 
       <li><a href="http://www.smushit.com/ysmush.it/">SmushIt</a></li> 
       <li><a href="https://compressor.io/">Compressor.io</a></li> 
       <li><a href="https://tinypng.com/">TinyPNG</a></li> 
       <li><a href="https://jakearchibald.github.io/svgomg/">SVGOMG</a></li> 
      </ul> 
      <p>The new kid on the compression block is Google's <a href="https://github.com/google/guetzli">Guetzli</a> which promises 20-30% typical compression. Though it takes a lot longer to compress, a fair compromise.</p> 
      <p>There's a lot to image optimisation, if you are interested The Book of Speed's chapter on <a href="http://www.bookofspeed.com/chapter5.html">Optimizing Images</a> is very good.</p> 
      <p>As a side note: When talking to designers use the term <q>image optimisation</q> rather than <q>image compression</q>, they'll listen a little longer.</p> 
      <h3><abbr title="Scalable Vector Graphics">SVGs</abbr></h3> 
      <p>If you're converting SVG to data-URI to embed into a style sheet or <abbr title="Hyper Text Markup Language">HTML</abbr> please consider this method instead: <a href="/articles/embedding-svg-cross-browser/">Cross browser SVG URI background-images</a>.</p> 
      <p>There's also a tool to <a href="/tools/svg-to-background-image-conversion/">convert SVG URIs to <abbr title="Cascading Style Sheet">CSS</abbr> background-images</a>.</p> 
      <h3>Cache the CSS</h3> 
      <p>Data-URI embedded images should be stored in cacheable files such as flat, not dynamic, HTML and style sheets. Data-URI style sheets should be cached (<q>me love you</q>) long-term, but rename the file on each update. See this <a href="https://www.mnot.net/cache_docs/">Caching Tutorial</a> for more details or perhaps <a href="https://developers.google.com/web/fundamentals/performance/optimizing-content-efficiency/http-caching">Google's developer docs</a>.</p> 
      <h3>Base64 file sizes</h3> 
      <p>File size increases by approximately 30% which may be more than offset by serving your files gzipped or deflated. If you're not doing this already, get on it. This article may help: <a href="http://betterexplained.com/articles/how-to-optimize-your-site-with-gzip-compression/">How To Optimize Your Site With GZIP Compression</a></p> 
      <p>Personally, I use <a href="https://github.com/farhadi/SmartOptimizer">SmartOptimizer</a> which minifies, gzips and caches files on the server thereby reducing server overheads. Google's <a href="https://github.com/pagespeed/mod_pagespeed">mod_pagespeed</a> does something similar.</p> 
      <h3>Browser support</h3> 
      <p>Full list of base64 data-URI <a href="http://caniuse.com/#feat=datauri">supported browsers</a>.</p> 
      <p>IEv8 has the lowest maximum data-URI size of 32 Kb.</p> 
      <p>Base64 data-URI embedded images are unsupported in IE 6 &amp; 7. You can circumvent by using the <abbr title="star">*</abbr> hack:</p> 
      <figure class="shadowlift"> 
       <figcaption>
        Language CSS
       </figcaption> 
       <pre><code class="language-css" spellcheck="false" contenteditable>.image {
  background-image:url(data:image/png;base64,iVBO…);

  /* Only IEv6 &amp; 7 see this */
  *background-image:url(image.png);
}    </code></pre> 
      </figure> 
      <h3>Mobile rendering</h3> 
      <p>I've read a few articles warning of poor rendering performance when using base64 data-URI images on mobile. Personally I've never noticed an issue but testing was far from exhaustive.</p> 
      <ul> 
       <li>Mobify: <a href="http://www.mobify.com/blog/base64-does-not-impact-data-uri-performance/">Base64 does not impact data-URI performance</a></li> 
       <li>Mobify: <a href="http://www.mobify.com/blog/data-uris-are-slow-on-mobile/">Data-URIs are slow on mobile</a></li> 
      </ul> 
      <h3>HTTP/1 and HTTP/2 transfer protocols</h3> 
      <p>The main use of base64 encoded data-URIs is to reduce the number of HTTP requests made by the browser. This is achieved by combining multiple images into a single cacheable <abbr title="Cascading Style Sheet">CSS</abbr> file, or inlining small images directly into the <abbr title="HyperText Markup Language">HTML</abbr>.</p> 
      <p>All well and good for HTTP/1 with its six maximum parallel connections, but now HTTP/2 is arriving which no longer requires multiple connections. If performance is your goal I suggest moving to <abbr title="Secure SHell">SSH</abbr> over a HTTP/2 connection.</p> 
      <p>See Illya Grigorik's presentation: <a href="https://docs.google.com/presentation/d/1r7QXGYOLCh4fcUq0jDdDwKJWNqWK1o4xMtYpKZCJYjM/present?slide=id.g518e3c87f_0_228">HTTP/2 - Let's optimise</a>.</p>

    function initCopyButton() {
      var btn_copy = document.getElementById("btn_copy");
      if (btn_copy) {

        btn_copy.addEventListener("click", function(e){

      var clipboard = new Clipboard('#btn_copy');

      clipboard.on('success', function(e) {
        var btn = e.trigger;
        if (btn && btn.textContent === btn.initialTextContent) {
          btn.textContent = btn.textContent.replace("Copy", "Copied");
          btn.classList.add("btn-copied");
          setTimeout(function(){
            // reset copy button
            btn.textContent = btn.initialTextContent;
            btn.classList.remove("btn-copied");
          }, 2000);
        }
      });
    }
