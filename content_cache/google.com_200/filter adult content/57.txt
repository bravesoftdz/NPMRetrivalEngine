         <p></p>
         <p>A content expression is the combination of functions, text, and phrases used in a <a class="jive-link-wiki-small" data-containerid="2024" data-containertype="14" data-objectid="1648" data-objecttype="102" href="https://community.mimecast.com/docs/DOC-1648" style="color: #0082cb; border: 0px; font-weight: inherit; padding: 1px 0px 1px calc(12px + 0.35ex);">Content Examination Definition</a> to assist with Data Leak Prevention (DLP). A content expression could be a word, phrase, regular expression, or hash algorithm matching a specific document. The content expression is entered into the Content Examination definition, which scans messages looking for any of the search criteria to match.</p>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <p>These search terms are used to prevent the accidental or malicious data loss through company email. Depending on your requirements, some or all of the examples below may be applicable to make your email system comply with your company's security policy. Mimecast has the capabilities to validate regular expressions using both Java and Perl regex engines. These can be specified by either using the invocation language perlregex or javaregex.</p>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>

         <h1 id="jive_content_id_Using_Content_Expressions">Using Content Expressions</h1>
         <p>&nbsp;</p>
         <p>Content expressions are entered into the "Word / Phrase Match List" field in a Content Examination definition. The activation score is a required field, and is the total score used to trigger the definition should a match be found in a message.</p>
         <div __default_attr="info" alert="info" class="jive_macro_alert jive_text_macro" name="alert">
          It is highly recommended to test Content Examination Definitions to ensure that the correct results are achieved as required.
         </div>

         <h2 id="jive_content_id_BooleanOperators">Boolean&nbsp;Operators</h2>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <p>You can use boolean&nbsp;operators (i.e. AND / OR) to help minimize the number of false positives experienced. This is achieved by specifying two search terms that must exist in a message.</p>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <h3 id="jive_content_id_Text_Strings_and_Regular_Expression">Text Strings and Regular Expression</h3>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <p>The following search terms would require the word 'Credit Card' along with a valid credit card number before a content hit is found. If only one of the specified search terms is found, no content hit will be logged.</p>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <div class="j-rte-table">
          <table class="j-table jiveBorder" style="border: 1px solid #c6c6c6;" width="100%">
           <thead>
            <tr style="background-color: #efefef;">
             <th style="width: 25%;"><span style="color: #505050;"><strong>Word / Phrase Match List</strong></span></th>
             <th><span style="color: #505050;"><strong>Email Content Required to Trigger Definition</strong></span></th>
            </tr>
           </thead>
           <tbody>
            <tr>
             <td style="width: 25%;"><span>Credit Card 4111 1111 1111 1111</span></td>
             <td><span>1 ("Credit Card") AND (regex,cardnumber 4(?&lt;=\b(?&lt;!\.)4)\d{3}[\W\s]?\d{4}[\W\s]?\d{4}[\W\s]?\d{4}\b)</span></td>
            </tr>
           </tbody>
          </table>
         </div>
         <div __default_attr="info" alert="info" class="jive_macro_alert jive_text_macro" name="alert">
          Each search term must be wrapped in brackets () and followed by the AND operator and your next search term. Phrases must be wrapped in brackets and speech marks. For example, ("Credit Card"). Only one AND operator can be used per line in the Search Word/Phrase match list.
         </div>

         <h2 id="jive_content_id_Proximity_Operators">Proximity Operators</h2>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <p>The Proximity operator allows administrators to specify if a Content Policy triggers when a search term is found in a specified number of characters of another search term. If a match is found outside of the specified distance no content match is triggered.</p>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <p>Considerations:</p>
         <ul>
          <li>There is no limit on the distance specified by the Proximity operator.</li>
          <li>Mimecast Managed Reference Dictionaries (MMRDs) and Reference Dictionaries can be used in conjunction with the 'Proximity' operator.</li>
          <li>The Proximity value is calculated by the number of characters from the end of the first search word / phrase, until the first character of the second search word / phrase (including blank spaces and special characters). Take the following example:</li>
         </ul>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <p>"Here is my credit card number for you to use 4111 1111 1111". The proximity value would be 17, as there are 17 characters between the end of Credit Card and the first character of the credit card number specified (4111 1111 1111). Remember that blank spaces count as characters, so you'll need to include them in your proximity value.</p>
         <p><a href="https://community.mimecast.com/servlet/JiveServlet/showImage/102-1613-31-89279/promimity.png"><img alt="Proximity Example" class="image-2 jive-image j-img-original" height="50" src="https://community.mimecast.com/servlet/JiveServlet/downloadImage/102-1613-31-89279/promimity.png" style="" width="395"></a></p>

         <p>You wish to hold all messages containing references to credit card numbers.&nbsp;T<span>he "creditcard" entity finds all credit card numbers, regardless of the credit card type. For example, the following would match any credit card number found in the specified areas of an email (header, body, attachment).</span></p>
         <blockquote class="jive-quote" style="background: none #f6f6f6; border-left: 2px solid #cccccc; margin: 10px 0px; padding: 10px 20px;">
          1 detect creditcard
         </blockquote>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <h3 id="jive_content_id_Multiple_Entities_with_Operators">Multiple Entities with Operators</h3>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <p>You want to hold messages that contain a&nbsp;piece of PII (Personal Identifiable Information) and a date of birth, that are a specific distance from each other.</p>
         <blockquote class="jive_macro_quote jive-quote jive_text_macro" style="background: none #f6f6f6; border-left: 2px solid #cccccc; margin: 10px 0px; padding: 10px 20px;">
          1 (detect aba) Proximity:50 (detect date_dob)
         </blockquote>
         <p>This criteria detects any instances of an ABA number, and any instances of a Date of Birth (DOB), within a range of 50 characters from each other before a match is made.</p>
         <div __default_attr="success" alert="success" class="jive_macro_alert jive_text_macro" name="alert">
          By default the "Proximity" operator has a default distance of 300 characters. Specifying a number value after proximity overrides the default distance.
         </div>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <h3 id="jive_content_id_Excluding_Entities">Excluding Entities</h3>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <p>An administrator is using the PII entity group.&nbsp;They are seeing a high number of false positives with the "Phone Number" entity, and would like to check if excluding this resolves the problem. To do this they can:</p>
         <ol>
          <li>Remove the entry for the PII entity group from the Content Examination definition.</li>
          <li>Enter all the individual entities that they wish to use.</li>
         </ol>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <div class="j-rte-table">
          <table class="j-table jiveBorder" style="border: 1px solid #c6c6c6;" width="100%">
           <thead>
            <tr style="background-color: #efefef;">
             <th>Original Word / Phase Match List</th>
             <th>New Word / Phrase Match List</th>
            </tr>
           </thead>
           <tbody>
            <tr>
             <td>
              <blockquote class="jive_macro_quote jive-quote jive_text_macro">
               <p>1 detect PII</p>
              </blockquote></td>
             <td>
              <blockquote class="jive_macro_quote jive-quote jive_text_macro">
               <p>1 detect Name<br>1 detect DOB<br>1 detect SSN<br>1 detect MedicareID<br>1 detect FAX<br>1 detect VIN<br>1 detect IP<br>1 detect EmailAddress<br>1 detect URL</p>
              </blockquote></td>
            </tr>
           </tbody>
          </table>
         </div>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <h3 id="jive_content_id_Negative_Score">Negative Score</h3>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <p>Using the same example of the PII entity group, but wanting to exclude the "Phone Number" entity,&nbsp;a negative score for the entity can be applied. If a match is found for the search term, the negative score is applied to the number of hits total. This reduces the overall score, and the chance of the Content Examination policy being applied.&nbsp;To do this they can:</p>
         <ol style="padding: 0px 0px 0px 30px;">
          <li style="margin: 0.2em 0px;">Leave&nbsp;the PII entity group in&nbsp;the Content Examination definition.</li>
          <li style="margin: 0.2em 0px;">Apply a negative score for the "Phone Number" entity.</li>
         </ol>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <div class="j-rte-table">
          <table class="j-table jiveBorder" style="border: 1px solid #c6c6c6;" width="100%">
           <thead style="border: inherit solid inherit;">
            <tr style="background-color: #efefef;">
             <th>Original Word / Phase Match List</th>
             <th>New Word / Phrase Match List</th>
            </tr>
           </thead>
           <tbody style="border: inherit solid inherit;">
            <tr style="border: inherit solid inherit;">
             <td>
              <blockquote class="jive_macro_quote jive-quote jive_text_macro" style="background: none #f6f6f6; border-left: 2px solid #cccccc; margin: 10px 0px; padding: 10px 20px;">
               <p>1 detect PII</p>
              </blockquote></td>
             <td>
              <blockquote class="jive_macro_quote jive-quote jive_text_macro" style="background: none #f6f6f6; border-left: 2px solid #cccccc; margin: 10px 0px; padding: 10px 20px;">
               <p>1 detect&nbsp;PII<br>-1 detect PhoneNumber</p>
              </blockquote></td>
            </tr>
           </tbody>
          </table>
         </div>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <h3 id="jive_content_id_Adding_Entities">Adding Entities</h3>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <p>There are reports of Canadian Social Security Numbers (SIN) being present in messages&nbsp;being sent externally. On investigation it is discovered the SIN number entity is not present in the policy.&nbsp;To do this they can:</p>
         <ol style="padding: 0px 0px 0px 30px;">
          <li style="margin: 0.2em 0px;">Add the SIN entity to&nbsp;the Content Examination definition.</li>
         </ol>
         <blockquote class="jive_macro_quote jive-quote jive_text_macro">
          1 detect SIN
         </blockquote>
         <h3 id="jive_content_id_Combining_Entities_and_Phrases">Combining Entities and Phrases</h3>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <p>You want to search for the term "Admission Date" followed by a date in Month / Day / Year format.&nbsp;This can be achieved by using the following policy syntax.</p>
         <blockquote class="jive_macro_quote jive-quote jive_text_macro">
          <p>1 ("Admission Date") Proximity (detect date_mdy)</p>
         </blockquote>
         <ul>
          <li>1 is the line score applied when a match is found.</li>
          <li>("Admission Date")&nbsp;is the first check performed. This must be&nbsp;in brackets to mark the boundaries of the search text.</li>
          <li>Proximity&nbsp;is the operator. In this case the phrase "Admission Date" needs to be within 300 characters of a date in Month/Day/Year format.</li>
          <li>(detect date_mdy)&nbsp;is the second check performed. Again this must be in brackets to mark the boundaries of the search term.</li>
         </ul>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <h1 id="jive_content_id_Incorrect_Content_Matching_When_Using_Spreadsheets">Incorrect Content Matching When Using Spreadsheets</h1>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <p>Occasionally an attachment is blocked because of a match (usually numeric) although the content does not appear to be in the spreadsheet. This is as a result of some document types (e.g. Microsoft Excel) that have internal formatting features. These can cause Mimecast to match incorrectly. Examples include:</p>
         <ul>
          <li>Spreadsheet columns that have an internal numbering scheme (invisible to the user) which when analyzed appear to be numeric content.</li>
          <li>Dates that are stored internally in a long integer format.</li>
         </ul>
         <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
         <p>This internal numbering can be mistaken for Social Security Numbers, Credit Card numbers, etc. by the Mimecast content text analyzer. Unfortunately, there is no automatic way to avoid this, and the incorrectly held attachment must be released manually.</p>
