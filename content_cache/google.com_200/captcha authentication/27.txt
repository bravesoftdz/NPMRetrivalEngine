            <h1 id="markdown-header-thinbus-javascript-secure-remote-password-srp">Thinbus Javascript Secure Remote Password (SRP)</h1> 
            <p>This package provides a Javascript <a href="http://srp.stanford.edu/">Secure Remote Password</a> <a href="http://srp.stanford.edu/doc.html#papers">SRP-6a</a> implementation for web browsers to perform a zero-knowledge proof-of-password to a web server. It comes with compatible Java classes which use the <a href="https://bitbucket.org/connect2id/nimbus-srp">Nimbus SRP6a Java</a> library. </p> 
            <p>There are some implementations of SRP which are compatible with the Thinbus client code. This allows you to use Thibus client code in the broswer but other languages on the server: </p> 
            <ol> 
             <li><a href="https://bitbucket.org/simon_massey/thinbus-srp-spring-demo/overview">thinbus-srp-spring-demo</a> A Spring MVC application which uses the Thinbus JavaScript library to create accounts and login users with Spring Security. </li> 
             <li><a href="https://bitbucket.org/simon_massey/thinbus-php/overview">thinbus-php</a> Uses the Thinbus Javascript library to do SRP authentication to PHP server code. It also includes a PHP SRP client that you can use for server-to-server authentication or for generating temporary passwords. </li> 
             <li><a href="https://github.com/SthPhoenix/pysrp_thinbus">pysrp_thinbus</a> is a fork of <a href="https://github.com/cocagne/pysrp">pysrp</a> which is compatible with Thinbus so that you can use Python on the server. </li> 
            </ol> 
            <p>The spring demo app has been checked on IE8+, Edge, Chrome, FireFox, and Safari. </p>

            <h2 id="markdown-header-using">Using</h2> 
            <p>Check the <a href="https://bitbucket.org/simon_massey/thinbus-srp-spring-demo/overview">Thinbus Spring Demo</a>. It may or may not be running on the <a href="http://thinbus-n00p.rhcloud.com/">demo server</a> if not the build tool can run it locally for you. </p> 
            <p>For the definitions of the values discussed below please refer to the <a href="http://srp.stanford.edu/design.html">SRP design page</a>. The following sequence diagram shows how to register a user with an SRP salt and verifier as demonstrated by the <a href="https://bitbucket.org/simon_massey/thinbus-srp-spring-demo/overview">Thinbus Spring Demo</a>. </p> 
            <p><img alt="Thinbus SRP Register Diagram" src="http://simonmassey.bitbucket.io/thinbus/register.png" title="Thinbus SRP Register Diagram"></p> 
            <p>In the diagram above the user is shown a standard registration form which includes email and password fields. They enter their email and password and click the register button. JavaScript then generates their random <code>salt</code> and uses the salt, email and password to generate an SRP <code>verififer</code>. Only the <code>salt</code> and the <code>verifier</code> are transmitted to the server and they are saved into the database keyed by the users email. </p> 
            <p><strong>Note</strong> Always use browser developer tools to inspect what you actually post to the server and only post the values shown in the sequence diagram as defined in the <a href="http://srp.stanford.edu/design.html">SRP design page</a>. It is a protocol violation and security bug if the raw password is accidently transmitted to the server even if it is ignored by the server. </p> 
            <p>The following sequence diagram shows how to login a registered user. </p> 
            <p><img alt="Thinbus SRP Login Diagram" src="http://simonmassey.bitbucket.io/thinbus/login.png" title="Thinbus SRP Login Diagram"></p> 
            <p>In the diagram above the user is shown a standard login form. They enter their email and password and click the login button. JavaScript then makes an AJAX call using their email to load their <code>salt</code> and a one-time server challenge <code>B</code>. JavaScript creates a one-time client challenge <code>A</code> and uses all the information to compute a password proof <code>M1</code>. It then posts to the server the email, <code>A</code>, and <code>M1</code> as the users credentials. The server uses all the information to check the password proof. Only the email, client challenge <code>A</code> and the password proof <code>M1</code> are transmitted to the server. </p> 
            <p><strong>Note</strong> As per RFC 2945 the user ID (usually their email) is concatenated to their password when generating the verifier. This means that if a user changes <em>either</em> their email address or their password you need to generate a new verifier and replace the old one in the database. </p> 
            <p><strong>Note</strong> Always use browser developer tools to inspect what you actually post to the server and only post the values shown in the sequence diagram as defined in the <a href="http://srp.stanford.edu/design.html">SRP design page</a>. It is a protocol violation and a security bug to accidently transmit to the server anything else even if it is ignored by the server. </p> 
            <p><strong>Note</strong> the JavaScript client object (typically <code>SRP6JavascriptClientSessionSHA256</code>) must be destroyed after each login attempt. The object is intended to be a temporary object and should be deleted to erase all traces of the password. You must also destroy the password form field the user typed their password into. The normal way to achieve destroying any traces of the password is to unload the login page after every login attempt. This is trivial to do by reloading the login page upon authentication failure or by loading a main landing page upon successful login. </p> 
            <p><strong>Note</strong> that the server has to remember the one-time server challenge <code>B</code> that it gave to the browser in order to check the users password proof. This requires storing the one-time challenge value either in the database, the server session or a server cache for the short duration of the login protocol. You cannot pass this value back to the server from the client without compromising security. The server should not use any values transmitted from the client other than those shown in the sequence diagram and named in the <a href="http://srp.stanford.edu/design.html">SRP design page</a>.</p> 
            <p>There is an optional step <code>client.step3(M2)</code> where <code>M2</code> is the server's proof of a shared session key to the client. You can return <code>M2</code> from the server to check the browser has a matching shared secret if you wish to use that for further cryptography. If your web application is distributed as a native mobile application such that the client is running trusted JavaScript then the <code>M2</code> proof is an additional check of the authenticity of the server; it confirms to trusted JavaScript that the server knows the verifier matching the user password. </p> 
            <p><strong>Note</strong> if you want to use the shared session key for follow-on cryptography you should use <code>client.getSessionKey()</code> to retrieved the session key from the thinbus object and destroy the thinbus object as discussed above. The typical way to do this is to put the session key into browser local session storage. Then you can unload the login page then load a main landing page that collects the session key from storage. </p> 
            <h2 id="markdown-header-custom-configuration">Custom Configuration</h2> 
            <p>SHA-256 is the strongest hash algorithm Java 1.7/1.8 supports out of the box so it is recommended. The Javascript SHA-256 version is in <code>thinbus-srp6a-sha256-min.js</code>. The corresponding Java server SRP session class is <code>SRPJavascriptServerSessionSHA256</code>. The Java code is configured via constructor parameters. The JavaScript code is configured by defining an <code>SRP6CryptoParams</code> object literal before you include the main thinbus file: </p> 
            <div class="codehilite language-Javascript">
             <pre><span></span><span class="kd">var</span> <span class="nx">SRP6CryptoParams</span><span class="o">=</span> <span class="p">{</span>
    <span class="nx">N_base10</span><span class="o">:</span> <span class="s2">"19502997308733...</span>
<span class="s2">    g_base10: "</span><span class="mi">2</span><span class="s2">", </span>
<span class="s2">    k_base16: "</span><span class="mi">1</span><span class="nx">a3d1769e1d6337</span><span class="p">....</span>
<span class="p">}</span>
</pre>
            </div> 
            <p>An extra implementation detail is that the JavaScript must be configure with <code>k</code>. In the SRP protocol <code>k</code> is computed from <code>N</code> and <code>g</code> which is why the Java code does not need it. The catch is that Nimbus uses the <code>java.net.BigInteger</code> byte array constructor when generating <code>k</code>. This byte array constructor is not available in JavaScript so the constant value computed by the Java must be added to the Javascript configuration. The <code>toString()</code> of the Java class will print each of <code>N</code>, <code>g</code> and <code>k</code> in the correct formats to configure the Javascript. </p> 
            <h2 id="markdown-header-creating-a-custom-large-safe-prime">Creating A Custom Large Safe Prime</h2> 
            <p>You can use openssl to create your own large safe prime. To help with this there is a class which parses the output of the openssl safe prime generation command: </p> 
            <div class="codehilite language-sh">
             <pre><span></span><span class="c1"># create your parameters set &lt;bit-length&gt; (recommended minimum of 2048)</span>
openssl dhparam -text &lt;bit-length&gt; <span class="p">|</span> tee /tmp/my_dhparam.txt

<span class="c1"># build the runnable jar-with-dependencies </span>
mvn assembly:assembly

<span class="c1"># use the jar name which matches the output of the assembly command. set &lt;hash&gt; to the name of the java hashing algorithm to use e.g. "SHA-256"</span>
java -jar target/thinbus-srp6a-js-&lt;version&gt;-jar-with-dependencies.jar /tmp/my_dhparam.txt &lt;hash&gt;
</pre>
            </div> 
            <p>This will output something like: </p> 
            <div class="codehilite language-sh">
             <pre><span></span>bits:1024
hashing to create <span class="s1">'k'</span> using SHA-256
computing
N base10: <span class="m">19502997308733</span>...
g base10: <span class="m">2</span>
k base16: 1a3d1769e1d6337...
</pre>
            </div> 
            <p>Else you could try the online version of that tool if it is currently up and running over on the <a href="http://thinbus-n00p.rhcloud.com/dhparam">demo server</a>. </p> 
            <p>You then use the <code>N</code> and <code>g</code> value to configure the Java session and use the <code>N</code>, <code>g</code> and <code>k</code> values to configure the Javascript session as outlined above. Also see <code>TestSRP6JavascriptClientSessionSHA256.js</code> which configures matching Java and Javascript sessions and tests them against one another. You should edit that test to use your own safe prime values and confirm that the test passes before attempting the use your configuration with a web browser. </p> 
            <p>Using the demo 2048 bit prime a modern developer workstation takes less than 90ms to do the math. Trying out smaller 1024 bit primes on a low spec 2010 mac the browser takes between 0.05s and 0.10s to run the main srp work. The timings depend on which of Firefox, Chrome or Safari is used. YMMV as Javascript runtimes and mobile hardware may vary considerably so you should test the user experience on all the browsers you are targeting. </p> 
            <h2 id="markdown-header-javascript-code">Javascript Code</h2> 
            <p>Other JavaScript source files in the jar show the original copyright of the libraries and the un-minified client session: </p> 
            <ul> 
             <li><code>js/biginteger.js</code> BigInteger math package. </li> 
             <li><code>js/isaac.js</code> A random number generator which aims to be secure. </li> 
             <li><code>js/random.js</code> A random number class which tries to use <code>window.crypto</code> or <code>window.msCrypto</code> random numbers else fall-backs to the <code>isaac.js</code> generator. </li> 
             <li><code>js/sha256.js</code> The Crypto.JS SHA256 hash algorithm. Recommended. </li> 
             <li><code>js/sha1.js</code> The Crypto.JS SHA1 hash algorithm. Optional. </li> 
             <li><code>js/thinbus-srp6client.js</code> The SRP client session</li> 
            </ul> 
            <h2 id="markdown-header-recommendations">Recommendations</h2> 
            <ul> 
             <li>Use Thinbus SRP over HTTPS. Configure your webserver to mark session cookies as secure to prevent accidental use of HTTP. Configure <a href="https://en.wikipedia.org/wiki/HTTP_Strict_Transport_Security#HSTS_mechanism_overview">HSTS</a> to force HTTPS with your service. If your customers use a company supplied computer going via a corporate web proxy then HTTPS may be <a href="http://security.stackexchange.com/questions/63304/how-can-my-employer-be-a-man-in-the-middle-when-i-connect-to-gmail">decrypted and monitored</a>. HTTPS may be compromised due to things like <a href="http://nakedsecurity.sophos.com/2013/12/09/serious-security-google-finds-fake-but-trusted-ssl-certificates-for-its-domains-made-in-france/">bad certs in the wild</a>. HTTPS may be compromised by bugs or misconfigurations such as <a href="http://en.wikipedia.org/wiki/Heartbleed">Heartbleed</a>. HTTPS alone cannot protected against leaking passwords into error messages in your webserver logs. SRP over HTTPS is much safer than either used alone. </li> 
             <li>Add a javascript password strength meter and only allow users to register a verifier for a strong password. The best cryptography in the world won't protect your users if they use "password" as their password.</li> 
             <li>Use a custom large safe prime number <code>N</code> using the instructions above. <strong>Tip:</strong> Check on the browsers and hardware you are targeting that the math runs fast enough for a good user experience for your chosen bit length. </li> 
             <li>Make the salt column in the database <code>not null</code> and add a uniqueness constraint. </li> 
             <li>Use symmetric AES encryption with a key only visible at the webserver to encrypt the verifier <code>v</code> value within the database. This protects against off-site database backups being used in an offline dictionary attack against <code>v</code>. </li> 
             <li>You can prevent privileged accounts from logging in using legacy browsers by checking <code>random16byteHex.isWebCryptoAPI()</code> when fetching the user salt; simply abort the protocol for privileged accounts when secure random numbers are not available at the browser. If you allow the use of browsers that don't have the <code>WebCryptoAPI</code> secure random number APIs then the fallback random generator hashes <code>window.cookie</code> as part of the generator seed. Consider adding a secure random cookie to help seed the fallback generator; see PRNG.md for more info.</li> 
             <li>Don't include any JS files <a href="http://stackoverflow.com/a/3613162/329496">or any CSS files</a> from external sites onto your login page. </li> 
             <li>Count the number of failed password attempts and present the user with a CAPTCHA after a dozen attempts. This slows down scripted online dictionary attacks. Consider suspending the account (possibly temporarily) after a large number of contiguous failed attempts to defeat someone carefully researching a user then trying to guess their likely password. </li> 
            </ul>
