        <p> However, this pertains to the cross-origin&nbsp;requests (as opposed to same-origin request), which turns&nbsp;out to be the&nbsp;unbreakable wall for requesting page content from pure JavaScript.</p> 
        <h3>Same Origin Policy</h3> 
        <!-- shortcode box --> 
        <div class="shortcode clearfix box gray note">
         Same-origin policy restricts how a document or script loaded from one origin can interact with a resource from another origin. It is a critical security mechanism for isolating potentially malicious documents. 
         <a href="https://developer.mozilla.org/en-US/docs/Web/Security/Same-origin_policy" target="_blank">Source</a>.&nbsp;
        </div> 
        <!-- /shortcode box --> Same-origin policy requires that in web requests made from the client side (mysite.com) both domain name and protocol (http, https) should be equal.&nbsp;So security limitations do not allow you to request another domain website (example.com) or any&nbsp;of its resources.
        <p></p>

        <h2>Cross-Origin Resource Sharing (CORS)</h2> 
        <p>The main concept is that&nbsp;a&nbsp;target server may allow some other origins (or all of them) to request its resources.&nbsp;Server&nbsp;configured for allowing cross-origin requests is useful&nbsp;for&nbsp;the cross-domain API access of&nbsp;its resources.</p>

        <h2>How does WordPress load foreign&nbsp;page shots into its admin panel</h2> 
        <p>WordPress CMS can load of foreign resources with server-side call (if having access to wp-admin  just&nbsp;visit: &lt;site&gt;/wp-admin/edit-comments.php).         <p><img class="aligncenter wp-image-8215 size-full" src="http://scraping.pro/wp-content/uploads/wp_load_link_preview.png" alt="wp load link preview" width="580" height="417"></p> 
        <p>Now the CMS makes HTTP request to its own server, embedding the link to the foreign resource. Obviously the WordPress server makes request to the resource by provided link of interest and returns the content:<br> <img class="aligncenter size-full wp-image-8216" src="http://scraping.pro/wp-content/uploads/wp_load_link_request.png" alt="wp load resource request" width="580" height="305"></p> 
        <p>The only the thing is that the content returned by WordPress being an image: <em>content-type:&nbsp;image/jpeg. </em>&nbsp;You can program server to return HTML code, but thats server-side data extraction.</p> 
        <h2>Conclusion</h2> 
        <p>The client-side (from your browser) scraping with JavaScript is not practical today. (1) The browser capabilities are far less compared to web servers (speed, memory, etc.). (2) The Same-Origin Policy safeguards sites from cross-origin&nbsp;requests, avoiding&nbsp;XSS attacks&nbsp;threat. CORS is limited scope applicable. (3) Ive also tried to&nbsp;emulate the cross-domain HTTP request by a virtual form submission to&nbsp;load a&nbsp;result into an iframe, but this failed&nbsp;since the browser restrictions forbid scripts to handle raw response&nbsp;HTML cause of XSS attacks threat. (4) The last option&nbsp;is the indirect&nbsp;requesting thru a&nbsp;domain server (mysite.com, who actually extracts). Eg. WordPress loading foreign pages previews.</p> 
        <p>Feel free to add more to this topic (using comments).</p>
