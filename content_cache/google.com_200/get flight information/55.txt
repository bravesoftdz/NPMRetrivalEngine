         <h1>Sample Application</h1> 
         <p class="edit-link" style="font-size: 17px; opacity: 0.6; text-align: right;"><a target="_blank" href="https://github.com/couchbase/docs-cb4/edit/4.6/content/sdk/sample-application.dita">Edit on GitHub</a></p> 
         <p class="shortdesc">Couchbase Travel is a sample web application that demonstrates how to interact with the Couchbase query services via the SDKs</p> 
         <p class="p">Each SDK comes with its own implementation of the backend for the application. You can download the complete source code and then build and run the app. While the app runs, you can get a peek of what is happening in the backend via the "Narration" (a console like UI element that can display N1QL queries for instance). It is only visible in components that communicate with the server (but can also be collapsed). <br><img class="image" src="http://static.couchbaseinc.hosting.ca.onehippo.com/images/server/4.6/20170920-104712/travel-app-narration.png" width="650"><br></p> 
         <p class="p">The documentation for the travel app illustrates the data model and walks through the N1QL and FTS queries used to select flights and search for hotels.</p> 
         <p class="p">The <a class="xref" href="https://github.com/couchbaselabs/try-cb-frontend" target="_blank">travel app front-end</a> is the same in each SDK implementation and is built on an <a class="xref" href="https://angular.io" target="_blank">Angular2</a> stack.</p> 
         <p class="p">The focus of this tutorial is the backend, which is a REST API available in the following languages:</p> 
         <ul class="ul"> 
          <li class="li"><a class="xref" href="nodejs/sample-app-backend.html" title="The Node.js SDK tutorial bridges the gap between simple and advanced concepts by walking through a complete web application.">Node.js</a></li> 
          <li class="li"><a class="xref" href="java/sample-app-backend.html" title="The Java SDK tutorial bridges the gap between simple and advanced concepts by walking through a complete web application.">Java</a></li> 
          <li class="li"><a class="xref" href="dotnet/sample-app-backend.html" title="The .NET SDK tutorial bridges the gap between simple and advanced concepts by walking through a complete web application.">.NET</a></li> 
          <li class="li"><a class="xref" href="python/sample-app-backend.html" title="The Python SDK tutorial bridges the gap between simple and advanced concepts by walking through a complete web application.">Python</a></li> 
          <li class="li"><a class="xref" href="go/sample-app-backend.html" title="The Go SDK tutorial bridges the gap between simple and advanced concepts by walking through a complete web application.">Go</a></li> 
          <li class="li"><a class="xref" href="c/sample-app-backend.html">C/C++ (basic interaction with data model only)</a></li> 
         </ul> 
         <p class="p">Please refer to the travel-app documentation of your SDK of choice for specifics for that particular SDK, like backend dependencies, source-code checkout and running instructions.</p> 
         <p class="p">The application allows users to find flights by entering airports and travel dates. For the airport entries, the app uses a N1QL query bound to an Angular type ahead directive to suggest airport names based on the first few letters entered.</p> 
         <section class="section" id="travel-sample-app-generic__note-installing"> 
          <h2 class="title sectiontitle">Generic set up</h2> 
          <div class="p">
            You'll need: 
           <ul class="ul"> 
            <li class="li">Your favorite editor or IDE</li> 
            <li class="li">Your SDK of choice and its specific dependencies</li> 
            <li class="li">The sample app source code for your SDK from GitHub (check the instructions and make sure to use the implementation compatible with the v2 API)</li> 
            <li class="li">A local <samp class="ph codeph">Couchbase 4.5</samp> (or greater) installation (make sure that the <samp class="ph codeph">travel-sample</samp> bucket has been loaded from 4.5 and that there is, at least, one node with data, query, index and full text search services in the cluster)</li> 
            <li class="li">That's it!</li> 
           </ul> 
          </div> 
          <p class="p">To start, it is easiest if you run Couchbase Server and the travel sample app on the same machine. It's not required to run your development environment this way, and advanced Multi-Dimensional Scaling (MDS) configurations are supported. It's just easier to start a development environment with components running locally.</p> 
          <p class="p">Download <a class="xref" href="http://www.couchbase.com/download" target="_blank">Couchbase Server 4.6 </a>and install it. As you follow the download instructions and setup wizard, make sure you keep all the services (<samp class="ph codeph">data</samp>, <samp class="ph codeph">query</samp>, and <samp class="ph codeph">index</samp>) selected. Make sure also to install the sample bucket named <samp class="ph codeph">travel-sample</samp> (introduced in Couchbase Server 4.0) because it contains the data used in this tutorial.</p> 
          <div class="note warning"> 
           <span class="warningtitle">Warning:</span> It is important to install the 
           <samp class="ph codeph">travel-sample</samp> bucket from a 4.5 or later version of Couchbase Server, as it has been enriched with hotels data. If you have installed the sample bucket from an earlier version, delete the bucket and follow the instructions below. 
          </div> 
          <p class="p">If you already have Couchbase Server 4.5 or later installed but did not install the <samp class="ph codeph">travel-sample</samp> bucket (or you installed it from a 4.0 version), open the Couchbase Web Console and select <span class="ph menucascade"><span class="ph uicontrol">Settings</span> &gt; <span class="ph uicontrol">Sample Buckets</span></span>. Select the <samp class="ph codeph">travel-sample</samp> checkbox, and then click <span class="ph uicontrol">Create</span>. A notification box in the upper-right corner disappears when the bucket is ready to use.</p> 
          <p class="p">Finally, in order to be able to search for hotels, an FTS index called <samp class="ph codeph">hotels</samp> must be created on the <samp class="ph codeph">travel-sample</samp> bucket.           <h2 class="title sectiontitle">Architecture</h2> 
          <p class="p">The application serves an HTML file named <span class="ph filepath">index.html</span> (in an <span class="ph filepath">src/app/</span> folder usually contained in a root folder dedicated to frontend files, eg. the <span class="ph filepath">public</span> path in the Node.js application). This file contains references to various included script files for front side JavaScript components such as Bootstrap and jQuery. It also contains a small script snippet that bootstraps the Angular2 application, as well as an Angular loop (<samp class="ph codeph">{{#each scripts.polyfills}}</samp>) to load <span class="keyword apiname">polyfills</span> (implementation of html5/web component elements that can be used as a backup if the browser doesn't provide the corresponding feature natively).</p> 
          <p class="p">Angular2 works in terms of <span class="keyword apiname">Components</span>, and the main component that gets displayed at bootstrap is in <span class="ph filepath">src/app/try.component.ts</span>. It is in charge of displaying the common elements of the single-page app, and defines <span class="keyword apiname">Routes</span>, determining which component to show depending on the URL. The place at which said components are shown in the html is represented by a component <span class="keyword apiname">tag</span> provided by the angular <span class="keyword apiname">Router</span>: <samp class="ph codeph">&lt;router-outlet&gt;&lt;/router-outlet&gt;</samp>.</p> 
          <p class="p">The application is then split into several packages. Each talks to a specific subset of the REST API provided by the backend. Each package has a corresponding main Angular <span class="keyword apiname">Component</span> that provides the methods to interact with the API or perform work. Their corresponding views then use these methods to populate data in the application.</p> 
          <p class="p">The backend REST API, in its version associated with Couchbase Server 4.5 (version 2), can be found in the <a class="xref" href="https://github.com/couchbaselabs/try-cb-frontend/blob/master/documentation/try-cb-api-spec-v2.adoc" target="_blank">couchbaselabs/try-cb-frontend</a> repository on GitHub.</p> 
          <figure class="fig fignone"> 
           <span class="figcap">Figure 3. The Application Services</span> 
           <img class="image" src="http://static.couchbaseinc.hosting.ca.onehippo.com/images/server/4.6/20170920-104712/travel-app-archi.png" width="650"> 
          </figure>

         <section class="section"> 
          <h2 class="title sectiontitle">The Home component</h2> 
          <div class="p"> 
           <span class="ph filepath">src/app/+home</span> is the main page, dealing with flight searches. In the backend, these searches are performed using 
           <span class="keyword apiname">N1QL</span> queries. It has three main methods: 
           <ul class="ul"> 
            <li class="li">Two at <span class="keyword apiname">getToAirport</span> and <span class="keyword apiname">getFromAirport</span> which are both REST API calls to the <span class="ph filepath">/api/airports</span> REST endpoint</li> 
            <li class="li">And one at <span class="keyword apiname">findFlights</span>, which is a REST API call to the <span class="ph filepath">/api/flightPaths/{to}/{from}</span> endpoint</li> 
           </ul> 
          </div> 
          <figure class="fig fignone"> 
           <span class="figcap">Figure 4. Home Component call hierarchy</span> 
           <img class="image" src="http://static.couchbaseinc.hosting.ca.onehippo.com/images/server/4.6/20170920-104712/travel-app-archi-home.png" width="650"> 
          </figure> 
          <p class="p">The application attempts to find an airport based on the codes used for the <samp class="ph codeph">name</samp> (case sensitive), and codes for Federal Aviation Administration (FAA) or International Civil Aviation Organization (ICAO). The backend method allows the user to search by FAA code, ICAO code or airport name. It then prepares a N1QL query based on the selection criteria (using bucket information from its configuration). The query is then sent to the Couchbase query service, all by using the backend's associated SDK.</p> 
          <p class="p">Using the data model, if you enter <kbd class="ph userinput">SEA</kbd>, <kbd class="ph userinput">KSEA</kbd>, or <kbd class="ph userinput">Seattle</kbd>, the typeahead directive gives you the option to select <span class="ph uicontrol">Seattle Tacoma Intl.</span> To see the results of the query, watch the terminal window in which the application is running. The example is showing an output for the N1QL statements:</p> 
          <pre><code class="pre codeblock language-sql">QUERY: SELECT airportname FROM `travel-sample` WHERE faa ='SEA'
QUERY: SELECT airportname FROM `travel-sample` WHERE icao ='KSEA'
QUERY: SELECT airportname FROM `travel-sample` WHERE airportname LIKE 'Seattle%'</code></pre> 
          <p class="p">After the airports are selected and a leave date is entered, the application tries to find route and schedule information for an airline that services the requested flight path. The call to the REST API <span class="ph filepath">/api/flightPaths/{to}/{from}</span> on the server is initiated when the user clicks the <span class="ph uicontrol">Find Flights</span> button.</p> 
          <p class="p">The elements in the form bind their controls to corresponding objects in the component, which allows to perform some validation checking among other things. To learn more about forms and validation in Angular2, please refer to the <a class="xref" href="https://angular.io/docs/ts/latest/guide/forms.html" target="_blank">official documentation</a>. The submit button calls the component's <samp class="ph codeph">findFlights()</samp> function, which in turn invokes the server REST API <span class="ph filepath">/api/flightPaths/{to}/{from}/</span> function either once for a one way flight or twice for a round-trip flight. When it requests the return flight for a round trip, it just uses the reverse <samp class="ph codeph">to</samp> and <samp class="ph codeph">from</samp> sequence. While <samp class="ph codeph">to</samp> and <samp class="ph codeph">from</samp> are made part of the URL, the API also expects an additional query parameter, <samp class="ph codeph">leaves</samp>, that contains the requested travel date.</p> 
          <p class="p">In the backend, the code corresponding to the <span class="keyword apiname">flightPaths</span> endpoint performs several important steps for processing the request to find flights:</p> 
          <ul class="ul"> 
           <li class="li">It determines the FAA identifier for the <samp class="ph codeph">to</samp> and <samp class="ph codeph">from</samp> airports.</li> 
           <li class="li">It searches for routes and schedule information based on the source and destination airports and the requested dates. The collection of schedule documents is nested in the route document for each route. They are grouped by day of the week (1-7), and the requested dates are compared to the day of the week to see what flights are available on those particular dates. It returns a list of flights that includes data for the following fields: airline, flight, departure time ("utc" field), from, to, price (randomly assigned by the backend) and aircraft ("equipment" field). </li> 
          </ul> 
          <p class="p"> The Query can be seen in the terminal window that is running the backend application: </p> 
          <pre><code class="pre codeblock language-sql">QUERY:
SELECT a.name, s.flight, s.utc, r.sourceairport, r.destinationairport, r.equipment
FROM `travel-sample` r
UNNEST r.schedule s
JOIN `travel-sample` a
ON KEYS r.airlineid
WHERE r.sourceairport='SEA' AND r.destinationairport='MCO' AND s.day=6
ORDER BY a.name</code></pre> 
         </section> 
         <section class="section"> 
          <h2 class="title sectiontitle">N1QL query anatomy</h2> 
          <p class="p">The Couchbase Query API is a powerful tool for efficient retrieval of information from a document data store. In each SDK-specific travel application, queries about flights are created by a dedicated module (for example in Node.js, the <span class="ph filepath">flightPath.js</span> module). Here's an example of a query that finds flights between Seattle-Tacoma International Airport (SEA) and Orlando International Airport (MCO), followed by a description of what's happening in the query:</p> 
          <pre><code class="pre codeblock language-sql">SELECT a.name, s.flight, s.utc, r.sourceairport, r.destinationairport, r.equipment
FROM `travel-sample` r
UNNEST r.schedule s
JOIN `travel-sample` a ON KEYS r.airlineid
WHERE r.sourceairport='SEA' AND r.destinationairport='MCO' AND s.day=6
ORDER BY a.name</code></pre> 
          <p class="p">N1QL provides JOIN functionality, something previously not possible in a document database. For two documents to be joined in the result of a SELECT statement, one of them must contain a field whose value is equal to the Couchbase key of the other document. The following example shows two documents that demonstrate that requirement and a SELECT statement that joins them:</p> 
          <pre><code class="pre codeblock">"keyA" is the Couchbase KV key for Doc A.
Doc A: { some fields }

Doc B: { some fields "joinField": "keyA" }

SELECT * FROM default b JOIN default a ON KEYS b.joinField</code></pre> 
          <p class="p">The data model for the travel application includes an <samp class="ph codeph">airlineid</samp> field in each <samp class="ph codeph">route</samp> document. That <samp class="ph codeph">airlineid</samp> field is used as a foreign key identifier and corresponds to the key for an <samp class="ph codeph">airline</samp> document. To select the airline name <samp class="ph codeph">a.name</samp>, the query uses the following clause: <samp class="ph codeph">JOIN `travel-sample` a ON KEYS r.airlineid</samp>.</p> 
          <p class="p">One of the powerful features available in the N1QL query language is the ability to <samp class="ph codeph">UNNEST</samp> or flatten, the results returned in the <samp class="ph codeph">SELECT</samp> statement. This is frequently needed when working with JSON documents, which may have rich heirarchies. In the data model for the travel application, each route document contains a nested collection of schedule documents. To alleviate a complicated JSON parsing code pattern for the return results, you can have the query execution <samp class="ph codeph">UNNEST</samp> the schedule documents, so they become the root-level fields in the returned results.</p> 
          <p class="p">Standard SQL syntax is used in the <samp class="ph codeph">WHERE</samp> clause for the <samp class="ph codeph">SELECT</samp> statement. The result set is ordered by the <samp class="ph codeph">a.name</samp> field, which contains the airline name.</p> 
          <p class="p">Refer to the <a class="xref" href="n1ql-query.html#toplevel" title="You can query for documents in Couchbase using the N1QL query language, a language based on SQL, but designed for structured and flexible JSON documents. Querying can solve typical programming tasks such as finding a user profile by email address, facebook login, or user ID.">Querying with N1QL</a> section for more information on using N1QL with the SDKs.</p> 
         </section> 
         <section class="section"> 
          <h2 class="title sectiontitle">The Hotels component</h2> 
          <p class="p"><span class="ph filepath">src/app/+hotels</span> allows to search for hotels. In the backend these searches are performed using <span class="keyword apiname">FTS</span> queries, performed using the <span class="ph filepath">/api/hotel/</span> REST API endpoint.</p> 
          <figure class="fig fignone"> 
           <span class="figcap">Figure 5. Hotels component call hierarchy</span> 
           <img class="image" src="http://static.couchbaseinc.hosting.ca.onehippo.com/images/server/4.6/20170920-104712/travel-app-archi-hotel.png" width="650"> 
          </figure> 
          <p class="p">The form is bound to the component using a <span class="keyword apiname">FormBuilder</span>, and it has two optional text inputs: a description and a location field. When submitting the form, the component's <span class="keyword apiname">findHotels</span> method is called with a JSON representation of the form's content. The <span class="keyword apiname">findHotels</span> method will use these values to determine if it should hit a more specific endpoint, in the case where a particular description keyword or location are requested by the form.</p> 
          <p class="p">For instance, to search for an hotel in France with a description matching "golf" (in a full text search sense, possibly including alternative spellings, words with the same root, etc...), one should use the <span class="ph filepath">/api/hotel/France/golf</span> endpoint. The possible endpoints are:</p> 
          <ul class="ul"> 
           <li class="li"><span class="ph filepath">/api/hotel</span> to find all hotels</li> 
           <li class="li"><span class="ph filepath">/api/hotel/{description}/</span> to find all hotels matching a keyword/phrase in their name or description.</li> 
           <li class="li"><span class="ph filepath">/api/hotel/{description}/{location}/</span> to find all hotels matching the keyword/phrase and a location keyword (in either of the location-related fields: country, city, state, address...)</li> 
           <li class="li"><span class="ph filepath">/api/hotel/*/{location}/</span> to find all hotels within a location (not using keyword)</li> 
          </ul> 
          <p class="p">As a second step, for each hit returned by the FTS service the information that we want to display is collected by the backend using the subdocument API. This allows the client to request that the cluster to only transmit specific parts of the hotel document to be sent back to the client for display.</p> 
          <p class="p">Refer to the <a class="xref" href="subdocument-operations.html#subdoc-operations" title="sub-document operations can be used to efficiently access parts of documents. Sub-document operations may be quicker and more network-efficient than full-document operations such as upsert, update and get because they only transmit the accessed sections of the document over the network. Sub-document operations are also atomic, allowing safe modifications to documents with built-in concurrency control.">Sub-Document Operations</a> section for more information on using the Subdocument API with the SDKs.</p> 
          <p class="p"> Both parts of the request can be individually seen in the narration component: <br><img class="image" src="http://static.couchbaseinc.hosting.ca.onehippo.com/images/server/4.6/20170920-104712/travel-app-narration.png" width="650"><br></p>

         <section class="section"> 
          <h2 class="title sectiontitle">FTS query anatomy</h2> 
          <p class="p">The Couchbase Search API (FTS) is a powerful tool for searching text inside a document data store. In each SDK-specific travel application, queries about hotels are created by a dedicated module (for example in Java, the <span class="ph filepath">Hotel</span> service). Here's an example of a query that finds hotels mentioning "golf" in France, followed by a description of what's happening in the query:</p> 
          <pre><code class="pre codeblock language-json">{"query":{"conjuncts": [
    {"field":"type","term":"hotel"},
    {"disjuncts":[
        {"field":"country","match_phrase":"France"},
        {"field":"city","match_phrase":"France"},
        {"field":"state","match_phrase":"France"},
        {"field":"address","match_phrase":"France"}
    ]},
    {"disjuncts":[
        {"field":"description","match_phrase":"golf"},
        {"field":"name","match_phrase":"golf"}
    ]}
]},
"size":100}</code></pre> 
          <p class="p">The <samp class="ph codeph">query</samp> is actually a composite (<samp class="ph codeph">conjuncts</samp>) of 3 sub-queries: a <samp class="ph codeph">term</samp> query and two <samp class="ph codeph">disjuncts</samp> queries.</p> 
          <ul class="ul"> 
           <li class="li">The term query matches a term as-is, without performing any analysis: it must match <strong class="ph b">exactly</strong> what is stored in the index (which could be just the root of a word). Here we use it to restrict our search to the <samp class="ph codeph">hotel</samp> <span class="keyword apiname">type</span>.</li> 
           <li class="li">A disjunction query matches if any of its sub-queries matches. We use the first only when there is a <samp class="ph codeph">location</samp> provided, to search for it in any of the <span class="keyword apiname">country</span>, <span class="keyword apiname">city</span>, <span class="keyword apiname">state</span> and <span class="keyword apiname">address</span> fields. We use a <span class="keyword apiname">match phrase</span> sub-query on each, which will attempt to find several keywords, analyzing them to remove stop words, include close results like words from the same root, etc...</li> 
           <li class="li">The second disjunction query is added whenever a description is provided for the search. It performs two match phrase queries on the description, respectively in the <span class="keyword apiname">name</span> and <span class="keyword apiname">content</span> fields of the landmark.</li> 
           <li class="li">There are quite a few other FTS query types. For instance, a match query is much like a match phrase query, except it only considers one term to search for, and can apply fuzzyness (eg. to match close words with a slightly different spelling).</li> 
          </ul> 
          <p class="p">Finally, we notice that at the same level as the <samp class="ph codeph">query</samp> entry there is also a <samp class="ph codeph">size</samp> attribute. This is one of the many tunings applicable to the whole query, a top-level parameter that limits the results to 100 hits.</p> 
          <p class="p">Refer to the <a class="xref" href="full-text-search-overview.html#full-text-search" title="Couchbase Server Full Text Search (FTS) enables you to create, manage, and query full text indexes on JSON documents stored in a Couchbase bucket. You should use search instead of query when your application requires natural language processing when searching.">Full Text Search</a> section for more information on using FTS with the SDKs.</p>
