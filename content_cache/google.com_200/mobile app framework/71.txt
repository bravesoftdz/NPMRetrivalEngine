           <h1 name="3793" id="3793" class="graf graf--h3 graf--leading graf--title">Isomorphic JavaScript: The Future of Web&nbsp;Apps</h1>
           <p name="b1f1" id="b1f1" class="graf graf--p graf-after--h3">By Spike Brehm</p>
           <blockquote name="7328" id="7328" class="graf graf--blockquote graf-after--p">
            This post has been 
            <a href="http://venturebeat.com/2013/11/08/the-future-of-web-apps-is-ready-isomorphic-javascript/" data-href="http://venturebeat.com/2013/11/08/the-future-of-web-apps-is-ready-isomorphic-javascript/" class="markup--anchor markup--blockquote-anchor" rel="nofollow noopener" target="_blank">cross-posted</a> on VentureBeat.
           </blockquote>
           <p name="53e3" id="53e3" class="graf graf--p graf-after--blockquote">At Airbnb, we’ve learned a lot over the past few years while building rich web experiences. We dove into the single-page app world in 2011 with our <a href="https://m.airbnb.com/" data-href="https://m.airbnb.com/" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">mobile web site</a>, and have since launched <a href="https://www.airbnb.com/wishlists" data-href="https://www.airbnb.com/wishlists" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">Wish Lists</a> and our newly-redesigned <a href="https://www.airbnb.com/s" data-href="https://www.airbnb.com/s" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">search page</a>, among others. Each of these is a large JavaScript app, meaning that the bulk of the code runs in the browser in order to support a more modern, interactive experience.</p>
           <p name="fe34" id="fe34" class="graf graf--p graf-after--p">This approach is commonplace today, and libraries like Backbone.js, Ember.js, and Angular.js have made it easier for developers to build these rich JavaScript apps. We have found, however, that these types of apps have some critical limitations. To explain why, let’s first take a quick detour through the history of web apps.</p>
           <h4 name="cff2" id="cff2" class="graf graf--h4 graf-after--p">JavaScript Grows&nbsp;Up</h4>
           <p name="bf10" id="bf10" class="graf graf--p graf-after--h4">Since the dawn of the Web, the browsing experience has worked like this: a web browser would request a particular page (say, “http://www.geocities.com/"), causing a server somewhere on the Internet to generate an HTML page and send it back over the wire. This has worked well because browsers weren’t very powerful and HTML pages represented documents that were mostly static and self-contained. JavaScript, created to allow web pages to be more dynamic, didn’t enable much more than image slideshows and date picker widgets.</p>
           <p name="4b69" id="4b69" class="graf graf--p graf-after--p">After years of advances in personal computing, creative technologists have pushed the web to its limits, and web browsers have evolved to keep up. Now, the Web has matured into a fully-featured application platform, and fast JavaScript runtimes and HTML5 standards have enabled developers to create the rich apps that before were only possible on native platforms.</p>
           <h4 name="8a1e" id="8a1e" class="graf graf--h4 graf-after--p">The Single-Page App</h4>
           <p name="e705" id="e705" class="graf graf--p graf-after--h4">It wasn’t long before developers started to build out entire applications in the browser using JavaScript, taking advantage of these new capabilities. Apps like Gmail, the classic example of the single-page app, could respond immediately to user interactions, no longer needing to make a round-trip to the server just to render a new page.</p>
           <p name="468c" id="468c" class="graf graf--p graf-after--p">Libraries like Backbone.js, Ember.js, and Angular.js are often referred to as client-side MVC (Model-View-Controller) or MVVM (Model-View-ViewModel) libraries. The typical client-side MVC architecture looks something like this:</p>

           <p name="bc1a" id="bc1a" class="graf graf--p graf-after--figure">The bulk of the application logic (views, templates, controllers, models, internationalization, etc.) lives in the client, and it talks to an API for data. The server could be written in any language, such as Ruby, Python, or Java, and it mostly handles serving up an initial barebones page of HTML. Once the JavaScript files are downloaded by the browser, they are evaluated and the client-side app is initialized, fetching data from the API and rendering the rest of the HTML page.</p>
           <p name="cd25" id="cd25" class="graf graf--p graf-after--p">This is great for the user because once the app is initially loaded, it can support quick navigation between pages without refreshing the page, and if done right, can even work offline.</p>
           <p name="16fb" id="16fb" class="graf graf--p graf-after--p">This is great for the developer because the idealized single-page app has a clear separation of concerns between the client and the server, promoting a nice development workflow and preventing the need to share too much logic between the two, which are often written in different languages.</p>
           <h3 name="2c9b" id="2c9b" class="graf graf--h3 graf-after--p">Trouble in&nbsp;Paradise</h3>
           <p name="f7e0" id="f7e0" class="graf graf--p graf-after--h3">In practice, however, there are a few fatal flaws with this approach that prevent it from being right for many use cases.</p>
           <h4 name="423b" id="423b" class="graf graf--h4 graf-after--p">SEO</h4>
           <p name="865e" id="865e" class="graf graf--p graf-after--h4">An application that can only run in the client-side cannot serve HTML to crawlers, so it will have poor SEO by default. Web crawlers function by making a request to a web server and interpreting the result; but if the server returns a blank page, it’s not of much value. There are workarounds, but not without jumping through some hoops.</p>
           <h4 name="235b" id="235b" class="graf graf--h4 graf-after--p">Performance</h4>
           <p name="a461" id="a461" class="graf graf--p graf-after--h4">By the same token, if the server doesn’t render a full page of HTML but instead waits for client-side JavaScript to do so, users will experience a few critical seconds of blank page or loading spinner before seeing the content on the page. There are plenty of studies showing the drastic effect a slow site has on users, and thus revenue. <a href="http://www.strangeloopnetworks.com/resources/infographics/web-performance-and-ecommerce/amazon-100ms-faster-1-revenue-increase/" data-href="http://www.strangeloopnetworks.com/resources/infographics/web-performance-and-ecommerce/amazon-100ms-faster-1-revenue-increase/" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">Amazon claims</a> that each 100ms reduction in page load time raises revenue by 1%. Twitter spent a year and 40 engineers <a href="https://blog.twitter.com/2012/improving-performance-twittercom" data-href="https://blog.twitter.com/2012/improving-performance-twittercom" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">rebuilding their site to render on the server instead of the client</a>, claiming a 5x improvement in perceived loading time.</p>
           <h4 name="52cf" id="52cf" class="graf graf--h4 graf-after--p">Maintainability</h4>
           <p name="9e0e" id="9e0e" class="graf graf--p graf-after--h4">While the ideal case can lead to a nice, clean separation of concerns, inevitably some bits of application logic or view logic end up duplicated between client and server, often in different languages. Common examples are date and currency formatting, form validations, and routing logic. This makes maintenance a nightmare, especially for more complex apps.</p>
           <p name="e385" id="e385" class="graf graf--p graf-after--p">Some developers, myself included, feel bitten by this approach?—?it’s often only after having invested the time and effort to build a single-page app that it becomes clear what the drawbacks are.</p>
           <h3 name="1852" id="1852" class="graf graf--h3 graf-after--p">A Hybrid&nbsp;Approach</h3>
           <p name="49d4" id="49d4" class="graf graf--p graf-after--h3">At the end of the day, we really want a hybrid of the new and old approaches: we want to serve fully-formed HTML from the server for performance and SEO, but we want the speed and flexibility of client-side application logic.</p>
           <p name="2b57" id="2b57" class="graf graf--p graf-after--p">To this end, we’ve been experimenting at Airbnb with “Isomorphic JavaScript” apps, which are JavaScript applications that can run both on the client-side and the server-side.</p>
           <p name="b01e" id="b01e" class="graf graf--p graf-after--p">An isomorphic app might look like this, dubbed here “Client-server MVC”:</p>

           <p name="536e" id="536e" class="graf graf--p graf-after--figure">In this world, some of your application and view logic can be executed on both the server and the client. This opens up all sorts of doors?—?performance optimizations, better maintainability, SEO-by-default, and more stateful web apps.</p>
           <p name="23a1" id="23a1" class="graf graf--p graf-after--p">With <a href="http://nodejs.org/" data-href="http://nodejs.org/" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">Node.js</a>, a fast, stable server-side JavaScript runtime, we can now make this dream a reality. By creating the appropriate abstractions, we can write our application logic such that it runs on both the server and the client?—?the definition of isomorphic JavaScript.</p>
           <h4 name="e1be" id="e1be" class="graf graf--h4 graf-after--p">Isomorphic JavaScript in the&nbsp;Wild</h4>
           <p name="ccde" id="ccde" class="graf graf--p graf-after--h4">This idea isn’t new?—?Nodejitsu <a href="http://blog.nodejitsu.com/scaling-isomorphic-javascript-code" data-href="http://blog.nodejitsu.com/scaling-isomorphic-javascript-code" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">wrote a great description</a> of isomorphic JavaScript architecture in 2011?—?but it’s been slow to adopt. There have been a few isomorphic frameworks to spring up already.</p>
           <p name="9e1f" id="9e1f" class="graf graf--p graf-after--p">Mojito was the first open-source isomorphic framework to get any press. It’s an advanced, full-stack Node.js-based framework, but its dependence on YUI and Yahoo!-specific quirks haven’t led to much popularity in the JavaScript community since they <a href="http://developer.yahoo.com/blogs/ydn/yahoo-mojito-now-open-source-52490.html" data-href="http://developer.yahoo.com/blogs/ydn/yahoo-mojito-now-open-source-52490.html" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">open sourced it</a> in April 2012.</p>
           <p name="8c83" id="8c83" class="graf graf--p graf-after--p"><a href="http://www.meteor.com/" data-href="http://www.meteor.com/" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">Meteor</a> is probably the most well-known isomorphic project today. Meteor is built from the ground up to support real-time apps, and the team is building an entire ecosystem around its package manager and deployment tools. Like Mojito, it is a large, opinionated Node.js framework, however it’s done a much better job engaging the JavaScript community, and its much-anticipated 1.0 release is just around the corner. Meteor is a project to keep tabs on?—?it’s got an all-star team, and it’s raised $11.2 M from Andreessen Horowitz?—?unheard of for a company wholly focused on releasing an open-source product.</p>
           <p name="b317" id="b317" class="graf graf--p graf-after--p"><a href="https://asana.com/" data-href="https://asana.com/" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">Asana</a>, the task management app founded by Facebook cofounder Dustin Moskovitz, has an interesting isomorphic story. Not hurting for funding, considering Moskovitz’ status as youngest billionaire in the world, Asana spent years in R&amp;D developing their closed-source <a href="https://asana.com/luna" data-href="https://asana.com/luna" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">Luna</a> framework, one of the most advanced examples of isomorphic JavaScript around. Luna, originally built on <a href="https://code.google.com/p/v8cgi/" data-href="https://code.google.com/p/v8cgi/" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">v8cgi</a> in the days before Node.js existed, allows a complete copy of the app to run on the server for every single user session. It runs a separate server process for each user, executing the same JavaScript application code on the server that is running in the client, enabling a whole class of advanced optimizations, such as robust offline support and snappy real-time updates.</p>
           <p name="c2f9" id="c2f9" class="graf graf--p graf-after--p">We launched an isomorphic library of our own earlier this year. Called <a href="http://nerds.airbnb.com/weve-launched-our-first-nodejs-app-to-product/" data-href="http://nerds.airbnb.com/weve-launched-our-first-nodejs-app-to-product/" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">Rendr</a>, it allows you to build a Backbone.js + Handlebars.js single-page app that can also be fully rendered on the server-side. Rendr is a product of our experience rebuilding the <a href="https://m.airbnb.com/" data-href="https://m.airbnb.com/" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">Airbnb mobile web</a> app to drastically improve pageload times, which is especially important for users on high-latency mobile connections. Rendr strives to be a library rather than a framework, so it solves fewer of the problems for you compared to Mojito or Meteor, but it is easy to modify and extend.</p>
           <h3 name="e749" id="e749" class="graf graf--h3 graf-after--p">Abstraction, Abstraction, Abstraction</h3>
           <p name="ab52" id="ab52" class="graf graf--p graf-after--h3">That these projects tend to be large, full-stack web frameworks speaks to the difficulty of the problem. The client and server are very dissimilar environments, and so we must create a set of abstractions that decouple our application logic from the underlying implementations, so we can expose a single API to the application developer.</p>
           <h4 name="822a" id="822a" class="graf graf--h4 graf-after--p">Routing</h4>
           <p name="c685" id="c685" class="graf graf--p graf-after--h4">We want a single set of routes that map URI patterns to route handlers. Our route handlers need to be able to access HTTP headers, cookies, and URI information, and specify redirects without directly accessing window.location (browser) or req and res (Node.js).</p>
           <h4 name="2378" id="2378" class="graf graf--h4 graf-after--p">Fetching and persisting data</h4>
           <p name="9909" id="9909" class="graf graf--p graf-after--h4">We want to describe the resources needed to render a particular page or component independently from the fetching mechanism. The resource descriptor could be a simple URI pointing to a JSON endpoint, or for larger applications, it may be useful to encapsulate resources in models and collections and specify a model class and primary key, which at some point would get translated to a URI.</p>
           <h4 name="21f4" id="21f4" class="graf graf--h4 graf-after--p">View rendering</h4>
           <p name="cf51" id="cf51" class="graf graf--p graf-after--h4">Whether we choose to directly manipulate the DOM, stick with string-based HTML templating, or opt for a UI component library with a DOM abstraction, we need to be able to generate markup isomorphically. We should be able to render any view on either the server or the client, dependent on the needs of our application.</p>
           <h4 name="2d0a" id="2d0a" class="graf graf--h4 graf-after--p">Building and packaging</h4>
           <p name="cdb2" id="cdb2" class="graf graf--p graf-after--h4">It turns out writing isomorphic application code is only half the battle. Tools like <a href="http://gruntjs.com/" data-href="http://gruntjs.com/" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">Grunt</a> and <a href="http://browserify.org/" data-href="http://browserify.org/" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">Browserify</a> are essential parts of the workflow to actually get the app up and running. There can be a number of build steps: compiling templates, including client-side dependencies, applying transforms, minification, etc. The simple case is to combine all application code, views and templates into a single bundle, but for larger apps, this can result in hundreds of kilobytes to download. A more advanced approach is to create dynamic bundles and introduce asset lazy-loading, however this quickly gets complicated. Static-analysis tools like <a href="http://esprima.org/" data-href="http://esprima.org/" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">Esprima</a> can allow ambitious developers to attempt advanced optimization and <a href="http://en.wikipedia.org/wiki/Metaprogramming" data-href="http://en.wikipedia.org/wiki/Metaprogramming" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">metaprogramming</a> to reduce boilerplate code.</p>
           <h3 name="eba5" id="eba5" class="graf graf--h3 graf-after--p">Composing Together Small&nbsp;Modules</h3>
           <p name="628f" id="628f" class="graf graf--p graf-after--h3">Being first to market with an isomorphic framework means you have to solve all these problems at once. But this leads to large, unwieldy frameworks that are hard to adopt and integrate into an already-existing app. As more developers tackle this problem, we’ll see an explosion of small, reusable modules that can be integrated together to build isomorphic apps.</p>
           <p name="5bd6" id="5bd6" class="graf graf--p graf-after--p">It turns out that most JavaScript modules can already be used isomorphically with little to no modification. For example, popular libraries like Underscore, Backbone.js, Handlebars.js, Moment, and even jQuery can be used on the server.</p>
           <p name="4e37" id="4e37" class="graf graf--p graf-after--p">To demonstrate this point, I’ve created a sample app called <a href="https://github.com/spikebrehm/isomorphic-tutorial" data-href="https://github.com/spikebrehm/isomorphic-tutorial" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">isomorphic-tutorial</a> that you can check out on GitHub. By combining together a few modules, each that can be used isomorphically, it’s easy to create a simple isomorphic app in just a few hundred lines of code. It uses <a href="https://github.com/flatiron/director" data-href="https://github.com/flatiron/director" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">Director</a> for server- and browser-based routing, <a href="https://github.com/visionmedia/superagent" data-href="https://github.com/visionmedia/superagent" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">Superagent</a> for HTTP requests, and <a href="https://github.com/wycats/handlebars.js" data-href="https://github.com/wycats/handlebars.js" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">Handlebars.js</a> for templating, all built on top of a basic <a href="https://github.com/visionmedia/express" data-href="https://github.com/visionmedia/express" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">Express.js</a> app. Of course, as an app grows in complexity, one has to introduce more layers of abstraction, but my hope is that as more developers experiment with this, there will be new libraries and standards to emerge.</p>
           <h3 name="103a" id="103a" class="graf graf--h3 graf-after--p">The View From&nbsp;Here</h3>
           <p name="9cd6" id="9cd6" class="graf graf--p graf-after--h3">As more organizations get comfortable running Node.js in production, it’s inevitable that more and more web apps will begin to share code between their client and server code. It’s important to remember that isomorphic JavaScript is a spectrum?—?it can start with just sharing templates, progress to be an entire application’s view layer, all the way to the majority of the app’s business logic. Exactly what and how JavaScript code is shared between environments depends entirely on the application being built and its unique set of constraints.</p>
           <p name="37bb" id="37bb" class="graf graf--p graf-after--p">Nicholas C. Zakas h<a href="http://www.nczonline.net/blog/2013/10/07/node-js-and-the-new-web-front-end/" data-href="http://www.nczonline.net/blog/2013/10/07/node-js-and-the-new-web-front-end/" class="markup--anchor markup--p-anchor" rel="nofollow noopener" target="_blank">as a nice description</a> of how he envisions apps will begin to pull their UI layer down to the server from the client, enabling performance and maintainability optimizations. An app doesn’t have to rip out its backend and replace it with Node.js to use isomorphic JavaScript, essentially throwing out the baby with the bathwater. Instead, by creating sensible APIs and RESTful resources, the traditional backend can live alongside the Node.js layer.</p>
           <p name="b324" id="b324" class="graf graf--p graf-after--p">At Airbnb, we’ve already begun to retool our client-side build process to use Node.js-based tools like Grunt and Browserify. Our main Rails app may never be entirely supplanted by a Node.js app, but by embracing these tools it gets ever easier to share certain bits of JavaScript and templates between environments.</p>
           <p name="efa2" id="efa2" class="graf graf--p graf-after--p">You heard it here first?—?within a few years, it will be rare to see an advanced web app that isn’t running some JavaScript on the server.</p>
