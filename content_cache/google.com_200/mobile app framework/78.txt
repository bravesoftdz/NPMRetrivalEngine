      <p>I suppose if you’re reading this you know what a <a title="Should You Be Learning a HTML5 Mobile Application Framework?" href="http://www.joshmorony.com/should-you-be-learning-a-html5-mobile-application-framework/">HTML5 mobile application</a> is, but you might not know <strong>how to actually get one on the Apple app store or Google Play</strong>.</p>
      <script>AmpedSense.OptimizeAdSpot('PL'); AmpedSense.OptimizeAdSpot('PC'); AmpedSense.OptimizeAdSpot('PR');</script> 
      <p>It might feel a little bit <em>naughty</em> or wrong at first, but you can submit an application built purely with web technology to both of these stores just like you would a native application (even if you don’t have a Mac)… and despite what some native developers might tell you, <strong>they look, feel and perform just as well as native applications in most cases too</strong>.</p> 
      <p>There’s a few steps between here and there though. There’s a few conceptual differences to understand when creating a HTML5 mobile application, and you can easily get stuck somewhere along the way.</p> 
      <p>I’ve created this simple step-by-step guide to point you in the right direction no matter where you are along your app store journey – so let’s get right into it.</p> 
      <h2>1. Understand the difference between native, hybrid and web applications</h2> 
      <p>Before you even get started, you should understand <em>why</em> you’re building a mobile application with HTML5 and what the difference is exactly between <strong>web</strong> and <strong>hybrid</strong> (if you’re submitting your app to app stores, you will be building a hybrid application) and <strong>native</strong>.</p> 
      <p>If you’re a little short on time, here’s a little infographic I made that explains the conceptual difference in a very simplified way. Please note that there are <strong>a lot&nbsp;</strong>of factors this infographic leaves out, and a hybrid approach is not always the best solution:</p> 
      <p><a href="https://www.joshmorony.com/wp-content/uploads/2014/01/nativewebhybrid.png"><img class="aligncenter size-large wp-image-1155" src="https://www.joshmorony.com/wp-content/uploads/2014/01/nativewebhybrid-655x1024.png" alt="The Difference Between Native, Web and Hybrid Mobile Applications" width="640" height="1001" srcset="https://www.joshmorony.com/wp-content/uploads/2014/01/nativewebhybrid-655x1024.png 655w, https://www.joshmorony.com/wp-content/uploads/2014/01/nativewebhybrid-192x300.png 192w, https://www.joshmorony.com/wp-content/uploads/2014/01/nativewebhybrid-640x1001.png 640w, https://www.joshmorony.com/wp-content/uploads/2014/01/nativewebhybrid.png 1141w" sizes="(max-width: 640px) 100vw, 640px"></a></p> 
      <p>It’s easy enough to understand what a <strong>native application</strong> is: an application that was coded using the native language of the platform. A native application has immediate access to everything that platform has to offer with no restrictions whatsoever. The downside to this approach is that separate applications need to built for each platform.</p> 
      <p>A <strong>web application</strong> is an application that is powered by a browser through the Internet. A web application is typically <strong>built using HTML, CSS and JavaScript</strong> and can be served either through a desktop or mobile browser. A web application can be built to mimic the feel and behaviour of a normal native application, but instead just runs through a browser. Using this approach means you won’t have access to all the bells and whistles a native application does and you <strong>can’t distribute your application through most app stores</strong>.</p> 
      <p>A <strong>hybrid application</strong> is a mix of the native and web approach that allows you to build the application once and submit it to multiple app stores. A hybrid application uses a native packager like <a href="http://phonegap.com/">PhoneGap</a> which essentially wraps up a browser into a native application and displays your web application through it. With this approach, users of the application will no longer be able to see the browser the app is being run through (so it will just appear like a normal native application to them), you have access to native API’s through Phonegap and you can distribute your application through app stores.</p> 
      <p>The most obvious question that might arise from this is:</p> 
      <div class="su-quote su-quote-style-default">
       <div class="su-quote-inner su-clearfix">
        If Hybrid applications are so great, why isn’t everybody using them?
       </div>
      </div> 
      <p>To drill down into all <strong>the differences between a Hybrid and Native approach would take a blog post of its own</strong>. But in short, the main benefit of building native is that it&nbsp;offers&nbsp;the best speed and performance. In most cases the choice just comes down to personal preference though. Whilst native applications have the potential for much higher performance, in most cases it doesn’t make any noticeable difference.</p> 
      <p>You might need a high end gaming PC to play some games, but normal and applications and even most games will run just fine on a normal PC. Similarly, most applications will run just fine with a hybrid solution, but some applications may require the extra performance a native application can provide.</p> 
      <p>I have elaborated on <a title="What’s the Difference between Native, Hybrid and Web Mobile App Development?" href="http://www.joshmorony.com/whats-the-difference-between-native-hybrid-and-web-mobile-app-development/">the difference between web, hybrid and native</a> before if you want to check that out.</p> 
      <h2>2. Design your application</h2> 
      <p>Before committing to a particular approach, make sure you have your <strong>requirements defined</strong> and that your approach is going to meet those requirements.</p>
      <script>AmpedSense.OptimizeAdSpot('1L'); AmpedSense.OptimizeAdSpot('1C'); AmpedSense.OptimizeAdSpot('1R');</script> 
      <p>If you’re creating a <strong>pure web application </strong>(a HTML5 mobile application that does not require native APIs or to be submitted to app stores), then you only have access to features that are available to the browser. Make sure all of the functionality you require is available <a href="http://mobilehtml5.org/">here</a>. Don’t worry if you later decide to include native functionality, <strong>it’s extremely easy to convert your HTML5 mobile web app to a hybrid app</strong>.</p> 
      <p>If you want <strong>access to some native API’s</strong> or want to <strong>submit your application to app stores</strong>&nbsp;(or both) then a <strong>hybrid approach</strong> is going to be your best bet. We’ll talk more about integrating PhoneGap to access native API’s and wrap your application later, but make sure the functionality you require is available through <a href="http://docs.phonegap.com/en/4.0.0/cordova_plugins_pluginapis.md.html#Plugin%20APIs">PhoneGap by default</a>&nbsp;or through a <a href="http://docs.phonegap.com/en/4.0.0/cordova_plugins_pluginapis.md.html#Plugin%20APIs">PhoneGap plugin</a>.</p> 
      <p>When designing your application keep in mind the <a href="https://developer.apple.com/library/ios/documentation/UserExperience/Conceptual/MobileHIG/">iOS Human Interface Guidelines</a> and <a href="https://developer.android.com/design/index.html">Android Design guidelines</a>. Keep in mind that some of the information in these guidelines will be specific to native applications, so don’t worry too much about it. Just make sure you follow good design principles that are prevalent in most mobile applications.</p> 
      <p>If you’re building an application to be distributed through the Apple and Google Play app stores, <strong>you need to play by their rules</strong>. Make sure you comply with the <a href="https://developer.apple.com/app-store/review/guidelines/">Apple App Store Review Guidelines</a>&nbsp;and the <a href="https://play.google.com/about/developer-content-policy.html">Google Play Developer Program Policies</a>.</p> 
      <p>Apple especially can be a bit fickle with their application of the guidelines, but if you break any of them <strong>your application may be rejected</strong> and you will have to fix the problem and resubmit (and the review queue takes about 10 days to get through on average). Here’s a few “<em>gotchas</em>” to watch out for that might see your application be rejected:</p> 
      <ul> 
       <li>Your app is more like a web page than a mobile application</li> 
       <li>You don’t handle online / offline states</li> 
       <li>Your app is slow or unresponsive</li> 
      </ul> 
      <p>Apple is much more strict than Google in the review process, for more information check out my other blog post on the <a href="http://www.joshmorony.com/5-reasons-apple-might-reject-your-html5-mobile-app/">Apple review process</a>.</p> 
      <h2>3. Build your application using a HTML5 mobile framework</h2> 
      <p>HTML5 mobile frameworks are the corner stone of developing HTML5 mobile apps. They fulfill a role similar to that of <a href="https://jquery.com/">jQuery</a> in normal websites.</p> 
      <p>On a website jQuery allows you to easily create animations, show and hide things, and so on. When building a HTML5 mobile application, frameworks like <a title="Learning the Ionic Framework as a Sencha Touch Developer: Part 1" href="http://www.joshmorony.com/learning-the-ionic-framework-as-a-sencha-touch-developer-part-1/">Ionic</a> and <a href="https://www.sencha.com/products/touch/">Sencha Touch</a> allow you to <strong>easily create typical mobile elements</strong> like scrolling lists, buttons and screen transitions.</p> 
      <p>HTML5 mobile frameworks are not strictly just for creating the user interface though, most also have ways to <strong>store and manage data</strong> (data that would populate a scrolling list for example).</p> 
      <p>If you were to try and recreate the behaviour of a <strong>native smooth scrolling list with accerlation and deceleration</strong> in plain old JavaScript you would likely not achieve a good result without an enormous amount of effort. In <strong>Ionic</strong> though, you can just do something like this:</p> 
      <pre class="brush: jscript; title: ; notranslate" title="">&lt;ion-list&gt;

        Item 1
    &lt;/ion-item&gt;
    &lt;ion-item&gt;

    &lt;ion-item&gt;
        Item 20
    &lt;/ion-item&gt;
    &lt;ion-item&gt;
        Item 21

&lt;ion-list&gt;

      <p>See <a href="http://ionicframework.com/docs/api/directive/ionList/">this page in the Ionic documentation</a> for an interactive demonstration.</p> 
      <p>These frameworks implement advanced techniques like recycling DOM elements in the list that are off screen to improve performance, so I highly recommend that you use one. Getting the same level of performance as a native application is a difficult challenge, and it’s something these frameworks have done very well.</p> 
      <p>There are many frameworks out there to choose from, but the most popular today are probably:</p>

      <h2>4. Test your application through the browser</h2> 
      <p>The great thing about developing mobile applications with HTML5 is that <strong>you can test it right through your browser</strong>.</p> 
      <p>Different frameworks may be run in different ways but typically, you will be able to access your application through a local web server by going to:</p> 
      <pre>http://localhost/MyApp</pre> 
      <p>in your browser. If you’re using Chrome Dev Tools you can even emulate the device you are designing for by clicking the mobile icon in the top left:</p> 
      <p><a href="https://www.joshmorony.com/wp-content/uploads/2015/04/chrome-dev-tools.jpg"><img class="aligncenter wp-image-1168" src="https://www.joshmorony.com/wp-content/uploads/2015/04/chrome-dev-tools.jpg" alt="Chrome Dev Tools" width="600" height="368" srcset="https://www.joshmorony.com/wp-content/uploads/2015/04/chrome-dev-tools.jpg 911w, https://www.joshmorony.com/wp-content/uploads/2015/04/chrome-dev-tools-300x184.jpg 300w, https://www.joshmorony.com/wp-content/uploads/2015/04/chrome-dev-tools-640x392.jpg 640w" sizes="(max-width: 600px) 100vw, 600px"></a></p> 
      <p>From here you can debug your application to your hearts content using the Console, Network, Timeline etc.</p> 
      <p><strong>If you’re building a web application, then once you’ve finished this stage your app is all good and ready to go</strong>. But if you want to create a <strong>hybrid application</strong> that can access native API’s and be submitted to app stores, it’s now time to package the application.</p> 
      <h2>5. Package your application</h2> 
      <p>You can use a technology like <a href="http://phonegap.com/">PhoneGap</a> to wrap your web application in a native wrapper and act as a bridge between your application and the native API’s of the device. If you want to build iOS applications <strong>but don’t have a Mac</strong> you can use the <a href="http://build.phonegap.com/">PhoneGap Build</a> service.</p> 
      <p>I elaborate more on <a title="Freebie! Learn to Use PhoneGap Build with This eBook" href="http://www.joshmorony.com/freebie-learn-to-use-phonegap-build-with-this-ebook/">how PhoneGap works</a> in this post, but the simplified version of what happens is:</p> 
      <ol> 
       <li>PhoneGap creates a native application that contains a web view</li> 
       <li>All of the resources for your application are stored inside this native application</li> 
       <li>PhoneGap loads your HTML5 application into the web view (the web view can not be seen by the user)</li> 
       <li>The web view displays your application to the user</li> 
      </ol> 
      <p>So really, our application&nbsp;is still&nbsp;a web application since it’s being powered by the browser, but now we also have access to the device through PhoneGap.</p> 
      <div class="bonus-box">
       <strong>BONUS CONTENT:</strong>
       <a href="#ck_modal" rel="ck_modal">Grab my free PhoneGap Build eBook</a> to learn more about how to implement PhoneGap in your own applications.
      </div> 
      <h2>6. Access Native APIs (optional)</h2> 
      <p>You can just use PhoneGap to package your application so that it can be submitted to app stores, but why stop there? We have access to all this magical native functionality now!</p> 
      <p>With PhoneGap we can access just about everything that a native application can. Rather than interacting with the device directly, we use PhoneGap to pass messages between the device and our application.</p> 
      <p><a href="https://www.joshmorony.com/wp-content/uploads/2015/01/vibrate.png"><img class="aligncenter size-full wp-image-828" src="https://www.joshmorony.com/wp-content/uploads/2015/01/vibrate.png" alt="HTML5 app interacting with the device" width="500" height="200" srcset="https://www.joshmorony.com/wp-content/uploads/2015/01/vibrate.png 500w, https://www.joshmorony.com/wp-content/uploads/2015/01/vibrate-300x120.png 300w" sizes="(max-width: 500px) 100vw, 500px"></a></p> 
      <p>Typically, the native functionality will be available through some <strong>Javascript object</strong>. We will use the <a href="http://docs.phonegap.com/en/edge/cordova_camera_camera.md.html">Camera API</a> as an example. To trigger the camera all we need to do is run the following bit of JavaScript:</p>
      <script>AmpedSense.OptimizeAdSpot('3L'); AmpedSense.OptimizeAdSpot('3C'); AmpedSense.OptimizeAdSpot('3R');</script> 
      <pre class="brush: jscript; title: ; notranslate" title="">navigator.camera.getPicture(onSuccess, onFail, { quality: 50,
    destinationType: Camera.DestinationType.DATA_URL
});

function onSuccess(imageData) {
    var image = document.getElementById('myImage');
    image.src = "data:image/jpeg;base64," + imageData;
}

function onFail(message) {
    alert('Failed because: ' + message);
}
</pre> 
      <p>This tells PhoneGap to tell the device to launch the camera, and then we can grab the resulting image data from the <strong>onSuccess</strong> function. In this case the functionality is available through the&nbsp;<strong>camera&nbsp;</strong>object which can be found at&nbsp;<strong>navigator.camera</strong>. Sometimes though, like in the case of the <a title="Monetizing Your PhoneGap Build App with AdMob" href="http://www.joshmorony.com/monetizing-your-phonegap-build-app-with-admob/">AdMob plugin</a>, the functionality will be available directly through an object that is available globally like:</p> 
      <pre class="brush: jscript; title: ; notranslate" title="">
admob.someMethod();

</pre> 
      <p>If you want more information on how to include and use plugins in your PhoneGap project, take a look in my <a href="#ck_modal" rel="ck_modal">free PhoneGap Build Quickstart Guide</a>.</p> 
      <h2>7. Test your application on a device</h2> 
      <p>I mentioned before that one of the great things about HTML5 applications is that you can test them through your browser. Once you start integrating PhoneGap and native API’s though <strong>testing isn’t so great</strong>.</p> 
      <p><strong>To test an application that is accessing native API’s you will need to run the application on an actual device</strong>. But how do you debug without your browser debugging tools?</p> 
      <p>Fortunately, there are a few options for debugging directly on your device. I believe the best option though is <a title="Genuitec Release EPIC Debugger ‘GapDebug’ for PhoneGap Apps" href="http://www.joshmorony.com/genuitec-release-epic-debugger-gapdebug-for-phonegap-apps/">GapDebug</a>. GapDebug allows you to <strong>easily install .ipa files for iOS or .apk files for Android</strong> onto your device and provides a debugging interface that almost exactly replicates the Chrome Dev Tools. Changes that you make through the debugging tools will be reflected live onto the application on your device.</p> 
      <p>It’s an important step in any case to test on a device, whether you’re using native functionality or not the behaviour on an actual device can be different than when it is viewed through a desktop browser.</p> 
      <h2>8. Sign your application</h2> 
      <p><strong>To distribute your applications on app stores you will need to “sign” your application</strong>, and in the case of iOS you need to sign your application before you can even install it on a device.</p> 
      <p>For an iOS application you will need to <a title="How to Create an iOS Provisioning Profile and P12 with Windows" href="http://www.joshmorony.com/how-to-create-an-ios-provisioning-profile-and-p12-with-windows/">create a .p12 personal information file and a provisioning profile</a>. You can easily sign an iOS application if you have a Mac and XCode but it is also possible to sign an iOS application without a Mac (the article I linked above describes how to do that).</p> 
      <p>To sign an Android application, you simply need to <strong>create a single keystore file</strong>.</p> 
      <p>For more detailed information on how to sign iOS and Android applications using PhoneGap Build,&nbsp;again, take a look at my <a href="#ck_modal" rel="ck_modal">free PhoneGap Build eBook</a>.</p> 
      <h2>9. Distribute your application</h2> 
      <p>You’ll need to sign up for the <a href="https://developer.apple.com/programs/ios/">iOS Developer Program</a> (in fact, you’ll have to do this to create your provisioning profile and such anyway) and as a <a href="https://play.google.com/apps/publish/">Google Play Developer</a>.</p> 
      <p>You will need to prepare your app store listings for both platforms and provide some details about your application. To submit to Google Play you will simply be able to upload your&nbsp;<strong>.apk&nbsp;</strong>file through the developer console. To submit to the Apple App Store without a Mac though you will need to use the&nbsp;<strong>Application Loader&nbsp;</strong>program, <strong>which is only available on Macs</strong>. You can <a title="The Challenges of Developing iOS Applications on a PC" href="http://www.joshmorony.com/the-challenges-of-developing-ios-applications-on-a-pc/">use a service like Macincloud.com to get around this</a> though.</p> 
      <p>My <a href="#ck_modal" rel="ck_modal">PhoneGap Build eBook</a> goes through this process in a lot more detail. Thankfully, this is the last small hurdle you’ll need to get past! (and cross your fingers that Apple or Google don’t reject your application).</p> 
      <p>For a more complete walkthrough of creating an application from scratch and submitting it to app stores, take a look at <a href="https://joshmorony.com/mobile-development-for-web-developers/">Mobile Development for Web Developers</a>.</p>
