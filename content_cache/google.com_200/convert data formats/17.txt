              hit._highlightResult[key].value = hit._highlightResult[key].value.replace(/__ais-highlight__/g, '<em>').replace(/__\/ais-highlight__/g, '</em>');

              /* We do not deal with arrays. */

              hit._snippetResult[key].value = hit._snippetResult[key].value.replace(/__ais-highlight__/g, '<em>').replace(/__\/ais-highlight__/g, '</em>');

      <h1 class="p-name"> Localizing Times in&nbsp;JavaScript </h1> 
     </header> 
     <p class="author-byline"> <span class="author-and-date"> By <a href="https://css-tricks.com/author/chriscoyier/"> <span class="p-author"> Chris Coyier </span> </a> <span class="slashes">On</span> <time datetime="2015-08-12" class="dt-published"> August 12, 2015 </time> </span> </p> 
     <div class="article-content"> 
      <div class="module module-ad"> 
       <div id="bsap_1279518" class="bsarocks bsap_3469a2a501a9e18091036aa0c89f9dcb"></div> 
      </div> 
      <p>Say you have a <strong>time</strong> you'd like to display on your site. You've entered it in <em>your</em> time zone. You can always be specific and say something like: <strong>3:00 PM Eastern Standard Time</strong>. Then leave it up to whoever is reading to convert it to their local time. <a href="http://everytimezone.com/">Every Time Zone</a> is a great site for that.</p> 
      <p>But it can be nice to localize the time <em>for</em> the reader. Time zone conversion is notoriously confusing. And it's the type of thing computers are good at. JavaScript know what time zone the reader is in, after all.</p> 
      <p><span id="more-206436"></span></p> 
      <p>I'm sure there are a number of ways to do this. I had to do it recently on a new project, and I followed Dave Rupert's lead, who implemented it on <a href="http://shoptalkshow.com/">ShopTalk's website</a> to display the localized time for our next live show.</p> 
      <h3>Libraries</h3> 
      <p><a href="http://momentjs.com/">Moment.js</a> and <a href="http://momentjs.com/timezone/">Moment Timezone</a> are a pair that can get this done. And if you need to do any other heavier date and time manipulation or relative calculations (e.g. "32 minutes ago", "12 de Agosto de 2015"), you'll be in good hands with these libraries.</p> 
      <p>We'll also need to detect the time zone itself, and thankfully there is <a href="https://bitbucket.org/pellepim/jstimezonedetect">a library for that</a>.</p> 
      <p>So we'll be using:</p> 
      <ul> 
       <li>jstz.js</li> 
       <li>moment.js</li> 
       <li>moment-timezone.js</li> 
      </ul> 
      <p>Package or link those up however you will.</p> 
      <h3>Step 1: Get the Timezone</h3> 
      <p>Might as well stash it in localStorage after you've gotten it. That requires less effort on the browser's part to retrieve next time. </p> 
      <pre rel="JavaScript"><code class="language-javascript">if (!sessionStorage.getItem('timezone')) {
  var tz = jstz.determine() || 'UTC';
  sessionStorage.setItem('timezone', tz.name());
}
var currTz = sessionStorage.getItem('timezone');</code></pre> 
      <h3>Step 2: Get a Time Value</h3> 
      <p>Moment expects a date/time value (in <a href="https://en.wikipedia.org/wiki/Coordinated_Universal_Time">UTC</a>) like:</p> 
      <pre><code>2015-08-12T14:30Z</code></pre> 
      <p>If we're only interested in the time, we can create the date portion of that string from Moment itself of today's date (just makes the formatting way easier). The date actually kinda matters since UGHGKGJHGH daylight savings.</p> 
      <pre rel="JavaScript"><code class="language-javascript">var date = moment().format("YYYY-MM-DD");</code></pre> 
      <p>Then we'll create the final string, assuming we have a variable with the time we want:</p> 
      <pre rel="JavaScript"><code class="language-javascript">var stamp = date + "T" + theTime + "Z";</code></pre> 
      <p>And turn it into a Moment object:</p> 
      <pre rel="JavaScript"><code class="language-javascript">var momentTime = moment(stamp);</code></pre> 
      <h3>Step 3: Localize the time</h3> 
      <p>Now we can adjust/localize that time with Moment Timezone:</p> 
      <pre rel="JavaScript"><code class="language-javascript">var tzTime = momentTime.tz(currTz);</code></pre> 
      <p>And properly format it for display:</p> 
      <pre rel="JavaScript"><code class="language-javascript">var formattedTime = tzTime.format('h:mm A');</code></pre> 
      <h3>Step 4: Use it</h3> 
      <p>Now we can drop it on the page anywhere we want. If you output the time zone as well, you could even replace the time currently output on the page, since even if detecting the time zone fails it will fall back to UTC and output that as the time zone.</p> 
      <pre rel="JavaScript"><code class="language-javascript">output.textContent = "Time in " + currTz + ": " + formattedTime;</code></pre> 
      <h3>Adjusting Times</h3> 
      <p>Sometimes it makes sense (perhaps from a CMS perspective) to ask people to input time in a local time zone. But in order to do the conversion correctly, it makes more sense to have that time in UTC. </p> 
      <p>Moment itself can help with adjusting times, like:</p> 
      <pre rel="JavaScript"><code class="language-javascript">moment().subtract(4, 'hours');
moment().add(3, 'hours');</code></pre> 
      <p>Server side languages can help too. For instance PHP can do math on times:</p> 
      <pre rel="PHP"><code class="language-javascript">&lt;?php var $sixHoursAgo = strtotime("-6 hours", time()); ?&gt;</code></pre> 
      <p>It probably makes the most sense to do that with <a href="http://momentjs.com/timezone/">Moment Timezone</a>, because it's not always an exact formula (again UGHADKGHK daylight savings). </p> 
      <h3>Demo</h3> 
      <p>In this demo we use an HTML5 time input and adjust it on the fly to a localized time.</p> 
      <p data-height="248" data-theme-id="1" data-slug-hash="WvWeNa" data-default-tab="result" data-user="chriscoyier" class="codepen">See the Pen <a href="http://codepen.io/chriscoyier/pen/WvWeNa/">Adjusting Times to Local Time Zones</a> by Chris Coyier (<a href="http://codepen.io/chriscoyier">@chriscoyier</a>) on <a href="http://codepen.io">CodePen</a>.</p> 
      <h3>Other Ways</h3> 
      <p>Do you have any experience with this? How did you handle it? </p> 
      <p>The dependencies we used here are not insignificant! It appears there are native JavaScript methods (e.g. <a href="https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Date/getTimezoneOffset">getTimezoneOffset()</a>) that have pretty good support that could be used to similar effect. Any experience relying on those?</p>

            <p>I had to do similar things on a project related to the Olympics, where the users timezone, the timezone of the Olympic events and the timezones of the servers this site ran on all interacted in annoying ways. We found the simplest solution was to have dates/times in the server-generated HTML delivered in the timezone local to the servers and then re-format them on the client side in the timezone of the users device.</p> 
            <p>But what made this easier to manage was using data attributes to store a JS-compatible date string. So our servers in New York would deliver <code>&lt;span data-datetime="2015-08-18T18:30:00.000Z"&gt;Aug 18, 2:30pm EDT&lt;/span&gt;</code>, but a user in California would see <code>Aug 18, 11:30am</code>. The attr let us grab all the dates that needed formatting with a simple CSS query and the value was parsed without having to do any string manipulation.</p> 
            <p>We didnt have Moment.js back when we did this project, but that would have made life much easier. Having a browser-native <code>Intl.DateTimeFormat</code> would have been even better, and happily thats almost a reality now.</p>
