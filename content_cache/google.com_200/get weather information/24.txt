        <h1 class="page-title">Configuration Options</h1> 
        <p>The global configuration for your WeatherBlox setup manages things like link formatting, icon locations and mappings, and the units to display. The library provides all of the defaults necessary to get started without worrying about any of these settings, but you can override these defaults to provide additional customization.</p> 
        <p>Refer to the <a href="https://www.aerisweather.com/support/docs/toolkits/aeris-weatherblox/customization/configuration">configuration documentation</a> for more information on how to override these defaults for both the JavaScript and API methods.</p> 
        <p>Below are the available configuration options:</p> 
        <table class="table"> 
         <tbody> 
          <tr> 
           <td width="30%"><code>links</code></td> 
           <td>Defines the URL formatting to use for various types of links used throughout the views and layouts</td> 
          </tr> 
          <tr> 
           <td><code>links.loc</code></td> 
           <td>Defines how to format locations when used in a URL via the <code>{{loc}}</code> variable</td> 
          </tr> 
          <tr> 
           <td><code>links.local.main</code></td> 
           <td>URL to the main local weather layout</td> 
          </tr> 
          <tr> 
           <td><code>links.local.history</code></td> 
           <td>URL to the local daily and monthly observation layouts</td> 
          </tr> 
          <tr> 
           <td><code>links.local.advisory</code></td> 
           <td>URL to the local advisory layout</td> 
          </tr> 
          <tr> 
           <td><code>links.maps</code></td> 
           <td>URL to the main maps layout</td> 
          </tr> 
          <tr> 
           <td><code>icons.path</code></td> 
           <td>Base URL path to load weather icons from. The <a href="https://www.aerisweather.com/support/docs/api/reference/icon-list">icon names</a> provided by the Aeris Weather API will be appended to the end of this URL path along with the value defined by <code>icons.ext</code>.</td> 
          </tr> 
          <tr> 
           <td><code>icons.ext</code></td> 
           <td>The file extension to use for the weather icons from the source specified by <code>icon.path</code></td> 
          </tr> 
          <tr> 
           <td><code>icons.mapping</code></td> 
           <td>An object that maps a single weather icon name to a regular expression to use against the weather icon names provided by the Aeris Weather API. Using this mapping will allow you to provide a single icon name that can be used for multiple API-specific icon names, such as using "sunny" for "sunny", "clear" and "fair". This reduces the total number of icons you need to customize and also allows you to define your own icon naming convention. Refer to the <a href="#icon-mapping">Icon Mapping</a> section for more details.</td> 
          </tr> 
          <tr> 
           <td><code>unitsType</code></td> 
           <td>The default unit type to display, either <code>Units.Type.Imperial</code> or <code>Units.Type.Metric</code></td> 
          </tr> 
          <tr> 
           <td><code>units</code></td> 
           <td>The unit strings to display next to values for both imperial and metric values. These are specified per data type they represent.</td> 
          </tr> 
         </tbody> 
        </table> 
        <p>The following is the default configuration object for WeatherBlox. Note that only the <code>links</code> options are currently overridable using the API method:</p> 
        <pre>
<code class="language-javascript">{
	links: {
		loc: '{{place.state}}/{{place.name}}',
		local: {
			main: '/local/{{loc}}.html',
			history: {
				day: '/local/{{loc}}/history/{{year}}/{{month}}/{{date}}.html',
				month: '/local/{{loc}}/history/{{year}}/{{month}}.html'
			},
			advisory: '/local/{{loc}}/advisories.html'
		},
		maps: {
			main: '/maps/{{regionSlug}}/{{layers}}.html'
		}
	},
	icons: {
		path: '//cdn.aerisapi.com/wxblox/icons/',
		ext: 'png',
		mapping: {}
	},
	unitsType: Units.Type.Imperial,
	units: {
		imperial: {
			temp: '°F',
			speed: ' mph',
			distance: ' mi',
			height: ' ft',
			pressure: ' in',
			percent: '%',
			precip: '"',
			rain: '"',
			snow: '"',
			phrase: ''
		},
		metric: {
			temp: '°C',
			speed: ' kmh',
			distance: ' km',
			height: ' m',
			pressure: ' mb',
			percent: '%',
			precip: ' mm',
			rain: ' mm',
			snow: ' cm',
			phrase: 'met'
		}
	}
}</code></pre> 
        <p><a name="icon-mapping"></a></p> 
        <h3>Icon Mapping</h3> 
        <p>Using the <code>icons.mapping</code> configuration option, you can provide your own method for mapping your custom icon set to the icon names provided by the Aeris Weather API. The object should be key-value pairs where the value is a regular expression string used for matching against the API icon names. For instance, the Aeris Weather API will append an "n" to the end of all icon names for their night counterparts, but not all weather icons need a nighttime representation if there isn't a sun or moon displayed. Using this icon mapping option can eliminate the need to duplicate an icon for both day and night representations.</p> 
        <p>Below is the default mapping used by WeatherBlox views:</p> 
        <pre>
<code class="language-javascript">{
	sunny: '(sunny|clear|fair)',
	clearn: '(sunny|clear|fair)n',
	hazy: '(hazy|smoke|dust)',
	hazyn: '(hazy|smoke|dust)n',
	drizzle: 'drizzlef?',
	tstorms: 'tstorms?w?',
	fog: 'fogn?',
	wind: '(wind|pcloudywn?|mcloudywn?)',
	snowflurries: '(snow)?flurries',
	sleet: '(sleet|freezingrain)',
	rainandsnow: '(rain(and)?snow|rainsnow)',
	showers: 'showersw?n?',
	blizzard: 'snoww'
}</code></pre>

        <h1 class="page-title">Injecting Custom HTML</h1> 
        <p>You may need to inject your own custom HTML content inside of a WeatherBlox view or layout, such as inserting ads or other analytics content. This is easy if you're using the client-side integration, or if you're using the server-side integration with our PHP package that provides utility methods for inserting your content.</p> 
        <h3>Custom HTML Via JavaScript</h3> 
        <p>Inserting your custom HTML into a WeatherBlox view via client-side is simple by using JavaScript for prepending or appending your HTML content into an existing DOM element within a rendered view or layout. Note that you will need to insert your HTML within an event handler for the view's <code>render:done</code> event:</p> 
        <pre>
<code class="language-javascript">var $ = Aeris.wxblox.$;
var view = new Aeris.wxblox.layouts.local.Main(target);
view.on('render:done', function(e) {
    $('.awxb-view-obs .obs').append('&lt;div&gt;This is just an custom content block area.&lt;/div&gt;');
});
view.load({
    p: 'seattle,wa'
});</code></pre> 
        <p>The WeatherBlox library exposes its internal DOM selector utility via the Aeris.wxblox.$ property that functions similarly to jQuery for&nbsp;you to&nbsp;use to select&nbsp;and insert&nbsp;HTML content. Alternatively, you can use jQuery or your method of choice for accessing and manipulating the DOM.</p> 
        <h3>Custom HTML Via Server-Side</h3> 
        <p>The easiest way to insert your own HTML into WeatherBlox views&nbsp;via server-side is by using our <a href="https://www.aerisweather.com/support/docs/toolkits/aeris-weatherblox/getting-started/server-side">PHP package</a> and associated <code>View</code> class for requesting and rendering the view. This class offers methods that make it easy for you to insert your own custom content into the view's HTML before outputting it to your page.&nbsp;</p> 
        <p>Use the following methods on your <code>View</code> instance to insert your HTML into the desired location within the view:</p> 
        <table class="table"> 
         <tbody> 
          <tr> 
           <td width="30%"><code>find($selector, $context)</code></td> 
           <td>Searches the view's HTML for the element specified by the $selector string and within the optional <code>$context</code> selector string. If found, this method will return a <a href="http://php.net/manual/en/class.domnode.php" target="_blank">DOMNode</a> as it uses PHP's <a href="http://php.net/manual/en/class.domxpath.php" target="_blank">DOMXPath</a> internally for performing the queries and converts your selector strings to the proper XPath format.</td> 
          </tr> 
          <tr> 
           <td><code>prepend($selector, $html)</code></td> 
           <td>Inserts the provided <code>$html</code> content as the first child within the DOM element specified by <code>$selector</code> if it exists within the view's HTML</td> 
          </tr> 
          <tr> 
           <td><code>append($selector, $html)</code></td> 
           <td>Inserts the provided <code>$html</code> content as the last child within the DOM element specified by <code>$selector</code> if it exists within the view's HTML</td> 
          </tr> 
          <tr> 
           <td><code>insertBefore($selector, $html)</code></td> 
           <td>Inserts the provided <code>$html</code> content before the DOM element specified by <code>$selector</code> if it exists within the view's HTML</td> 
          </tr> 
          <tr> 
           <td><code>insertAfter($selector, $html)</code></td> 
           <td>Inserts the provided <code>$html</code> content after the DOM element specified by <code>$selector</code> if it exists within the view's HTML</td> 
          </tr> 
         </tbody> 
        </table> 
        <p>For instance, if you want to insert a custom block of HTML content in a local weather layout after the observations block but before the short-term forecast phrasing, you're code could be:</p> 
        <pre>
<code class="language-php">$view = new Aeris\WxBlox\View('layouts/local', 'seattle,wa');
$view-&gt;insertAfter('.awxb-view-obs', '&lt;div&gt;This is just an custom content block area.&lt;/div&gt;');
$content = $view-&gt;html();
...
echo $content;</code></pre> 
        <p>You can also insert custom HTML into an existing DOM container within the view using <code>prepend()</code> and <code>append()</code>. For example, you want to insert custom HTML into the Observation view before existing child nodes within the .obs .weather selector target:</p> 
        <pre>
<code class="language-php">$view = new Aeris\WxBlox\View('obs', 'seattle,wa');
$view-&gt;prepend('.awxb-view-obs .obs .weather', '&lt;div&gt;This is just an custom content block area.&lt;/div&gt;');
$content = $view-&gt;html();
...
echo $content;</code></pre> 
        <p>If you're not using PHP or our library for interacting with the WeatherBlox API, you can use regular expressions or a DOM helper library for your language of choice in order to manipulate a view's HTML before rendering it to the page.</p>

        <h1 class="page-title">Configuration</h1> 
        <p>You may need to customize the configuration of a WeatherBlox view beyond just the API data request parameters, such as overriding the default options that control the final output.&nbsp;Most WeatherBlox views&nbsp;have additional customization options that you can override via their constructor method when you create an instance of them. Refer to the&nbsp;documentation for each <a href="https://www.aerisweather.com/support/docs/toolkits/aeris-weatherblox/reference/views">view</a> or <a href="https://www.aerisweather.com/support/docs/toolkits/aeris-weatherblox/reference/layouts">layout</a> for additional information regarding the configuration options currently supported.</p> 
        <h3>Global Configuration</h3> 
        <p>The global configuration for your WeatherBlox setup manages things like link formatting, icon locations and mappings, and the units to display. The library provides all of the defaults necessary to get started without worrying about any of these settings, but you can override these defaults to provide additional customization.</p> 
        <p>Both JavaScript and API methods of using WeatherBlox support the ability to customize link formatting, which can appear throughout the rendered views and are used to link to different areas of your site. For example, the NearbyObservations view contains links for each location that, when clicked, will take the user to the weather page for that location. Or for the Maps layout, clicking a new map layer or region will take the user to the page for that content (if not using JavaScript event handlers for the links).</p> 
        <p>Below is the default configuration object, specifically related to the available link types taken from the <a href="https://www.aerisweather.com/support/docs/toolkits/aeris-weatherblox/reference/configuration-options">full configuration object</a>:</p> 
        <pre>
<code class="language-javascript">{
	links: {
		loc: '{{place.state}}/{{place.name}}',
		local: {
			main: '/local/{{loc}}.html',
			radar: '/local/{{loc}}/radar.html',
			history: {
				day: '/local/{{loc}}/history/{{year}}/{{month}}/{{date}}.html',
				month: '/local/{{loc}}/history/{{year}}/{{month}}.html'
			},
			forecast: {
				day: '/local/{{loc}}/forecast/{{year}}/{{month}}/{{date}}.html'
			},
			advisory: '/local/{{loc}}/advisories.html',
			calendar: '/local/{{loc}}/calendar/{{year}}/{{month}}.html',
			maps: '/local/{{loc}}/maps.html'
		},
		maps: {
			main: '/maps/{{regionSlug}}/{{layers}}.html',
			default: '/maps/us/radar.html'
		}
	}
}</code></pre> 
        <p>To override these values using WeatherBlox via JavaScript, you just need to use the Aeris.wxblox.config.set() method to set your desired value for the configuration's key path. For instance, you can use a callback function instead of a string to format locations as part of a URL, the links.loc configuration key:</p> 
        <pre>
<code class="language-javascript">Aeris.wxblox.config.set('links.loc', function(data) {
	let place = data.place;
	return (place.state) ? place.country+'/'+place.state+'/'+place.name : place.country+'/'+place.name;
});</code></pre> 
        <p>For the API method using our PHP package, the approach is similar except you would use the Config class in the package instead of the above JavaScript object:</p> 
        <pre>
<code class="language-php">Aeris\WxBlox\Config::getInstance()-&gt;set('links.loc', function($place) {
	if (!empty($place) &amp;&amp; isset($place['country'])) {
		if (strtolower($place['country']) == 'us') {
			return strtolower(implode('/', array($place['country'], $place['state'], $place['name'])));
		}
		return strtolower(implode('/', array($place['country'], $place['name'])));
	}
	return '';
});</code></pre> 
        <p>If you're using WeatherBlox via the API but not our PHP package for loading and rendering views, you would need to use your own method for performing the necessary link formatting and replacement before outputting your view to the page.</p> 
        <h3>Configuring Views - JavaScript</h3> 
        <p>To customize view options using the JavaScript (client-side) method, pass an optional object as the second parameter to a view's constructor method after the DOM target, which should contain the options you wish to override for the view.</p> 
        <p>For example, to override the default layout controls on a MapViewer instance, you would need to provide a custom value for the controls.layers configuration option:</p> 
        <pre>
<code class="language-javascript">var view = new Aeris.wxblox.views.MapViewer('#map', { 
	controls: { 
		layers: [{
			code: "radar", 
			title: "Radar"
		}] 
	}
});</code></pre> 
        <h4>Configuring Layouts</h4> 
        <p>Configuring a WeatherBlox view that's part of a layout container follows a similar approach as explained above except that you would need to assign your options to the key associated with the view you're targeting within the layout.</p> 
        <p>For example, in the <code>local.Main</code> layout, an instance of the same MapViewer view used in the examples above is displayed along with local weather information. This view is associated with the <code>maps</code> key within the layout. Therefore, you would just set your configuration options for this view on the layout's <code>maps</code> key instead:</p> 
        <pre>
<code class="language-javascript">var view = new Aeris.wxblox.layouts.local.Main('#local-view', { 
	maps: {
		controls: { 
			layers: [{
				code: "radar", 
				title: "Radar"
			}] 
		}
	}
});</code></pre> 
        <p>Refer to each layout's specific documentation for more information on accessing and targeting individual views contained within them.</p> 
        <h3>Configuring Views - API</h3> 
        <p>To customize view options using the API (server-side) method, provide an <code>opts</code> parameter with your request, which <strong>must</strong> be a JSON-encoded string that corresponds to the supported configuration options for the requested view.</p> 
        <p>For example, to override the default layout controls on the MapViewer view (maps/viewer), you would need to provide a custom value for the <code>controls.layers</code> configuration option. The resulting JSON object for this option would look similar to:</p> 
        <pre>
<code class="language-javascript">{ controls: { layers: [ {code: "radar", title: "Radar"} ] } }</code></pre> 
        <p>If you're using our PHP package and associated <code>View</code> class, just pass your configuration overrides as an array to an <code>opts</code> key when instantiating your instance:</p> 
        <pre>
<code class="language-php">$opts = array(
    'controls' =&gt; array(
        'layers' =&gt; 
            array('code' =&gt; 'radar', 'title' =&gt; 'Radar')
        )
    );
$view = new Aeris\WxBlox\View('maps/viewer', '98109', $opts);
</code></pre> 
        <p>When requesting WeatherBlox views directly from our servers, you must pass your JSON-encoded string to the <code>opts</code> URL parameter:</p> 
        <pre>
<code class="language-php">$opts = array(
    'controls' =&gt; array(
        'layers' =&gt; 
            array('code' =&gt; 'radar', 'title' =&gt; 'Radar')
        )
    );
$optsStr = json_encode($opts);
$url = ‘http://wxblox.aerisapi.com/{{API_KEY}}/{{SECRET_KEY}}/views/maps/viewer/98109?opts=$optsStr;
</code></pre> 
        <p>The above encoded URL that is requested&nbsp;would then be:</p> 
        <pre>
<code class="language-javascript">http://wxblox.aerisapi.com/{{API_KEY}}/{{SECRET_KEY}}/views/maps/viewer/98109?opts=%7B%22maps%22%3A%7B%22controls%22%3A%7B%22layers%22%3A%5B%7B%22code%22%3A%22radar%22%2C%22title%22%3A%22Radar%22%7D%5D%7D%7D%7D</code></pre> 
        <h4>Configuring Layouts</h4> 
        <p>Configuring a WeatherBlox view that's part of a layout container follows a similar approach as explained above except that you would need to assign your options to the key associated with the view you're targeting within the layout.</p> 
        <p>For example, in the <code>local.Main</code> (<em>layouts/local</em>) layout, an instance of the same MapViewer view used in the examples above is displayed along with local weather information. This view is associated with the <code>maps</code> key within the layout. Therefore, you would just set your configuration options for this view on the layout's <code>maps</code> key instead:</p> 
        <pre>
<code class="language-php">$opts = array(
    'maps' =&gt; array(
        'controls' =&gt; array(
            'layers' =&gt; 
                array('code' =&gt; 'radar', 'title' =&gt; 'Radar')
            )
        )
    );
$view = new Aeris\WxBlox\View('layouts/local', '98109', $opts);</code></pre> 
        <p>Refer to each layout's specific documentation for more information on accessing and targeting individual views contained within them.</p>

        <h1 class="page-title">Custom CSS</h1> 
        <p>All styling for WeatherBlox views and layouts is scoped under the .awxb-view or .awxb-component namespaces. To reduce conficts with CSS on your own pages where these views will be included, the specificity is increased by duplicating the namespace:&nbsp;.awxb-view.awxb-view and&nbsp;.awxb-component.awxb-component respectively. You'll need to replicate this same namespace duplication when overriding the default styles:</p> 
        <pre>
<code class="language-css">.awxb-component.awxb-component {
    font-size: 1.2rem;
}
.awxb-component.awxb-component . awxb-local .obs .wx .temps .temp {
    font-size: 3.5rem;
}</code></pre> 
        <p>We recommend using a CSS precompiler, such as SASS, which will make overriding our default CSS considerably easier especially when a lot of overrides or customization is required:</p> 
        <pre>
<code class="language-css">.awxb-component.awxb-component {
    font-size: 1.2rem;

    .awxb-local {
        .obs {
            .wx {
                .temps {
                    .temp {
                        font-size: 3.5rem;
                    }
                }
            }
        }
    }
}</code></pre>

        <h1 class="page-title">API Accesses &amp;&nbsp;Caching</h1> 
        <p>When adding WeatherBlox views to your website, they will make calls to the Aeris API and/or Aeris Maps Platform (AMP) to obtain the required weather information. Since WeatherBlox views&nbsp;utilize your Aeris account for requesting data, it's important to understand that WeatherBlox view requests will also count against your maximum daily access for our data and/or maps API and that the total accesses varies depending on the view requested.&nbsp;</p> 
        <p>The documentation for each weather view and layout will provide information on the number of data and/or map API&nbsp;accesses that will be used to request and render the necessary weather data. &nbsp;In general, each weather view uses&nbsp;1-3 data API calls to fetch and display the weather information, whereas layouts can use considerably more as they are comprised of multiple individual views.</p> 
        <p>While the two implementation methods for WeatherBlox, client-side and server-side, generally&nbsp;consume the same number of hits&nbsp;for weather data (except for AMP requests which varies), caching these requests can greatly reduce the total number of accesses required that count against your daily allowed maximum for your account.</p> 
        <h3>WeatherBlox JavaScript&nbsp;Data Usage (Client-Side)</h3> 
        <p>When integrating WeatherBlox via <a href="https://www.aerisweather.com/support/docs/toolkits/aeris-weatherblox/getting-started/client-side">JavaScript</a>, all data API requests are made from the web visitor's browser. &nbsp;Therefore, each time a page containing WeatherBlox views&nbsp;is requested by a user,&nbsp;it will count as additional API calls. The exception to this would be any browser-based caching that is enabled, which would then cache each API request based on those requests' cache headers.</p> 
        <p>For example, if you&nbsp;include&nbsp;a single&nbsp;<a href="https://www.aerisweather.com/support/docs/toolkits/aeris-weatherblox/reference/views">Advisories</a> view that&nbsp;will consume 1 weather API access per page view. So if this page receives&nbsp;10,000 views throughout the day, then there would be 10,000 API accesses against your account's limit.</p> 
        <p>Alternatively, if you&nbsp;include&nbsp;the <a href="https://www.aerisweather.com/support/docs/toolkits/aeris-weatherblox/reference/views">Observations</a> view, then 1 to 3 API accesses will be used&nbsp;per page view&nbsp;depending on options (e.g. having&nbsp;threats and advisories enabled). If this page receives 10,000 views per day, then up to 30,000 daily accesses to the Aeris API will be required from your account.</p> 
        <h3>WeatherBlox API&nbsp;Data Usage (Server-Side)</h3> 
        <p>When integrating WeatherBlox on a high-volume site,&nbsp;your account's total API accesses against your daily limit&nbsp;can quickly get consumed. In such cases, using the <a href="https://www.aerisweather.com/support/docs/toolkits/aeris-weatherblox/getting-started/server-side">WeatherBlox API</a> instead of the JavaScript method may be the preferred solution. While requiring additional development efforts, you have greater control over caching responses and views,&nbsp;potentially lowering total usage significantly.</p> 
        <p>For example, if your website caters to a single location&nbsp;and you have 1,000 views of a page containing the <a href="https://www.aerisweather.com/support/docs/toolkits/aeris-weatherblox/reference/views">Observations</a> view in one&nbsp;hour, the following compares how caching WeatherBlox API responses significantly lowers your accesses usage:</p> 
        <ul> 
         <li>By using JavaScript method, a total of 1,000 to&nbsp;3,000 data API accesses would be required&nbsp;during the 1-hour time frame.</li> 
         <li>By using the WeatherBlox API and caching the weather content for 10 minutes, resulting in fetching the data from the WeatherBlox API&nbsp;6 times per hour, only 12 to&nbsp;24 data API accesses would be required (1-3 data API requests for the Observation View + 1 WeatherBlox API request * 6 updates).</li> 
        </ul> 
        <p class="alert alert-info"><strong>Heads up!</strong> &nbsp;When using the WeatherBlox API method, each call to the WeatherBlox API will count as a single access to the Aeris API. This is in addition to the data API calls the weather data requests required by the view.</p> 
        <p>The result is that&nbsp;using the WeatherBlox API integration method can&nbsp;drastically lower the total accesses used by your account.</p> 
        <h3>WeatherBlox&nbsp;AMP Usage</h3> 
        <p>Many of the WeatherBlox views and layouts include weather maps utilizing the <a href="https://www.aerisweather.com/support/docs/aeris-maps">Aeris Mapping Platform (AMP)</a>. &nbsp;The total AMP map units consumed by these views will be highly dependent upon the configuration of the views, such as the size of the maps, total&nbsp;layers, and animation playback.&nbsp;Review the <a href="https://www.aerisweather.com/support/docs/aeris-maps/getting-started/map-units-accesses">map units documentation</a> for more information on calculating AMP usage.</p>

        <h1 class="page-title">API Usage</h1> 
        <p>The WeatherBlox <a href="../client-side/">JavaScript integration</a> may be the simplest, for some use cases you need additional control, which the WeatherBlox API (Server Side) provides. You may need to request one or more WeatherBlox views and insert them into your HTML before the page is rendered by the browser, or you may not want to rely on JavaScript for rendering large blox of content on your page for indexing, searching or SEO purposes. In these cases, you can use our server-side method where you request weather blox from our servers as HTML and then insert them into your pages before serving those pages to the browser in the response.</p> 
        <p>The benefit of using the server-side (API) method is that you can control caching of rendered WeatherBlox content on your backend, reducing the total number of accesses against your API account. However, the server-side method for using WeatherBlox is not as quick and easy as the client-side method as it requires additional development.</p> 
        <p>There are two different approaches to using the server-side method. You can use our <a href="#package">PHP package</a> for quicker integration without worrying about performing the necessary requests to our servers or configuring the view. Or, you can request the WeatherBlox HTML <a href="#direct">directly from our servers</a>.</p> 
        <p><a name="package"></a></p> 
        <h3>Using the PHP Package</h3> 
        <p>Using our PHP package for integrating WeatherBlox into your own pages offers the quickest and easiest server-side solution. Not only does it eliminate the need for you to perform the requests to our servers directly, but it also handles a lot of the necessary view configuration and rendering based on your global configuration before returning the final HTML to your script, such as replacing and formatting links.</p> 
        <p>Use the following steps to get started using the PHP package:</p> 
        <p><strong>Step 1:</strong> Include the package in your application either as a dependency via Composer or by downloading the files directly from our <a href="https://github.com/aerisweather/wxblox-php">Github repo</a>.</p> 
        <p><strong>Step 2:</strong> Import the library's <em>View.php</em> file into your script where you need to request WeatherBlox content:</p> 
        <pre>
<code class="language-php">require_once('./lib/Aeris/WxBlox/View.php');</code></pre> 
        <p><strong>Step 3:</strong> Include the client-side&nbsp;CSS and script files for WeatherBlox from our CDN:&nbsp;</p> 
        <pre>
<code class="language-javascript">&lt;link href="https://cdn.aerisapi.com/wxblox/aeris-wxblox.css" rel="stylesheet" /&gt;
&lt;script src="https://cdn.aerisapi.com/wxblox/aeris-wxblox.min.js"&gt;&lt;/script&gt;
&lt;script src="https://cdn.aerisapi.com/wxblox/aeris-wxblox-ui.min.js"&gt;&lt;/script&gt;</code></pre> 
        <p><strong>Step 4:</strong> Configure the library in your script with your API access keys and any other configuration options you wish to override, such as link formats:</p> 
        <pre>
<code class="language-php">Aeris\WxBlox\Config::getInstance()-&gt;setAccess(‘{{API_KEY}}’, '{{SECRET_KEY}}');</code></pre> 
        <p><strong>Step 5:</strong> Instantiate an instance of <code>View</code>, passing the desired view or layout type and any additional API request parameters or <a href="https://www.aerisweather.com/support/docs/toolkits/aeris-weatherblox/customization/configuration">configuration options</a>:</p> 
        <pre>
<code class="language-php">$opts = array('limit' =&gt; 5);
$view = new Aeris\WxBlox\View('forecasts/detailed', '98109', $opts);</code></pre> 
        <p><strong>Step 6:</strong> Request and return the view's HTML via the View's <code>html()</code> method:</p> 
        <pre>
<code class="language-php">$content = $view-&gt;html();</code></pre> 
        <p><strong>Step 7:</strong> Output the returned HTML where you want the view's content to appear within your own HTML:</p> 
        <pre>
<code class="language-javascript">&lt;div class="local-forecast"&gt;&lt;?php echo $content; ?&gt;&lt;/div&gt;</code></pre> 
        <p><a name="direct"></a></p> 
        <h3>Directly Requesting HTML</h3> 
        <p>An alternative to using our PHP package for requesting WeatherBlox content is to request them directly from our servers. While this reduces the dependency on our scripts and PHP, the downside is that view options and links will not be automatically applied and formatted from the server's response. You will need to perform any output customization, especially link formatting, on your own as needed before outputting the final WeatherBlox content to your page.</p> 
        <p>To use the direct-request method via a URL:</p> 
        <p><strong>Step 1:</strong>&nbsp;Include the client-side&nbsp;CSS and script files for WeatherBlox from our CDN:&nbsp;</p> 
        <pre>
<code class="language-javascript">&lt;link href="https://cdn.aerisapi.com/wxblox/aeris-wxblox.css" rel="stylesheet" /&gt;
&lt;script src="https://cdn.aerisapi.com/wxblox/aeris-wxblox.min.js"&gt;&lt;/script&gt;

Note that some views or layouts have additional URL requirements, so refer to the view and layout reference for more information per WeatherBlox type:</p>

        <h1 class="page-title">JavaScript&nbsp;Usage</h1> 
        <p>When using the client-side (JavaScript) method for integrating WeatherBlox views into your application, the views are rendered after your main page has loaded and all requests are performed asynchronously. This method is the easiest to get started with as it doesn't require any server-side development in order to output the weather views into your HTML. However, using the JavaScript method can also increase your API usage as you can't control the amount of <a href="../caching-api-usage/">caching</a> that is performed.</p> 
        <p>Once you have an active Aeris API account and access keys for your application, follow these steps to get started using the JavaScript method:</p> 
        <p><strong>Step 1:</strong> Include the library CSS and script files from our CDN:</p> 
        <pre>
<code class="language-javascript">&lt;link href="https://cdn.aerisapi.com/wxblox/aeris-wxblox.css" rel="stylesheet" /&gt;
&lt;script src="https://cdn.aerisapi.com/wxblox/aeris-wxblox.js"&gt;&lt;/script&gt;</code></pre> 
        <p><strong>Step 2:</strong> Set your access keys:</p> 
        <pre>
<code class="language-javascript">Aeris.wxblox.setAccess('CLIENT_KEY', 'CLIENT_SECRET');</code></pre> 
        <p><strong>Step 3:</strong> <a href="https://www.aerisweather.com/support/docs/toolkits/aeris-weatherblox/customization/configuration">Configure</a> your WeatherBlox views and/or layouts and set their destination DOM targets where they will be rendered:</p> 
        <pre>
<code class="language-javascript">var view = new Aeris.wxblox.views.NearbyObservations('#obs');</code></pre> 
        <p><strong>Step 4:</strong> Add any desired event handlers. Refer to the general view or layout documentation for more information regarding the supported events for each view:</p> 
        <pre>
<code class="language-javascript">view.on('render:after', function(e) {
     console.log('after render');
});
</code></pre> 
        <p><strong>Step 5:</strong> Load and/or render your view, providing any additional API request parameters to control the view's output:</p> 
        <pre>
<code class="language-javascript">view.load({
     p: 'austin,tx'
});</code></pre> 
        <h3>Customization</h3>

        <h1 class="page-title">Legacy Graphics to AMP Migration Guide</h1> 
        <p>All great things must come to an end, including the Aeris Legacy Graphics. Originally started in 2004, these graphics are being end of lifed on <strong>June 30th, 2017</strong>. Fret not, as our Aeris Maps Platform (AMP) is ready for you to use today and has been our primary graphics platform since Feburary 2016. The question now is, how do I go about transitioning from the Legacy Graphics to AMP? We understand you are busy so we have created this document to help ease the cutover.</p> 
        <h4>Overview</h4> 
        <ul> 
         <li><a href="#Features">Features</a></li> 
         <li><a href="#Accessing Your Options">Accessing Your Options</a></li> 
         <li><a href="#Legends">Legends</a></li> 
         <li><a href="#Adding Non-Layer Items">Adding Non-Layer Items</a></li> 
         <li><a href="#Layer Options">Layer Options</a></li> 
         <li><a href="#Examples">Examples</a></li> 
         <li><a href="#Comparison-Chart">Comparison Chart</a></li> 
         <li><a href="#Support">Support</a></li> 
         <li><a href="#Helpful Links">Helpful Links</a></li> 
        </ul> 
        <h4><a name="Features"></a>Features</h4> 
        <p>Why AMP? We wanted to introduce more flexibility and allow you to tailor our weather data in a way that would give your audience a unique experience. Below is a list of features you have with the Legacy Graphics and AMP, followed by a list of features you will gain by using AMP.</p> 
        <p><strong>Existing Features</strong></p> 
        <ul> 
         <li>Weather maps for your applications</li> 
         <li>Ability to title and brand your map with limited customization</li> 
         <li>Built-in Legends</li> 
         <li>Easy integration</li> 
         <li>Limited global coverage</li> 
         <li>Limited map size options</li> 
        </ul> 
        <p><strong>New Features Only Offered In AMP</strong></p> 
        <ul> 
         <li>Selection of base and overlay layers with built-in styling options</li> 
         <li>Customize your legend with the Legend Generator</li> 
         <li>Build the perfect map with our Map Builder</li> 
         <li>Access to forecast models (NDFD, GFS, NAM, HRRR, etc.)</li> 
         <li>Generate maps with custom width and height</li> 
         <li>Center your map on any global location</li> 
         <li>Use bounding box options to draw your maps boundaries</li> 
         <li>Integrate our layers into most any 3rd party mapping library</li> 
         <li>Expanded global coverage</li> 
         <li>Integrate your map into any web page or mobile application with our SDK's</li> 
         <li>Extensive and expanding online documentation</li> 
        </ul> 
        <h4><a name="Accessing Your Options"></a>Accessing Your Options</h4> 
        <p>The Legacy Graphics offered the handy Access Helper which provided a few options to help create your weather map. The options were fairly limited, allowing you to select a weather map, pre-defined region, and toggle the title bar, legend and point data. AMP offers a unique tool for creating maps called the Map Builder. With the Map Builder you have all these same options; however, they have been presented a little differently. You can access the Map Builder by logging into your Aeris account&nbsp;and locating the Map Builder tab. Below is a list of common features found in the Access Helper along with a brief summary of where to find similar features in AMP.</p> 
        <p><strong><i>Date Type &amp; Contour Data Set</i></strong></p> 
        <p>The purpose of <i>Contour Data Set</i> was to identify the category of weather maps (current, forecast, convective outlook, etc.). From there, you would have various options in the <i>Data Type</i> dropdown that directly correlated with the 'Contour Data Set' selection. With the Map Builder, we have all these options listed under the 'Layer Options' section. Click through the various categories to see what layer options you have per your subscription package. The&nbsp;<a href="https://www.aerisweather.com/support/docs/aeris-maps/reference/map-layers">Map Layers page</a>&nbsp;that provides additional details.</p> 
        <p><strong><i><a name="Region"></a>Region</i></strong></p> 
        <p>The Legacy Graphics gave you a number of regions to choose from; however, this did not provide much wiggle room if you wanted to change the map coverage area. You were locked into images that only fit a 4:3 ratio. The region selector is a thing of the past and now you can define your own region. With AMP we build your map based on the parameters you've passed, allowing&nbsp;you to generate a custom sized map, centered on any global location, at user defined zoom level. This can be accomplished in the Map Options section of the map builder. If you would prefer to generate your map with a bounding box instead, AMP supports this as well. Check out the <a href="https://www.aerisweather.com/support/docs/aeris-maps/map-access/static-maps">bounding box and center point options</a> in our online docs for more information.</p> 
        <p><strong><i>Point Data</i></strong></p> 
        <p>Here you had the ability to add text data for the applicable layers. Instead of toggling this option in the Map Builder, there are separate layers for each text layer we offer. Accessing the metric version of each text layer is done by adding the appropriate modifier to the layer name. For example, calling the <font color="firebrick">temperatures-text</font> layer will default to the imperial (F) unit of measurement. By adding <font color="firebrick">-metric</font> to the layer name you can access the metric (C) unit of measurement - full layer name, in this case, would be <font color="firebrick">temperatures-text-metric</font>.</p> 
        <p><strong><i>Size</i></strong></p> 
        <p>As previously mentioned in the <a href="#Region">Region</a> section above, you no longer have to choose from our very short list of size options. You tell us the size and AMP will take care of the rest.</p> 
        <p><strong><i>Legend and Title</i></strong></p> 
        <p>There are 3 options in this section: Legend Bar + Title Bar, Legend Bar only, and Title Bar only. In order to allow customization, AMP provides a few different options for adding these components to your map. Check out the <a href="#Adding Non-Layer Items">Adding Non-Layer Items</a> section below for more information.</p> 
        <h4><a name="Legends"></a>Legends</h4> 
        <p>Another great feature of AMP is the Legend Generator which can be accessed by logging into your Aeris account and finding the <i>Legend Generator</i> tab. The Legend Generator allows you to customize various components of your legend with the only limitation being the color ramp that is used. Each data set has an out-the-box legend that is ready for you to use in the Legend Generator if you do not need to customize your legend.</p> 
        <h4><a name="Adding Non-Layer Items"></a>Adding Non-Layer Items</h4> 
        <p>There are a number of items that you may want to include in or on your map that is not included in the map layers. Common items include:</p> 
        <ul> 
         <li>Legends</li> 
         <li>Map Title</li> 
         <li>Branding/Watermarks</li> 
         <li>Timestamp</li> 
        </ul> 
        <p>Here are 3 options we would recommend for adding these items to your map:</p> 
        <ol> 
         <li>HTML/CSS - Visit our <a href="https://www.aerisweather.com/support/docs/aeris-maps/branding-and-labels/branding-html">HTML/CSS Branding page</a> for details examples.</li> 
         <li>AMP Visualizer - Our developer have created a .js library for creating, displaying, and animation AMP imagery. Check out the <a href="https://www.aerisweather.com/support/docs/aeris-maps/branding-and-labels/branding-visualizer">AMP Visualizer docs</a> for more information.</li> 
         <li>Proxy script - Fetch the desired AMP image and add your own brand/title/legend on your servers before serving up the image on your application.</li> 
        </ol> 
        <h4><a name="Layer Options"></a>Layer Options</h4> 
        <p>In the <a href="https://www.aerisweather.com/support/docs/aeris-maps/reference/map-layers">Map Layers documentation</a>, there is a column titled, <i>modifiers</i>. Modifiers can do a number of different things depending on what layer you are requesting.</p> 
        <p>Looking at the Base Maps category, you'll notice <font color="firebrick">flat</font> and <font color="firebrick">terrain</font> both have a <font color="firebrick">-dk</font> modifiers. This means you could access <font color="firebrick">flat</font> as a base map OR <font color="firebrick">flat-dk</font> to render this base with a dark theme. In the Overlays category, all of these layers have the <font color="firebrick">-dk</font> theme and a few of them have the large theme option listed as <font color="firebrick">-lg</font>. You can also use any combination of these modifiers. For example, here are a few images with the <font color="firebrick">flat</font> base map and all combinations available with the <font color="firebrick">admin</font> layer:</p> 
        <p><font color="firebrick">admin</font></p> 
        <p><img alt="admin over flat" src="/img/docs/admin.png" style="margin:0"></p> 
        <p><font color="firebrick">admin-dk</font></p> 
        <p><img alt="admin-dk over flat" src="/img/docs/admin-dk.png" style="margin:0"></p> 
        <p><font color="firebrick">admin-lg</font></p> 
        <p><img alt="admin-lg over flat" src="/img/docs/admin-lg.png" style="margin:0"></p> 
        <p><font color="firebrick">admin-dk-lg</font></p> 
        <p><img alt="admin-dk-lg over flat" src="/img/docs/admin-dk-lg.png" style="margin:0"></p> 
        <p>Some layers also have data specific modifiers. As mentioned in the <i>Point Data</i> section above, the metric modifier can be accessed in order to display temperatures in Celsius. Another data specific modifier we offer allows you to render model specific data. For example, under the Forecasts category, you will see the <i>ftemperatures</i> layer. This layer has 4 model modifiers: HRRR, NAM, GFS, and NDFD. Add any of these modifiers to the layer name to your request to access the model specific data. Multiple modifiers cannot be used for model specific data. However, if you needed access all models you could make multiple layer requests with the appropriate modifiers.</p> 
        <h4><a name="Examples"></a>Examples</h4>

         <li>Log in to your account to register your application for an API access key. Each application requires its own unique access key. Check out the <a href="https://www.aerisweather.com/support/docs/api">Aeris API docs</a>&nbsp;for more info.*</li>

        <h3>Google Maps Configuration for the AerisDemo App</h3> 
        <p>The Aeris Maps library project currently supports only Google maps. To use the Google Maps API, you'll&nbsp;need to register for a an API key&nbsp;in the Google Maps API Console.&nbsp;Visit the&nbsp;<a href="https://developers.google.com/maps/documentation/android-api/?hl=en_US">Google Maps Android API page</a>&nbsp;and click the "GET A KEY" &nbsp;button for detailed instructions on getting your Google Maps key.&nbsp;</p> 
        <p>Once you have your Google Maps API account set up:</p> 
        <ol> 
         <li>In the Services page, verify that the “Google Maps Android API v2” is enabled.</li> 
         <li>In the left navigation bar, click API Access.</li> 
         <li>In the resulting page, click Create New Android Key.</li> 
         <li>In the resulting dialog, enter the SHA-1 fingerprint, then a semicolon, then The Demo project’s package name. For example: BB:0D:AC:74:D3:21:E1:43:67:71:9B:62:91:AF:A1:66:6E:44:5D:75;com.example.demoaerisproject</li> 
         <li>The Google API Console responds by displaying Key for Android apps (with certificates) followed by a forty-character API key, for example:&nbsp;AIzaSyBdVl-cTICSwYKrZ95SuvNw7dbMuDt1KG0</li> 
        </ol>

        <h1 class="page-title">Version 2.0&nbsp;Migration Guide</h1> 
        <p>Welcome to version 2.0&nbsp;of AerisWeather's Android Software Development Kit!</p> 
        <p>There are several architectural changes that you should be aware of before jumping into the upgrade&nbsp;from 1.x. The following guide will help you when migrating from a pre-2.0&nbsp;version of the SDK within your own projects.</p> 
        <h4>Summary</h4> 
        <ul> 
         <li><a href="#New Endpoints">New Endpoints</a></li> 
         <li><a href="#Tile Data">New Tile Layers</a></li> 
         <li><a href="#Point Data">New Point Data</a> Layers</li> 
         <li><a href="#Polygon Data">New Polygon</a> Layers</li> 
         <li>Support for the Google Play Services up to and including 10.0.1</li> 
         <li>Updated <a href="#Package Prefix">Package Prefix</a> from hamweather to aerisweather</li> 
         <li>New <a href="#Maven Repo">Maven Repository</a> for com.aerisweather</li> 
         <li>Better (more consistent) <a href="#File Names">File&nbsp;Naming</a></li> 
         <li>Various Bug Fixes and Performance Enhancements</li> 
        </ul> 
        <h4>Requirements</h4> 
        <p>The Aeris Android SDK officially supports Android 4.4 (KitKat) and up, Java SE 8 and&nbsp;Google Play Services through 10.0.1. For the most part, these will be backward compatible, but if your project requires supporting an earlier version, you should refer to the previous versions of the Aeris Android libraries&nbsp;which can be found via the Central Repository under "com.hamweather".</p> 
        <h4><a name="Package Prefix"></a>Package Prefix</h4> 
        <p>To bring the Android libs more in line with our current corporate branding, the package&nbsp;prefix "hamweather",&nbsp;has been replaced with "aerisweather". All of the references to hamweather in the code and comments have been updated accordingly.&nbsp;</p> 
        <p><i><small><font color="firebrick">To Do:<br> Once you update to the 2.x libraries, you will need to change any references of "hamweather" in your code to "aerisweather"</font></small></i></p> 
        <h4><a id="File Names" name="File Names"></a>File Names</h4> 
        <p>Along with the renaming of the package prefix, we have also updated the naming of the libraries themselves to provide a more consistent feel and convention.&nbsp;</p> 
        <p>The previously named <code>aeris-android-lib</code> is now <code>aeris-core-lib</code>.</p> 
        <p>The previously named <code>aeris-maps-library</code> is now <code>aeris-maps-lib</code>.</p> 
        <p>The previously named <code>aeris-android-lib.jar</code> is now renamed to <code>aeris-core-lib.jar</code></p> 
        <p><i><small><font color="firebrick">To Do:<br> Typically the jar file is not used directly, so the only change needed for here will be in your <a href="#Gradle Dep">build.gradle</a> where you will change the name as well as the version.</font></small></i></p> 
        <h4><a id="Maven Repo" name="Maven Repo"></a>Maven Repository</h4> 
        <p>A new repository has been created to provide access to the newly named aerisweather packages and libraries. The&nbsp;new repository can be found in the Central Repository under "com.aerisweather".&nbsp;</p> 
        <p>Official releases of the Aeris Android SDK libraries can be found in the Releases repository of the Central Repo. The following is all that is needed to ensure access to the new repository:</p> 
        <p>&nbsp;</p> 
        <pre>
<code class="language-java">buildscript {
&nbsp; &nbsp; 
    repositories {

&nbsp; &nbsp; &nbsp; &nbsp; mavenCentral()
&nbsp;       
&nbsp;       jcenter()

    }
}</code></pre> 
        <p>...</p> 
        <pre>
<code class="language-java">repositories {

&nbsp; &nbsp; mavenCentral()

&nbsp; &nbsp; jcenter()

}

</code>
</pre> 
        <h4><a id="Gradle Dep" name="Gradle Dep"></a>Gradle Dependency</h4> 
        <p>To reference the version 2.0.0&nbsp;Aeris Android libraries in your Gradle file, use:</p> 
        <pre>
<code class="language-java">
dependencies {
    compile ('com.aerisweather:aeris-maps-lib:2.0.0@aar') {

    compile 'com.google.android.gms:play-services-maps:10.0.1'

        <p><i><small><font color="firebrick">To Do:<br> Update the build.gradle file(s) with the above referenced dependencies.</font></small></i></p> 
        <h2>Core Library</h2> 
        <h4><a id="New Endpoints" name="New Endpoints"></a>New Endpoints</h4> 
        <ul> 
         <li>Convective Outlook -&nbsp;The&nbsp;convective/outlook&nbsp;endpoint provides convective outlook information based on the SPC convective outlooks.&nbsp; Coverage of this endpoint is for the US only.</li> 
         <li>Dought Monitor -&nbsp;The&nbsp;droughts/monitor&nbsp;endpoint provides access to the US drought monitor information from the National Drought Mitigation Center.&nbsp; Coverage of this endpoint is for the US only, and data is updated weekly.</li> 
         <li>Fires Outlook -&nbsp;The&nbsp;fires/outlook&nbsp;endpoint provides fire weather outlook information based on the SPC fire weather outlooks.&nbsp; Coverage of this endpoint is for the US only.</li> 
         <li>Indicies -&nbsp;Indices for health and outdoor activities as based on common weather observations and forecasts 
          <ul> 
           <li>Arthritis</li> 
           <li>Cold and Flu</li> 
           <li>Migraines</li> 
           <li>Sinus</li> 
           <li>Outdoors</li> 
           <li>Golf</li> 
           <li>Biking</li> 
           <li>Swimming</li> 
           <li>Campfires</li> 
           <li>Bees</li> 
          </ul> </li> 
         <li>Phrases Outlook</li> 
         <li>Storm Cells Summary (Beta) -&nbsp;The stormcells/summary endpoint provides an overall summary on current storm cells based on search / filter criteria.&nbsp; Example uses include obtaining the number of storms cells, maximum hail currently reported within storm cells,&nbsp; number of storm cells with a high probability of hail within a 50 mile radius.&nbsp; This endpoint can also provide polygons of the storm cell groups included in the summary. Coverage area includes the 50 US states, Puerto Rico and Guam</li> 
         <li>Storm Reports Summary -&nbsp;The storm reports data set provides access to all local storm reports (LSR) as transmitted by the National Weather Service (NWS). These reports are received from storm spotters, such as amateur radio operators, storm chasers, law enforcement, emergency management, or citizens about severe weather conditions in their area.</li> 
        </ul> 
        <h4>Endpoint Type - Spelling Error</h4> 
        <p>In the core library, we corrected a spelling error in the STORMCELLS endopint type.</p> 
        <p><code>STORMECELLS</code> has been corrected to <code>STORMCELLS</code></p> 
        <p><i><small><font color="firebrick">To Do:<br> If you reference this endpoint in your code, you will need to update to the correct spelling.</font></small></i></p> 
        <h2>Maps Library</h2> 
        <h4><a id="Tile Data" name="Tile Data"></a>Tile Data / AMP Layers</h4> 
        <p>Version 2 of the SDK includes support for our latest and greatest mapping service AMP (Aeris Mapping Platform).&nbsp;The previous group of Tiles or Overlays has been removed, since AMP provides all of those layers and more.</p> 
        <p>For more info on our new mapping service, check out <a href="https://www.aerisweather.com/develop/maps">AMP (Aeris Mapping Platform)</a>.</p> 
        <p>&nbsp;</p> 
        <p>For detailed information regarding how to use the AMP layers in your Android application, please refer to the <a href="https://www.aerisweather.com/support/docs/toolkits/aeris-android-sdk/getting-started/weather-maps">Maps section</a> of the Aeris Android SDK documentation.</p> 
        <h4><a id="Point Data" name="Point Data"></a>Point Data</h4> 
        <p><b>AerisPointData Class</b></p> 
        <ul> 
         <li>Added another getRequest method with an option to bypass and replace the&nbsp;default parameters.</li> 
         <li>For an example, see the onResume sample method below and note this line: 
          <ul> 
           <li>AerisRequest request = pointData.getRequest(Action.CLOSEST, params, null, true);</li> 
          </ul> </li> 
        </ul> 
        <p><b>New Layer - Records</b> (for a full explanation of the layer see the API endpoint docs for&nbsp;<a href="https://www.aerisweather.com/support/docs/api/reference/endpoints/records">Records</a>)</p> 
        <p>The records data set provides access to daily US records (RER) as transmitted by the National Weather Service (NWS).</p> 
        <ul> 
         <li>Default parameters: from=-7days, limit=500</li> 
        </ul> 
        <p>Examples</p>
