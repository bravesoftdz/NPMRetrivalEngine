              <br>
              <br>client.packagename = (function () {
              <br>&nbsp;&nbsp;&nbsp; "use strict";
              <br>
              <br>&nbsp; //all of your code stuff here for example
              <br>
              <br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; const exampleFunction1 = function() {
              <br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; //stuffff
              <br>&nbsp; &nbsp;&nbsp; &nbsp;&nbsp; };
              <br>
              <br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; const exampleFunction2 = function(avariable) {
              <br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; //thingys
              <br>&nbsp; &nbsp;&nbsp; &nbsp;&nbsp; };
              <br>
              <br>
              <br>&nbsp;&nbsp;&nbsp; return Object.freeze({
              <br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; exampleFunction1,
              <br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; exampleFunction2

              <br>}());
              <br>

               So two things
              <br>
              <br>1. instead of using the call function option just execute a script and call the function either as a JavaScript function or call it like I do with the run_function call that I showed in the last post
              <br>
              <br>2. While and for loops are a bad thing in JavaScript, why you ask because JavaScript is single threaded and they are blocking functions, which means while either are in effect nothing else is processing. So if you have an array you are looping through or a counter you are stepping through,nothing else is going on inside the client. Affliction notices are not coming in, attack notices, GMCP, etc... this is one of the biggest issues with Nexus vs Mudlet, MUSH etc... They can all have these nice big arrays and for loops and while loops and not impact the client. Now you might think oh hey it only takes a few milliseconds to process them so no biggie, I will tell you it gets to be a major issue in PvP when your for/while loops are firing over and over.&nbsp;

               Ok, I was wrong. You can still overload commands, just the syntax has changed.
              <br>
              <br>So if you set an alias for "look" you can attach the following as a script:
              <br>
              <br>
              <div>
               &nbsp; &nbsp; &nbsp;//Overloaded look script
              </div>
              <div>
               &nbsp; &nbsp; &nbsp;send_command("look", 1) //the 1 tells the system "not to expand" the command that's sent, which basically means, don't repeat it if it matched the alias.
              </div>&nbsp; &nbsp; &nbsp;display_notice("This is an example of a basic command that's been overloaded to do something else.")&nbsp;
              <br>
              <br>One oddity you may notice is that the display notice actually processes before the look command. That was something we noticed a lot of in MKO too, so many of the notices for things actually had a delay attached. Also only "look" is overloaded "l" will send look as normal.

              <br>
