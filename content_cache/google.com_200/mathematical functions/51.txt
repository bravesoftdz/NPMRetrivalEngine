           <p name="70f2" id="70f2" class="graf graf--p graf-after--figure">Before we can tackle what a pure function is, it’s probably a good idea to take a closer look at functions. There may be a different way to look at them that will make functional programming easier to understand.</p>
           <h4 name="c9b0" id="c9b0" class="graf graf--h4 graf-after--p">What is a Function?</h4>
           <p name="79ff" id="79ff" class="graf graf--p graf-after--h4">A <strong class="markup--strong markup--p-strong">function</strong> is a process which takes some input, called <strong class="markup--strong markup--p-strong">arguments</strong>, and produces some output called a <strong class="markup--strong markup--p-strong">return value</strong>. Functions may serve the following purposes:</p>
           <ul class="postList">
            <li name="2b73" id="2b73" class="graf graf--li graf-after--p"><strong class="markup--strong markup--li-strong">Mapping:</strong> Produce some output based on given inputs. A function <strong class="markup--strong markup--li-strong">maps</strong> input values to output values.</li>
            <li name="7efa" id="7efa" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">Procedures:</strong> A function may be called to perform a sequence of steps. The sequence is known as a procedure, and programming in this style is known as <strong class="markup--strong markup--li-strong">procedural programming</strong>.</li>
            <li name="e02d" id="e02d" class="graf graf--li graf-after--li"><strong class="markup--strong markup--li-strong">I/O:</strong> Some functions exist to communicate with other parts of the system, such as the screen, storage, system logs, or network.</li>
           </ul>
           <h4 name="a77b" id="a77b" class="graf graf--h4 graf-after--li">Mapping</h4>
           <p name="1f55" id="1f55" class="graf graf--p graf-after--h4">Pure functions are all about mapping. Functions map input arguments to return values, meaning that for each set of inputs, there exists an output. A function will take the inputs and return the corresponding output.</p>
           <p name="39eb" id="39eb" class="graf graf--p graf-after--p"><em class="markup--em markup--p-em">`Math.max()`</em> takes numbers as arguments and returns the largest number:</p>
           <pre name="8cba" id="8cba" class="graf graf--pre graf-after--p">Math.max(2, 8, 5); // 8</pre>
           <p name="d539" id="d539" class="graf graf--p graf-after--pre">In this example, 2, 8, &amp; 5 are <em class="markup--em markup--p-em">arguments</em>. They’re values passed into the function.</p>
           <p name="3b80" id="3b80" class="graf graf--p graf-after--p"><em class="markup--em markup--p-em">`Math.max()`</em> is a function that takes any number of arguments and returns the largest argument value. In this case, the largest number we passed in was 8, and that’s the number that got returned.</p>
           <p name="1888" id="1888" class="graf graf--p graf-after--p">Functions are really important in computing and math. They help us process data in useful ways. Good programmers give functions descriptive names so that when we see the code, we can see the function names and understand what the function does.</p>
           <p name="eace" id="eace" class="graf graf--p graf-after--p">Math has functions, too, and they work a lot like functions in JavaScript. You’ve probably seen functions in algebra. They look something like this:</p>
           <p name="2b48" id="2b48" class="graf graf--p graf-after--p"><em class="markup--em markup--p-em">f</em>(<em class="markup--em markup--p-em">x</em>) = 2<em class="markup--em markup--p-em">x</em></p>
           <p name="af1f" id="af1f" class="graf graf--p graf-after--p">Which means that we’re declaring a function called f and it takes an argument called x and multiplies x by 2.</p>
           <p name="4ba8" id="4ba8" class="graf graf--p graf-after--p">To use this function, we simply provide a value for x:</p>
           <p name="10c2" id="10c2" class="graf graf--p graf-after--p"><em class="markup--em markup--p-em">f</em>(2)</p>
           <p name="85c8" id="85c8" class="graf graf--p graf-after--p">In algebra, this means exactly the same thing as writing:</p>
           <p name="308c" id="308c" class="graf graf--p graf-after--p">4</p>
           <p name="a028" id="a028" class="graf graf--p graf-after--p">So any place you see <em class="markup--em markup--p-em">f</em>(2) you can substitute 4.</p>
           <p name="2027" id="2027" class="graf graf--p graf-after--p">Now let’s convert that function to JavaScript:</p>
           <pre name="a931" id="a931" class="graf graf--pre graf-after--p">const double = x =&gt; x * 2;</pre>
           <p name="dc0e" id="dc0e" class="graf graf--p graf-after--pre">You can examine the function’s output using <em class="markup--em markup--p-em">`console.log()`</em>:</p>
           <pre name="361f" id="361f" class="graf graf--pre graf-after--p">console.log( double(5) ); // 10</pre>
           <p name="06eb" id="06eb" class="graf graf--p graf-after--pre">Remember when I said that in math functions, you could replace <em class="markup--em markup--p-em">`f(2)`</em> with <em class="markup--em markup--p-em">`4`</em>? In this case, the JavaScript engine replaces <em class="markup--em markup--p-em">`double(5)`</em> with the answer, <em class="markup--em markup--p-em">`10`</em>.</p>
           <p name="487c" id="487c" class="graf graf--p graf-after--p">So, <em class="markup--em markup--p-em">`console.log( double(5) );`</em> is the same as <em class="markup--em markup--p-em">`console.log(10);`</em></p>
           <p name="79fb" id="79fb" class="graf graf--p graf-after--p">This is true because <em class="markup--em markup--p-em">`double()`</em> is a pure function, but if <em class="markup--em markup--p-em">`double()` </em>had side-effects, such as saving the value to disk or logging to the console, you couldn’t simply replace <em class="markup--em markup--p-em">`double(5)` </em>with 10 without changing the meaning.</p>
           <p name="0ab5" id="0ab5" class="graf graf--p graf-after--p">If you want referential transparency, you need to use pure functions.</p>
           <h4 name="8171" id="8171" class="graf graf--h4 graf-after--p">Pure Functions</h4>
           <p name="28d4" id="28d4" class="graf graf--p graf-after--h4">A <strong class="markup--strong markup--p-strong">pure function</strong> is a function which:</p>
           <ul class="postList">
            <li name="0244" id="0244" class="graf graf--li graf-after--p">Given the same input, will always return the same output.</li>
            <li name="92a4" id="92a4" class="graf graf--li graf-after--li">Produces no side effects.</li>
            <li name="4185" id="4185" class="graf graf--li graf-after--li">Relies on no external mutable state.</li>
           </ul>
           <blockquote name="8629" id="8629" class="graf graf--pullquote graf-after--li">
            A dead giveaway that a function is impure is if it makes sense to call it without using its return value. For pure functions, that’s a&nbsp;noop.
           </blockquote>
           <p name="2d7e" id="2d7e" class="graf graf--p graf-after--pullquote">I recommend that you favor pure functions. Meaning, if it is practical to implement a program requirement using pure functions, you should use them over other options. Pure functions take some input and return some output based on that input. They are the simplest reusable building blocks of code in a program. Perhaps the most important design principle in computer science is KISS (Keep It Simple, Stupid). I prefer Keep It Stupid Simple. Pure functions are stupid simple in the best possible way.</p>
           <p name="495d" id="495d" class="graf graf--p graf-after--p">Pure functions have many beneficial properties, and form the foundation of <strong class="markup--strong markup--p-strong">functional programming</strong>. Pure functions are completely independent of outside state, and as such, they are immune to entire classes of bugs that have to do with shared mutable state. Their independent nature also makes them great candidates for parallel processing across many CPUs, and across entire distributed computing clusters, which makes them essential for many types of scientific and resource-intensive computing tasks.</p>
           <p name="6ede" id="6ede" class="graf graf--p graf-after--p">Pure functions are also extremely independent?—?easy to move around, refactor, and reorganize in your code, making your programs more flexible and adaptable to future changes.</p>
           <h4 name="3d2e" id="3d2e" class="graf graf--h4 graf-after--p">The Trouble with Shared&nbsp;State</h4>
           <p name="5c44" id="5c44" class="graf graf--p graf-after--h4">Several years ago I was working on an app that allowed users to search a database for musical artists and load the artist’s music playlist into a web player. This was around the time Google Instant landed, which displays instant search results as you type your search query. AJAX-powered autocomplete was suddenly all the rage.</p>
           <p name="ffa7" id="ffa7" class="graf graf--p graf-after--p">The only problem was that users often type faster than an API autocomplete search response can be returned, which caused some strange bugs. It would trigger race conditions, where newer suggestions would be replaced by outdated suggestions.</p>
           <p name="cc5f" id="cc5f" class="graf graf--p graf-after--p">Why did that happen? Because each AJAX success handler was given access to directly update the suggestion list that was displayed to users. The slowest AJAX request would always win the user’s attention by blindly replacing results, even when those replaced results may have been newer.</p>
           <p name="99f5" id="99f5" class="graf graf--p graf-after--p">To fix the problem, I created a suggestion manager?—?a single source of truth to manage the state of the query suggestions. It was aware of a currently pending AJAX request, and when the user typed something new, the pending AJAX request would be canceled before a new request was issued, so only a single response handler at a time would ever be able to trigger a UI state update.</p>
           <p name="74b2" id="74b2" class="graf graf--p graf-after--p">Any sort of asynchronous operation or concurrency could cause similar race conditions. Race conditions happen if output is dependent on the sequence of uncontrollable events (such as network, device latency, user input, randomness, etc…). In fact, if you’re using shared state and that state is reliant on sequences which vary depending on indeterministic factors, for all intents and purposes, the output is impossible to predict, and that means it’s impossible to properly test or fully understand. As Martin Odersky (creator of Scala) puts it:</p>
           <blockquote name="c93b" id="c93b" class="graf graf--blockquote graf-after--p">
            <em class="markup--em markup--blockquote-em">non-determinism = parallel processing + mutable state</em>
           </blockquote>
           <p name="68bc" id="68bc" class="graf graf--p graf-after--blockquote">Program determinism is usually a desirable property in computing. Maybe you think you’re OK because JS runs in a single thread, and as such, is immune to parallel processing concerns, but as the AJAX example demonstrates, a single threaded JS engine does not imply that there is no concurrency. On the contrary, there are many sources of concurrency in JavaScript. API I/O, event listeners, web workers, iframes, and timeouts can all introduce indeterminism into your program. Combine that with shared state, and you’ve got a recipe for bugs.</p>
           <p name="2e3f" id="2e3f" class="graf graf--p graf-after--p">Pure functions can help you avoid those kinds of bugs.</p>
           <h4 name="6d14" id="6d14" class="graf graf--h4 graf-after--p">Given the Same Input, Always Return the Same&nbsp;Output</h4>
           <p name="4688" id="4688" class="graf graf--p graf-after--h4">With our <em class="markup--em markup--p-em">`double()`</em> function, you can replace the function call with the result, and the program will mean the same thing?—?<em class="markup--em markup--p-em">`double(5)`</em> will always mean the same thing as <em class="markup--em markup--p-em">`10`</em> in your program, regardless of context, no matter how many times you call it or when.</p>
           <p name="0ad7" id="0ad7" class="graf graf--p graf-after--p">But you can’t say the same thing about all functions. Some functions rely on information other than the arguments you pass in to produce results.</p>
           <p name="742e" id="742e" class="graf graf--p graf-after--p">Consider this example:</p>
           <pre name="62bf" id="62bf" class="graf graf--pre graf-after--p">Math.random(); // =&gt; 0.4011148700956255<br>Math.random(); // =&gt; 0.8533405303023756<br>Math.random(); // =&gt; 0.3550692005082965</pre>
           <p name="c41c" id="c41c" class="graf graf--p graf-after--pre">Even though we didn’t pass any arguments into any of the function calls, they all produced different output, meaning that <em class="markup--em markup--p-em">`Math.random()` </em>is <strong class="markup--strong markup--p-strong">not pure</strong>.</p>
           <p name="a2e4" id="a2e4" class="graf graf--p graf-after--p"><em class="markup--em markup--p-em">`Math.random()`</em> produces a new random number between 0 and 1 every time you run it, so clearly you couldn’t just replace it with 0.4011148700956255 without changing the meaning of the program.</p>
           <p name="a8b4" id="a8b4" class="graf graf--p graf-after--p">That would produce the same result every time. When we ask the computer for a random number, it usually means that we want a different result than we got the last time. What’s the point of a pair of dice with the same numbers printed on every side?</p>
           <p name="455f" id="455f" class="graf graf--p graf-after--p">Sometimes we have to ask the computer for the current time. We won’t go into the details of how the time functions work. For now, just copy this code:</p>
           <pre name="65a0" id="65a0" class="graf graf--pre graf-after--p">const time = () =&gt; new Date().toLocaleTimeString();</pre>
           <pre name="9c4c" id="9c4c" class="graf graf--pre graf-after--pre">time(); // =&gt; "5:15:45 PM"</pre>
           <p name="e9c0" id="e9c0" class="graf graf--p graf-after--pre">What would happen if you replaced the <em class="markup--em markup--p-em">`time()`</em> function call with the current time?</p>
           <p name="ddc2" id="ddc2" class="graf graf--p graf-after--p">It would always say it’s the same time: the time that the function call got replaced. In other words, it could only produce the correct output once per day, and only if you ran the program at the exact moment that the function got replaced.</p>
           <p name="240f" id="240f" class="graf graf--p graf-after--p">So clearly, <em class="markup--em markup--p-em">`time()`</em> isn’t like our <em class="markup--em markup--p-em">`double()`</em> function.</p>
           <p name="03d9" id="03d9" class="graf graf--p graf-after--p"><strong class="markup--strong markup--p-strong">A function is only pure if, given the same input, it will always produce the same output</strong>. You may remember this rule from algebra class: the same input values will always map to the same output value. However, many input values may map to the same output value. For example, the following function <strong class="markup--strong markup--p-strong">is pure</strong>:</p>
           <pre name="fe5b" id="fe5b" class="graf graf--pre graf-after--p">const highpass = (cutoff, value) =&gt; value &gt;= cutoff;</pre>
           <p name="3f57" id="3f57" class="graf graf--p graf-after--pre">The same input values will always map to the same output value:</p>
           <pre name="7c76" id="7c76" class="graf graf--pre graf-after--p">highpass(5, 5); // =&gt; true<br>highpass(5, 5); // =&gt; true<br>highpass(5, 5); // =&gt; true</pre>
           <p name="a728" id="a728" class="graf graf--p graf-after--pre">Many input values may map to the same output value:</p>
           <pre name="4e85" id="4e85" class="graf graf--pre graf-after--p">highpass(5, 123); // true<br>highpass(5, 6);   // true<br>highpass(5, 18);  // true</pre>
           <pre name="33da" id="33da" class="graf graf--pre graf-after--pre">highpass(5, 1);   // false<br>highpass(5, 3);   // false<br>highpass(5, 4);   // false</pre>
           <h3 name="4f73" id="4f73" class="graf graf--h3 graf-after--pre">Pure Functions Produce No Side&nbsp;Effects</h3>
           <p name="133f" id="133f" class="graf graf--p graf-after--h3">A pure function produces no side effects, which means that it can’t alter any external state.</p>
           <h3 name="1f41" id="1f41" class="graf graf--h3 graf-after--p">Immutability</h3>
           <p name="5c1c" id="5c1c" class="graf graf--p graf-after--h3">JavaScript’s object arguments are references, which means that if a function were to mutate a property on an object or array parameter, that would mutate state that is accessible outside the function. Pure functions must not mutate external state.</p>
           <p name="dbd3" id="dbd3" class="graf graf--p graf-after--p">Consider this mutating, <strong class="markup--strong markup--p-strong">impure</strong> <em class="markup--em markup--p-em">`addToCart()`</em> function:</p>

           <p name="0127" id="0127" class="graf graf--p graf-after--p">The problem with this is that we’ve just mutated some shared state. Other functions may be relying on that cart object state to be what it was before the function was called, and now that we’ve mutated that shared state, we have to worry about what impact it will have on the program logic if we change the order in which functions have been called. Refactoring the code could result in bugs popping up, which could screw up orders, and result in unhappy customers.</p>

This is more complex state than you’ll typically be dealing with. For most things, you can break it down into smaller chunks.</p>
           <p name="7148" id="7148" class="graf graf--p graf-after--p">For example, Redux lets you compose reducers rather than deal with the entire app state inside each reducer. The result is that you don’t have to create a deep clone of the entire app state every time you want to update just a small part of it. Instead, you can use non-destructive array methods, or<em class="markup--em markup--p-em"> `Object.assign()` </em>to update a small part of the app state.</p>
