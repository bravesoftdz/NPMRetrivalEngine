      <h1 class="post-title"><a href="https://blogs.windows.com/buildingapps/2013/07/17/whats-new-in-webview-in-windows-8-1/">What’s new in WebView in Windows 8.1</a></h1> 
      <div class="author">
        By 
       <a href="https://blogs.windows.com/buildingapps/author/windowsappsteam/" title="Posts by Windows Apps Team" class="author url fn" rel="author">Windows Apps Team</a> 
      </div> 
      <div id="addthis-2478" class="addthis_toolbox addthis_default_style expanded" addthis:url="https://blogs.windows.com/buildingapps/2013/07/17/whats-new-in-webview-in-windows-8-1/" addthis:title="What’s new in WebView in Windows 8.1"> 
       <a class="addthis_button_facebook"><i class="icon icon-facebook"></i> <span>Share</span></a> 
       <a class="addthis_button_twitter"><i class="icon icon-twitter"></i> <span>Tweet</span></a> 
       <a class="addthis_button_reddit"><i class="icon icon-reddit"></i> <span>Share</span></a> 
       <a class="addthis_button_linkedin"><i class="icon icon-linkedin"></i> <span>Share</span></a> 
       <a class="addthis_button_skype" href="#"><i class="icon icon-skype"></i> <span>Skype</span></a> 
      </div> 
     </header> 
     <p>Incorporating web code and content into apps is becoming more the norm for app development. With Windows 8.1 Preview, you have choices in how you do that. Developers might be looking to integrate an advertising SDK, collecting web analytics about who’s using their app, or leveraging mature online web apps and services. To support existing code and content from the web, we’ve made more investment in the WebView control for Windows 8.1. </p> 
     <p>You may remember that the XAML framework included a WebView control, which enables you host web content in your XAML app. After getting feedback from developers who used this control in their Windows 8 apps, we made some improvements for Windows 8.1. To take advantage of these changes, make sure your app manifest is <a href="http://msdn.microsoft.com/en-us/library/windows/apps/dn263114.aspx" target="_blank">updated to require Windows 8.1</a>. </p> 
     <h2>WebView support in Windows Store apps written in HTML and JavaScript</h2> 
     <p>With Windows 8.1, the WebView control is now available to use in HTML apps. Previously, apps written in HTML and JavaScript needed to use an iframe if they wanted to host web content that’s not part of the app. When you use the WebView control for site hosting you get better isolation, navigation, <a href="http://windows.microsoft.com/en-US/windows7/SmartScreen-Filter-frequently-asked-questions?woldogcb=0" target="_blank">smart screen</a>, and support for sites that can’t be hosted in an iframe. For more info see <a href="http://channel9.msdn.com/Events/Build/2013/3-179" target="_blank">the talk John Hazen gave at the //Build conference</a>. </p> 
     <p>In HTML apps, the WebView control is instantiated with the x-ms-webview element: </p> 
     <blockquote> 
      <pre class="csharpcode"><span class="kwrd">&lt;</span><span class="html">x-ms-webview</span> <span class="attr">id</span><span class="kwrd">="WebView"</span> <span class="attr">src</span>=”<span class="attr">http:</span>//<span class="attr">www</span>.<span class="attr">microsoft</span>.<span class="attr">com</span>” <span class="attr">style</span>=”<span class="attr">width:400px</span>; <span class="attr">height:400px</span>;”<span class="kwrd">&gt;</span>
<span class="kwrd">&lt;/</span><span class="html">x-ms-webview</span><span class="kwrd">&gt;</span>
</pre> 
     </blockquote> 
     <p>The same underlying control is used for both the HTML and XAML versions of the control, so they have the same functionality. The only difference is some of the API names and data types are customized to fit in with the semantics of each platform. </p> 
     <h2>WebView z-order layering changes</h2> 
     <p>In Windows 8, the WebView control was implemented as an overlay window over the top of the app’s immersive window. It didn’t participate nicely in the app’s layering, and would sit on top of everything else such as popups, combo boxes, and the app bar.      <h2>Browsing functionality</h2> 
     <p>One of the scenarios for using WebView in apps is to provide a lightweight browsing experience. For example, an RSS feed app might provide a browsing experience for links in the content. WebView now includes additional API surface to make that easier: </p> 
     <ul> 
      <li>Navigation history is now tracked by the control, and the API includes: 
       <ul> 
        <li><a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.goback.aspx" target="_blank">GoBack()</a> and <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.goforward.aspx" target="_blank">GoForward()</a> methods to move through the history </li> 
        <li><a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.cangoback.aspx" target="_blank">CanGoBack</a> and <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.cangoforward.aspx" target="_blank">CanGoForward</a> properties to indicate if the navigation history supports those operations</li> 
       </ul> </li> 
      <li><a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.refresh.aspx" target="_blank">Refresh()</a> method </li> 
      <li><a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.stop.aspx" target="_blank">Stop()</a> method</li> 
     </ul> 
     <p>The event lifecycle for navigation has been updated and extended. The main flow is now: </p> 
     <ul> 
      <li><a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.navigationstarting.aspx" target="_blank">NavigationStarting</a> – Indicates the WebView is starting a navigation due to an API call, user interaction, script etc. To enable filtering of the navigation, the eventArgs includes a cancel property which, when set to true, cancel the navigation. </li> 
      <li><a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.contentloading.aspx" target="_blank">ContentLoading</a> – Indicates the HTML content has been received and is being loaded into the control. </li> 
      <li><a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.domcontentloaded.aspx" target="_blank">DOMContentLoaded</a> – Indicates that the main page has finished loading. If you need to use <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.invokescriptasync.aspx" target="_blank">InvokeScriptAsync</a>, wait for this event. </li> 
      <li><a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.navigationcompleted.aspx" target="_blank">NavigationCompleted</a> – Indicates the navigation is complete, or that a failure has occurred and why.</li> 
     </ul> 
     <p>The exceptions to this flow are: </p> 
     <ul> 
      <li><a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.unviewablecontentidentified.aspx" target="_blank">UnviewableContentIdentified</a> – Is fired when a user navigates to content other than a webpage. The WebView control is only capable of displaying HTML content. It doesn’t support displaying standalone images, downloading files, viewing Office documents, etc. This event is fired so the app can decide how to handle the situation. <a href="http://msdn.microsoft.com/en-us/library/windows/apps/hh701471.aspx" target="_blank">Windows.System.Launcher.LaunchFileAsync()</a> can be called so that the shell determines the right app to handle the file. </li> 
      <li><a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.unsafecontentwarningdisplaying.aspx" target="_blank">UnsafeContentWarningDisplaying</a> – Is fired when the SmartScreen filter has detected that the page may be unsafe, and the SmartScreen shield is shown. SmartScreen detection is asynchronous, so can occur during and after navigation. When this event is fired, the page content can no longer be interacted with using <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.invokescriptasync.aspx" target="_blank">InvokeScriptAsync</a> etc.</li> 
     </ul> 
     <p>These events are demonstrated in the first scenario of the <a href="http://go.microsoft.com/fwlink/p/?linkid=238582" target="_blank">WebView sample</a>. </p> 
     <p>As webpages can contain iframes, and each iframe has its own navigation sequence, WebView now includes a set of parallel events for detecting the lifecycle of iframes within the page. These events are: </p> 
     <ul> 
      <li><a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.framenavigationstarting.aspx" target="_blank">FrameNavigationStarting</a> </li> 
      <li><a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.framecontentloading.aspx" target="_blank">FrameContentLoading</a> </li> 
      <li><a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.framedomcontentloaded.aspx" target="_blank">FrameDOMContentLoaded</a> </li> 
      <li><a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.framenavigationcompleted.aspx" target="_blank">FrameNavigationCompleted</a></li> 
     </ul> 
     <p>They follow the same pattern as the main page navigation events. </p> 
     <h2>Content sources</h2> 
     <p>In Windows 8, you could use WebView to load content from: </p> 
     <ul> 
      <li>The internet using the <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.navigate.aspx" target="_blank">Navigate()</a> method and <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.source.aspx" target="_blank">Source</a> property (src in HTML) </li> 
      <li>Content from the app package, using “ms-appx-web:///directory/file” protocol as the Uri with the above APIs </li> 
      <li>The app can programmatically supply HTML markup using the <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.navigatetostring.aspx" target="_blank">NavigateToString()</a> method</li> 
     </ul> 
     <p>To improve scenarios where WebView is used to host embedded web content, we have added two content sources – Local State directories and the StreamUriResolver. </p> 
     <h3>Local state directories</h3> 
     <p>Sometimes you might want to create and/or store content locally, but then also load the content into WebView. For example, if you have an HTML game with downloadable levels, you can store the levels in the local state directories so they are available offline. </p> 
     <p>WebView can now load content from the local and temporary app state folders. These are the same folders that are discoverable using the <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.storage.applicationdata.localfolder.aspx" target="_blank">LocalFolder</a> and <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.storage.applicationdata.temporaryfolder.aspx" target="_blank">TemporaryFolder</a> APIs, and can be used for storing app content. The app can create directories and files under these folders, and then navigate to them using the ms-appdata protocol. </p> 
     <p>To enable an app to have multiple islands of content that are isolated from each other, you are required to create a sub directory under the local or temporary folder’s and store the content within that directory. The Uri scheme is: </p> 
     <ul> 
      <li>“ms-appdata:///local/TopLevelDirectory/file” for files from the local state and </li> 
      <li>“ms-appdata:///temp/TopLevelDirectory/file” for files from the app’s temporary state folder</li> 
     </ul> 
     <p>Each of the top level directories under the local or temp folder is isolated from each other, and browsing to a folder is not enabled unless explicitly navigated to using the navigate API. For example, an eBook app might have a set of different books that it wants to store locally. It can create a folder per book, and then navigate to a book. If each book has its own folder under the LocalFolder, each book would be isolated from each other and cannot access content from other book’s folders. </p> 
     <p>The following code will create a “NavigateToState” folder under the local state folder and then copy a file from the app package to it. </p> 
     <p><strong>C#</strong> </p> 
     <blockquote> 
      <pre class="csharpcode">async <span class="kwrd">void</span> createHtmlFileInLocalState()
{
    StorageFolder stateFolder = await ApplicationData.Current.LocalFolder.CreateFolderAsync(<span class="str">"NavigateToState"</span>, CreationCollisionOption.OpenIfExists);
    StorageFile htmlFile = await Windows.ApplicationModel.Package.Current.InstalledLocation.GetFileAsync(<span class="str">"htmlhtml_example2.html"</span>);
    await htmlFile.CopyAsync(stateFolder, <span class="str">"test.html"</span>, NameCollisionOption.ReplaceExisting);
    loadFromLocalState.IsEnabled = <span class="kwrd">true</span>;
}
</pre> 
     </blockquote> 
     <p><strong>JavaScript</strong> </p> 
     <blockquote> 
      <pre class="csharpcode"><span class="kwrd">function</span> createFileInState()
{
    Windows.Storage.ApplicationData.current.localFolder.createFolderAsync(<span class="str">"NavigateToState"</span>, Windows.Storage.CreationCollisionOption.openIfExists).then(<span class="kwrd">function</span> (stateFolder) {
        Windows.ApplicationModel.Package.current.installedLocation.getFileAsync(<span class="str">"htmlsimple_example.html"</span>).then(<span class="kwrd">function</span> (htmlFile) {
            <span class="kwrd">return</span> htmlFile.copyAsync(stateFolder, <span class="str">"simple_example.html"</span>, Windows.Storage.CreationCollisionOption.failIfExists);
        });
    });
}
</pre> 
     </blockquote>

     <h3>Custom URI resolving</h3> 
     <p>The <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.navigatetostring.aspx" target="_blank">NavigateToString</a> method provides an easy way to navigate to a single HTML document. However it’s common for webpages to reference other content such as css, scripts, images, and fonts that aren’t handled in this way. To solve this problem in Windows 8.1, we’ve added a new capability to WebView to use an <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.web.iuritostreamresolver.aspx" target="_blank">UriToStreamResolver</a> object to provide custom URI resolution.</p> 
     <p>This enables the host app to intercept requests under a specific URI pattern and supply the content as a stream. It can supply the content programmatically for the resources for a webpage, or series of pages, without needing to persist the content to the file system first. A scenario where this may be useful is to present encrypted content – the content can be persisted on disk as encrypted files, or package such as a cab, and then when content is requested it can be decrypted on the fly and supplied to the WebView. </p> 
     <p>To use this capability, you create an UriToStreamResolver object, and pass it to WebView using the <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.navigatetolocalstreamuri.aspx" target="_blank">NavigateToLocalStreamUri()</a> method. The UriToStreamResolver object is a WinRT object that needs to implement the IUriToStreamResolver interface, which has one method: </p> 
     <blockquote> 
      <pre class="csharpcode">IAsyncOperation&lt;IInputStream&gt; UriToStreamAsync(Uri uri)</pre> 
     </blockquote> 
     <p>&nbsp; </p> 
     <p>This method takes the URI and returns the stream. The URI is in the form of “ms-local-stream://appname_KEY/folder/file”. </p> 
     <p>For example, the following code shows a resolver that serves files from the app package. </p> 
     <blockquote> 
      <pre class="csharpcode"><span class="rem">/// &lt;summary&gt;</span>
<span class="rem">/// Sample URI resolver object for use with NavigateToLocalStreamUri</span>
<span class="rem">/// This sample uses the local storage of the package as an example of how to write a resolver.</span>
<span class="rem">/// &lt;/summary&gt;</span>
<span class="kwrd">public</span> <span class="kwrd">sealed</span> <span class="kwrd">class</span> StreamUriWinRTResolver : IUriToStreamResolver
{
    <span class="rem">/// &lt;summary&gt;</span>
    <span class="rem">/// The entry point for resolving a Uri to a stream.</span>
    <span class="rem">/// &lt;/summary&gt;</span>
    <span class="rem">/// &lt;param name="uri"&gt;&lt;/param&gt;</span>
    <span class="rem">/// &lt;returns&gt;&lt;/returns&gt;</span>
    <span class="kwrd">public</span> IAsyncOperation&lt;IInputStream&gt; UriToStreamAsync(Uri uri)
    {
        <span class="kwrd">if</span> (uri == <span class="kwrd">null</span>)
        {
            <span class="kwrd">throw</span> <span class="kwrd">new</span> Exception();
        }
        <span class="kwrd">string</span> path = uri.AbsolutePath;
        <span class="rem">// Because of the signature of this method, it can't use await, so we </span>
        <span class="rem">// call into a separate helper method that can use the C# await pattern.</span>
        <span class="kwrd">return</span> getContent(path).AsAsyncOperation();
    }
    <span class="rem">/// &lt;summary&gt;</span>
    <span class="rem">/// Helper that maps the path to package content and resolves the Uri</span>
    <span class="rem">/// Uses the C# await pattern to coordinate async operations</span>
    <span class="rem">/// &lt;/summary&gt;</span>
    <span class="kwrd">private</span> async Task&lt;IInputStream&gt; getContent(<span class="kwrd">string</span> path)
    {
        <span class="rem">// We use a package folder as the source, but the same principle should apply</span>
        <span class="rem">// when supplying content from other locations</span>
        <span class="kwrd">try</span>
        {
            Uri localUri= <span class="kwrd">new</span> Uri(<span class="str">"ms-appx:///html"</span> + path);
            StorageFile f = await StorageFile.GetFileFromApplicationUriAsync(localUri);
            IRandomAccessStream stream = await f.OpenAsync(FileAccessMode.Read);
            <span class="kwrd">return</span> stream.GetInputStreamAt(0);
        }
        <span class="kwrd">catch</span> (Exception) { <span class="kwrd">throw</span> <span class="kwrd">new</span> Exception(<span class="str">"Invalid path"</span>); }
    }
}
</pre> 
     </blockquote> 
     <p>&nbsp; </p> 
     <p>Multiple resolvers can be created, and handed to WebView, so to map the resolver instance to the URI, it uses a key as part of the hostname field, which is why the uri format is “ms-local-stream://appname_<b>KEY</b>/folder/file”. The best way to create this URI is to use the <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.buildlocalstreamuri.aspx" target="_blank">BuildLocalStreamUri()</a> method passing a key string and a relative URI for the first page. When you call NavigateToStreamURI, you pass the URI and the resolver object, and that instance of the resolver will be associated with the key of the URI. </p> 
     <p>The resolver will be remembered for the lifetime of the WebView instance. Until the instance of WebView is destroyed, or the key is associated with a new resolver object, it continues to be used to resolve URIs with that key. </p> 
     <p>In JavaScript apps, it is not possible to code an UriResolver object in JavaScript. However, you can use a resolver that is written in C++, C# or VB. For more details, see Scenario 4 in the <a href="http://go.microsoft.com/fwlink/p/?linkid=238582" target="_blank">WebView SDK Sample</a>. </p> 
     <h2>Capturing a snapshot of the WebView as a bitmap</h2> 
     <p>One of the common requests we’ve had for WebView is the ability to capture a snapshot of the WebView as a Bitmap, which can then be manipulated as required by the app. This can now be done using the <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.capturepreviewtostreamasync.aspx" target="_blank">CapturePreviewToStreamAsync()</a> or capturePreviewToBlobAsync() methods depending on the platform. </p> 
     <p>For example, the following code creates a file in the local state folder and save the bitmap of the WebView into the file. </p> 
     <p><strong>C#</strong> </p> 
     <blockquote> 
      <pre class="csharpcode"><span class="kwrd">private</span> async <span class="kwrd">void</span> Btn_Click(<span class="kwrd">object</span> sender, RoutedEventArgs e)
{
    StorageFile f = await ApplicationData.Current.LocalFolder.CreateFileAsync(<span class="str">"test.png"</span>, CreationCollisionOption.ReplaceExisting);
    IRandomAccessStream output = await f.OpenAsync(FileAccessMode.ReadWrite);
    await WebView1.CapturePreviewToStreamAsync(output);
    await output.FlushAsync();
    output.Dispose();
}
</pre> 
     </blockquote> 
     <p><strong>JavaScript</strong> </p> 
     <blockquote> 
      <pre class="csharpcode"><span class="kwrd">function</span> captureBitmap() {
    <span class="kwrd">var</span> webviewControl = document.getElementById(<span class="str">"webview"</span>);
    Windows.Storage.ApplicationData.current.localFolder.createFileAsync(<span class="str">"test.png"</span>, Windows.Storage.CreationCollisionOption.replaceExisting).then(<span class="kwrd">function</span> (file) {
        file.openAsync(Windows.Storage.FileAccessMode.readWrite).then(<span class="kwrd">function</span> (stream) {
            <span class="kwrd">var</span> captureOperation = webviewControl.capturePreviewToBlobAsync();
            captureOperation.oncomplete = <span class="kwrd">function</span> (completeEvent) {
                <span class="kwrd">var</span> inputStream = completeEvent.target.result.msDetachStream();
                Windows.Storage.Streams.RandomAccessStream.copyAsync(inputStream, stream).then(<span class="kwrd">function</span> () {
                    stream.flushAsync().done(<span class="kwrd">function</span> () {
                        inputStream.close();
                        stream.close();
                    });
                });
            };
            captureOperation.start();
        });
    });
}
</pre> 
     </blockquote> 
     <p>The data can also be captured to an in memory stream and manipulated in memory, for example scaling it using the BitMapTransform object. When capturing a snapshot, the captured image is bounded by the area of the content within the control’s viewport. </p> 
     <h2>Script notify changes in XAML</h2> 
     <p>To improve the security of WebView, we’ve restricted when window.external.notify() can be used from WebView content. These restrictions prevent untrusted content, or content that has been tampered with, from sending messages that are executed without validation to the host. For content to be able to send notifications the following conditions apply: </p> 
     <ul> 
      <li>The source of the page should be from the local system via NavigateToString(), NavigateToStream() or ms-appx-web:///</li> 
     </ul> 
     <p>Or </p> 
     <ul> 
      <li>The source of the page is delivered via https:// and the site domain name is listed in the app content URI’s section of the package manifest.</li> 
     </ul> 
     <p>As part of this change, <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.allowedscriptnotifyuris.aspx" target="_blank">AllowedScriptNotifyUris</a> and <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.anyscriptnotifyuri.aspx" target="_blank">AnyScriptNotifyUri</a> are deprecated and will return errors in Windows 8.1 apps. </p> 
     <h2>XAML API changes and deprecation</h2> 
     <p>As the XAML and HTML versions of the WebView are based around the same core component, we wanted to move the APIs to be consistent between the controls. This has resulted in a few API changes on the XAML side: </p> 
     <ul> 
      <li><a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.invokescript.aspx" target="_blank">InvokeScript()</a> and <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.datatransferpackage.aspx" target="_blank">DataTransferPackage</a> are no longer supported. They are replaced by <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.invokescriptasync.aspx" target="_blank">InvokeScriptAsync()</a> and <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.captureselectedcontenttodatapackageasync.aspx" target="_blank">CaptureSelectedContentToDataPackageAsync()</a> methods. </li> 
      <li><a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.allowedscriptnotifyuris.aspx" target="_blank">AllowedScriptNotifyUris</a> and <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.anyscriptnotifyuri.aspx" target="_blank">AnyScriptNotifyUri</a> are no longer supported. They are replaced by the <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.notifyeventargs.aspx" target="_blank">INotifyEventArgs</a>’ <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.notifyeventargs.callinguri.aspx" target="_blank">CallingUri</a> property that allows the app to get the URI of the page that fired ScriptNotify. </li> 
      <li>The <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.loadcompleted.aspx" target="_blank">LoadCompleted</a> and <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.navigationfailed.aspx" target="_blank">NavigationFailed</a> events have been replaced by <a href="http://msdn.microsoft.com/en-us/library/windows/apps/windows.ui.xaml.controls.webview.navigationcompleted.aspx" target="_blank">NavigationCompleted</a>.</li> 
     </ul> 
     <p>In XAML these APIs are no longer supported, which means that Windows 8.1 apps can still be built using the API, but will receive compiler warnings. </p> 
     <h2>In closing</h2> 
     <p>We have made significant improvements to the WebView control for XAML and have fixed some of the main pain points that developers have felt in Windows 8. Plus as an HTML and JavaScript developer, you can now use WebView in your apps as well. We hope you’ll take time to try out the new version of the control and provide feedback on the <a href="http://social.msdn.microsoft.com/Forums/windowsapps/en-US/home?forum=w81prevwCsharp" target="_blank">XAML</a> and <a href="http://social.msdn.microsoft.com/Forums/windowsapps/en-US/home?forum=w81prevwhtml5" target="_blank">HTML</a> forums. We look forward to seeing the apps you can create using it. </p>
