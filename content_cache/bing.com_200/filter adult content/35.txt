      <p>A <b>Java servlet</b> is a <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a> <a href="/wiki/Computer_program" title="Computer program">program</a> that extends the capabilities of a <a href="/wiki/Server_(computing)" title="Server (computing)">server</a>. Although servlets can respond to any types of requests, they most commonly implement applications hosted on <a href="/wiki/Web_server" title="Web server">Web servers</a>.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup> Such Web servlets are the <a href="/wiki/Java_(software_platform)" title="Java (software platform)">Java</a> counterpart to other <a href="/wiki/Dynamic_web_page" title="Dynamic web page">dynamic Web content</a> technologies such as <a href="/wiki/PHP" title="PHP">PHP</a> and <a href="/wiki/ASP.NET" title="ASP.NET">ASP.NET</a>.</p>

      <h2><span class="mw-headline" id="Introduction">Introduction</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Java_servlet&amp;action=edit&amp;section=1" title="Edit section: Introduction">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>A Java servlet processes or stores a <a href="/wiki/Java_class" class="mw-redirect" title="Java class">Java class</a> in <a href="/wiki/Java_EE" class="mw-redirect" title="Java EE">Java EE</a> that conforms to the Java Servlet API,<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup> a standard for implementing Java classes that respond to requests. Servlets could in principle communicate over any <a href="/wiki/Client%E2%80%93server_model" title="Client–server model">client–server</a> protocol, but they are most often used with the <a href="/wiki/Hypertext_Transfer_Protocol" title="Hypertext Transfer Protocol">HTTP protocol</a>. Thus "servlet" is often used as shorthand for "HTTP servlet".<sup id="cite_ref-r1_3-0" class="reference"><a href="#cite_note-r1-3">[3]</a></sup> Thus, a <a href="/wiki/Software_developer" title="Software developer">software developer</a> may use a servlet to add <a href="/wiki/Dynamic_web_page" title="Dynamic web page">dynamic content</a> to a <a href="/wiki/Web_server" title="Web server">web server</a> using the <a href="/wiki/Java_platform" class="mw-redirect" title="Java platform">Java platform</a>. The generated content is commonly <a href="/wiki/HTML" title="HTML">HTML</a>, but may be other data such as <a href="/wiki/XML" title="XML">XML</a>. Servlets can maintain <a href="/wiki/State_(computer_science)" title="State (computer science)">state</a> in <a href="/wiki/Session_(computer_science)" title="Session (computer science)">session</a> variables across many server transactions by using <a href="/wiki/HTTP_cookie" title="HTTP cookie">HTTP cookies</a>, or <a href="/wiki/URL_rewriting" class="mw-redirect" title="URL rewriting">URL rewriting</a>.</p> 
      <p>To deploy and run a servlet, a <a href="/wiki/Web_container" title="Web container">web container</a> must be used. A web container (also known as a servlet container) is essentially the component of a web server that interacts with the servlets. The web container is responsible for managing the lifecycle of servlets, mapping a URL to a particular servlet and ensuring that the URL requester has the correct access rights.</p> 
      <p>The Servlet <a href="/wiki/Application_programming_interface" title="Application programming interface">API</a>, contained in the <a href="/wiki/Java_package" title="Java package">Java package</a> hierarchy <code><a rel="nofollow" class="external text" href="//docs.oracle.com/javaee/7/api/javax/servlet/package-summary.html">javax.servlet</a></code>, defines the expected interactions of the web container and a servlet.<sup id="cite_ref-r1_3-1" class="reference"><a href="#cite_note-r1-3">[3]</a></sup></p> 
      <p>A <code><a rel="nofollow" class="external text" href="//docs.oracle.com/javaee/7/api/javax/servlet/Servlet.html">Servlet</a></code> is an <a href="/wiki/Object_(computer_science)" title="Object (computer science)">object</a> that receives a request and generates a response based on that request. The basic Servlet package defines Java objects to represent servlet requests and responses, as well as objects to reflect the servlet's configuration parameters and execution environment. The package <code><a rel="nofollow" class="external text" href="//docs.oracle.com/javaee/7/api/javax/servlet/http/package-summary.html">javax.servlet.http</a></code> defines <a href="/wiki/HTTP" class="mw-redirect" title="HTTP">HTTP</a>-specific subclasses of the generic servlet elements, including session management objects that track multiple requests and responses between the web server and a client. Servlets may be packaged in a <a href="/wiki/WAR_file_format_(Sun)" class="mw-redirect" title="WAR file format (Sun)">WAR file</a> as a <a href="/wiki/Web_application" title="Web application">web application</a>.</p> 
      <p>Servlets can be generated automatically from <a href="/wiki/JavaServer_Pages" title="JavaServer Pages">JavaServer Pages</a> (JSP) by the <a href="/wiki/JavaServer_Pages_compiler" class="mw-redirect" title="JavaServer Pages compiler">JavaServer Pages compiler</a>. The difference between servlets and JSP is that servlets typically embed HTML inside Java code, while JSPs embed Java code in HTML. While the direct usage of servlets to generate HTML (as shown in the example below) has become rare, the higher level MVC web framework in Java EE (<a href="/wiki/JavaServer_Faces" title="JavaServer Faces">JSF</a>) still explicitly uses the servlet technology for the low level request/response handling via the <code><a rel="nofollow" class="external text" href="//docs.oracle.com/javaee/7/api/javax/faces/webapp/FacesServlet.html">FacesServlet</a></code>. A somewhat older usage is to use servlets in conjunction with JSPs in a pattern called "<a href="/wiki/JSP_model_2_architecture" title="JSP model 2 architecture">Model 2</a>", which is a flavor of the <a href="/wiki/Model%E2%80%93view%E2%80%93controller" title="Model–view–controller">model–view–controller</a>.</p> 
      <p>The current version of Servlet is 3.1. This is the upgraded version.</p> 
      <h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Java_servlet&amp;action=edit&amp;section=2" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>The Servlet1 specification was created by Pavni Diwanji<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup> while she worked at <a href="/wiki/Sun_Microsystems" title="Sun Microsystems">Sun Microsystems</a>, with version 1.0 finalized in June 1997. Starting with version 2.2, the specification was developed under the <a href="/wiki/Java_Community_Process" title="Java Community Process">Java Community Process</a>. As of September 2017<sup class="plainlinks noprint asof-tag update" style="display:none;"><a class="external text" href="//en.wikipedia.org/w/index.php?title=Java_servlet&amp;action=edit">[update]</a></sup>, the current version of the Servlet specification is 4.0.</p> 
      <p>In his blog on <a href="/wiki/Java.net" title="Java.net">java.net</a>, Sun veteran and <a href="/wiki/GlassFish" title="GlassFish">GlassFish</a> lead Jim Driscoll details the history of servlet technology.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup> <a href="/wiki/James_Gosling" title="James Gosling">James Gosling</a> first thought of servlets in the early days of <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>, but the concept did not become a product until Sun shipped the <span class="clarify-content" style="padding-left:0.1em; padding-right:0.1em; color:#595959; border:1px solid #DDD;"><i>Java Web Server</i></span><sup class="noprint Inline-Template Template-Clarify" style="margin-left:0.1em; white-space:nowrap;">[<i><a href="/wiki/Wikipedia:Please_clarify" title="Wikipedia:Please clarify"><span title="This passage needs to be better explained. (September 2012)">clarify</span></a></i>]</sup> product. This was before what is now the <a href="/wiki/Java_Platform,_Enterprise_Edition" title="Java Platform, Enterprise Edition">Java Platform, Enterprise Edition</a> was made into a specification.</p>

      <p>The advantages of using servlets are their fast performance and ease of use combined with more power over traditional <a href="/wiki/Common_Gateway_Interface" title="Common Gateway Interface">CGI</a> (Common Gateway Interface). Traditional CGI scripts written in Java have a number of performance disadvantages:</p> 
      <ul> 
       <li>When an HTTP request is made, a new process is created each time the CGI script is called. The overhead associated with process creation can dominate the workload especially when the script does relatively fast operations. Thus, process creation will take more time for CGI script execution. In contrast, for servlets, each request is handled by a separate Java thread <i>within</i> the web server process, thereby avoiding the overhead associated with forking processes within the <a href="/wiki/HTTP_daemon" class="mw-redirect" title="HTTP daemon">HTTP daemon</a>.</li> 
       <li>Simultaneous CGI requests will load the CGI script to be copied into memory once per request. With servlets, there is only one copy that persists across requests and is shared between threads.</li> 
       <li>Only a single instance answers all requests concurrently. This reduces memory usage and eases the management of persistent data.</li> 
       <li>A servlet can be run by a servlet container in a restrictive environment, called a <a href="/wiki/Sandbox_(computer_security)" title="Sandbox (computer security)">sandbox</a>. This is similar to an <a href="/wiki/Applet" title="Applet">applet</a> that runs in the sandbox of the web browser. This enables restricted use of potentially harmful servlets.<sup id="cite_ref-r1_3-2" class="reference"><a href="#cite_note-r1-3">[3]</a></sup> CGI programs can of course also sandbox themselves, since they are simply OS processes.</li> 
      </ul> 
      <p>Technologies like <a href="/wiki/FastCGI" title="FastCGI">FastCGI</a> and its derivatives (including <a href="/wiki/Simple_Common_Gateway_Interface" title="Simple Common Gateway Interface">SCGI</a>, <a href="/wiki/Apache_JServ_Protocol" title="Apache JServ Protocol">AJP</a>) do not exhibit the performance disadvantages of CGI, incurred by the constant process spawning. They are, however, roughly as simple as CGI. They are therefore also in contrast with servlets which are substantially more complex.</p> 
      <h2><span class="mw-headline" id="Life_cycle_of_a_servlet">Life cycle of a servlet</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Java_servlet&amp;action=edit&amp;section=4" title="Edit section: Life cycle of a servlet">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>Three methods are central to the life cycle of a servlet. These are <code>init()</code>, <code>service()</code>, and <code>destroy()</code>. They are implemented by every servlet and are invoked at specific times by the server.</p> 
      <ul> 
       <li>During initialization stage of the servlet <a href="/wiki/Object_lifetime" title="Object lifetime">life cycle</a>, the web container initializes the servlet instance by calling the <a rel="nofollow" class="external text" href="http://docs.oracle.com/javaee/7/api/javax/servlet/Servlet.html#init"><code>init()</code></a> method, passing an object implementing the <a rel="nofollow" class="external text" href="http://docs.oracle.com/javaee/7/api/javax/servlet/ServletConfig.html"><code>javax.servlet.ServletConfig</code></a> interface. This configuration object allows the servlet to access <a href="/wiki/Attribute%E2%80%93value_pair" title="Attribute–value pair">name-value</a> initialization parameters from the web application.</li> 
       <li>After initialization, the servlet instance can service client requests. Each <a href="/wiki/HTTP_request#Request_message" class="mw-redirect" title="HTTP request">request</a> is serviced in its own separate thread. The web container calls the <code>service()</code> method of the servlet for every request. The <code>service()</code> method determines the kind of request being made and dispatches it to an appropriate method to handle the request. The difference between servlets and JSP is that servlets typically embed HTML inside Java code, while JSPs embed Java code in HTML. The developer of the servlet must provide an implementation for these methods. If a request is made for a method that is not implemented by the servlet, the method of the parent class is called, typically resulting in an error being returned to the requester.</li> 
       <li>Finally, the web container calls the <code>destroy()</code> method that takes the servlet out of service. The <code>destroy()</code> method, like <code>init()</code>, is called only once in the lifecycle of a servlet.</li> 
      </ul> 
      <p>The following is a typical user scenario of these methods.</p> 
      <ol> 
       <li>Assume that a user requests to visit a <a href="/wiki/Uniform_resource_locator" class="mw-redirect" title="Uniform resource locator">URL</a>. 
        <ul> 
         <li>The browser then generates an HTTP request for this URL.</li> 
         <li>This request is then sent to the appropriate server.</li> 
        </ul> </li> 
       <li>The HTTP request is received by the web server and forwarded to the servlet container. 
        <ul> 
         <li>The container maps this request to a particular servlet.</li> 
         <li>The servlet is dynamically retrieved and loaded into the address space of the container.</li> 
        </ul> </li> 
       <li>The container invokes the <code>init()</code> method of the servlet. 
        <ul> 
         <li>This method is invoked only when the servlet is first loaded into memory.</li> 
         <li>It is possible to pass initialization parameters to the servlet so that it may configure itself.</li> 
        </ul> </li> 
       <li>The container invokes the <code>service()</code> method of the servlet. 
        <ul> 
         <li>This method is called to process the HTTP request.</li> 
         <li>The servlet may read data that has been provided in the HTTP request.</li> 
         <li>The servlet may also formulate an HTTP response for the client.</li> 
        </ul> </li> 
       <li>The servlet remains in the container's address space and is available to process any other HTTP requests received from clients. 
        <ul> 
         <li>The <code>service()</code> method is called for each HTTP request.</li> 
        </ul> </li> 
       <li>The container may, at some point, decide to unload the servlet from its memory. 
        <ul> 
         <li>The algorithms by which this decision is made are specific to each container.</li> 
        </ul> </li> 
       <li>The container calls the servlet's <code>destroy()</code> method to relinquish any resources such as file handles that are allocated for the servlet; important data may be saved to a persistent store.</li> 
       <li>The memory allocated for the servlet and its objects can then be garbage collected.</li> 
      </ol> 
      <h2><span class="mw-headline" id="Example">Example</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Java_servlet&amp;action=edit&amp;section=5" title="Edit section: Example">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>The following example servlet prints how many times its <code>service()</code> method was called.</p> 
      <p>Note that <code>HttpServlet</code> is a subclass of <code>GenericServlet</code>, an implementation of the <code>Servlet</code> interface.</p> 
      <p>The <code>service()</code> method of <code>HttpServlet</code> class dispatches requests to the methods <code>doGet()</code>, <code>doPost()</code>, <code>doPut()</code>, <code>doDelete()</code>, and so on; according to the HTTP request. In the example below <code>service()</code> is overridden and does not distinguish which HTTP request method it serves.</p> 
      <div class="mw-highlight mw-content-ltr" dir="ltr"> 
       <pre>
<span class="kn">import</span> <span class="nn">java.io.IOException</span><span class="o">;</span>
 
<span class="kn">import</span> <span class="nn">javax.servlet.ServletConfig</span><span class="o">;</span>
<span class="kn">import</span> <span class="nn">javax.servlet.ServletException</span><span class="o">;</span>
<span class="kn">import</span> <span class="nn">javax.servlet.http.HttpServlet</span><span class="o">;</span>
<span class="kn">import</span> <span class="nn">javax.servlet.http.HttpServletRequest</span><span class="o">;</span>
<span class="kn">import</span> <span class="nn">javax.servlet.http.HttpServletResponse</span><span class="o">;</span>

<span class="kd">public</span> <span class="kd">class</span> <span class="nc">ServletLifeCycleExample</span> <span class="kd">extends</span> <span class="n">HttpServlet</span> <span class="o">{</span>
 
    <span class="kd">private</span> <span class="kt">int</span> <span class="n">count</span><span class="o">;</span>
 
    <span class="nd">@Override</span>
    <span class="kd">public</span> <span class="kt">void</span> <span class="nf">init</span><span class="o">(</span><span class="kd">final</span> <span class="n">ServletConfig</span> <span class="n">config</span><span class="o">)</span> <span class="kd">throws</span> <span class="n">ServletException</span> <span class="o">{</span>
        <span class="kd">super</span><span class="o">.</span><span class="na">init</span><span class="o">(</span><span class="n">config</span><span class="o">);</span>
        <span class="n">getServletContext</span><span class="o">().</span><span class="na">log</span><span class="o">(</span><span class="s">"init() called"</span><span class="o">);</span>
        <span class="n">count</span> <span class="o">=</span> <span class="mi">0</span><span class="o">;</span>
    <span class="o">}</span>
 
    <span class="nd">@Override</span>
    <span class="kd">protected</span> <span class="kt">void</span> <span class="nf">service</span><span class="o">(</span><span class="kd">final</span> <span class="n">HttpServletRequest</span> <span class="n">request</span><span class="o">,</span> <span class="kd">final</span> <span class="n">HttpServletResponse</span> <span class="n">response</span><span class="o">)</span> <span class="kd">throws</span> <span class="n">ServletException</span><span class="o">,</span> <span class="n">IOException</span> <span class="o">{</span>
        <span class="n">getServletContext</span><span class="o">().</span><span class="na">log</span><span class="o">(</span><span class="s">"service() called"</span><span class="o">);</span>
        <span class="n">count</span><span class="o">++;</span>
        <span class="n">response</span><span class="o">.</span><span class="na">getWriter</span><span class="o">().</span><span class="na">write</span><span class="o">(</span><span class="s">"Incrementing the count to "</span> <span class="o">+</span> <span class="n">count</span><span class="o">);</span>
    <span class="o">}</span>
 
    <span class="nd">@Override</span>
    <span class="kd">public</span> <span class="kt">void</span> <span class="nf">destroy</span><span class="o">()</span> <span class="o">{</span>
        <span class="n">getServletContext</span><span class="o">().</span><span class="na">log</span><span class="o">(</span><span class="s">"destroy() called"</span><span class="o">);</span>
    <span class="o">}</span>
<span class="o">}</span>
</pre>
