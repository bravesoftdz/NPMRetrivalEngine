             <div class="post-text" itemprop="text"> 
              <p>If you have <a href="http://en.wikipedia.org/wiki/Spring_Framework" rel="noreferrer">Spring</a> in you classpath then the following will do it.</p> 
              <p>Find all classes in a package that are annotated with XmlRootElement:</p> 
              <pre><code>private List&lt;Class&gt; findMyTypes(String basePackage) throws IOException, ClassNotFoundException
{
    ResourcePatternResolver resourcePatternResolver = new PathMatchingResourcePatternResolver();
    MetadataReaderFactory metadataReaderFactory = new CachingMetadataReaderFactory(resourcePatternResolver);

    List&lt;Class&gt; candidates = new ArrayList&lt;Class&gt;();
    String packageSearchPath = ResourcePatternResolver.CLASSPATH_ALL_URL_PREFIX +
                               resolveBasePackage(basePackage) + "/" + "**/*.class";
    Resource[] resources = resourcePatternResolver.getResources(packageSearchPath);
    for (Resource resource : resources) {
        if (resource.isReadable()) {
            MetadataReader metadataReader = metadataReaderFactory.getMetadataReader(resource);
            if (isCandidate(metadataReader)) {
                candidates.add(Class.forName(metadataReader.getClassMetadata().getClassName()));
            }
        }
    }
    return candidates;
}

private String resolveBasePackage(String basePackage) {
    return ClassUtils.convertClassNameToResourcePath(SystemPropertyUtils.resolvePlaceholders(basePackage));
}

private boolean isCandidate(MetadataReader metadataReader) throws ClassNotFoundException
{
    try {
        Class c = Class.forName(metadataReader.getClassMetadata().getClassName());
        if (c.getAnnotation(XmlRootElement.class) != null) {
            return true;
        }
    }
    catch(Throwable e){
    }
    return false;
}
</code></pre>

              <pre><code>public static ArrayList&lt;String&gt;getClassNamesFromPackage(String packageName) throws IOException{
    ClassLoader classLoader = Thread.currentThread().getContextClassLoader();
    URL packageURL;
    ArrayList&lt;String&gt; names = new ArrayList&lt;String&gt;();;

    packageName = packageName.replace(".", "/");
    packageURL = classLoader.getResource(packageName);

    if(packageURL.getProtocol().equals("jar")){
        String jarFileName;
        JarFile jf ;
        Enumeration&lt;JarEntry&gt; jarEntries;
        String entryName;

        // build jar file name, then loop through zipped entries
        jarFileName = URLDecoder.decode(packageURL.getFile(), "UTF-8");
        jarFileName = jarFileName.substring(5,jarFileName.indexOf("!"));
        System.out.println("&gt;"+jarFileName);
        jf = new JarFile(jarFileName);
        jarEntries = jf.entries();
        while(jarEntries.hasMoreElements()){
            entryName = jarEntries.nextElement().getName();
            if(entryName.startsWith(packageName) &amp;&amp; entryName.length()&gt;packageName.length()+5){
                entryName = entryName.substring(packageName.length(),entryName.lastIndexOf('.'));
                names.add(entryName);
            }
        }

    // loop through files in classpath
    }else{
    URI uri = new URI(packageURL.toString());
    File folder = new File(uri.getPath());
        // won't work with path which contains blank (%20)
        // File folder = new File(packageURL.getFile()); 
        File[] contenuti = folder.listFiles();
        String entryName;
        for(File actual: contenuti){
            entryName = actual.getName();
            entryName = entryName.substring(0, entryName.lastIndexOf('.'));
            names.add(entryName);
        }
    }
    return names;
}
</code></pre>

             <div class="post-text" itemprop="text"> 
              <p>Spring has implemented an excellent classpath search function in the <a href="http://static.springsource.org/spring/docs/2.0.x/api/org/springframework/core/io/support/PathMatchingResourcePatternResolver.html" rel="noreferrer"><code>PathMatchingResourcePatternResolver</code></a>. If you use the <code>classpath*</code>: prefix, you can find all the resources, including classes in a given hierarchy, and even filter them if you want. Then you can use the children of <a href="http://static.springsource.org/spring/docs/2.5.x/api/org/springframework/core/type/filter/AbstractTypeHierarchyTraversingFilter.html" rel="noreferrer"><code>AbstractTypeHierarchyTraversingFilter</code></a>, <code>AnnotationTypeFilter</code> and <code>AssignableTypeFilter</code> to filter those resources either on class level annotations or on interfaces they implement.</p>

              <p>Java 1.6.0_24:</p> 
              <pre><code>public static File[] getPackageContent(String packageName) throws IOException{
    ArrayList&lt;File&gt; list = new ArrayList&lt;File&gt;();
    Enumeration&lt;URL&gt; urls = Thread.currentThread().getContextClassLoader()
                            .getResources(packageName);
    while (urls.hasMoreElements()) {
        URL url = urls.nextElement();
        File dir = new File(url.getFile());
        for (File f : dir.listFiles()) {
            list.add(f);
        }
    }
    return list.toArray(new File[]{});
}
</code></pre>

              <p><a href="http://scannotation.sourceforge.net/" rel="nofollow">Scannotation</a> and <a href="https://github.com/ronmamo/reflections" rel="nofollow">Reflections</a> use class path scanning approach:</p> 
              <pre><code>Reflections reflections = new Reflections("my.package");
Set&lt;Class&lt;? extends Object&gt;&gt; classes = reflections.getSubTypesOf(Object.class);
</code></pre> 
              <p>Another approach is to use <a href="http://docs.oracle.com/javase/6/docs/api/javax/annotation/processing/Processor.html" rel="nofollow">Java Pluggable Annotation Processing API</a> to write annotation processor which will collect all annotated classes at compile time and build the index file for runtime use. This mechanism is implemented in <a href="https://github.com/atteo/classindex" rel="nofollow">ClassIndex</a> library:</p> 
              <pre><code>Iterable&lt;Class&gt; classes = ClassIndex.getPackageClasses("my.package");
</code></pre>

              <p>That functionality is still suspiciously missing from the Java reflection API as far as I know. You can get a package object by just doing this:</p> 
              <pre><code>Package packageObj = Package.getPackage("my.package");
</code></pre> 
              <p>But as you probably noticed, that won't let you list the classes in that package. As of right now, you have to take sort of a more filesystem-oriented approach.</p> 
              <p>I found some sample implementations in <a href="http://forums.sun.com/thread.jspa?threadID=341935" rel="nofollow noreferrer">this</a> post </p> 
              <p>I'm not 100% sure these methods will work when your classes are buried in JAR files, but I hope one of those does it for you.</p> 
              <p>I agree with @skaffman...if you have another way of going about this, I'd recommend doing that instead.</p>

             <div class="post-text" itemprop="text"> 
              <p>I happen to have implemented it, and it works in most cases. Since it is long, I put it in a <a href="http://dl.getdropbox.com/u/1961549/StackOverflow/SO1456930/ClassDiscovery.java" rel="nofollow noreferrer">file here</a>.</p> 
              <p>The idea is to find the location of the class source file which is available in most cases (a known exception are JVM class files -- as far as I've tested). If the code is in a directory, scan through all files and only spot class files. If the code is in a JAR file, scan all entries.</p> 
              <p>This method can only be used when:</p> 
              <ol> 
               <li><p>You have a class that is in the same package you want to discover, This class is called a SeedClass. For example, if you want to list all classes in 'java.io', the seed class may be <code>java.io.File</code>.</p></li> 
               <li><p>Your classes are in a directory or in a JAR file it has source file information (not source code file, but just source file). As far as I've tried, it work almost 100% except the JVM class (those classes come with the JVM).</p></li> 
               <li><p>Your program must have permission to access ProtectionDomain of those classes. If your program is loaded locally, there should be no problem.</p></li> 
              </ol> 
              <p>I've tested the program only for my regular usage, so it may still have problem.</p> 
              <p>I hope this helps.</p>

              <p><a href="http://sourceforge.net/projects/extcos/" rel="nofollow">eXtcos</a> looks promising. Imagine you want to find all the classes that:</p> 
              <ol> 
               <li>Extend from class "Component", and store them</li> 
               <li>Are annotated with "MyComponent", and</li> 
               <li>Are in the “common” package. </li> 
              </ol> 
              <p>With eXtcos this is as simple as</p> 
              <pre><code>ClasspathScanner scanner = new ClasspathScanner();
final Set&lt;Class&gt; classStore = new ArraySet&lt;Class&gt;();

Set&lt;Class&gt; classes = scanner.getClasses(new ClassQuery() {
    protected void query() {
        select().
        from(“common”).
        andStore(thoseExtending(Component.class).into(classStore)).
        returning(allAnnotatedWith(MyComponent.class));
    }
});
</code></pre>

             <div class="post-text" itemprop="text"> 
              <ol> 
               <li><p>Bill Burke has written a (nice <a href="http://bill.burkecentral.com/2008/01/14/scanning-java-annotations-at-runtime/" rel="nofollow">article</a> about class scanning] and then he wrote <strong><a href="http://scannotation.sourceforge.net/" rel="nofollow">Scannotation</a></strong>.</p></li> 
               <li><p><strong>Hibernate</strong> has this already written:</p> 
                <ul> 
                 <li>org.hibernate.ejb.packaging.Scanner</li> 
                 <li>org.hibernate.ejb.packaging.NativeScanner</li> 
                </ul></li> 
               <li><p><a href="http://docs.jboss.org/weld/reference/latest/en-US/html_single/#lookup" rel="nofollow"><strong>CDI</strong></a> might solve this, but don't know - haven't investigated fully yet </p></li> 
              </ol> 
              <p>.</p> 
              <pre><code>@Inject Instance&lt; MyClass&gt; x;
...
x.iterator() 
</code></pre> 
              <p>Also for annotations:</p> 
              <pre><code>abstract class MyAnnotationQualifier
extends AnnotationLiteral&lt;Entity&gt; implements Entity {}

            <td class="answercell"> 
             <div class="post-text" itemprop="text"> 
              <p>You could try my library <a href="https://github.com/lukehutch/fast-classpath-scanner" rel="nofollow">FastClasspathScanner</a>. Scanning the classpath is not as simple as checking the <code>java.class.path</code> property, and recursively scanning for classes there, because there are many ways that the classpath can be specified (e.g. you can add <code>Class-Path</code> entries to a jarfile's manifest). Using Class.forName() actually initializes the class, which may not be what you want, etc. FastClasspathScanner handles these complexities for you.</p> 
              <p>To enumerate all classes in a package, do the following:</p> 
              <pre><code>String packageName = "my.package";
Set&lt;String&gt; classNames = new FastClassPathScanner(packageName)
    .scan()
    .getNamesOfAllClasses();
String packageNamePrefix = packageName + ".";
for (String className : classNames) {
    if (className.startsWith(packageNamePrefix) {
        System.out.println("Found class: " + className);
   }
}
</code></pre> 
              <p>You need the check <code>if (className.startsWith(packageNamePrefix)</code> because if a class refers to another class (e.g. as a superclass), and it's outside the whitelisted package prefix <code>my.package</code>, it will be included in the set returned by <code>.getNamesOfAllClasses()</code>.</p>

             <div class="post-text" itemprop="text"> 
              <p>Back when applets were common place, one might have a URL on the classpath. When the classloader required a class, it would search all the locations on the classpath, including http resources. Because you can have things like URLs and directories on the classpath, there is no easy way to get a definitive list of the classes.</p> 
              <p>However, you can get pretty close. Some of the Spring libraries are doing this now. You can get all the jar's on the classpath, and open them up like files. You can then take this list of files, and create a data structure containing your classes.</p>

              <p>Brent - the reason the association is one way has to do with the fact that any class on any component of your CLASSPATH can declare itself in any package (except for java/javax). Thus there just is no mapping of ALL the classes in a given "package" because nobody knows nor can know. You could update a jar file tomorrow and remove or add classes. It's like trying to get a list of all people named John/Jon/Johan in all the countries of the world - none of us is omniscient therefore none of us will ever have the correct answer.</p>
