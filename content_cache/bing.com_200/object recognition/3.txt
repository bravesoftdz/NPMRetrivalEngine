       <h2>Class declaration</h2>
       <p>A <em>class</em> is a template for manufacturing objects. You declare a class by specifying the <code>class</code> keyword followed by a non-reserved identifier that names it. A pair of matching open and close brace characters (<code>{</code> and <code>}</code>) follow and delimit the class's body. This syntax appears below:</p>
       <pre class="prettyprint"><code>class <em>identifier</em>
{
   // class body
}</code></pre>
       <p>By convention, the first letter of a class's name is uppercased and subsequent characters are lowercased (for example, <code>Employee</code>). If a name consists of multiple words, the first letter of each word is uppercased (such as <code>SavingsAccount</code>). This naming convention is called <a href="https://en.wikipedia.org/wiki/CamelCase" target="_blank" rel="nofollow"><em>camelcasing</em></a>.</p>
       <p>The following example declares a class named <code>Book</code>:</p>
       <aside class="nativo-promo smartphone" id=""> 
       </aside>
       <pre class="prettyprint"><code>class Book
{
   // class body
}</code></pre>
       <p>A class's body is populated with fields, methods, and constructors. Combining these language features into classes is known as <em>encapsulation</em>. This capability lets us program at a higher level of abstraction (classes and objects) rather than focusing separately on data structures and functionality.</p>
       <aside class="sidebar"> 
        <h3>Utility classes</h3> 
        <p>A class can be designed to have nothing to do with object manufacturing. Instead, it exists as a placeholder for class fields and/or class methods. Such a class is known as a <em>utility class</em>. An example of a utility class is the Java standard class library's <code>Math</code> class. See the <a href="http://www.javaworld.com/PDFLINK">Fields and methods in Java</a> primer for another example.</p> 
       </aside>
       <h2>Multi-class applications and main()</h2>
       <p>A Java application is implemented by one or more classes. Small applications can be accommodated by a single class, but larger applications often require multiple classes. In that case one of the classes is designated as the <em>main class</em> and contains the <code>main()</code> entry-point method. For example, Listing 1 presents an application built using three classes: <code>A</code>, <code>B</code>, and <code>C</code>; <code>C</code> is the main class.</p>
       <h4>Listing 1. A Java application with multiple classes</h4>
       <pre class="prettyprint"><code>class A
{
}

class B
{
}

class C
{
   public static void main(String[] args)
   {
      System.out.println("Application C entry point");
   }
}</code></pre>
       <p>You could declare these three classes in a single source file, such as <code>D.java</code>. You would then compile this source file as follows:</p>
       <pre><code>javac D.java</code></pre>
       <p>The compiler generates three class files: <code>A.class</code>, <code>B.class</code>, and <code>C.class</code>. Run this application via the following command:</p>
       <aside class="nativo-promo tablet desktop" id=""> 
       </aside>
       <pre><code>java C</code></pre>
       <p>You should observe the following output:</p>
       <pre><code>Application C entry point</code></pre>
       <p>Alternatively, you could declare each class in its own source file. By convention, the source file's name matches the class name. You would declare <code>A</code> in <code>A.java</code>, for instance. You could then compile these source files separately:</p>
       <pre><code>javac A.java
javac B.java
javac C.java</code></pre>
       <p>To save time, you could compile all three source files at once by replacing the file name with an asterisk (but keep the <code>.java</code> file extension):</p>
       <pre><code>javac *.java</code></pre>
       <p>Either way, you would run the application via the following command:</p>
       <pre><code>java C</code></pre>
       <aside class="sidebar"> 
        <h3>Public classes</h3> 
        <p>Java lets you declare a class with public access via the <code>public</code> keyword. When you declare a class <code>public</code>, you must store it in a file with the same name. For example, you would store <code>public class C {}</code> in <code>C.java</code>. You may declare only one <code>public</code> class in a source file.</p> 
       </aside>
       <p>When designing multi-class applications, you will designate one of these classes as the main class and locate the <code>main()</code> method in it. However, there is nothing to prevent you from declaring <code>main()</code> methods in the other classes, perhaps for testing purposes. This technique is shown in Listing 2.</p>
       <h4>Listing 2. Declaring more than one <code>main()</code> method</h4>
       <pre class="prettyprint"><code>class A
{
   public static void main(String[] args)
   {
      System.out.println("Testing class A");
   }
}

class B
{
   public static void main(String[] args)
   {
      System.out.println("Testing class B");
   }
}

class C
{
   public static void main(String[] args)
   {
      System.out.println("Application C entry point");
   }
}</code></pre>
       <p>After compiling the source code, you would execute the following commands to test the helper classes <code>A</code> and <code>B</code>, and to run the application class <code>C</code>:</p>
       <pre><code>java A
java B
java C</code></pre>
       <p>You would then observe the following lines of output, one line per <code>java</code> command:</p>
       <pre><code>Testing class A
Testing class B
Application C entry point</code></pre>
       <aside class="sidebar"> 
        <h3>Be careful with main()</h3> 
        <p>Placing a <code>main()</code> method in each class can be confusing, especially if you forget to document the main class. Also, you might forget to remove these methods before putting the application into production, in which case their presence would add bulk to the application. Furthermore, someone might run one of the supporting classes, which could disrupt the application's environment.</p> 
       </aside>
       <h2>Fields: Describing attributes</h2>
       <p>A class models a real-world entity in terms of state (attributes). For example, a vehicle has a color and a checking account has a balance. A class can also include non-entity state. Regardless, state is stored in variables that are known as <em>fields</em>. A field declaration has the following syntax:</p>
       <pre class="prettyprint"><code>[static] <em>type</em> <em>identifier</em> [ = <em>expression</em> ] ;</code></pre>
       <p>A field declaration optionally begins with keyword <code>static</code> (for a non-entity attribute) and continues with a <em><code>type</code></em> that's followed by a non-reserved <em><code>identifier</code></em> that names the field. The field can be explicitly initialized by specifying <code>=</code> followed by an <em><code>expression</code></em> with a compatible type. A semicolon terminates the declaration.</p>
       <p>The following example declares a pair of fields in <code>Book</code>:</p>
       <pre class="prettyprint"><code>class Book
{
   String title;
   int pubYear; // publication year
}</code></pre>
       <p>The <code>title</code> and <code>pubYear</code> field declarations are identical to the variable declarations I presented in "<a href="http://www.javaworld.com/article/2925720/learn-java/elementary-java-language-features.html" target="_blank">Java 101: Elementary Java language features</a>." These fields are known as <em>instance fields</em> because each object contains its own copy of them.</p>
       <p>The <code>title</code> and <code>pubYear</code> fields store values for a specific book. However, you might want to store state that is independent of any particular book. For example, you might want to record the total number of <code>Book</code> objects created. Here's how you would do it:</p>
       <pre class="prettyprint"><code>class Book
{
   // ...

   static int count;
}</code></pre>
       <p>This example declares a <code>count</code> integer field that stores the number of <code>Book</code> objects created. The declaration begins with the <code>static</code> keyword to indicate that there is only one copy of this field in memory. Each <code>Book</code> object can access this copy, and no object has its own copy. For this reason, <code>count</code> is known as a <em>class field</em>.</p>
       <h2>Initialization</h2>
       <p>The previous fields were not assigned values. When you don't explicitly initialize a field, it's implicitly initialized with all of its bits set to zero. You interpret this default value as <code>false</code> (for <code>boolean</code>), <code>'\u0000'</code> (for <code>char</code>), <code>0</code> (for <code>int</code>), <code>0L</code> (for <code>long</code>), <code>0.0F</code> (for <code>float</code>), <code>0.0</code> (for <code>double</code>), or <code>null</code> (for a reference type).</p>
       <p>However, it is also possible to explicitly initialize a field when the field is declared. For example, you could specify <code>static int count = 0;</code> (which isn't necessary because <code>count</code> defaults to 0), <code>String logfile = "log.txt";</code>, <code>static int ID = 1;</code>, or even <code>double sinPIDiv2 = Math.sin(Math.PI / 2);</code>.</p>
       <p>Although you can initialize an instance field through direct assignment, it's more common to perform this initialization in a constructor, which I'll demonstrate later. In contrast, a class field (especially a class constant) is typically initialized through direct assignment of an expression to the field.</p>
       <h2>Lifetime and scope</h2>
       <p>An instance field is born when its object is created and dies when the object is <a href="http://www.javaworld.com/article/2078645/java-se/jvm-performance-optimization-part-3-garbage-collection.html" target="_blank">garbage collected</a>. A class field is born when the class is loaded and dies when the class is unloaded or when the application ends. This property is known as <em>lifetime</em>.</p>
       <p>Instance and class fields are accessible from their declarations to the end of their declaring classes. Furthermore, they are accessible to external code in an object context only (for instance fields) or object and class contexts (for class fields) when given suitable access levels. This property is known as <em>scope</em>.</p>
       <h2>Methods: Describing behaviors</h2>
       <p>In addition to modeling the state of a real-world entity, a class also models its behaviors. For example, a vehicle supports movement and a checking account supports deposits and withdrawals. A class can also include non-entity behaviors. Regardless, Java programmers use <em>methods</em> to describe behaviors. A method declaration has the following syntax:</p>
       <pre class="prettyprint"><code>[static] <em>returnType</em> <em>identifier</em> ( [<em>parameterList</em>] )
{
   // method body
}</code></pre>
       <p>A method declaration optionally begins with keyword <code>static</code> (for a non-entity behavior) and continues with a <em><code>returnType</code></em> that's followed by a non-reserved <em><code>identifier</code></em> that names the method. The name is then followed by a round bracket-delimited optional <em><code>parameterList</code></em>. A brace-delimited body containing code to execute when the method is called follows.</p>
       <p>The <em>return type</em> identifies the type of values that are returned from the method via the <code>return</code> statement, which I'll discuss later. For example, if a method returns strings, its return type would be set to <code>String</code>. When a method doesn't return a value, its return type is set to <code>void</code>.</p>
       <p>The <em>parameter list</em> is a comma-separated list of parameter declarations: each declaration consists of a type followed by a non-reserved identifier that names the parameter. A <em>parameter</em> is a variable that receives an <em>argument</em> (an expression value whose type is compatible with its corresponding parameter) when a method or constructor is called.</p>
       <p>A parameter is local to its method or constructor. It comes into existence when the method or constructor is called and disappears when the method or constructor returns to its caller. In other words, its lifetime is the method execution. A parameter can be accessed by any code within the method. Its scope is the entire method.</p>
       <p>The following example declares four methods in the <code>Book</code> class:</p>
       <pre class="prettyprint"><code>class Book
{
   // ...

   String getTitle()
   {
      return title;
   }

   int getPubYear()
   {
      return pubYear;
   }

   void setTitle(String _title)
   {
      title = _title;
   }

   void setPubYear(int _pubYear)
   {
      pubYear = _pubYear;
   }
}</code></pre>
       <p>The <code>getTitle()</code> and <code>getPubYear()</code> methods return the values of their respective fields. They use the <code>return</code> statement to return these values to the caller. Note that the type of this statement's expression must be compatible with the method's return type.</p>
       <p>The <code>setTitle()</code> and <code>setPubYear()</code> methods let you set the values of the <code>title</code> and <code>pubYear</code> fields. Their return types are set to keyword <code>void</code> to indicate that they don't return any values to their callers. All four methods are known as <em>instance methods</em> because they affect only the objects on which they are called.</p>
       <aside class="sidebar"> 
        <h3>Setters and getters</h3> 
        <p>The "<code>set</code>" prefix identifies <code>setTitle()</code> and <code>setPubYear()</code> as <em>setter methods</em>, meaning that they set values. Similarly, the "<code>get</code>" prefix identifies <code>getTitle()</code> and <code>getPubYear()</code> as <em>getter methods</em>, which means that they get values.</p> 
        <p>If you're wondering about the need for setter/getter methods in lieu of directly accessing <code>title</code> and <code>pubYear</code>, you'll find some good reasons in the "<a href="http://stackoverflow.com/questions/10407877/what-is-the-point-of-getters-and-setters" target="_blank" rel="nofollow">What is the point of getters and setters?</a>" topic at StackOverflow.com.</p> 
       </aside>
       <p>The <code>getTitle()</code>, <code>getPubYear()</code>, <code>setTitle()</code>, and <code>setPubYear()</code> methods affect a single object's copies of the <code>title</code> and <code>pubYear</code> fields. However, you might want to declare a method that's independent of any particular book. For example, you might want to introduce a method that outputs the number of <code>Book</code> objects, as follows:</p>
       <pre class="prettyprint"><code>class Book
{
   // ...

   static void showCount()
   {
      System.out.println("count = " + count);
   }
}</code></pre>
       <p>This example declares a <code>showCount()</code> method that will output the value of the <code>count</code> field. The declaration begins with the <code>static</code> keyword to indicate that this method belongs to the class and cannot access individual object state; no objects need to be created. For this reason, <code>showCount()</code> is known as a <em>class method</em>.</p>
       <h2>Local variables</h2>
       <p>Within a method or constructor, you can declare additional variables as part of its implementation. These variables are known as <em>local variables</em> because they are local to the method/constructor. They exist only while the method or constructor is executing and cannot be accessed from outside the method/constructor. Consider the following example:</p>
       <pre class="prettyprint"><code>static void average(double[] values)
{
   double sum = 0.0;
   for (int i = 0; i &lt; values.length; i++)
      sum += values[i];
   System.out.println("Average: " + (sum / values.length));
}</code></pre>
       <p>This example presents a method named <code>average</code> for calculating and outputting the average of an array of <code>double</code>s. In addition to parameter <code>values</code>, it declares local variables <code>sum</code> and <code>i</code> to help in the calculation. <code>sum</code>'s lifetime and scope range from its point of declaration to the end of the method. <code>i</code>'s lifetime and scope are confined to the <code>for</code> loop.</p>
