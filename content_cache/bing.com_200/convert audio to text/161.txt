  <h2><a name="Header_1">JRecord</a></h2> 
  <p>This project provides code to </p>
  <ol compact type="1"> 
   <li>Read and write files of length based records (both fixed length records and Length field based records). </li>
   <li>Read and write CSV files. </li>
   <li>Read and Write Flat Fixed width files (Text and Binary) via either a Xml-Record-Layout or a Cobol Copybook.. </li>
   <li>Read and write XML files (via StAX parser). </li>
   <li>Common IO routines across all File Types (XML, CSV, Fixed field Width). </li>
   <li>Support for various Flat file formats (Fixed, Delimited, Length based Files (i.e. Mainframe VB). </li>
  </ol> 
  <p> </p>
  <p></p>
  <h3><a name="Header_2">Guiding Principles</a></h3> 
  <p> </p>
  <ol compact type="1"> 
   <li>Method of reading / writing files should be the same no matter what the file format is. </li>
   <li>Programmer should be shielded from the files format. </li>
   <li>You should be able to edit files using the files description (Cobol Copybook, Xml or what ever). </li>
   <li>Package should be flexible - easy to add new file formats or Record-Layout formats. </li>
  </ol> 
  <p> </p>
  <p></p>
  <h3><a name="Header_3">Package Structure</a></h3> 
  <p>The JRecord package consists of </p>
  <ul compact> 
   <li>Low level line IO routines (ByteReader / ByteWriter). These classes read a line from a file as an array of Bytes (Formats supported include Fixed Length and various Length based file formats). These could be of interest to any one reading / writing Legacy Files. </li>
   <li>Classes to read / write File-Descriptions to/from external files </li>
   <li>High level IO Routines Supporting field level access. </li>
  </ul> 
  <p> </p>
  <p> </p>
  <p></p>
  <h3><a name="Header_4">Issues</a></h3> 
  <ol compact type="1"> 
   <li>Code was written as part of the <a href="http://record-editor.sourceforge.net" target="_top">RecordEditor</a> not as a Java library, This has both advantages and disadvantages: 
    <ul compact> 
     <li>The <a href="http://record-editor.sourceforge.net" target="_top">RecordEditor</a> has had 20,000 downloads over several years. Many parts of the package have been well tested </li>
     <li>Some code is there for the RecordEditor making the JRecord package larger that it would be otherwise be. </li>
     <li>Classes minimizing space at the expense of speed. This makes sense in a Editor but not for reading / writing in Java.. </li>
     <li>Package is still being adapted for use as a Java package, More work is needed. </li>
    </ul> </li>
   <li>A way of tailoring the conversion of Cobol Copybooks is needed. </li>
   <li>Currently the System has 2 ways of representing a Record-Layout 
    <table cellpadding="3"> 
     <tbody>
      <tr valign="TOP">
       <td><p><b>LayoutDetail</b></p></td>
       <td><p>The original format it holds only the bare minimum needed by the package </p></td>
      </tr>
      <tr valign="TOP">
       <td><p><b>ExternalRecord</b></p></td>
       <td><p>A general purpose interchange format. This was introduced latter on to support conversion between file formats / Database (RecordEditor). </p></td>
      </tr>
     </tbody>
    </table> Having two classes representing the same thing is not ideal. I may provide greater support for ExternalRecord's through out the package. There is a class <b>ToLayoutDetail</b> that can convert one to the other </li>
  </ol> 
  <p> </p>
  <p></p>
  <h3><a name="Header_5">To Do</a></h3> 
  <ol compact type="1"> 
   <li>Support for more files Structures </li>
   <li>Support for Shorter Type-Names. </li>
   <li>Greater Flexibility in importing Cobol Copybooks </li>
   <li>Support for more Cobol dialects. </li>
   <li>More work on the Documentation. </li>
   <li>More options for parsing XML files (would be useful to convert child elements to fields). </li>
   <li>One possible change is to have the <b>Open</b> methods return the object. This would allow <pre>        AbstractLineReader reader = new XmlLineReader(true).open(filename);
    or
        AbstractLineReader reader = LineIOProvider.getInstance()
				.getLineReader(Constants.IO_FIXED_LENGTH).open(filename);                                                  
</pre>any thoughts anybody. </li>
   <li>Bettetr class naming standards. </li>
  </ol>
