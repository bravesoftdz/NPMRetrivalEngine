        <h1 itemprop="name headline" class="content-stripe__title">Using Java to Print PDF Documents</h1> 
       </div> 
      </div> 
     </div> 
    </div> 
    <div class="content-stripe__bg content-stripe__bg--desktop-opaque"></div> 
   </div> 
   <div class="page-content"> 
    <div class="grid"> 
     <div class="col-tablet-2-3"> 
      <div class="page-content__copy" itemprop="articleBody">  
       <p> Printing has been possible since the beginnings of Java. It's not the most elegant aspect of the language, which reflects it's evolution over the years from a single <code>Toolkit</code> method in Java 1.0 to the <code>PrinterJob</code> of Java 1.2 and the current <code>PrintService</code> approach in 1.4. Thankfully the API is now fairly stable, although fixes were made in Java 5 and 6; printing is one area where a newer version of Java does help. </p> 
       <p> The packages for printing are the <a href="http://docs.oracle.com/javase/8/docs/api/index.html?java/awt/print/package-summary.html">java.awt.print</a> and <a href="http://docs.oracle.com/javase/8/docs/api/index.html?javax/print/package-summary.html">javax.print</a> packages - the latter dates from Java 1.4 and supplements, rather than replaces, the original API. To print a PDF from Java the process broadly works as follows: </p> 
       <ol> 
        <li>Search for and select a <i>PrintService</i> to print to </li> 
        <li>Create a <i>PrinterJob</i> and assign it to this service </li> 
        <li>Assign the PDF and the attributes for this print job, then print</li> 
       </ol> 
       <p> Of these steps, the only PDF-specific one is the last, so in practice printing a PDF in Java is like printing any other type of document, but we'll go through all the steps in detail. All the code below assumes you've imported <code>java.awt.print</code>, <code>javax.print</code> and its sub-packages. </p> 
       <h3>Selecting a PrintService</h3> 
       <p> In order to chose a printer you must first know what you're printing, which means choosing a <a href="http://docs.oracle.com/javase/8/docs/api/index.html?javax/print/DocFlavor.html">DocFlavor</a>. In the case of a PDF it's a <a href="http://docs.oracle.com/javase/8/docs/api/javax/print/DocFlavor.SERVICE_FORMATTED.html#PAGEABLE">PAGEABLE</a> object, which means that the contents are a (potentially) multi-page object which will be formatted by the print API. </p> 
       <p> You also get to specify a set of <a href="http://docs.oracle.com/javase/8/docs/api/index.html?javax/print/attribute/PrintRequestAttribute.html">PrintRequestAttributes</a> to qualify your search for a printer - for instance, you may need a printer than can print double-sided or that can print to A3, and there are attributes for each of these. Here's what the code would look like if we were looking for a printer that can print double-sided: </p> 
       <pre class="brush:java">
DocFlavor flavor = DocFlavor.SERVICE_FORMATTED.PAGEABLE;
PrintRequestAttributeSet patts = new HashPrintRequestAttributeSet();
patts.add(Sides.DUPLEX);
PrintService[] ps = PrintServiceLookup.lookupPrintServices(flavor, patts);
if (ps.length == 0) {
    throw new IllegalStateException("No Printer found");
}
PrinterJob job = PrinterJob.getPrinterJob();
job.setPrintService(ps[0]);
</pre> 
       <h3>Spooling the print job</h3> 
       <p> Now we need to specify the document and the job attributes. Just like the <code>PrintRequestAttribute</code> helps to specify the type of printer you're looking for, the <a href="http://docs.oracle.com/javase/8/docs/api/index.html?javax/print/attribute/DocAttribute.html">DocAttribute</a> specifies details about the print job itself - the name of the the printjob, whether it's double-sided, which pages to print and so on. </p> 
       <p> How you specify the document depends on what sort of "flavor" you specified. We chose the <code>PAGEABLE</code> flavor so we need to pass in a <a href="http://docs.oracle.com/javase/8/docs/api/index.html?java/awt/print/Pageable.html">Pageable</a> object - this is the <a href="/products/pdf/docs/api/index.html?org/faceless/pdf2/PDFParser.html">PDFParser</a> object, which implements Pageable. Here's how to print to our double-sided printer: </p> 
       <pre class="brush:java">
DocumentAttributeSet datts = new HashDocumentAttributeSet();
datts.add(Sides.DUPLEX);
PDFParser parser = new PDFParser(pdf);
job.setPageable(parser);
job.print(datts);
</pre> 
       <p> Nine times out of ten that's all you'll need to know, but if your print job doesn't run as expected you'll need to peek behind the curtain to know why. </p> 
       <h3>The gory details</h3> 
       <p> The first thing that happens is our <i>Pageable</i> object is asked for a sequence of <a href="http://docs.oracle.com/javase/8/docs/api/index.html?java/awt/print/Printable.html">Printable</a> objects, which represent a single page. <i>Printable</i> has a single <code>print</code> method which is given a Graphics2D object implementing <a href="http://docs.oracle.com/javase/8/docs/api/index.html?java/awt/print/PrinterGraphics.html">PrinterGraphics</a>, and our PDF API simply "draws" the PDF to that Graphics object, as if we were rendering to the screen. This Graphics object will convert our graphics operations to a format suitable for printing, and in practice that means PostScript. </p> 
       <h3>PostScript</h3> 
       <p> PostScript is the grandaddy of PDF, and if you're reading about printing you've already heard of it. It can represent text, vector graphics and images, so if your PDF contains a vector shape it's going to be reproduced almost exactly in PostScript. Other graphics operations (like transparency) can't be done in PostScript, and the Java print layer handles this by rasterizing the page to a bitmap and including that bitmap in the spool file. </p> 
       <p> The PrintGraphics doesn't know in advance if a page has transparency, so printing in Java means potentially painting the page twice - a best effort first, then falling back to a bitmap if that's not possible (actually the page image is often printed in "stripes", so the <code>print</code> method may be called several times). </p> 
       <p> However the main consequence of rasterization is that a 600dpi image of a page is a lot bigger than the PDF page itself, even before you get into PostScript's tendency to drastically overinflate. It's not uncommon to see spool files of several MB per page, so avoiding this if possible is a good idea. </p> 
       <p> If you know in advance that your page is going to be rasterized you can avoid the page being painted twice by setting the "bfo.printasimage" option on the PDF before printing: </p>
       <pre class="brush:java">
pdf.setOption("bfo.printasimage", "300");
</pre> will rasterize the PDF as a 300-dpi bitmap before printing (you can also set the System property 
       <code>sun.java2d.print.pipeline</code> to 
       <code>raster</code>). This is also useful when the print layer gives incorrect results, or when you want to print the image at a lower DPI than your printer is capable of. 
       <p></p> 
       <h3>What to avoid for better PostScript</h3> 
       <ul class="spacedlist"> 
        <li><b>Compositing</b>,or more specifically any sort of transparency (including image transparency) or any composites other than <a href="http://docs.oracle.com/javase/8/docs/api/java/awt/AlphaComposite.html#SRC_OVER">SRC_OVER</a>. In PDF, composites are called "Blend Modes" and you'll see them used when blending one image over another, or even when printing a page containing a "highlight" annotation (which is done with a Multiply composite). Any of these will cause the page to be rasterized </li> 
        <li><b>Patterns</b>. Tiling, shading, anything other than a solid opaque <code>java.awt.Color</code> will cause the page to be rasterized </li> 
        <li><b>Bitmap fonts</b> - sadly, we still see these quite a bit in documents created by older versions of GhostScript. Bitmap fonts have one image per glyph which acts as a mask. The mask is done with transparency, and this will cause the page to be rasterized </li> 
        <li><b>Embedded Fonts</b> - Java has some deep-seated issues with fonts, which are used in a very different way inside a PDF. Many fonts that are valid in PDF will throw exceptions if parsed by the AWT, so the PDF API has it's own font parsers for TrueType, Type 1 and Type 2 (compact) fonts. However, due to other assumptions within the AWT we can't just pass these fonts in to the Graphics2D for rendering: we have to draw each glyph as a <a href="http://docs.oracle.com/javase/8/docs/api/index.html?java/awt/Shape.html">Shape</a>. In the AWT this largely boils down to the same operation, but for PostScript it means that each letter in the font has to be drawn as a path. This will lead to a much larger file, although usually it's not as bad as rasterizing. We're edging towards another approach to embedded fonts, but at the time of writing this is still the case. </li> 
        <li><b>Colors outside the sRGB gamut.</b> Not nearly as awkward as the above points, but even if your PDF contains CMYK colors and your PostScript is destined for a CMYK printer, the colors are still going to be converted to RGB by the Java print layer, then back again by the printer. Don't expect photo-realistic printing with Java. </li> 
       </ul> 
       <h3>Scaling</h3> 
       <p> Barring the issues above, printing in recently versions of Java is good enough for most documents. However there are a few other tips we can offer: The most common printing questions we get are about scaling (and not just when printing from Java). </p> 
       <p> PDF is a page language and a page in PDF should represent a page on the printer. If they're the same size that's usually OK (provided your printer driver isn't trying to add margins, of course), but if the sizes differ there are several options. </p>
       <ul> 
        <li>If the output page is too small the image can be clipped or scaled down to fit.</li> 
        <li>If the output page is bigger than the PDF page, the image can be scaled up or just centered in the page. </li> 
       </ul> 
       <p></p> 
       <p> You can control this with the <code>print.scaling</code> PDF option added in Acrobat 8: set this option to <code>None</code> to prevent <i>all</i> page scaling. If you're printing to Java with our API you can choose from a few other options as well: </p>
       <ul> 
        <li><b>Fit</b> - scale the page up or down to fit the printable area, preserving the aspect ratio </li> 
        <li><b>FitUnlocked</b> - as for "Fit" but don't preserve the aspect ratio </li> 
        <li><b>ShrinkToFit</b> - like Fit, but will only ever scale the page down (our default if not specified) </li> 
        <li><b>ShrinkToFitUnlocked</b> - like ShrinkToFit, but don't preserve the aspect ratio. </li> 
       </ul> For examples, if you want to fit a Letter page exactly to A4 and don't mind losing the aspect ratio of the text, do this before spooling the file with Java 
       <pre class="brush:java">
pdf.setOption("print.scaling", "FitUnlocked");
</pre> 
       <p></p> 
       <h3>Duplex and other print options</h3> 
       <p> You can specify other options in the PDF - see our <a href="/products/pdf/docs/api/org/faceless/pdf2/PDF.html#setOption(java.lang.String,%20java.lang.Object)">setOption</a> method. However other than scaling as described above these will only set the defaults in the print dialog of our viewer. If you're printing from your own code and want to honours these options, you'll need to turn them into appropriate <a href="http://docs.oracle.com/javase/8/docs/api/index.html?javax/print/attribute/PrintRequestAttribute.html">PrintRequestAttributes</a> or <a href="http://docs.oracle.com/javase/8/docs/api/index.html?javax/print/attribute/DocAttribute.html">DocAttributes</a> yourself. </p>
