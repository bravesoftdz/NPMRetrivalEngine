      <p>I built my first mobile application in 2012 using the <a href="https://www.joshmorony.com/beginners-guide-to-sencha-touch/">Sencha Touch</a> framework and <a title="Freebie! Learn to Use PhoneGap Build with This eBook" href="http://www.joshmorony.com/freebie-learn-to-use-phonegap-build-with-this-ebook/">PhoneGap Build</a>. Since then mobile applications have become my full time focus, and I have experimented with a few different frameworks like <a title="Getting Started with the Famo.us Mobile Framework" href="http://www.joshmorony.com/getting-started-with-the-famo-us-mobile-framework/">Famo.us</a>, <a title="Creating HTML5 Mobile Games with Phaser and PhoneGap" href="http://www.joshmorony.com/creating-html5-mobile-games-with-phaser-and-phonegap/">Phaser</a> and more recently <a title="Learning the Ionic Framework as a Sencha Touch Developer: Part 1" href="http://www.joshmorony.com/learning-the-ionic-framework-as-a-sencha-touch-developer-part-1/">Ionic</a>.</p>
      <script>AmpedSense.OptimizeAdSpot('PL'); AmpedSense.OptimizeAdSpot('PC'); AmpedSense.OptimizeAdSpot('PR');</script> 
      <p>I’ve been building <strong>HTML5 mobile applications</strong> for around three years now, and am confident in the skills I have and the tools I use, to build mobile applications that look and <strong>perform just as well as native applications</strong>. Back in 2012 HTML5 mobile frameworks were already quite powerful, but they just keep getting better and better.</p> 
      <p>There were <em>a lot</em> of obstacles and roadblocks I hit along the way though. This was partly due to the fact that HTML5 was my first&nbsp;attempt at&nbsp;mobile app development (I had only ever done a small amount of native Android development previously), but mostly because there’s a lot more twists and turns for HTML5 mobile developers than there is for native developers.</p> 
      <p>If you have a web background, it may be easier to learn a HTML5 mobile framework than pick up a native language like Objective C, but you have to learn how to deal with the complications that using HTML5 presents like:</p> 
      <ul> 
       <li>Accessing Native APIs (use PhoneGap or PhoneGap Build)</li> 
       <li>Managing the DOM (a bloated DOM can lead to poor performance)</li> 
       <li>Giving the UI a native feel (this is mostly handled by the framework)</li> 
       <li>Creating App Store ready builds (again, use PhoneGap or PhoneGap Build)</li> 
       <li>Signing Applications (if you don’t have a Mac <a title="How to Create an iOS Provisioning Profile and P12 with Windows" href="http://www.joshmorony.com/how-to-create-an-ios-provisioning-profile-and-p12-with-windows/">you can use OpenSSL</a>)</li> 
      </ul> 
      <p>All of these things are easy enough to overcome once you know how to do it and have had a bit of practice, but discovering how to do everything is a little more difficult than doing it the “normal” native way.</p> 
      <p>There’s quite a bit I’ve learned, discovered or surmised over the time that I have been developing <a title="What’s the Difference between Native, Hybrid and Web Mobile App Development?" href="http://www.joshmorony.com/whats-the-difference-between-native-hybrid-and-web-mobile-app-development/">hybrid mobile applications</a>. These are the top 7&nbsp;lessons&nbsp;that I think someone getting started with HTML5 mobile development should know.</p> 
      <h3>1. HTML5 is NOT&nbsp;better than&nbsp;native, but it rarely matters</h3> 
      <p>Despite what I said above about HTML5 mobile applications performing just as well as native mobile applications, high end <strong>native applications still have the potential to be better</strong>. The <a href="http://en.wikipedia.org/wiki/Pareto_principle">pareto principle</a>, or the “80/20” rule, applies really well here. With HTML5 you can create an app that feels and performs <strong>about 80% as well as a well designed native app</strong>. If you want, or the clients wants, that last 20% – the beautifully smooth and magical experience – it’s probably going to cost 80% more.</p> 
      <p>HTML5 does a great job across most platforms which drastically <strong>reduces cost by only having to develop a single version of the application</strong>, but to get a <em>perfect</em>&nbsp;app, an investment into developing native applications for each supported platform will be required.</p> 
      <p>I’ve turned down clients who seemed to want a Facebook level experience within their application, and told them that if they want that level of polish then HTML5 is not the answer. Often though these are the same people that don’t realise the level of investment required to create an application like that. HTML5 can do some amazing things and <a href="http://www.sencha.com/blog/the-making-of-fastbook-an-html5-love-story/">Fastbook by Sencha</a> is a great example of this, but the engineering required to get that level of quality would be out of reach for most developers.</p> 
      <p>Some people have the impression that because there’s a general consensus that native is better in terms of performance, that a HTML5 mobile app will be sub standard or poor performing; “if you want a <em>real app</em> then you should go native”. The fact is though that in the majority of cases you wouldn’t even be able to tell the difference, only when it comes to these high end apps where companies spend tens of thousands of dollars or more creating the perfect experience. A well designed HTML5 mobile application can outperform an average native application, and a lot more apps are using web tech these days than you probably realise (<a href="https://www.netflix.com">Netflix</a> being one of the bigger “out and proud” HTML5 users).</p> 
      <p>When dealing with clients though, by far the most common thing I’ve noticed is: <em>they do not care if it’s HTML5 or native</em>. In fact, despite explanation, they rarely ever understand the difference. Deliver what they want and they will be happy no matter what you make it with.</p>

      <h3>2. It is possible to create iOS applications without a Mac</h3> 
      <p>It is an extremely common misconception that you need a Mac to develop iOS applications, even if you’re using HTML5. <strong>This is not true</strong>.</p> 
      <p>You couldn’t be blamed for thinking it though, most tools will tell you that you need the iOS SDK only available on Macs to build for iOS. Take a look at what happens when I try to add iOS as a platform from a windows computer for example:</p> 
      <div id="attachment_1124" style="width: 618px" class="wp-caption aligncenter">
       <a href="https://www.joshmorony.com/wp-content/uploads/2015/04/7warning.jpg"><img class="size-full wp-image-1124" src="https://www.joshmorony.com/wp-content/uploads/2015/04/7warning.jpg" alt="iOS Platform Warning" width="608" height="28" srcset="https://www.joshmorony.com/wp-content/uploads/2015/04/7warning.jpg 608w, https://www.joshmorony.com/wp-content/uploads/2015/04/7warning-300x14.jpg 300w" sizes="(max-width: 608px) 100vw, 608px"></a>
       <p class="wp-caption-text">WARNING: Applications for platform ios can not be built on this OS</p>
      </div> 
      <p>After seeing this message, one might reasonably conclude that they need a Mac if they want to build iOS applications. This is partly true: if you want to build the application on your machine you <em>will</em> require a Mac, but there are ways around it.</p> 
      <p>By using the <a href="http://build.phonegap.com/">PhoneGap Build</a> cloud service to build the application (this avoids the requirement of having the SDK as the application is built on their end, not yours) and using <a title="How to Create an iOS Provisioning Profile and P12 with Windows" href="http://www.joshmorony.com/how-to-create-an-ios-provisioning-profile-and-p12-with-windows/">OpenSSL to create a personal information file</a>&nbsp;necessary for signing your application (which otherwise would require a Mac to create) you can easily create iOS applications on a Windows machine as well.</p> 
      <h3>3. You need to use a HTML5 framework, and you need to choose a good one</h3> 
      <p>As a caveat to the first point, if you’re not using a decent HTML5 mobile framework then <strong>your app will be nowhere near as good as native</strong>. You should never attempt to build a mobile application without using a UI framework like Sencha Touch or Ionic, except in very specific circumstances.</p> 
      <div class="bonus-box">
       Want to learn how to build apps or mobile games with two powerful HTML5 frameworks? Take a look at my 
       <a href="https://joshmorony.com/mobile-development-for-web-developers/">Mobile Development for Web Developers Course.</a>
      </div> 
      <p>HTML5 mobile frameworks have been engineered to handle all the unique challenges of <strong>creating native like behaviour with JavaScript</strong>. This includes things like screen transitions, scrolling lists with acceleration and deceleration, viewport bouncing and so on. The frameworks also <strong>improve performance</strong> by doing things like recycling DOM elements. It would require a monstrous effort to get anything that performs anywhere near as well as a native application without using one of these frameworks.</p> 
      <p>Even still though, some frameworks are just not as good as others and may not perform that well. There’s so many available today, and of course some are good and some are bad. I can tell you that from my experience <strong>Sencha Touch</strong> and <strong>Ionic</strong> are both well performing frameworks and you won’t go wrong with them. There’s still plenty of frameworks that I have never even been able to take a look at though.</p> 
      <p>The amount of frameworks available is a double edged sword. It’s great that there’s options out there as it increases competition between the frameworks as they strive to be better than their competitors. At the same time though, it’s possible that a framework you have chosen to use may stop being actively developed, it could be overtaken by a competitor, something completely new could come out that blows it out of the water and so on.</p> 
      <p>As a technologist you’re probably used to keeping up with all the latest developments and constantly learning new skills. Native developers don’t have to worry about these things though (well, to a lesser extent at least), as the language the device uses never changes.</p>

      <h3>4. HTML5 can do EVERYTHING native can do</h3> 
      <p>Technically speaking, HTML5 by itself can only do what the browser can do. When compared to everything a native application can do <strong>that isn’t a lot</strong> (a browser is very sandboxed).</p> 
      <p>With <a href="http://phonegap.com/">PhoneGap</a> though, that doesn’t really matter. The browser can talk to PhoneGap and PhoneGap can talk to the device, so the browser has access to everything that a native application does. The process may be a little more convoluted, but it’s certainly possible – and since there’s PhoneGap plugins available for just about everything you’ll ever need to do it’s pretty easy to implement native functionality.</p> 
      <p>If you want to learn more about PhoneGap and PhoneGap Build you can <a href="#ck_modal" rel="ck_modal">download my PhoneGap Build Quickstart Guide for free</a>.</p> 
      <h3>5. There are elitist native developers and defensive HTML5 developers</h3> 
      <p>Some native mobile application developers portray an attitude that “<em>real developers</em>” code native mobile apps. Some HTML5 mobile app developers get defensive of their chosen tech and also go on the attack. This creates an extremely toxic environment that’s completely unnecessary.</p> 
      <p>I think a lot of the animosity boils down to the fact that developers on either side of the argument are scared that the time they have invested into their tech is going to become redundant. There’s benefits to using both in different circumstances, so to outright dismiss either option is not reasonable.</p> 
      <p>As developers, we often pride ourselves on our knowledge. It is easy to associate your self worth and identity with the skills you possess. Criticism of your chosen tech can often feel like a personal attack.</p> 
      <p>If you’re getting into mobile application development – or anything really – always keep learning and approach everything with humility. Research the pros and cons of different options and don’t believe everything everyone says, as they often have vested interests.</p> 
      <h3>6. The DOM is the biggest determining factor for performance</h3> 
      <p>Javascript performance is getting better and better, <a title="What Does iOS8 Mean for HTML5 Mobile App Developers" href="http://www.joshmorony.com/what-does-ios8-mean-for-html5-mobile-app-developers/">Apples new WKWebView with the NitroJS engine touts an impressive 400% increase</a>, but <strong>that doesn’t mean HTML5 mobile applications will perform 4x better</strong>.</p> 
      <p>The biggest limiter of performance for HTML5 mobile applications are DOM operations. The DOM, for those uninitiated, is the <a href="http://www.w3schools.com/js/js_htmldom.asp">Document Object Model</a> – it is essentially a tree of nodes that represents the structure of a web page.</p> 
      <p>As the DOM becomes more complex (more elements, more nesting) the tree structure of the DOM becomes more complex and takes longer to navigate and modify.</p> 
      <p>The frameworks you use will often have smart ways of reducing impact of manipulating the DOM (e.g most good frameworks recycle DOM elements in a scrolling list when they go off screen), but it’s still something you need to pay attention to. Whenever you’re building HTML5 mobile applications always keep this in mind and look up strategies for keeping the DOM as simple as possible.</p>
      <script>AmpedSense.OptimizeAdSpot('3L'); AmpedSense.OptimizeAdSpot('3C'); AmpedSense.OptimizeAdSpot('3R');</script> 
      <h3>7. Debugging is sometimes awesome, sometimes difficult</h3> 
      <p>The benefit of using web technology to build mobile applications is that you can test it right in your browser – no emulators or devices needed. You can debug in an environment that you’re probably already familiar with, something like Chrome Dev Tools.</p> 
      <p>Most of the time though, you will also be using something like PhoneGap. PhoneGap can not be tested through the browser since it accesses the native API’s of the device (obviously not available through your browser).</p> 
      <p>To test any PhoneGap functionality you will need to test on a device, and then things become a little more difficult. You can use something like <a title="Genuitec Release EPIC Debugger ‘GapDebug’ for PhoneGap Apps" href="http://www.joshmorony.com/genuitec-release-epic-debugger-gapdebug-for-phonegap-apps/">GapDebug</a> to provide you with debugging tools similar to the browser whilst your app is installed on a device. This makes it easy enough to test, but the annoying part is having to create a new build with PhoneGap and install it on your device to test any changes.</p> 
      <p><strong>NOTE:</strong> When testing with GapDebug, if your app isn’t working and there are no errors in the console, try this:</p> 
      <p>1. Open up the debugger for your app<br> 2. Wait for it to load<br> 3. Go to the ‘Resources’ section and click the refresh icon</p> 
      <p>This will ensure the debugger is active when your app is starting up, so you won’t miss any errors that occur right at the start. It’s very common for HTML5 mobile applications to get stuck at the start up screen when installed on a device (but not when run through the browser) – this is usually because some error is occurring on the device that you can’t see (usually related to a PhoneGap plugin).</p>

Meanwhile native development would require separate sources / developers / products for separate platforms, and created technical debt relating to those specific techs.</p> 
          <p>But the reason developers would be more inclined to use the front-end web tech stack they’re familiar with is also more involved than simply preferring HTML: there’s a notion that specific application development experience and code — for instance in the techniques involved in creating a performant scrolling grid, taking a pre-existing widget, or even forking existing web app code — could be valuable economies in what would otherwise be a task to build something from scratch using unfamiliar tech.</p>
