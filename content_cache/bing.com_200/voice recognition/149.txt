  <h2 id="introduction"><span class="secno">2 </span>Introduction</h2> 
  <p><em>This section is non-normative.</em> </p>
  <p>The HTML Speech Input API aims to provide web developers with features that are typically not available when using standard speech recognition software to replace keyboard input in a web browser. The API itself is agnostic of the underlying speech recognition implementation and can support both server based as well as embedded recognizers. The API is designed to enable both one-off speech input and continuous speech input requests. Speech recognition results are provided to the web page as a list of hypotheses along with other relevant information for each hypothesis. </p>
  <h3>Automatic actions at the end of a spoken input</h3> This API allows web applications to be notified at the completion of successful and failed speech input. For e.g. at the end of a successful speech input session, a web application could perform: 
  <ol> 
   <li>Safe and idempotent actions, e.g. search or web site navigation. The actions have no side effects, and the cost of correcting the action is very low, e.g. by editing the query on the search results page or going back in the browser history.</li> 
   <li>Undoable actions, e.g. archiving e-mail or editing a document. It is ok to take action immediately, since actions can be easily undone, e.g. by an undo option in the app.</li> 
   <li>Final, time critical, but not dangerous actions, e.g. game inputs. These actions cannot be undone, but ease or speed of input is more important than correctness.</li> 
   <li>Final and dangerous actions, e.g. composing and sending a message (e-mail, SMS, IM, etc). Actions cannot be undone, so the user must be able to verify the action before it is taken.</li> 
  </ol> Types 1, 2 &amp; 3 work best when the web page can take action immediately at the end of speech input. This requires some facility other than a purely transparent speech input method that simulates keyboard input, since web pages typically do not want to take actions directly on text 'change' events. 
  <h3>Speech recognition grammars</h3> This API allows web applications to specify grammars that the speech recognizer should use when recognizing the user's speech. Specifying a grammar is useful for apps which have limited vocabulary, for e.g. commands, navigation within page, maps etc. Such applications would not work as well with free-form text input. The existing HTML attribute 
  <code>pattern</code> can be used to restrict the allowed inputs, but regular expressions are less expressive than context-free grammars. Also, SRGS grammars can include semantic annotations. 
  <h3>Application-specific handling of speech recognition hypotheses</h3> This API gives the web application access to more information than just the most likely recognized utterance. Some applications can provide a better user experience when they have access to the list of recognition hypotheses produced by the speech recognizer. For example: 
  <ul> 
   <li>A web search application can accept speech input, and perform a search immediately when the input is recognized. If it has access to the additional recognition hypothesis (aka N-best list), it can display that on the search results page and let the user chose the correct query if the input was misrecognized. For example, Google search might display search results for "recognize speech", and show a link with the text "Did you say 'wreck a nice beach'?".</li> 
   <li>An application may accept input that can only be validated programmatically (i.e. that can't be expressed with a regular expression or a context free grammar). Examples may include credit card numbers, user names etc. In order to check that a certain combination of digits form a valid credit card number one can apply a simple algorithm to the input. Given a set of recognition hypotheses, the app can eliminate the invalid inputs by applying the algorithm and looking at the result.</li> 
  </ul> 
  <h3>Examples</h3> 
  <p>The following code extracts illustrate how to use speech input in various cases: </p>
  <h4>Web search by voice</h4> 
  <div class="example"> 
   <pre>
    &lt;script type="text/javascript"&gt;
      function startSearch(event) {
        event.target.form.submit();
      }
    &lt;/script&gt;

    &lt;form action="http://www.google.com/search"&gt;
    &lt;input type="search" name="q" speech required onspeechchange="startSearch"&gt;
    &lt;/form&gt;
   </pre> Behavior 
   <ol> 
    <li>User clicks speech input element.</li> 
    <li>Speech input element shows that it's active, starts capturing audio as user speaks.</li> 
    <li>Endpointer detects end of speech once user stops speaking.</li> 
    <li>Speech recognizer returns results and the <code>onspeechchange</code> event is dispatched.</li> 
    <li><code>startSearch()</code> is called by <code>onspeechchange</code> event handler on speech input element.</li> 
    <li>Search results are loaded.</li> 
   </ol> 
  </div>

      // These will most likely be set in a UI.

      var toLang = "es";

      function handleSpeechInput(event) {
        var text = event.target.value;
        var callback = function(result) {
          if (result.translation)
            speak(result.translation, toLang);

  <h2 id="scope"><span class="secno">3 </span>Scope</h2> 
  <p><em>This section is non-normative.</em> </p>
  <p>This specification is limited to extending existing HTML elements with attributes and methods for allowing speech input. Speech recognition grammars are specified using SRGS and semantic interpretation of hypotheses are specified using SISR. </p>
  <p>The scope of this specification does not include providing a new markup language of any kind. </p>
  <p>The scope of this specification does not include defining or interpreting any definition or interpretation of dialog management instructions </p>
  <p>The scope of this specification does not include interfacing with telephony systems of any kind. </p>
  <h2 id="security"><span class="secno">4 </span>Security and privacy considerations</h2> 
  <ol> 
   <li>User agents must only start speech input sessions with explicit, informed user consent. User consent can include, for example: 
    <ul> 
     <li>User click on a visible speech input element which has an obvious graphical representation showing that it will start speech input.</li> 
     <li>Accepting a permission prompt shown as the result of a call to <code>startSpeechInput</code>.</li> 
     <li>Consent previously granted to always allow speech input for this web page.</li> 
    </ul> </li> 
   <li>User agents must give the user an obvious indication when audio is being recorded.</li> 
   <ul>

  <h3>Implementation considerations</h3> 
  <p><em>This section is non-normative.</em></p> 
  <ol> 
   <li>Spoken password inputs can be problematic from a security perspective, but it is up to the user to decide if they want to speak their password.</li> 
   <li>Speech input could potentially be used to eavesdrop on users. Malicious webpages could use tricks such as hiding the input element or otherwise making the user believe that it has stopped recording speech while continuing to do so. They could also potentially style the input element to appear as something else and trick the user into clicking them. An example of styling the file input element can be seen at <a href="http://www.quirksmode.org/dom/inputfile.html">http://www.quirksmode.org/dom/inputfile.html</a>. The above recommendations are intended to reduce this risk of such attacks.</li> 
  </ol> 
  <h2 id="api_description"><span class="secno">5 </span>API Description</h2> 
  <h3 id="extending_html_elements"><span class="secno">5.1 </span>Extending HTML elements</h3> The API adds a set of new attributes and methods to 
  <a href="http://dev.w3.org/html5/spec/the-input-element.html#htmlinputelement">HTMLInputElement</a> and 
  <a href="http://dev.w3.org/html5/spec/the-button-element.html#htmltextareaelement">HTMLTextAreaElement</a>. 
  <pre class="idl">
  interface HTMLInputElement : HTMLElement {
    ...

    // speech input attributes
    attribute boolean <a href="#attrib-speech">speech</a>;
    attribute DOMString <a href="#attrib-grammar">grammar</a>;
    attribute short <a href="#attrib-maxresults">maxresults</a>;
    attribute long <a href="#attrib-nospeechtimeout">nospeechtimeout</a>;

    // speech input <a href="http://dev.w3.org/html5/spec/webappapis.html#event-handler-idl-attributes">event handler IDL attributes</a>
    attribute <a href="http://dev.w3.org/html5/spec/webappapis.html#function">Function</a> <a href="#event-capture-start">oncapturestart</a>();
    attribute <a href="http://dev.w3.org/html5/spec/webappapis.html#function">Function</a> <a href="#event-speech-start">onspeechstart</a>();
    attribute <a href="http://dev.w3.org/html5/spec/webappapis.html#function">Function</a> <a href="#event-speech-change">onspeechchange</a>(in <a href="#speech-input-result-list">SpeechInputEvent</a> event);
    attribute <a href="http://dev.w3.org/html5/spec/webappapis.html#function">Function</a> <a href="#event-speech-end">onspeechend</a>();
    attribute <a href="http://dev.w3.org/html5/spec/webappapis.html#function">Function</a> <a href="#event-speech-error">onspeecherror</a>(in <a href="#speech-input-error">SpeechInputError</a> error);

    // speech input methods
    void <a href="#method-start-speech-input">startSpeechInput</a>();
    void <a href="#method-stop-speech-input">stopSpeechInput</a>();
    void <a href="#method-cancel-speech-input">cancelSpeechInput</a>();
  };
  </pre> 
  <p>The <dfn id="attrib-speech"><code>speech</code></dfn> attribute indicates to the user agent that it should accept speech input for that element. This attribute is applicable for HTMLInputElement in the following states: </p>
  <ul> 
   <li>Text</li> 
   <li>Search</li> 
   <li>URL</li> 
   <li>Telephone</li> 
   <li>E-mail</li> 
   <li>Password</li> 
  </ul> 
  <p>The <dfn id="attrib-grammar"><code>grammar</code></dfn> attribute gives the address of an external application-specific grammar, e.g. "http://example.com/grammars/pizza-order.grxml". The attribute, if present, must contain a <a href="http://dev.w3.org/html5/spec/urls.html#valid-non-empty-url-potentially-surrounded-by-spaces">valid non-empty URL potentially surrounded by spaces</a>. The implementation should use the grammar to guide the speech recognizer. Implementations must support <a href="http://www.w3.org/TR/speech-grammar/">SRGS grammars</a> and <a href="http://www.w3.org/TR/semantic-interpretation/">SISR</a> annotations. </p>
  <p>The <dfn id="attrib-maxresults"><code>maxresults</code></dfn> attribute specifies the maximum number of items that the implementation should place in the <code>results</code> sequence. If not set, the maximum number of elements in the results sequence is implementation dependent. </p>
  <p>The <dfn id="attrib-nospeechtimeout"><code>nospeechtimeout</code></dfn> attribute specifies the time in milliseconds to wait for start of speech, after which the audio capture times out. If not set, the timeout is implementation dependent. </p>
  <p>The <dfn id="event-capture-start"><code>capturestart</code></dfn> event is dispatched when audio capture starts for the active speech input session. </p>
  <p>The <dfn id="event-speech-start"><code>speechstart</code></dfn> event is dispatched when the active speech input session detects that the user has started speaking. </p>
  <p>The <dfn id="event-speech-change"><code>speechchange</code></dfn> event is dispatched when a set of complete and valid utterances have been recognized. This event bubbles. A complete utterance ends when the implementation detects end-of-speech or if the <code>stopSpeechInput()</code> method was invoked. </p>
  <p>Some implementations may dispatch the <code>change</code> event for elements when their value changes. When the new value was obtained as the result of a speech input session, such implementations must dispatch the <code>speechchange</code> event prior to the <code>change</code> event. </p>
  <p>The <dfn id="event-speech-end"><code>speechend</code></dfn> event is dispatched when the active speech input session detects that the user has stopped speaking. </p>
  <p>The <dfn id="event-speech-error"><code>speecherror</code></dfn> event is dispatched when the active speech input session resulted in an error. </p>
  <p>When the <dfn id="method-start-speech-input"><code>startSpeechInput()</code></dfn> method is invoked, a new speech input session is started. If there was already an active speech input session, that session is cancelled, as if <code>cancelSpeechInput</code> had been called. If the user has not already given consent to this web page to accept speech input, implementations must not start the speech input session until the user has given consent. </p>
  <p>When the <dfn id="method-stop-speech-input"><code>stopSpeechInput()</code></dfn> method is invoked, if there was an active speech input session and this element had initiated it, the user agent must gracefully stop the session as if end-of-speech was detected. The user agent must perform speech recognition on audio that has already been recorded and the relevant events must be fired if necessary. If there was no active speech input session, if this element did not initiate the active speech input session or if end-of-speech was already detected, this method must return without doing anything. </p>
  <p>When the <dfn id="method-cancel-speech-input"><code>cancelSpeechInput()</code></dfn> method is invoked, if there was an active speech input session and this element had initiated it, the user agent must abort the session, discard any pending/buffered audio data and fire no events for the pending data. If there was no active speech input session or if this element did not initiate it, this method must return without doing anything. </p>
  <h3 id="speech_input_event_interface"><span class="secno">5.2 </span>SpeechInputEvent interface</h3> 
  <pre class="idl">
  [NoInterfaceObject]
  interface <dfn id="speech-input-event">SpeechInputEvent</dfn> {
    readonly attribute <a href="#speech-input-result-list">SpeechInputResultList</a> <a href="#attrib-event-results">results</a>;
    void <a href="#method-results-feedback">feedback</a>(DOMString correctUtterance);
  };
  </pre> 
  <p>The <a href="#attrib-event-results"><code>results</code></a> attribute returns a <a href="#speech-input-result-list"><code>SpeechInputResultList</code></a> object containing all the recognition results. </p>
  <p>The <dfn id="method-results-feedback"><code>feedback(correctUtterance)</code></dfn> method is used to give feedback on the speech recognition results. Implementations may use this to improve the accuracy of future speech recognition requests. </p>
  <ul> 
   <li>Setting <code>correctUtterance</code> to null indicates that none of the items in <code>results</code> were a satisfactory match to the original speech.</li> 
   <li>Setting <code>correctUtterance</code> to the utterance of an item in <code>results</code> indicates that the item is the most satisfactory match to the original speech.</li> 
   <li>Setting <code>correctUtterance</code> to a value not matching any of the utterance values in <code>results</code> indicates that the given value is the expected match to the original speech.</li> 
  </ul> 
  <h3 id="speech_input_result_list_interface"><span class="secno">5.3 </span>SpeechInputResultList interface</h3> 
  <pre class="idl">
  [NoInterfaceObject]
  interface <dfn id="speech-input-result-list">SpeechInputResultList</dfn> {
    readonly attribute unsigned long <a href="#attrib-results-length">length</a>;
    <a href="#speech-input-result">SpeechInputResult</a> <a href="#method-results-item">item</a>(in unsigned long index);
  };
  </pre> 
  <p>The <dfn id="attrib-results-length"><code>length</code></dfn> attribute must return the number of results represented by the list. </p>
  <p>The <dfn id="method-results-item"><code>item(index)</code></dfn> method must return the indexth result in the list. If there is no indexth result in the list, then the method must return null. </p>
  <h3 id="speech_input_result_interface"><span class="secno">5.4 </span>SpeechInputResult interface</h3> In ECMAScript, 
  <a href="#speech-input-results"><code>SpeechInputResult</code></a> objects are represented as regular native objects with optional properties named 
  <code>utterance</code>, 
  <code>confidence</code> and 
  <code>interpretation</code>. 
  <pre class="idl">
  [NoInterfaceObject]
  interface <dfn id="speech-input-result">SpeechInputResult</dfn> {
    readonly attribute DOMString <a href="#attrib-utterance">utterance</a>;
    readonly attribute float <a href="#attrib-confidence">confidence</a>;
    readonly attribute object <a href="#attrib-interpretation">interpretation</a>;
  };
  </pre> 
  <p>The <dfn id="attrib-utterance"><code>utterance</code></dfn> attribute must, on getting, return the text of recognized speech. </p>
  <p>The <dfn id="attrib-confidence"><code>confidence</code></dfn> attribute must, on getting, return a value in the inclusive range [0.0, 1.0] indicating the quality of the match. The higher the value, the more confident the recognizer is that this matches what the user spoke. </p>
  <p>The <dfn id="attrib-interpretation"><code>interpretation</code></dfn> attribute must, on getting, return the result of semantic interpretation of the recognized speech, using semantic annotations in the grammar. If no grammar was specified or if the grammar contained no semantic annotations for the utterance, this value should be the same as utterance. </p>
  <h3 id="speech_input_error_interface"><span class="secno">5.5 </span>SpeechInputError interface</h3> 
  <pre class="idl">
  [NoInterfaceObject]
  interface <dfn id="speech-input-error">SpeechInputError</dfn> {
    const unsigned short <a href="#error-aborted">ABORTED</a> = 1;
    const unsigned short <a href="#error-audio">AUDIO</a> = 2;
    const unsigned short <a href="#error-network">NETWORK</a> = 3;
    const unsigned short <a href="#error-no-speech">NO_SPEECH</a> = 4;
    const unsigned short <a href="#error-no-match">NO_MATCH</a> = 5;
    const unsigned short <a href="#error-bad-grammar">BAD_GRAMMAR</a> = 6;
    const unsigned short <a href="#error-permission-denied">PERMISSION_DENIED</a> = 7;
    const unsigned short <a href="#error-unsupported-language">UNSUPPORTED_LANGUAGE</a> = 8;
    readonly attribute unsigned short <a href="#attrib-error-code">code</a>;
    readonly attribute DOMString <a href="#attrib-error-message">message</a>;
  };
  </pre> 
  <p>The <dfn id="attrib-error-code"><code>code</code></dfn> attribute must return the appropriate code from the following list: </p>
  <dl> 
   <dt>
    <dfn id="error-aborted"><code>ABORTED</code></dfn> (numeric value 1)
   </dt> 
   <dd>
    The user or a script aborted speech input.
   </dd> 
   <dt>
    <dfn id="error-audio"><code>AUDIO</code></dfn> (numeric value 2)
   </dt> 
   <dd>
    There was an error with recording audio.
   </dd> 
   <dt>
    <dfn id="error-network"><code>NETWORK</code></dfn> (numeric value 3)
   </dt> 
   <dd>
    There was a network error, for implementations that use server-side recognition.
   </dd> 
   <dt>
    <dfn id="error-no-speech"><code>NO_SPEECH</code></dfn> (numeric value 4)
   </dt> 
   <dd>
    No speech heard before 
    <code>nospeechtimeout</code>.
   </dd> 
   <dt>
    <dfn id="error-no-match"><code>NO_MATCH</code></dfn> (numeric value 5)
   </dt> 
   <dd>
    Speech was heard, but could not be interpreted in the specified language and language model.
   </dd> 
   <dt>
    <dfn id="error-bad-grammar"><code>BAD_GRAMMAR</code></dfn> (numeric value 6)
   </dt> 
   <dd>
    There was an error in the speech recognition grammar.
   </dd> 
   <dt>
    <dfn id="error-permission-denied"><code>PERMISSION_DENIED</code></dfn> (numeric value 7)
   </dt> 
   <dd>
    The user did not consent to starting speech input.
   </dd> 
  </dl> 
  <p>The <dfn id="attrib-error-message"><code>message</code></dfn> attribute must return an error message describing the details of the error encountered. This attribute is primarily intended for debugging and developers should not use it directly in their application user interface. </p>
  <h3 id="existing_attributes"><span class="secno">5.6 </span>Notes about existing attributes</h3> 
  <p>The <a href="http://dev.w3.org/html5/spec/elements.html#attr-lang"><code>lang</code></a> attribute, if present, sets the speech input language. If this attribute is not set the implementation must fall back to the language of the closest ancestor that has a <code>lang</code> attribute, and finally to the language of the document. </p>
  <p>The <a href="http://dev.w3.org/html5/spec/common-input-element-attributes.html#the-pattern-attribute"> <code>pattern</code></a> attribute, if present, should be used to guide the speech recognizer. If <code>grammar</code> was also specified, <code>grammer</code> takes precedence over <code>pattern</code>. </p>
  <p>For <a href="http://dev.w3.org/html5/spec/the-input-element.html#htmlinputelement">HTMLInputElement</a> the <code>value</code> attribute must be the utterance of the most likely recognition result after a successful recognition. This is equivalent to <code>results[0].utterance</code>. In the case of unsuccessful recognition <code>value</code> must remain unaffected. </p>
  <h3 id="single_multi_differences"><span class="secno">5.7 </span>Differences between single line and multi-line elements</h3> 
  <p></p>
  <ul> 
   <li>Successful speech input to <a href="http://dev.w3.org/html5/spec/the-input-element.html#htmlinputelement">HTMLInputElement</a> must set the <code>value</code> attribute to the utterance of most likely speech recognition hypothesis.</li> 
   <li>Successful speech input to <a href="http://dev.w3.org/html5/spec/the-button-element.html#htmltextareaelement">HTMLTextAreaElement</a> must insert the utterance of the most likely speech recognition hypothesis at the current caret position.</li> 
  </ul> 
  <h3 id="backwards_compatibility"><span class="secno">5.8 </span>Backwards Compatibility</h3> 
  <p>A DOM application can use the <code>hasFeature(feature, version)</code> method of the <code>DOMImplementation</code> interface with parameter values "SpeechInput" and "1.0" (respectively) to determine whether or not this module is supported by the implementation. </p>
  <p>Implementations that don't support speech input will ignore the additional attributes and events defined in this module and the HTML elements with these attributes will continue to work with other forms of input. </p>

  <h3 id="requirements_section"><span class="secno">7.2 </span>Requirements</h3> 
  <ul> 
   <li> <a href="#fpr-feat">3.1 Web Authoring Feature Requirements</a> 
    <ul> 
     <li> <a href="#fpr-feat-ss">3.1.1 Web Authoring Feature Speech System Requirements</a> <br><br>Yes 
      <ul> 
       <li><a href="#fpr40">FPR40. Web applications must be able to use barge-in (interrupting audio and TTS output when the user starts speaking).</a><br> (Requires use of Javascript to abort audio playback when <code>onspeechstart</code> event is received.)</li> 
      </ul>No
      <ul> 
       <li><a href="#fpr8">FPR8. User agent (browser) can refuse to use requested speech service.</a></li> 
       <li><a href="#fpr11">FPR11. If the web apps specify speech services, it should be possible to specify parameters.</a></li> 
       <li><a href="#fpr12">FPR12. Speech services that can be specified by web apps must include network speech services.</a></li> 
       <li><a href="#fpr31">FPR31. User agents and speech services may agree to use alternate protocols for communication.</a></li> 
       <li><a href="#fpr33">FPR33. There should be at least one mandatory-to-support codec that isn't encumbered with IP issues and has sufficient fidelity &amp; low bandwidth requirements.</a></li> 
       <li><a href="#fpr58">FPR58. Web application and speech services must have a means of binding session information to communications.</a></li> 
      </ul> </li> 
     <li> <a href="#fpr-feat-reco">3.1.2 Web Authoring Feature Recognition Requirements</a> <br><br>Yes
      <ul> 
       <li><a href="#fpr2">FPR2. Implementations must support the XML format of SRGS and must support SISR.</a></li> 
       <li><a href="#fpr19">FPR19. User-initiated speech input should be possible.</a></li> 
       <li><a href="#fpr21">FPR21. The web app should be notified that capture starts.</a></li> 
       <li><a href="#fpr22">FPR22. The web app should be notified that speech is considered to have started for the purposes of recognition.</a></li> 
       <li><a href="#fpr23">FPR23. The web app should be notified that speech is considered to have ended for the purposes of recognition.</a></li> 
       <li><a href="#fpr24">FPR24. The web app should be notified when recognition results are available.</a></li> 
       <li><a href="#fpr25">FPR25. Implementations should be allowed to start processing captured audio before the capture completes.</a></li> 
       <li><a href="#fpr26">FPR26. The API to do recognition should not introduce unneeded latency.</a></li> 
       <li><a href="#fpr34">FPR34. Web application must be able to specify domain specific custom grammars.</a></li> 
       <li><a href="#fpr35">FPR35. Web application must be notified when speech recognition errors or non-matches occur.</a></li> 
       <li><a href="#fpr42">FPR42. It should be possible for user agents to allow hands-free speech input.</a></li> 
       <li><a href="#fpr43">FPR43. User agents should not be required to allow hands-free speech input.</a></li> 
       <li><a href="#fpr47">FPR47. When speech input is used to provide input to a web app, it should be possible for the user to select alternative input methods.</a></li> 
       <li><a href="#fpr50">FPR50. Web applications must not be prevented from integrating input from multiple modalities.</a></li> 
       <li><a href="#fpr54">FPR54. Web apps should be able to customize all aspects of the user interface for speech recognition, except where such customizations conflict with security and privacy requirements in this document, or where they cause other security or privacy problems.</a></li> 
       <li><a href="#fpr59">FPR59. While capture is happening, there must be a way for the web application to abort the capture and recognition process.</a></li> 
      </ul>No
      <ul> 
       <li><a href="#fpr4">FPR4. It should be possible for the web application to get the recognition results in a standard format such as EMMA.</a></li> 
       <li><a href="#fpr27">FPR27. Speech recognition implementations should be allowed to add implementation specific information to speech recognition results.</a></li> 
       <li><a href="#fpr28">FPR28. Speech recognition implementations should be allowed to fire implementation specific events.</a></li> 
       <li><a href="#fpr48">FPR48. Web application author must be able to specify a domain specific statistical language model.</a></li>(This would be easy to support by allowing an SLM URL in the grammar attribute). 
       <li><a href="#fpr56">FPR56. Web applications must be able to request NL interpretation based only on text input (no audio sent).</a></li> 
       <li><a href="#fpr57">FPR57. Web applications must be able to request recognition based on previously sent audio.</a></li> 
      </ul> </li> 
     <li> <a href="#fpr-feat-syn">3.1.3 Web Authoring Feature Synthesis Requirements</a> <br><br>No
      <ul> 
       <li><a href="#fpr3">FPR3. Implementation must support SSML.</a></li> 
       <li><a href="#fpr29">FPR29. Speech synthesis implementations should be allowed to fire implementation specific events.</a></li> 
       <li><a href="#fpr41">FPR41. It should be easy to extend the standard without affecting existing speech applications.</a></li> 
       <li><a href="#fpr46">FPR46. Web apps should be able to specify which voice is used for TTS.</a></li> 
       <li><a href="#fpr51">FPR51. The web app should be notified when TTS playback starts.</a></li> 
       <li><a href="#fpr52">FPR52. The web app should be notified when TTS playback finishes.</a></li> 
       <li><a href="#fpr53">FPR53. The web app should be notified when the audio corresponding to a TTS &lt;mark&gt; element is played back.</a></li> 
       <li><a href="#fpr60">FPR60. Web application must be able to programatically abort tts output.</a></li> 
      </ul> </li> 
    </ul> </li> 
   <li> <a href="#fpr-conv">3.2 Web Authoring Convenience Requirements</a> 
    <ul> 
     <li> <a href="#fpr-conv-ss">3.2.1 Web Authoring Convenience Speech System Requirements</a> <br><br>No
      <ul> 
       <li><a href="#fpr7">FPR7. Web apps should be able to request speech service different from default.</a></li> 
       <li><a href="#fpr9">FPR9. If browser refuses to use the web application requested speech service, it must inform the web app.</a></li> 
       <li><a href="#fpr10">FPR10. If browser uses speech services other than the default one, it must inform the user which one(s) it is using.</a></li> 
       <li><a href="#fpr30">FPR30. Web applications must be allowed at least one form of communication with a particular speech service that is supported in all UAs.</a></li> 
      </ul> </li> 
     <li> <a href="#fpr-conv-reco">3.2.2 Web Authoring Convenience Recognition Requirements</a> <br><br>Yes
      <ul> 
       <li><a href="#fpr5">FPR5. It should be easy for the web appls to get access to the most common pieces of recognition results such as utterance, confidence, and nbests.</a></li> 
       <li><a href="#fpr6">FPR6. Browser must provide default speech resource.</a></li> 
       <li><a href="#fpr36">FPR36. User agents must provide a default interface to control speech recognition.</a></li> 
       <li><a href="#fpr38">FPR38. Web application must be able to specify language of recognition.</a></li> 
       <li><a href="#fpr39">FPR39. Web application must be able to be notified when the selected language is not available.</a></li> 
       <li><a href="#fpr44">FPR44. Recognition without specifying a grammar should be possible.</a></li> 
       <li><a href="#fpr45">FPR45. Applications should be able to specify the grammars (or lack thereof) separately for each recognition.</a></li> 
      </ul> </li> 
     <li> <a href="#fpr-conv-syn">3.2.3 Web Authoring Convenience Synthesis Requirements</a> <br><br>No
      <ul> 
       <li><a href="#fpr13">FPR13. It should be easy to assign recognition results to a single input field.</a></li> 
       <li><a href="#fpr14">FPR14. It should not be required to fill an input field every time there is a recognition result.</a></li> 
       <li><a href="#fpr15">FPR15. It should be possible to use recognition results to multiple input fields.</a></li> 
       <li><a href="#fpr61">FPR61. Aborting the TTS output should be efficient.</a></li> 
      </ul> </li> 
    </ul> </li> 
   <li> <a href="#fpr-sec">3.3 Security and Privacy Requirements</a> 
    <ul> 
     <li> <a href="#fpr-sec-ss">3.3.1 Security and Privacy Speech System Requirements</a> <br><br>Yes
      <ul> 
       <li><a href="#fpr16">FPR16. User consent should be informed consent.</a></li> 
       <li><a href="#fpr20">FPR20. The spec should not unnecessarily restrict the UA's choice in privacy policy.</a></li> 
      </ul>No
      <ul> 
       <li><a href="#fpr55">FPR55. Web application must be able to encrypt communications to remote speech service.</a></li> 
      </ul> </li> 
     <li> <a href="#fpr-sec-reco">3.3.2 Security and Privacy Recognition Requirements</a> <br><br>Yes
      <ul> 
       <li><a href="#fpr1">FPR1. Web applications must not capture audio without the user's consent.</a></li> 
       <li><a href="#fpr17">FPR17. While capture is happening, there must be an obvious way for the user to abort the capture and recognition process.</a></li> 
       <li><a href="#fpr18">FPR18. It must be possible for the user to revoke consent.</a></li> 
       <li><a href="#fpr49">FPR49. End users need a clear indication whenever microphone is listening to the user</a></li> 
      </ul>No
      <ul> 
       <li><a href="#fpr37">FPR37. Web application should be given captured audio access only after explicit consent from the user.</a></li> 
      </ul> </li> 
    </ul> </li>
