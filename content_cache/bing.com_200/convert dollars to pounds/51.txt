  <h2 id="DISCLAIMER">Disclaimer</h2> 
  <div class="important">
   <p>This essay does not describe an existing computer program, just one that <em>should</em> exist. This essay is about a suggested <a class="project" href="projects.html">student project</a> in <span class="writteninjava">Java</span> programming. This essay gives a rough overview of how it might work. I have <strong>no</strong> source, object, specifications, file layouts or anything else useful to implementing this project. Everything I have prepared to help you is right here.</p> 
   <p>This project outline is not like the artificial, tidy little problems you are spoon-fed in school, when all the facts you need are included, nothing extraneous is mentioned, the answer is fully specified, along with hints to nudge you toward a single expected canonical solution. This project is much more like the real world of messy problems where it is up to you to fully the define the end point, or a series of ever more difficult versions of this project and research the information yourself to solve them.</p> 
   <p>Everything I have to say to help you with this project is written below. I am <strong>not</strong> prepared to help you implement it; or give you any additional materials. I have too many other projects of my own.</p> 
   <p>Though I am a programmer by profession, I don’t do people’s <a class="jgloss" href="../jgloss/homework.html">homework</a> for them. That just robs them of an education.</p> 
   <p>You have my full permission to implement this project in any way you please and to keep all the profits from your endeavour.</p>
  </div> 
  <p>Please do not email me about this project without reading the disclaimer above.</p> 
  <!-- /generated by Include -->In many places throughout this website I mention prices in Canadian dollars. These are mostly ball park prices of a variety of things, typically not products I sell. I need to have a way so that people could easily convert that price to their local currency. Ideally the system would remember their preferred currency and automatically display all displayed currencies in that form. They would not need to key in the from-country to-country and amount as they would with a conventional currency converter, such as the one at 
  <a class="com" href="https://www.oanda.com/currency/converter/">Oanda</a>. 
  <p>It would also be sufficient to run some batch process once a week to update my 
   <!-- macro Acronym HTML -->
   <!-- generated --><a class="acronymdef" href="../jgloss/html.html">HTML</a> (<span class="means"><span class="ac">H</span>yper<span class="ac">t</span>ext <span class="ac">M</span>arkup <span class="ac">L</span>anguage</span>)
   <!-- /generated by Acronym --> and reupload it. I don’t need up to the second accurate exchange rates.</p> 
  <p>The programming is fairly simple. It could be done with 
   <!-- macro Acronym CGI_c -->
   <!-- generated --><a class="acronymdef" href="../jgloss/cgi.html">CGI</a> (<span class="means"><span class="ac">C</span>ommon <span class="ac">G</span>ateway <span class="ac">I</span>nterface</span>)
   <!-- /generated by Acronym -->, JavaScript or even a Java Applet. The most difficult part is political, getting access to a file of current exchange rates. You could extract it by simulating one of the free converters, or you might make special arrangements with a company such as Oanda to let you periodically download it to your own server using their now defunct FXCommerce service. See the <a class="applet" href="../applet/currcon.manual.html">CurrCon manual</a> for other sources.</p> 
  <p>It might be implemented much the way a <a class="jgloss" href="../jgloss/hitcounter.html">hit counter</a> is, using digit <span class="ext">*.gif</span>s or a numeric gif to display the value. You might use a cookie to track preferred currency. You might have a way to display/change that preferred currency either as a flag icon or as a country code and/or country name.</p> 
  <p>Oanda implemented a now defunct scheme very similar to what I have asked for called FXCommerce that requires JavaScript and cookies. A typical convertible price has to be encoded in your 
   <!-- macro Acronym HTML -->
   <!-- generated --><span class="acronym"><abbr title="Hypertext Markup Language">HTML</abbr></span>
   <!-- /generated by Acronym --> with JavaScript like this:</p> 
  <div class="javascript">
   &lt;script type=
   <span class="dq">text/javascript</span>&gt;document.write(OANDAconvert(45.95, 
   <span class="quoted">USD</span>));&lt;/script&gt;
  </div>They have recently added some code to hide the script to make this more difficult. 
  <p>This means the price is

With Oanda there is no default currency and no display of the current currency country, either in words or icon. In their scheme, typically you use two columns to display, one in US$ and one in local currency that starts out saying <span class="quoted">Convert This!</span> until the user selects a local currency, which wrecks the layout by being too long a string. In my scheme, you would need only one column.</p>

  <p>The interesting thing about is the way all currencies on a page will flip if you change even one. See the <a class="ggloss" href="../ggloss/laser.html">gay glossary</a> to see it in action.</p> 
  <p>Another way to handle this is to prepare your pages with a servlet using 
   <!-- macro Acronym JSP -->
   <!-- generated --><a class="acronymdef" href="../jgloss/jsp.html">JSP</a> (<span class="means"><span class="ac">J</span>ava <span class="ac">S</span>erver <span class="ac">P</span>ages</span>)
   <!-- /generated by Acronym -->, or displaying columns of currencies, recalculated daily. You can get a list of the day’s conversion factors by screensrapinig in from their FxDaily page.</p> 
  <p>You might try a similar but slightly simpler project first, a metric/English measure converter, so you can prepare your web pages in either metric or English units and have the viewer decide in which system they prefer to view the page. The conversions are more complex, but the conversion factors don’t change daily the way currency conversion factors do.</p> 
  <p>The main cleverness in the program is selecting a suitable unit for a particular dimension, e.g. is inches, feet or yards most appropriate for 59 cm?</p> 
  <p>The user could even invent her own personal set of measures, e.g. <em>hands</em> where hand is defined as the width of <em>her</em> hand, or <em>strides</em> where a stride is the length of <em>her</em> stride, or <em>fingers</em> for the volume displaced by <em>her index finger</em> up to the knuckle. This lets everyone satisfy Henry VIII-sized egos.</p> 
  <p>You could even invent a set of personal cooking units of measure, like the <em>pinch</em> and <em>sifterful</em>. You could compose recipes in your personal units and others could read them translated to whatever units they wanted. It would be trivial to add recipe doubling or halving to serve any number of people.</p>
