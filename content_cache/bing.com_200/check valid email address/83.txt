           <p>10 Feb 2010 - <em>As a followup to reader comments, the author made updates to the final paragraph under <a href="#validation">Validation</a> Only the schema is thread safe, validators and schema factories are not.</em></p>
           <p>Validation is a powerful tool. It enables you to quickly check that input is roughly in the form you expect and quickly reject any document that is too far away from what your process can handle. If there's a problem with the data, it's better to find out earlier than later.</p>
           <p>In the context of Extensible Markup Language (XML), validation normally involves writing a detailed specification for the document's contents in any of several schema languages such as the World Wide Web Consortium (W3C) XML Schema Language (XSD), RELAX NG, Document Type Definitions (DTDs), and Schematron. Sometimes validation is performed while parsing, sometimes immediately after. However, it's usually done before any further processing of the input takes place. (This description is painted with broad strokes -- there are exceptions.)</p>
           <p>Until recently, the exact Application Programming Interface (API) by which programs requested validation varied with the schema language and parser. DTDs and XSD were normally accessed as configuration options in Simple API for XML (SAX), Document Object Model (DOM), and Java API for XML Processing (JAXP). RELAX NG required a custom library and API. Schematron might use the Transformations API for XML(TrAX); and still other schema languages required programmers to learn still more APIs, even though they were performing essentially the same operation. </p>
           <p> Java 5 introduced the <code>javax.xml.validation</code> package to provide a schema-language-independent interface to validation services. This package is also available in Java 1.3 and later when you install JAXP 1.3 separately. Among other products, an implementation of this library is included with Xerces 2.8. </p>
           <h2 id="validation" class="ibm-h2">Validation</h2>
           <p> The <code>javax.xml.validation</code> API uses three classes to validate documents: <code>SchemaFactory</code>, <code>Schema</code>, and <code>Validator</code>. It also makes extensive use of the <code>javax.xml.transform.Source</code> interface from TrAX to represent the XML documents. In brief, a <code>SchemaFactory</code> reads the schema document (often an XML file) from which it creates a <code>Schema</code> object. The <code>Schema</code> object creates a <code>Validator</code> object. Finally, the <code>Validator</code> object validates an XML document represented as a <code>Source</code>. </p>
           <p><a href="#listing1">Listing 1</a> shows a simple program to validate a URL entered on the command line against the DocBook XSD schema. </p>
           <h5 id="listing1" class="ibm-h5">Listing 1. Validating an Extensible Hypertext Markup Language (XHTML) document</h5>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">import java.io.*;
import javax.xml.transform.Source;
import javax.xml.transform.stream.StreamSource;
import javax.xml.validation.*;
import org.xml.sax.SAXException;

public class DocbookXSDCheck {

    public static void main(String[] args) throws SAXException, IOException {

        // 1. Lookup a factory for the W3C XML Schema language
        SchemaFactory factory = 
            SchemaFactory.newInstance("http://www.w3.org/2001/XMLSchema");
        
        // 2. Compile the schema. 
        // Here the schema is loaded from a java.io.File, but you could use 
        // a java.net.URL or a javax.xml.transform.Source instead.
        File schemaLocation = new File("/opt/xml/docbook/xsd/docbook.xsd");
        Schema schema = factory.newSchema(schemaLocation);
    
        // 3. Get a validator from the schema.
        Validator validator = schema.newValidator();
        
        // 4. Parse the document you want to check.
        Source source = new StreamSource(args[0]);
        
        // 5. Check the document
        try {
            validator.validate(source);
            System.out.println(args[0] + " is valid.");
        }
        catch (SAXException ex) {
            System.out.println(args[0] + " is not valid because ");
            System.out.println(ex.getMessage());
        }  
        
    }

}</pre></span>
           <p> Here's some typical output when checking an invalid document using the version of Xerces bundled with Java 2 Software Development Kit (JDK) 5.0: </p>
           <p><code>file:///Users/elharo/CS905/Course_Notes.xml is not valid because cvc-complex-type.2.3: Element 'legalnotice' cannot have character [children], because the type's content type is element-only.</code></p>
           <p> You can easily change the schema to validate against, the document to validate, and even the schema language. However, in all cases, validation follows these five steps: </p>
           <ol>
            <li>Load a schema factory for the language the schema is written in.</li>
            <li>Compile the schema from its source.</li>
            <li>Create a validator from the compiled schema.</li>
            <li>Create a <code>Source</code> object for the document you want to validate. A <code>StreamSource</code> is usually simplest.</li>
            <li>Validate the input source. If the document is invalid, the <code>validate()</code> method throws a <code>SAXException</code>. Otherwise, it returns quietly. </li>
           </ol>
           <p>You can reuse the same validator and the same schema multiple times in series. However, only the schema is thread safe. Validators and schema factories are not. If you validate in multiple threads simultaneously, make sure each one has its own <code>Validator</code> and <code>SchemaFactory</code> objects.</p>
           <h3 id="N100CD" class="ibm-h3">Validate against a document-specified schema</h3>
           <p> Some documents specify the schema they expect to be validated against, typically using <code>xsi:noNamespaceSchemaLocation</code> and/or <code>xsi:schemaLocation</code> attributes like this: </p>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">&lt;document xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
  xsi:noNamespaceSchemaLocation="http://www.example.com/document.xsd"&gt;
  ...</pre></span>
           <p> If you create a schema without specifying a URL, file, or source, then the Java language creates one that looks in the document being validated to find the schema it should use. For example: </p>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">SchemaFactory factory = SchemaFactory.newInstance("http://www.w3.org/2001/XMLSchema");
Schema schema = factory.newSchema();</pre></span>
           <p> However, normally this isn't what you want. Usually the document consumer should choose the schema, not the document producer. Furthermore, this approach works only for XSD. All other schema languages require an explicitly specified schema location. </p>
           <h2 id="N100E5" class="ibm-h2">Abstract factories</h2>
           <p><code>SchemaFactory</code> is an abstract factory. The abstract factory design pattern enables this one API to support many different schema languages and object models. A single implementation usually supports only a subset of the numerous languages and models. However, once you learn the API for validating DOM documents against RELAX NG schemas (for instance), you can use the same API to validate JDOM documents against W3C schemas. </p>
           <p> For example, <a href="#listing2">Listing 2</a> shows a program that validates DocBook documents against DocBook's RELAX NG schema. It's almost identical to <a href="#listing1">Listing 1</a>. The only things that have changed are the location of the schema and the URL that identifies the schema language. </p>
           <h5 id="listing2" class="ibm-h5">Listing 2. Validating a DocBook document using RELAX NG</h5>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">import java.io.*;
import javax.xml.transform.Source;
import javax.xml.transform.stream.StreamSource;
import javax.xml.validation.*;
import org.xml.sax.SAXException;

public class DocbookRELAXNGCheck {

    public static void main(String[] args) throws SAXException, IOException {

        // 1. Specify you want a factory for RELAX NG
        SchemaFactory factory 
         = SchemaFactory.newInstance("http://relaxng.org/ns/structure/1.0");
        
        // 2. Load the specific schema you want. 
        // Here I load it from a java.io.File, but we could also use a 
        // java.net.URL or a javax.xml.transform.Source
        File schemaLocation = new File("/opt/xml/docbook/rng/docbook.rng");
        
        // 3. Compile the schema.
        Schema schema = factory.newSchema(schemaLocation);
    
        // 4. Get a validator from the schema.
        Validator validator = schema.newValidator();
        
        // 5. Parse the document you want to check.
        String input 
         = "file:///Users/elharo/Projects/workspace/CS905/build/Java_Course_Notes.xml";
        
        // 6. Check the document
        try {
            validator.validate(source);
            System.out.println(input + " is valid.");
        }
        catch (SAXException ex) {
            System.out.println(input + " is not valid because ");
            System.out.println(ex.getMessage());
        }  
        
    }

}</pre></span>
           <p> If you run this program with the stock Sun JDK and no extra libraries, you'll probably see something like this: </p>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">Exception in thread "main" java.lang.IllegalArgumentException: 
http://relaxng.org/ns/structure/1.0
	at javax.xml.validation.SchemaFactory.newInstance(SchemaFactory.java:186)
	at DocbookRELAXNGCheck.main(DocbookRELAXNGCheck.java:14)</pre></span>
           <p> This is because, out of the box, the JDK doesn't include a RELAX NG validator. When the schema language isn't recognized, <code>SchemaFactory.newInstance()</code> throws an <code>IllegalArgumentException</code>. However, if you install a RELAX NG library such as Jing and a JAXP 1.3 adapter, then it should produce the same answer the W3C schema does. </p>
           <h3 id="N1010C" class="ibm-h3">Identify the schema language</h3>
           <p> The <code>javax.xml.constants</code> class defines several constants to identify schema languages: </p>
           <ul class="ibm-bullet-list">
            <li><code>XMLConstants.W3C_XML_SCHEMA_NS_URI</code>: <code>http://www.w3.org/2001/XMLSchema</code></li>
            <li><code>XMLConstants.RELAXNG_NS_URI</code>: <code>http://relaxng.org/ns/structure/1.0</code></li>
            <li><code>XMLConstants.XML_DTD_NS_URI</code>: <code>http://www.w3.org/TR/REC-xml</code></li>
           </ul>
           <p> This isn't a closed list. Implementations are free to add other URLs to this list to identify other schema languages. Typically, the URL is the namespace Uniform Resource Identifier (URI) for the schema language. For example, the URL <code>http://www.ascc.net/xml/schematron</code> identifies Schematron schemas. </p>
           <p> Sun's JDK 5 only supports XSD schemas. Although DTD validation is supported, it isn't accessible through the <code>javax.xml.validation</code> API. For DTDs, you have to use the regular SAX <code>XMLReader</code> class. However, you can install additional libraries that add support for these and other schema languages. </p>
           <h3 id="N10135" class="ibm-h3">How schema factories are located</h3>
           <p>The Java programming language isn't limited to a single schema factory. When you pass a URI identifying a particular schema language to <code>SchemaFactory.newInstance()</code>, it searches the following locations in this order to find a matching factory:</p>
           <ol>
            <li> The class named by the <code>"javax.xml.validation.SchemaFactory:<em>schemaURL</em>"</code> system property </li>
            <li>The class named by the <code>"javax.xml.validation.SchemaFactory:<em>schemaURL</em>"</code> property found in the <code>$java.home/lib/jaxp.properties</code> file </li>
            <li><code>javax.xml.validation.SchemaFactory</code> service providers found in the META-INF/services directories of any available Java Archive (JAR) files </li>
            <li>A platform default <code>SchemaFactory</code>, <code>com.sun.org.apache.xerces.internal.jaxp.validation.xs.SchemaFactoryImpl</code> in JDK 5 </li>
           </ol>
           <p> To add support for your own custom schema language and corresponding validator, all you have to do is write subclasses of <code>SchemaFactory</code>, <code>Schema</code>, and <code>Validator</code> that know how to process your schema language. Then, install your JAR in one of these four locations. This is useful for adding constraints that are more easily checked in a Turing-complete language like Java than in a declarative language like the W3C XML Schema language. You can define a mini-schema language, write a quick implementation, and plug it into the validation layer. </p>
           <h2 id="N10169" class="ibm-h2">Error handlers</h2>
           <p> The default response from a schema is to throw a <code>SAXException</code> if there's a problem and do nothing if there isn't. However, you can provide a SAX <code>ErrorHandler</code> to receive more detailed information about the document's problems. For example, suppose you want to log all validation errors, but you don't want to stop processing when you encounter one. You can install an error handler such as that in <a href="#listing3">Listing 3</a>. </p>
           <h5 id="listing3" class="ibm-h5">Listing 3. An error handler that merely logs non-fatal validity errors</h5>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">import org.xml.sax.ErrorHandler;
import org.xml.sax.SAXException;
import org.xml.sax.SAXParseException;

public class ForgivingErrorHandler implements ErrorHandler {

    public void warning(SAXParseException ex) {
        System.err.println(ex.getMessage());
    }

    public void error(SAXParseException ex) {
        System.err.println(ex.getMessage());
    }

    public void fatalError(SAXParseException ex) throws SAXException {
        throw ex;
    }

}</pre></span>
           <p> To install this error handler, you create an instance of it and pass that instance to the <code>Validator</code>'s <code>setErrorHandler()</code> method: </p>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">  ErrorHandler lenient = new ForgivingErrorHandler();
  validator.setErrorHandler(lenient);</pre></span>
           <h2 id="N1018C" class="ibm-h2">Schema augmentation</h2>
           <p> Some schemas do more than validate. As well as providing a true-false answer to the question of whether a document is valid, they also <em>augment</em> the document with additional information. For example, they can provide default attribute values. They might also assign types like int or gYear to an element or attribute. The validator can create such type-augmented documents and write them onto a <code>javax.xml.transform.Result</code> object. All you need to do is pass a <code>Result</code> as the second argument to validate. For example, <a href="#listing4">Listing 4</a> both validates an input document and creates an augmented DOM document from the combination of the input with the schema. </p>
           <h5 id="listing4" class="ibm-h5">Listing 4. Augmenting a document with a schema</h5>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">import java.io.*;
import javax.xml.transform.dom.*;
import javax.xml.validation.*;
import javax.xml.parsers.*;
import org.w3c.dom.*;
import org.xml.sax.SAXException;

public class DocbookXSDAugmenter {

    public static void main(String[] args) 
      throws SAXException, IOException, ParserConfigurationException {

        SchemaFactory factory 
         = SchemaFactory.newInstance("http://www.w3.org/2001/XMLSchema");
        File schemaLocation = new File("/opt/xml/docbook/xsd/docbook.xsd");
        Schema schema = factory.newSchema(schemaLocation);
        Validator validator = schema.newValidator();
        
        DocumentBuilderFactory domFactory = DocumentBuilderFactory.newInstance();
        domFactory.setNamespaceAware(true); // never forget this
        DocumentBuilder builder = domFactory.newDocumentBuilder();
        Document doc = builder.parse(new File(args[0]));
        
        DOMSource source = new DOMSource(doc);
        DOMResult result = new DOMResult();
        
        try {
            validator.validate(source, result);
            Document augmented = (Document) result.getNode();
            // do whatever you need to do with the augmented document...
        }
        catch (SAXException ex) {
            System.out.println(args[0] + " is not valid because ");
            System.out.println(ex.getMessage());
        }  
        
    }

}</pre></span>
           <p> This procedure can't transform an arbitrary source into an arbitrary result. It doesn't work at all for stream sources and results. SAX sources can be augmented into SAX results, and DOM sources into DOM results; but SAX sources can't be augmented to DOM results or vice versa. If you need to do that, first augment into the matching result -- SAX for SAX and DOM for DOM -- and then use TrAX's identity transform to change the model. </p>
           <p> This technique isn't recommended, though. Putting all the information the document requires in the instance is far more reliable than splitting it between the instance and the schema. You might validate, but not everyone will. </p>
           <h2 id="N101AA" class="ibm-h2">Type information</h2>
           <p> The W3C XML Schema Language is heavily based on the notion of <em>types</em>. Elements and attributes are declared to be of type int, double, date, duration, person, PhoneNumber, or anything else you can imagine. The Java Validation API includes a means to report such types, although it's surprisingly independent of the rest of the package. </p>
           <p> Types are identified by an <code>org.w3c.dom.TypeInfo</code> object. This simple interface, summarized in <a href="#listing5">Listing 5</a>, tells you the local name and namespace URI of a type. You can also tell whether and how a type is derived from another type. Beyond that, understanding the type is up to your program. The Java language doesn't tell you what it means or convert the data to a Java type such as <code>double</code> or <code>java.util.Date</code>. </p>
           <h5 id="listing5" class="ibm-h5">Listing 5. The DOM TypeInfo interface</h5>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">package org.w3c.dom;

public interface TypeInfo {

  public static final int DERIVATION_RESTRICTION;
  public static final int DERIVATION_EXTENSION;
  public static final int DERIVATION_UNION;

  public String  getTypeName();
  public String  getTypeNamespace()
  public boolean isDerivedFrom(String namespace, String name, int derivationMethod);

}</pre></span>
           <p> To get <code>TypeInfo</code> objects, you ask the <code>Schema</code> object for a <code>ValidatorHandler</code> rather than a <code>Validator</code>. <code>ValidatorHandler</code> implements SAX's <code>ContentHandler</code> interface. Then, you install this handler in a SAX parser. </p>
           <p> You also install your own <code>ContentHandler</code> in the <code>ValidatorHandler</code> (not the parser); the <code>ValidatorHandler</code> will forward the augmented events on to your <code>ContentHandler</code>. </p>
           <p> The <code>ValidatorHandler</code> makes available a <code>TypeInfoProvider</code> that your <code>ContentHandler</code> can call at any time to find out the type of the current element or one of its attributes. It can also tell you whether an attribute is an ID, and whether the attribute was explicitly specified in the document or defaulted in from the schema. <a href="#listing6">Listing 6</a> summarizes this class. </p>
           <h5 id="listing6" class="ibm-h5">Listing 6. The TypeInfoProvider class</h5>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">package javax.xml.validation;

public abstract class TypeInfoProvider {

  public abstract TypeInfo getElementTypeInfo();
  public abstract TypeInfo getAttributeTypeInfo(int index);
  public abstract boolean  isIdAttribute(int index);
  public abstract boolean  isSpecified(int index);

}</pre></span>
           <p> Finally, you parse the document with the SAX <code>XMLReader</code>. <a href="#listing7">Listing 7</a> shows a simple program that uses all these classes and interfaces to print out the names of all the types of the elements in a document. </p>
           <h5 id="listing7" class="ibm-h5">Listing 7. Listing element types</h5>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">import java.io.*;
import javax.xml.validation.*;

import org.xml.sax.*;
import org.xml.sax.helpers.*;

public class TypeLister extends DefaultHandler {

    private TypeInfoProvider provider;
    
    public TypeLister(TypeInfoProvider provider) {
        this.provider = provider;
    }

    public static void main(String[] args) throws SAXException, IOException {

        SchemaFactory factory 
         = SchemaFactory.newInstance("http://www.w3.org/2001/XMLSchema");
        File schemaLocation = new File("/opt/xml/docbook/xsd/docbook.xsd");
        Schema schema = factory.newSchema(schemaLocation);
    
        ValidatorHandler vHandler = schema.newValidatorHandler();
        TypeInfoProvider provider = vHandler.getTypeInfoProvider();
        ContentHandler   cHandler = new TypeLister(provider);
        vHandler.setContentHandler(cHandler);
        
        XMLReader parser = XMLReaderFactory.createXMLReader();
        parser.setContentHandler(vHandler);
        parser.parse(args[0]);
        
    }
    
    public void startElement(String namespace, String localName,
      String qualifiedName, Attributes atts) throws SAXException {
        String type = provider.getElementTypeInfo().getTypeName();
        System.out.println(qualifiedName + ": " + type);
    }

}</pre></span>
           <p> Here's the start of the output from running this code on a typical DocBook document: </p>
           <span class="dw-code-nohighlight"><pre data-widget="syntaxhighlighter" class="brush: js; html-script: true; gutter: true;">book: #AnonType_book
title: #AnonType_title
subtitle: #AnonType_subtitle
info: #AnonType_info
copyright: #AnonType_copyright
year: #AnonType_year
holder: #AnonType_holder
author: #AnonType_author
personname: #AnonType_personname
firstname: #AnonType_firstname
othername: #AnonType_othername
surname: #AnonType_surname
personblurb: #AnonType_personblurb
para: #AnonType_para
link: #AnonType_link</pre></span>
           <p> As you can see, the DocBook schema assigns most elements anonymous complex types. Obviously, this will vary from one schema to the next. </p>
           <h2 id="N10219" class="ibm-h2">Conclusion</h2>
           <p> The world would be a poorer place if everyone spoke just one language. Programmers would be unhappy if they had only one programming language to choose from. Different languages suit different tasks better, and some tasks require more than one language. XML schemas are no different. You can choose from a plethora of useful schema languages. In Java 5 with <code>javax.xml.validation</code>, you have an API that can handle all of them. </p>
