        <h1 class="entry-title">Developing Custom Components with Oracle Mobile Application Framework</h1> 
        <div class="post-info">
         <span class="date published time" title="2014-07-07T09:16:07+00:00">July 7, 2014</span> by 
         <span class="author vcard"><span class="fn"><a href="http://www.ateam-oracle.com/author/jeevan/" rel="author">Jeevan Joseph</a></span></span> 
         <span class="post-comments"><a href="http://www.ateam-oracle.com/custom-components-with-maf/#comments">1 Comment</a></span> 
        </div>
        <div class="entry-content">
         <p id="Introduction">Introduction</p> 
         <p>Oracle’s Mobile Application Framework (MAF) is a great tool for web developers to enter the mobile application development space in an easy and seamless manner. MAF’s full support for JavaScript and CSS to create application behavior and styling, means developers can carry over the same skills of web development into the world of mobile apps. While MAF’s AMX component set is fully featured and is a great springboard to get your application jump-started, MAF developers are not limited by the component set that AMX offers. MAF supports custom components to be created in pure JavaScript and CSS. These custom components become custom tag libraries that can be reused. Custom components can even be used to wrap standard AMX components, making them ideal to customize or enhance the component set.</p> 
         <p id="Main_Article">Main Article</p> 
         <p>In this article we will take a brief look at how you can build and use a custom component. We will examine how the custom tag is created, registered, and the various artifacts that are required to support your custom component. To demonstrate the concepts, we will build a very simple custom component that is rendered like a notification badge inside the application. The component will have support for custom attributes that are used by the component implementation. In this article we also use Font Awesome, which is a font and CSS toolkit, which gives you glyphs and icons that are font characters instead of images. The finished notification badge looks like this <a href="http://www.ateam-oracle.com/wp-content/uploads/2014/07/finished1.png"><img src="http://www.ateam-oracle.com/wp-content/uploads/2014/07/finished1.png" alt="finished" width="365" height="80" class="aligncenter wp-image-25662 size-full" srcset="http://www.ateam-oracle.com/wp-content/uploads/2014/07/finished1.png 365w, http://www.ateam-oracle.com/wp-content/uploads/2014/07/finished1-300x65.png 300w" sizes="(max-width: 365px) 100vw, 365px"></a> The AMX code that accomplished this is simple a single line</p> 
         <pre class="brush: xml; gutter: true">&lt;notification:badge id="badge" class="fa fa-bell fa-2x appIcon" unread="#{applicationScope.unreadCount}"/&gt; 
</pre> 
         <p>This is obviously not an AMX component tag, and it has some non standard attributes like “unread”. This is how a custom component will appear inside an AMX page, and we are going to take a look at how this tag was created, and what happens at runtime to render the icon with a notification badge. The finished project can be downloaded <a href="http://www.ateam-oracle.com/wp-content/uploads/2014/07/MAFCustomComponentExample.zip">here</a>.</p> 
         <h3>The parts of a custom component</h3> 
         <p>A custom component is really just a JavaScript function, that is usually accompanied by some CSS styling. The CSS Styling is of course optional. The job of the JavaScript function is to render the component by generating HTML for it. Any AMX page is ultimately rendered as HTML by the MAF runtime on the device on to the device’s WebView. The DOM structure that a custom component’s JavaScript function generates is inserted at the location where the custom component tag is encountered in the AMX markup as the MAF runtime renders it. You can also pass parameters to your JavaScript function using custom attributes on the tag you have defined for your custom component. In the example below, the <code>unread</code> attribute is used by the JavaScript <code>render()</code> function to decide what&nbsp;HTML to generate and what CSS classes to add to the DOM nodes when rendering the component. In the AMX markup, the custom attribute values can be EL expressions like in our example. The EL expression is evaluated and the JavaScript function sees the evaluated value. For our simple example, we are nor triggering this from real push notifications, but instead just driven off of an &nbsp;input slider component’s value. For more complex custom components, you can obviously organize the code with additional helper functions, but the very minimum for the JavaScript code is to define what HTML to generate when our custom tag is encountered. The JavaScript for our custom component is below :</p> 
         <pre class="brush: javascript; gutter: true">(function () {
    // ---------------------------------------------------------------------------------------------
    // This is the Javascript for the notification:badge custom component
    // ---------------------------------------------------------------------------------------------
    var badge = adf.mf.api.amx.TypeHandler.register("http://xmlns.ateam.com/notification", "badge");
    badge.prototype.render = function (amxNode, id) {
        var rootElement = document.createElement("div");
        try {
            var link = document.createElement("a");
            //tag's custom attribute values being accessed in the JavaScript function
            var userClass = amx.getTextValue(amxNode.getAttribute("class"));
            var unreadCount = parseInt(amx.getTextValue(amxNode.getAttribute("unread")));

            link.setAttribute("href", "#");
            link.setAttribute("class", userClass);
            rootElement.appendChild(link);

            var badgeOverlay = document.createElement("span");
            badgeOverlay.className = "notification-badge";

            // Display a badge only if there are more than 0 unread notifications
            if (unreadCount &gt; 0) {
                badgeOverlay.appendChild(document.createTextNode(unreadCount));
            }
            else {
                badgeOverlay.className = "notification-badge-hidden";
            }
            link.appendChild(badgeOverlay);

        }
        catch (problem) {
            alert(problem);
            adf.mf.log.Framework.logp(adf.mf.log.level.SEVERE, "notification", "render", 
                                       "Problem with custom component creation: " + problem);
        }
        return rootElement;
    };

})();
</pre> 
         <h3>A closer look at the JavaScript and the APIs used</h3> 
         <p>How did MAF know to invoke the function you just wrote, when it encountered the <code>&lt;notification:badge&gt;</code>&nbsp;tag in the AMX page ? This is because we registered a <code>TypeHandler</code> for the tag within the tag name-space in the first line of code above. This is how custom tags are registered with the runtime and how the framework knows what function to call to generate the HTML for a custom component when that custom component’s tag is encountered. <a href="http://docs.oracle.com/middleware/mobile200/mobile/develop/maf-ui-custom-components.htm#BABHJAIB"><code>TypeHandler</code> is the primary API </a>for creating and managing custom UI components. The <a href="http://docs.oracle.com/middleware/mobile200/mobile/develop/maf-ui-custom-components.htm#BABJIBCB"><code>adf.mf.api.amx.TypeHandler.register</code></a> is a static API that lets you provide a name-space and a tag name to register that tag under that name space, and the API returns the <code>TypeHandler</code> subclass. This subclass can be used to prototype functions for this <code>TypeHandler</code> which is now registered for that (namespaced) tag. The function that should be prototyped mandatorily is on the returned <code>TypeHandler</code> is, unsurprisingly, <code>render()</code> which you see in the very next line of code. The <code>render()</code> function returns an <code>HTMLElement</code> subclass by creating&nbsp;the DOM hierarchy for this custom component, as shown in the code snippet above. This function can be arbitrarily complex and reply upon helper functions as well. The important thing is that it should prototype the <code>render()</code> function and return a single <code>HTMLElement</code> that is the root element for the custom component’s HTML.</p> 
         <h3>Putting it together</h3> 
         <p>Once you have the custom component’s JavaScript&nbsp;function and the optional CSS styling used by the component in a CSS file, you can start using it in your MAF application. Custom components are added as JavaScript and CSS additions at a feature level. To add the files to your feature, open the maf-feature.xml in your <em>ViewController&gt;ApplicationSources&gt;Meta-INF</em> directory. Select the feature you want to add your custom component to, and add the JavaScript&nbsp;and CSS&nbsp;files to the “includes”&nbsp;section.</p> 
         <p><a href="http://www.ateam-oracle.com/wp-content/uploads/2014/07/addition1.png"><img src="http://www.ateam-oracle.com/wp-content/uploads/2014/07/addition1.png" alt="addition" width="662" height="306" class="aligncenter wp-image-25665 size-full" srcset="http://www.ateam-oracle.com/wp-content/uploads/2014/07/addition1.png 662w, http://www.ateam-oracle.com/wp-content/uploads/2014/07/addition1-300x138.png 300w" sizes="(max-width: 662px) 100vw, 662px"></a></p> 
         <p>Once the assets are added to the feature, you can add the custom component to your AMX page. In our example we have the <em>notificationBadge.js</em> that does the registration and provides the <code>TypeHander</code> render prototyping, the <em>notification.css</em> that contains the CSS classes that we used to style the notification badge.&nbsp;&nbsp;The complete AMX code is short and concise :</p> 
         <pre class="brush: xml; gutter: true">&lt;?xml version="1.0" encoding="UTF-8" ?&gt;
&lt;amx:view xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" 
          xmlns:amx="http://xmlns.oracle.com/adf/mf/amx"
          xmlns:dvtm="http://xmlns.oracle.com/adf/mf/amx/dvt" 
          xmlns:notification="http://xmlns.ateam.com/notification"&gt;
 &lt;amx:panelPage id="pp1"&gt;
   &lt;amx:facet name="header"&gt;
     &lt;amx:outputText value="Custom Component Sample" id="ot1"/&gt;
   &lt;/amx:facet&gt;
   &lt;amx:facet name="primary"&gt;
     &lt;notification:badge id="badge" 
                      class="fa fa-bell fa-2x notification-icon"
                      unread="#{applicationScope.unreadCount}"/&gt;
   &lt;/amx:facet&gt;
   &lt;amx:inputNumberSlider label="# of Notifications" 
     id="ins1" minimum="0" maximum="15" 
     stepSize="1"
     value="#{applicationScope.unreadCount}"/&gt;
 &lt;/amx:panelPage&gt;
&lt;/amx:view&gt;
</pre> 
         <p class="note">Do not forget to add the xml namespace for your custom tag to the <code>&lt;amx:view&gt;</code> &nbsp;tag.<br> <code>&lt;amx:view xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"</code><br> <code>&nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; xmlns:amx="http://xmlns.oracle.com/adf/mf/amx"</code><br> <code>&nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; xmlns:dvtm="http://xmlns.oracle.com/adf/mf/amx/dvt" &nbsp; &nbsp;&nbsp;</code><br> <code><strong><span style="color: #ff0000;">&nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; xmlns:notification="http://xmlns.ateam.com/notification"</span></strong>&gt;</code></p> 
         <h3>Custom Components : Pros, Cons and Best practices</h3> 
         <p>With custom components in MAF, it becomes possible for web developers to implement sophisticated and highly customized mobile user experiences with the familiar&nbsp;JavaScript and CSS toolset. &nbsp;On the flip side though, custom components do come with certain limitations, at the time of writing this article, custom components do not offer a way to be packaged and shared as a single unit. This means that custom components need to be added as feature additions on all features that require it. The JDeveloper design time does not support using the property inspector to set the custom components properties. This is because the properties offered by a custom component are designed by the custom component creator and can be totally arbitrary, while no such meta-data is known to the JDeveloper design time.</p> 
         <p>When building your own custom components, it is useful to have a set of general development guideines. It is recommended to surround the JavaScript function in a<code> try... catch</code> block to easily troubleshoot and debug the code using JavaScript debugging methods described <a title="here (Android)" href="http://www.ateam-oracle.com/debugging-maf-applications-on-android/">here(Android) </a>and <a href="http://www.ateam-oracle.com/debugging-maf-applications-on-ios/">here (iOS)</a>. Do not&nbsp;assume the DOM structure or develop a custom component that assumes and depends on a certain DOM structure for the rest of the DOM tree. This makes the component fragile and changes to the DOM structure either in the rest of the application code or in the AMX&nbsp;component implementation (perhaps in a later framework release) might break the custom component. Always use the APIs to introspect the AmxNode structure than relying on an observed DOM structure. In most cases like our example above, the markup that is generated also needs to be styled. For styling, any CSS class available through the skin or through a CSS file added to the feature can be used. In our example, we add a separate CSS file that contains CSS classes specific to the custom component. This keeps the skin, which is applicable to any MAF application, devoid of custom component styling. When developing CSS classes, it is a good practice to follow a naming convention like <span style="color: #000000;">&lt;namespace&gt;-&lt;component&gt;-&lt;element&gt;</span>&nbsp;, which is the structure that AMX&nbsp;uses. For example,class names like&nbsp;<em>notification-badge</em> and <em>notification-badge-text</em> would make it clear what component the CSS class belongs to, and would also be better at avoiding collisions.</p> 
         <h3>Dialing it up a notch : Nesting AMX components inside your custom component</h3> 
         <p>Taking the custom components to the next level, the first question would be if you can nest standard AMX components inside your custom component tag, and the answer is yes. This is particularly useful when you are building a custom component that wraps one of more AMX components to add specialized UI behavior.&nbsp;The <a href="http://docs.oracle.com/middleware/mobile200/mobile/develop/maf-ui-custom-components.htm#BABHJAIB">TypeHandler</a>, <a href="http://docs.oracle.com/middleware/mobile200/mobile/develop/maf-ui-custom-components.htm#BABJHGGC">AmxNode</a> and <a href="http://docs.oracle.com/middleware/mobile200/mobile/develop/maf-ui-custom-components.htm#BABCFIJJ">AmxTag </a>APIs <a href="http://docs.oracle.com/middleware/mobile200/mobile/develop/maf-ui-custom-components.htm#BABGHAFA">among others</a>, make the process of mixing and interacting with AMX components simple and easy by letting you have programmatic access to these AMXNodes. Two of the most important functions on the AmxNode API to programmatically access and manipulate the custom component’s nested child components would be the would be the <code>getChildren()</code> API that returns an array of AMXNodes that are the children of the current node. This lets you use the API to manipulate the child nodes before they are rendreded. Just as with the custom component you created above, the standard AMX components also generate their HTML root elements that are HTMLElement subclasses when they are rendered by the runtime. The <code>renderDescendants()</code> API rendders the child nodes and returns an array of HTMLElements that are the root elements created by the child AmxNodes. Our simple example here to get the reader introduced to custom components does not leverage these APIs, and stands to prove that even with very simple implementations meaningful custom components can be built. The framework also ships with a set of samples that&nbsp;show how these APIs work. The samples can be found in the&nbsp;<code style="color: #000000;">PublicSamples.zip</code><span style="color: #222222;">&nbsp;file within the&nbsp;</span><code style="color: #000000;"><span class="codeinlineitalic" style="font-style: italic;">jdev_install</span></code><code style="color: #000000;">/jdeveloper/jdev/extensions/oracle.maf/Samples</code></p> 
         <p>I hope that this has whet your appetite for more on custom components, and we shall dive deeper in to custom components and the APIs in a future post. You can download the sample application used in the article here&nbsp;:&nbsp;<a href="http://www.ateam-oracle.com/wp-content/uploads/2014/07/MAFCustomComponentExample.zip">MAFCustomComponentExample</a></p>
