        <h1 class="entry-title">Choix d’un framework JS pour site mobile en single-page</h1> 
        <div class="entry-meta"> 
         <span class="date"><a href="http://lapetitepausetechnique.net/2013/09/choix-framework-js-site-mobile-en-single-page/" title="Permalien vers Choix d’un framework JS pour site mobile en single-page" rel="bookmark"><time class="entry-date" datetime="2013-09-24T13:44:23+00:00">24/09/2013</time></a></span>
         <span class="categories-links"><a href="http://lapetitepausetechnique.net/category/developpement/javascript/angularjs/" rel="category tag">AngularJS</a>, <a href="http://lapetitepausetechnique.net/category/developpement/" rel="category tag">Développement</a>, <a href="http://lapetitepausetechnique.net/category/developpement/php/drupal/" rel="category tag">Drupal</a>, <a href="http://lapetitepausetechnique.net/category/developpement/javascript/" rel="category tag">Javascript</a></span>

       <div class="entry-content"> 
        <p>Le site Web que je gère au quotidien repose sur&nbsp;<a href="https://drupal.org/" onclick="__gaTracker('send', 'event', 'outbound-article', 'https://drupal.org/', 'Drupal');" title="Drupal.org" target="_blank">Drupal</a>. Pour faire une version mobile, la 1ère idée qui vient est simplement de faire un thème mobile et de continuer de se servir de Drupal pour desservir les pages.&nbsp;Mais je suis joueur, et j’avais envie depuis longtemps d’aller taquiner le JS de façon plus poussée, en faisant autre chose que du jQuery (<a href="http://stackoverflow.com/questions/14994391/how-do-i-think-in-angularjs-if-i-have-a-jquery-background/15012542#15012542" onclick="__gaTracker('send', 'event', 'outbound-article', 'http://stackoverflow.com/questions/14994391/how-do-i-think-in-angularjs-if-i-have-a-jquery-background/15012542#15012542', 'Article très intéressant sur le sujet');" title="Structurer ses développements JS" target="_blank">Article très intéressant sur le sujet</a>), étant une technologie de plus en plus utilisée, tant pour le mobile que pour des applications Web. <img title="Lire la suite…" src="http://www.lapetitepausetechnique.net/wp-includes/js/tinymce/plugins/wordpress/img/trans.gif" alt=""></p> 
        <p><span style="line-height: 1.5;">L’idée est donc naturellement venue de créer un site mobile «&nbsp;à part&nbsp;», appelant des Web-services Drupal qui retourneront le contenu (et seulement le contenu utilisé = perf).&nbsp;J’espère ainsi apprendre de nouvelle technologies, créer une application structurée, innovante, performante…</span></p> 
        <p><span id="more-530"></span></p> 
        <p>Voici donc plusieurs mois que j’étudie les<strong> frameworks JS</strong> avant de faire un&nbsp;<strong>choix</strong>&nbsp;pour la création de ce&nbsp;<strong>site Web mobile</strong>.<br> J’ai donc pu essayer&nbsp;<strong><a href="http://emberjs.com/" onclick="__gaTracker('send', 'event', 'outbound-article', 'http://emberjs.com/', 'EmberJS');" title="EmberJS" target="_blank">EmberJS</a>,&nbsp;<a href="http://jquerymobile.com/" onclick="__gaTracker('send', 'event', 'outbound-article', 'http://jquerymobile.com/', 'jQuery Mobile');" title="jQueryMobile" target="_blank">jQuery Mobile</a>,&nbsp;<a href="http://backbonejs.org/" onclick="__gaTracker('send', 'event', 'outbound-article', 'http://backbonejs.org/', 'Backbone.js');" title="BackboneJS" target="_blank">Backbone.js</a>&nbsp;et&nbsp;<a href="http://angularjs.org/" onclick="__gaTracker('send', 'event', 'outbound-article', 'http://angularjs.org/', 'AngularJS');" title="AngularJS" target="_blank">AngularJS</a></strong>.<br> L’objectif ici n’est pas de comparer les frameworks points par points, notamment sur le paradigme&nbsp;<a href="http://fr.wikipedia.org/wiki/Mod%C3%A8le-vue-contr%C3%B4leur" onclick="__gaTracker('send', 'event', 'outbound-article', 'http://fr.wikipedia.org/wiki/Mod%C3%A8le-vue-contr%C3%B4leur', 'MVC');" title="Qu'est ce que le MVC ?" target="_blank">MVC</a>&nbsp;et d’ajouter un énième article sur le sujet, mais de faire un retour d’expérience concret sur le choix qui a été fait.</p> 
        <p>Venant du monde PHP, technologie «&nbsp;serveur&nbsp;», le <strong>JS</strong> oblige à repenser un peu sa façon de concevoir un site. Quel que soit le framework, on retrouve principalement le principe du «&nbsp;<strong><a href="http://fr.wikipedia.org/wiki/Application_Web_Monopage" onclick="__gaTracker('send', 'event', 'outbound-article', 'http://fr.wikipedia.org/wiki/Application_Web_Monopage', 'single page application');" title="Single page application" target="_blank">single page application</a></strong>«&nbsp;. La partie MVC est donc pour partie déportée côté client.<br> C’est donc le JS qui s’occupe du routing en fonction de l’URL, d’appeler les traitements métiers qui vont bien, qui calcule la vue (template) et la restitue. Il y a souvent des appels Ajax à des Web Services lors du traitement (récupération de contenu, sauvegarde, authentification, etc.)</p> 
        <p>Cette techno permet également l’utilisation de services comme <a href="http://phonegap.com/" target="_blank">phoneGap</a>&nbsp;qui permet d’encapsuler un site HTML/CSS/JS en single-app, dans une application mobile disponible ensuite pour Android, iPhone, Windows Phone, etc. C’est pas non plus magique, il y a quelques éléments à prendre en compte (car l’appli créée est ensuite exécutée côté mobile, appelant le point d’entrée en local : files://index.html, apportant son lot de difficultés).<br> Je vous ai recensé les ressources que j’ai pu trouver sur le sujet <a title="PhoneGap : transformer votre single-page Web en une application mobile native" href="http://www.lapetitepausetechnique.net/2013/09/phonegap-transformer-votre-single-page-web-en-une-application-mobile-native/">dans un nouvel article sur PhoneGap</a>.</p> 
        <p><a name="frameworks-testes-et-choix"></a></p> 
        <h2>Frameworks testés et choix</h2> 
        <p>On trouve beaucoup de ressources sur chacun. AngularJS et EmberJS disposent même de <strong>tutoriaux progressifs</strong>, pratiques pour évaluer ces frameworks sans y passer trop de temps, mais trop léger pour répondre à tous les questions, bien souvent.<br> En plus de ces tutoriaux, vous avez <strong><a href="http://todomvc.com/" target="_blank">la célèbre application «&nbsp;TODO List&nbsp;»</a></strong>&nbsp;développée avec un grand nombre de frameworks JS, permettant de voir les différences de chacun. Mais selon votre besoin, idem, ce sera limité, car les fonctionnalités explorées restent «&nbsp;de base&nbsp;», ce qui est normal pour une application de démo…</p> 
        <p><strong>EmberJS :</strong>&nbsp;bien que la doc soit bien faite, il souffre, je trouve, de trop de conventions de nommages, et m’a semblé de prime abord complexe, mais maintenant que je commence à maîtriser, peut-être qu’en essayant de nouveau, je verrai ça d’un nouvel oeil. Mais je souhaitais une techno dans laquelle on entre sans trop de difficulté (notamment pour mon équipe de développeurs qui devra se faire la main), et avec laquelle il est facile de retrouver ces petits. Je n’ai pas eu ce sentiment avec EmberJS.</p> 
        <p><strong>jQuery mobile</strong>&nbsp;: il est bien fait, intègre son système de routing, mais pas de MVC =&gt; pas cool ! Du coup, soit on le couple à un framework MVC, mais dans ce cas, bonjour l’usine à gaz (choix de qui doit s’occuper du routing, etc.), soit on oublie le framework, et on fonctionne «&nbsp;à la jquery&nbsp;» classique, sans grande structure de code/projet. Depuis, ils ont créé le projet M En gros, ça aurait pu être un bon choix si j’avais conservé la logique métier côté Drupal, car JQM est bien pour la restitution, widgets adaptatifs, etc. Un peu bridé en terme de design (complexe à customiser).</p> 
        <p><strong>Backbone.js</strong> a été mon choix pendant quelques semaines, j’ai commencé un site avec (recherche, liste, détail), mais j’ai vite trouvé que le code se complexifiait très vite pour un besoin simple, du fait de devoir créer à chaque fois des modèles, des collections, l’attachement des événements est lourd, etc.<br> Ca m’a servi de découvrir ce framework d’autant que Drupal 8 et WordPress tendent à l’utiliser dans leurs versions futures. De là à créer un projet complet reposant dessus, je trouve trop lourd les développements/maintenance associés.&nbsp;Et enfin, il ne comporte pas d’emblée d’outil de tests unitaires/fonctionnels, bien qu’une multitude de plugins existent.<br> [EDIT] <a title="D’Angular.js à Backbone.js… le retour" href="http://lapetitepausetechnique.net/2014/05/dangular-js-a-backbone-js-le-retour/">Depuis, j’ai revu ma position !&nbsp;</a>[/EDIT]</p> 
        <p><strong>AngularJS</strong> est mon choix final, bien que j’y soit venu puis reparti (vers backbone) puis revenu, et là, c’est devenu beaucoup plus limpide et clair. Il faut un certain temps pour rentrer dedans, je me retrouve complètement dans ce graphique :&nbsp;<a href="http://www.bennadel.com/blog/2439-My-Experience-With-AngularJS-The-Super-heroic-JavaScript-MVW-Framework.htm">http://www.bennadel.com/blog/2439-My-Experience-With-AngularJS-The-Super-heroic-JavaScript-MVW-Framework.htm</a> <span style="line-height: 1.5;">Je ne détaille pas la raison du choix, car il vaut nettement mieux essayer et mettre les mains dans le cambouis pour évaluer ces frameworks, d’autant que des tonnes d’articles les expliquent déjà.</span></p> 
        <p>&nbsp;</p> 
        <h2>Technologies gravitant autour du JS</h2> 
        <p>En plus de votre framework, on dispose d’un grand nombre d’outils géniaux pour automatiser des tâches, compresser, tester, pour installer des librairies, etc.</p> 
        <ul> 
         <li><a href="http://gruntjs.com/" target="_blank"><strong>Grunt </strong></a>: tâches en ligne de commande («&nbsp;à la&nbsp;» <strong>drush</strong> de Drupal, <strong>Ant</strong> de Java ou <strong>phing</strong> de PHP) permettant de compiler les fichiers LESS, passer un coup de jsHint pour vérifier la syntaxe JS, lancer les tests, créer une version de dev ou de prod compilée/compressée/concaténée…</li> 
         <li><strong><a href="http://bower.io/" target="_blank">Bower</a></strong> : outil permettant d’installer des librairies/dépendances, avec gestion des versions, exactement comme Composer en PHP. Les librairies sont hébergées sur github.com, et doivent disposer d’un package.json ou bower.json contenant les informations de version, le nom de la librairie, etc.</li> 
         <li><a href="https://npmjs.org/" target="_blank"><strong>Npm</strong></a> : un peu comme bower, mais ce sont des packages pour NodeJS. Lorsqu’on développe en NodeJS, on l’utilise pour télécharger les dépendances, et gérer les versions de chacune, exactement comme Composer.</li> 
         <li><span style="line-height: 1.5;"><a href="http://nodejs.org/" target="_blank"><strong>NodeJS</strong></a> : souvent les applications Web ont un backend en NodeJS, j’ai donc également étudié ce formidable serveur, hyper performant. Dans le cas de mon application, il ne sert que pour tous les modules nécessaires à <strong>grunt</strong> (jshint, concat, uglify, karma, ngmin, html2js, recess, …)<br> </span></li> 
        </ul> 
        <h2>Association d’une librairie graphique (UI)</h2> 
        <p>Le framework JS apporte tout le nécessaire pour structurer les développements, les tests, etc. Il manque cependant une brique essentielle : la gestion de l’interface graphique.</p> 
        <p>Ici, le choix dépend beaucoup du type de site/application que l’on souhaite.</p> 
        <ul> 
         <li>Soit on crée tout de A à Z, ce qui peut s’avérer très coûteux en temps, pour bien gérer les soucis de navigateurs (décuplés par le nombre de terminaux), mais peut aussi permettre d’arriver exactement à ce que l’on souhaite, de façon légère et optimisée. Ça dépend aussi des compétences en interne à votre équipe.</li> 
         <li>Soit on s’appuye sur un <strong>framework CSS</strong>, voire <strong>UI</strong> : 
          <ul> 
           <li><strong>jQuery mobile</strong> est bien mais pour le coupler avec AngularJS, c’est carrément lourd ! ( <a href="https://github.com/opitzconsulting/jquery-mobile-angular-adapter" target="_blank">Adaptateur</a> )</li> 
           <li></li> 
           <li><strong><a href="http://getbootstrap.com/" onclick="__gaTracker('send', 'event', 'outbound-article', 'http://getbootstrap.com/', 'Bootstrap');" style="line-height: 1.5;" target="_blank">Bootstrap</a></strong><span style="line-height: 1.5;">&nbsp;est une bonne solution : il permet de disposer d’une grille adaptative, de composants tout fais, de widgets, etc. Permet d’aller plus loin que le mobile, ce qui est une bonne chose (tablette, etc.) car responsive. Se customize assez facilement. Reste complètement indépendant du framework JS choisi (très bon point)</span></li> 
           <li><strong><a href="http://designmodo.github.io/Flat-UI/" onclick="__gaTracker('send', 'event', 'outbound-article', 'http://designmodo.github.io/Flat-UI/', 'FlatUI');" style="line-height: 1.5;" target="_blank">FlatUI</a></strong><span style="line-height: 1.5;">&nbsp;ou autre : surcouche à Bootstrap 2, intéressant, surtout que suivant la mouvance actuellement du «&nbsp;flat design&nbsp;». Mais s’appuie pour le moment sur bootstrap 2, qui vient de passer en 3 récemment.</span></li> 
           <li><strong><a href="http://justspamjustin.github.io/junior/#home" onclick="__gaTracker('send', 'event', 'outbound-article', 'http://justspamjustin.github.io/junior/#home', 'Junior');" style="line-height: 1.5;" target="_blank">Junior</a></strong></li> 
           <li><span style="line-height: 1.5;">Autres frameworks payant (kendo, etc.)</span></li> 
          </ul> </li> 
        </ul> 
        <p>Pour inspiration, des sites proposent des <strong>exemples d’interfaces mobile</strong>, une mine pour trouver des idées :&nbsp;<a href="http://www.mobile-patterns.com/detail-views" target="_blank">http://www.mobile-patterns.com/detail-views</a></p> 
        <p><strong style="font-family: Bitter, Georgia, serif; font-size: 30px; line-height: 1.3;">Conclusion</strong></p> 
        <p>Toutes ces technologies JavaScript sont <strong>de plus en plus matures</strong>, bien qu’évoluant encore très vite. D’autre frameworks voient régulièrement le jour (je pense à <a href="http://www.meteor.com/" onclick="__gaTracker('send', 'event', 'outbound-article', 'http://www.meteor.com/', 'Meteor');" title="Meteor JS" target="_blank">Meteor</a> par exemple). Le choix dépend beaucoup de vos besoins (site web ? application ? mobile ? backend disposant de Webservices ?).</p> 
        <p>Quel que soit votre choix, vous aurez <strong>quasiment toujours besoin de plugins ou librairies externes à votre framework</strong> pour ajouter les fonctionnalités manquantes. Celles-ci dépendent du framework, mais il manque toujours quelque-chose, qu’on trouve généralement très facilement, et bien souvent hébergé sur un github.com, donc gérable via bower.</p>
