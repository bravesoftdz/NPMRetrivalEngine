                    <span class="rendered_qtext">I have been asked this question so many times in so many permutations, so I eventually made a video about it (linked below). Basically, my stance is that there is no such thing as a best tool in general (think Leatherman vs a Power Screwdriver) so you have to know what needs your app specifically will have in order to pick the right set of tools. <br><br>The problem is that you won't know what you're building until you've actually built it (Instagram was first called Burbn, and was a completely different service) so you need to build a prototype/MVP/thingee first - and to use that you should just use whatever the heck you know already. This is because you're picking tools in the dark anyway, there is a 99% chance the tools you're picking are going to be wrong, so you might as well pick the one that you know really well.<br><br>After you have your prototype, you should analyze it in order to find out what your needs are, and then take those needs with you when you wade through the vast amount of tools available, which will allow you to find the ones that are best for your app, and it's going to be a set of tools, not a single god-Leatherman. <br><br>

                    <span class="rendered_qtext"><p class="qtext_para"><i>tl;dr - The React ecosystem is the most energized and creative in fronted frameworks in 2015 and 2016 but it won't always be. </i><b><i>More important is truly understanding vanilla JavaScript</i></b><i>, Object-oriented and functional programming and experimenting building your application without a framework so you truly </i><b><i>feel</i></b><i> the problems frameworks are trying to solve. That will make you able to pick up any framework quickly regardless of what your team choses or what new framework emerges</i></p><p class="qtext_para"><b>Why do we have frameworks?</b></p><p class="qtext_para">Frameworks exist because handling complexity in larger applications can be extremely hard - and often it's the same complexity each time you build a new application.</p><p class="qtext_para">So we create frameworks to give our applications a predictable structure and give us some of the standard features we need in most applications in a 'plug and play' type way. Features like handling requests to the server, or maintaining data consistent across different parts of your web page or web app (you like a tweet and you want the heart to appear but also for the number of likes to increase by 1). Handling that 'state' - your live data - is complex and frameworks can give you a structure for it</p><p class="qtext_para">Frameworks also exist because as engineers we love to build tools and publish them - it's exciting and creative. Other engineers love to challenge themselves with new tools and so get excited by exploring these new frameworks</p><p class="qtext_para"><b>Whats on trend in 2015 and 2016?</b></p><p class="qtext_para">Right now React is the trend and glamorous framework - 2 years from now it likely won't be. People sometimes ask 'Which frameworks do you teach at Codesmith?' We decided to teach React at <span class="qlink_container"><a href="http://codesmith.io" rel="noopener nofollow" target="_blank" onclick="return Q.openUrl(this, 22868522);" class="external_link" data-qt-tooltip="codesmith.io">Codesmith</a></span> (the coding school I cofounded) since the early days and we've been lucky to have the support and endorsement of the React lead at Facebook - <span class="qlink_container"><a href="https://www.youtube.com/watch?v=KVZ-P-ZI6W4" rel="noopener nofollow" target="_blank" onclick="return Q.openUrl(this, 22868522);" class="external_link" data-qt-tooltip="youtube.com">Tom Occhino</a></span>.</p><p class="qtext_para">But which framework we chose to teach at Codesmith is far less relevant than people think. Much more important is developing the capacity to quickly learn any new framework (or even language)</p><p class="qtext_para"><b>How to choose a framework</b></p><p class="qtext_para">To achieve this versatile ability to pick up any new framework quickly, <b>focus on a deep understanding of JavaScript, functional and object oriented programming and an understanding application architecture before anything else</b>.</p>
                     <div class="qtext_image_wrapper">
                      <canvas class="qtext_image_placeholder landscape qtext_image zoomable_in zoomable_in_feed" width="602.0" height="287.7245724572457"></canvas>
                      <img class="landscape qtext_image zoomable_in zoomable_in_feed lazy_loaded lazy_loading" src="" data-src="https://qph.ec.quoracdn.net/main-qimg-46ffc4072ae7666a9010423e0672884a.webp" master_src="https://qph.ec.quoracdn.net/main-qimg-00f4d09fa5cd00db331b5b2ab88c2ab1" master_w="1111" master_h="531">
                     </div><p class="qtext_para">Do what students at <span class="qlink_container"><a href="http://codesmith.io" rel="noopener nofollow" target="_blank" onclick="return Q.openUrl(this, 22868522);" class="external_link" data-qt-tooltip="codesmith.io">Codesmith</a></span> (the 'masters program of coding bootcamps' that I teach at) do - first build your application with vanilla Javascript (or with jQuery - although here's a fantastic resource to avoid jQuery - <span class="qlink_container"><a href="http://youmightnotneedjquery.com/" rel="noopener nofollow" target="_blank" onclick="return Q.openUrl(this, 22868522);" class="external_link" data-qt-tooltip="youmightnotneedjquery.com">You Might Not Need jQuery</a></span>). Consider structuring your app in MVC style - which you don't need a framework to do - <span class="qlink_container"><a href="https://danmartensen.svbtle.com/how-to-write-mvc-without-a-framework" rel="noopener nofollow" target="_blank" onclick="return Q.openUrl(this, 22868522);" class="external_link" data-qt-tooltip="svbtle.com">Rediscovering MVC and How to Write without a Framework  Dan Martensen</a></span></p><p class="qtext_para"><b>Then once you have a prototype together and you have to start handling more complexity - now analyze your application</b> (how is your data's flowing, what are the most complex interactions, how and how often are you interacting with the server).</p><p class="qtext_para">Assess the different frameworks - but <b>now you'll understand </b><b><i>why</i></b><b> you need a framewor</b>k - what problems it's solving - and it'll make you far more effective at using it (whichever framework you choose)</p></span>

                    <span class="rendered_qtext">I do accept/confirm that there is no one go here like React or Angular but as a front end developer I can suggest that pure technology stack that is easy on your browsers and your code editors too, not to mention the coder :). <br><br>Starting a project in 2015 needs a huge amount of knowledge on a number of frameworks, libraries and concepts. (see my stars <span class="qlink_container"><a href="https://github.com/ahmadalibaloch/" rel="noopener nofollow" target="_blank" onclick="return Q.openUrl(this, 15920220);" class="external_link" data-qt-tooltip="github.com">github (Ahmad Ali)</a></span>) You will need a base revolving around a framework or concept and a large number of libraries helping you out for the hacks and fixes needed for that framework or concept. Because simply nothing is perfect. :).<br><br>
                     <div class="qtext_image_wrapper">
                      <canvas class="qtext_image_placeholder landscape qtext_image zoomable_in zoomable_in_feed" width="294" height="294"></canvas>
                      <img class="landscape qtext_image zoomable_in zoomable_in_feed lazy_loaded lazy_loading" src="" data-src="https://qph.ec.quoracdn.net/main-qimg-3ad8208c245d07e889060f120a3a1b3e.webp" master_src="https://qph.ec.quoracdn.net/main-qimg-3ad8208c245d07e889060f120a3a1b3e" master_w="294" master_h="294">
                     </div>For all of this tree of technologies to be useful for you in a best way, learn ES6. <i>Javascript with ES6 has developed to that level that you can rely on it for your next project even without a framework</i> with facilities like modules, classes, proxies and promises and much more mentioned here <span class="qlink_container"><a href="https://github.com/lukehoban/es6features" rel="noopener nofollow" target="_blank" onclick="return Q.openUrl(this, 15920220);" class="external_link" data-qt-tooltip="github.com">ES6 features</a></span>.<br><br>I will make my answer short with a list of frameworks I think are going to make it easy and will well survive for a couple of years based on the recent concepts.<br>
                     <ol>
                      <li>Tools like <span class="qlink_container"><a href="https://webpack.github.io/" rel="noopener nofollow" target="_blank" onclick="return Q.openUrl(this, 15920220);" class="external_link" data-qt-tooltip="webpack.github.io">Webpack module bundler</a></span>, Jspm and<span class="qlink_container"><a href="https://babeljs.io/" rel="noopener nofollow" target="_blank" onclick="return Q.openUrl(this, 15920220);" class="external_link" data-qt-tooltip="babeljs.io"> Babeljs.io</a></span> are pre-requisites if you are starting a project today (Sept 2015). They are also supported by editors . For Visual Studio users its a lot easy for ES6 and much new things like LESS with <span class="qlink_container"><a href="https://github.com/madskristensen/WebCompiler" rel="noopener nofollow" target="_blank" onclick="return Q.openUrl(this, 15920220);" class="external_link" data-qt-tooltip="github.com">WebCompiler</a></span>.</li>
                      <li><i>Angularjs 1,2</i> and frameworks based on MVC, MVVM models that well support the RESTful apis and your backend frameworks.</li>
                      <li><span class="qlink_container"><a href="https://www.firebase.com/" rel="noopener nofollow" target="_blank" onclick="return Q.openUrl(this, 15920220);" class="external_link" data-qt-tooltip="firebase.com"><i>Firebase</i></a></span><i> </i>, Meteor etc live databases supported (sort of) frameworks which give you relief on the backend and connection of backend to client code. Concentrate on the business.</li>
                      <li><i>React methodology</i> based on events occurring within the data which improves understanding of the business logic and is easy on browsers.</li>
                     </ol><br>Angularjs is advanced and well supported framework and it helps. React is comparatively new and this "new thing" is killing it. Its not getting the ground but its a good thing. Angularjs 2 itself has the React flow in its working. Live database and cloud are another good selection but again they are still in testing phase for a real big project.</span>
