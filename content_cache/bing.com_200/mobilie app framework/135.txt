    <p>Android Studio is designed to make testing simple. With just a few clicks, you can set up a JUnit test that runs on the local JVM or an instrumented test that runs on a device. Of course, you can also extend your test capabilities by integrating test frameworks such as <a href="https://github.com/mockito/mockito">Mockito</a> to test Android API calls in your local unit tests, and <a href="https://developer.android.com/topic/libraries/testing-support-library/index.html#Espresso">Espresso</a> or <a href="https://developer.android.com/topic/libraries/testing-support-library/index.html#UIAutomator">UI Automator</a> to exercise user interaction in your instrumented tests. You can generate Espresso tests automatically using <a href="https://developer.android.com/studio/test/espresso-test-recorder.html">Espresso Test Recorder</a>.</p> 
    <p>This page provides the basic information about how to add new tests to your app and run them from Android Studio.</p> 
    <p>For a more complete how-to guide for writing tests, see <a href="https://developer.android.com/training/testing/index.html">Testing Apps on Android</a>.</p> 
    <h2 id="test_types_and_location">Test types and location</h2> 
    <p>The location of your test code depends on the type of test you are writing. Android Studio provides source code directories (source sets), for the following two types of tests:</p> 
    <dl> 
     <dt>
      <b>Local unit tests</b>
     </dt> 
     <dd> 
      <p>Located at <code><var>module-name</var>/src/test/java/</code>.</p> 
      <p>These are tests that run on your machine's local Java Virtual Machine (JVM). Use these tests to minimize execution time when your tests have no Android framework dependencies or when you can mock the Android framework dependencies. </p> 
      <p>At runtime, these tests are executed against a modified version of <code>android.jar</code> where all <code>final</code> modifiers have been stripped off. This lets you use popular mocking libraries, like Mockito.</p> 
     </dd> 
     <dt>
      <b>Instrumented tests</b>
     </dt> 
     <dd> 
      <p>Located at <code><var>module-name</var>/src/androidTest/java/</code>.</p> 
      <p>These are tests that run on a hardware device or emulator. These tests have access to <code><a href="https://developer.android.com/reference/android/app/Instrumentation.html">Instrumentation</a></code> APIs, give you access to information such as the <code><a href="https://developer.android.com/reference/android/content/Context.html">Context</a></code> of the app you are testing, and let you control the app under test from your test code. Use these tests when writing integration and functional UI tests to automate user interaction, or when your tests have Android dependencies that mock objects cannot satisfy.</p> 
      <p>Because instrumented tests are built into an APK (separate from your app APK), they must have their own <a href="https://developer.android.com/guide/topics/manifest/manifest-intro.html"><code>AndroidManifest.xml</code></a> file. However, Gradle automatically generates this file during the build so it is not visible in your project source set. You can add your own manifest file if necessary, such as to specify a different value for `minSdkVersion` or register run listeners just for your tests. When building your app, Gradle merges multiple manifest files into one manifest. </p>
     </dd> 
    </dl> 
    <p>The Gradle build interprets these test source sets in the same manner as it does for your project's <a href="https://developer.android.com/studio/build/index.html#sourcesets">app source sets</a>, which allows you to <a href="#create_instrumented_test_for_a_build_variant">create tests based on build variants</a>.</p> 
    <p>When you create a new project or add an app module, Android Studio creates the test source sets listed above and includes an example test file in each. You can see them in the <strong>Project</strong> window as shown in figure 1.</p> 
    <p><img src="https://developer.android.com/studio/images/test/project-window-tests_2-2_2x.png" alt="" width="827"> </p>
    <p class="img-caption"><b>Figure 1.</b> Your project's <b>(1)</b> instrumented tests and <b>(2)</b> local JVM tests are visible in either the <b>Project</b> view (left) or <b>Android</b> view (right).</p>
    <p></p> 
    <h2 id="add_a_new_test">Add a new test</h2> 
    <p>To create either a local unit test or an instrumented test, you can create a new test for a specific class or method by following these steps:</p> 
    <ol> 
     <li>Open the Java file containing the code you want to test.</li> 
     <li>Click the class or method you want to test, then press Ctrl+Shift+T (??T).</li> 
     <li>In the menu that appears, click <strong>Create New Test</strong>.</li> 
     <li>In the <strong>Create Test</strong> dialog, edit any fields and select any methods to generate, and then click <strong>OK</strong>.</li> 
     <li>In the <strong>Choose Destination Directory</strong> dialog, click the source set corresponding to the type of test you want to create: <strong>androidTest </strong>for an instrumented test or <strong>test</strong> for a local unit test. Then click <strong>OK</strong>.</li> 
    </ol> 
    <p>Alternatively, you can create a generic Java file in the appropriate test source set as follows:</p> 
    <ol> 
     <li>In the <strong>Project </strong>window on the left, click the drop-down menu and select the <strong>Project</strong> view.</li> 
     <li>Expand the appropriate module folder and the nested <strong>src </strong> folder. To add a local unit test, expand the <strong>test</strong> folder and the nested <strong>java</strong> folder; to add an instrumented test, expand the <strong>androidTest</strong> folder and the nested <strong>java</strong> folder.</li> 
     <li>Right-click on the Java package directory and select <strong>New &gt; Java Class</strong>.</li> 
     <li>Name the file and then click <strong>OK</strong>.</li> 
    </ol> 
    <p>Also be sure you specify the test library dependencies in your app module's <code>build.gradle</code> file:</p> 
    <pre class="prettyprint notranslate" translate="no"><code>dependencies {
    // Required for local unit tests (JUnit 4 framework)
    testCompile 'junit:junit:4.12'

    // Required for instrumented tests
    androidTestCompile 'com.android.support:support-annotations:24.0.0'
    androidTestCompile 'com.android.support.test:runner:0.5'
}
</code></pre> 
    <p>For other optional library dependencies and more information about how to write your tests, see <a href="https://developer.android.com/training/testing/unit-testing/local-unit-tests.html">Building Local Unit Tests</a> and <a href="https://developer.android.com/training/testing/unit-testing/instrumented-unit-tests.html">Building Instrumented Unit Tests</a>.</p> 
    <h4 id="dependency_conflicts">Resolve conflicts between the app and test APK</h4> 
    <p>Both your app APK and its test APK share the same classpath. So, you can get a build error if both APKs depend on different versions of the same library. If, for some reason, Gradle isn't able to detect this version conflict, your app may behave unexpectedly during runtime or crash.</p> 
    <p>To resolve this issue, simply make sure that you use the same version of each dependency that is common to both your app and its instrumented tests. To help you manage version numbers across your project, consider <a href="https://developer.android.com/studio/build/gradle-tips.html#configure-project-wide-properties">creating project-wide properties</a>.</p> 
    <p>In some cases, version conflicts may occur from transitive dependencies (which are dependencies of the dependencies you configure in your build). To resolve this version conflict, explicitly declare the transitive dependency in the build script for your app or its instrumented tests, and use the higher available version number. To investigate transitive dependencies in your project, <a href="https://developer.android.com/studio/build/dependencies.html#view_the_dependency_tree">view the dependency tree</a>.</p> 
    <p><a name="sourcesets"></a></p> 
    <h3 id="create_instrumented_test_for_a_build_variant">Create instrumented test for a build variant</h3> 
    <p>If your project includes <a href="https://developer.android.com/studio/build/build-variants.html">build variants</a> with unique source sets, then you might want corresponding instrumented test source sets. Creating instrumented tests in source sets that correspond to your build variants helps keep your test code organized and allows you to run only the tests that apply to a given build variant.</p> 
    <p>To add a testing source set for your build variant, follow these steps:</p> 
    <ol> 
     <li>In the <strong>Project</strong> window on the left, click the drop-down menu and select the <strong>Project</strong> view.</li> 
     <li>Within the appropriate module folder, right-click the <strong>src</strong> folder and click <strong>New &gt; Directory</strong>.</li> 
     <li>For the directory name, enter "androidTest<em>VariantName</em>." For example, if you have a build variant called "MyFlavor" then the directory name should be "androidTestMyFlavor." Then click <strong>OK</strong>.</li> 
     <li>Right-click on the new directory and click <strong>New &gt; Directory</strong>.</li> 
     <li>Enter "java" as the directory name, and then click <strong>OK</strong>.</li> 
    </ol> 
    <p>Now you can add tests to this new source set by following the steps above to add a new test. When you reach the <strong>Choose Destination Directory</strong> dialog, select the new variant test source set.</p> 
    <p>The instrumented tests in <code>src/androidTest/</code> source set are shared by all build variants. When building a test APK for the "MyFlavor" variant of your app, Gradle combines both the <code>src/androidTest/</code> and <code>src/androidTestMyFlavor/</code> source sets.</p> 
    <p>For example, the following table shows how instrumentation test files should reside in source sets that correspond to the app's code source sets.</p> 
    <table> 
     <tbody>
      <tr> 
       <th>Path to app class </th> 
       <th>Path to matching instrumentation test class </th> 
      </tr> 
      <tr> 
       <td><code>src/main/java/Foo.java</code> </td> 
       <td><code>src/androidTest/java/AndroidFooTest.java</code> </td> 
      </tr> 
      <tr> 
       <td><code>src/myFlavor/java/Foo.java</code> </td> 
       <td><code>src/androidTestMyFlavor/java/AndroidFooTest.java</code> </td> 
      </tr> 
     </tbody>
    </table> 
    <p>Just as it does for your app source sets, the Gradle build merges and overrides files from different test source sets. In this case, the <code>AndroidFooTest.java</code> file in the "androidTestMyFlavor" source set overrides the version in the "androidTest" source set. For more information about how source sets are merged, see <a href="https://developer.android.com/studio/build/index.html#sourcesets">Configure Your Build</a>.</p> 
    <p>Another reason you should use build variants for your app and test source sets is to create hermetic tests through mock dependencies. That is, you can create a product flavor for your app that contains fake implementations of dependencies (such as network requests or device sensor data that is ordinarily flaky), and then add a corresponding mock test source set. For more information, see the blog post about <a href="http://android-developers.blogspot.com/2015/12/leveraging-product-
flavors-in-android.html">leveraging product flavors for hermetic testing</a>.</p> 
    <h3 id="configure_instrumentation_manifest_settings">Configure instrumentation manifest settings</h3> 
    <p>When Gradle builds your test APK, it automatically generates the <a href="https://developer.android.com/guide/topics/manifest/manifest-intro.html"><code>AndroidManifest.xml</code></a> file and configures it with the <a href="https://developer.android.com/guide/topics/manifest/instrumentation-element.html"><code>&lt;instrumentation&gt;</code></a> node. One of the reasons Gradle configures this node for you is to make sure that the <a href="https://developer.android.com/guide/topics/manifest/instrumentation-element.html#trgt"><code>targetPackage</code></a> property specifies the correct package name of the app under test. You can change some of the other settings for this node by either creating another manifest file in the test source set or configuring your module-level <code>build.gradle</code> file, as shown in the following code sample.</p> 
    <pre>
android {
  ...
  // Each product flavor you configure can override properties in the
  // defaultConfig {} block. To learn more, go to <a href="https://developer.android.com/studio/build/build-variants.html#product-flavors">Configure Product Flavors</a>.
  defaultConfig {
    ...
    // Specifies the <a href="https://developer.android.com/studio/build/application-id.html">application ID</a> for the test APK.
    <a href="http://google.github.io/android-gradle-dsl/current/com.android.build.gradle.internal.dsl.ProductFlavor.html#com.android.build.gradle.internal.dsl.ProductFlavor:testApplicationId">testApplicationId</a> "com.test.foo"
    // Specifies the fully-qualified class name of the test instrumentation runner.
    <a href="http://google.github.io/android-gradle-dsl/current/com.android.build.gradle.internal.dsl.ProductFlavor.html#com.android.build.gradle.internal.dsl.ProductFlavor:testInstrumentationRunner">testInstrumentationRunner</a> "android.test.InstrumentationTestRunner"
    // If set to 'true', enables the instrumentation class to start and stop profiling.
    // If set to false (default), profiling occurs the entire time the instrumentation
    // class is running.
    <a href="http://google.github.io/android-gradle-dsl/current/com.android.build.gradle.internal.dsl.ProductFlavor.html#com.android.build.gradle.internal.dsl.ProductFlavor:testHandleProfiling">testHandleProfiling</a> true
    // If set to 'true', indicates that the Android system should run the instrumentation
    // class as a functional test. The default value is 'false'
    <a href="http://google.github.io/android-gradle-dsl/current/com.android.build.gradle.internal.dsl.ProductFlavor.html#com.android.build.gradle.internal.dsl.ProductFlavor:testFunctionalTest">testFunctionalTest</a> true
  }
}
...
</pre> 
    <h3 id="change_the_test_build_type">Change the test build type</h3> 
    <p>By default, all tests run against the debug build type. You can change this to another build type by using the <code>testBuildType</code> property in your module-level <code>build.gradle</code> file. For example, if you want to run your tests against your "staging" build type, edit the file as shown in the following snippet.</p> 
    <pre class="prettyprint notranslate" translate="no"><code>android {
    ...
    testBuildType "staging"
}
</code></pre> 
    <p><a name="test_options"></a></p> 
    <h3 id="configure_gradle_test_options">Configure Gradle test options</h3> 
    <p>The <a href="https://developer.android.com/studio/releases/gradle-plugin.html">Android plugin for Gradle</a> allows you to specify certain options for all or just some of your unit tests. In the module-level <code>build.gradle</code> file, use the <a href="http://google.github.io/android-gradle-dsl/current/com.android.build.gradle.internal.dsl.TestOptions.html"><code>testOptions {}</code></a> block to specify options that change how Gradle runs all your tests.</p> 
    <pre>
android {
  ...
  // Encapsulates options for running tests.
  testOptions {
    // Changes the directory where Gradle saves test reports. By default, Gradle saves test reports
    // in the <code><var>path_to_your_project</var>/<var>module_name</var>/build/outputs/reports/</code> directory.
    // '$rootDir' sets the path relative to the root directory of the current project.
    reportDir "$rootDir/test-reports"
    // Changes the directory where Gradle saves test results. By default, Gradle saves test results
    // in the <code><var>path_to_your_project</var>/<var>module_name</var>/build/outputs/test-results/</code> directory.
    // '$rootDir' sets the path relative to the root directory of the current project.
    resultsDir "$rootDir/test-results"
  }
}
</pre> 
    <p>To specify options for only local unit tests, configure the <a href="http://google.github.io/android-gradle-dsl/current/com.android.build.gradle.internal.dsl.TestOptions.UnitTestOptions.html"><code>unitTests {}</code></a> block inside <code>testOptions {}</code>.</p> 
    <pre>
android {
  ...
  testOptions {
    ...
    // Encapsulates options for local unit tests.
    unitTests {
      // By default, local unit tests throw an exception any time the code you are testing tries to access
      // Android platform APIs (unless you <a href="https://developer.android.com/training/testing/unit-testing/local-unit-tests.html#mocking-dependencies">mock Android dependencies</a> yourself or with a testing
      // framework like Mockito). However, you can enable the following property so that the test
      // returns either null or zero when accessing platform APIs, rather than throwing an exception.
      returnDefaultValues true

      // Encapsulates options for controlling how Gradle executes local unit tests. For a list
      // of all the options you can specify, read <a href="https://docs.gradle.org/current/javadoc/org/gradle/api/tasks/testing/Test.html">Gradle's reference documentation</a>.
      all {
        // Sets JVM argument(s) for the test JVM(s).
        jvmArgs '-XX:MaxPermSize=256m'

        // You can also check the task name to apply options to only the tests you specify.
        if (it.name == 'testDebugUnitTest') {
          systemProperty 'debug', 'true'
        }
        ...
      }
    }
  }
}
</pre> 
    <h2 id="run_a_test">Run a test</h2> 
    <p>To run a test, proceed as follows:</p> 
    <ol> 
     <li>Be sure your project is synchronized with Gradle by clicking <strong>Sync Project </strong> <img src="https://developer.android.com/studio/images/buttons/toolbar-sync-gradle.png" class="inline-icon"> in the toolbar.</li> 
     <li>Run your test in one of the following ways:
      <ul> 
       <li>In the <strong>Project</strong> window, right-click a test and click <strong>Run</strong> <img src="https://developer.android.com/studio/images/buttons/toolbar-run.png" class="inline-icon">.</li> 
       <li>In the Code Editor, right-click a class or method in the test file and click <strong>Run</strong> <img src="https://developer.android.com/studio/images/buttons/toolbar-run.png" class="inline-icon"> to test all methods in the class.</li> 
       <li>To run all tests, right-click on the test directory and click <strong>Run tests</strong> <img src="https://developer.android.com/studio/images/buttons/toolbar-run.png" class="inline-icon">.</li> 
      </ul> </li> 
    </ol> 
    <p>By default, your test runs using Android Studio's default run configuration. If you'd like to change some run settings such as the instrumentation runner and deployment options, you can edit the run configuration in the <strong>Run/Debug Configurations </strong>dialog (click <strong>Run &gt; Edit Configurations</strong>).

information, see Create and Edit Run/Debug Configurations.
--></p> 
    <h3 id="view_test_coverage">View test coverage</h3> 
    <p>The test coverage tool is available for local unit tests to track the percentage and areas of your app code that your unit tests have covered. Use the test coverage tool to determine whether you have adequately tested the elements, classes, methods, and lines of code that make up your app.</p> 
    <p>There are a number of ways to run a unit test, and they are described on the IntelliJ <a class="external-link" href="https://www.jetbrains.com/help/idea/2016.2/running-with-coverage.html"> Running with Coverage</a> page. The following procedure shows how to run a unit test inline from the editor:</p> 
    <ol> 
     <li>Double-click the unit test you want to run.</li> 
     <li>In the editor, place your cursor in the line you want to run with coverage. 
      <ul> 
       <li>If you place your cursor in the class declaration, all test methods in that class run.</li> 
       <li>If you place your cursor in a method declaration, all code in that method runs.</li> 
       <li>If you place your cursor on a particular line within a method, only that line runs.</li> 
      </ul></li> 
     <li>Right-click the line where you placed your cursor.</li> 
     <li>In the context menu, choose <b>Run <var>test-name</var> with coverage</b>. <p>The <a class="external-link" href="https://www.jetbrains.com/help/idea/2016.2/coverage-tool-window.html"> coverage tool window</a> appears.</p></li> 
    </ol> 
    <p>Figure 2 shows the coverage tool window for a calculator unit test that tests for addition, subtraction, multiplication, and division.</p> 
    <p><img src="https://developer.android.com/studio/images/test/test-code-coverage-tool_2x.png" alt="" width="722"> </p>
    <p class="img-caption"><b>Figure 2.</b> See the code coverage percentages for your application.</p>
    <p></p> 
    <p> For more information about local unit tests, see <a href="https://developer.android.com/training/testing/unit-testing/local-unit-tests.html">Building Local Unit Tests</a>. </p> 
    <h2 id="view_the_test_results">View the test results</h2> 
    <p>When you run a JUnit or instrumented test, the results appear in the <strong>Run</strong> window. A green bar means all tests succeeded and a red bar means at least one test failed. Figure 1 shows a successful test run.</p> 
    <p><img src="https://developer.android.com/studio/images/test/run_window_2x.png" alt="" width="841"> </p>
    <p class="img-caption"><b>Figure 1.</b> Test results appear in the Run window.</p>
    <p></p> 
    <p>The <strong>Run</strong> window displays the tests in a tree view on the left, and the results and messages for the current test suite in the output pane on the right. Use the toolbars, context menus, and status icons to manage the test results, as follows:</p> 
    <ol class="callouts"> 
     <li>Use the <b>run toolbar</b> to rerun the current test, stop the current test, rerun failed tests (not shown because it is available for unit tests only), pause output, and dump threads.</li> 
     <li>Use the <b>testing toolbar</b> to filter and sort test results. You can also expand or collapse nodes, show test coverage, and import or export test results. </li> 
     <li>Click the <b>context menu</b> <img class="inline-icon" src="https://developer.android.com/studio/images/buttons/runpane-cog-button.png" alt=""> to track the running test, show inline statistics, scroll to the stacktrace, open the source code at an exception, auto scroll to the source, and select the first failed test when the test run completes.</li> 
     <li><b>Test status icons</b> indicate whether a test has an error, was ignored, failed, is in progress, has passed, is paused, was terminated, or was not run. </li> 
     <li>Right-click a line in the tree view to display a context menu that lets you run the tests in debug mode, open the test source code file, or jump to the line in the source code being tested.</li> 
    </ol> 
    <p>For more information about the <strong>Run</strong> window, its toolbars, and context menus, see the IntelliJ page, <a class="external-link" href="https://www.jetbrains.com/help/idea/2016.2/test-runner-tab.html">Test Runner Tab</a>.</p> 
    <h3 id="view_inline_statistics">View inline statistics</h3> 
    <p>To find out how long your tests took to run, do the following:</p> 
    <ol> 
     <li>Click the cog icon <img class="inline-icon" src="https://developer.android.com/studio/images/buttons/runpane-cog-button.png" alt="">. </li>
     <li>In the drop-down list, select <b>Show Inline Statistics</b>. <p> The elapsed time in milliseconds displays to the right of the tests. </p> </li> 
    </ol> 
    <h3 id="compare_strings">Compare strings</h3> 
    <p> If a unit test contains <code>assertEquals()</code> failures from comparing two string objects, you can see what the differences are between the two string objects to find out what caused the failures, as follows: </p> 
    <ol> 
     <li>In the output pane, click the <b>Click to see difference</b> link. </li> 
     <li>In the <b>Differences viewer</b>, explore the differences as described on the IntelliJ page, <a class="external-link" href="https://www.jetbrains.com/help/idea/2016.2/differences-viewer-for-files.html"> Differences viewer for files</a>.</li> 
    </ol> 
    <h3 id="export_test_results">Export test results</h3> 
    <p>You can export your test results in XML or HTML format, as follows:</p> 
    <ol> 
     <li>Click <b>Export Test Results</b> <img class="inline-icon" src="https://developer.android.com/studio/images/buttons/runpane-export-test-results.png" alt="">.</li> 
     <li>In the <b>Export Test Results</b> dialog, provide format and output information, and click <b>OK</b>. <p>The exported test results save to the specified folder. </p> </li> 
    </ol> 
    <h3 id="import_test_results">Import test results</h3> 
    <p>You can import your exported test results, as follows:</p> 
    <ol> 
     <li>Click <b>Import Test Results</b> <img class="inline-icon" src="https://developer.android.com/studio/images/buttons/runpane-import-test-results.png" alt="">.</li> 
     <li>In the drop-down menu, select the file you want to import. <p>The imported tests results appear in the <b>Run</b> window. </p> </li> 
    </ol>
