        <p>In today’s Java tutorial, we will cover the topic of <strong>imports</strong>. What are <strong>Java imports</strong>? How are imports used in Java code? Why do I need imports? Is there any easy way to manage imports? These are the questions I hope to answer in order to give you a full understanding of why these imports used.</p> 
        <h2>What are Java Imports?</h2> 
        <p>Well, since you are already familiar with <a href="https://howtoprogramwithjava.com/java-package">Java packages</a>, Java imports flow naturally from packages. In Java, there are TONS of useful <strong>built in classes and methods</strong> that allow us to do things like:</p> 
        <ul> 
         <li>Read the contents of a file / Create a file and populate it with contents</li> 
         <li>Compare dates with each other (i.e. see if one date is before or after another)</li> 
         <li>Send emails to anyone</li> 
        </ul> 
        <p>Okay, that’s great, so <strong>what does that have to do with imports</strong>? Well, all of these classes are nicely organized in packages, and if you want to USE these classes, you’ll need to import them into your project. So, this means that before you can play around with Dates, you’ll need to import the <code>Date</code> object first!</p> 
        <h2>What do Java imports look like in Code?</h2> 
        <p>Well, if we continue on our example of Dates, then it would look like this:</p>

        <p>If you have a keen eye, you’ll spot in the <strong>imports</strong> at the very <strong>beginning of this code segment</strong>. All this code means, is that in the package “java.util”, there are Classes called “Date” and “Calendar”, and we want to use them in our program! The reason that we need to tell Java that we are going to use these Classes is for the sake of brevity and specificity. A great example of this is the <code>Date</code> class. You may or may not know this, but there are more than one <code>Date</code> classes in Java. There is a <code>java.util.Date</code> class and a <code>java.sql.Date</code> class.</p> 
        <p>So, you can imagine how it would be impossible for Java to decide which <code>Date</code> class you’re actually referring to right? So you just need to <strong>specify which one you’d like to use</strong>. </p> 
        <h2>Why do I need Java Imports</h2> 
        <p>Well, the truth is that you don’t actually NEED to use imports. You could get away with referring to the <strong>full package name</strong> of the Class you with to use. So the code above could be re-written like so:</p>

        <p>Notice that the <strong>import statements at the top are gone</strong>, and we’ve now prefixed every <code>Date</code> and <code>Calendar</code> reference with <code>java.util</code>. This is correct code and will compile just fine. But it’s a bit <strong>annoying</strong> to have to write out the <code>java.util</code> every time you reference <code>Date</code> or <code>Calendar</code>. So this is why we have import statements in Java. It just <strong>makes life easier</strong>!</p> 
        <h2>Is there an easy way to use Java Imports?</h2> 
        <p>There is! In your Spring STS IDE, you can use a shortcut key that will automatically detect any Classes that haven’t yet been imported and attempt to automatically determine what package that Class exists in and put in the import statement in code. The shortcut keys are <strong>Ctrl-Shift-O</strong> (that’s Control-Shift-“Oh”, not zero), try it out yourself, copy/paste the following code, you’ll see that there are errors with all the <code>Date</code> and <code>Calendar</code> classes. Hit Ctrl-Shift-O, Spring STS will ask you if you want <code>java.util.Date</code> or <code>java.sql.Date</code>, choose <code>java.util.Date</code> and voila! The import code is automatically added to the top of your Class file!</p>

        <h2>Importing ALL Classes in a Package</h2> 
        <p>There is one other “shortcut” method of importing classes in Java, and that’s by using a wildcard (*). Say for instance you just want to import ALL of the classes that belong in the <code>java.util</code> package, you could just use the code <code>import java.util.*</code>. This will import everything in <code>java.util</code>, but it’s important to note that it won’t import any sub-packages within <code>java.util</code>. However, this “importing method” is not really used much anymore due to the fact that IDEs (like Spring STS) are able to import classes automatically by organizing your import (i.e. via Ctrl-Shift-O shortcut key). But for the sake of completion, I figured I would talk about the wildcard imports. Use them if you like!</p> 
        <h2>One final note</h2> 
        <p>For those of you who may be asking the question, “Why can I use things like <code>String</code> or <code>Integer</code> without having to do imports?”. Excellent question and very astute observation! This is because the most commonly used Classes reside in the <code>java.lang</code> package, and this package is essentially auto-imported for us for convenience. How nice of Java to do that for us ?</p> 
        <p>I hope that little Java tutorial helped clear the air on what Java imports are and how they are used. If something is still not clear, please ask your questions in the comments section and I’ll respond ASAP, I promise! So for now, keep coding and keep learning!</p>
