             <div class="post-text" itemprop="text"> 
              <p>Recently I did an exercise for a interview pre-screening, but was not selected. They did not give much detailed feedback apart from "the code was not up to what they wanted".</p> 
              <p>It would be really helpful to understand as to whether the code written is up to production standards and what is it that I could be missing.</p> 
              <p>Here are the requirements:</p> 
              <blockquote> 
               <p>Our business wants to move in the bitcoin market and we’ve been asked to implement the first cut of a simple RFQ (request-for-quote) service. The API looks like this and cannot be changed</p> 
               <pre><code>public interface RfqService {
    Optional&lt;Quote&gt; quoteFor(String currency, int amount);
}

public class Quote {
    public final double bid;
    public final double ask;
    public Quote(double bid, double ask) {
        this.bid = bid;
        this.ask = ask;
    }
}
</code></pre> 
               <p>The job of our engine is to respond with a bid (the price we buy at) and ask (the price we sell at) by finding an exact client offer to match on each side. To do so, our service has access to the company’s live order board which given a single payment currency (e.g. USD) will return a list of clients’ pending orders. </p> 
               <pre><code>public interface LiveOrderBoard {
    List&lt;Order&gt; ordersFor(String currency);
}
</code></pre> 
               <p>The response from our service should be a single bid/ask quote. If the request cannot be fulfilled on both sides by our internal RFQ service, the response should be empty.</p> 
               <pre><code>Bid = highest buy order – X
Ask = lowest sell order + X
</code></pre> 
               <p>The difference between the client order and our quote (i.e. the X amount) is how we make money. X is 0.02 irrespective of the currency.</p> 
               <h3>Worked examples</h3> 
               <p>RFQ for 200 bitcoins paying in USD should return:</p> 
               <pre><code>Bid (we buy) = 232.69
Ask (we sell) = 232.75
</code></pre> 
               <p>RFQ for 500 bitcoins cannot be fulfilled because there’s no matching Sell order. </p> 
               <h3>The solution</h3> 
               <p>Please provide an implementation of the RFQ engine in Java which will allow the business to enter the bitcoin market. No UI or persistence is required but your solution should be able to demonstrate the RFQ business rules given some sample inputs of your choice. We don’t require an implementation of the live orders board. If you find the requirements unclear, you’re free to make any assumptions along the way but please state them explicitly in a readme file within your solution. It should contain everything you would normally check in to your source code management system, and the code should be of production quality.</p> 
              </blockquote> 
              <pre><code>package com.trade.service;

import java.math.BigDecimal;
import java.util.List;
import java.util.Optional;
import java.util.logging.Level;
import java.util.logging.Logger;

/**
  * Implements the RFQService interface. Provides a mechanism to get a      quotes for bitcoins
*/
public class RFQServiceEngine implements RfQService {

    private static Logger log = Logger.getLogger("RFQServiceEngine");

    private LiveOrderBoard liveOrderBoard; //reference to the live order board

    private static final BigDecimal commission = new BigDecimal(0.02); // Our commission

    public RFQServiceEngine(LiveOrderBoard liveOrderBoard) {
        // We have a depencency on the live order board
        this.liveOrderBoard = liveOrderBoard;
    }

    /**
     *  Returns a quote
     * @param currency - The type of currency to trade the bitcoins in
     * @param amount - The number of bitcoins needed
     * @return -  A quote containing the ask and bid price if we can exactly match the request, else an empty quote
     */
    public Optional&lt;Quote&gt; quoteFor(String currency, int amount) {
        try {
            // No use going any further if we don't have a liveorder board OR the amount needed is not positive
            if (liveOrderBoard != null &amp;&amp; amount &gt; 0) {
                // have the whole code in a try catch block , so if the below conversion to currency goes tits up,
                // we log the error and send a empty quote
                final TradeCurrency cur = TradeCurrency.valueOf(currency);
                List&lt;Order&gt; orders = liveOrderBoard.ordersFor(currency);

                Optional&lt;Order&gt; sell = getLowestSellOrder(amount, cur, orders);

                Optional&lt;Order&gt; buy = getHighestBuyOrder(amount, cur, orders);

                if (buy.isPresent() &amp;&amp; sell.isPresent()) {
                    // We deal with price as BigDecimal, but the Quote object needs a double
                    double bid = buy.get().getPrice().subtract(commission).doubleValue();
                    double ask = sell.get().getPrice().add(commission).doubleValue();
                    return Optional.of(new Quote(bid, ask));
                }
            }
            // If we don't find any exact matching Buy or Sell orders
            return Optional.empty();

        } catch (IllegalArgumentException e) {
            //Since the signature of quoteFor does not have a throws exception, we return a empty quote.
            // Logs should be able to highlight that the error occurred
            log.log(Level.SEVERE,"Illegal argument passed to quote for",e);
            return Optional.empty();
        } catch (Exception e) {
            log.log(Level.SEVERE,"Exception occurred in quote for",e);
            return Optional.empty();
        }
    }

    /**
     *  Helper method to get the Highest buy order from the order book
     * @param quantity - No of bitcoins requested
     * @param cur - the currency to trade in
     * @param orders - The list of orders from the live order board
     * @return - the highest buy order for the {cur} and {amount}
     */
    private Optional&lt;Order&gt; getHighestBuyOrder(int quantity, TradeCurrency cur, List&lt;Order&gt; orders) {
        return orders.stream().
                            filter(order -&gt; cur.equals(order.getTradeCurrency()) &amp;&amp;
                                    TradeDirection.BUY.equals(order.getTradeDirection()) &amp;&amp;
                                    quantity == order.getQuantity()).
                            max((p1, p2) -&gt; p1.getPrice().compareTo(p2.getPrice()));
    }



    /**
     *  Helper method to get the lowest sell order from the order book
     * @param quantity - No of bitcoins requested
     * @param cur - the currency to trade in
     * @param orders - The list of orders from the live order board
     * @return - the lowest sell order for the {cur} and {amount}
     */
    private Optional&lt;Order&gt; getLowestSellOrder(int quantity, TradeCurrency cur, List&lt;Order&gt; orders) {
        return orders.stream().
                            filter(order -&gt; cur.equals(order.getTradeCurrency()) &amp;&amp;
                                    TradeDirection.SELL.equals(order.getTradeDirection()) &amp;&amp;
                                    quantity == order.getQuantity()).
                            min((p1, p2) -&gt; p1.getPrice().compareTo(p2.getPrice()));

    }
}
</code></pre>

              <p>In these screening situations, you only get a few seconds to make a good first impression. Unfortunately, this line already makes a bad impression:</p> 
              <blockquote> 
               <pre><code>public class RFQServiceEngine implements RfQService {
</code></pre> 
              </blockquote> 
              <p>The inconsistent capitalization shows a <strong>lack of attention to detail</strong>. If the problem explicitly states that you have to implement the given <code>RfqService</code> interface exactly, you can't write <code>implements RfQService</code>. And the class should be named <code>RfqServiceEngine</code>.</p> 
              <hr> 
              <p>The <code>LiveOrderBoard</code> should be passed in like this:</p> 
              <pre><code>private final LiveOrderBoard orders;

public RfqServiceEngine(LiveOrderBoard liveOrderBoard) {
    if (liveOrderBoard == null) {
        throw new NullPointerException();
    }
    this.orders = liveOrderBoard;
}
</code></pre> 
              <p>Marking the <code>LiveOrderBoard</code> as a <code>final</code> field makes it clear that it cannot be swapped out after the constructor. I don't think it makes any sense to allow it to be <code>null</code>, and it makes even less sense to decline all future quotes as a result of it being <code>null</code>. (The last thing a financial firm wants is to lose millions of dollars due to a hidden bug. If something is wrong, just make it <strong>fail fast</strong>.)</p> 
              <p>Don't write <strong>worthless comments</strong> like this — it's a dead giveaway that you are a beginner.</p> 
              <blockquote> 
               <pre><code>//reference to the live order board
</code></pre> 
              </blockquote> 
              <hr> 
              <p>It is probably good that you used <code>BigDecimal</code> to represent money, to avoid floating-point imprecision. You didn't have to do it, since the specified interface has <code>double</code>s. Personally, if I were given this problem for an interview, I would just go with <code>double</code>, but write a big comment like this in protest:</p> 
              <pre><code>// FIXME: Money amounts should ideally be represented using BigDecimal,
// since doubles can lead to rounding errors.  Unfortunately, we are
// stuck with doubles since that is what the Quote class uses.
private static final double commission = 0.02;
</code></pre> 
              <p>However, having chosen to use <strong><code>BigDecimal</code></strong>, you screwed it up by writing <code>new BigDecimal(0.02)</code> instead of <code>new BigDecimal("0.02")</code>. The <a href="https://docs.oracle.com/javase/8/docs/api/java/math/BigDecimal.html#BigDecimal-double-">JavaDoc</a> warns you that the <code>BigDecimal(double)</code> constructor is unpredictable, but you should be able to figure out yourself that writing <code>0.02</code> already risks losing precision.</p> 
              <p>Here also, <code>// Our commission</code> is a superfluous comment.</p> 
              <hr> 
              <p>The question simply represented currencies as strings. <strong>What is this <code>TradeCurrency</code> class?</strong> </p> 
              <blockquote> 
               <pre><code>final TradeCurrency cur = TradeCurrency.valueOf(currency);
</code></pre> 
              </blockquote> 
              <p>Did you include the <code>TradeCurrency</code> class with your submission? Why did you invent something to overcomplicate a simple problem?</p> 
              <hr> 
              <p>You calculated the bid and ask prices, but did you actually <strong>check that the spread yields a positive profit</strong>? That may be the logic flaw that doomed your chances with the company — imagine how much money the firm could lose by accidentally making unprofitable trades!</p>

              <h3><code>Comparator</code> usage</h3> 
              <p>In Java 8, you can use <a href="https://docs.oracle.com/javase/8/docs/api/java/util/Comparator.html#comparing-java.util.function.Function-" rel="nofollow"><code>Comparator.comparing(Function)</code></a> to give you a <code>Comparator</code> on the <code>Order.getPrice()</code> method, when you use it as a <a href="https://docs.oracle.com/javase/tutorial/java/javaOO/methodreferences.html" rel="nofollow">method reference</a>:</p> 
              <pre><code>Comparator&lt;Order&gt; BY_PRICE = Comparator.comparing(Order::getPrice);
</code></pre> 
              <h3>Code deduplication</h3> 
              <p>Both <code>getHighestBuyOrder</code> and <code>getLowestSellOrder</code> are similar except for one of the predicate conditions and the terminal operation (<code>max(Comparator)</code> vs <code>min(Comparator)</code> respectively).</p> 
              <p>As such, you can consider the following refactoring:</p> 
              <pre><code>private static Stream&lt;Order&gt; stream(List&lt;Order&gt; orders, TradeCurrency currency,
            TradeDirection direction, int quantity) {
    // assuming TradeCurrency and TradeDirection are enums
    return orders.stream()
                    .filter(o -&gt; currency == o.getTradeCurrency())
                    .filter(o -&gt; direction == o.getTradeDirection())
                    .filter(o -&gt; quantity == o.getQuantity());
}

private static Optional&lt;Order&gt; getHighestBuyOrder(List&lt;Order&gt; orders,
            TradeCurrency currency, int quantity) {
    return stream(orders, currency, TradeDirection.BUY, quantity).max(BY_PRICE);
}

private static Optional&lt;Order&gt; getLowestSellOrder(List&lt;Order&gt; orders,
            TradeCurrency currency, int quantity) {
    return stream(orders, currency, TradeDirection.SELL, quantity).min(BY_PRICE);
}
</code></pre> 
              <h3>Exception handling</h3> 
              <p>Avoid catching on a generic <code>Exception</code> as it can cause confusion in the long run. For starters, it adversely affects logic control the moment you encounter any kind of exception - checked or runtime, minor or major.</p> 
              <p>It also potentially hides newer errors when you change your code: maybe you need to be concerned about a new checked exception that you can safely resume processing the remaining results, but because of this broad <code>catch Exception</code> condition, you'll be left wondering why you sometimes/always end up with an empty <code>Optional</code>.</p> 
              <p>Also, the <code>throws ...</code> declaration is for <em>checked</em> exceptions, so you are still able throw runtime exceptions, if it's severe enough.</p> 
              <h3><code>return</code> early</h3> 
              <p>If you invert your <code>if</code> condition check on <code>liveOrderBoard</code> and <code>amount</code> to <code>return</code> early, you can reduce one level of nesting.</p> 
              <h3>Style</h3> 
              <pre><code>public class RFQServiceEngine implements RfQService
</code></pre> 
              <p>Since <code>RfqService</code> is given in proper <code>camelCasing</code> (for some definition of 'proper'), you may want to stick to that convention and call your class <code>RfqServiceEngine</code>.</p> 
              <p>I noticed that in your <code>getHighestBuyOrder</code> and <code>getLowestSellOrder</code> methods, the dot (<code>.</code>) is separated from the method call when spanning across lines. I think that's slightly <em>unconventional</em>, unless it already is your convention. :)</p> 
              <p>Also, you should be using <code>{@code cur}</code> instead of <code>{cur}</code> in your Javadoc for the monospaced font to be used and rendered correctly.</p> 
              <h3>Putting it altogether</h3> 
              <p>The refactored code for <code>quoteFor(String, int)</code> can look something like:</p> 
              <pre><code>public Optional&lt;Quote&gt; quoteFor(String currency, int amount) {
    if (liveOrderBoard == null) {
        // illustrating plausible runtime exceptions
        throw new IllegalStateException("No live order board configured.");
    }
    if (currency &lt;= 0) {
        return Optional.empty();
    }
    TradeCurrency tradeCurrency;
    try {
        tradeCurrency = TradeCurrency.valueOf(currency);
    } catch (IllegalArgumentException | NullPointerException e) {
        // log something first?
        return Optional.empty();
    }
    List&lt;Order&gt; orders = liveOrderBoard.ordersFor(currency);
    Optional&lt;Order&gt; buy = getHighestBuyOrder(order, tradeCurrency, amount);
    Optional&lt;Order&gt; sell = getLowestSellOrder(orders, tradeCurrency, amount);
    if (buy.isPresent() &amp;&amp; sell.isPresent()) {
        double bid = buy.get().getPrice().subtract(commission).doubleValue();
        double ask = sell.get().getPrice().add(commission).doubleValue();
        return Optional.of(new Quote(bid, ask));
    }
    return Optional.empty();
}
</code></pre>

              <p><strong>Fail Early</strong></p> 
              <p>Your class has a dependency on <code>liveOrderBoard</code>. Check it in your constructor and throw an exception if it is null, don't defer the check until you use it. It's better to not let your class be constructed in an inconsistent state than to have to check it for every public method.</p> 
              <p><strong>Tests</strong></p> 
              <p>One of the requirements was that you submit everything you would normally check into source control. I would expect to at least see tests that covered the example cases provided (you may have written them, but not added them here for review). Testing styles vary and can be the source of many string opinions so it it worth considering them as much as the 'production' code.</p> 
              <p><strong>Comments</strong></p> 
              <p>It is very tempting to over comment code when submitting it for an interview. I would try to avoid it, the code should be as much like your real code as possible. Write your code so that what you are doing and why is as obvious as you can and reserve comments for more complex explanations. I know you save the live order board in your constructor because you have a dependency, you don't need to tell me...</p>
