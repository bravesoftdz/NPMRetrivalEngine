       <div class="entry-content single"> 
        <p><em>09/04/2017: post updated to include a reference to desktop applications developed with C++, since CPP projects are supported as well by the Desktop Bridge Packaging Project.</em></p> 
        <p><a href="https://blogs.msdn.microsoft.com/appconsult/2017/02/21/an-easier-way-to-generate-the-packages-for-a-desktop-bridge-converted-app/" target="_blank">In one of the previous posts</a> I’ve written on this blog we’ve seen how, thanks to a combination of a JavaScript project for the Universal Windows Platform and a special Visual Studio extension called <strong>Desktop Bridge Debugging Project</strong>, it is possible to create a packaged version of a .NET desktop application in an easier way. Thanks to this approach, you are able to edit some parts of the manifest in a visual way; generate the app packages for sideloading or Store distribution using the same Visual Studio wizard used for regular UWP applications; debug the packaged version of your application.</p> 
        <p>However, the experience wasn’t 100% optimal: since the JavaScript project of the Universal Windows Platform wasn’t born for this specific scenario, but it was used as a workaround to avoid the compilation of the packaged application, there was still some manual work to do. However, I have a great news for you: the packaging experience is about to become a lot better and you can try it starting from today! Visual Studio 2017 Update 4, in fact, will include a new project called <strong>Desktop Bridge Packaging Project</strong>, which will make the overall experience of packaging a .NET desktop application using Visual Studio a whole lot easier. You can start playing with this new project today, since last week Microsoft has released the first preview of the Update 4 through the separate <strong>Visual Studio Preview </strong>channel, which is a different edition of Visual Studio that can be installed side-by-side with the regular one and that contains all the features that are still in beta. When they will be finalized, they will be moved from the preview to the stable channel.</p> 
        <p>Let’s see how to use the Desktop Bridge Packaging Project and how it can make our life much easier.</p> 
        <h3>Installing Visual Studio Preview and packaging a .NET application</h3> 
        <p>The first step is to install <strong>Visual Studio 2017 Preview</strong>, which you can get from <a title="https://www.visualstudio.com/vs/preview/" href="https://www.visualstudio.com/vs/preview/">https://www.visualstudio.com/vs/preview/</a>. Make sure to install, as minimum requirement for the purpose of this post:</p> 
        <ol> 
         <li>The tools for developing Universal Windows Platform apps. </li>
         <li>The tools for developing desktop application based on the .NET Framework (like Windows Forms or WPF) or C++.</li> 
        </ol> 
        <p>Now let’s start by opening your .NET desktop application with Visual Studio 2017 Preview. For the purpose of this post, I’m going to reuse a simple Windows Forms application that I wrote a while ago as a sample for the Convert phase of the Desktop Bridge journey, which you can find on GitHub at <a title="https://github.com/qmatteoq/DesktopBridge/tree/master/3.%20Convert/Convert" href="https://github.com/qmatteoq/DesktopBridge/tree/master/3.%20Convert/Convert">https://github.com/qmatteoq/DesktopBridge/tree/master/3.%20Convert/Convert</a> However, the new Desktop Bridge Packaging Project will work with any Windows Forms, WPF project or C++ project. Just remember that, in case you’re using one of the first two technologies, the minimum version of the .NET Framework supported by the Desktop Bridge is 4.0, even if it’s highly recommended to target at least .NET 4.6.1.</p> 
        <p>The application is very simple. The UI includes two buttons, which both create a text file on the desktop: the only difference is that the first one includes a sample text, the second one saves instead a serialized version of an entity that represents a person by using the popular JSON.NET library. The goal of this second sample was to show how to handle, in a packaged version of a desktop application, a dependency from a 3rd party library. Both operations create the file directly on the desktop of the user, which is a task that a Universal Windows Platform app would need to perform in a different way (for example, by using the <strong>FileSavePicker</strong> API to ask to the user where he wants to save the file).</p> 
        <p>Now let’s right click on the Visual Studio solution and choose <strong>Add –&gt; New project</strong>. In the Universal Windows Platform section, you will find a new template called <strong>Desktop Bridge Packaging Project:</strong></p> 
        <p><a href="https://msdnshared.blob.core.windows.net/media/2017/08/packaging-project.png"><img title="packaging project" style="border-left-width: 0px;border-right-width: 0px;border-bottom-width: 0px;padding-top: 0px;padding-left: 0px;padding-right: 0px;border-top-width: 0px" border="0" alt="packaging project" src="https://msdnshared.blob.core.windows.net/media/2017/08/packaging-project_thumb.png" width="800" height="555"></a></p> 
        <p>Give to the project a meaningful name (I like to use the naming convention <strong>&lt;name of the original desktop project&gt;.Package</strong>) and press <strong>OK.</strong> The first thing you will be asked is which version of the Universal Windows Platform SDK you want to target. This choice doesn’t really have a real impact, since we’re talking about a packaged version of a desktop application and not a full Universal Windows Platform app. However, make sure to choose, as Minimum version, at least the 14393 SDK (Anniversary Update), since it’s the first edition of Windows that added support to the Desktop Bridge. If you target a prior version as minimum, you’ll get an error during the first compilation.</p> 
        <p><a href="https://msdnshared.blob.core.windows.net/media/2017/08/image522.png"><img title="image" style="border-left-width: 0px;border-right-width: 0px;border-bottom-width: 0px;padding-top: 0px;padding-left: 0px;padding-right: 0px;border-top-width: 0px" border="0" alt="image" src="https://msdnshared.blob.core.windows.net/media/2017/08/image_thumb416.png" width="800" height="321"></a></p> 
        <p>&nbsp;</p> 
        <p>This how your solution will look like:</p> 
        <p><a href="https://msdnshared.blob.core.windows.net/media/2017/08/image523.png"><img title="image" style="border-left-width: 0px;border-right-width: 0px;border-bottom-width: 0px;padding-top: 0px;padding-left: 0px;padding-right: 0px;border-top-width: 0px" border="0" alt="image" src="https://msdnshared.blob.core.windows.net/media/2017/08/image_thumb417.png" width="558" height="480"></a></p> 
        <p>&nbsp;</p> 
        <p>As you can see, the <strong>Convert.Package </strong>project looks like a Universal Windows Platform one. It has an <strong>Assets</strong> folder, where the various images for icons and tiles are stored; it has a <strong>Package.appxmanifest</strong>, which is the manifest file of the application; it has a temporary certificate, which is used to sign the package for testing purposes. However, instead of having a <strong>References</strong> section with the list of all the libraries used by the application, it has a new section called <strong>Applications.</strong> Its purpose is exactly to specify which applications should be included in the app package. Right click on it, choose <strong>Add reference </strong>and you will see a list automatically filtered to display only the other projects that are part of the solution which are full applications and not just libraries. In my case, for example, I’m able to see only the Windows Forms app called <strong>Convert</strong>, I can’t add a reference to a generic library:</p> 
        <p><a href="https://msdnshared.blob.core.windows.net/media/2017/08/image524.png"><img title="image" style="border-left-width: 0px;border-right-width: 0px;border-bottom-width: 0px;padding-top: 0px;padding-left: 0px;padding-right: 0px;border-top-width: 0px" border="0" alt="image" src="https://msdnshared.blob.core.windows.net/media/2017/08/image_thumb418.png" width="800" height="552"></a></p> 
        <p>Click on it and you will find the <strong>Convert </strong>application listed under <strong>Applications</strong>, like if it’s a normal reference. Now there’s another step to take which is required because, <a href="https://blogs.msdn.microsoft.com/appconsult/2016/11/28/desktop-bridge-multiple-desktop-processes-a-single-uwp-container/" target="_blank">as we have seen in another post on this blog</a>, an app package can contain multiple executables, which can be launched between each other. This is a very important scenario to keep in mind, especially <a href="https://blogs.msdn.microsoft.com/appconsult/2017/06/15/porting-and-testing-your-classic-desktop-applications-on-windows-10-s-with-the-desktop-bridge/" target="_blank">when it comes to Windows 10 S compatibility</a>: a Desktop Bridge app running on Windows 10 S, in fact, can launch another process using the <strong>Process.Start()</strong> method of the .NET Framework only if it’s part of the same app package. If it’s an external process, you will get an exception and the operation will fail. However, despite an app package can contain multiple executables, it can have only one entry point, which is the process that is launched when the user clicks on the tile or on the icon in the Start menu. As such, by using the <strong>Applications</strong> section of the Desktop Bridge Packaging Project, we can add a reference to multiple applications, in case our Visual Studio solution contains multiple .NET desktop projects. However, only one of them can be the starting point, which is the one that, if you remember when we had to manually edit the manifest file, was set using the <strong>&lt;Application&gt; </strong>tag like in the following example:</p> 
        <pre class="brush: xml;">&lt;Application Id="Convert" Executable="win32\Convert.exe" EntryPoint="Windows.FullTrustApplication"&gt;
</pre> 
        <p>Thanks to the new Packaging project, we don’t have anymore to manually set the entry point, but it’s enough to right click on the application we want in the <strong>Applications </strong>section (in my sample, the <strong>Convert</strong> one) and choose <strong>Set as entry point</strong>.</p> 
        <p><a href="https://msdnshared.blob.core.windows.net/media/2017/08/image525.png"><img title="image" style="border-left-width: 0px;border-right-width: 0px;border-bottom-width: 0px;padding-top: 0px;padding-left: 0px;padding-right: 0px;border-top-width: 0px" border="0" alt="image" src="https://msdnshared.blob.core.windows.net/media/2017/08/image_thumb419.png" width="800" height="300"></a></p> 
        <p>That’s all regarding the basic configuration. Now you can just set the <strong>Convert.Package</strong> as startup project, choose a specific CPU architecture (x86 or x64) and press the <strong>Start debugging</strong> button. The packaged version of your desktop application will be deployed and the debugger automatically attached, giving you in just one step both the advantages of the previous approach based on JavaScript and Desktop Bridge Debugging Project: automatic generation and deployment of the AppX file and a real-time debugging experience of the converted version. The debugging experience is very helpful especially if, <a href="https://blogs.msdn.microsoft.com/appconsult/2016/11/03/desktop-bridge-identify-the-applications-context/" target="_blank">in combination with the context identification library available on NuGet</a>, you are planning to detect if your application is running as a classic desktop one or packaged through the Desktop Bridge and, only in the second case, take advantage of some APIs of the Universal Windows Platform (for example, to display a toast notification, update a live tile or integrate voice commands with Cortana).</p> 
        <p>As you can notice, compared to the previous approach based on the JavaScript project, there are multiple advantages:</p> 
        <ol> 
         <li>There is no need anymore to manually setup a post build process that takes care of copying the output of the build of the Win32 project inside a folder of the JavaScript project. </li>
         <li>No more manual editing of the manifest file. Out of the box, the included manifest is already tailored for a Desktop Bridge application. You just need to set which is the entry point but, as you have seen, the operation is done with a simple right click. </li>
         <li>The manifest of a Desktop Bridge is fully supported by the Visual Studio editor. If you remember what happened when you opened the <strong>Package.appxmanifest</strong> file of a JavaScript project used in a Desktop Bridge scenario, there was a persistent error in the <strong>Application</strong> tab of the editor, since the entry point of the manifest wasn’t recognized as valid: it was expecting the path and name of the HTML page used as entry point of the UWP application. Now, instead, it’s set to the special value <strong>Windows.FullTrustApplication</strong> and correctly recognized.<br> <p><a href="https://msdnshared.blob.core.windows.net/media/2017/08/image526.png"><img title="image" style="border-left-width: 0px;border-right-width: 0px;border-bottom-width: 0px;padding-top: 0px;padding-left: 0px;padding-right: 0px;border-top-width: 0px" border="0" alt="image" src="https://msdnshared.blob.core.windows.net/media/2017/08/image_thumb420.png" width="800" height="173"></a></p> </li>
         <li> <p><a href="https://blogs.msdn.microsoft.com/appconsult/2017/02/21/an-easier-way-to-generate-the-packages-for-a-desktop-bridge-converted-app/" target="_blank">If you remember the blog post</a> about setting up the JavaScript project, there was some extra work to do in case your desktop application at some point started to have a dependency from a new library. By default, in fact, thanks to the post-build task, all the DLLs that were part of the library were copied inside the JavaScript project, but they weren’t included in the app package, triggering an error at runtime related to one or more missing DLLs if you forgot to include them in the build output. Now, instead, everything is handled automatically by the Desktop Bridge Packaging Project. As you may have noticed, the application used in this sample has a dependency from the library <strong>Newtonsoft.Json.dll</strong> (JSON.NET, used for the serialization of JSON data), but everything is completely transparent: the packaged version of our desktop application is deployed and launched by Visual Studio without having to perform any extra step.</p> </li> 
        </ol> 
        <p>Additionally, <a href="https://blogs.msdn.microsoft.com/appconsult/2016/10/17/desktop-bridge-converting-an-installer/" target="_blank">compared to the Desktop App Converter</a> and <a href="https://blogs.msdn.microsoft.com/appconsult/2016/10/28/desktop-bridge-manually-converting-a-desktop-application/" target="_blank">the manual approach to create a package</a>, you keep getting the same advantages that were offered also by the JavaScript project and Visual Studio 2017:</p> 
        <ol> 
         <li>Automatic generation of all required assets starting from a single image, thanks to the new tool added in Visual Studio 2017 in the <strong>Assets </strong>section of the manifest visual editor. </li>
         <li>Generation of the packages (both for the Store and for sideloading) using the Visual Studio wizard (available when you right click on the Desktop Bridge Packaging Project and you choose <strong>Store –&gt; Create App Packages</strong>), instead of having to deal with command line tools like <strong>makeappx</strong> or <strong>signtool.</strong> </li>
         <li>Editing of the information stored in the manifest (like application’s name, capabilities, extensions, etc.) in a visual way, instead of having to manually edit the XML file. </li>
         <li>In case of Store publishing (which, remember, requires first <a href="https://developer.microsoft.com/en-us/windows/projects/campaigns/desktop-bridge" target="_blank">to nominate your application</a>, since your account requires a special capability), you can easily assign an identity to the application from one of the names you have reserved on the Dev Center thanks to the Visual Studio wizard <strong>Store –&gt; Associate App with the Store. </strong>Previously, you needed to manually copy the required information (like publisher or identity name) from the Dev Center website and paste them inside the XML of the manifest file.</li> 
        </ol> 
        <h3></h3> 
        <h3>Wrapping up</h3> 
        <p>As you have seen, using the Desktop Bridge to package a desktop application based on the .NET Framework is easier than ever. This way, you can take advantage of all the benefits of the Desktop Bridge technology: you can distribute it using the Store or the Store for Business (other than the traditional distribution channels); you can improve the deployment experience; you can start leveraging the benefits of the Universal Windows Platform and plan a migration path without having to immediately rewrite from scratch your existing desktop application. You can download the sample project used in this post from <a title="https://github.com/qmatteoq/DesktopBridge/tree/master/Extras/PackagingProject" href="https://github.com/qmatteoq/DesktopBridge/tree/master/Extras/PackagingProject">https://github.com/qmatteoq/DesktopBridge/tree/master/Extras/PackagingProject</a> Just remember that, to open it, you need to install Visual Studio 2017 Preview first from <a title="https://www.visualstudio.com/vs/preview/" href="https://www.visualstudio.com/vs/preview/">https://www.visualstudio.com/vs/preview/</a>. If you’ll try to open it with the regular Visual Studio 2017 version, you will get an error, since the Desktop Bridge Packaging Project won’t be recognized.</p> 
        <p>And if you’re a Windows developer, don’t miss also another important feature coming with Visual Studio 2017 Update 4: support to .NET Standard Libraries 2.0 in UWP apps built with the upcoming Fall Creators Update SDK. Thanks to this new feature, you’ll be able to leverage lot of highly requested APIs from the .NET Framework also in a UWP application, like <strong>DataSet</strong> or <strong>SqlClient</strong> to perform queries directly on a database. You can learn more in the following blog post: <a title="https://blogs.msdn.microsoft.com/dotnet/2017/08/25/uwp-net-standard-2-0-preview/" href="https://blogs.msdn.microsoft.com/dotnet/2017/08/25/uwp-net-standard-2-0-preview/https://blogs.msdn.microsoft.com/dotnet/2017/08/25/uwp-net-standard-2-0-preview/">https://blogs.msdn.microsoft.com/dotnet/2017/08/25/uwp-net-standard-2-0-preview/</a><a title="https://blogs.msdn.microsoft.com/dotnet/2017/08/25/uwp-net-standard-2-0-preview/" href="https://blogs.msdn.microsoft.com/dotnet/2017/08/25/uwp-net-standard-2-0-preview/">https://blogs.msdn.microsoft.com/dotnet/2017/08/25/uwp-net-standard-2-0-preview/</a></p> 
        <p>Happy packaging!</p>
