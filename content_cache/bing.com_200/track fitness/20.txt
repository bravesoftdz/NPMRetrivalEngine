         <p>The <code>:login</code> key passed to <code>ActiveShipping::FedEx.new()</code> is really the FedEx meter number, not the FedEx login.</p> 
         <p>When developing with test credentials, be sure to pass <code>test: true</code> to <code>ActiveShipping::FedEx.new()</code>.</p> 
         <h2><a id="user-content-tests" class="anchor" href="#tests" aria-hidden="true">
           <svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewbox="0 0 16 16" width="16">
            <path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path>
           </svg></a>Tests</h2> 
         <p>You can run the unit tests with:</p> 
         <pre><code>bundle exec rake test:unit
</code></pre> 
         <p>and the remote tests with:</p> 
         <pre><code>bundle exec rake test:remote
</code></pre> 
         <p>The unit tests mock out requests and responses so that everything runs locally, while the remote tests actually hit the carrier servers. For the remote tests, you'll need valid test credentials for any carriers' tests you want to run. The credentials should go in <a href="https://github.com/Shopify/active_shipping/blob/master/test/credentials.yml"><code>~/.active_shipping/credentials.yml</code></a>. For some carriers, we have public credentials you can use for testing in <code>.travis.yml</code>. Remote tests missing credentials will be skipped.</p> 
         <h2><a id="user-content-contributing" class="anchor" href="#contributing" aria-hidden="true">
           <svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewbox="0 0 16 16" width="16">
            <path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path>
           </svg></a>Contributing</h2> 
         <p>See <a href="https://github.com/Shopify/active_shipping/blob/master/CONTRIBUTING.md">CONTRIBUTING.md</a>.</p> 
         <p>We love getting pull requests! Anything from new features to documentation clean up.</p> 
         <p>If you're building a new carrier, a good place to start is in the <a href="https://github.com/Shopify/active_shipping/blob/master/lib/active_shipping/carrier.rb"><code>Carrier</code> base class</a>.</p> 
         <p>It would also be good to familiarize yourself with <a href="https://github.com/Shopify/active_shipping/blob/master/lib/active_shipping/location.rb"><code>Location</code></a>, <a href="https://github.com/Shopify/active_shipping/blob/master/lib/active_shipping/package.rb"><code>Package</code></a>, and <a href="https://github.com/Shopify/active_shipping/blob/master/lib/active_shipping/response.rb"><code>Response</code></a>.</p> 
         <p>You can use the <a href="https://github.com/Shopify/active_shipping/blob/master/test/console.rb"><code>test/console.rb</code></a> to do some local testing against real endpoints.</p> 
         <p>To log requests and responses, just set the <code>logger</code> on your Carrier class to some kind of <code>Logger</code> object:</p> 
         <div class="highlight highlight-source-ruby">
          <pre><span class="pl-c1">ActiveShipping</span>::<span class="pl-c1">USPS</span>.logger <span class="pl-k">=</span> <span class="pl-c1">Logger</span>.<span class="pl-k">new</span>(<span class="pl-c1">STDOUT</span>)</pre>
         </div>

           <svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewbox="0 0 16 16" width="16">

           </svg></a>Anatomy of a pull request</h3> 
         <p>Any new features or carriers should have passing unit <em>and</em> remote tests. Look at some existing carriers as examples.</p> 
         <p>When opening a pull request, include description of the feature, why it exists, and any supporting documentation to explain interaction with carriers.</p>
