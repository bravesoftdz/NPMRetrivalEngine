       <h2><a href="showtutorial.php?tutorialid=291">A Tutorial on importing packages in Java</a></h2> 
       <p class="para"><strong>By:</strong> Emiley J <strong>Viewed:</strong> 155346 times &nbsp;<a href="forprinting.php?tutorialid=291">

       <p class="para">Given that packages exist and are a good mechanism for compartmentalizing diverse classes from each other, it is easy to see why all of the built-in Java classes are stored in packages. There are no core Java classes in the unnamed default package; all of the standard classes are stored in some named package. Since classes within packages must be fully qualified with their package name or names, it could become tedious to type in the long dot-separated package path name for every class you want to use. For this reason, Java includes the <b>import </b>statement to bring certain classes, or entire packages, into visibility. Once imported, a class can be referred to directly, using only its name. The<b>import </b>statement is a convenience to the programmer and is not technically needed to<br>write a complete Java program. If you are going to refer to a few dozen classes in your application, however, the <b>import </b>statement will save a lot of typing.</p> 
       <p class="para">In a Java source file, <b>import </b>statements occur immediately following the <b>package</b> statement (if it exists) and before any class definitions. This is the general form of the <b>import</b>statement:</p> 
       <code>import <i>pkg1</i>[.<i>pkg2</i>].(<i>classname</i>|*);</code> 
       <p class="para">Here, <i>pkg1 </i>is the name of a top-level package, and <i>pkg2 </i>is the name of a subordinate package inside the outer package separated by a dot (<b>.</b>). There is no practical limit on the depth of a package hierarchy, except that imposed by the file system. Finally, you specify either an explicit <i>classname </i>or a star (<b>*</b>), which indicates that the Java compiler should import the entire package. This code fragment shows both forms in use:</p> 
       <code>import java.util.Date; <br>import java.io.*;</code> 
       <blockquote>
         Caution: The star form may increase compilation time—especially if you import several large packages. For this reason it is a good idea to explicitly name the classes that you want to use rather than importing whole packages. However, the star form has absolutely no effect on the run-time performance or size of your classes. 
       </blockquote> 
       <p class="para">All of the standard Java classes included with Java are stored in a package called <b>java</b>. The basic language functions are stored in a package inside of the <b>java </b>package called <b>java.lang</b>. Normally, you have to import every package or class that you want to use, but since Java is useless without much of the functionality in <b>java.lang</b>,it is implicitly imported by the compiler for all programs. This is equivalent to the following line being at the top of all of your programs:&nbsp;</p> 
       <code>import java.lang.*;</code> 
       <p class="para">If a class with the same name exists in two different packages that you import using the star form, the compiler will remain silent, unless you try to use one of the classes. In that case, you will get a compile-time error and have to explicitly name the class specifying its package. Any place you use a class name, you can use its fully qualified name, which includes its full package hierarchy. For example, this fragment uses an import statement:</p> 
       <code>import java.util.*; <br>class MyDate extends Date { <br>}</code> 
       <p class="para">The same example without the <b>import </b>statement looks like this:</p> 
       <code>class MyDate extends java.util.Date { <br>}</code> 
       <p class="para">When a package is imported, only those items within the package declared as <b>public </b>will be available to non-subclasses in the importing code. For example, if you want the <b>Balance </b>class of the package <b>MyPack </b>shown earlier to be available as a stand-alone class for general use outside of <b>MyPack</b>, then you will need to declare it as <b>public </b>and put it into its own file, as shown here:</p> 
       <code> package MyPack; <br>/* Now, the Balance class, its constructor, and its <br>show() method are public. This means that they can <br>be used by non-subclass code outside their package. <br>*/ <br>public class Balance { <br>String name; <br>double bal; <br>public Balance(String n, double b) { <br>name = n; <br>bal = b; <br>} <br>public void show() { <br>if(bal&lt;0) <br>System.out.print("—&gt; "); <br>System.out.println(name + ": $" + bal); <br>} <br>} </code> 
       <p class="para">As you can see, the <b>Balance </b>class is now <b>public</b>. Also, its constructor and its <b>show( )</b> method are <b>public</b>, too. This means that they can be accessed by any type of code outside the <b>MyPack </b>package. For example, here <b>TestBalance</b>imports <b>MyPack </b>and is then able to make use of the <b>Balance </b>class:</p> 
       <code> import MyPack.*; <br>class TestBalance { <br>public static void main(String args[]) { <br>/* Because Balance is public, you may use Balance <br>class and call its constructor. */ <br>Balance test = new Balance("J. J. Jaspers", 99.88); <br>test.show(); // you may also call show() <br>} <br>} </code> 
       <p class="para">As an experiment, remove the <b>public </b>specifier from the <b>Balance</b>class and then try <br>compiling <b>TestBalance</b>. As explained, errors will result.</p>
