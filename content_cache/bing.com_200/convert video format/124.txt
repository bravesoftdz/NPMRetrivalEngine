                <div class="jive-rendered-content">
                 <p>Keep in mind that .f4f format is directly playable through many Flash Players on the web, also the hls-vod and hds-vod Apache modules can produce the manifest and playlist .f4m and .m3u8 on the fly with the .f4f as well. I would recommend leaving the files in .f4f as if they become corrupted, they are easily recoverable. Eseentially when you scrub a .f4f to .mp4 losslessly you are removing the interval metadata stored within the file rendering it subsceptible to not being recoverable if something would happen to the media the data is contained on.</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p>It is also a good idea to "rename" the entire contents of a /stream/_definst_/ live stream as the meta data and control files can be used to generate .m3u8 manifests when streaming over HLS and future developments of software come about you will have the entire set of data of the storeed live stream.&nbsp; This is essentialy what the new HLS Packager that is bundled with AMS 5.x.x when taking .mp4 from other sources to generate your manifests and playlists .m3u8 and stream via HTTP HLS.</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p>You can essentially modify the main.asc on interactive and enterprise editions of AMS to simply "Rename" the stream directories, and then sym link the /stream/_definst_/ to the /application/vod/media to directly replay a live stream through RTMP VOD, or hls-vod or hds-vod seamlessly. Renaming the directories is a perfect solution rather than "copying" as the main.asc if copying a large stream will render the "pkgr" application not responsive to immediately rebroadcast a stream until the contents of the directory is copied to another location.&nbsp; You can then pass particular "stream id strings" in which the main.asc can parse and perfom a specific function, as by defualt "record" and "append" modes preform those fucntions as such.&nbsp; append meaning adding to the existing .f4f, and record meaning to erase and start over the .f4f.&nbsp; You could pass a string to rename the directory to what the string parameter contains. i.e.&nbsp;&nbsp; &amp;adbe-rename-mode=rename.to.%"</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p>Hope this helps you utilize the FMS &amp; AMS suites effectively in your application.</p>

                <div class="jive-rendered-content">
                 <p>We needed to convert some files in Wowza's format (f4v/f4a atoms) to mp4 so we could use them with AMS.&nbsp; Here's what I did:</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p>Used this script: <a class="jive-link-external-small" href="https://forums.adobe.com/external-link.jspa?url=https%3A%2F%2Fgithub.com%2FK-S-V%2FScripts%2Fwiki" rel="nofollow" target="_blank">https://github.com/K-S-V/Scripts/wiki</a></p>
                 <p>Looks like for your case you can use it to do offline processing of your f4f fragments.</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p>This script puts the files in an FLV container, but I wanted mp4 as you do.&nbsp; For this I used ffmpeg to do "passthrough" re-encoding, basically rebox it in the mp4 container without any transcoding.&nbsp; Note - I had trouble with ffmpeg versions older than v1.0.1 with audio/video out of sync.&nbsp; The newest RPM version of it was way out of date for CentOS so I had to manually compile/install v1.0.1.</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p>I used this syntax and it worked for me:</p>
                 <p>ffmpeg -i input_file.flv -acodec copy -vcodec copy output_file.mp4</p>
                 <p><br>Also, if you're looking to script this, you can add "-loglevel panic" to make ffmpeg silent.&nbsp; Hope this helps, it took my team and I a lot of research to work this out.</p>

                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p>It is very simple to remux an MP4 ISO by using ffmpeg from the command line. The latest version of ffmpeg will take the HLS .m3u8 stream and convert to to ISO MP4 in real time.</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p>You can have the main.asc call a php script to call ffmpeg, and start recording to a MP4 ISO container on every publish to the livepkgr application.</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p>You can then complete set the server to default dismanagement and duration, as the .f4f's can be discarded, as they are merely inplace to provide a cache store for your clients to watch the live video.</p>
                 <p style="min-height: 8pt; padding: 0px;">&nbsp;</p>
                 <p>It is then very simple to symlink a directory store from /webroot/vod/ of the MP4 ISO that is immediately replayable through the hls-vod application creating an instant on demand of the content being live streamed to FMS.</p>
