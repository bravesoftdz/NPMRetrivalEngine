   <h1 id="firstHeading" class="firstHeading" lang="en">HTTP Live Streaming</h1>

      <p><b>HTTP Live Streaming</b> (also known as <b>HLS</b>) is an <a href="/wiki/HTTP" class="mw-redirect" title="HTTP">HTTP</a>-based media streaming communications protocol implemented by <a href="/wiki/Apple_Inc." title="Apple Inc.">Apple Inc.</a> as part of its <a href="/wiki/QuickTime" title="QuickTime">QuickTime</a>, <a href="/wiki/Safari_(web_browser)" title="Safari (web browser)">Safari</a>, <a href="/wiki/OS_X" class="mw-redirect" title="OS X">OS X</a>, and <a href="/wiki/IOS" title="IOS">iOS</a> software. It resembles <a href="/wiki/Dynamic_Adaptive_Streaming_over_HTTP" title="Dynamic Adaptive Streaming over HTTP">MPEG-DASH</a> in that it works by breaking the overall stream into a sequence of small HTTP-based file downloads, each download loading one short chunk of an overall potentially unbounded transport stream. As the stream is played, the client may select from a number of different alternate streams containing the same material encoded at a variety of data rates, allowing the streaming session to adapt to the available data rate. At the start of the streaming session, HLS downloads an <a href="/wiki/Extended_M3U" class="mw-redirect" title="Extended M3U">extended M3U playlist</a> containing the metadata for the various sub-streams which are available.<sup id="cite_ref-2" class="reference"><a href="#cite_note-2">[2]</a></sup></p> 
      <p>Since its requests use only standard HTTP transactions, HTTP Live Streaming can traverse any <a href="/wiki/Firewall_(computing)" title="Firewall (computing)">firewall</a> or <a href="/wiki/Proxy_server" title="Proxy server">proxy server</a> that lets through standard HTTP traffic, unlike UDP-based protocols such as <a href="/wiki/Real-time_Transport_Protocol" title="Real-time Transport Protocol">RTP</a>. This also allows content to be offered from conventional HTTP servers as origin and delivered over widely available HTTP-based <a href="/wiki/Content_delivery_network" title="Content delivery network">content delivery networks</a>.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup></p> 
      <p>HLS also specifies a standard encryption mechanism<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup> using <a href="/wiki/Advanced_Encryption_Standard" title="Advanced Encryption Standard">AES</a> and a method of secure-key distribution using <a href="/wiki/HTTPS" title="HTTPS">HTTPS</a> with either a device-specific realm login or <a href="/wiki/HTTP_cookie" title="HTTP cookie">HTTP cookie</a> which together provide a simple <a href="/wiki/Digital_Rights_Management" class="mw-redirect" title="Digital Rights Management">DRM</a> system. Later versions of the protocol also provide for <a href="/wiki/Trick_mode" title="Trick mode">trick-mode</a> fast-forward and rewind and for integration of subtitles. <a href="/wiki/UpLynk" class="mw-redirect" title="UpLynk">upLynk</a> has also added the AES scrambling and <a href="/wiki/Base-64" class="mw-redirect" title="Base-64">base-64</a> encoding of the DRM content-key with a 128-bit device-specific key for registered commercial <a href="/wiki/SWF" title="SWF">SWF</a> applications together with a sequential <a href="/wiki/Initialization_Vector" class="mw-redirect" title="Initialization Vector">initialization Vector</a> for each chunk to its implementation of the standard.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup></p> 
      <p>Apple has documented HTTP Live Streaming as an <a href="/wiki/Internet_Draft" title="Internet Draft">Internet Draft</a> (Individual Submission), the first stage in the process of publishing it as a <a href="/wiki/Request_for_Comments" title="Request for Comments">Request for Comments</a> (RFC). As of December 2015, the authors of that document have requested the RFC Independent Stream Editor (ISE) to publish the document as an informational (non-standard) RFC outside of the <a href="/wiki/IETF" class="mw-redirect" title="IETF">IETF</a> consensus process.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup> August 2017, RFC8216 was published to describe version 7 of the protocol.<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">[7]</a></sup></p>

      <h2><span class="mw-headline" id="Architecture">Architecture</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=HTTP_Live_Streaming&amp;action=edit&amp;section=1" title="Edit section: Architecture">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>HTTP Live Streaming uses a conventional <a href="/wiki/Web_server" title="Web server">web server</a> to distribute audiovisual content and requires specific software to fit into the proper format transmission in real time. The service architecture comprises:</p> 
      <dl> 
       <dt>
        Server
       </dt> 
       <dd>
        Codify and encapsulate the input video flow in a proper format for the delivery. Then, it is prepared for distribution by segmenting it into different files. In the process of intake, the video is coded and segmented to generate video fragments and index file. 
        <ul> 
         <li>Encoder: codify video files in <a href="/wiki/H.264/MPEG-4_AVC" title="H.264/MPEG-4 AVC">H.264</a> format and audio in <a href="/wiki/Advanced_Audio_Coding" title="Advanced Audio Coding">AAC</a>, <a href="/wiki/MP3" title="MP3">MP3</a>, <a href="/wiki/Dolby_AC-3" class="mw-redirect" title="Dolby AC-3">AC-3</a> or <a href="/wiki/Dolby_Digital_Plus" title="Dolby Digital Plus">EC-3</a>.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">[8]</a></sup> This is encapsulated by <a href="/wiki/MPEG_transport_stream" title="MPEG transport stream">MPEG-2 Transport Stream</a> to carry it</li> 
         <li>Segmenter: divides the MPEG-2 TS file into fragments of equal length, kept as <tt>.ts</tt> files. It also creates an index file that contains references of the fragmented files, saved as <a href="/wiki/M3U" title="M3U">.m3u8</a></li> 
        </ul> 
       </dd> 
      </dl> 
      <dl> 
       <dt>
        Distributor
       </dt> 
       <dd>
        Formed by standard web Server, accepts requests from clients and delivers all the resources (.m3u8 playlist file and .ts segment files) needed for 
        <a href="/wiki/Streaming_media" title="Streaming media">streaming</a>.
       </dd> 
       <dt>
        Client
       </dt> 
       <dd>
        Request and download all the files and resources, assembling them so that they can be presented to the user as a continuous flow video. The client software downloads first the index file through a 
        <a href="/wiki/URL" title="URL">URL</a> and then the several media files available. The playback software assembles the sequence to allow continued display to the user.
       </dd> 
      </dl> 
      <h2><span class="mw-headline" id="Features">Features</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=HTTP_Live_Streaming&amp;action=edit&amp;section=2" title="Edit section: Features">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>HTTP Live Streaming provides mechanisms for players to adapt to unreliable network conditions without causing user-visible playback stalling. For example, on an unreliable wireless network, HLS allows the player to use a lower quality video, thus reducing bandwidth usage. HLS videos can be made highly available by providing multiple servers for the same video, allowing the player to swap seamlessly if one of the servers fails.</p> 
      <h3><span class="mw-headline" id="Adaptability">Adaptability</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=HTTP_Live_Streaming&amp;action=edit&amp;section=3" title="Edit section: Adaptability">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p>To enable a player to adapt to the bandwidth of the network, the original video is encoded in several distinct <a href="/wiki/Video_quality" title="Video quality">quality levels</a>. The server serves an index, called a "master playlist", of these <a href="/wiki/Video_encoding" class="mw-redirect" title="Video encoding">encodings</a>, called "variant streams". The player can then choose between the variant streams during playback, changing back and forth seamlessly as network conditions change.</p> 
      <h3><span class="mw-headline" id="Availability">Availability</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=HTTP_Live_Streaming&amp;action=edit&amp;section=4" title="Edit section: Availability">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p>The master playlist can list duplicate variant streams that have the same level of quality but are hosted by different servers. This enables the player to swap to a different server for a given bitrate if one of the servers fails. This grants a degree of <a href="/wiki/High_availability" title="High availability">high availability</a> to serving streams.</p> 
      <p>This can be combined with adaptability by listing multiple variant streams for each distinct quality level.</p> 
      <h3><span class="mw-headline" id="Using_fragmented_MP4">Using fragmented MP4</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=HTTP_Live_Streaming&amp;action=edit&amp;section=5" title="Edit section: Using fragmented MP4">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p>On WWDC2016 Apple announced<sup id="cite_ref-9" class="reference"><a href="#cite_note-9">[9]</a></sup> the inclusion of byte-range addressing for fragmented <a href="/wiki/MP4" class="mw-redirect" title="MP4">MP4</a> files, or fMP4, allowing content to be played in HLS without the need to multiplex it into <a href="/wiki/MPEG_transport_stream" title="MPEG transport stream">MPEG-2 Transport Stream</a>. The industry considered this as a step towards compatibility between HLS and <a href="/wiki/MPEG-DASH" class="mw-redirect" title="MPEG-DASH">MPEG-DASH</a>.<sup id="cite_ref-10" class="reference"><a href="#cite_note-10">[10]</a></sup><sup id="cite_ref-11" class="reference"><a href="#cite_note-11">[11]</a></sup></p>

       <li><a href="/wiki/Tata_Communications" title="Tata Communications">Tata Communications</a> <a href="/wiki/Content_delivery_network" title="Content delivery network">CDN</a> supports HLS for live and on-demand streams.</li> 
       <li><a href="/wiki/TVersity" class="mw-redirect" title="TVersity">TVersity</a> supports HLS in conjunction with on-the-fly transcoding for playback of any video content on iOS devices.</li> 
       <li><a href="/wiki/Unified_Origin" class="mw-redirect" title="Unified Origin">Unified Origin</a> supports HLS in live and VOD mode.</li> 
       <li><a href="/wiki/Unreal_Media_Server" title="Unreal Media Server">Unreal Media Server</a> supports low latency HLS as of version 9.5.</li> 
       <li><a href="/wiki/Ustream" title="Ustream">Ustream</a> supports HLS delivery of live broadcasts. The ingested stream is re-transcoded if the original audio and video codec falls outside HLS requirements.</li> 
       <li><a href="/wiki/VLC_Media_Player" class="mw-redirect" title="VLC Media Player">VLC Media Player</a> supports HLS for serving live and on-demand streams as of version 2.0</li>
