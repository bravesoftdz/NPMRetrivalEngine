       <dt>
        <span class="section"><a href="#ol_uek2_64rn">1.1.1. Unbreakable Enterprise Kernel Release 2</a></span>
       </dt>
      </dl>
     </div>
     <p>Oracle Linux 6 Update 4 ships with two sets of kernel packages:</p>
     <div class="itemizedlist">
      <ul class="itemizedlist" style="list-style-type: disc; ">
       <li class="listitem"><p>Unbreakable Enterprise Kernel Release 2 (<code class="literal">kernel-uek-2.6.39-400.17.1.el6uek</code>)</p></li>
       <li class="listitem"><p>Red Hat Compatible Kernel (<code class="literal">kernel-2.6.32-358.el6</code>)</p></li>
      </ul>
     </div>
     <p>By default, both the Unbreakable Enterprise Kernel and the Red Hat Compatible Kernel for the specific architecture (i386 or x86_64) are installed and the system boots the Unbreakable Enterprise Kernel.</p>
     <p>To make your system use the Red Hat Compatible Kernel by default:</p>
     <div class="orderedlist">
      <ol class="orderedlist" type="1">
       <li class="listitem"><p>Edit <code class="literal">/etc/grub.conf</code> and change the value of the <code class="literal">default</code> parameter to indicate the Red Hat Compatible Kernel. (Each entry for a bootable kernel in the file starts with a <code class="literal">title</code> definition. The entries are effectively numbered from 0 upwards, where 0 corresponds to the first entry in the file, 1 to the second entry, and so on. To view the GRUB manual, use the <span class="command"><strong>info grub</strong></span> command.)</p></li>
       <li class="listitem"><p>Edit <code class="literal">/etc/sysconfig/kernel</code> and change the setting for the default kernel package type from <code class="literal">DEFAULTKERNEL=kernel-uek</code> to <code class="literal">DEFAULTKERNEL=kernel</code>.</p></li>
      </ol>
     </div>

        <div>
         <h3 class="title"><a name="ol_uek2_64rn"></a>1.1.1.&nbsp;Unbreakable Enterprise Kernel Release 2</h3>
        </div>
       </div>
      </div>
      <p>The Unbreakable Enterprise Kernel Release 2 (UEK R2) is based on the upstream kernel 3.0.36 stable source tree.</p>
      <p>The Unbreakable Enterprise Kernel supports a wide range of hardware and devices. In close cooperation with hardware and storage vendors, the following device drivers have been updated by Oracle in the 2.6.39-400 kernel.</p>
      <h4><a name="idp147696"></a>Storage Adapter Drivers</h4>
      <p><span class="bold"><strong>Broadcom</strong></span></p>
      <div class="itemizedlist">
       <ul class="itemizedlist" style="list-style-type: disc; ">
        <li class="listitem"><p>NetXtreme II iSCSI driver (<code class="literal">bnx2i</code>) updated to 2.7.4.1f.</p></li>
        <li class="listitem"><p>NetXtreme II Fibre Channel over Ethernet driver (<code class="literal">bnx2fc</code>) updated to 2.2.17.</p></li>
       </ul>
      </div>
      <p><span class="bold"><strong>Emulex</strong></span></p>

      <div class="section">

        <div>
         <div>
          <h4 class="title"><a name="ol_lvm_64rn"></a>1.2.2.1.&nbsp;LVM RAID10 Support</h4>
         </div>
        </div>
       </div>
       <p>You can create, resize, and remove RAID10 volumes in LVM, where striping is laid out across an array of mirrors.</p>
       <p>To create a RAID 10 logical volume, use the following form of the <span class="command"><strong>lvcreate</strong></span> command:</p>
       <pre class="programlisting"># lvcreate --type raid10 -m <em class="replaceable"><code>mirrors</code></em> -i <em class="replaceable"><code>stripes</code></em> -L <em class="replaceable"><code>lv_size</code></em> -n <em class="replaceable"><code>lv_name</code></em> <em class="replaceable"><code>vol_grp</code></em></pre>
       <p>For example, the following command would create a 200 GB RAID10 volume named <code class="literal">myr10vol</code> with four stripes and two mirrors in the <code class="literal">myvg</code> volume group:</p>
       <pre class="programlisting"># lvcreate --type raid10 -m 1 -i 4 -L 200G -n myr10vol myvg</pre>
       <p>The <span class="command"><strong>-m</strong></span> option specifies the number of additional copies of the data, not the total number of copies.</p>
      </div>
      <div class="section">
       <div class="titlepage">
        <div>
         <div>
          <h4 class="title"><a name="ol_kst_64rn"></a>1.2.2.2.&nbsp;New Boot and Kickstart Options</h4>
         </div>
        </div>
       </div>
       <p>The following new boot options are available:</p>
       <pre class="programlisting">bond=<em class="replaceable"><code>bondname</code></em>:<em class="replaceable"><code>bondslave1</code></em>[,<em class="replaceable"><code>bondslave2</code></em>]...:[<em class="replaceable"><code>opt1</code></em>[, <em class="replaceable"><code>opt2</code></em>]...]</pre>
       <p>Specifies the bonded network interface, the network connections to be bonded to the interface, and any additional options.</p>
       <p> </p>
       <pre class="programlisting">vlanid=<em class="replaceable"><code>tag</code></em></pre>
       <p>Specifies a network device's numeric 802.1q tag to allow installation over a VLAN.</p>
       <p>The following new Kickstart options are available with the <code class="literal">network</code> keyword:</p>
       <pre class="programlisting">--bondslaves=<em class="replaceable"><code>bondslave1</code></em>[,<em class="replaceable"><code>bondslave2</code></em>]... --bondopts=<em class="replaceable"><code>opt1</code></em>[, <em class="replaceable"><code>opt2</code></em>]...</pre>
       <p>Specifies the network connections to be bonded to the network interface, and any additional options.</p>
       <pre class="programlisting">--vlanid=<em class="replaceable"><code>tag</code></em></pre>
       <p>Specifies a network device's numeric 802.1q tag to allow installation over a VLAN.</p>
       <p>The new Kickstart <code class="literal">fcoe</code> keyword allows you to enable Fibre Channel over Ethernet (FCoE) devices in addition to Enhanced Disk Drive Services (EDD) discovered devices. The following options are available with the <code class="literal">fcoe</code> keyword:</p>
       <pre class="programlisting">--autovlan</pre>
       <p>Specifies that VLANs should be discovered automatically.</p>
       <pre class="programlisting">--dcb=<em class="replaceable"><code>setting1</code></em>[,<em class="replaceable"><code>setting2</code></em>]...</pre>
       <p>Specifies settings for Data Center Bridging (DCB).</p>
       <pre class="programlisting">--nic=<em class="replaceable"><code>device</code></em></pre>
       <p>Specifies the name of the FCoE device to activate.</p>
      </div>
      <div class="section">
       <div class="titlepage">
        <div>
         <div>
          <h4 class="title"><a name="ol_kpdn_64rn"></a>1.2.2.3.&nbsp;Persistent Device Naming</h4>
         </div>
        </div>
       </div>
       <p><code class="literal">udev</code> maintains persistent device names for devices such as <code class="literal">/dev/sdb</code> by creating symbolic links such as <code class="literal">/dev/disk/by-path/pci-0000:00:0d.0-scsi-1:0:0:0</code> and <code class="literal">/dev/disk/by-uuid/e8d40553-43f2-4ae6-8e4b-38e04e7ee41c</code>. The kernel message log now records each <code class="literal">udev</code> persistent device name mapping in the following format:</p>
       <pre class="programlisting">udev-alias: <em class="replaceable"><code>device_name</code></em> (<em class="replaceable"><code>udev_persistent_name1</code></em> <em class="replaceable"><code>udev_persistent_name2</code></em> ... )</pre>
      </div>
      <div class="section">
       <div class="titlepage">
        <div>
         <div>
          <h4 class="title"><a name="ol_kpmu_64rn"></a>1.2.2.4.&nbsp;PMU Support for Uncore and Load Latency</h4>
         </div>
        </div>
       </div>
       <p>The uncore feature of the <code class="literal">perf</code> event subsystem implements Performance Monitoring Unit (PMU) support for the Intel Xeon Processor X55xx and X56xx processor families. Multiple processor cores can share physical uncore subsystems, including the L3 cache. Uncore PMU support allows packages to collect performance data, including load latency at various levels in the cache and memory hierarchy, which ranges from micro-operation dispatch up to globally observable data. Debugging in <code class="literal">perf</code> is made possible by the implementation of PMU event parsing.</p>
      </div>
     </div>
    </div>

       <div>
        <h2 class="title" style="clear: both"><a name="ol_tpf_64rn"></a>1.3.&nbsp;Technology Preview Features</h2>
       </div>
      </div>
     </div>
     <p>The following features included in the Unbreakable Enterprise Kernel Release 2 are still under development, but are made available for testing and evaluation purposes.</p>
     <div class="itemizedlist">
      <ul class="itemizedlist" style="list-style-type: disc; ">
       <li class="listitem"><p>Kernel module signing facility</p><p>Applies cryptographic signature checking to modules on module load, checking the signature against a ring of public keys compiled into the kernel. GPG is used to do the cryptographic work and determines the format of the signature and key data. </p></li>
       <li class="listitem"><p>Linux Containers (lxc)</p><p>Based on the Linux Cgroups and name spaces functionality, containers allow you to safely and securely run multiple applications or instances of an operating system on a single host without risking them interfering with each other. Containers are lightweight and resource-friendly, which saves both rack space and power. In order to get started with containers, you need to install the <code class="literal">lxc</code> package, which is included in the package repository of the Unbreakable Enterprise Kernel. </p></li>
       <li class="listitem"><p>Transcendent memory (tmem)</p><p>Transcendent Memory provides a new approach for improving the utilization of physical memory in a virtualized environment by claiming underutilized memory in a system and making it available where it is most needed. From the perspective of an operating system, tmem is fast pseudo-RAM of indeterminate and varying size that is useful primarily when real RAM is in short supply. To learn more about this technology and its use cases, see the Transcendent Memory project page on oss.oracle.com: <a class="ulink" href="http://oss.oracle.com/projects/tmem/" target="_top">http://oss.oracle.com/projects/tmem/</a> </p></li>
       <li class="listitem"><p>Distributed Replicated Block Device (DRBD)</p><p>A shared-nothing, synchronously replicated block device (<span class="emphasis"><em>RAID1 over network</em></span>), designed to serve as a building block for high availability (HA) clusters. It requires a cluster manager (for example, pacemaker) to implement automatic failover. </p></li>
      </ul>
     </div>

       <li class="listitem"><p>LVM API</p></li>
       <li class="listitem"><p>LVM RAID support</p></li>
       <li class="listitem"><p>Matahari</p></li>
       <li class="listitem"><p>Open multicast ping (<code class="literal">omping</code>)</p></li>
       <li class="listitem"><p>System Information Gatherer and Reporter (SIGAR)</p></li>
       <li class="listitem"><p>Trusted Platform Module (TPM)</p></li>
       <li class="listitem"><p>Trusted Boot</p></li>
       <li class="listitem"><p><code class="literal">vios-proxy</code></p></li>
      </ul>
     </div>

     <h4><a name="idp449584"></a>Oracle Clusterware Fails to Start on ASM Storage with SELinux Enabled</h4>
     <p>If the SELinux policy packages have not been updated recently, Cluster Ready Services (CRS) might fail to start with messages such as the following in <code class="literal">/var/log/messages</code>:</p>
     <pre class="programlisting">SELinux is preventing /usr/lib/oracleasm/oracleasm-instantiate-disk from
associate access on the filesystem DATA1.</pre>
     <p>The solution is to upgrade the <code class="literal">selinux-policy</code> and <code class="literal">selinux-policy-targeted</code> packages to ensure that you are running a version no earlier than 3.7.195.0.1.el6_4.5: </p>
     <pre class="programlisting"># yum update 'selinux-policy*'</pre>
     <p>After upgrading the packages, reboot the system. (Bug ID 13925445)</p>
     <h4><a name="idp454128"></a>Support for Large Memory 32-bit Systems</h4>
     <p>Releases of Oracle Linux prior to Oracle Linux 5 supplied a <span class="emphasis"><em>hugemem</em></span> kernel to allow a system to address up to 64 GB of memory in 32-bit mode. The hugemem kernel is no longer available in Oracle Linux 5 and later releases.</p>
     <p>The Unbreakable Enterprise Kernel (UEK) supports a maximum of 16 GB of memory for 32-bit kernels on bare metal and hardware virtualized machine (HVM) systems, and 8 GB for fully paravirtualized machine (PVM) systems. 32-bit PVM guest operating systems must be located in the first 128 GB of physical memory on the host.</p>
     <p>The Red Hat Compatible Kernel (RHCK) has the same limitations, except that PVM systems can have up to 16 GB of memory. The limitation of 8 GB for PVM on UEK was chosen for reasons of reliability.</p>
     <p>A 32-bit system uses the PAE (physical address extension) memory feature to map physical memory beyond 4 GB into the 32-bit address space that is available to each process. A 64-bit system can address memory beyond 4 GB without requiring an extra layer of memory abstraction.</p>
     <p>Oracle Linux on x86_64 includes 32-bit libraries, which allow applications built for both 64-bit and 32-bit Linux to run on the same system. This capability provides scalability to virtually unlimited memory sizes, while retaining the ability to run 32-bit applications. Oracle recommends this configuration for any system with more than 4 GB of memory. (Bug ID 16974301)</p>
     <h4><a name="idp458720"></a>Linux Containers 32-bit Support</h4>
     <p>The Linux Containers package (<code class="literal">lxc</code>) is not available for the i386 architecture.</p>
     <h4><a name="idp460304"></a>Linux Containers Interoperability with SELinux</h4>
     <p>Running with SELinux enabled on the host can cause issues with Linux Containers. The workaround is to disable SELinux altogether by setting <code class="literal">SELINUX=disabled</code> in <code class="literal">/etc/selinux/config</code> and rebooting the system. Using the <span class="command"><strong>setenforce Permissive</strong></span> command is not sufficient as the <code class="literal">selinuxfs</code> pseudo file system remains mounted. (Bug ID 15967411)</p>
     <h4><a name="idp463456"></a>Linux Containers Default Configuration Location</h4>
     <p>The default location for a container's configuration has changed from <code class="literal">/etc/lxc/<em class="replaceable"><code>name</code></em></code> to <code class="literal">/container/<em class="replaceable"><code>name</code></em></code> in <code class="literal">lxc 0.8.0</code>.</p>
     <p>To start a container that you created with a previous update of Oracle Linux, specify the <span class="command"><strong>-f</strong></span> option to <span class="command"><strong>lxc-start</strong></span>, for example: </p>
     <pre class="programlisting"># lxc-start -n ol6u3 -f /etc/lxc/ol6u3/config</pre>
     <p>To convert an existing container to use the new location:</p>
     <div class="orderedlist">
      <ol class="orderedlist" type="1">
       <li class="listitem"><p>Move the container's configuration directory to <code class="literal">/container/<em class="replaceable"><code>name</code></em></code>:</p><pre class="programlisting"># mv /etc/lxc/<em class="replaceable"><code>name</code></em> /container </pre></li>
       <li class="listitem"><p>Edit the <code class="literal">/container/<em class="replaceable"><code>name</code></em>/config</code> file and change the values of any <code class="literal">lxc.rootfs</code> and <code class="literal">lxc.mount</code> parameters to refer to <code class="literal">/container</code> instead of <code class="literal">/etc/lxc</code>.</p><p>For example, if the <code class="literal">config</code> file contained the following entries:</p><pre class="programlisting">lxc.rootfs = /etc/lxc/example/rootfs
lxc.mount.entry=/lib /etc/lxc/example/rootfs/lib none ro,bind 0 0
lxc.mount.entry=/usr/lib /etc/lxc/example/rootfs/usr/lib none ro,bind 0 0
lxc.mount.entry=/lib64 /etc/lxc/example/rootfs/lib64 none ro,bind 0 0
lxc.mount.entry=/usr/lib64 /etc/lxc/example/rootfs/usr/lib64 none ro,bind 0 0</pre><p>you would change these entries to read:</p><pre class="programlisting">lxc.rootfs = /container/example/rootfs
lxc.mount.entry=/lib /container/example/rootfs/lib none ro,bind 0 0
lxc.mount.entry=/usr/lib /container/example/rootfs/usr/lib none ro,bind 0 0
lxc.mount.entry=/lib64 /container/example/rootfs/lib64 none ro,bind 0 0
lxc.mount.entry=/usr/lib64 /container/example/rootfs/usr/lib64 none ro,bind 0 0</pre></li>
      </ol>
     </div>
     <p>After converting the container, you do not need to specify the <span class="command"><strong>-f</strong></span> option to <span class="command"><strong>lxc-start</strong></span>. (Bug ID 15967411)</p>
     <h4><a name="idp478736"></a>Broadcom NetXtreme II 10Gbps Network Adapter Driver</h4>
     <p>When using the <code class="literal">bnx2x</code> driver in a bridge, disable Transparent Packet Aggregation (TPA) by including <code class="literal">options bnx2x disable_tpa=1</code> in <code class="literal">/etc/modprobe.conf</code>. (Bug ID 14626070)</p>
     <h4><a name="idp481296"></a>Btrfs File System Balancing</h4>
     <p>Running <span class="command"><strong>btrfs filesystem balance</strong></span> converts a non-RAID or concatenated file system setup to RAID-0 after adding a new device. Do not run this command if you do not intend to convert the profile of the file system after adding the new device. (Bug ID 13715389)</p>
     <h4><a name="idp483104"></a>Btrfs Conversion Does Not Preserve SELinux Security Contexts</h4>
     <p>Converting an existing <code class="literal">ext2</code>, <code class="literal">ext3</code>, or <code class="literal">ext4</code> root file system to <code class="literal">btrfs</code> does not carry over the associated security contexts that are stored as part of a file's extended attributes. With SELinux enabled and set to enforcing mode, you might experience many <code class="literal">permission denied</code> errors after reboot, and the system might be unbootable. To avoid this problem, enforce automatic file system relabeling run at bootup time. To trigger automatic relabeling, create an empty file named <code class="literal">autorelabel</code> (for example, by using <span class="command"><strong>touch</strong></span>) in the file system's root directory before rebooting the system after the initial conversion. The presence of this file instruct SELinux to recreate the security attributes for all files on the file system. If you forget to do this and rebooting fails, either temporarily disable SELinux completely by adding <code class="literal">selinux=0</code> to the kernel boot parameters, or disable enforcing of the SELinux policy by adding <code class="literal">enforcing=0</code>. (Bug ID 13806043)</p>
     <h4><a name="idp489056"></a>Btrfs RAID1 Failure Results in a Kernel Panic</h4>
     <p>A failing RAID1 disk might result in a kernel panic with the error <code class="literal">kernel: BTRFS error (device (null)) in btree_writepage_io_failed_hook:3662: IO failure (Error occurred while writing out btree at offset)</code>. (Bug ID 16262571)</p>
     <h4><a name="idp490832"></a>btrfs subvolume get-default Command</h4>
     <p>The <span class="command"><strong>btrfs subvolume get-default</strong></span> command lists all existing subvolumes instead of only the default subvolume. (Bug ID 13815433)</p>
     <h4><a name="idp492496"></a>btrfs filesystem defragment Command</h4>
     <p>The <span class="command"><strong>btrfs filesystem defragment</strong></span> command exits with an exit code of 20 even if it succeeds. (Bug ID 13714531)</p>
     <h4><a name="idp494144"></a>Btrfs File Size</h4>
     <p>Commands such as <span class="command"><strong>du</strong></span> might show inconsistent results for file sizes in a btrfs file system when the number of bytes that is under delayed allocation is changing. (Bug ID 13096268)</p>
     <h4><a name="idp495872"></a>Btrfs Hard Link Limit</h4>
     <p>Btrfs has a limit of 237 hard links to a file. Attempting to create more than this number of links results in the error <code class="literal">Too many links</code>. (Bug ID 16278563)</p>
     <h4><a name="idp497536"></a>Error Message Following First Reboot on an HP ProLiant Server</h4>
     <p>You might see a message similar to the following during the first reboot of an HP ProLiant server:</p>
     <pre class="programlisting">[Firmware Bug]: the BIOS has corrupted hw-PMU resources (MSR 186 is 43003c)</pre>
     <p>You can safely ignore this message. The functionality and performance of the operating system and the server are not affected.</p>
     <h4><a name="idp499792"></a>Mellanox ConnectX Drivers</h4>
     <p>The Mellanox ConnectX core, Ethernet, and InfiniBand drivers are supported only for the x86_64 architecture.</p>
     <p>(Bug ID 16228063)</p>
     <h4><a name="idp501424"></a>udevd Message</h4>
     <p>A message similar to the following might be recorded in <code class="literal">dmesg</code> or <code class="literal">/var/log/messages</code> at boot time:</p>
     <pre class="programlisting">udevd (<em class="replaceable"><code>pid</code></em>): /proc/<em class="replaceable"><code>pid</code></em>/oom_adj is deprecated, please use /proc/<em class="replaceable"><code>pid</code></em>/oom_score_adj instead.</pre>
     <p>The <code class="literal">udevd</code> process uses the deprecated <code class="literal">oom_adj</code> kernel interface to prevent it from being killed if the system runs short of memory. You can safely ignore the message as the action still succeeds. To prevent the message from occurring, install the package <code class="literal">udev-147-2.42.el6.arch.rpm</code> or higher. (Bug ID 13655071, 13712009)</p>
     <h4><a name="idp507072"></a>Unable to Register Oracle Linux Guest with ULN</h4>
     <p>Registering an Oracle Linux guest running under Virtual Box with the Unbreakable Linux Network (ULN) might fail with a server communication error. The workaround is to run the following command as <code class="literal">root</code> on the guest:</p>
     <pre class="programlisting"># echo "uuid=`uuidgen -t`" &gt;&gt; /etc/sysconfig/rhn/up2date</pre>
     <p>and then run <span class="command"><strong>uln_register</strong></span> again. (Bug ID 14696776)</p>
     <h4><a name="idp510064"></a>xguest Package Not Installable with SELinux Disabled</h4>
     <p>If the <code class="literal">xguest</code> package fails to install with a <code class="literal">PREIN</code> script error, enable SELinux by setting <code class="literal">SELINUX=enabled</code> in <code class="literal">/etc/selinux/config</code>, reboot the system, and reinstall the <code class="literal">xguest</code> package. (Bug ID 13495388)</p>
     <h4><a name="idp513536"></a>X Window System Does Not Run in a PVHVM guest</h4>
     <p>If you install an Oracle Linux 6 Update 4 (x86_64) PVHVM guest with either the Desktop or the Software Development Workstation installation options, the X Window System is not accessible after installation when you boot the guest into run level 5. This problem is seen in OVM 3.0 and later.</p>
     <p>Use the following workaround:</p>
     <div class="orderedlist">
      <ol class="orderedlist" type="1">
       <li class="listitem"><p>Boot the guest into run level 3 by appending 3 to the <code class="literal">kernel</code> command line in GRUB, for example:</p><p> </p><pre class="programlisting">kernel /vmlinuz-2.6.39-400.15.0.el6uek.x86_64 ... rd_NO_DM 3</pre></li>
       <li class="listitem"><p>After the guest boots, log in as <code class="literal">root</code>, and uninstall the <code class="literal">xorg-x11-drv-cirrus</code> package, for example:</p><p> </p><pre class="programlisting"># rpm -ev --nodeps xorg-x11-drv-cirrus</pre></li>
      </ol>
     </div>
     <p>You can then either reboot the system into run level 5 or use the <span class="command"><strong>init 5</strong></span> command to switch to run level 5. The X Window System will be accessible on subsequent boots to run level 5. (Bug ID 16280196)</p>
     <h4><a name="idp521376"></a>Oracle VM 3.0 Guests Crash During Oracle Database Installation</h4>
     <p>PVHVM guests on Oracle VM 3.0 crash during Oracle Database installation if the value of the maximum memory (<code class="literal">maxmem</code>) parameter set for the guest is greater than the amount specified at boot time (<code class="literal">memory</code>). To avoid this issue, ensure that the values of the <code class="literal">maxmem</code> and <code class="literal">memory</code> parameters are the same. This issue has been resolved in Oracle VM 3.1.1. (Bug ID 13396734)</p>
     <h4><a name="idp524608"></a>Booting UEK R2 as a 32-bit PVHVM guest</h4>
     <p>When booting UEK R2 as a 32-bit PVHVM guest, you can safely ignore the kernel message <code class="literal">register_vcpu_info failed: err=-38</code>, which might be displayed. (Bug ID 13713774)</p>
     <h4><a name="idp526304"></a>Post-installation Anaconda Errors</h4>
     <p>In certain cases, after successfully completing installation and rebooting the system, it is possible for errors such as the following to occur:</p>
     <pre class="programlisting">Error in sys.excepthook:
 Traceback (most recent call last):
   File "/usr/lib/python2.6/site-packages/meh/handler.py", line 161, in
 (lambda)
   File "/usr/lib/anaconda/exception.py", line 44, in handleException
   File "/usr/lib/python2.6/site-packages/meh/handler.py", line 106, in
 handleException
   File "/usr/lib/anaconda/gui.py", line 1169, in mainExceptionWindow
 ImportError: No module named ui.gui</pre>
     <p>These errors can safely be ignored.</p>
     <h4><a name="idp529296"></a>FCoE Target Service</h4>
     <p>The upstream release has added support for FCoE target service. This service is not supported with the previous release of the Unbreakable Enterprise Kernel (2.6.32). To use this service, boot your system into the Unbreakable Enterprise Kernel Release 2 (2.6.39) or the Red Hat Compatible Kernel.</p>
     <h4><a name="idp530704"></a>mlx4_core Conflicts Between the mlnx_en and ofa Packages</h4>
     <p>Both the <code class="literal">mlnx_en</code> and <code class="literal">ofa</code> packages contain <code class="literal">mlx4_core</code>. Only one of these packages should be installed. Attempting to install both packages on a single server results in a package conflict error. If you have a Mellanox Ethernet Controller, install <code class="literal">mlnx_en</code>. If you have a Mellanox InfiniBand Controller, install <code class="literal">ofa</code>. If your system has both controllers, use <code class="literal">ofa</code> as it supports both the Ethernet and InfiniBand controllers.</p>
     <h4><a name="idp534784"></a>Kdump Service Fails to Start for UEK</h4>
     <p>When configuring the crash kernel for the UEK, only standard crash kernel settings (for example, <code class="literal">crashkernel=128M@32M</code>) are supported. The new settings used by the Red Hat Compatible Kernel (for example, <code class="literal">crashkernel=auto</code>) are not supported and cause the <code class="literal">kdump</code> service to fail to start. (Bug ID 13495212)</p>
     <h4><a name="idp537472"></a>iTCO_wdt Errors</h4>
     <p>If you see the boot-time <code class="literal">dmesg</code> error <code class="literal">iTCO_wdt: failed to reset NO_REBOOT flag, device disabled by hardware/BIOS</code> with UEK R2 or <code class="literal">iTCO_wdt: failed to reset NO_REBOOT flag, reboot disabled by hardware</code> with UEK, add the line <code class="literal">blacklist iTCO_wdt</code> to <code class="literal">/etc/modprobe.d/blacklist-watchdog</code>.</p>
     <h4><a name="idp540960"></a>Paravirtualized Drivers in a Hardware Virtualized Guest</h4>
     <p>The Unbreakable Enterprise Kernel adds support for PV drivers in a HVM guest (PVHVM) on Oracle VM. The default is to present only PV drivers when running in an HVM guest. To run <code class="literal">kernel-uek</code> fully hardware virtualized, including the drivers, add the parameter <code class="literal">xen_emul_unplug=never</code> to the boot parameters in <code class="literal">/etc/grub.conf</code>, for example:</p>
     <p> </p>
     <pre class="programlisting">kernel /vmlinuz-2.6.32-300.2.1.el6uek ro root=/dev/VolGroup00/LogVol00 xen_emul_unplug=never</pre>
     <p>Adding this parameter makes the kernel also present the emulated drivers as previously (for example, the <code class="literal">8139cp</code> network driver).</p>
     <h4><a name="idp545552"></a>Incorrect Package Count</h4>
     <p>Selecting all packages in certain groups during installation might not show the correct package count. (Bug ID 11684244)</p>
     <h4><a name="idp546768"></a>Default Reverse Path Filtering Mode Affects Certain Oracle Products</h4>
     <p>Oracle Linux 6 defaults to reverse path filtering in strict mode. Some Oracle products and network storage devices work more reliably with reverse path filtering in loose mode. To enable loose mode, issue the following command (where <em class="replaceable"><code>iface</code></em> is the network interface, for example, <code class="literal">eth1</code>).</p>
     <p> </p>
     <pre class="programlisting"># sysctl net.ipv4.conf.<em class="replaceable"><code>iface</code></em>.rp_filter=2</pre>
     <p>The default setting is 1 for strict mode. (Bug ID 10649976)</p>
     <h4><a name="idp550768"></a>Receive Packet Steering errors</h4>
     <p>Certain network operations that utilize receive packet steering could cause errors on the system. (Bug ID 11071685)</p>
     <h4><a name="idp552000"></a>Spurious udev Messages During Failed Path Restoration</h4>
     <p>If failed paths are restored in a multipath configuration, you might see <code class="literal">udevd-work</code> error messages in <code class="literal">/var/log/messages</code>. The failed paths are restored despite these messages, which you can ignore. (Bug ID 11682171)</p>
     <h4><a name="idp554192"></a>Default NFS Mount Options</h4>
     <p>The default NFS mount option has changed to NFS v4. To mount an NFS v3 volume (the default in Oracle Linux 5), use the following mount options:</p>
     <p> </p>
     <pre class="programlisting">-o vers=3,mountproto=tcp</pre>
     <h4><a name="idp556256"></a>Setting the Serial Console in a Hardware Virtualized Guest</h4>
     <p>To set the serial console a hardware virtualized guest, use following settings in the guest:</p>
     <div class="itemizedlist">
      <ul class="itemizedlist" style="list-style-type: disc; ">
       <li class="listitem"><p>Add the following parameters to the kernel boot line in <code class="literal">/etc/grub.conf</code>:</p><pre class="programlisting">console=tty0 console=ttyS0,57600n8</pre></li>
       <li class="listitem"><p>Add the following line to <code class="literal">/etc/securetty</code>:</p><pre class="programlisting">ttyS0</pre></li>
      </ul>
     </div>
     <h4><a name="idp560832"></a>Oracle RDBMS Server 11gR2 Preinstall Settings Not Visible After Installation</h4>
     <p>On an x86_64 system, if you install the <code class="literal">pam.i386</code> package either manually or via a package dependency, and the <code class="literal">oracle-rdbms-server-11gR2-preinstall</code> package is also selected, this overwrites the settings for Oracle Database in <code class="literal">/etc/security/limits.conf</code>. This is most likely to occur during a Kickstart-automated installation that includes non-standard packages. To restore the settings, run the <span class="command"><strong>oracle-rdbms-server-11gR2-preinstall-verify</strong></span> script. (Bug ID 14212822)</p>
     <h4><a name="idp564176"></a>Unbreakable Linux Network</h4>
     <p>Following the first reboot after installing Oracle Linux 6, you are prompted to register your system with the Unbreakable Linux Network (ULN). If you did not configure your network during the installation, the registration process to ULN cannot succeed. To register your system, log in as <code class="literal">root</code>, configure the system's network manually, and run <span class="command"><strong>uln_register</strong></span>. </p>
     <h4><a name="idp566496"></a>Console Appears to Hang When Booting</h4>
     <p>On some hardware, the console may appear to hang during the boot process after starting <code class="literal">udev</code>. However, the system does boot properly and is accessible. A workaround to this problem is to add <code class="literal">nomodeset</code> as a kernel boot parameter in <code class="literal">/etc/grub.conf</code>. (Bug ID 10094052, 13485328)</p>
     <h4><a name="idp569168"></a>Default I/O Scheduler</h4>
     <p>For the Unbreakable Enterprise Kernel, <code class="literal">deadline</code> is the default I/O scheduler.</p>
     <p>For the Red Hat Compatible Kernel, <code class="literal">cfq</code> is the default I/O scheduler.</p>
     <h4><a name="idp571664"></a>sched_yield() Settings for Completely Fair Scheduler</h4>
     <p>For the Unbreakable Enterprise Kernel, the default setting is <code class="literal">kernel.sched_compat_yield=1</code>.</p>
     <p>For the Red Hat Compatible Kernel, the default setting is <code class="literal">kernel.sched_compat_yield=0</code>.</p>

        <h2 class="title" style="clear: both"><a name="ol_sup_64rn"></a>3.1.&nbsp;Supported Upgrade Paths</h2>
       </div>
      </div>
     </div>
     <p>Upgrading from Oracle Linux 6 GA, Update 1, Update 2, or Update 3 is supported. Upgrading from a beta release is not supported.</p>
     <p>In-place upgrading from a major version of Oracle Linux 5 or earlier is not supported. Although Anaconda provides an option to perform an upgrade, fresh installation is strongly recommended.</p>
     <p>If you have an Oracle Linux 5.8 system, you can use new features in Unbreakable Enterprise Kernel without upgrading to Oracle Linux 6 as Oracle Linux 5.8 includes the Unbreakable Enterprise Kernel.</p>

       <div>
        <h2 class="title" style="clear: both"><a name="ol_opkg_rn64"></a>3.2.&nbsp;Obtaining Oracle Linux 6 Update 4 Packages</h2>
       </div>
      </div>
     </div>
     <div class="toc">
      <dl>
       <dt>
        <span class="section"><a href="#ol_uln_rn64">3.2.1. About the Unbreakable Linux Network</a></span>
       </dt>
       <dt>
        <span class="section"><a href="#ol_pubyum_rn64">3.2.2. About Public Yum</a></span>
       </dt>
       <dt>
        <span class="section"><a href="#ol_olim_rn64">3.2.3. About Oracle Linux Installation Media</a></span>
       </dt>
      </dl>
     </div>
     <div xmlns="http://www.w3.org/1999/xhtml" class="note" style="margin-left: 0.5in; margin-right: 0.5in;">
      <div class="admon-title">
       Note
      </div>
      <p xmlns="">You use <span class="command"><strong>yum</strong></span> rather than <span class="command"><strong>up2date</strong></span> to manage packages with Oracle Linux 6. Using <span class="command"><strong>up2date</strong></span> is not supported.</p>
     </div>
     <p>You can download a full Oracle Linux installation media image from the Oracle Software Delivery Cloud at <a class="ulink" href="http://edelivery.oracle.com/linux" target="_top">http://edelivery.oracle.com/linux</a>. You can also obtain Oracle Linux packages from the Unbreakable Linux Network (ULN) and the Oracle Public Yum server.</p>
     <div class="section">
      <div class="titlepage">
       <div>
        <div>
         <h3 class="title"><a name="ol_uln_rn64"></a>3.2.1.&nbsp;About the Unbreakable Linux Network</h3>
        </div>
       </div>
      </div>
      <p>You have the option of registering a system with ULN when you install Oracle Linux 6 on a system. To register with ULN after installation, use the <span class="command"><strong>uln_register</strong></span> command.</p>
      <p>To obtain Oracle Linux updates from ULN, you must have an Oracle Linux support subscription. For more information about ULN, see <a class="ulink" href="http://linux.oracle.com" target="_top">http://linux.oracle.com</a>.</p>
      <p>During ULN registration, your server is automatically registered with the latest channels for the base repository and the Unbreakable Enterprise Kernel Release 2. If you have upgraded your system from a previous release and do not want to install UEK Release 2, you must manually unsubscribe the server from this channel.</p>
      <p>ULN also provides channels for Oracle-specific software packages such as Oracle's <code class="literal">ASMlib</code> userspace package and the Oracle Instant Client. To enable access to these packages, log in to ULN and subscribe your system to the Oracle Software channel.</p>
     </div>
     <div class="section">
      <div class="titlepage">
       <div>
        <div>
         <h3 class="title"><a name="ol_pubyum_rn64"></a>3.2.2.&nbsp;About Public Yum</h3>
        </div>
       </div>
      </div>
      <p>Oracle also provides all errata and updates for Oracle Linux via the Public Yum service, which includes updates to the base distribution, but does not include Oracle-specific software. You do not require an Oracle Linux support subscription to use this service. For more information on how to obtain updates from Public Yum, see <a class="ulink" href="http://public-yum.oracle.com" target="_top">http://public-yum.oracle.com</a>.</p>
      <p>By default, all new installations of Oracle Linux 6 Update 4 are automatically configured to use the public yum update service. If you subsequently register the system with ULN, the public yum service is automatically disabled.</p>
      <p>The following entries in the <code class="literal">/etc/yum.repos.d/public-yum-ol6.repo</code> file enable you to download the latest available packages for Oracle Linux 6 and the Unbreakable Enterprise Kernel Release 2:</p>
      <pre class="programlisting">[ol6_latest]
name=Oracle Linux $releasever Latest ($basearch)
baseurl=http://public-yum.oracle.com/repo/OracleLinux/OL6/latest/$basearch/
gpgkey=http://public-yum.oracle.com/RPM-GPG-KEY-oracle-ol6
gpgcheck=1
enabled=1

[ol6_UEK_latest]
name=Latest Unbreakable Enterprise Kernel for Oracle Linux $releasever ($basearch)
baseurl=http://public-yum.oracle.com/repo/OracleLinux/OL6/UEK/latest/$basearch/
gpgkey=http://public-yum.oracle.com/RPM-GPG-KEY-oracle-ol6
gpgcheck=1
enabled=1</pre>
     </div>
     <div class="section">
      <div class="titlepage">
       <div>
        <div>
         <h3 class="title"><a name="ol_olim_rn64"></a>3.2.3.&nbsp;About Oracle Linux Installation Media</h3>
        </div>
       </div>
      </div>
      <p>Oracle Linux 6 Update 4 contains two distinct repository sources on the installation media for the Red Hat Compatible Kernel and the Unbreakable Enterprise Kernel. To configure <span class="command"><strong>yum</strong></span> to use both repositories from an ISO image of the installation media, create the file <code class="literal">/etc/yum.repos.d/Media.repo</code> containing entries similar to the following:</p>
      <pre class="programlisting">[ol6_base_media]
name=Oracle Linux 6 Update 4 Base Media
baseurl=file:///media/<em class="replaceable"><code>ISOimage</code></em>/Server
gpgkey=file:///media/<em class="replaceable"><code>ISOimage</code></em>/RPM-GPG-KEY
gpgcheck=1
enabled=1

[ol6_uek_media]
name=Oracle Linux 6 Update 4 UEK Media
baseurl=file:///media/<em class="replaceable"><code>ISOimage</code></em>/UEK2
gpgkey=file:///media/<em class="replaceable"><code>ISOimage</code></em>/RPM-GPG-KEY
gpgcheck=1
enabled=1</pre>
      <p>Adjust the value of the <code class="literal">baseurl</code> and <code class="literal">gpgkey</code> parameters to match the mount point of the ISO image on your system. If you do not require one of the repositories, set the value of the corresponding <code class="literal">enabled</code> parameter to 0.</p>
     </div>
    </div>
    <div class="section">
     <div class="titlepage">
      <div>
       <div>
        <h2 class="title" style="clear: both"><a name="ol_appup_rn64"></a>3.3.&nbsp;Applying Updates</h2>
       </div>
      </div>
     </div>
     <p>Once you have set up the ULN channels, Public Yum repositories, or installation media repositories that <span class="command"><strong>yum</strong></span> should use, you can update all installed packages by running the following command:</p>
     <pre class="programlisting"># yum update</pre>
     <p>If your system is currently installed with Oracle Linux 6 GA, Update 1, Update 2, or Update 3, this command upgrades it to Update 4.</p>
     <p>You can use the following command to update a specific package:</p>
     <pre class="programlisting"># yum update <em class="replaceable"><code>package</code></em></pre>
     <p>For example, to update the Z-shell package (<code class="literal">zsh</code>), you would enter:</p>
     <pre class="programlisting"># yum update zsh</pre>
     <p>For more information, see the <code class="literal">yum(8)</code> manual page.</p>
    </div>
    <div class="section">
     <div class="titlepage">
      <div>
       <div>
        <h2 class="title" style="clear: both"><a name="ol_upuek2_rn64"></a>3.4.&nbsp;Upgrading Unbreakable Enterprise Kernel Release 2</h2>
       </div>
      </div>
     </div>
     <p>Oracle Linux 6 Update 4 ships with the latest Unbreakable Enterprise Kernel Release 2. If you upgrade your system from the installation media, there are two upgrade scenarios:</p>
     <div class="itemizedlist">
      <ul class="itemizedlist" style="list-style-type: disc; ">
       <li class="listitem"><p>If UEK R2 is not currently installed on the system, only the latest Red Hat Compatible Kernel is installed. The UEK R2 kernel is not installed.</p></li>
       <li class="listitem"><p>If UEK R2 is currently installed on the system, the latest version of that kernel is installed.</p></li>
      </ul>
     </div>
     <p><span class="command"><strong>yum</strong></span> uses whatever repositories you have configured on your system to upgrade it. You can find the latest UEK2 packages in the <code class="literal">ol6_i386_UEK_latest</code> and <code class="literal">ol6_x86_64_UEK_latest</code> repositories. If you want to install the latest UEK R2 kernel, subscribe your system to the correct channel on ULN, or configure the repository in the <code class="literal">/etc/yum.repos.d/public-yum-ol6.repo</code> file as shown here:</p>
     <pre class="programlisting">[ol6_UEK_latest]
name=Latest Unbreakable Enterprise Kernel for Oracle Linux $releasever ($basearch)
baseurl=http://public-yum.oracle.com/repo/OracleLinux/OL6/UEK/latest/$basearch/
gpgkey=http://public-yum.oracle.com/RPM-GPG-KEY-oracle-ol6
gpgcheck=1
enabled=1</pre>
