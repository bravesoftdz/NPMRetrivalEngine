         <h1 class="title topictitle1" id="ariaid-title1">Troubleshooting Kerberos</h1> 
         <div class="body conbody">
          <p class="shortdesc">Provide solutions to common Kerberos difficulties. </p> 
          <section class="section"> 
           <p class="p">Java errors from Kerberos problems can be obscure and difficult to interpret. To see the Kerberos error messages, enable Kerberos debugging by adding these settings to your JVM:</p> 
           <pre class="pre codeblock"><code>-Dsun.security.krb5.debug=true -Dsun.security.spnego.debug=true -Djavax.net.debug=all</code></pre> 
           <p class="p">You can also enable Kerberos debugging for the MapR-provided <code class="ph codeph">maprcli</code> and Hadoop clients by adding the following line to the <code class="ph codeph">env.sh</code> shell script:</p> 
           <pre class="pre codeblock"><code>#MAPRLOGIN_OPTS="${MAPRLOGIN_OPTS} -Dsun.security.krb5.debug=true -Dsun.security.spnego.debug=true -Djavax.net.debug=all"</code></pre> 
           <p class="p">Capture the Kerberos error to research the issue.</p> 
           <p class="p">The following sections list common Kerberos error conditions:</p> 
          </section> 
          <section class="section">
           <h2 class="title sectiontitle"> Incorrect JVM</h2> 
           <p class="p">Nodes often have multiple Java Virtual Machines (JVM) installed. The MapR <code class="ph codeph">env.sh</code> script automatically configures a JVM to use. To change the automatically configured JVM, set the value of the <code class="ph codeph">JAVA_HOME</code> environment variable in the <code class="ph codeph">/opt/mapr/conf/env.sh</code> file.</p> 
          </section> 
          <section class="section">
           <h2 class="title sectiontitle"> Incorrect Server Name</h2>

          <section class="section">
           <h2 class="title sectiontitle"> Invalid or missing keytab file</h2> 
           <p class="p">The <code class="ph codeph">keytab</code> file must be consistent with the key versions of the Kerberos principal. The following example shows an inconsistent keytab file:</p> 
           <pre class="pre codeblock"><code>kadmin:  getprinc mapr/realm1
        Principal: mapr/realm1@mapr
        Expiration date: [never]
        Last password change: Thu May 23 15:36:01 PDT 2013
        Password expiration date: [none]
        Maximum ticket life: 0 days 10:00:00
        Maximum renewable life: 7 days 00:00:00
        Last modified: Thu May 23 15:36:01 PDT 2013 (mapr/admin@mapr)
        Last successful authentication: Thu May 23 19:31:59 PDT 2013
        Last failed authentication: Thu May 23 15:35:41 PDT 2013
        Failed password attempts: 0
        Number of keys: 8
        Key: vno 15, aes256-cts-hmac-sha1-96, no salt
        Key: vno 15, arcfour-hmac, no salt
        Key: vno 15, des3-cbc-sha1, no salt
        Key: vno 15, des-cbc-crc, no salt
        Key: vno 15, des-cbc-md5, Version 4
        Key: vno 15, des-cbc-md5, Version 5 - No Realm
        Key: vno 15, des-cbc-md5, Version 5 - Realm Only
        Key: vno 15, des-cbc-md5, AFS version 3
        MKey: vno 1
        
        ktutil:  rkt mapr.keytab
        ktutil:  l
        slot KVNO Principal
        ---- ---- ---------------------------------------------------------------------
        1   14             mapr/realm1@mapr
        2   14             mapr/realm1@mapr
        3   14             mapr/realm1@mapr
        4   14             mapr/realm1@mapr
        ktutil:  q</code></pre> 
           <p class="p">Note that the key versions in the Kerberos principal <code class="ph codeph">/realm1</code> are 15, and the versions in the <code class="ph codeph">keytab</code> file are 14. This mismatch can result in errors about missing keys or mismatched encryption.</p> 
           <div class="note note">
            <span class="notetitle">Note:</span> 
            <p class="p">This error state can also be caused by the <code class="ph codeph">/opt/mapr/conf/mapr.keytab</code> file not being owned by the user <code class="ph codeph">mapr</code> or not being present. The <code class="ph codeph">keytab</code> file is owned by <code class="ph codeph">root</code> at generation. Be sure to use the <code class="ph codeph">chown</code> command to set the <code class="ph codeph">mapr</code> user as the owner:</p> 
            <pre class="pre codeblock"><code>$ chown mapr:mapr /opt/mapr/conf/mapr.keytab</code></pre> 
           </div> 
          </section> 
          <section class="section">
           <h2 class="title sectiontitle"> Incompatible encryption on Java runtime</h2> 
           <p class="p">Incompatible cryptography between the KDC and the JDK results in failed handshakes, leading to errors similar to the following:</p> 
           <pre class="pre codeblock"><code>Caused by: javax.security.auth.login.LoginException: Unable to obtain Principal Name for authentication </code></pre> 
           <p class="p">With debugging active, the following message is displayed:</p> 
           <pre class="pre codeblock"><code>&gt;&gt;&gt;DEBUG &lt;CCacheInputStream&gt;  client principal is username@hostname
        &gt;&gt;&gt;DEBUG &lt;CCacheInputStream&gt; server principal is X-CACHECONF:/krb5_ccache_conf_data/fast_avail/krbtgt/user@hostname
        &gt;&gt;&gt;DEBUG &lt;CCacheInputStream&gt; key type: 0
        &gt;&gt;&gt;DEBUG &lt;CCacheInputStream&gt; auth time: Wed Dec 31 16:00:00 PST 1969
        &gt;&gt;&gt;DEBUG &lt;CCacheInputStream&gt; start time: null
        &gt;&gt;&gt;DEBUG &lt;CCacheInputStream&gt; end time: Wed Dec 31 16:00:00 PST 1969
        &gt;&gt;&gt;DEBUG &lt;CCacheInputStream&gt; renew_till time: null
        &gt;&gt;&gt; CCacheInputStream: readFlags() 
        &gt;&gt;&gt; unsupported key type found the default TGT: 18
        Negotiate support not initiated, will fallback to other scheme if allowed. Reason:</code></pre> 
           <p class="p">This debug message indicates that the problem is an unsupported key type.</p> 
           <p class="p">Incompatible encryption errors can occur due to a <code class="ph codeph">keytab</code> file that is not present or contains outdated keys.</p> 
           <p class="p">Be sure to update the Java jurisdiction policy file. Jurisdiction policy files are available from <a class="xref" href="http://www.oracle.com/technetwork/java/javase/downloads/index.html" target="_blank">Oracle</a>.</p> 
           <p class="p">A persistent encryption incompatibility problem may require you to edit the <code class="ph codeph">krb5.conf</code> file to ensure compatible algorithms between Java and Kerberos.</p> 
          </section>

          <section class="section">
           <h2 class="title sectiontitle"> Kerberos and PAM validation</h2> 
           <p class="p">Standard Kerberos implementations are predicated on access to elevated user privileges that are not present on secure MapR clusters. In a MapR cluster, the MCS console and other components call PAM as an ordinary user process. This discrepancy in expected and actual privileges can cause a variety of obscure file permisson errors. Since different Kerberos PAM modules are available, error reports can vary.</p> 
           <p class="p">To diagnose this issue, attempt starting the MCS as the root user, or clear out the <code class="ph codeph">/tmp</code> folder. If there are no problems when starting MCS as root, or if clearing out the <code class="ph codeph">/tmp</code> folder enables a single login before errors appear again, the problem may lie in the Kerberos PAM configuration.</p> 
           <p class="p">To resolve this condition, prevent Kerberos from creating a ticket file. MapR security does not use Kerberos tickets. The Kerberos KDC is used to validate passwords. Typically the configuration file for PAM is in the <code class="ph codeph">/etc/pam.d</code> directory. See the documentation for your specific Kerberos PAM module for more information.</p> 
          </section>
