      <div class="entry-content"> 
       <p>With the coming changes in ASP.NET Core, our friend and intrepid reporter <a href="https://twitter.com/sethjuarez" target="_blank">Seth Juarez</a> sat down with ASP.NET Program Manager <a href="https://twitter.com/rustd" target="_blank">Pranav Rastogi</a> to discuss the updates and improvements in the new ASP.NET Core authentication system:</p> 
       <p><iframe src="https://channel9.msdn.com/Blogs/Seth-Juarez/ASPNET-Core-Authentication-with-Pranav-Rastogi/player" width="960" height="540" frameborder="0" allowfullscreen></iframe></p> 
       <p>Here are some of the highlights of their discussion and some sample code to get you started:</p> 
       <p>Pranav gave a quick definition of authentication compared to authorization:&nbsp; Authentication validates who the user is and authorization validates access to the actions a user wants to perform.</p> 
       <p>The authentication scenarios in ASP.NET are still the same as in previous versions: OAuth can be enabled with providers available for Facebook, Twitter, Microsoft Account, and Google.&nbsp; You can still manage a database of your own users with their own passwords on your application.&nbsp; Additionally, two-factor authentication is easy to add to your application with code samples commented out of the initial ASP.NET Core project templates.</p> 
       <p>The four authentication options by default in a new web application are:</p> 
       <ul> 
        <li>No authentication</li> 
        <li>Individual User Accounts – a security database will be created and you can configure individuals to access your application with a userid and password, an OAuth provider, or two-factor authentication, or any combination of these items.</li> 
        <li>Work and School Accounts – Using Azure Active Directory</li> 
        <li>Windows Authentication</li> 
       </ul> 
       <p>Configuration of your controllers and actions to restrict access to logged on users are very similar to how they behaved before, with the added benefit of being able to configure security policies that define a collection of security claims that a user has been granted.&nbsp; Claims can even be checked in the server-side code of razor views to determine segments of content to deliver to web visitors.</p> 
       <p>In the individual user account model, all access of the database goes through Entity Framework models by default.&nbsp; You can change the connections and structure of the repository used by Entity Framework to meet your needs, and entity framework will generate or connect to your repository appropriately.</p> 
       <p>If you are using Yeoman generators to start an application, they will allow you to choose templates that will either create an application with security enabled similar to Individual User Accounts OR without authentication.&nbsp; The secured code generated by Yeoman will use the Entity Framework SQLite provider in order to deliver a cross-platform compatible experience.</p> 
       <p>The following NuGet packages deliver the features necessary to enable individual user account access:</p> 
       <ul> 
        <li>Microsoft.AspNetCore.Authentication.Cookies</li> 
        <li>Microsoft.AspNetCore.Authentication.Facebook</li> 
        <li>Microsoft.AspNetCore.Authentication.Google</li> 
        <li>Microsoft.AspNetCore.Authentication.MicrosoftAccount</li> 
        <li>Microsoft.AspNetCore.Authentication.Twitter</li> 
        <li>Microsoft.AspNetCore.Identity.EntityFramework</li> 
       </ul> 
       <p>The packages for Facebook, Google, MicrosoftAccount, and Twitter are only needed if you are going to enable OAuth authentication from those sources.</p> 
       <p>Configure the entity framework storage with these two segments in Startup.cs – ConfigureServices method:</p> 
       <script src="https://gist.github.com/csharpfritz/3494551293f5a862d40c.js"></script> 
       <p>The default user model is stored in /Models/ApplicationUser.cs&nbsp; You can add properties to this class to store those properties with the user identified in your application.&nbsp; In the default project template, the username will be the same as the user’s email address.&nbsp; If you add properties to your user model, you will also want to expand the RegisterViewModel appropriately to receive any extra fields that you want your users to submit at registration time.</p> 
       <p>Default account management and authentication views are stored in the /Views/Account The Controllers/AccountController contains the actions to manage the user.&nbsp; The AccountController receives and provides a UserManager to manage the data about a user and the SignInManager is used to handle verification of a user’s credentials.</p> 
       <p>The _LoginPartial.cshtml demonstrates how to inspect the user object and determine if the user is authenticated and how to fetch the username</p> 
       <p>If you want to allow authentication from a third party provider like Facebook, there are <a href="https://azure.microsoft.com/en-us/documentation/articles/mobile-services-how-to-register-facebook-authentication/" target="_blank">guides on MSDN </a>that will instruct you how to establish an account with that provider and capture your app keys to access that service.&nbsp; These keys should be retrieved from configuration and stored for configuration using the secret manager.</p>
