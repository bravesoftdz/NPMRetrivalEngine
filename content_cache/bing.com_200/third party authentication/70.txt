      <h1 class="email-content-title">3rd party authentication and web api usage from the browser</h1> 
      <p class="email-content-subtitle"> Tag: <a href="http://www.howtobuildsoftware.com/index.php/built-with/javascript">javascript</a>,<a href="http://www.howtobuildsoftware.com/index.php/built-with/authentication">authentication</a>,<a href="http://www.howtobuildsoftware.com/index.php/built-with/client-side">client-side</a>,<a href="http://www.howtobuildsoftware.com/index.php/built-with/web-api">web-api</a>,<a href="http://www.howtobuildsoftware.com/index.php/built-with/third-party-api">third-party-api</a> </p> 
     </div> 
     <div class="email-content-body"> 
      <p> </p>
      <p>I'm having trouble wrapping my head around some concepts revolving around 3rd party authentication in a browser based webapp as well as requesting user information from these 3rd party application's web api. I'm pretty new to this topic so I'm trying to describe my understanding of the matter and ask my questions afterwards.</p> 
      <p>From what I understand the authentication process (on a high level) using a 3rd party application like twitter or facebook works as follows:</p> 
      <ol> 
       <li><p>Client (Browser) initiates the authorization process by requesting a specific url for this purpose e.g. <code>auth/facebook</code></p></li> 
       <li><p>Server handles this request by requesting user information/validation from the 3rd party application providing some <code>client ID</code> and <code>client secret</code> as well as a callback url.</p></li> 
       <li><p>Client gets redirected to some authorization page of the 3rd party application where he grants access to my application to use his information.</p></li> 
       <li><p>The 3rd party application calls my callback url and provides user information as well as some <code>id</code></p></li> 
       <li><p>My application now knows that this user is authenticated and can also do any kind of request to the 3rd party application "on behalf" of the User who gave me permission (restricted to the authorization level I requested).</p></li> 
      </ol> 
      <p>Assuming I got this more or less right, here comes my main question: What if, now that I got permission to the users information etc., I want to make requests to the 3rd party application from the browser directly. I would have to send my <code>client ID</code> / <code>client secret</code> to the browser in order to make requests directly from the browser. Isn't this a big security issue? Aren't these informations highly susceptible to man in the middle attacks and subsequently allow an attacker to use this information to request information from the 3rd party application?</p> 
      <p>Am I missing something? Or how are webapps using 3rd party webapis generally built?</p> 
      <p>Any help/hint is very welcome!</p> 
      <p></p>

      <p> </p>
      <p>The sort of third party authentication you're describing above is OAuth. Most providers now-a-days are using Oauth2: <a href="https://aaronparecki.com/articles/2012/07/29/1/oauth2-simplified" rel="nofollow">https://aaronparecki.com/articles/2012/07/29/1/oauth2-simplified</a> (this is a great blog post which explains it pretty simply).</p> 
      <p>Once a user has granted your application access to their account, you'll be given an access token (a long, random string) that can be used as an API key to access the provider's API.</p> 
      <p>So for instance, let's say you get a user to grant you Facebook access via their OAuth API. Once the user has accepted your permissions, you'll be given a token that you will store in the browser.</p> 
      <p>Using that token, you can then fire off API requests to the Facebook API service using that token in the HTTP Authorization header like so:</p> 
      <p>Authorization: Bearer </p> 
      <p>This will successfully identify you, and grant you access.</p>
