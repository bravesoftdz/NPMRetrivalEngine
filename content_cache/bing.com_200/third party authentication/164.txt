      <h1 class="entry-title">OAUTH 2.0 AUTHENTICATION FOR THIRD PARTY APPLICATIONS</h1> 
      <div class="entry-meta"> 
       <span class="entry-date"><a href="http://blog.mirketa.com/oauth-2-0-authentication-for-third-party-applications/" title="6:07 am" rel="bookmark">07/25/2016</a></span> by 
       <span class="author vcard"><a class="url fn n" href="http://blog.mirketa.com/author/sunaiyana/" title="View all posts by Sunaiyana Thakuria">Sunaiyana Thakuria</a></span> 
       <span class="comments-link"> · <a href="http://blog.mirketa.com/oauth-2-0-authentication-for-third-party-applications/#respond">Leave a comment</a></span> 
      </div>
      <!-- .entry-meta --> 
      <div class="entry-content"> 
       <p style="text-align: left;"><span style="text-align: left;">Most of the APIs now-a-days incorporate oauth 2.0 authentication. It is not as complicated as it may seem at times, provided the right links and documentation are found.&nbsp;Microsoft APIs are extremely helpful and useful, but to access them from a third party application is when its needed, for the entire process of registration and access token retrieval, to be followed; to comply with the oauth authentication in place.</span></p> 
       <p>For office365 (2016) APIs the links that would help access these APIs would be of the format&nbsp;<em><a href="https://outlook.office.com/api/%7bversion%7d/me/">https://outlook.office.com/api/{version}/me/</a><span style="text-decoration: underline;"></span></em></p> 
       <p style="margin-left: 40px;">me  represents the logged in user<br> {version}  v2.0 or v1.0<br> /  events (for outlook calendar API)</p> 
       <p><em><strong>The first step</strong></em>, as hundreds of websites mentions, is to register the application. To be a little more comprehensive on this point, I would like to mention that it is not required to deploy any kind of code or application into the registration portal.</p> 
       <p><em><a href="https://apps.dev.microsoft.com/">https://apps.dev.microsoft.com</a></em> is free if you have access to a Microsoft office account.</p> 
       <p>The registration process is a way of letting Microsoft know that a particular app is going to access its APIs. It is a good practice to name the app appropriately as it will appear on the screen when the application, that is being developed, navigates to the login page.</p> 
       <p><a href="http://www.mirketa.com/blog/wp-content/uploads/2016/07/mirketa_oAuthAppRegistration.png"><img class="wp-image-1358 aligncenter" alt="mirketa_oAuthAppRegistration" src="http://www.mirketa.com/blog/wp-content/uploads/2016/07/mirketa_oAuthAppRegistration.png" width="656" height="369"></a></p> 
       <p><em><strong>Application id</strong></em> is the client id which is needed to be provided in the headers when requests are made for authorisation code and access token.<br> <span id="more-1357"></span><br> Clicking on Generate new password will provide with a client secret. New key pairs can be generated as often as needed, but in case the developed application is deployed in a client environment, these details will have to be provided at the very beginning and there would be little chance for frequent changes. Therefore, as client secret will be displayed just one time it is advisable to make a note of it. These are generally valid for two years.</p> 
       <p><em><strong>Add platform button</strong></em> lets choose which type of application is going to be developed.</p> 
       <p><a href="http://www.mirketa.com/blog/wp-content/uploads/2016/07/mirketa-oAuthAddPlatform.png"><img class="wp-image-1359 aligncenter" alt="mirketa-oAuthAddPlatform" src="http://www.mirketa.com/blog/wp-content/uploads/2016/07/mirketa-oAuthAddPlatform.png" width="631" height="355"><br> </a>Definitely both can be selected (one at a time).</p> 
       <p><a href="http://www.mirketa.com/blog/wp-content/uploads/2016/07/mirketa_oAuthAddPlatform2.png"><img class="wp-image-1360 aligncenter" alt="mirketa_oAuthAddPlatform2" src="http://www.mirketa.com/blog/wp-content/uploads/2016/07/mirketa_oAuthAddPlatform2.png" width="631" height="355"></a></p> 
       <p>&nbsp;</p> 
       <p><em><strong>The most important step</strong></em> while registration is to take extra care when the redirect-uri is added. This is where Microsoft will send the access and refresh token after authentication.<br> Redirect-uris are very crucial and it is to be made absolutely sure that the links provided are available in the application and requests and responses to those links can be monitored. Multiple uris can be added here as well.</p> 
       <p>(N.B.- Uris in the format <em><a href="http://localhost:8080/">http://localhost:8080</a></em> are accepted, however any uri with a different system name appearing in place of localhost will not be accepted.)</p> 
       <p>OAuth 2.0 is a two-step authentication. The first layer sends an authorisation code, which could be referred to as a part of the key to the second layer which gives the access token.<br> <em><strong>To get the part of the key</strong> </em>it is required to execute a get request for the authorisation code.</p> 
       <p>The link to which the request is sent to:  &nbsp;<em><a href="https://login.microsoftonline.com/common/oauth2/v2.0/authorize">https://login.microsoftonline.com/common/oauth2/v2.0/authorize</a></em>.</p> 
       <p>The request header must contain: </p> 
       <p style="margin-left: 40px;">Response-type = code (always code for receiving authorisation code)<br> Client-id = your client id<br> Redirect-uri = uri which is available in your application and also registered with Microsoft dev portal for the &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp; &nbsp;same client id<br> Scope = what functions your app will be performing<br> (eg:-&nbsp;<a href="http://dfv"><i>openid offline_access profile https://outlook.office.com/mail.readwrite https://outlook.office.com/mail.send</i></a>)</p> 
       <p>This get request will take the application to the Microsoft login page where in a valid Microsoft user name and password will have to be provided and necessary permissions must be granted. Completion of this step will redirect the application to the redirect uri with an authorisation code.</p> 
       <p>This authorisation code can be captured from the query string parameters of the request.</p> 
       <p><em><strong>The next step</strong></em> is to obtain the access token which can be done by sending a post request to <em><a href="https://login.microsoftonline.com/common/oauth2/v2.0/token">https://login.microsoftonline.com/common/oauth2/v2.0/token</a></em>.</p> 
       <p>The post request for access token should contain: </p> 
       <p style="margin-left: 40px;">Grant-type = authorisation-code (captured from last request)<br> Scope= same as previous request<br> Redirect-uri=needs to be same as previous url or else this would give an error<br> Client-id=same client-id<br> Client-secret=the client secret that had been noted down</p> 
       <p>Sharing of client-secret is deemed as an offence and hence no website will ever share that detail and hence the registration process becomes even more pivotal.</p> 
       <p>Java code snippet to form the post request: </p> 
       <p style="margin-left: 40px;"><em>List&lt;NameValuePair&gt; pairs = new ArrayList&lt;NameValuePair&gt; ();</em><br> <em>pairs.add (new BasicNameValuePair (grant_type,authorization_code));</em><br> <em>pairs.add (new BasicNameValuePair (code, authorisation code));</em><br> <em>pairs.add (new BasicNameValuePair (scope, scope mentioned in get request for auth code));</em><br> <em>pairs.add (new BasicNameValuePair (redirect_uri, your redirect uri);</em><br> <em>pairs.add (new BasicNameValuePair (client_id, your client id);</em><br> <em>pairs.add (new BasicNameValuePair (client_secret, your client secret));</em></p> 
       <p><em>&nbsp; &nbsp; &nbsp; &nbsp; &nbsp; HttpPost post = new HttpPost(https://login.microsoftonline.com/common/oauth2/v2.0/token);</em><br> <em>&nbsp; &nbsp; &nbsp; &nbsp; &nbsp; HttpEntity postParams = new UrlEncodedFormEntity(pairs);</em></p> 
       <p><em>&nbsp; &nbsp; &nbsp; &nbsp; &nbsp; post.setHeader(Content-Type,application/x-www-form-urlencoded);</em><br> <em>&nbsp; &nbsp; &nbsp; &nbsp; &nbsp; post.setEntity(postParams);</em></p> 
       <p>This post request will again redirect the application to the redirect uri mentioned and this time the response content will contain the access token and the refresh token.<br> The access tokens for different APIs have different expiration time and for Microsoft APIs they are mostly valid for an hour. To avoid the pain of logging in again and again refresh tokens can be used to keep retrieving fresh access tokens. The refresh tokens look similar to the authorisation code and the post request for access token using refresh token are the same except that grant-type is needed to be set to refresh-token and in place of authorisation code the refresh token should be mentioned.</p> 
       <p>Once the access token is received, the authentication procedure is complete and these tokens can be used to access the endpoint APIs. The tokens can be saved to a file or a variable for the required function.</p> 
       <p>Requests to different APIs can vary widely, however, it is of utmost importance that the request header contains: -authorization = Bearer your-access-token, to perform the necessary actions.<br> ( eg: post.setHeader(authorization ,Bearer +access-token);).</p> 
       <p>&nbsp;</p> 
       <p>Key Points: </p> 
       <ul> 
        <ol> 
         <li>Registration of application.</li> 
         <li>Get request to authorisation API to receive authorisation code.</li> 
         <li>Post request to access token API to receive access token and refresh token.</li> 
         <li>Post or get (depending on requirement) to the endpoint API using the access token received.</li> 
        </ol> 
       </ul>
