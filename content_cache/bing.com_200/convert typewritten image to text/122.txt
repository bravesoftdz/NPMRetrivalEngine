      <h1 class="title">Data Type Conversion (Database Engine)</h1> 
      <div class="lw_vs"> 
       <div id="versionclick"> 
        <div>

      <div id="mainSection"> 
       <div id="mainBody"> 
        <p></p> 
        <div class="introduction">
         <p>Data types can be converted in the following scenarios:</p>
         <ul>
          <li><p>When data from one object is moved to, compared with, or combined with data from another object, the data may have to be converted from the data type of one object to the data type of the other.</p></li>
          <li><p>When data from a Transact-SQL result column, return code, or output parameter is moved into a program variable, the data must be converted from the SQL Server&nbsp;system data type to the data type of the variable.</p></li>
         </ul>
         <p>Data types can be converted either implicitly or explicitly:</p>
         <ul>
          <li><p>Implicit conversions are not visible to the user. </p><p>SQL Server automatically converts the data from one data type to another. For example, when a <span><span class="input">smallint</span></span> is compared to an <span><span class="input">int</span></span>, the <span><span class="input">smallint</span></span> is implicitly converted to <span><span class="input">int</span></span> before the comparison proceeds. Note that the query optimizer may generate a query plan to perform this conversion at any time. This may cause a runtime error for conversion failures such as loss of precision and an attempt to convert a nonnumeric string to a number. For more information, see <span><a href="https://technet.microsoft.com/en-us/library/ms188295(v=sql.105).aspx">Troubleshooting Errors and Warnings on Query Expressions</a></span>.</p></li>
          <li><p>Explicit conversions use the CAST or CONVERT functions.</p><p>The <a href="https://technet.microsoft.com/en-us/library/ms187928(v=sql.105).aspx">CAST and CONVERT</a> functions convert a value (a local variable, a column, or another expression) from one data type to another. For example, the following <span class="code">CAST</span> function converts the numeric value of <span class="code">$157.27</span> into a character string of <span class="code">'157.27'</span>:</p> 
           <div id="code-snippet-1" class="codeSnippetContainer" xmlns=""> 
            <div class="codeSnippetContainerTabs"> 
            </div> 
            <div class="codeSnippetContainerCodeContainer"> 
             <div class="codeSnippetToolBar"> 
              <div class="codeSnippetToolBarText"> 
               <a name="CodeSnippetCopyLink" style="display: none;" title="Copy to clipboard." href="javascript:if (window.epx.codeSnippet)window.epx.codeSnippet.copyCode('CodeSnippetContainerCode_93ad2936-85d1-40fb-98c6-da864aad3d7e');" ms.cmptyp="CodeSnippet">Copy</a> 
              </div> 
             </div> 
             <div id="CodeSnippetContainerCode_93ad2936-85d1-40fb-98c6-da864aad3d7e" class="codeSnippetContainerCode" dir="ltr"> 
              <div style="color:Black;">
               <pre>
CAST ( $157.27 AS VARCHAR(10) )
</pre>
              </div> 
             </div> 
            </div> 
           </div> <p>Use CAST instead of CONVERT if you want Transact-SQL program code to comply with ISO. Use CONVERT instead of CAST to take advantage of the style functionality in CONVERT.</p></li>
         </ul>
         <p>Some implicit and explicit data type conversions are not supported when you are converting the data type of one SQL Server object to another. For example, an <span><span class="input">nchar</span></span> value cannot be converted to an <span><span class="input">image</span></span> value. An <span><span class="input">nchar</span></span> can only be converted to <span><span class="input">binary</span></span> by using explicit conversion, an implicit conversion to <span><span class="input">binary</span></span> is not supported. However, an <span><span class="input">nchar</span></span> can be explicitly or implicitly converted to <span><span class="input">nvarchar</span></span>. </p>
         <p>When handling the <span><span class="input">sql_variant</span></span> data type, SQL Server supports implicit conversions of objects with other data types to the <span><span class="input">sql_variant</span></span> type. However, SQL Server does not support implicit conversions from <span><span class="input">sql_variant</span></span> data to an object with another data type.</p>
         <p>For more information about supported conversions between SQL Server objects, see <span><a href="https://technet.microsoft.com/en-us/library/ms187928(v=sql.105).aspx">CAST and CONVERT (Transact-SQL)</a></span>. </p>
         <p>When you convert between an application variable and a SQL Server result set column, return code, parameter, or parameter marker, the supported data type conversions are defined by the database API. For more information, see <span><a href="https://technet.microsoft.com/en-us/library/ms189969(v=sql.105).aspx">Moving Data to Program Variables</a></span>.</p>
        </div>

          <a id="_binary"></a>
          <h3 class="subHeading">Converting binary and varbinary Data</h3>
          <div class="subsection">
           <p>When data is converted from a string data type (<span><span class="input">char</span></span>, <span><span class="input">varchar</span></span>, <span><span class="input">nchar</span></span>, <span><span class="input">nvarchar</span></span>, <span><span class="input">binary</span></span>, <span><span class="input">varbinary</span></span>, <span><span class="input">text</span></span>, <span><span class="input">ntext</span></span>, or <span><span class="input">image</span></span>) to a <span><span class="input">binary</span></span> or <span><span class="input">varbinary</span></span> data type of unequal length, SQL Server pads or truncates the data on the right. When other data types are converted to <span><span class="input">binary</span></span> or <span><span class="input">varbinary</span></span>, the data is padded or truncated on the left. Padding is achieved by using hexadecimal zeros.</p>
           <p>Converting data to the <span><span class="input">binary</span></span> and <span><span class="input">varbinary</span></span> data types is useful if <span><span class="input">binary</span></span> data is the easiest way to move data around. Converting any value of any type to a binary value of large enough size and then back to the type, will always result in the same value if both conversions are taking place on the same version of SQL Server. The binary representation of a value might change from version to version of SQL Server.</p>
           <p>You can convert <span><span class="input">int</span></span>, <span><span class="input">smallint</span></span>, and <span><span class="input">tinyint</span></span> to <span><span class="input">binary</span></span> or <span><span class="input">varbinary</span></span>, but if you convert the <span><span class="input">binary</span></span> value back to an integer value, this value will be different from the original integer value if truncation has occurred. For example, the following <span class="code">SELECT</span> statement shows that the integer value <span class="code">123456</span> is usually stored as a binary <span class="code">0x0001e240</span>:</p> 
           <div id="code-snippet-2" class="codeSnippetContainer" xmlns=""> 
            <div class="codeSnippetContainerTabs">

          <h3 class="subHeading">Converting to bit Data</h3>
          <div class="subsection">
           <p>Converting to <span><span class="input">bit</span></span> promotes any nonzero value to 1.</p>
          </div>
          <a id="_character"></a>
          <h3 class="subHeading">Converting Character Data</h3>
          <div class="subsection">
           <p>When character expressions are converted to a character data type of a different size, values that are too long for the new data type are truncated. The <span><span class="input">uniqueidentifier</span></span> type is considered a character type for the purposes of conversion from a character expression, and therefore is subject to the truncation rules for converting to a character type. For more information, see <span><a href="https://technet.microsoft.com/en-us/library/ms187942(v=sql.105).aspx">uniqueidentifier (Transact-SQL)</a></span>.</p>
           <p>When a character expression is converted to a character expression of a different data type or size, such as from <span><span class="input">char(5)</span></span> to <span><span class="input">varchar(5)</span></span>, or <span><span class="input">char(20)</span></span> to <span><span class="input">char(15)</span></span>, the collation of the input value is assigned to the converted value. If a noncharacter expression is converted to a character data type, the default collation of the current database is assigned to the converted value. In either case, you can assign a specific collation by using the <a href="https://technet.microsoft.com/en-us/library/ms184391(v=sql.105).aspx">COLLATE</a> clause.</p>
           <div class="alert">
            <table summary="table">
             <tbody>
              <tr>
               <th scope="col"><img id="alert_note" alt="Note" src="https://i-technet.sec.s-msft.com/areas/global/content/clear.gif" title="Note" xmlns="" class="cl_IC101471"><strong>Note</strong></th>
              </tr>
              <tr>
               <td><p>Code page translations are supported for <span><span class="input">char</span></span> and <span><span class="input">varchar</span></span> data types, but not for <span><span class="input">text</span></span> data type. As with earlier versions of SQL Server, data loss during code page translations is not reported.</p></td>
              </tr>
             </tbody>
            </table>
           </div>
           <p>Character expressions that are being converted to an approximate <span><span class="input">numeric</span></span> data type can include optional exponential notation (a lowercase e or uppercase E followed by an optional plus (+) or minus (-) sign and then a number).</p>
           <p>Character expressions that are being converted to an exact <span><span class="input">numeric</span></span> data type must consist of digits, a decimal point, and an optional plus (+) or minus (-). Leading blanks are ignored. Comma separators, such as the thousands separator in 123,456.00, are not allowed in the string.</p>
           <p>Character expressions being converted to <span><span class="input">money</span></span> or <span><span class="input">smallmoney</span></span> data types can also include an optional decimal point and dollar sign ($). Comma separators, as in $123,456.00, are allowed.</p>
           <p>The following example shows how to convert data for display. This example converts sales data to character data before performing a string comparison and converts the current date to style 3, dd/mm/yy.</p> 
           <div id="code-snippet-5" class="codeSnippetContainer" xmlns=""> 
            <div class="codeSnippetContainerTabs"> 
            </div>

             <div class="codeSnippetToolBar">

              </div> 
             </div> 
             <div id="CodeSnippetContainerCode_1c671c08-4383-4d15-9caf-f8eb8d279fca" class="codeSnippetContainerCode" dir="ltr">

   CAST(SalesYTD AS varchar(12)),
   CONVERT(VARCHAR(12), GETDATE(), 3)

GO
</pre>
              </div> 
             </div> 
            </div> 
           </div> 
           <p>This example converts a <span class="code">uniqueidentifier</span> value to a <span class="code">char</span> data type.</p>

          <a id="_datetime"></a>
          <h3 class="subHeading">Converting Date and Time Data</h3>
          <div class="subsection">
           <p>When you convert to <span><span class="input">date</span></span> and <span><span class="input">time</span></span> data types, SQL Server&nbsp;rejects all values it cannot recognize as dates or times. For an overview of all Transact-SQL date and time data types and functions, see <span><a href="https://technet.microsoft.com/en-us/library/ms186724(v=sql.105).aspx">Date and Time Functions (Transact-SQL)</a></span>.</p>
           <p>The following example converts <span class="code">date</span> and <span class="code">datetime2</span> values to <span class="code">varchar</span> and <span class="code">binary</span> data types, respectively.</p> 
           <div id="code-snippet-8" class="codeSnippetContainer" xmlns=""> 
            <div class="codeSnippetContainerTabs"> 
            </div> 
            <div class="codeSnippetContainerCodeContainer"> 
             <div class="codeSnippetToolBar"> 
              <div class="codeSnippetToolBarText"> 
               <a name="CodeSnippetCopyLink" style="display: none;" title="Copy to clipboard." href="javascript:if (window.epx.codeSnippet)window.epx.codeSnippet.copyCode('CodeSnippetContainerCode_ec07a006-efa1-451f-b979-8a273e4568e1');" ms.cmptyp="CodeSnippet">Copy</a> 
              </div> 
             </div> 
             <div id="CodeSnippetContainerCode_ec07a006-efa1-451f-b979-8a273e4568e1" class="codeSnippetContainerCode" dir="ltr"> 
              <div style="color:Black;">
               <pre>
DECLARE @mydate date;
SET @mydate = '4/05/98';

SELECT CAST(@mydate AS varchar) AS DATE_VARCHAR;
GO

DECLARE @mydate  datetime2;
SET @mydate     = '4/05/98';

SELECT  CAST(@mydate AS binary) AS DATE_BINARY;
GO
</pre>
              </div> 
             </div> 
            </div> 
           </div> 
           <p>Here is the result set.</p>
           <p><span class="code">(1 row(s) affected)</span></p>
           <p></p>
           <p><span class="code">DATE_VARCHAR </span></p>
           <p><span class="code">------------------------------ </span></p>
           <p><span class="code">Apr 5 1998 </span></p>
           <p></p>
           <p><span class="code">(1 row(s) affected)</span></p>
           <p></p>
           <p><span class="code">DATE_BINARY </span></p>
           <p><span class="code">-------------------------------------------------------------- </span></p>
           <p><span class="code">0x0700000000008B210B</span></p>

          <a id="_float"></a>
          <h3 class="subHeading">Converting float and real Data</h3>
          <div class="subsection">
           <p>Values of <span><span class="input">float</span></span> are truncated when they are converted to any integer type.</p>
           <p>When you want to convert from <span><span class="input">float</span></span> or <span><span class="input">real</span></span> to character data, using the STR string function is usually more useful than CAST( ). This is because STR enables more control over formatting. For more information, see <span><a href="https://technet.microsoft.com/en-us/library/ms189527(v=sql.105).aspx">STR (Transact-SQL)</a></span> and <span><a href="https://technet.microsoft.com/en-us/library/ms174318(v=sql.105).aspx">Built-in Functions (Transact-SQL)</a></span>. </p>
          </div>
          <a id="_money"></a>
          <h3 class="subHeading">Converting money Data</h3>
          <div class="subsection">
           <p>When you convert to <span><span class="input">money</span></span> from integer data types, units are assumed to be in monetary units. For example, the integer value of 4 is converted to the <span><span class="input">money</span></span> equivalent of 4 monetary units.</p>
           <p>The following example converts <span><span class="input">smallmoney</span></span> and <span><span class="input">money</span></span> values to <span><span class="input">varchar</span></span> and <span><span class="input">decimal</span></span> data types, respectively.</p> 
           <div id="code-snippet-9" class="codeSnippetContainer" xmlns=""> 
            <div class="codeSnippetContainerTabs"> 
            </div> 
            <div class="codeSnippetContainerCodeContainer"> 
             <div class="codeSnippetToolBar"> 
              <div class="codeSnippetToolBarText"> 
               <a name="CodeSnippetCopyLink" style="display: none;" title="Copy to clipboard." href="javascript:if (window.epx.codeSnippet)window.epx.codeSnippet.copyCode('CodeSnippetContainerCode_4c2e1aa0-c777-4fcf-8c1f-bfda65009735');" ms.cmptyp="CodeSnippet">Copy</a> 
              </div> 
             </div> 
             <div id="CodeSnippetContainerCode_4c2e1aa0-c777-4fcf-8c1f-bfda65009735" class="codeSnippetContainerCode" dir="ltr"> 
              <div style="color:Black;">
               <pre>
USE AdventureWorks2008R2;
GO
DECLARE @mymoney_sm smallmoney;
SET  @mymoney_sm = 3148.29;
SELECT  CAST(@mymoney_sm AS varchar) AS 'SM_MONEY varchar';
GO
DECLARE @mymoney    money;
SET  @mymoney    = 3148.29;
SELECT  CAST(@mymoney AS decimal)    AS 'MONEY DECIMAL';

GO
</pre>
              </div> 
             </div>

           <p><span class="code">SM_MONEY VARCHAR </span></p>

           <p><span class="code">(1 row(s) affected)</span></p>
           <p></p>
           <p><span class="code">MONEY DECIMAL </span></p>
           <p><span class="code">---------------------- </span></p>
           <p><span class="code">3148 </span></p>
           <p></p>
           <p><span class="code">(1 row(s) affected)</span></p>

          <a id="_decimal"></a>
          <h3 class="subHeading">Converting decimal and numeric Data</h3>
          <div class="subsection">
           <p>For the <span><span class="input">decimal</span></span> and <span><span class="input">numeric</span></span> data types, SQL Server considers each specific combination of precision and scale as a different data type. For example, <span><span class="input">decimal(5,5)</span></span> and <span><span class="input">decimal(5,0)</span></span> are considered different data types.</p>
           <p>In Transact-SQL statements, a constant with a decimal point is automatically converted into a <span><span class="input">numeric</span></span> data value, using the minimum precision and scale necessary. For example, the constant 12.345 is converted into a <span><span class="input">numeric</span></span> value with a precision of 5 and a scale of 3.</p>
           <p>Converting from <span><span class="input">decimal</span></span> or <span><span class="input">numeric</span></span> to <span><span class="input">float</span></span> or <span><span class="input">real</span></span> can cause some loss of precision. Converting from <span><span class="input">int</span></span>, <span><span class="input">smallint</span></span>, <span><span class="input">tinyint</span></span>, <span><span class="input">float</span></span>, <span><span class="input">real</span></span>, <span><span class="input">money</span></span>, or <span><span class="input">smallmoney</span></span> to either <span><span class="input">decimal</span></span> or <span><span class="input">numeric</span></span> can cause overflow.</p>
           <p>By default, SQL Server uses rounding when converting a number to a <span><span class="input">decimal</span></span> or <span><span class="input">numeric</span></span> value with a lower precision and scale. However, if the SET ARITHABORT option is ON, SQL Server raises an error when overflow occurs. Loss of only precision and scale is not sufficient to raise an error.</p>
          </div>
          <a id="_integer"></a>
          <h3 class="subHeading">Converting integer Data</h3>
          <div class="subsection">
           <p>When integers are implicitly converted to a character data type, if the integer is too large to fit into the character field, SQL Server enters ASCII character 42, the asterisk (*).</p>
           <p>Integer constants greater than 2,147,483,647 are converted to the <span><span class="input">decimal</span></span> data type, not the <span><span class="input">bigint</span></span> data type. The following example shows that when the threshold value is exceeded, the data type of the result changes from an <span><span class="input">int</span></span> to a <span><span class="input">decimal</span></span>.</p> 
           <div id="code-snippet-10" class="codeSnippetContainer" xmlns=""> 
            <div class="codeSnippetContainerTabs">
