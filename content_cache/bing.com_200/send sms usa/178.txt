          <section id="introduction" class="level-1"> 
           <header>
             Introduction 
           </header> 
           <section id="introduction-overview" class="level-2"> 
            <p>The SMS API sends SMS messages to one or more AT&amp;T Wireless mobile phones in a single request.</p> 
            <p>This API contains methods for sending and receiving messages and querying for the status of previously submitted SMS messages. For Mobile Originating messages, the SMS API allows you to poll the delivery status of the message or request that delivery status notifications be sent to a registered callback listener URI as soon as the message arrives.</p> 
           </section> 
          </section>

            <header>
             Get App Key &amp; App Secret
            </header> 
            <p class="qs-content">To start working with AT&amp;T APIs, create a developer account or sign in to your existing account on the <a href="https://developer.att.com/developer/mvc/auth/login?destPage=%2F">AT&amp;T Developer Program website</a>.</p> 
            <p class="qs-content">Click on the My Apps selection to set up a new app account by providing an Application Name, choosing SMS from the list of available APIs and, clicking submit to get an App Key and App Secret.</p> 
            <p class="qs-content"> <strong> Applicable Scopes : </strong> SMS </p> 
           </section>

          <section id="introduction-considerations" class="level-1"> 
           <header>
            Considerations
           </header> 
           <p>The RESTful SMS API provided by AT&amp;T has the following considerations.</p> 
           <ul> 
            <li>You can send an SMS message to a maximum of 10 recipients at a time.</li> 
            <li>All the messages sent are retried for up to 72 hours and any undelivered messages get deleted after that.</li> 
            <li>AT&amp;T supports two signatures of SMS API GSMA One and AT&amp;T. If the application wants to use multiple short codes to send SMS messages, then consider using GSMA One API. Using the GSMA One API enables you to specify a short code as a parameter in the URI to indicate the originating sender address for the SMS message.</li> 
            <li>Any SMS message that uses the Unicode encoding must be handled by the app by converting the hexadecimal value of the body using the first byte as the key for the encoding.</li> 
           </ul> 
          </section> 
          <section id="introduction-provisioning" class="level-2"> 
           <header>
            Provisioning
           </header> 
           <p>The RESTful SMS API provided by AT&amp;T requires that you provision your app as follows.</p> 
           <ul> 
            <li>By default, an online short code is assigned to your application for the reception of Mobile Originating SMS messages via a callback URI, so that the application gets a notification as soon as a message is received. You must register this callback URI during provisioning.</li> 
            <li>You can bring your own short code (custom short code) to use for your application. The process for on-boarding a custom short code typically takes 1-2 weeks. For more information, please see the <a href="/sms/support/sms-api-faqs" target="faqsWindow">FAQs</a>.</li> 
            <li>The GSMA One API can support multiple short codes for an application. When using the GMA One API to send an SMS message from the application, the sender address should always be set to the highest numbered short code available for the application.</li> 
            <li>For delivery status of an SMS message you can have a callback or you can poll for the message.</li> 
           </ul> 
          </section>

             <div id="get-sms-att" style="display: block;"> 
              <section id="resources-get-sms-att-overview" class="level-3"> 
               <header>
                Overview
               </header> 
               <p>The Get SMS method receives multiple SMS messages sent to the short code associated with the app account from the AT&amp;T Wireless network.</p> 
               <p>Each time this method is requested all SMS messages are received for the short code associated with the app account.</p> 
               <p>This method removes the messages from AT&amp;T and therefore it retrieves only those SMS messages received since the previous Get SMS method request.</p> 
               <p>If there are more messages than could fit into a single response, the app must invoke multiple method requests to retrieve the complete list of the messages.</p> 
               <p>The totalNumberOfPendingMessages parameter in the SMS API response indicates that there are any more messages left on the server that are not sent in the current response.</p> 
               <p>An app that supports receiving SMS messages must invoke this request at least once every 30 minutes; otherwise the messages are automatically removed from the API Gateway.</p> 
               <p>For apps that expect to receive a lot of SMS messages, it is recommended that the app invokes this request frequently, but with a maximum frequency of one every 5 minutes.</p> 
               <p><strong>Note:</strong> If the developer's app is expecting a huge number of Mobile Originated (MO) messages, then it is recommended to use the Receive SMS callback by registering a listener URI for the app where the API Gateway immediately delivers the incoming messages as soon as each message arrives in the AT&amp;T Wireless network.</p> 
               <p><strong>Note:</strong> This method is only allowed for a short code that is set up as an offline short code.</p> 
               <p>The process to obtain an offline short code for your app is described below.</p> 
               <p>Add one or more offline short codes as follows.</p> 
               <ol> 
                <li><span>Your first step depends upon your app account.</span> 
                 <ul> 
                  <li>If you have not created an SMS app account, then create an app account with the API set to SMS and click on the Manage Short Code link after submitting your app account.</li> 
                  <li>If you have already created an SMS app account, go to the My Apps page, click on the link to your SMS app account, and navigate to the Manage Short Code page.</li> 
                 </ul> </li>

              <section id="resources-get-sms-gsma-overview" class="level-3"> 
               <header>
                Overview
               </header> 
               <p>The Get SMS method receives multiple SMS messages sent to the short code associated with the app account from the AT&amp;T Wireless network.</p> 
               <p>Each time this method is requested all of the SMS messages are received for the short code associated with the app account.</p> 
               <p>This method removes the messages from AT&amp;T and therefore it retrieves only those SMS messages received since the previous Get SMS method request.</p> 
               <p>If there are more messages than could fit into a single response, the app must invoke multiple method requests to retrieve the complete list of the messages.</p> 
               <p>The totalNumberOfPendingMessages parameter in the SMS API response indicates that there are any more messages left on the server that are not sent in the current response.</p> 
               <p>An app that supports receiving SMS messages must invoke this request at least once every 30 minutes; otherwise the messages are automatically removed from the API Gateway.</p> 
               <p>For apps that expect to receive a lot of SMS messages, it is recommended that the app invokes this request frequently, but with a maximum frequency of one every 5 minutes.</p> 
               <p><strong>Note:</strong> If the developer's app is expecting a huge number of Mobile Originated (MO) messages, then it is recommended to use the Receive SMS callback by registering a listener URI for the app where the API Gateway immediately delivers the incoming messages as soon as each message arrives in the AT&amp;T Wireless network.</p> 
               <p><strong>Note:</strong> This method is only allowed for a short code that is set up as an offline short code.</p> 
               <p>The process to obtain an offline short code for your app is described below.</p> 
               <p>Add one or more offline short codes as follows.</p> 
               <ol> 
                <li>Your first step depends upon your app account. 
                 <ul> 
                  <li>If you have not created an SMS app account, then create an app account with the API set to SMS and click on the Manage Short Code link after submitting your app account.</li> 
                  <li>If you have already created an SMS app account, go to the My Apps page, click on the link to your SMS app account, and navigate to the Manage Short Code page.</li> 
                 </ul> </li> 
                <li>Click on the Add Short Code link and click the Save button without adding any URI information.</li> 
               </ol> 
               <p><strong>Note:</strong> If you want the SMS messages sent from the user's mobile device to be forwarded to your app immediately by the API Gateway, then you must register your SMS listener URI after creating the SMS app.</p> 
               <p>This enables you to utilize the Receive SMS method.</p> 
               <p><strong>Note:</strong> This method removes all of the messages for the registrationID parameter value that are stored on the AT&amp;T Wireless network at the time that this method is requested.</p>

               <header>
                Overview
               </header> 
               <p>The Get SMS Delivery Status method requests the status of a previously submitted SMS delivery request that was accepted by the API Gateway for delivery to the destination mobile device.</p> 
               <p>The Get SMS Delivery Status method is one of two ways in which the delivery status of a sent message may be obtained from the API Gateway.</p> 
               <p>The other way is to register an endpoint URI with the API Gateway which will be notified of the final status of the message. For a given message, only one of these mechanisms can be used.</p> 
               <p>The indicator of which mechanism will be used is the notifyDeliveryStatus parameter of the Send SMS method request.</p> 
               <p>To request the delivery status of a sent message using the Get SMS Delivery Status method, the notifyDeliveryStatus parameter of the Send SMS method must be set to false or should not be present in the request, this causes the messageId parameter value to be returned in the Send SMS method response.</p> 
               <p>The message identifier that was returned in the Send SMS method response must be provided in the Get SMS Delivery Status method request.</p> 
               <p><strong>Note:</strong> The delivery status of the message is removed from the API Gateway and is not available for query, when one of the following events occurs.</p> 
               <ul> 
                <li>Thirty minutes have elapsed and the message has reached a final state of DeliveredToTerminal or DeliveryImpossible.</li> 
                <li>The API Gateway responds to the first delivery status query after the message has reached the final state.</li> 
               </ul>

               <header>
                Overview
               </header> 
               <p>The Get SMS Delivery Status method requests the status of a previously submitted SMS delivery request that was accepted by the API Gateway for delivery to the destination mobile device.</p> 
               <p>The message identifier returned in the response of the corresponding Send SMS method request must be provided in this method request.</p>

               <ul> 
                <li>Thirty minutes have elapsed and the message has reached a final state of DeliveredToTerminal or DeliveryImpossible.</li> 
                <li>The API Gateway responds to the first delivery status query after the message has reached the final state.</li>

             <div id="receive-sms-att" style="display: block;"> 
              <section id="resources-receive-sms-att-overview" class="level-3"> 
               <header>
                Overview
               </header> 
               <p>The Receive SMS callback receives the Mobile Originated (MO) SMS messages in the app immediately for messages that are destined for the specified short codes of the app account.</p> 
               <p>In this case, the API Gateway acts as a client and the developer's app acts as a server.</p> 
               <p>The developer exposes the app functionality to the API Gateway in the form of a URI listener. This URI is used by the API Gateway to forward the received messages.</p> 
               <p><strong>Note:</strong> The developer must register the URI or the system will not be able push the notification to the app for the Mobile Originated (MO) Messages destined for the short code.</p> 
               <p><strong>Note:</strong> If the URI provided is HTTPS and port is not defined, then the default SSL port 443 will be used. The use of a transport mechanism other than HTTPS is not supported in the URI.</p> 
               <p>When a user sends a message to AT&amp;T short code, the system determines the app to which the short code is associated, and then forwards the message to the listener URI provisioned with the short code of the app account.</p> 
               <p>The messages are removed from the system after delivery is acknowledged by the app or until all the retries as per the retry configuration are exhausted.</p> 
               <p>The app must respond to the notification with an appropriate HTTP Status Code class of 2xx for positive acknowledgment.</p> 
               <p>The following explanation for the handling of the HTTP Status Codes sent by the user's app is sent in response to a notification from the API Gateway.</p> 
               <ul> 
                <li>In case of the HTTP Status Code class of 2xx, the server assumes positive acknowledgement that the notification has been sent successfully.</li> 
                <li>In case of no response, timeout condition or HTTP Status Code classes other than 2xx, the server assumes that either the API Gateway is temporarily unavailable or cannot receive and process the notification at that point in time.</li> 
               </ul> 
               <p>If the app SMS Mobile Originated (MO) URI listener is unavailable, busy, or otherwise unable to process and acknowledge the Mobile Originated (MO) message, then it will be retried as per the retry configuration set in the network.</p> 
               <p>Typically the retry period is 72 hours after which the message will be discarded and no further retries would be attempted.</p> 
              </section>

             <div id="receive-sms-delivery-status-att" style="display: block;"> 
              <section id="resources-receive-sms-delivery-status-att-overview" class="level-3"> 
               <header>
                Overview
               </header> 
               <p><span>The Receive SMS Delivery Status callback receives notifications of the final delivery status of the SMS messages sent by the app.</span> <br> <span>In this case, the API Gateway acts as a client and the developer's app acts as a server.</span> <br> <span>The developer provides the app listener endpoint URI to the API Gateway during the app account registration process or with an update to the app account against the appropriate short code associated with the app.</span> <br> <span>There exists a one-to-one association between the URI and a short code.</span> <br> <span>The URI is used by the API Gateway to notify the final delivery status of the SMS messages that were earlier accepted for delivery by the API Gateway against a specified short code.</span> <br> <span><strong>Note:</strong> The delivery notification is delivered to the app only if requested in the Send SMS method request by setting the notifyDeliveryStatus parameter value to true.</span> <br> <span><strong>Note:</strong> There is only one notification message per destination address when the SMS message is being delivered to multiple recipients in a single request.</span> <br> <span><strong>Note:</strong> All of the notifications in this case might not be necessarily at the same time as it depends on the delivery of the SMS message to each recipient.</span> <br> <span><strong>Note:</strong> If the URI provided is HTTPS and port is not defined, then the default SSL port 443 is used.</span> <br> <span>The use of a transport mechanism other than HTTPS is not supported in the URI.</span> <br> <span>When an SMS message that was earlier accepted for delivery reaches the final state and the app has requested for delivery notification in the Send SMS method request, a notification message is sent to the endpoint URI registered by the app for the specified short code.</span> <br> <span>The SMS message reaches the final status when it is delivered to the user's mobile device or when it expires in the network after multiple retries.</span> <br> <span>The number of retries is based on the Network retry configuration, which is currently set to 72 hours.</span> <br> <span>If the notifyDeliveryStatus parameter value is set to true in the Send SMS method request and if the notification endpoint URI is not provided or associated to the corresponding short code of the app in the app account, then the delivery notification status message might become lost.</span> <br> <span>The app listener is expected to respond to the notification with the appropriate HTTP Status Code class of 2xx for positive acknowledgement.</span> <br> <span>The following explanations describe the handling of HTTP Status Codes sent by the user's app, in response to a notification from the server.</span> </p> 
               <ul> 
                <li><span>In case of the HTTP Status Code class of 2xx, the server assumes positive acknowledgement that the notification has been sent successfully.</span></li> 
                <li><span>In case of no response, timeout condition or HTTP Status Code classes other than 2xx, the server assumes that either the API Gateway is temporarily unavailable or cannot receive and process the notification at that point in time.</span></li> 
               </ul> 
               <p><span>If the registered app delivery notification URI listener is unavailable, busy or otherwise unable to process and acknowledge the delivery status notification message then it will be retried as per the retry configuration set in the AT&amp;T Wireless network.</span> <br> <span>Typically the retry period is 72 hours after which the message will be discarded and no further retries would be attempted.</span> </p> 
              </section>

               <p>If you have more than one short code assigned to your app account and require the abilty to specify a specific short code as the originating sender address, then consider using the GSMA OneAPI provider of this method.</p>

              <section id="resources-send-sms-gsma-overview" class="level-3"> 
               <header>
                Overview
               </header> 
               <p>The Send SMS method sends an SMS message to one or more AT&amp;T Wireless mobile devices.</p> 
               <p>The SMS messages are processed synchronously and sent asynchronously to a destination on the AT&amp;T Network.</p> 
               <p>A unique identifier (ID) is returned in the response that may be used to query the status of the message that has been sent.</p> 
               <p><strong>Note:</strong> Using this provider of this method enables you to specify the sender originator address of the SMS message as a parameter in the URI, if you have one or more short codes assigned to your app account.</p> 
              </section>

                 <span class="label">notifyDeliveryStatus</span> 
                 <span class="data-type">Boolean</span> 
                 <span class="placement">Body</span> 
                </div> 
               </div> 
               <div class="grid-right"> 
                <p>Indicates that a delivery status notification is expected by the app.</p> 
                <p>If this parameter is set to “True”, then the AT&amp;T API Gateway sends a push notification to indicate the final delivery status of this message. The notification is sent to the endpoint URI registered against the given short code in your app’s short code management profile. If there is more than one recipient specified in the address parameter, one notification is sent per recipient.</p> 
                <p>If this parameter value is set to “False”, then your app does not receive any delivery status notifications from the AT&amp;T API Gateway.</p> 
                <p><strong>Note:</strong>If this flag is set to “True” but you have not registered any endpoint URI, a bad request error is thrown back in the message response.</p> 
                <p><strong>Note:</strong> This parameter is not part of the GSMA One API specification for this method. This is an additional parameter that is added to the AT&amp;T version of the GSMA One API signature.</p>
