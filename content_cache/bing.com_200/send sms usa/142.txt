      <p><b>Swing</b> is a <a href="/wiki/Graphical_user_interface" title="Graphical user interface">GUI</a> <a href="/wiki/Widget_toolkit" title="Widget toolkit">widget toolkit</a> for <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>. It is part of <a href="/wiki/Oracle_Corporation" title="Oracle Corporation">Oracle</a>'s <a href="/wiki/Java_Foundation_Classes" title="Java Foundation Classes">Java Foundation Classes</a> (JFC)&nbsp;– an <a href="/wiki/Application_programming_interface" title="Application programming interface">API</a> for providing a <a href="/wiki/Graphical_user_interface" title="Graphical user interface">graphical user interface</a> (GUI) for Java programs.</p> 
      <p><a rel="nofollow" class="external text" href="https://www.programmermart.com/">Swing</a> was developed to provide a more sophisticated set of GUI <a href="/wiki/Software_component" class="mw-redirect" title="Software component">components</a> than the earlier <a href="/wiki/Abstract_Window_Toolkit" title="Abstract Window Toolkit">Abstract Window Toolkit (AWT)</a>. Swing provides a native <a href="/wiki/Look_and_feel" title="Look and feel">look and feel</a> that emulates the look and feel of several platforms, and also supports a <a href="/wiki/Pluggable_look_and_feel" title="Pluggable look and feel">pluggable look and feel</a> that allows applications to have a look and feel unrelated to the underlying platform. It has more powerful and flexible components than AWT. In addition to familiar components such as buttons, check boxes and labels, Swing provides several advanced components such as tabbed panel, scroll panes, trees, tables, and lists.</p> 
      <p>Unlike AWT components, <a rel="nofollow" class="external text" href="https://www.programmermart.com/">Swing components</a> are not implemented by platform-specific code. Instead, they are written entirely in Java and therefore are platform-independent. The term "lightweight" is used to describe such an element.<sup id="cite_ref-1" class="reference"><a href="#cite_note-1">[1]</a></sup></p> 
      <p>Though Swing is intended to be replaced by <a href="/wiki/JavaFX" title="JavaFX">JavaFX</a>, it will remain part of the Java SE specification for the foreseeable future.<sup id="cite_ref-FAQs_2-0" class="reference"><a href="#cite_note-FAQs-2">[2]</a></sup></p>

      <h2><span class="mw-headline" id="History">History</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Swing_(Java)&amp;action=edit&amp;section=1" title="Edit section: History">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p>The <a href="/wiki/Internet_Foundation_Classes" title="Internet Foundation Classes">Internet Foundation Classes</a> (IFC) were a <a href="/wiki/Graphics_library" title="Graphics library">graphics library</a> for Java originally developed by <a href="/wiki/Netscape_Communications_Corporation" class="mw-redirect" title="Netscape Communications Corporation">Netscape Communications Corporation</a> and first released on December 16, 1996. On April 2, 1997, <a href="/wiki/Sun_Microsystems" title="Sun Microsystems">Sun Microsystems</a> and <a href="/wiki/Netscape_Communications_Corporation" class="mw-redirect" title="Netscape Communications Corporation">Netscape Communications Corporation</a> announced their intention to incorporate IFC with other technologies to form the <a href="/wiki/Java_Foundation_Classes" title="Java Foundation Classes">Java Foundation Classes</a>.<sup id="cite_ref-3" class="reference"><a href="#cite_note-3">[3]</a></sup> The "Java Foundation Classes" were later renamed "Swing."</p> 
      <p><a rel="nofollow" class="external text" href="https://www.programmermart.com/">Swing</a> introduced a mechanism that allowed the <a href="/wiki/Look_and_feel" title="Look and feel">look and feel</a> of every component in an application to be altered without making substantial changes to the application code. The introduction of support for a <a href="/wiki/Pluggable_look_and_feel" title="Pluggable look and feel">pluggable look and feel</a> allows Swing components to emulate the appearance of native components while still retaining the benefits of platform independence. Originally distributed as a separately downloadable library, Swing has been included as part of the <a href="/wiki/Java_Platform,_Standard_Edition" title="Java Platform, Standard Edition">Java Standard Edition</a> since release 1.2.<sup id="cite_ref-4" class="reference"><a href="#cite_note-4">[4]</a></sup> The Swing classes and components are contained in the <code><a rel="nofollow" class="external text" href="//docs.oracle.com/javase/9/docs/api/javax/swing/package-summary.html">javax.swing</a></code> <a href="/wiki/Java_package" title="Java package">package</a> hierarchy.</p> 
      <h2><span class="mw-headline" id="Architecture">Architecture</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Swing_(Java)&amp;action=edit&amp;section=2" title="Edit section: Architecture">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <p><a rel="nofollow" class="external text" href="https://www.programmermart.com/">Swing</a> is a platform-independent, "<a href="/wiki/Model-view-controller" class="mw-redirect" title="Model-view-controller">model-view-controller</a>" <a href="/wiki/GUI" class="mw-redirect" title="GUI">GUI</a> framework for Java, which follows a single-<a href="/wiki/Thread_(computing)" title="Thread (computing)">threaded</a> programming model.<sup id="cite_ref-5" class="reference"><a href="#cite_note-5">[5]</a></sup> Additionally, this framework provides a layer of abstraction between the code structure and graphic presentation of a Swing-based GUI.</p> 
      <h3><span class="mw-headline" id="Foundations">Foundations</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Swing_(Java)&amp;action=edit&amp;section=3" title="Edit section: Foundations">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p>Swing is platform-independent because it is completely written in Java. Complete documentation for all Swing classes can be found in the <a rel="nofollow" class="external text" href="http://docs.oracle.com/javase/6/docs/api/">Java API Guide</a> for Version 6 or the <a rel="nofollow" class="external text" href="http://docs.oracle.com/javase/8/docs/api/">Java Platform Standard Edition 8 API Specification</a> for Version 8.</p> 
      <h4><span class="mw-headline" id="Extensible">Extensible</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Swing_(Java)&amp;action=edit&amp;section=4" title="Edit section: Extensible">edit</a><span class="mw-editsection-bracket">]</span></span></h4> 
      <p>Swing is a highly modular-based architecture, which allows for the "plugging" of various custom implementations of specified framework interfaces: Users can provide their own custom implementation(s) of these components to override the default implementations using Java's inheritance mechanism.<sup id="cite_ref-6" class="reference"><a href="#cite_note-6">[6]</a></sup></p> 
      <p><a rel="nofollow" class="external text" href="https://www.programmermart.com/">Swing</a> is a <b>component-based framework</b>, whose components are all ultimately derived from the <tt>javax.swing.JComponent</tt> class. Swing objects asynchronously fire events, have bound properties, and respond to a documented set of methods specific to the component. Swing components are <a href="/wiki/Java_Beans" class="mw-redirect" title="Java Beans">Java Beans</a> components, compliant with the Java Beans Component Architecture specifications.</p> 
      <h4><span class="mw-headline" id="Configurable">Configurable</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Swing_(Java)&amp;action=edit&amp;section=5" title="Edit section: Configurable">edit</a><span class="mw-editsection-bracket">]</span></span></h4> 
      <p>Swing's heavy reliance on runtime mechanisms and indirect composition patterns allows it to respond at run time to fundamental changes in its settings. For example, a Swing-based application is capable of <a href="/wiki/Hot_swapping" title="Hot swapping">hot swapping</a> its user-interface during runtime. Furthermore, users can provide their own look and feel implementation, which allows for uniform changes in the look and feel of existing Swing applications without any programmatic change to the application code.</p> 
      <dl> 
       <dt>
        Lightweight UI
       </dt> 
      </dl> 
      <p>Swing's high level of flexibility is reflected in its inherent ability to override the native host <a href="/wiki/Operating_system" title="Operating system">operating system</a> (OS)'s GUI controls for displaying itself. Swing "paints" its controls using the Java 2D APIs, rather than calling a native user interface toolkit. Thus, a Swing component does not have a corresponding native OS GUI component, and is free to render itself in any way that is possible with the underlying graphics GUIs.</p> 
      <p>However, at its core, every Swing component relies on an <a href="/wiki/Abstract_Window_Toolkit" title="Abstract Window Toolkit">AWT</a> container, since (Swing's) <code><a rel="nofollow" class="external text" href="//docs.oracle.com/javase/9/docs/api/javax/swing/JComponent.html">JComponent</a></code> extends (AWT's) Container. This allows Swing to plug into the host OS's GUI management framework, including the crucial device/screen mappings and user interactions, such as key presses or mouse movements. Swing simply "transposes" its own (OS-agnostic) semantics over the underlying (OS-specific) components. So, for example, every Swing component paints its rendition on the graphic device in response to a call to component.paint(), which is defined in (AWT) Container. But unlike AWT components, which delegated the painting to their OS-native "heavyweight" widget, Swing components are responsible for their own rendering.</p> 
      <p>This transposition and decoupling is not merely visual, and extends to Swing's management and application of its own OS-independent semantics for events fired within its component containment hierarchies. Generally speaking, the Swing architecture delegates the task of mapping the various flavors of OS GUI semantics onto a simple, but generalized, pattern to the AWT container. Building on that generalized platform, it establishes its own rich and complex GUI semantics in the form of the <code><a rel="nofollow" class="external text" href="//docs.oracle.com/javase/9/docs/api/javax/swing/JComponent.html">JComponent</a></code> model.</p> 
      <h4><span class="mw-headline" id="Loosely_coupled_and_MVC">Loosely coupled and MVC</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Swing_(Java)&amp;action=edit&amp;section=6" title="Edit section: Loosely coupled and MVC">edit</a><span class="mw-editsection-bracket">]</span></span></h4> 
      <p>The Swing library makes heavy use of the <a href="/wiki/Model-view-controller" class="mw-redirect" title="Model-view-controller">Model/View/Controller</a> software <a href="/wiki/Design_pattern_(computer_science)" class="mw-redirect" title="Design pattern (computer science)">design pattern</a>,<sup id="cite_ref-7" class="reference"><a href="#cite_note-7">[7]</a></sup> which conceptually decouples the data being viewed from the user interface controls through which it is viewed. Because of this, most Swing components have associated <i>models</i> (which are specified in terms of Java <a href="/wiki/Interface_(computer_science)" class="mw-redirect" title="Interface (computer science)">interfaces</a>), and the programmers can use various default implementations or provide their own. The framework provides default implementations of model interfaces for all of its concrete components. The typical use of the Swing framework does not require the creation of custom models, as the framework provides a set of default implementations that are transparently, by default, associated with the corresponding <code><a rel="nofollow" class="external text" href="//docs.oracle.com/javase/9/docs/api/javax/swing/JComponent.html">JComponent</a></code> child class in the Swing library. In general, only complex components, such as tables, trees and sometimes lists, may require the custom model implementations around the application-specific data structures. To get a good sense of the potential that the Swing architecture makes possible, consider the hypothetical situation where custom models for tables and lists are wrappers over <a href="/wiki/Data_Access_Object" class="mw-redirect" title="Data Access Object">DAO</a> and/or <a href="/wiki/Ejb" class="mw-redirect" title="Ejb">EJB</a> services.</p> 
      <p>Typically, Swing component model objects are responsible for providing a concise interface defining events fired, and accessible properties for the (conceptual) data model for use by the associated JComponent. Given that the overall MVC pattern is a loosely coupled collaborative object relationship pattern, the model provides the programmatic means for attaching event listeners to the data model object. Typically, these events are model centric (ex: a "row inserted" event in a table model) and are mapped by the JComponent <a href="/wiki/Subclass_(computer_science)#Subclasses_and_superclasses" class="mw-redirect" title="Subclass (computer science)">specialization</a> into a meaningful event for the GUI component.</p> 
      <p>For example, the <code><a rel="nofollow" class="external text" href="//docs.oracle.com/javase/9/docs/api/javax/swing/JTable.html">JTable</a></code> has a model called <code><a rel="nofollow" class="external text" href="//docs.oracle.com/javase/9/docs/api/javax/swing/table/TableModel.html">TableModel</a></code> that describes an interface for how a table would access tabular data. A default implementation of this operates on a two-dimensional <a href="/wiki/Array_data_structure" title="Array data structure">array</a>.</p> 
      <p>The view component of a Swing JComponent is the object used to graphically represent the conceptual GUI control. A distinction of Swing, as a GUI framework, is in its reliance on programmatically rendered GUI controls (as opposed to the use of the native host OS's GUI controls). Prior to <a href="/wiki/Java_version_history#Java_SE_6_Update_10" title="Java version history">Java 6 Update 10</a>, this distinction was a source of complications when mixing AWT controls, which use native controls, with Swing controls in a GUI (see <a href="/wiki/Abstract_Window_Toolkit#Mixing_AWT_and_Swing_components" title="Abstract Window Toolkit">Mixing AWT and Swing components</a>).</p> 
      <p>Finally, in terms of visual composition and management, Swing favors <a href="/wiki/Layout_manager" title="Layout manager">relative layouts</a> (which specify the positional relationships between components) as opposed to absolute layouts (which specify the exact location and size of components). This bias towards "fluid"' visual ordering is due to its origins in the <a href="/wiki/Java_applet" title="Java applet">applet</a> operating environment that framed the design and development of the original Java GUI toolkit. (Conceptually, this view of the layout management is quite similar to that which informs the rendering of HTML content in browsers, and addresses the same set of concerns that motivated the former.)</p> 
      <h3><span class="mw-headline" id="Relationship_to_AWT">Relationship to AWT</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Swing_(Java)&amp;action=edit&amp;section=7" title="Edit section: Relationship to AWT">edit</a><span class="mw-editsection-bracket">]</span></span></h3>

      <p>Since early versions of Java, a portion of the <a href="/wiki/Abstract_Window_Toolkit" title="Abstract Window Toolkit">Abstract Window Toolkit</a> (AWT) has provided platform-independent APIs for user interface components. In AWT, each component is rendered and controlled by a native peer component specific to the underlying windowing system.</p> 
      <p>By contrast, Swing components are often described as <i>lightweight</i> because they do not require allocation of native resources in the operating system's windowing toolkit. The AWT components are referred to as <i>heavyweight components</i>.<sup id="cite_ref-8" class="reference"><a href="#cite_note-8">[8]</a></sup></p> 
      <p>Much of the Swing API is generally a complementary extension of the AWT rather than a direct replacement. In fact, every Swing lightweight interface ultimately exists within an AWT heavyweight component because all of the top-level components in Swing (<code><a rel="nofollow" class="external text" href="//docs.oracle.com/javase/9/docs/api/javax/swing/JApplet.html">JApplet</a></code>, <code><a rel="nofollow" class="external text" href="//docs.oracle.com/javase/9/docs/api/javax/swing/JDialog.html">JDialog</a></code>, <code><a rel="nofollow" class="external text" href="//docs.oracle.com/javase/9/docs/api/javax/swing/JFrame.html">JFrame</a></code>, and <code><a rel="nofollow" class="external text" href="//docs.oracle.com/javase/9/docs/api/javax/swing/JWindow.html">JWindow</a></code>) extend an AWT top-level container. Prior to <a href="/wiki/Java_version_history#Java_SE_6_Update_10" title="Java version history">Java 6 Update 10</a>, the use of both lightweight and heavyweight components within the same window was generally discouraged due to <a href="/wiki/Z-order" title="Z-order">Z-order</a> incompatibilities. However, later versions of Java have fixed these issues, and both Swing and AWT components can now be used in one GUI without Z-order issues.</p> 
      <p>The core rendering functionality used by Swing to draw its lightweight components is provided by <a href="/wiki/Java_2D" title="Java 2D">Java 2D</a>, another part of JFC.</p>

      <h3><span class="mw-headline" id="Relationship_to_SWT">Relationship to SWT</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Swing_(Java)&amp;action=edit&amp;section=8" title="Edit section: Relationship to SWT">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <p>The <a href="/wiki/Standard_Widget_Toolkit" title="Standard Widget Toolkit">Standard Widget Toolkit</a> (SWT) is a competing toolkit originally developed by <a href="/wiki/IBM" title="IBM">IBM</a> and now maintained by the <a href="/wiki/Eclipse_(software)" title="Eclipse (software)">Eclipse</a> <a href="/wiki/Free_software_community" class="mw-redirect" title="Free software community">community</a>. SWT's implementation has more in common with the heavyweight components of AWT. This confers benefits such as more accurate fidelity with the underlying native windowing toolkit, at the cost of an increased exposure to the native platform in the programming model.</p> 
      <p>There has been significant debate and speculation about the performance of SWT versus Swing; some hinted that SWT's heavy dependence on <a href="/wiki/Java_Native_Interface" title="Java Native Interface">JNI</a> would make it slower when the GUI component and Java need to communicate data, but faster at rendering when the data model has been loaded into the GUI, but this has not been confirmed either way.<sup id="cite_ref-9" class="reference"><a href="#cite_note-9">[9]</a></sup> A fairly thorough set of benchmarks in 2005 concluded that neither Swing nor SWT clearly outperformed the other in the general case.<sup id="cite_ref-10" class="reference"><a href="#cite_note-10">[10]</a></sup></p> 
      <h2><span class="mw-headline" id="Examples">Examples</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Swing_(Java)&amp;action=edit&amp;section=9" title="Edit section: Examples">edit</a><span class="mw-editsection-bracket">]</span></span></h2> 
      <h3><span class="mw-headline" id="A_basic_example">A basic example</span><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Swing_(Java)&amp;action=edit&amp;section=10" title="Edit section: A basic example">edit</a><span class="mw-editsection-bracket">]</span></span></h3> 
      <div class="thumb tright"> 
       <div class="thumbinner" style="width:194px;">
        <a href="/wiki/File:Swing_example_on_Windows_7.png" class="image"><img alt="" src="//upload.wikimedia.org/wikipedia/en/d/d7/Swing_example_on_Windows_7.png" width="192" height="74" class="thumbimage" data-file-width="192" data-file-height="74"></a>

      <p>The following is a rather simple Swing-based program. It displays a window (a <code><a rel="nofollow" class="external text" href="//docs.oracle.com/javase/9/docs/api/javax/swing/JFrame.html">JFrame</a></code>) containing a label and a button.</p> 
      <div class="mw-highlight mw-content-ltr" dir="ltr"> 
       <pre>
<span class="kn">import</span> <span class="nn">java.awt.FlowLayout</span><span class="o">;</span>
<span class="kn">import</span> <span class="nn">javax.swing.JButton</span><span class="o">;</span>
<span class="kn">import</span> <span class="nn">javax.swing.JFrame</span><span class="o">;</span>
<span class="kn">import</span> <span class="nn">javax.swing.JLabel</span><span class="o">;</span>
<span class="kn">import</span> <span class="nn">javax.swing.WindowConstants</span><span class="o">;</span>
<span class="kn">import</span> <span class="nn">javax.swing.SwingUtilities</span><span class="o">;</span>
 
<span class="kd">public</span> <span class="kd">class</span> <span class="nc">SwingExample</span> <span class="kd">implements</span> <span class="n">Runnable</span> <span class="o">{</span>

    <span class="nd">@Override</span>
    <span class="kd">public</span> <span class="kt">void</span> <span class="nf">run</span><span class="o">()</span> <span class="o">{</span>
        <span class="c1">// Create the window</span>
        <span class="n">JFrame</span> <span class="n">f</span> <span class="o">=</span> <span class="k">new</span> <span class="n">JFrame</span><span class="o">(</span><span class="s">"Hello, !"</span><span class="o">);</span>
        <span class="c1">// Sets the behavior for when the window is closed</span>
        <span class="n">f</span><span class="o">.</span><span class="na">setDefaultCloseOperation</span><span class="o">(</span><span class="n">WindowConstants</span><span class="o">.</span><span class="na">EXIT_ON_CLOSE</span><span class="o">);</span>
        <span class="c1">// Add a layout manager so that the button is not placed on top of the label</span>
        <span class="n">f</span><span class="o">.</span><span class="na">setLayout</span><span class="o">(</span><span class="k">new</span> <span class="n">FlowLayout</span><span class="o">());</span>
        <span class="c1">// Add a label and a button</span>
        <span class="n">f</span><span class="o">.</span><span class="na">add</span><span class="o">(</span><span class="k">new</span> <span class="n">JLabel</span><span class="o">(</span><span class="s">"Hello, world!"</span><span class="o">));</span>

        <span class="c1">// Arrange the components inside the window</span>
        <span class="n">f</span><span class="o">.</span><span class="na">pack</span><span class="o">();</span>
        <span class="c1">// By default, the window is not visible. Make it visible.</span>
        <span class="n">f</span><span class="o">.</span><span class="na">setVisible</span><span class="o">(</span><span class="kc">true</span><span class="o">);</span>
    <span class="o">}</span>
 
    <span class="kd">public</span> <span class="kd">static</span> <span class="kt">void</span> <span class="nf">main</span><span class="o">(</span><span class="n">String</span><span class="o">[]</span> <span class="n">args</span><span class="o">)</span> <span class="o">{</span>
        <span class="n">SwingExample</span> <span class="n">se</span> <span class="o">=</span> <span class="k">new</span> <span class="n">SwingExample</span><span class="o">();</span>
        <span class="c1">// Schedules the application to be run at the correct time in the event queue.</span>
        <span class="n">SwingUtilities</span><span class="o">.</span><span class="na">invokeLater</span><span class="o">(</span><span class="n">se</span><span class="o">);</span>
    <span class="o">}</span>

<span class="o">}</span>
</pre>
      </div> 
      <p>Notice how all instantiation and handling of Swing components are done by creating an instance of the class, which implements the Runnable interface. This is then run on the <a href="/wiki/Event_Dispatch_Thread" class="mw-redirect" title="Event Dispatch Thread">Event Dispatch Thread</a> by use of the method <code><a rel="nofollow" class="external text" href="//docs.oracle.com/javase/9/docs/api/javax/swing/SwingUtilities.html#invokeLater(java.lang.Runnable)">SwingUtilities.invokeLater(Runnable)</a></code>), created in the main method (see <a href="/wiki/Event_dispatching_thread#Swing_and_thread_safety" title="Event dispatching thread">Swing and thread safety</a>). Although Swing code can be run without using this technique (for instance, by not implementing Runnable and moving all commands from the run method to the main method), it is considered to be good form, as Swing is not <a href="/wiki/Thread_safety" title="Thread safety">thread-safe</a>, meaning that invoking resources from multiple threads can result in thread interference and memory consistency errors.<sup id="cite_ref-11" class="reference"><a href="#cite_note-11">[11]</a></sup></p>

      <p>In this example let javax.swing.JFrame be super class and add our own widget(s) to it (in this case, a JButton).</p>

      <p>The layout is set to null using the <code><a rel="nofollow" class="external text" href="//docs.oracle.com/javase/9/docs/api/java/awt/Container.html#setLayout-java.awt.LayoutManager-">Container.setLayout(LayoutManager)</a></code> method since JFrame uses java.awt.BorderLayout as its default layout-manager.
