     <h2> <span class="mw-headline" id="Overview"> Overview </span></h2> 
     <p>JavaScript code used for the web needs special consideration to ensure that it meets the high standards expected of all code shipped by Fedora, while still being useful and complying with conventions already used on millions of websites. Additionally, certain libraries typically used on the web can also be useful in a server-side context (by nodejs or rubygem-execjs), so it's important to package JavaScript so it meets the standards required of locally executed code as well. </p>
     <p>Please note that this section really only applies to JavaScript libraries intended for use on the web. Server-side JavaScript runtimes like Node.js <a href="/wiki/Packaging:Node.js" title="Packaging:Node.js">have their own guidelines</a>, and software like GNOME which embeds JavaScript for extensions have their own directories and policies as well. </p> 
     <h2> <span class="mw-headline" id="Rationale"> Rationale </span></h2> 
     <h3> <span class="mw-headline" id="Duplication"> Duplication </span></h3> 
     <p>We have a standard to avoid <a href="/wiki/Packaging:Guidelines#Duplication_of_system_libraries" title="Packaging:Guidelines">duplication of system libraries</a>. This applies as much to JavaScript libraries as it does to C, python, Java, and any other programming language. The main concern is security. JavaScript code is not vulnerable to buffer overflows like C but it can be subject to attacks from malicious data that attempts to send the user to alternate web sites, steal their information, or execute code via jsonp requests to an alternate server. Duplicating third party libraries in an application leads to these problems sticking with an application even after the upstream library has fixed the issues. It is therefore prohibited. </p>
     <p>There are also some JavaScript libraries which are intended to be used on the local system, not served via a web server to a browser. These libraries clearly have all the standard reasons to avoid duplication. </p> 
     <h3> <span class="mw-headline" id="Licensing"> Licensing </span></h3> 
     <p>It is common to see third-party applications consuming multiple JavaScript libraries. When all of these libraries are placed together in one directory, there is little information to tell what the relationship is between the files. This leads to problems when incompatible licenses are found. For instance, consider a situation where Foo.js is licensed with Apache Software License-2.0, Bar.js and Baz.js licensed with LGPLv2+, and Qux.js with MIT. We have to then audit the code in Foo.js to determine what symbols are potentially being used in other files, and then audit Bar.js and Baz.js to see if those symbols are being used there. </p>
     <p>Having separate packages for each library makes diagnosing these problems much easier. </p> 
     <h2> <span class="mw-headline" id="Naming_Guidelines"> Naming Guidelines </span></h2> 
     <p>The name of a JavaScript library package must start with <i>js-</i> then the upstream name. For example: <i>js-jquery</i>. </p> 
     <h2> <span class="mw-headline" id="BuildRequires"> BuildRequires </span></h2> 
     <p>To ensure the presence of the necessary RPM macros, all packages that provide JavaScript in <code>%{_jsdir}</code> must have: </p> 
     <pre>BuildRequires:  web-assets-devel</pre> 
     <h2> <span class="mw-headline" id="Requires"> Requires </span></h2> 
     <p>To ensure the availability of the necessary directories, all packages that provide JavaScript in <code>%{_jsdir}</code> must have: </p> 
     <pre>Requires:  web-assets-filesystem</pre> 
     <p>JavaScript packages <b>must not</b> have Requires on any HTTP daemon-specific configuration package, such as <code>web-assets-httpd</code>, since they could be used by any HTTP daemon. </p> 
     <h2> <span class="mw-headline" id="RPM_Macros"> RPM Macros </span></h2>

     <h2> <span class="mw-headline" id="Install_Location"> Install Location </span></h2> 
     <ul>
      <li> If a JavaScript library can be executed locally or consists purely of JavaScript code, it must be installed into a subdirectory of <code>%{_jsdir}</code>. </li>
     </ul> 
     <ul>
      <li> If a package contains JavaScript code, but is never useful outside the browser (e.g. if it is some sort of HTML user interface library) it may instead install to <code>%{_assetdir}</code>. For more information, see <a href="/wiki/Packaging:Web_Assets" title="Packaging:Web Assets">the Web Assets guidelines</a>. </li>
     </ul> 
     <ul>
      <li> If a package contains JavaScript code that is only used as part of a web application, and it is not useful to any other applications whatsoever, it may continue to ship that code along with the application. However, that does not absolve it from complying with the rest of these guidelines. </li>
     </ul> 
     <ul>
      <li> If a package contains JavaScript code that is not useful on the web, but only in locally run software (e.g. <a href="/wiki/Packaging:Node.js" title="Packaging:Node.js">Node.js</a> or GNOME shell extensions), it should use the appropriate directory for its runtime, not <code>%{_jsdir}</code>. </li>
     </ul> 
     <div class="messagebox" style="background-color: #def3fe; border: 1px solid #c5d7e0; color: black; padding: 5px; margin: 1ex 0; min-height: 35px; padding-left: 45px;"> 
      <div style="float: left; margin-left: -40px;">
       <a href="/wiki/File:Note.png" class="image"><img alt="Note.png" src="/w/uploads/c/cc/Note.png" width="35" height="35"></a>
      </div> 
      <div>
       <b>Huh?</b>
       <br>Stuff like jquery-ui will never be used server-side, so splitting the JS and non-JS portions and providing compat symlinks and such seems like an unnecessary amount of work. Hence this exception.
      </div> 
     </div> 
     <h2> <span class="mw-headline" id="Server_Location"> Server Location </span></h2> 
     <p>JavaScript code is included as part of the general <a href="/wiki/Packaging:Web_Assets" title="Packaging:Web Assets">Web Assets framework</a>. Therefore, <code>%{_jsdir}</code> is available on Fedora-provided web servers by default at <code>/_sysassets/javascript/</code>. </p>
     <p>For instance, jQuery may be installed in <code>%{_jsdir}/jquery/jquery-min.js</code>, so web applications that need to use it can simply include this HTML tag: </p> 
     <pre>
&lt;script type="text/javascript" src="/_sysassets/javascript/jquery/jquery-min.js"&gt;&lt;/script&gt;
</pre> 
     <p>Regardless, web applications may want to make subdirectories of <code>%{_jsdir}</code> available under their own directory via aliases or symlinks for compatibility purposes or to eliminate needless deviation from upstream. </p> 
     <h2> <span class="mw-headline" id="Compilation.2FMinification"> Compilation/Minification </span></h2> 
     <p>If a JavaScript library typically is shipped as minified or compiled code, it <b>must</b> be compiled or minified as part of the RPM build process. Shipping pre-minified or pre-compiled code is unacceptable in Fedora. </p>
     <p>The compiler or minifier used by upstream should be used to compile or minify the code. If the minifier used by upstream is unable to be included in Fedora, an alternative minifier may be used. <a href="/wiki/JavaScript/Minification_Issues" title="JavaScript/Minification Issues">See this page for a list of known problem areas and suggestions for workarounds.</a> </p> 
     <div class="messagebox" style="background-color: #def3fe; border: 1px solid #c5d7e0; color: black; padding: 5px; margin: 1ex 0; min-height: 35px; padding-left: 45px;"> 
      <div style="float: left; margin-left: -40px;">
       <a href="/wiki/File:Note.png" class="image"><img alt="Note.png" src="/w/uploads/c/cc/Note.png" width="35" height="35"></a>
      </div> 
      <div>
       <b>Huh?</b>
       <br>There are compiler/minifiers like the Google Closure compiler that will probably never be acceptable for Fedora due to bundled libraries, etc., so it's not a bad compromise to permit alternative minifiers to be used.
      </div> 
     </div> 
     <p>Additionally, the uncompiled/unminified version must be included alongside the compiled/minified version. </p>
     <p>Minified JavaScript is not useful for JavaScript run locally, as the entire point of minification is to reduce HTTP transfer times. Therefore, JavaScript not intended for the web (e.g. for Node.js or GNOME Shell extensions) <b>must not</b> be minified. </p> 
     <h2> <span class="mw-headline" id="Static_Inclusion_of_Libraries"> Static Inclusion of Libraries </span></h2> 
     <p>A single minified JavaScript file may contain bundled code from other JavaScript libraries. This may be to reduce HTTP transfer times for incredibly common libraries, like jQuery, which bundles sizzle.js, or when web applications only utilize parts of large JavaScript libraries like Dojo. </p>
     <p>JavaScript files intended for the web may include the source of other JavaScript libraries (in whole or in part) without an exception from the <a href="/wiki/Packaging_Committee" title="Packaging Committee">FPC</a>, provided the following conditions are met: </p> 
     <ul>
      <li> The bundled library must be shipped as a separate package in Fedora, which also meets these guidelines. </li>
      <li> The package that statically includes the library must use the code from that separate package. It <b>must not</b> bundle its own version of the library. </li>
      <li> The package that statically includes the library must include virtual provides in the form <code>Provides: js-includes(&lt;package name&gt;)</code> for all libraries it statically includes in this fashion. <code>web-assets-devel</code> provides a <code>%js_includes</code> macro that will automatically add the version present in the mock buildroot to the virtual provides. To use it, just add <code>%js_includes &lt;package name&gt;</code> anywhere before <code>%prep</code> in your spec file. </li>
     </ul> 
     <p>Again, this is only useful for the web. Locally used JavaScript runtimes (like Node.js or GNOME Shell extensions) <b>must</b> have a mechanism for utilizing libraries, and they are not permitted to utilize this exception to the <a href="/wiki/Packaging:No_Bundled_Libraries" title="Packaging:No Bundled Libraries" class="mw-redirect">standard bundled libraries policy</a>. </p> 
     <h2> <span class="mw-headline" id="Wrappers_for_Other_Languages_or_Environments"> Wrappers for Other Languages or Environments </span></h2> 
     <p>Sometimes there may exist a simple wrapper from a foreign language (like Ruby via rubygem-execjs or Java via rhino) or server-side JavaScript environment (like Node.js) to a pure JavaScript library. Such packages should delete the bundled library code and instead point to and Require the code provided by the primary Fedora package for that library. </p> 
     <h2> <span class="mw-headline" id="Node.js_Modules_that_contain_browser.2Fpure-JS_components"> Node.js Modules that contain browser/pure-JS components </span></h2> 
     <p>Some Node.js modules include parts that can be used in the browser or by other server-side JavaScript engines. Such packages should be shipped as one SRPM that contains two packages: </p> 
     <ul>
      <li> One <code>js-foo</code> package that contains the pure JavaScript portion, following these guidelines. </li>
     </ul> 
     <ul>
      <li> One <code>nodejs-foo</code> package that contains the Node.js module portion, following the <a href="/wiki/Packaging:Node.js" title="Packaging:Node.js">Node.js guidelines</a>. This may symlink to the necessary files or directories of the <code>js-foo</code> package. </li>
     </ul>
