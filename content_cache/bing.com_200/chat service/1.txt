            <h2>Creating a Chat Application using Java EE 7, Websockets and GlassFish 4</h2> 
            <small>August 13th, 2013 by <a href="/about/" rel="author">Micha Kops</a></small> 
            <div class="entry"> 
             <p>Java EE 7 is out now and so I was curious to play around with the new specifications and APIs from in this technology stack.</p> 
             <p>That’s why I didn’t hesitate to add yet another websocket-chat tutorial to the existing ones on the internet in favour of gathering some experience with this technology and a possible integration using a GlassFish 4 server, the new Java API for JSON Processing for data serialization combined with custom websocket encoders/decoders and finally adding some Bootstrap and jQuery on the client side.</p>

             <span id="The_Websocket_Endpoint"><h3>The Websocket Endpoint</h3></span> 
             <p>This is our endpoint implementation. Some short facts here about the implementation.</p> 
             <ul> 
              <li><em>@ServerEndpoint </em>defines a new endpoint – the value specifies the URL path and allows PathParams as we’re used know from JAX-RS.</li> 
              <li>So <em>value=”/chat/{room}”</em> allows the client to open a websocket connection to an URL like <em>ws://&lt;IP&gt;:&lt;PORT&gt;/&lt;CONTEXT_PATH&gt;/chat/room</em> .. e.g. ws://0.0.0.0:8080/hascode/chat/java</li> 
              <li>The value in curly braces may be injected as a path parameter in the lifecycle callback methods in the endpoint using <em>javax.websocket.server.PathParam</em></li> 
              <li>In addition to the URL schema, we’re specifying an encoder and a decoder class. We need this because we’re using a custom DTO object to pass our chat data between server and client</li> 
              <li>When a client opens the first connection to the server, he passes the chat-room he wants to enter as a path-parameter and we’re storing this value in the user properties map using <em>session.getUserProperties()</em>..</li> 
              <li>Now when a chat participant posts a new message over the tcp connection to the server, we’re iterating over all open session, and each session that is assigned to the room the session/message is bound to, receives the decoded and re-encoded message</li> 
              <li>If we wanted to send a simple text or binary message we could have used <em>session.getBasicRemote().sendBinary()</em> or <em>session.getBasicRemote().sendText()</em>…</li> 
             </ul>

             <span id="Bootstrap_Javascript_.._the_Client_Side_of_Life8230"><h3>Bootstrap, Javascript .. the Client Side of Life…</h3></span> 
             <p>Finally we’re adding the following <em>index.html</em> to the <em>src/main/webapp</em> directory (shortened, please feel free to have a look a the tutorial sources for the full code)</p> 
             <p>There are only three interesting API calls that we should remember here – the rest of the javascript adds some functions to the user interface etc …</p> 
             <ul> 
              <li>A websocket URL follows this schema: <em>ws://IP:PORT/CONTEXT_PATH/ENDPOINT_URL e.g ws://0.0.0.0:8080/hascode/chat/java<br> </em></li> 
              <li>A new websocket connection is created using the native <em>WebSocket</em> object e.g. <em>var wsocket = new WebSocket(‘ws://0.0.0.0:8080/hascode/chat/java’);</em></li> 
              <li>Registering a callback function to receive incoming messages from the server goes like this: <em>wsocket.onmessage = yourCallbackFunction;</em></li> 
              <li>Sending a message to the server is done by <em>wsocket.send()</em> .. pass a string, binary data .. whatever you like ..</li> 
              <li>Closing a connection is simply done by <em>wsocket.close()</em></li> 
              <li>There is a lot of useful information that I did not add to this tutorial from keepalive-pings to the handshake protocol and other features .. one good starting point for detailed information about websockets might be<em> </em><a onclick="javascript:pageTracker._trackPageview('/outgoing/tools.ietf.org/html/rfc6455');" href="http://tools.ietf.org/html/rfc6455" target="_blank">IETF RFC 6455</a><em> </em></li> 
             </ul>
