            <h1>Java EE Container Security</h1> 
            <p>There are two aspects covered by Java EE container security: authentication and authorization. <b>Authentication</b> is the process of verifying that users are who they claim to be. Typically this is performed by the user providing credentials such as a password. <b>Authorization</b>, or access control, is the process of restricting operations to specific users or categories of users. The EJB specification provides two kinds of authorization: declarative and programmatic, as we shall see later in the article.</p> 
            <p>The Java EE security model introduces a few concepts common to both authentication and authorization. A principal is an entity that we wish to authenticate. The format of a principal is application-specific but an example is a username. A role is a logical grouping of principals. For example, we can have administrator, manager, and employee roles. The scope over which a common security policy applies is known as a security domain, or realm.</p> 
            <h1>Authentication</h1> 
            <p>For authentication, every Java EE compliant application server provides the Java Authentication and Authorization Service (JAAS) API. JAAS supports any underlying security system. So we have a common API regardless of whether authentication is username/password verification against a database, iris or fingerprint recognition for example. The JAAS API is fairly low level and most application servers provide authentication mechanisms at a higher level of abstraction. These authentication mechanisms are application-server specific however. We will not cover JAAS any further here, but look at authentication as provided by the GlassFish application server.</p> 
            <h1>GlassFish Authentication</h1> 
            <p>There are three actors we need to define on the GlassFish application server for authentication purposes: <b>users</b>, <b>groups</b>, and <b>realms</b>. A user is an entity that we wish to authenticate. A user is synonymous with a principal. A group is a logical grouping of users and is not the same as a role. A group's scope is global to the application server. A role is a logical grouping of users whose scope is limited to a specific application. Of course for some applications we may decide that roles are identical to groups. For other applications we need some mechanism for mapping the roles onto groups. We shall see how this is done later. A realm, as we have seen, is the scope over which a common security policy applies.</p> 
            <p>GlassFish provides three kinds of realms: <b>file</b>, <b>certificate</b>, and <b>admin-realm</b>. The file realm stores user, group, and realm credentials in a file named keyfile. This file is stored within the application server file system. A file realm is used by web clients using http or EJB application clients. The certificate realm stores a digital certificate and is used for authenticating web clients using https. The admin-realm is similar to the file realm and is used for storing administrator credentials. GlassFish comes pre-configured with a default file realm named file.</p> 
            <p>We can add, edit, and delete users, groups, and realms using the GlassFish administrator console. We can also use the create-file-user option of the asadmin command line utility. To add a user named scott to a group named bankemployee, in the file realm, we would use the command:</p> 
            <pre>&lt;target name="create-file-user"&gt;<br>    &lt;exec executable="${glassfish.home}/bin/asadmin"<br>          failonerror="true"<br>          vmlauncher="false"&gt;<br><span style="font-weight: bold;">   &lt;arg line="create-file-user --user admin</span><br style="font-weight: bold;"><span style="font-weight: bold;">   --passwordfile userpassword --groups bankemployee scott"/&gt;</span><br>    &lt;/exec&gt;<br>&lt;/target&gt;</pre>
            <ul> 
             <li><i>--user</i> specifies the GlassFish administrator username, admin in our example.</li> 
             <li><i>--passwordfile</i> specifies the name of the file containing password entries. In our example this file is userpassword. Users, other than GlassFish administrators, are identified by AS_ADMIN_USERPASSWORD. In our example the content of the userpassword file is:<br> <pre>AS_ADMIN_USERPASSWORD=xyz</pre><p>&nbsp;This indicates that the user's password is xyz.</p></li> 
             <li><i>--groups</i> specifies the groups associated with this user (there may be more than one group). In our example there is just one group, named bankemployee. Multiple groups are colon delineated. For example if the user belongs to both the bankemployee and bankcustomer groups, we would specify:</li> 
             <pre>--groups bankemployee:bankcustomer</pre>
             <li>The final entry is the operand which specifies the name of the user to be created. In our example this is scott.</li> 
            </ul> 
            <p>There is a corresponding asadmin delete-file-user option to remove a user from the file realm.</p> 
            <h2>Mapping Roles to Groups</h2> 
            <p>The Java EE specification specifies that there must be a mechanism for mapping local application specific roles to global roles on the application server. Local roles are used by an EJB for authorization purposes. The actual mapping mechanism is application server specific. As we have seen in the case of GlassFish, the global application server roles are called groups. In GlassFish, local roles are referred to simply as roles. Suppose we want to map an employee role to the <span style="font-style: italic;">bankemployee</span> group. We would need to create a GlassFish specific deployment descriptor, sun-ejb-jar.xml, with the following 
             <security-role-mapping>
               element:
             </security-role-mapping></p> 
            <pre>&lt;security-role-mapping&gt;<br>    &lt;role-name&gt;employee&lt;/role-name&gt;<br>    &lt;group-name&gt;bankemployee&lt;/group-name&gt;<br>&lt;/security-role-mapping&gt;</pre>
            <p>We also need to access the configuration-security screen in the administrator console. We then <i>disable</i> the Default Principal To Role Mapping flag. If the flag is enabled then the default is to map a group onto a role with the same name. So the <span style="font-style: italic;">bankemployee</span> group will be mapped to the <span style="font-style: italic;">bankemployee</span> role.</p> 
            <p></p>
            <center>
             <img style="border: 0pt none;" src="/sites/default/files/Article-Images/ejb3-article-1.png" border="0" alt="EJB 3 Security" title="EJB 3 Security">
            </center>
            <p></p> 
            <p>We can leave the default values for the other properties on the configuration-security screen. Many of these features are for advanced use where third party security products can be plugged in or security properties customized. Consequently we will give only a brief description of these properties here.</p> 
            <ul> 
             <li><b>Security Manager:</b> This refers to the JVM security manager which performs code-based security checks. If the security manager is disabled GlassFish will have better performance. However, even if the security manager is disabled, GlassFish still enforces standard Java EE authentication/authorization.</li> 
             <li><b>Audit Logging:</b> If this is enabled, GlassFish will provide an audit trail of all authentication and authorization decisions through <b>audit modules</b>. Audit modules provide information on incoming requests, outgoing responses and whether authorization was granted or denied. Audit logging applies for web-tier and ejb-tier authentication and authorization. A default audit module is provided but custom audit modules can also be created.</li> 
             <li><b>Default Realm:</b> This is the default realm used for authentication. Applications use this realm unless they specify a different realm in their deployment descriptor. The default value is file. Other possible values are admin-realm and certificate. We discussed GlassFish realms in the previous section.</li> 
             <li><b>Default Principal:</b> This is the user name used by GlassFish at run time if no principal is provided. Normally this is not required so the property can be left blank.</li> 
             <li><b>Default Principal Password:</b> This is the password of the default principal.</li> 
             <li><b>JACC:</b> This is the class name of a JACC (Java Authorization Contract for Containers) provider. This enables the GlassFish administrator to set up third-party plug in modules conforming to the JACC standard to perform authorization.</li> 
             <li><b>Audit Modules:</b> If we have created custom modules to perform audit logging, we would select from this list.</li> 
             <li><b>Mapped Principal Class:</b> This is only applicable when Default Principal to Role Mapping is enabled. The mapped principal class is used to customize the java.security.Principal implementation class used in the default principal to role mapping. If no value is entered, the com.sun.enterprise.deployment.Group implementation of java.security.Principal is used.</li> 
            </ul> 
            <h2>Authenticating an EJB Application Client</h2> 
            <p>Suppose we want to invoke an EJB, BankServiceBean, from an application client. We also want the application client container to authenticate the client. There are a number of steps we first need to take which are application server specific. We will assume that all roles will have the same name as the corresponding application server groups. In the case of GlassFish we need to use the administrator console and enable Default Principal To Role Mapping.</p> 
            <p>Next we need to define a group named <span style="font-style: italic;">bankemployee</span> with one or more associated users.</p> 
            <p>An EJB application client needs to use <b>IOR (Interoperable Object Reference)</b> authentication. The IOR protocol was originally created for <b>CORBA (Common Object Request Broker Architecture)</b> but all Java EE compliant containers support IOR. An EJB deployed on one Java EE compliant vendor may be invoked by a client deployed on another Java EE compliant vendor. Security interoperability between these vendors is achieved using the IOR protocol. In our case the client and target EJB both happen to be deployed on the same vendor, but we still use IOR for propagating security details from the application client container to the EJB container.</p> 
            <p>IORs are configured in vendor specific XML files rather than the standard ejb-jar.xml file. In the case of GlassFish, this is done within the &lt;ior-security-config&gt; element within the <span style="font-style: italic;">sun-ejb-jar.xml</span> deployment descriptor file. We also need to specify the invoked EJB, BankServiceBean, in the deployment descriptor. An example of the <span style="font-style: italic;">sun-ejb-jar.xml</span> deployment descriptor is shown below:</p> 
            <pre>&lt;?xml version="1.0" encoding="UTF-8"?&gt;<br>&lt;!DOCTYPE sun-ejb-jar PUBLIC "-//Sun Microsystems, Inc.//DTD<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; Application Server 9.0 EJB 3.0//EN"<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; "http://www.sun.com/software/appserver/dtds/sun-ejb-jar_3_0-0.dtd"&gt;<br>&lt;sun-ejb-jar&gt;<br>&nbsp; &lt;enterprise-beans&gt;<br>&nbsp;&nbsp;&nbsp; &lt;ejb&gt;<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;ejb-name&gt;BankServiceBean&lt;/ejb-name&gt;<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;ior-security-config&gt;<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;as-context&gt;<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;auth-method&gt;USERNAME_PASSWORD&lt;/auth-method&gt;<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;realm&gt;default&lt;/realm&gt;<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;required&gt;true&lt;/required&gt;<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;/as-context&gt;<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;/ior-security-config&gt;<br>&nbsp;&nbsp;&nbsp; &lt;/ejb&gt;<br>&nbsp; &lt;/enterprise-beans&gt;<br>&lt;/sun-ejb-jar&gt;</pre>
            <ul> 
             <li>The as in &lt;as-context&gt; stands for the IOR <b>authentication service</b>. This specifies authentication mechanism details.</li> 
             <li>The &lt;auth-method&gt; element specifies the authentication method. This is set to <span style="font-style: italic;">USERNAME_PASSWORD</span> which is the only value for an application client.</li> 
             <li>The &lt;realm&gt; element specifies the realm in which the client is authenticated.</li> 
             <li>The &lt;required&gt; element specifies whether the above authentication method is required to be used for client authentication.</li> 
            </ul> 
            <p>When creating the corresponding EJB JAR file, the <span style="font-style: italic;">sun-ejb-jar.xml</span> file should be included in the META-INF directory, as follows:</p> 
            <pre>&lt;target name="package-ejb" depends="compile"&gt;<br>&nbsp;&nbsp;&nbsp; &lt;jar jarfile="${build.dir}/BankService.jar"&gt;<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;fileset dir="${build.dir}"&gt;<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;include name="ejb30/session/**" /&gt;<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;include name="ejb30/entity/**" /&gt;<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;/fileset&gt;<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;metainf dir="${config.dir}"&gt;<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;include name="persistence.xml" /&gt;<br><span style="font-weight: bold;">&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;include name="sun-ejb-jar.xml" /&gt;</span><br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; &lt;/metainf&gt;<br>&nbsp;&nbsp;&nbsp; &lt;/jar&gt;<br>&lt;/target&gt;</pre>
            <p>As soon as we run the application client, GlassFish will prompt with a username and password form, as follows:</p> 
            <p></p>
            <center>
             <img style="border: 0pt none;" src="/sites/default/files/Article-Images/ejb3-article-2.png" border="0" alt="EJB 3 Security" title="EJB 3 Security" width="335" height="161">
            </center>
            <p></p> 
            <p>If we reply with the username scott and password xyz the program will run. If we run the application with an invalid username or password we will get the following error message:</p> 
            <pre>javax.ejb.EJBException: nested exception is: java.rmi.AccessException: <br>                                        CORBA NO_PERMISSION 9998 .....</pre>
            <h1>EJB Authorization</h1> 
            <p>Authorization, or access control, is the process of restricting operations to specific roles. In contrast with authentication, EJB authorization is completely application server independent. The EJB specification provides two kinds of authorization: declarative and programmatic. With declarative authorization all security checks are performed by the container. An EJB's security requirements are declared using annotations or deployment descriptors. With programmatic authorization security checks are hard-coded in the EJBs code using API calls. However, even with programmatic authorization the container is still responsible for authentication and for assigning roles to principals.</p> 
            <h2>Declarative Authorization</h2> 
            <p>As an example, consider the <span style="font-style: italic;">BankServiceBean</span> stateless session bean with methods <span style="font-style: italic;">findCustomer()</span>, <span style="font-style: italic;">addCustomer()</span> and <span style="font-style: italic;">updateCustomer()</span>:</p> 
            <pre>package ejb30.session;<br>import javax.ejb.Stateless;<br>import javax.persistence.EntityManager;<br>import ejb30.entity.Customer;<br>import javax.persistence.PersistenceContext;<br>import javax.annotation.security.RolesAllowed;<br>import javax.annotation.security.PermitAll;<br>import java.util.*;<br>@Stateless<br>@RolesAllowed("bankemployee")<br>public class BankServiceBean implements BankService {<br>   @PersistenceContext(unitName="BankService")<br>   private EntityManager em;<br>   private Customer cust;<br>   @PermitAll<br>   public Customer findCustomer(int custId) {<br>        return ((Customer) em.find(Customer.class, custId));<br>   }<br>   public void addCustomer(int custId, String firstName,<br>                                        String lastName) {<br>          cust = new Customer();<br>          cust.setId(custId);<br>          cust.setFirstName(firstName);<br>          cust.setLastName(lastName);<br>          em.persist(cust);<br>   }<br>   public void updateCustomer(Customer cust) {<br>   Customer mergedCust = em.merge(cust);<br>   }<br>}</pre>
            <p>We have prefixed the bean class with the annotation:</p> 
            <pre>@RolesAllowed("bankemployee")</pre>
            <p>This specifies the roles allowed to access any of the bean's method. So only users belonging to the <span style="font-style: italic;">bankemployee</span> role may access the <span style="font-style: italic;">addCustomer()</span> and <span style="font-style: italic;">updateCustomer()</span> methods. More than one role can be specified by means of a brace delineated list, as follows:</p> 
            <pre>@RolesAllowed({"bankemployee", "bankcustomer"})</pre>
            <p>We can also prefix a method with <span style="font-style: italic;">@RolesAllowed</span>, in which case the method annotation will override the class annotation. The @PermitAll annotation allows unrestricted access to a method, overriding any class level <span style="font-style: italic;">@RolesAllowed</span> annotation.</p> 
            <p>As with EJB 3 in general, we can use deployment descriptors as alternatives to the <span style="font-style: italic;">@RolesAllowed</span> and <span style="font-style: italic;">@PermitAll</span> annotations.</p> 
            <h3>Denying Authorization</h3> 
            <p>Suppose we want to deny all users access to the <span style="font-style: italic;">BankServiceBean.updateCustomer()</span> method. We can do this using the <span style="font-style: italic;">@DenyAll</span> annotation:</p> 
            <pre>@DenyAll<br>    public void updateCustomer(Customer cust) {<br>       Customer mergedCust = em.merge(cust);<br>    }</pre>
            <p>Of course if you have access to source code you could simply delete the method in question rather than using <span style="font-style: italic;">@DenyAll</span>. However suppose you do not have access to the source code and have received the EJB from a third party. If you in turn do not want your clients accessing a given method then you would need to use the &lt;exclude-list&gt; element in the <span style="font-style: italic;">ejb-jar.xml</span> deployment descriptor:</p> 
            <pre>&lt;?xml version="1.0" encoding="UTF-8"?&gt;<br>&lt;ejb-jar version="3.0" xmlns="http://java.sun.com/xml/ns/javaee"<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; xsi:schemaLocation="http://java.sun.com/xml/ns/javaee<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; http://java.sun.com/xml/ns/javaee/ejb-jar_3_0.xsd"&gt;<br>&lt;enterprise-beans&gt;<br>&lt;session&gt;<br>&lt;ejb-name&gt;BankServiceBean&lt;/ejb-name&gt;<br>&lt;/session&gt;<br>&lt;/enterprise-beans&gt;<br>&lt;assembly-descriptor&gt;<br><span style="font-weight: bold;">&lt;exclude-list&gt;</span><br style="font-weight: bold;"><span style="font-weight: bold;">&lt;method&gt;</span><br style="font-weight: bold;"><span style="font-weight: bold;">   &lt;ejb-name&gt;BankServiceBean&lt;/ejb-name&gt;</span><br style="font-weight: bold;"><span style="font-weight: bold;">   &lt;method-name&gt;updateCustomer&lt;/method-name&gt;</span><br style="font-weight: bold;"><span style="font-weight: bold;">&lt;/method&gt;</span><br style="font-weight: bold;"><span style="font-weight: bold;">&lt;/exclude-list&gt;</span><br>&lt;/assembly-descriptor&gt;<br>&lt;/ejb-jar&gt;</pre>
            <h3>EJB Security Propagation</h3> 
            <p>Suppose a client with an associated role invokes, for example, EJB A. If EJB A then invokes, for example, EJB B then by default the client's role is propagated to EJB B. However, you can specify with the <span style="font-style: italic;">@RunAs</span> annotation that all methods of an EJB execute under a specific role.</p> 
            <p>For example, suppose the <span style="font-style: italic;">addCustomer()</span> method in the <span style="font-style: italic;">BankServiceBean</span> EJB invokes the <span style="font-style: italic;">addAuditMessage()</span> method of the <span style="font-style: italic;">AuditServiceBean</span> EJB:</p> 
            <pre>@Stateless<br>@RolesAllowed("bankemployee")<br>public class BankServiceBean implements BankService {<br><span style="font-weight: bold;">private @EJB AuditService audit;</span><br>....<br>&nbsp;&nbsp;&nbsp;&nbsp; public void addCustomer(int custId, String firstName,<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; String lastName) {<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; cust = new Customer();<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; cust.setId(custId);<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; cust.setFirstName(firstName);<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; cust.setLastName(lastName);<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; em.persist(cust);<br><span style="font-weight: bold;">&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; audit.addAuditMessage(1, "customer add attempt");</span><br>&nbsp;&nbsp;&nbsp;&nbsp; }<br>&nbsp;&nbsp;&nbsp;&nbsp; ...<br>}</pre>
            <p>Note that only a client with an associated role of <span style="font-style: italic;">bankemploye</span>e can invoke <span style="font-style: italic;">addCustomer()</span>. If we prefix the <span style="font-style: italic;">AuditServiceBean</span> class declaration with <span style="font-style: italic;">@RunAs("bankauditor")</span> then the container will run any method in <span style="font-style: italic;">AuditServiceBean</span> as the <span style="font-style: italic;">bankauditor</span> role, regardless of the role which invokes the method. Note that the <span style="font-style: italic;">@RunAs</span> annotation is applied only at the class level, <span style="font-style: italic;">@RunAs</span> cannot be applied at the method level.</p> 
            <pre>@Stateless<br>@RunAs("bankauditor")<br>public class AuditServiceBean implements AuditService {<br>          @PersistenceContext(unitName="BankService")<br>          private EntityManager em;<br>          @TransactionAttribute(<br>                   TransactionAttributeType.REQUIRES_NEW)<br>          public void addAuditMessage (int auditId,<br>                                   String message) {<br>             Audit audit = new Audit();<br>             audit.setId(auditId);<br>             audit.setMessage(message);<br>             em.persist(audit);<br>          }<br>}<br></pre>
            <h2>Programmatic Authorization</h2> 
            <p>With programmatic authorization the bean rather than the container controls authorization. The <span style="font-style: italic;">javax.ejb.SessionContext</span> object provides two methods which support programmatic authorization: <span style="font-style: italic;">getCallerPrincipal()</span> and <span style="font-style: italic;">isCallerInRole()</span>. The <span style="font-style: italic;">getCallerPrincipal()</span> method returns a <span style="font-style: italic;">java.security.Principal</span> object. This object represents the caller, or principal, invoking the EJB. We can then use the <span style="font-style: italic;">Principal.getName()</span> method to obtain the name of the principal. We have done this in the <span style="font-style: italic;">addAccount()</span> method of the <span style="font-style: italic;">BankServiceBean</span> as follows:</p> 
            <pre>Principal cp = ctx.getCallerPrincipal();<br>System.out.println("getname:" + cp.getName());</pre>
            <p>The <span style="font-style: italic;">isCallerInRole()</span> method checks whether the principal belongs to a given role. For example, the code fragment below checks if the principal belongs to the bankcustomer role. If the principal does not belong to the bankcustomer role, we only persist the account if the balance is less than 99.</p> 
            <pre>if (ctx.isCallerInRole("bankcustomer")) {<br>&nbsp;&nbsp;&nbsp; em.persist(ac);<br>} else if (balance &lt; 99) {<br>&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; em.persist(ac);<br>&nbsp; }</pre>
            <p>When using the <span style="font-style: italic;">isCallerInRole()</span> method, we need to declare all the security role names used in the EJB code using the class level <span style="font-style: italic;">@DeclareRoles</span> annotation:</p> 
            <pre>@DeclareRoles({"bankemployee", "bankcustomer"})</pre>
            <p>The code below shows the <span style="font-style: italic;">BankServiceBean</span> EJB with all the programmatic authorization code described in this section:</p> 
            <pre>package ejb30.session;<br>import javax.ejb.Stateless;<br>import javax.persistence.EntityManager;<br>import ejb30.entity.Account;<br>import javax.persistence.PersistenceContext;<br>import javax.annotation.security.RolesAllowed;<br>import java.security.Principal;<br>import javax.annotation.Resource;<br>import javax.ejb.SessionContext;<br>import javax.annotation.security.DeclareRoles;<br>import java.util.*;<br>@Stateless<br>@DeclareRoles({"bankemployee", "bankcustomer"})<br>public class BankServiceBean implements BankService {<br>       @PersistenceContext(unitName="BankService")<br>          private EntityManager em;<br>          private Account ac;<br>          @Resource SessionContext ctx;<br>          @RolesAllowed({"bankemployee", "bankcustomer"})<br>          public void addAccount(int accountId, double balance,<br>              String accountType) {<br>              ac = new Account();<br>              ac.setId(accountId);<br>              ac.setBalance(balance);<br>              ac.setAccountType(accountType);<br>              Principal cp = ctx.getCallerPrincipal();<br>              System.out.println("getname:" + cp.getName());<br>              if (ctx.isCallerInRole("bankcustomer")) {<br>                 em.persist(ac);<br>              } else if (balance &lt; 99) {<br>                   em.persist(ac);<br>                }<br>          }<br>      .....<br>}<br></pre>
            <p>Where we have a choice declarative authorization is preferable to programmatic authorization. Declarative authorization avoids having to mix business code with security management code. We can change a bean's security policy by simply changing an annotation or deployment descriptor instead of modifying the logic of a business method. However, some security rules, such as the example above of only persisting an account within a balance limit, can only be handled by programmatic authorization. Declarative security is based only on the principal and the method being invoked, whereas programmatic security can take state into consideration.</p> 
            <h1>Java EE Web Container Security</h1> 
            <p>Because an EJB is typically invoked from the web-tier by a servlet, JSP page or JSF component, we will briefly mention Java EE web container security. The web-tier and EJB tier share the same security model. So the web-tier security model is based on the same concepts of principals, roles and realms.</p> 
            <h2>Web-Tier Authorization</h2> 
            <p>Web-tier authorization is similar to EJB authorization but with a few differences. Both programmatic and declarative authorization is supported for the web-tier. For programmatic authorization, the EJB methods of <span style="font-style: italic;">getCallerPrincipal()</span> and <span style="font-style: italic;">isCallerInRole()</span> have their equivalents of <span style="font-style: italic;">getUserPrincipal()</span> and <span style="font-style: italic;">isUserInRole()</span> in the <span style="font-style: italic;">HttpServletRequest</span> interface. These methods would then be used in a servlet or JSP page. To use these methods we would need to add the <span style="font-style: italic;">@DeclareRoles</span> annotation or the &lt;security-role&gt; deployment descriptor element as with EJBs. The <span style="font-style: italic;">@RunAs</span> annotation can also be used at servlet class level to specify that a servlet runs as a specified role.</p> 
            <p>There is no annotation equivalent for web-tier declarative authorization; we have to use the web.xml deployment descriptor instead. In the code fragment below, the bankemployee and bankcustomer roles have access to all URLs in the application (specified by the URL pattern of /*). Of course we can extend this to allow given roles access to specific URLs.</p> 
            <pre>&lt;security-constraint&gt;<br>&lt;web-resource-collection&gt;<br>&lt;web-resource-name&gt;bank service example&lt;/web-resource-name&gt;<br>&lt;url-pattern&gt;/*&lt;url-pattern&gt;<br>&lt;/web-resource-collection&gt;<br>&lt;auth-constraint&gt;<br>&lt;role-name&gt;bankemployee&lt;/role-name&gt;<br>&lt;role-name&gt;bankcustomer&lt;/role-name&gt;<br>&lt;/auth-constraint&gt;<br>&lt;/security-constraint&gt;</pre> 
            <p>Recall that the mechanism for mapping local application specific roles to global roles (or roles to groups) on the application server is application server specific. In the case of GlassFish we would use the &lt;security-role-mapping&gt; element in the sun-web.xml deployment descriptor as we did in the <span style="font-style: italic;">sun-ejb.xml </span>deployment descriptor for an EJB application client.</p> 
            <h2>Transport Layer Security</h2> 
            <p>Before we move on to web-tier authentication, we will briefly describe transport layer security, as we refer to this when we configure web-tier authentication. The transport layer is characterized by the <b>Secure Sockets Layer (SSL)</b> protocol and its successor the <b>Transport Layer Security (TLS)</b> protocol. Messages are encrypted using public-key encryption, and users are authenticated using digital certificates. The application server presents a certificate to the client's web browser. The purpose of the certificate is to verify that the site is what it claims to be. This is known as <b>server-side authentication</b>. The server may request the client to present their digital certificate. This is known as <b>client-side authentication</b>. <b>Mutual authentication</b> occurs when both server-side and client-side authentication are enabled. The SSL or TLS layers are typically used underneath http. In this case the https URL scheme is used to indicate a secure http connection.</p> 
            <p>The SSL and TLS protocols correspond to the transport layer of the <b>Open System Interconnection (OSI)</b> communication model.</p> 
            <h2>Web-Tier Authentication</h2> 
            <p>We have seen that an EJB application client uses IOR authentication and only one authentication mechanism, <span style="font-style: italic;">USERNAME_PASSWORD</span>, is available. Web applications have a number of authentication mechanisms:<b> basic</b>,<b> form</b>, and<b> client</b> certificate. Before we can authenticate a web application user we must have set up valid usernames, passwords and roles on the web or application server. This process is vendor specific. We have seen how to do this for the GlassFish application server.</p> 
            <p><b>Basic authentication</b> requires a username and password to be entered on a login form supplied by the web container. The username and password are transmitted as part of an http request and authenticated by the web container. This mechanism is not secure. If the transmission is intercepted, the username and password can easily be decoded. Basic authentication is specified in the &lt;auth-method&gt; element within the &lt;login-config&gt; element in the web.xml deployment descriptor as follows:</p> 
            <pre>&lt;login-config&gt;<br>&lt;auth-method&gt;BASIC&lt;/auth-method&gt;<br>&lt;/login-config&gt;</pre> 
            <p><b>Form authentication</b> is similar to basic authentication except that the login and error screens can be customized. Form authentication takes places over http and like basic authentication, is not secure. Form authentication is specified in the web.xml deployment descriptor as follows:</p> 
            <pre>&lt;login-config&gt;<br>&lt;auth-method&gt;FORM&lt;/auth-method&gt;<br>&lt;/login-config&gt;</pre> 
            <p><b>Client certificate authentication</b> requires the web server to authenticate a client using the client's public key certificate. The authentication uses https (http over SSL) so is more secure than either basic or form authentication. Client certificate authentication is specified in the web.xml deployment descriptor as follows:</p> 
            <pre>&lt;login-config&gt;<br>&lt;auth-method&gt;CLIENT&lt;/auth-method&gt;<br>&lt;/login-config&gt;</pre> 
            <h2>Example of Web-Tier Authentication and Authorization</h2> 
            <p>We will take the example of <span style="font-style: italic;">AddCustomerServlet</span>, which adds a customer to the database. In this example we will see that only users belonging to the <span style="font-style: italic;">bankemployee</span> role can access the servlet using basic authentication. Basic authentication requires a username and password to be entered on a login form supplied by the web container before any web component such as a servlet can be accessed.</p> 
            <p>The only modification we need to make is in the web.xml file. We have listed the entire web.xml file below with the security modifications highlighted:</p> 
            <pre>&lt;?xml version="1.0" encoding="UTF-8"?&gt;<br>&lt;web-app version="2.5" xmlns="http://java.sun.com/xml/ns/javaee"<br>         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"<br>         xsi:schemaLocation="http://java.sun.com/xml/ns/javaee<br>         http://java.sun.com/xml/ns/javaee/web-app_2_5.xsd"&gt;<br>&lt;servlet&gt;<br>   &lt;servlet-name&gt;AddCustomerServlet&lt;/servlet-name&gt;<br>   &lt;servlet-class&gt;ejb30.AddCustomerServlet&lt;/servlet-class&gt;<br>&lt;/servlet&gt;<br>&lt;servlet-mapping&gt;<br>   &lt;servlet-name&gt;AddCustomerServlet&lt;/servlet-name&gt;<br>   &lt;url-pattern&gt;/addCustomer&lt;/url-pattern&gt;<br>&lt;/servlet-mapping&gt;<br><span style="font-weight: bold;">&lt;security-constraint&gt;</span><br style="font-weight: bold;"><span style="font-weight: bold;">   &lt;web-resource-collection&gt;</span><br style="font-weight: bold;"><span style="font-weight: bold;">      &lt;web-resource-name&gt;bank service&lt;/web-resource-name&gt;</span><br style="font-weight: bold;"><span style="font-weight: bold;">      &lt;url-pattern&gt;/*&lt;/url-pattern&gt;</span><br style="font-weight: bold;"><span style="font-weight: bold;">   &lt;/web-resource-collection&gt;</span><br style="font-weight: bold;"><span style="font-weight: bold;">   &lt;auth-constraint&gt;</span><br style="font-weight: bold;"><span style="font-weight: bold;">      &lt;role-name&gt;bankemployee&lt;/role-name&gt;</span><br style="font-weight: bold;"><span style="font-weight: bold;">   &lt;/auth-constraint&gt;</span><br style="font-weight: bold;"><span style="font-weight: bold;">&lt;/security-constraint&gt;</span><br style="font-weight: bold;"><span style="font-weight: bold;">&lt;login-config&gt;</span><br style="font-weight: bold;"><span style="font-weight: bold;">    &lt;auth-method&gt;BASIC&lt;/auth-method&gt;</span><br style="font-weight: bold;"><span style="font-weight: bold;">    &lt;realm-name&gt;file&lt;/realm-name&gt;</span><br style="font-weight: bold;"><span style="font-weight: bold;">&lt;/login-config&gt;</span><br>&lt;/web-app&gt;<br></pre> 
            <p>To run the servlet enter <span style="font-style: italic;">http://localhost:8080/BankService/index.html</span> in a browser. The address <span style="font-style: italic;">http://localhost:8080</span> is the URL for web applications as shown by GlassFish on startup. If GlassFish provides a port other than 8080 then modify the URL accordingly. As soon as the URL has been entered, the web container will present a login form requesting the username and password. If a user belonging to the <span style="font-style: italic;">bankemployee</span> role has been correctly entered we proceed to the servlet, otherwise the web container will generate an error page.{/literal}</p> 
            <h1>Summary</h1> 
            <p>We have seen that there are two aspects to Java EE container security: authentication and authorization. Authentication is the process of verifying that users are who they claim to be. Authorization is the process of restricting operations to specific users or categories of users. The Java EE security model deals with principals, roles and realms. A principal is an entity that we wish to authenticate. A role is a logical grouping of principals. A realm is the scope over which a common security policy applies.</p> 
            <p>Defining principals, roles, and realms is application server specific; we saw how to do this for GlassFish.</p> 
            <p>We saw how to configure the GlassFish application client container to authenticate an EJB application client.</p> 
            <p>There are two kinds of EJB authorization mechanisms: declarative and programmatic. With declarative authorization, security is managed by the container. The EJB specifies the desired access control using annotations. With programmatic authorization, the EJB controls authorization.</p> 
            <p>Finally we had a brief look at web-tier authorization and authentication.</p>
