       <p>A project in the works here at Vodori involves making a set of <a href="http://en.wikipedia.org/wiki/SOAP" target="_blank">SOAP</a> web service calls to an external system. This is generally a pretty routine exercise: set up the correct client code from the WSDL, integrate some kind of delivery mechanism, and then make the calls. At least, it’s routine when it isn’t hitting one roadblock after another.</p> 
       <h3><strong>Getting started: Axis to Axis2 to Spring-WS</strong></h3> 
       <p>For our purposes, we started out using <a href="http://ws.apache.org/axis/" target="_blank">Apache Axis</a> since it was a pre-existing piece of our platform. We used the existing WSDL2JAVA process to generate the client and connectivity code.</p> 
       <p>However, when we tried to actually connect to the system, we received a 401 response—our credentials were denied for failing an NTLM authentication. Digging around, I determined that the most likely culprit was a version mismatch between the NTLM support in Axis and the latest NTLMv2 support by the web service server.</p> 
       <p>We tried switching out to the newer <a href="http://axis.apache.org/axis2/java/core/" target="_blank">Axis2 project</a>, which boasted better NTLM support. Unfortunately, a change in the WSDL2JAVA process consolidated all of the code into a massive 11 MB, 226k-line Java source file. Including that file in our project ground IntelliJ to a halt, and it started throwing OutOfMemoryException errors whenever it would try to compile.</p> 
       <p>At this point, we switched to <a href="http://jaxb.java.net/" target="_blank">JAXB2’s xjc jar</a> to recreate the client files and Spring’s <a href="http://static.springsource.org/spring-ws/sites/2.0/" target="_blank">Spring-WS package</a> to handle the transport. This provided a simpler code setup and much greater visibility into how the calls were being made. Since we’re living in 2012 and not 2006, we probably should have started here in the first place.</p>

       <h3><strong>More problems</strong></h3> 
       <p>With the simpler setup in place, we were able to isolate the source of our troubles: <a href="http://hc.apache.org/httpclient-3.x/" target="_blank">Apache’s HTTPClient</a>. This library is pretty rock-solid and has been around for a while, and <em>everyone</em> uses it. It is the default (i.e., only) option for Axis, Axis2, and Spring-WS’s latest release.</p> 
       <p>One drawback: the library doesn’t support the latest authentication schemes, and Apache has since replaced it with the <a href="http://hc.apache.org/" target="_blank">HTTPComponents project</a> and its own HTTPClient class. A major side effect of this change was a wholesale break from the old 3.X HTTPClient codebase and package structure, precluding its use as a drop-in replacement.</p> 
       <p>Of course, we aren’t the only people encountering such snags. The ubiquity of these tools within the Java and Spring ecosystems has generated plenty of discussion and advice to draw upon.&nbsp;Looking at the Spring-WS site and its JIRA, I came across issue <a href="https://jira.springsource.org/browse/SWS-563" target="_blank">SWS-563</a>, which is a request for exactly what I needed.</p> 
       <p>The bad news is, it won’t see the light of day until 2.1 (or 2.1M1 for now), and Spring hasn’t yet published that version’s release calendar. The good news is, attached to SWS-563 are the three files updated by the Spring team to solve this issue. Downloading them and plugging them into my local codebase overwrote them in the Classpath, thus enabling me (for now) to use the most recent HTTPClient code.</p> 
       <p>Believing I had solved the issue, I ran my unit tests once more to validate the connection.</p> 
       <p>They failed.</p>

       <h3><strong><del><br>Third</del></strong><strong> <del>Fourth</del> Eighth time’s the charm</strong></h3> 
       <p>Being stubborn—and a geek—I decided to take a deeper look at the <a href="http://www.innovation.ch/personal/ronald/ntlm.html" target="_blank">authentication messages</a>. I logged out the actual NTLM handshake messages, ran them through a Base64Decoder, and analyzed the structure.</p> 
       <p>I found that part of the NTLMv2 handshake, our Type-3 message, was failing due to incorrect formatting. I chalked this up to the HTTPClient still not working correctly, so I went back to the Apache site and noticed this message in their <a href="http://hc.apache.org/httpcomponents-client-ga/ntlm.html" target="_blank">NTLM guide</a>:</p> 
       <blockquote> 
        <p><em>“HttpClient as of version 4.1 supports NTLMv1 and NTLMv2 authentication protocols out of the box using a custom authentication engine. However, there are still known compatibility issues with newer Microsoft products as the default NTLM engine implementation is still relatively new.”</em></p> 
       </blockquote> 
       <p>Luckily, they include some sample code that uses the Samba team’s <a href="http://jcifs.samba.org/" target="_blank">JCIFS</a> implementation as their NTLM engine. Pulling down that code, creating the relevant classes, and wiring it in was easier than I expected. With all of these components in place—JAXB2 2.X generated files, Spring-WS transport code, the Spring 2.1M1 files, HTTPComponents, and JCIFS—the web services connected successfully.</p>
