     <p>From <a rel="nofollow" class="external text" href="http://www.freedesktop.org/wiki/Software/polkit/">polkit homepage</a>: </p> 
     <dl>
      <dd>
       <i>polkit is an application-level toolkit for defining and handling the policy that allows unprivileged processes to speak to privileged processes: It is a framework for centralizing the decision making process with respect to granting access to privileged operations for unprivileged applications.</i>
      </dd>
     </dl> 
     <p>Polkit is used for controlling system-wide privileges. It provides an organized way for non-privileged processes to communicate with privileged ones. In contrast to systems such as sudo, it does not grant root permission to an entire process, but rather allows a finer level of control of centralized system policy. </p>
     <p>Polkit works by delimiting distinct actions, e.g. running GParted, and delimiting users by group or by name, e.g. members of the wheel group. It then defines how – if at all – those users are allowed those actions, e.g. by identifying as members of the group by typing in their passwords. </p>

     <h2><span class="mw-headline" id="Installation">Installation</span></h2> 
     <p>Install the <span class="plainlinks archwiki-template-pkg"><a rel="nofollow" class="external text" href="https://www.archlinux.org/packages/?name=polkit">polkit</a></span> package. </p> 
     <h3><span class="mw-headline" id="Authentication_agents">Authentication agents</span></h3> 
     <p>An authentication agent is used to make the user of a session prove that the user of the session really is the user (by authenticating as the user) or an administrative user (by authenticating as an administrator). The <span class="plainlinks archwiki-template-pkg"><a rel="nofollow" class="external text" href="https://www.archlinux.org/packages/?name=polkit">polkit</a></span> package contains a textual authentication agent called 'pkttyagent', which is used as a general fallback. </p>
     <p>If you are using a graphical environment, make sure that a graphical authentication agent is installed and <a href="/index.php/Autostarting" title="Autostarting">autostarted</a> on login. </p>
     <p><a href="/index.php/Cinnamon" title="Cinnamon">Cinnamon</a>, <a href="/index.php/Deepin" class="mw-redirect" title="Deepin">Deepin</a>, <a href="/index.php/GNOME" title="GNOME">GNOME</a>, <a href="/index.php/GNOME_Flashback" class="mw-redirect" title="GNOME Flashback">GNOME Flashback</a>, <a href="/index.php/KDE" title="KDE">KDE</a>, <a href="/index.php/LXDE" title="LXDE">LXDE</a>, <a href="/index.php/LXQt" title="LXQt">LXQt</a>, <a href="/index.php/MATE" title="MATE">MATE</a>, theShell and <a href="/index.php/Xfce" title="Xfce">Xfce</a> have an authentication agent already. In other desktop environments, you have to choose one of the following implementations: </p> 
     <ul>
      <li> <span class="plainlinks archwiki-template-pkg"><a rel="nofollow" class="external text" href="https://www.archlinux.org/packages/?name=lxqt-policykit">lxqt-policykit</a></span>, which provides <code>/usr/bin/lxqt-policykit-agent</code></li> 
      <li> <span class="plainlinks archwiki-template-pkg"><a rel="nofollow" class="external text" href="https://www.archlinux.org/packages/?name=lxsession">lxsession</a></span>, which provides <code>/usr/bin/lxpolkit</code></li> 
      <li> <span class="plainlinks archwiki-template-pkg"><a rel="nofollow" class="external text" href="https://www.archlinux.org/packages/?name=mate-polkit">mate-polkit</a></span>, which provides <code>/usr/lib/mate-polkit/polkit-mate-authentication-agent-1</code></li> 
      <li> <span class="plainlinks archwiki-template-pkg"><a rel="nofollow" class="external text" href="https://aur.archlinux.org/packages/polkit-efl-git/">polkit-efl-git</a></span><sup><small>AUR</small></sup>, which provides <code>/usr/bin/polkit-efl-authentication-agent-1</code></li> 
      <li> <span class="plainlinks archwiki-template-pkg"><a rel="nofollow" class="external text" href="https://www.archlinux.org/packages/?name=polkit-gnome">polkit-gnome</a></span>, which provides <code>/usr/lib/polkit-gnome/polkit-gnome-authentication-agent-1</code></li> 
      <li> <span class="plainlinks archwiki-template-pkg"><a rel="nofollow" class="external text" href="https://www.archlinux.org/packages/?name=polkit-kde-agent">polkit-kde-agent</a></span>, which provides <code>/usr/lib/polkit-kde/polkit-kde-authentication-agent-1</code></li> 
      <li> <span class="plainlinks archwiki-template-pkg"><a rel="nofollow" class="external text" href="https://aur.archlinux.org/packages/ts-polkitagent/">ts-polkitagent</a></span><sup><small>AUR</small></sup>, which provides <code>/usr/lib/ts-polkitagent</code></li> 
      <li> <span class="plainlinks archwiki-template-pkg"><a rel="nofollow" class="external text" href="https://aur.archlinux.org/packages/xfce-polkit-git/">xfce-polkit-git</a></span><sup><small>AUR</small></sup>, which provides <code>/usr/lib/xfce-polkit/xfce-polkit</code></li>
     </ul> 
     <h2><span class="mw-headline" id="Configuration">Configuration</span></h2> 
     <div style="padding: 5px; margin: 0.50em 0; background-color: #FFDDDD; border: thin solid #DDBBBB; overflow: hidden;">
      <strong> Warning: </strong>Do not amend the default permission files of packages, as these may be be overwritten on package upgrades.
     </div> 
     <p>Polkit definitions can be divided into two kinds: </p> 
     <ul>
      <li> <b>Actions</b> are defined in XML <code>.policy</code> files located in <code>/usr/share/polkit-1/actions</code>. Each action has a set of default permissions attached to it (e.g. you need to identify as an administrator to use the GParted action). The defaults can be overruled but editing the actions files is NOT the correct way.</li> 
      <li><b>Authorization rules</b> are defined in JavaScript <code>.rules</code> files. They are found in two places: 3rd party packages can use <code>/usr/share/polkit-1/rules.d</code> (though few if any do) and <code>/etc/polkit-1/rules.d</code> is for local configuration. </li>
     </ul> 
     <p>Polkit operates on top of the existing permissions systems in Linux – group membership, administrator status – it does not replace them. The .rules files designate a subset of users, refer to one (or more) of the actions specified in the actions files and determine with what restrictions these actions can be taken by that/those user(s). As an example, a rules file could overrule the default requirement for all users to authenticate as an admin when using GParted, determining that some specific user doesn't need to. Or isn't allowed to use GParted at all. </p> 
     <div style="padding: 5px; margin: 0.50em 0; background-color: #DDDDFF; border: thin solid #BBBBDD; overflow: hidden;">
      <strong> Note: </strong>This does not preclude running GParted by means which do not respect polkit, such as the command line. Therefore, polkit should be used to expand access to privileged services for unprivileged users, rather than try to curtail the rights of (semi-)privileged users. For security purposes, 
      <a href="/index.php/Sudo" title="Sudo">sudoers</a> is still the way to go.
     </div> 
     <h3><span class="mw-headline" id="Actions">Actions</span></h3> 
     <div style="padding: 5px; margin: 0.50em 0; background-color: #DDFFDD; border: thin solid #BBDDBB; overflow: hidden;">
      <strong> Tip: </strong>To display Policykit actions in a graphical interface, install the 
      <span class="plainlinks archwiki-template-pkg"><a rel="nofollow" class="external text" href="https://aur.archlinux.org/packages/polkit-explorer/">polkit-explorer</a></span>
      <sup><small>AUR</small></sup> package.
     </div> 
     <p>The actions available to you via polkit will depend on the packages you have installed. Some are used in multiple desktop environments <i>(org.freedesktop.*)</i>, some are DE-specific <i>(org.gnome.*)</i> and some are specific to a single program <i>(org.archlinux.pkexec.gparted.policy)</i>. The command <code>pkaction</code> lists all the actions defined in <code>/usr/share/polkit-1/actions</code> for quick reference. </p>
     <p>To get an idea of what polkit can do, here are a few commonly used groups of actions: </p> 
     <ul>
      <li> <b><a href="/index.php/Systemd" title="Systemd">systemd-logind</a></b> <i>(org.freedesktop.login1.policy)</i> actions regulated by polkit include powering off, rebooting, suspending and hibernating the system, including when other users may still be logged in.</li> 
      <li> <b><a href="/index.php/Udisks" title="Udisks">udisks</a></b> <i>(org.freedesktop.udisks2.policy)</i> actions regulated by polkit include mounting file systems and unlocking encrypted devices.</li> 
      <li> <b><a href="/index.php/NetworkManager" title="NetworkManager">NetworkManager</a></b> <i>(org.freedesktop.NetworkManager.policy)</i> actions regulated by polkit include turning on and off the network, wifi or mobile broadband.</li>
     </ul> 
     <p>Each action is defined in an <code>&lt;action&gt;</code> tag in a .policy file. The <code>org.archlinux.pkexec.gparted.policy</code> contains a single action and looks like this: </p> 
     <pre>&lt;?xml version="1.0" encoding="UTF-8"?&gt;
&lt;!DOCTYPE policyconfig PUBLIC
 "-//freedesktop//DTD PolicyKit Policy Configuration 1.0//EN"
 "http://www.freedesktop.org/software/polkit/policyconfig-1.dtd"&gt;
&lt;policyconfig&gt;

  &lt;action id="org.archlinux.pkexec.gparted"&gt;
    &lt;message&gt;Authentication is required to run the GParted Partition Editor&lt;/message&gt;
    &lt;icon_name&gt;gparted&lt;/icon_name&gt;
    &lt;defaults&gt;
      &lt;allow_any&gt;auth_admin&lt;/allow_any&gt;
      &lt;allow_inactive&gt;auth_admin&lt;/allow_inactive&gt;
      &lt;allow_active&gt;auth_admin&lt;/allow_active&gt;
    &lt;/defaults&gt;
    &lt;annotate key="org.freedesktop.policykit.exec.path"&gt;/usr/bin/gparted&lt;/annotate&gt;
    &lt;annotate key="org.freedesktop.policykit.exec.allow_gui"&gt;true&lt;/annotate&gt;
  &lt;/action&gt;

&lt;/policyconfig&gt;
</pre> 
     <p>The attribute <b>id</b> is the actual command sent to <a href="/index.php/D-Bus" title="D-Bus">D-Bus</a>, the <b>message</b> tag is the explanation to the user when authentication is required and the <b>icon_name</b> is sort of obvious. </p>
     <p>The <b>defaults</b> tag is where the permissions or lack thereof are located. It contains three settings: <b>allow_any</b>, <b>allow_inactive</b>, and <b>allow_active</b>. Inactive sessions are generally remote sessions (SSH, VNC, etc.) whereas active sessions are logged directly into the machine on a TTY or an X display. allow_any is the setting encompassing both scenarios. </p>
     <p>For each of these settings the following options are available: </p> 
     <ul>
      <li> <i>no</i>: The user is not authorized to carry out the action. There is therefore no need for authentication.</li> 
      <li> <i>yes</i>: The user is authorized to carry out the action without any authentication.</li> 
      <li> <i>auth_self</i>: Authentication is required but the user need not be an administrative user.</li> 
      <li> <i>auth_admin</i>: Authentication as an administrative user is required.</li> 
      <li> <i>auth_self_keep</i>: The same as auth_self but, like sudo, the authorization lasts a few minutes.</li> 
      <li> <i>auth_admin_keep</i>: The same as auth_admin but, like sudo, the authorization lasts a few minutes.</li>
     </ul> 
     <p>These are default setting and unless overruled in later configuration will be valid for all users. </p>
     <p>As can be seen from the GParted action, users are required to authenticate as administrators in order to use GParted, regardless of whether the session is active or inactive. </p> 
     <h3><span class="mw-headline" id="Authorization_rules">Authorization rules</span></h3> 
     <p>Authorization rules that overrule the default settings are laid out in a set of directories as described above. For all purposes relating to personal configuration of a single system, only <code>/etc/polkit-1/rules.d</code> should be used. </p>
     <p>The <code>addRule()</code> method is used for adding a function that may be called whenever an authorization check for action and subject is performed. Functions are called in the order they have been added until one of the functions returns a value. Hence, to add an authorization rule that is processed before other rules, put it in a file in <code>/etc/polkit-1/rules.d</code> with a name that sorts before other rules files, for example <code>00-early-checks.rules</code>. </p>
     <p>The layout of the .rules files is fairly self-explanatory: </p> 
     <pre>/* Allow users in admin group to run GParted without authentication */
polkit.addRule(function(action, subject) {
    if (action.id == "org.archlinux.pkexec.gparted" &amp;&amp;
        subject.isInGroup("admin")) {
        return polkit.Result.YES;
    }
});
</pre> 
     <p>Inside the function, we check for the specified action ID <i>(org.archlinux.pkexec.gparted)</i> and for the user's group <i>(admin)</i>, then return a value "yes". </p> 
     <h3><span class="mw-headline" id="Administrator_identities">Administrator identities</span></h3> 
     <p>The <code>addAdminRule()</code> method is used for adding a function that may be called whenever administrator authentication is required. The function is used to specify what identities may be used for administrator authentication for the authorization check identified by action and subject. Functions added are called in the order they have been added until one of the functions returns a value. </p>
     <p>The default configuration for administrator identities is contained in the file <code>50-default.rules</code> so any changes to that configuration should be made by copying the file to, say, <code>40-default.rules</code> and editing that file. </p> 
     <pre style="margin-bottom: 0; border-bottom:none; padding-bottom:0.8em;">/etc/polkit-1/rules.d/50-default.rules</pre> 
     <pre style="margin-top: 0; border-top-style:dashed; padding-top: 0.8em;">
polkit.addAdminRule(function(action, subject) {
    return ["unix-group:wheel"];
});</pre> 
     <p>The only part to edit (once copied) is the return array of the function: as whom should a user authenticate when asked to authenticate as an administrative user? If she herself is a member of the group designated as admins, she only need enter her own password. If some other user, e.g. root, is the only admin identity, she would need to enter in root's password. The format of the user identification is the same as the one used in designating authorities. </p>
     <p>The Arch default is to make all members of the group <b>wheel</b> administrators. A rule like below will have polkit ask for the root password instead of the users password for Admin authentication. </p> 
     <pre style="margin-bottom: 0; border-bottom:none; padding-bottom:0.8em;">/etc/polkit-1/rules.d/49-rootpw_global.rules</pre> 
     <pre style="margin-top: 0; border-top-style:dashed; padding-top: 0.8em;">
/* Always authenticate Admins by prompting for the root
 * password, similar to the rootpw option in sudo
 */
polkit.addAdminRule(function(action, subject) {
    return ["unix-user:root"];
});
</pre> 
     <h2><span class="mw-headline" id="Examples">Examples</span></h2> 
     <h3><span class="mw-headline" id="Debugging.2Flogging">Debugging/logging</span></h3> 
     <p>The following rule logs detailed information about any requested access. </p> 
     <pre style="margin-bottom: 0; border-bottom:none; padding-bottom:0.8em;">/etc/polkit-1/rules.d/00-log-access.rules</pre> 
     <pre style="margin-top: 0; border-top-style:dashed; padding-top: 0.8em;">
polkit.addRule(function(action, subject) {
    polkit.log("action=" + action);
    polkit.log("subject=" + subject);
});</pre> 
     <h3><span class="mw-headline" id="Disable_suspend_and_hibernate">Disable suspend and hibernate</span></h3> 
     <p>The following rule disables suspend and hibernate for all users. </p> 
     <pre style="margin-bottom: 0; border-bottom:none; padding-bottom:0.8em;">/etc/polkit-1/rules.d/10-disable-suspend.rules</pre> 
     <pre style="margin-top: 0; border-top-style:dashed; padding-top: 0.8em;">
polkit.addRule(function(action, subject) {
    if (action.id == "org.freedesktop.login1.suspend" ||
        action.id == "org.freedesktop.login1.suspend-multiple-sessions" ||
        action.id == "org.freedesktop.login1.hibernate" ||
        action.id == "org.freedesktop.login1.hibernate-multiple-sessions")
    {
        return polkit.Result.NO;
    }
});</pre> 
     <h3><span class="mw-headline" id="Bypass_password_prompt">Bypass password prompt</span></h3> 
     <p>To achieve something similar to the <a href="/index.php/Sudo" title="Sudo">sudo</a> <code>NOPASSWD</code> option and get authorized solely based on <a href="/index.php/Users_and_groups" title="Users and groups">user/group</a> identity, you can create custom rules in <code>/etc/polkit-1/rules.d/</code>. This allows you to override password authentication either <a href="#For_specific_actions">only for specific actions</a> or <a href="#Globally">globally</a>. See <a rel="nofollow" class="external autonumber" href="https://gist.github.com/4013294/ccacedd69d54de7f2fd5881b546d5192d6a2bddb">[1]</a> for an example rule set. </p> 
     <h4><span class="mw-headline" id="Globally">Globally</span></h4> 
     <p>Create the following file as root: </p> 
     <pre style="margin-bottom: 0; border-bottom:none; padding-bottom:0.8em;">/etc/polkit-1/rules.d/49-nopasswd_global.rules</pre> 
     <pre style="margin-top: 0; border-top-style:dashed; padding-top: 0.8em;">
/* Allow members of the wheel group to execute any actions
 * without password authentication, similar to "sudo NOPASSWD:"
 */
polkit.addRule(function(action, subject) {
    if (subject.isInGroup("wheel")) {
        return polkit.Result.YES;
    }
});
</pre> 
     <p>Replace <code>wheel</code> by any group of your preference. </p>
     <p>This will result in automatic authentication for <b>any</b> action requiring admin rights via Polkit. As such, be careful with the group you choose to give such rights to. </p> 
     <h4><span class="mw-headline" id="For_specific_actions">For specific actions</span></h4> 
     <p>Create the following file as root: </p> 
     <pre style="margin-bottom: 0; border-bottom:none; padding-bottom:0.8em;">/etc/polkit-1/rules.d/49-nopasswd_limited.rules</pre> 
     <pre style="margin-top: 0; border-top-style:dashed; padding-top: 0.8em;">
/* Allow members of the wheel group to execute the defined actions 
 * without password authentication, similar to "sudo NOPASSWD:"
 */
polkit.addRule(function(action, subject) {
    if ((action.id == "org.archlinux.pkexec.gparted" ||
	 action.id == "org.libvirt.unix.manage") &amp;&amp;
        subject.isInGroup("wheel"))
    {
        return polkit.Result.YES;
    }
});
</pre> 
     <p>The <code>action.id</code>s selected here are just (working) examples for GParted and <a href="/index.php/Libvirt" title="Libvirt">Libvirt</a>, but you can replace them by any other of your liking as long as they exist (custom made or supplied by a package), and so can you define any group instead of <code>wheel</code>. </p>
     <p>The <code>||</code> operator is used to delimit actions (logical OR), and <code>&amp;&amp;</code> means logical AND and must be kept as the last operator. </p> 
     <h4><span class="mw-headline" id="Udisks">Udisks</span></h4> 
     <p><a href="/index.php/File_manager" class="mw-redirect" title="File manager">File managers</a> may ask for a password when trying to mount a storage device, or yield a <i>Not authorized</i> or similar error. See <a href="/index.php/Udisks#Configuration" title="Udisks">Udisks#Configuration</a> for details. </p> 
     <h3><span class="mw-headline" id="Allow_management_of_individual_systemd_units_by_regular_users">Allow management of individual systemd units by regular users</span></h3> 
     <p>By checking for certain values passed to the polkit policy check, you can give specific users or groups the ability to manage specific units. As an example, you might want regular users to start and stop <a href="/index.php/Wpa_supplicant" class="mw-redirect" title="Wpa supplicant">wpa_supplicant</a>: </p> 
     <pre style="margin-bottom: 0; border-bottom:none; padding-bottom:0.8em;">/etc/polkit-1/rules.d/10-wifimanagement.rules</pre> 
     <pre style="margin-top: 0; border-top-style:dashed; padding-top: 0.8em;">
polkit.addRule(function(action, subject) {
    if (action.id == "org.freedesktop.systemd1.manage-units") {
        if (action.lookup("unit") == "wpa_supplicant.service") {
            var verb = action.lookup("verb");
            if (verb == "start" || verb == "stop" || verb == "restart") {
                return polkit.Result.YES;
            }
        }
    }
});
</pre>
