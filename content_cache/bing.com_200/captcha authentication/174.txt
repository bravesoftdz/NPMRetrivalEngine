         <div class="section" id="overview"> 
          <h2>Overview<a class="headerlink" href="#overview" title="Permalink to this headline">¶</a></h2> 
          <p>Ops Manager stores configuration settings both globally in the Ops Manager Application Database and locally on each server. Global settings apply to all your Ops Manager servers. Local settings apply to the server on which they are configured. Any local settings on a server override the global settings.</p> 
          <p>You configure global settings through the Ops Manager interface during installation. You can edit global settings at any time through the <span class="guilabel">Admin</span> interface by clicking the <span class="guilabel">General</span> tab and then clicking <span class="guilabel">Ops Manager Config</span>.</p> 
          <p>You configure local settings through a server’s <code class="docutils literal"><span class="pre">conf-mms.properties</span></code> file. Each server’s <code class="docutils literal"><span class="pre">conf-mms.properties</span></code> must contain the connection string and authentication settings for accessing the Ops Manager Application Database. The <code class="docutils literal"><span class="pre">conf-mms.properties</span></code> file also contains any overrides of global settings specific to that server.</p> 
          <p>The location of the <code class="docutils literal"><span class="pre">conf-mms.properties</span></code> file depends on how you installed Ops Manager, as described in the table below.</p> 
          <table border="1" class="colwidths-given docutils"> 
           <colgroup> 
            <col width="35%"> 
            <col width="65%"> 
           </colgroup> 
           <thead valign="bottom"> 
            <tr class="row-odd">
             <th class="head">Install method</th> 
             <th class="head">conf-mms.properties location</th> 
            </tr> 
           </thead> 
           <tbody valign="top"> 
            <tr class="row-even">
             <td><code class="docutils literal"><span class="pre">rpm</span></code> or <code class="docutils literal"><span class="pre">deb</span></code> package</td> 
             <td><code class="docutils literal"><span class="pre">/opt/mongodb/mms/conf/</span></code></td> 
            </tr> 
            <tr class="row-odd">
             <td><code class="docutils literal"><span class="pre">tar.gz</span></code> archive</td> 
             <td><code class="docutils literal"><span class="pre">&lt;install-directory&gt;/conf/</span></code></td> 
            </tr> 
            <tr class="row-even">
             <td><code class="docutils literal"><span class="pre">msi</span></code> file (Windows)</td> 
             <td><p class="first"><code class="docutils literal"><span class="pre">&lt;install-folder&gt;\Server\Config</span></code></p> <p class="last">By default, this is: <code class="docutils literal"><span class="pre">C:\MMSData\Server\Config</span></code>.</p> </td> 
            </tr>

If you change this default, you must also change the port specified in the <code class="xref mongodb mongodb-setting docutils literal"><span class="pre">URL</span> <span class="pre">to</span> <span class="pre">Access</span> <span class="pre">Ops</span> <span class="pre">Manager</span></code> setting.</p>

           </dt>

           </dd>
          </dl>

</pre>

A load balancer placed in front of the Ops Manager Application servers must not return cached content.</p> 
            <p>See <a class="reference internal" href="../../tutorial/configure-application-high-availability/"><span class="doc">Configure a Highly Available Ops Manager Application</span></a> for more information.</p>

          <h2>Authentication through LDAP<a class="headerlink" href="#authentication-through-ldap" title="Permalink to this headline">¶</a></h2> 
          <p>These settings configure Ops Manager to use an LDAP server for authentication. If you use LDAP authentication, users must belong to an LDAP group to log into Ops Manager. You must create LDAP groups for each Ops Manager <a class="reference internal" href="../user-roles/"><span class="doc">user role</span></a>.</p> 
          <p><em>Settings that begin with “mms.ldap.global.role”</em> assign Ops Manager <a class="reference internal" href="../user-roles/#global-roles"><span class="std std-ref">global roles</span></a> to the members of the specified LDAP groups. Specify groups using the format used by the LDAP attribute specified in the <code class="xref mongodb mongodb-setting docutils literal"><span class="pre">LDAP</span> <span class="pre">User</span> <span class="pre">Group</span></code> setting. You can specify multiple groups using the <code class="docutils literal"><span class="pre">;;</span></code> delimiter. To change the default delimiter, use the <code class="xref mongodb mongodb-setting docutils literal"><span class="pre">mms.ldap.group.separator</span></code> setting. Each Ops Manager global role provides its level of access to all the Ops Manager <a class="reference internal" href="../../tutorial/create-group/"><span class="doc">groups</span></a> in the deployment. To provide access to specific groups, use <a class="reference internal" href="../user-roles/"><span class="doc">group-level roles</span></a>.</p>

         <div class="section" id="multi-factor-authentication-mfa-settings"> 
          <h2>Multi-Factor Authentication (MFA) Settings<a class="headerlink" href="#multi-factor-authentication-mfa-settings" title="Permalink to this headline">¶</a></h2> 
          <dl class="setting"> 
           <dt id="Multi-factor-Auth-Level"> 
            <code class="descname">Multi-factor Auth Level</code>
            <a class="headerlink" href="#Multi-factor-Auth-Level" title="Permalink to this definition">¶</a>
           </dt> 
           <dd>
            <p><em>Type</em>: string</p> 
            <p><em>Default</em>: OFF</p> 
            <p>Configures the <a class="reference internal" href="../../tutorial/manage-two-factor-authentication/"><span class="doc">two-factor authentication</span></a> “level”:</p> 
            <ul class="simple"> 
             <li><code class="docutils literal"><span class="pre">OFF</span></code>: Disables two-factor authentication. Ops Manager does not use two-factor authentication.</li> 
             <li><code class="docutils literal"><span class="pre">OPTIONAL</span></code>: Users can choose to set up two-factor authentication for their Ops Manager account.</li> 
             <li><code class="docutils literal"><span class="pre">REQUIRED_FOR_GLOBAL_ROLES</span></code>: Users who possess a <a class="reference internal" href="../user-roles/#global-roles"><span class="std std-ref">global role</span></a> <strong>must</strong> set up two-factor authentication. Two factor authentication is optional for all other users.</li> 
             <li><code class="docutils literal"><span class="pre">REQUIRED</span></code>: <strong>All</strong> users must set up two-factor authentication for their Ops Manager account.</li> 
            </ul> 
            <p>Two-factor authentication is recommended for the security of your Ops Manager deployment.</p> 
            <p>Corresponds to configuration file setting: <code class="docutils literal"><span class="pre">mms.multiFactorAuth.level</span></code>.</p> 
            <div class="admonition warning"> 
             <p class="first admonition-title">Warning</p> 
             <p class="last">If enabling <code class="docutils literal"><span class="pre">mms.multiFactorAuth.level</span></code> through the configuration file, you must create a <a class="reference internal" href="../../tutorial/manage-users/#add-users"><span class="std std-ref">user account</span></a> first before updating the configuration file.          <div class="section" id="backup-snapshots"> 
          <span id="automation-backup-compression-settings"></span>
          <h2>Backup Snapshots<a class="headerlink" href="#backup-snapshots" title="Permalink to this headline">¶</a></h2> 
          <p>The following settings determine:</p> 
          <ul class="simple"> 
           <li>How much Ops Manager compresses file system store snapshots.</li> 
           <li>How frequently Ops Manager takes snapshots.</li> 
           <li>How long Ops Manager stores snapshots.</li> 
          </ul> 
          <p>To set these values, click the <span class="guilabel">Admin</span> link, then the <span class="guilabel">General</span> tab, then the <span class="guilabel">Ops Manager Config</span> page, and then the <span class="guilabel">Backup</span> section.</p> 
          <div class="admonition seealso"> 
           <p class="first admonition-title">See also</p> 
           <p class="last">See <a class="reference internal" href="../../core/backup-preparations/#snapshot-frequency-and-retention"><span class="std std-ref">Snapshot Frequency and Retention Policy</span></a> to learn more about how often snapshots are taken and how long they can be retained.</p> 
          </div> 
          <dl class="setting"> 
           <dt id="File-System-Store-Gzip-Compression-Level"> 
            <code class="descname">File System Store Gzip Compression Level</code>
            <a class="headerlink" href="#File-System-Store-Gzip-Compression-Level" title="Permalink to this definition">¶</a>
           </dt> 
           <dd>
            <p><em>Type</em>: integer</p> 
            <p><em>Default</em>: 6</p> 
            <p>Determines how much Ops Manager compresses file system-based snapshots. The level ranges from <code class="docutils literal"><span class="pre">0</span></code> to <code class="docutils literal"><span class="pre">9</span></code>:</p> 
            <ul class="simple"> 
             <li><code class="docutils literal"><span class="pre">0</span></code> provides no compression.</li> 
             <li><code class="docutils literal"><span class="pre">1</span></code> to <code class="docutils literal"><span class="pre">9</span></code> increases the degree of compression at a cost of how fast the snapshot is compressed. Level <code class="docutils literal"><span class="pre">1</span></code> compresses snapshots the least but at the fastest speed. Level <code class="docutils literal"><span class="pre">9</span></code> compresses snapshots the most but at the slowest speed.</li> 
            </ul> 
            <div class="admonition note"> 
             <p class="first admonition-title">Note</p> 
             <p class="last">Changing <span class="guilabel">File System Store Gzip Compression Level</span> affects new snapshots only. It does not affect the compression level of existing snapshots.</p> 
            </div> 
            <p><span class="guilabel">File System Store Gzip Compression Level</span> corresponds to the <code class="docutils literal"><span class="pre">backup.fileSystemSnapshotStore.gzip.compressionLevel</span></code> configuration file setting.</p> 
           </dd>
          </dl> 
          <dl class="setting"> 
           <dt id="Snapshot-Interval"> 
            <code class="descname">Snapshot Interval</code>
            <a class="headerlink" href="#Snapshot-Interval" title="Permalink to this definition">¶</a>
           </dt> 
           <dd>
            <p><em>Type</em>: integer</p> 
            <p><em>Default</em>: 24</p> 
            <p>Specifies the time, in hours, between two consecutive snapshots.</p> 
            <p><span class="guilabel">Snapshot Interval (Hours)</span> corresponds to the <code class="docutils literal"><span class="pre">brs.snapshotSchedule.interval</span></code> configuration file setting.</p> 
           </dd>
          </dl>

           </dt> 
           <dd>
            <p><em>Type</em>: integer</p> 
            <p><em>Default</em>: 2</p> 
            <p>Specifies how many days an interval snapshot is stored.</p> 
            <p><span class="guilabel">Base Retention of Snapshots (in Days)</span> corresponds to the <code class="docutils literal"><span class="pre">brs.snapshotSchedule.retention.base</span></code> configuration file setting.</p>

           </dt> 
           <dd>

            <p>Specifies how many days a daily snapshot is stored.</p> 
            <p><span class="guilabel">Daily Retention of Snapshots (in Days)</span> corresponds to the <code class="docutils literal"><span class="pre">brs.snapshotSchedule.retention.daily</span></code> configuration file setting.</p> 
           </dd>
          </dl> 
          <dl class="setting"> 
           <dt id="Weekly-Retention-of-Snapshots"> 
            <code class="descname">Weekly Retention of Snapshots</code>
            <a class="headerlink" href="#Weekly-Retention-of-Snapshots" title="Permalink to this definition">¶</a>

          <dl class="setting"> 
           <dt id="Monthly-Retention-of-Snapshot"> 
            <code class="descname">Monthly Retention of Snapshot</code>
            <a class="headerlink" href="#Monthly-Retention-of-Snapshot" title="Permalink to this definition">¶</a>
           </dt> 
           <dd>
            <p><em>Type</em>: integer</p>

         <div class="section" id="default-monitoring-data-retention"> 
          <span id="id4"></span>
          <h2>Default Monitoring Data Retention<a class="headerlink" href="#default-monitoring-data-retention" title="Permalink to this headline">¶</a></h2> 
          <p>Ops Manager gathers metric data at a 10-second granularity. The <span class="guilabel">Default Monitoring Data Retention</span> table determines how long Ops Manager stores metric data. For each increasing granularity level, Ops Manager computes the data based on the averages from the previous granularity level.</p> 
          <p>The table determines the default settings for new groups. If you change the settings, Ops Manager prompts you whether to also apply the settings to existing groups. To change the settings for a specific group without changing the Ops Manager default settings, see <a class="reference internal" href="../../core/administration-interface/#groups-page-admin-ui"><span class="std std-ref">Groups Page</span></a>.</p> 
          <p>Increasing the retention period for a granularity requires more storage on the <a class="reference internal" href="../../core/system-overview/#mms-application-database"><span class="std std-ref">Ops Manager Application Database</span></a>.</p> 
          <div class="admonition note"> 
           <p class="first admonition-title">Note</p> 
           <p class="last">Decreasing the retention period for existing groups does not immediately recovery available disk space on the file system and can actually use more disk space in the short term during the transition to the shorter retention period.</p> 
          </div> 
         </div>

         <div class="section" id="monitoring-agent-session-failover"> 
          <span id="standby-monitoring-agent"></span>
          <span id="config-monitoring-agent-failover"></span>
          <h2>Monitoring Agent Session Failover<a class="headerlink" href="#monitoring-agent-session-failover" title="Permalink to this headline">¶</a></h2> 
          <p><strong>Beginning with Monitoring Agent version 5.0.0</strong>, Ops Manager can distribute monitoring assignments among up to 100 running Monitoring Agents. One agent is the primary agent and the others share in monitoring responsibilities. If an agent fails, Ops Manager redistributes that agent’s monitoring assignments. If you run more than 100 Monitoring Agents, the additional agents run as standby agents that are completely idle, except to log their status as standby agents and to periodically ask Ops Manager whether they should receive monitoring assignments.</p> 
          <div class="admonition note"> 
           <p class="first admonition-title">Note</p> 
           <p class="last">Also beginning with version 5.0.0, the Monitoring Agent stores monitoring metrics at 10-second granularity.</p> 
          </div> 
          <p><strong>Prior to Monitoring Agent 5.0.0</strong>, only the primary Monitoring Agent handles monitoring assignments. All additional running agents are standby agents.</p> 
          <p>The following settings tune the interval Ops Manager uses to determine if a Monitoring Agent is unaccessible and the frequency with which standby agents poll Ops Manager to determine if they should receive monitoring assignments.</p> 
          <p>To add the following settings, click the <span class="guilabel">Admin</span> link, then the <span class="guilabel">General</span> tab, then the <span class="guilabel">Ops Manager Config</span> page, and then the <span class="guilabel">Custom</span> section.</p> 
          <dl class="setting"> 
           <dt id="mms.monitoring.agent.session.timeoutMillis">

            <a class="headerlink" href="#mms.monitoring.agent.session.timeoutMillis" title="Permalink to this definition">¶</a>
           </dt> 
           <dd>
            <p><em>Type</em>: integer</p> 
            <p><em>Default</em>: <code class="docutils literal"><span class="pre">90000</span></code></p> 
            <p>The interval that Ops Manager uses to determine if a standby agent should start monitoring. If Ops Manager does not hear from a Monitoring Agent for the duration specified, Ops Manager promotes a standby agent. Configuring the timeout below <cite>90000</cite> (90 seconds) will cause Ops Manager to fail at startup with a configuration error.</p> 
           </dd>
          </dl> 
          <dl class="setting"> 
           <dt id="mms.monitoring.agent.standbyCollectionFactor"> 
            <code class="descclassname">mms.monitoring.agent.</code>

            <a class="headerlink" href="#mms.monitoring.agent.standbyCollectionFactor" title="Permalink to this definition">¶</a>
           </dt> 
           <dd>
            <p><em>Type</em>: Integer</p> 
            <p><em>Default</em>: <code class="docutils literal"><span class="pre">4</span></code></p> 
            <p>Specifies how frequently a standby agent checks in with Ops Manager to see if it should start monitoring. The following values are permitted:</p> 
            <ul class="simple"> 
             <li><code class="docutils literal"><span class="pre">1</span></code>: the standby agents check every 55 seconds.</li> 
             <li><code class="docutils literal"><span class="pre">2</span></code>: the standby agents check in at twice the rate as <code class="docutils literal"><span class="pre">1</span></code>, or approximately every 27 seconds.</li> 
             <li><code class="docutils literal"><span class="pre">3</span></code>: the standby agents check approximately every 18 seconds</li> 
             <li><code class="docutils literal"><span class="pre">4</span></code>: the standby agents check approximately every 14 seconds.</li> 
            </ul> 
           </dd>
          </dl>

             <li>Authentication credentials for the <code class="xref mongodb mongodb-parameter docutils literal"><span class="pre">authentication</span> <span class="pre">mechanism</span></code> used on the Ops Manager Application database.</li> 
            </ul> 
            <p>See the following example connection strings:</p> 
            <ul> 
             <li><p class="first"><strong>Replica Sets:</strong> If you use a <a class="reference external" href="https://docs.mongodb.com/manual/reference/glossary/#term-replica-set" title="(in mongodb-manual v3.4)"><span class="xref std std-term">replica set</span></a> for the database’s <a class="reference internal" href="../../tutorial/prepare-backing-mongodb-instances/"><span class="doc">backing instance</span></a>, specify all members of the replica set, as shown in the example below for a replica set named <code class="docutils literal"><span class="pre">appdbRS</span></code>. If you omit the port number, Ops Manager uses the default <code class="docutils literal"><span class="pre">27017</span></code> port for all hosts.</p> 
              <div class="highlight-ini">
               <div class="highlight">
                <pre><span></span><span class="na">mongo.mongoUri</span><span class="o">=</span><span class="s">mongodb://db1.example.com:40000,db2.example.com:40000,db3.example.com:40000</span>
</pre>
               </div> 
              </div> </li> 
             <li><p class="first"><strong>Default MongoDB Authentication:</strong> For a MongoDB instance using the MongoDB <code class="docutils literal"><span class="pre">SCRAM-SHA-1</span></code> or <code class="docutils literal"><span class="pre">MONGODB-CR</span></code> challenge-response mechanisms, the connection string must include authentication credentials. The Ops Manager Application must authenticate as a MongoDB user with the following roles:</p> 
              <ul class="simple"> 
               <li><a class="reference external" href="https://docs.mongodb.com/manual/reference/built-in-roles/#readWriteAnyDatabase" title="(in mongodb-manual v3.4)"><code class="xref mongodb mongodb-authrole docutils literal"><span class="pre">readWriteAnyDatabase</span></code></a></li> 
               <li><a class="reference external" href="https://docs.mongodb.com/manual/reference/built-in-roles/#dbAdminAnyDatabase" title="(in mongodb-manual v3.4)"><code class="xref mongodb mongodb-authrole docutils literal"><span class="pre">dbAdminAnyDatabase</span></code></a>.</li> 
               <li><a class="reference external" href="https://docs.mongodb.com/manual/reference/built-in-roles/#clusterAdmin" title="(in mongodb-manual v3.4)"><code class="xref mongodb mongodb-authrole docutils literal"><span class="pre">clusterAdmin</span></code></a> if the database is a sharded cluster, otherwise <a class="reference external" href="https://docs.mongodb.com/manual/reference/built-in-roles/#clusterMonitor" title="(in mongodb-manual v3.4)"><code class="xref mongodb mongodb-authrole docutils literal"><span class="pre">clusterMonitor</span></code></a></li> 
              </ul> <p>Prefix the hostname with the MongoDB username and password in the form <code class="docutils literal"><span class="pre">&lt;username&gt;:&lt;password&gt;@</span></code></p> 
              <div class="highlight-ini">
               <div class="highlight">
                <pre><span></span><span class="na">mongo.mongoUri</span><span class="o">=</span><span class="s">mongodb://mongodbuser1:password@mydb1.example.com:40000</span>
</pre>
               </div> 
              </div> </li> 
             <li><p class="first"><strong>x.509 Certificate Authentication:</strong> For a MongoDB instance using <code class="docutils literal"><span class="pre">MONGODB-X509</span></code> authentication, you must first add the value of the <strong>subject</strong> from the client certificate as a MongoDB user, as described in <a class="reference external" href="https://docs.mongodb.com/manual/tutorial/configure-x509-client-authentication">Use x.509 Certificates to Authenticate Clients</a> in the MongoDB manual. The client certificate is contained in the PEM file you specify in the <code class="xref mongodb mongodb-setting docutils literal"><span class="pre">mongodb.ssl.PEMKeyFile</span></code> setting. Once you have created the user, prefix the host specified in <code class="xref mongodb mongodb-setting docutils literal"><span class="pre">mongo.mongoUri</span></code> with the name of the new user and append <code class="docutils literal"><span class="pre">authMechanism=MONGODB-X509</span></code> after the specified port:</p> 
              <div class="highlight-ini">
               <div class="highlight">
                <pre><span></span><span class="na">mongo.mongoUri</span><span class="o">=</span><span class="s">mongodb://&lt;new_mongodb_user&gt;@mydb1.example.com:40000/?authMechanism=MONGODB-X509</span>
</pre>
               </div> 
              </div> </li> 
             <li><p class="first"><strong>LDAP Authentication:</strong> For a MongoDB instance using LDAP, prefix the hostname with the MongoDB username and password in the form <code class="docutils literal"><span class="pre">&lt;username&gt;:&lt;password&gt;@</span></code>, and append the <code class="docutils literal"><span class="pre">authMechanism=PLAIN&amp;authSource=$external</span></code> options after the port:</p> 
              <div class="highlight-ini">
               <div class="highlight">
                <pre><span></span><span class="na">mongo.mongoUri</span><span class="o">=</span><span class="s">mongodb://mongodbuser1:password@mydb1.example.com:40000/?authMechanism=PLAIN&amp;authSource=$external</span>
</pre>
               </div> 
              </div> </li> 
             <li><p class="first"><strong>Kerberos Authentication:</strong> For a MongoDB instance using Kerberos, prefix the hostname with the Kerberos user principal and specify the authentication mechanism, <code class="docutils literal"><span class="pre">authMechanism=GSSAPI</span></code>, after the port.</p> <p>Kerberos user principal names have the form <code class="docutils literal"><span class="pre">&lt;username&gt;@&lt;KERBEROS</span> <span class="pre">REALM&gt;</span></code>. You must escape the user principal, replacing symbols with the URL encoded representation. A Kerberos user principal of <code class="docutils literal"><span class="pre">username@REALM.EXAMPLE.COM</span></code> would therefore become <code class="docutils literal"><span class="pre">username%40REALM.EXAMPLE.COM</span></code>.</p> <p>This is an example of Kerberos authentication:</p> 
              <div class="highlight-ini">
               <div class="highlight">
                <pre><span></span><span class="na">mongo.mongoUri</span><span class="o">=</span><span class="s">mongodb://username%40REALM.EXAMPLE.COM@mydb1.example.com:40000/?authMechanism=GSSAPI</span>
</pre>
               </div> 
              </div> <p>To enable Kerberos authentication between the Ops Manager Application and the <a class="reference internal" href="../../core/system-overview/#backup-database"><span class="std std-ref">Backup Data Storage</span></a>, see <a class="reference internal" href="#configuration-kerberos-settings"><span class="std std-ref">Kerberos Authentication to the Application Database</span></a>.</p> 
              <div class="admonition seealso"> 
               <p class="first admonition-title">See also</p> 
               <p class="last"><code class="xref mongodb mongodb-data docutils literal"><span class="pre">authMechanism</span></code> and <code class="xref mongodb mongodb-data docutils literal"><span class="pre">authSource</span></code> in the MongoDB manual.</p> 
              </div> </li> 
            </ul> 
           </dd>
          </dl>

          <h2>Encrypt User Credentials<a class="headerlink" href="#encrypt-user-credentials" title="Permalink to this headline">¶</a></h2> 
          <p>For configuration settings that store credentials, you can either store the credentials in plain text or use the Ops Manager <code class="docutils literal"><span class="pre">credentialstool</span></code> to encrypt the credentials. If you choose to store credentials in plain text, reduce the permissions on the <a class="reference internal" href="#conf-mms-properties"><span class="std std-ref">conf-mms.properties</span></a> file on each server.</p> 
          <div class="admonition note"> 
           <p class="first admonition-title">Note</p> 
           <p><strong>Protect Plain Text Passwords</strong></p>

               <div class="last highlight-sh">
                <div class="highlight">
                 <pre><span></span>mongo.mongoUri<span class="o">=</span>mongodb://da83ex3s:a4fbcf3a1@mydb1.example.net:40000/admin
mongo.encryptedCredentials<span class="o">=</span><span class="nb">true</span>
</pre>
                </div> 
               </div> 
              </div> 
              <div class="admonition important"> 
               <p class="first admonition-title">Important</p> 
               <p class="last">The <code class="docutils literal"><span class="pre">conf-mms.properties</span></code> file can contain multiple <code class="xref mongodb mongodb-setting docutils literal"><span class="pre">mongo.mongoUri</span></code> settings. If <code class="xref mongodb mongodb-setting docutils literal"><span class="pre">mongo.encryptedCredentials</span></code> is <code class="docutils literal"><span class="pre">true</span></code>, you must encrypt all user credentials found in the various <code class="xref mongodb mongodb-setting docutils literal"><span class="pre">mongo.mongoUri</span></code> settings.</p> 
              </div> </li> 
            </ol> 
           </div>
